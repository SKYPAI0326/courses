# 操作型講義 — 課程內容組織原則 SSOT

> 本檔為**內容組織原則的人類閱讀真相源**。當課程要採「低原理、高操作」模式時，所有改寫遵守本檔。
> 本檔與 `design-tokens.md`（視覺真相源）並存：視覺仍守 V4 CCS 無印風，本檔只規範**內容組織與示範元件**。
> 由 2026-05-21 `office-ai` 課程改寫經驗萃取而來（commit 1a79f81–1861cd2，8 個 commits）。

---

## 0. 何時採用

### 適用情境
- **商業培訓**（非自學工程師教育）：學員真實訴求是「結果」非「原理」
- **零基礎受眾**：學員旁邊沒有 Claude，看不懂原理散文
- **工具操作為主**：Gemini / NotebookLM / 各種 SaaS / no-code 工具教學
- 課程目標是「明天能用在工作上」而非「理解運作機制」

### 不適用情境
- 認證考試準備（如 iPAS、CCS）：考試會考原理，必須教
- 工程師技術深度課程：需要建立心智模型
- 學術導向課程

判斷口訣：**「課堂跑得動 ≠ 回家跑得出來」是核心擔憂時，採操作型講義。**

## 1. 核心原則

### 1.1 商業培訓哲學
- 學員真實訴求是**結果**，不是原理
- 禁止套用「學員應該想學原理」假設
- 「為什麼」可以保留，但要砍到一句話以下；超過一句話就改為「實例示範」

### 1.2 教學動線
```
傳統：原理講解 → 範例 → 練習
操作型：示範完整流程 → 學員模仿 → 一句話 takeaway
       ↑ AI 對話框 + 預期輸出 ↑
```

### 1.3 三大要素
每個改寫單元至少要有：
1. **可直接複製的 prompt**（在 dialog-box 內）
2. **AI 大概會回什麼的示範**（在 expected-output 內）
3. **≤ 50 字的 takeaway 或 punchline**（替代原理段）

## 2. 三件套元件（HTML / CSS 規格）

### 2.1 元件家族總覽

| Class | 用途 | 與既有元件區隔 |
|------|------|--------------|
| `.dialog-box` | 教師示範要貼進 AI 的 prompt | `drill-prompt-box` 留給「學員自己貼」 |
| `.expected-output` | 模擬 AI 會回什麼（含 `.expected-note` 提醒實際輸出每次略有差異） | `output-box` 留給「帶走產物」（表格、模板） |
| `.demo-flow` | 多輪對話完整示範（含 `.demo-step` / `.demo-takeaway`） | `step-list` 留給「純步驟」、`drill-card` 留給「對照演練」 |

### 2.2 DOM 結構範本

**dialog-box（單個 prompt 示範）**：
```html
<div class="dialog-box">
  <div class="dialog-label">背景</div>
  我是行政人員，需要寫通知信給全體員工
</div>
```

**expected-output（必含 expected-note）**：
```html
<div class="expected-output">
  <div class="expected-label">Gemini 大概會回</div>
  <div class="expected-body">
    <p><strong>主旨：</strong>...</p>
    <p>...</p>
  </div>
  <div class="expected-note">實際輸出每次略有差異，重點是結構與完整度。</div>
</div>
```

**demo-flow（多輪對話 + takeaway）**：
```html
<div class="demo-flow">
  <div class="demo-flow-label">教師示範 · 一份逐字稿走完整流程</div>
  <div class="demo-step">
    <div class="demo-step-title">第 1 步：拿一份逐字稿</div>
    <!-- dataset-link 或 dialog-box -->
  </div>
  <div class="demo-step">
    <div class="demo-step-title">第 2 步：貼進 NotebookLM</div>
    <!-- dialog-box（教師示範要貼什麼） -->
    <!-- expected-output（AI 大概會回什麼） -->
  </div>
  <div class="demo-takeaway">≤ 50 字的精煉版原理 punchline</div>
</div>
```

### 2.3 視覺規格紅線（守 V4 CCS）

- **底色**：純白底 + 1px 細邊 + 左色條（仿 `drill-card` 視覺語法）
- **左色條色彩**：
  - dialog-box: `var(--c-text)` 深色 = 使用者輸入
  - expected-output: `var(--c-a2)` 灰藍 = AI 回應
  - demo-takeaway: `var(--c-a1)` 橘色 = 重點
- **禁**：`box-shadow`、`gradient`
- **border-radius**：≤ 8px（用 `var(--radius-sm)` = 3px 或 `var(--radius)` = 4px）
- **字型**：只用 V4 15 階白名單（見 `design-tokens.md` §2）
  - dialog-box body：`font-family: 'Courier New', monospace; font-size: .8rem`
  - expected-output body：`font-family: 'Noto Sans TC', sans-serif; font-size: .85rem`
- **lint 紅線**：
  - `demo-takeaway` **不掛 callout class**（避開「每頁 callout ≤ 4」上限）
  - `expected-output` 內模擬商業書信時，「您」一律改「你」（避免觸發 `check_you_form` WARN）

## 3. 改寫範式（per 單元 SOP）

### 3.1 五步流程

```
Step 1：識別「待砍候選」段落
   ↓
Step 2：套用「原理 → 操作」轉換公式
   ↓
Step 3：寫 expected-output（刻意留瑕疵）
   ↓
Step 4：跑 lint 驗證
   ↓
Step 5：小 commit（每完成一批）
```

### 3.2 Step 1 — 識別待砍候選

掃描單元頁，凡符合以下三 marker 之一的段落列為候選：
1. **解釋型 body-text**：以「為什麼」「原理是」「之所以」「這是因為」「背後的邏輯」開頭
2. **連續 3 段以上的純散文**（無 grid、無 callout、無 drill）
3. **概念清單型 grid**：若該主題在前面章節介紹過，可降為一句話 + 連結

**判斷原則**：若這頁已偏操作（drill / step-list / output-box / error-grid 齊全），**不要硬砍原理**，改用「補強示範」（在 drill 後加 expected-output）。

### 3.3 Step 2 — 轉換公式

| 原理段元素 | 對應的操作版產出 |
|----------|----------------|
| 概念名詞（如「三要素」） | `demo-flow-label` 標題化，**不寫定義** |
| 「為什麼要這樣」（原理） | 砍掉，移到 demo 結束後的 `demo-takeaway`（≤ 50 字 punchline） |
| 抽象例子（「比方說……」） | 換成具名情境（給對象、時間、地點、字數） |
| 規則條列（「要明確、要具體、要限制」） | 改為多輪 `demo-flow`，每輪示範一條規則被應用 |
| 延伸閱讀 | 砍 |

### 3.4 Step 3 — expected-output 寫作原則

- **刻意留瑕疵**：保留 `[請填入主題]` / `[請補上聯絡窗口]` 等 placeholder，反映真實 AI 第一次輸出狀態
- **必含 `.expected-note`**：提醒「實際輸出每次略有差異」，避免學員誤以為截圖
- **A/B 對比**：當改寫的是 drill 的 A/B 版 prompt 時，兩版都要有 expected-output，並列展示「模糊指令 vs 結構化指令」的真實落差
- **長度**：每個 expected-output 內文約 50-150 字（不要寫成完整論文）
- **語氣**：模擬真實 AI 口吻（會用條列、會用粗體、偶爾自作主張加東西）
- **禁用「您」**：商業書信慣例用「您」，但 lint 會 WARN。一律改「你」（避免 Style Guide 違規）

### 3.5 Step 4 — lint 驗證

```bash
python3 docs/lint-page.py courses/<slug>/<file>.html
```

放行條件：
- **Exit 0**（0 BLOCKER）
- **WARN 數量**比 baseline 增加 ≤ 3（特別注意字型尺度、callout 數量、「您」字）
- 若 WARN 暴增，**立即檢修不要 commit**

### 3.6 Step 5 — 小 commit 規則

每改完一批（4 頁左右）跑完 lint 即 commit：

```
refactor(<slug>/partN): low-theory rewrite (batch N/5)

- CHN-1: drill 加 A/B 版 expected-output（描述 A/B 差別）
- CHN-2: ...
- PRACN: 在工具上方加教師示範 demo-flow

lint: 全 N 頁 0 BLOCKER 0 ERROR，X WARN 全為既有債務（無新增）。
```

方便逐批 revert 而不影響前批。

## 4. 課後素材包（offering）

操作型講義必附四件素材包，補上「課堂跑得動 ≠ 回家跑得出來」的缺口。

### 4.1 必備檔案結構

```
courses/<slug>/assets/
├── after-class-guide.md            # 30 分鐘復跑手冊（最重要，featured 卡片）
└── datasets/
    ├── <topic1>-sample.md           # 真實樣本（給 NotebookLM / 練習用）
    ├── <topic2>-sample.md
    └── prompts-all.md               # 全課程 Prompt 純文字總集
```

### 4.2 after-class-guide.md 範式

300 行左右，結構：

```markdown
# 回家 30 分鐘復跑手冊

> 課堂上跟著老師按了幾下，回家用真實工作資料跑不出來？...

## 共同前置：第一次跑前確認 3 件事（5 分鐘）
1. 資料份量
2. 資料敏感度
3. 工具選擇

## 案例 ① <主題> — 30 分鐘食譜
### 步驟 1：整理資料（5-10 分鐘）
### 步驟 2：套 prompt（3 分鐘）
### 步驟 3：跑 AI 並讀（3 分鐘）
### 步驟 4：追問修正（10 分鐘）
### 步驟 5：驗證輸出（5-9 分鐘）— **這步絕對不能省**

## 案例 ② ... ⑤ ...（每 Part 一個案例）

## 共同尾聲：跑出問題時的 3 個求救步驟
## 最重要的一個原則：AI 是助手不是主管，簽核責任永遠在你
```

### 4.3 datasets 樣本原則

- **虛構但真實感**：人名、公司名、數字均虛構，但結構模擬真實工作場景
- **含驗證點**：刻意埋「AI 容易誤判的陷阱」（如週報樣本含「結婚 8 週年晚餐」測試 AI 能否過濾私人事項）
- **檔尾附練習任務**：列「預期應該抓到什麼」+「驗證重點」

### 4.4 prompts-all.md 結構

## 5. 執行流程（per 課程 SOP）

### 5.1 第 0 批：前置（0.5h）

1. **git baseline**：執行改寫前 commit 一次乾淨基線
   ```bash
   git status --short courses/<slug>/  # 應為 clean
   ```
   若 clean 則直接用當前 HEAD SHA 作 baseline；若有變動則先 commit 一次

2. **本地副本**：
   ```bash
   cp -R courses/<slug> _backup_YYYYMMDD/<slug>
   ```
   寫 `_backup_YYYYMMDD/README.md` 記錄 baseline SHA + rollback 指令

3. **確認 `.gitignore` 含 `_backup_*/`**（避免誤 commit 巨檔）

4. **CSS 原型驗證**：選一頁（建議課程「最原理重」的單元）落地 dialog-box / expected-output / demo-flow 三件套 CSS，跑 lint 確認無新增 BLOCKER

### 5.2 第 1 批：樣板批（4-5h）

選最原理重的 Part 開做，含 3 個 CH + 1 PRAC（4 頁）。

**Stop & confirm 點**：第 1 批完成後**必須暫停**，由使用者目視確認對話框視覺風格。同意後才放第 2 批。視覺要改只改 1 頁，不擴散。

### 5.3 第 2-5 批：依優先順序推進其他 Part

優先順序判斷：
- **次高優先**：抽象度高、學員最容易卡關的 Part（如 office-ai 是 Part 3 NotebookLM）
- **中優先**：已偏操作但缺 expected-output 的 Part（主要是補強）
- **最後做**：邊際效益最低、最成熟的 Part

每批仍 4 頁左右，跑完 lint commit。

### 5.4 第 6 批：收尾（1-1.5h）

```bash
python3 docs/lint-page.py courses/<slug>/
python3 docs/check-integrity.py
python3 docs/build-search-index.py
python3 docs/build-sitemap.py
git add search-index.json sitemap.xml
git commit -m "chore(<slug>): rebuild search-index + sitemap (batch 6/6 收尾)"
git push
```

### 5.5 端到端驗證

最後跑這 6 項：

| # | 檢查項 | 通過標準 |
|---|--------|--------|
| 1 | 對話框 vs 預期輸出視覺一眼分得出 | 兩者邊框色 + 字體差異明顯（等寬 vs 一般） |
| 2 | demo-flow 在手機 viewport（< 768px）不爆版 | 對話框寬度自適應，文字不橫向滾動 |
| 3 | demo-takeaway 字數合理 | 抽 3 個數一下，平均 ≤ 50 字 |
| 4 | grep 該頁無「為什麼／原理是／之所以」開頭段落 | `grep -E "^.*<p.*>(為什麼|原理是|之所以|這是因為)" *.html` 應無結果 |
| 5 | 該頁仍有 ≥ 5 個 `.lesson-section` | lint 不再 WARN「lesson-section 僅 N 個」 |
| 6 | 既有 drill-card / tool-grid / faq-list 未被誤刪 | 對照 baseline 元件清單 |

## 6. 風險與緩解

| 風險 | 緩解 |
|------|------|
| 原理砍過頭，零基礎看不懂 | 每個 `demo-flow` 結尾必有 ≤ 50 字 `demo-takeaway` 作精煉版原理 |
| 學員以為 expected-output 是真實截圖 | 強制 `.expected-note` 提醒 + 等寬字 + 純白底（非氣泡風） |
| 多頁對話框風格漂移 | 第 1 批的樣板頁固定後，後續每批開工前對照樣板 |
| lint WARN 暴增（特別是「您」字、字型尺度、callout 上限） | expected-output 內統一用「你」；新 CSS 嚴格用 V4 15 階；`demo-takeaway` 不掛 callout class |
| AI 示範回應太完美 → 學員實際操作後失望 | 示範刻意留 `[請填入]` placeholder 與「需再補」標註 |
