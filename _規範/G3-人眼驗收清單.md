# G3 人眼驗收清單

**用途**：G3 講義驗收（每 Part 結束）時，由使用者（課程總監）逐條打勾，**lint 過不代表 G3 過**。
**適用**：每完成一個 Part 的講義（3-5 單元）後跑一次。
**原則**：lint 擋結構與合規，這份擋視覺與敘事。

---

## 使用方式

```
cd courses/<slug>/
cp ../../_規範/G3-人眼驗收清單.md ./_gates/G3-part{N}.md
# 開該 Part 全部 HTML，逐條打勾；未過者回頭修
```

或直接在 `_gates.md` 追加 G3-partN 段落，不另開檔。

---

## 第一層：排版視覺（打開前 3 秒的印象）

- [ ] **字型階一致**：同 Part 所有頁的 h1 / h2 / h3 / body 大小一致（不因單元內容長短而亂跳）
- [ ] **色塊平衡**：每頁 accent 色（`--c-a1`~`a5`）使用密度不過量，視覺重心落在標題與 CTA，不落在背景色塊
- [ ] **段落呼吸**：`section` 之間間距一致，不會某頁擠滿、某頁空曠
- [ ] **無孤兒欄**：grid 排版沒有 3+1 孤兒、2+1 孤兒（若類型為 concept-scenario-grid 或 tool-grid）
- [ ] **對齊一致**：同類元件（callout / card / step）左右邊界對齊，不會單頁突出

## 第二層：敘事連貫（跟讀測試 3 分鐘）

- [ ] **隨機點 1 頁跟做 3 分鐘**：能順到第幾步？卡住就是敘事有洞
- [ ] **案例銜接**：前一單元的案例（阿凱/雯姊）在下一單元有延續或明確結束，不會突兀切換
- [ ] **銜接語自然**：「接下來我們」「承上一節」這類語句分布合理，不過量也不缺失
- [ ] **Callout 使用節制**：info/tip/warn 三色 callout 平均分配，不會一頁五個 warn
- [ ] **大標題不只是裝飾**：每個 `<h2>` 下的內容真的在講那個主題，不是湊字數

## 第三層：AI 味偵測（用肉眼找違和）

- [ ] **無條列爆炸**：沒有「首先...其次...再者...最後」這種學術腔
- [ ] **案例不虛構**：阿凱/雯姊的具體行動可驗證（例如「他打開 Google Trends 搜尋『AI 行銷』」而非「他使用工具分析資料」）
- [ ] **數字有意義**：提到「提升 30%」「節省 2 小時」時，能跟上下文對得起來，不是隨便填
- [ ] **示意圖不說謊**：AI 生成的截圖 / 流程圖若含錯誤的介面 / 按鈕名，必須替換或標「示意圖」
- [ ] **無重複開場**：同 Part 內多個單元不會用同一種破題句型（例「你是否遇過...」）

## 第四層：試跑包驗證（按類型）

根據 `_outlines/{slug}.md` 的 `course_type` 跑對應驗證：

### A 類 skill-operation
- [ ] 試跑包 JSON 可直接匯入（不需手動改 endpoint / credential 名）
- [ ] `credential setup` 清單完整（列出要準備的 API key / OAuth）
- [ ] 至少一位非作者試匯入成功

### B 類 concept
- [ ] 題庫 / 分類練習有標準答案 + 解釋
- [ ] 策略範本可填寫、欄位不含歧義
- [ ] 隨機抽題作答，答案與講義一致

### C 類 programming
- [ ] `.py` 在乾淨環境（`venv`）可跑（requirements.txt 完整）
- [ ] 範例資料 / 測試輸入在包裡
- [ ] OAuth / credential 範本有 placeholder 且註解清楚

## 第五層：SEO / 連結 / 行為（lint 補強）

- [ ] **側邊導覽展開狀態對**：該 Part 在 module 頁是 active 的
- [ ] **前/後頁連結對**：上一頁 / 下一頁不會跳錯 Part
- [ ] **`<title>` 與 `og:title` 一致且描述具體**（非「課程講義 - CH1-1」這種通用字）
- [ ] **手機 viewport 可讀**：把視窗縮到 375px，字不會擠、按鈕可按
- [ ] **Progress bar 行為正常**：滾動到底顯 100%、localStorage 記得上次位置

---

## 評分原則

每層分三級：
- **Pass**：清單全勾
- **Soft-fail**：3 項以內未勾，記錄在 `_gates.md` 下一 Part 前修完
- **Hard-fail**：4 項以上未勾，或任一「試跑包驗證」未過，退回 G3 重產

Hard-fail 時，lesson-writer subagent 需重派，**不是**由 reviewer 出 issue list 修補 — reviewer 檢不到這層。

---

## 進化原則

每次 G3 跑完，若發現新的違和模式：
1. 能寫成 lint 規則 → 加進 `docs/lint-page.py`，本清單刪掉該條
2. 不能寫成 lint（純主觀判斷）→ 留在本清單
3. 若是 AI 味新套路 → 補進第三層 + 進飛輪規則

目標：**本清單每過一輪 G3 應變短，不變長**。
