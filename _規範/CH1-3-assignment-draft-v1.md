## 折疊參考解答（assignment-answer · 建議自己先試 12 分鐘再開）

### 診斷表（精簡：主病因 1 + 次病因 ≤ 1）

| 版本 | 主病因 | 次病因 | Lina 修了什麼 | 修對方向？|
|------|--------|--------|--------------|-----------|
| v1 | **太空泛**（沒定義什麼是待辦）| 格式錯（散文） | — | — |
| v2 | **不準確**（編造「維持氣氛」「持續溝通」）| 太冗長（湊很多無意義條目）| 加「要寫負責人和截止日期」← 只補了**輸出限制**，沒補**任務本質**（待辦定義）| ❌ **跳過求對直接求好**：以為加欄位就能解決 |
| v3 | **不準確**（編造 team retreat、brand guidelines）| **格式錯**（中英混用 + deadline 模糊 Q2/TBD）| 加「5 條最重要 + JSON 格式」← 補了**數量 + 結構**，但**限制數量反而誘導 AI 湊數編造**| ⚠ **仍不對**：Lina 兩輪都在「求好」未先「求對」 |

### Lina 哪一輪跳過「先對」？

**兩輪都跳過了**。「先對」的核心是「**先讓 AI 知道什麼是真正的待辦、什麼不是、缺資訊怎麼處理**」。

- v2 補了輸出欄位 → 是「求好」（讓輸出可用）
- v3 補了數量 + JSON → 是「求好/求快」混合
- 兩輪都沒處理「**待辦定義 + 不得編造**」這個本質性「先對」問題

正確順序應該是：v2 先補「待辦定義 + 不得編造 + 缺資訊填 null」（先對），v3 才能談「JSON 格式 + 5 條限制」（求好/求快）。

### v4 關鍵規則（核心交付答案）

要補的 6 條規則：

1. **角色定位**：「你是會議助理，**只整理會議裡明確說過的事，不得編造**」
2. **待辦定義**：「待辦 = 動作 + 負責人 + 大概截止，**且會議中明確被決議**的事項」
3. **不得補常識**：「**不從常識推測**團隊應該做什麼」（這是 Codex 點出的關鍵：「不得編造」太抽象，要寫「不得補常識」更可操作）
4. **缺資訊處理**：「owner/deadline 會議沒明說 → 填 `null`，由 Lina 補；**禁止用 TBD、Q2、Next month、End of quarter 這種模糊時間**」
5. **語言一致性**：「task 用中文（會議是中文進行），其他欄位英文（符合 Asana 慣例）」
6. **未決議分流**：「『待議』『討論未結論』的事項放 `pending_decisions` 陣列，不塞進 tasks 湊數」

### 完整 v4 prompt（進階交付）

```
【角色】你是松果科技的會議助理，擅長把口語化的會議錄音逐字稿轉成可匯入 Asana 的結構化待辦清單。**你只整理會議裡明確說過的事，不得編造，不得從常識推測團隊應該做什麼。**

【任務】我會貼上 1 小時上週新品發表會復盤會議的逐字稿（約 8000 字），請你抽出**真正的待辦事項**，輸出可直接匯入 Asana 的 JSON 格式。

【限制】
- **待辦定義**：必須有「動作 + 負責人 + 大概截止」三要素**且會議中明確被決議**的事項
- **不得補常識**：禁止憑「公司應該做的事」推測（如 team retreat、brand guidelines 這類沒在會議出現的議題一律不列）
- **缺資訊處理**：owner/deadline 會議沒明說就填 `null`，由 Lina 補
- **禁止模糊時間**：deadline 必須是 YYYY-MM-DD 或 null。**禁止 TBD、Q2、Next month、End of quarter 這類模糊表達**。會議明確說「下週」「下個月底」可推算成 YYYY-MM-DD
- **語言一致性**：task 用中文，其他欄位英文
- **未決議分流**：「待議題」「討論未結論」放 `pending_decisions` 陣列，**不塞進 tasks 湊數**

【輸出格式】
{
  "tasks": [
    {
      "task": "（中文，動詞開頭，≤ 30 字）",
      "owner": "（會議裡提到的人名，沒提填 null）",
      "deadline": "YYYY-MM-DD or null",
      "priority": "High|Medium|Low",
      "context": "（會議裡的決議原話節錄 ≤ 30 字）"
    }
  ],
  "pending_decisions": ["（未決議事項，待 Lina 補）"]
}

【AI 自我檢查 — 進階加分】
產出 JSON 後，請依以下 3 條句型逐項自檢。**任一不符合請重寫**：
1. 「**輸出前檢查每個 task 的 context 是否能在我貼的逐字稿裡找到原文？**」若找不到 → 該條移到 pending_decisions
2. 「**owner 與 deadline 是否真的是會議裡明確說的？**」若不確定 → 填 null
3. 「**task 是否都用中文、是否都是動詞開頭？**」若不是 → 改寫
```

### 常見錯誤（自我比對 · 5 條）

1. **誤診 v1 為「不準確」**：v1 還沒講具體事，怎麼算不準確？v1 主病是「太空泛」（什麼是待辦完全沒定義）。
2. **以為 v2→v3 是進步**：表面看格式變好，但實際**編造變嚴重**（從會議邊緣議題的延伸 → 完全不在會議的虛構議題）。注意「越改越好看」≠「越改越對」。
3. **v4 只補 JSON schema**：缺「不得編造 / 不得補常識 / 缺資訊填 null」這三個本質規則 → v4 還是會編造。
4. **AI 自我檢查寫成「請檢查格式」**：太模糊。要寫**自檢句型具體**（如「輸出前檢查每個 action 是否有逐字稿依據」）。
5. **跳過診斷直接寫 v4**：學員看到 Lina 抱怨就想動手寫，沒先理解病因 → 容易重蹈 Lina 覆轍（「我也補欄位了，怎麼還是爛？」）。

- v3：「字數限制 80 字，要熱情真誠」→ 變得肉麻、像詐騙信（**主管常見誤修：用模糊形容詞「熱情真誠」**）

學員診斷 v1/v2/v3 病因 — 特別是「**不照角色**」這條（Lina 一直沒設「Lina 本人」這個有 NDA 與技術背景的角色）+ 寫 v4。

**變體 2｜跨模型診斷對比**

同一個 prompt（v1 或 v2）分別丟給 ChatGPT、Claude、Gemini，比較它們各自會犯什麼病因 — 不同模型的「壞」方式不同：
- ChatGPT 容易**太冗長**（喜歡解釋）
- Claude 容易**太謹慎**（什麼都不確定 → 拒絕做）
- Gemini 容易**格式不一致**（有時表格有時段落）
