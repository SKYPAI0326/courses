## 情境敘述（lede · 約 290 字）

延續上週 — 你是松果科技的內容運營專員。今天主管 Lina 又丟一份檔案給你：「我這幾個 prompt 跑出來都很爛，你剛上完 prompt 課對吧？幫我改一下，下午 3 點前我要拿這幾個 prompt 給整個團隊用。」

打開檔案，你看到 3 個爛 prompt：每個都對應一個重要職場任務。Lina 不是不會寫 prompt — 她週會四場、客戶 escalation 兩三組同時跑，**只丟意圖不丟規格**就點送。結果：每個 prompt 跑出來的東西都接近「能用但勉強」，貼到正式檔案前還要再改半小時，等於 prompt 沒幫上忙。

你今天剛學完**四要素公式（角色／任務／限制／輸出）+ 6 種輸出格式 + Few-shot 範例法**，正好用在這。但要做的不只是「套公式」這麼機械 — 每個 prompt 缺的東西不同，要逐個診斷再改寫；其中至少 1 個必須加 Few-shot 範例（因為涉及風格控制）。

## 本作業拆解

- **核心 20 min（必做）**：
  - 3 個爛 prompt 全部填寫**診斷表**（固定欄位）
  - 3 個 prompt 全部用四要素**重寫**
  - 任務 2 **必加 Few-shot**（風格控制）
  - 每個重寫版指定**輸出格式名稱**（不必寫完整 schema/欄位）
- **進階 20 min（可選）**：
  - 任務 1 寫完整表格欄位、任務 3 寫完整 JSON schema
  - **before/after 一句話比較**（每個 prompt）
  - 任務 1、3 也加 Few-shot
  - 100 字反思「四要素中我最容易忘哪個」
  - 變體 1 或 2

本作業範圍只涉及本節三個子技巧（四要素 + 輸出格式 + Few-shot），不需深入 Chain-of-Thought（CH1-2 後段教過但屬延伸）、System Prompt（CH1-4 才教）。

## 任務清單（assignment-tasks）

**核心交付（必做）**：

1. **逐個診斷**：用**固定欄位**填寫 3 個 prompt 的診斷表 — 角色 ／ 任務 ／ 限制 ／ 輸出 ／ Few-shot 是否需要
2. **用四要素重寫** 3 個 prompt（每個重寫版至少 80 字）
3. **任務 2 必加 Few-shot**：在重寫版裡放 1–2 個 input → output 對照範例（沿用「背景補充」給的範例 A、B）
4. **每個重寫版指定具體輸出格式名稱**：「Markdown 表格 / JSON / 條列 / SOP / Email」其中之一

**進階交付（可選）**：

5. **完整輸出 schema**：任務 1 寫表格欄位（含計算規則）、任務 3 寫 JSON schema
6. **before/after 比較**：用一句話說「我的版本比 Lina 的具體在哪」（每個 prompt 都寫）
7. **任務 1、3 也加 Few-shot** 提升品質
8. **100 字反思**：四要素中哪個我最容易忘？為什麼？
9. **變體 1 或 2**

## 評分標準（rubric · 基本 4 條 + 加分 3 項）

**基本要求**：

1. **四要素完整性**：3 個重寫版都明確涵蓋角色、任務、限制、輸出（不能只寫「請寫一份報告」這種片段）
2. **輸出格式具體**：核心交付至少寫格式名稱；進階交付要寫出**表格欄位**或 **JSON schema**
3. **Few-shot 範例品質**：任務 2 的 Few-shot 必須是「**input → output 對照**」型範例（不只是「好範例」），完整呈現語氣與結構
4. **改寫後比原版具體可執行**：你的 prompt 一貼到 ChatGPT/Claude 就能跑出可用結果，不需再三輪追問補充

**加分項**（本節未必都教到，會做更好）：

- ➕ **任務 1、3 也加 Few-shot**：報告類、待辦類加 1–2 個歷史好範例
- ➕ **變體 1：System Prompt 改寫**（CH1-4 才教）：把任務 1 的 prompt 拆成「角色 + 規則」(System) 與「具體本月資料」(User) 兩段
- ➕ **跨模型驗證**：把同一 prompt 跑 ChatGPT 和 Claude，比較哪個輸出更接近期望（呼應 CH1-1 工具選擇）

### 重寫版（含預期輸出對照）

**Prompt 1 重寫**：

```
【角色】你是松果科技 SaaS B2B 客戶成功 + 內容運營雙部門的數據分析助手。

【任務】根據我給你的 200 筆客戶月度數據（CSV 格式），產出一份月度健康度報告，主要目的是讓 CSM 主管 + 內容運營主管在 2 分鐘內找出「本週需立刻聯絡的高風險客戶」。

【限制】
健康度評估維度（按重要性排序）：
1. 內容互動量（blog_views + onboarding_email_open/6 + in_app_guide_uses）
   → 加總後 ≥ 10 = 高 / 5–9 = 中 / < 5 = 低
2. 產品活躍度（logins + core_feature_usage）
   → logins ≥ 20 且 core_feature_usage ≥ 50% = 高 / logins ≥ 10 或 ≥ 30% = 中 / 都不到 = 低
3. 付費狀態：paid > trial > overdue > churned
4. 客訴次數：0 = 健康 / 1–2 = 注意 / ≥ 3 = 警示

續約風險判定規則：
- 高風險：以下任一 — 內容互動量低 + 產品活躍度低 ｜ overdue 狀態 ｜ 客訴 ≥ 3
- 中風險：內容互動量中 OR 產品活躍度中 OR 客訴 1–2 次
- 低風險：以上都不符合

「需立刻聯絡」定義：續約風險高 AND（產品活躍度 < 10 logins OR overdue OR 客訴 ≥ 3）

不要寫散文段落，要表格 + 排序。**不得編造資料** — 只能用我給的 5 筆，缺的欄位填 `unknown`。

【輸出格式】Markdown 表格，欄位依序：
| 客戶名 | 內容互動 | 產品活躍 | 付費 | 客訴 | 風險等級 | 建議行動 | 緊急度 |
表格依「緊急度」由高至低排序。表格後寫一句話「本月需立刻聯絡：N 個客戶」。
```

### Before/After 對比（每個 prompt 一句話）

- **P1**：Lina 版會給 800 字散文（不能給主管看），我的版會給含計算規則的緊急客戶表格（直接行動）
- **P2**：Lina 版會給「打官腔的歉信」，我的版加了 input→output 對照 Few-shot 後會接近 Lina 自己的真誠風格
- **P3**：Lina 版會給「會議概要」（沒法匯入），我的版直接給 Asana JSON 結構（含 null 容忍）

### 常見錯誤（自我比對 · 6 條）

1. **四要素缺漏**：只寫 3 個要素（最常缺「角色」，學員以為 AI 自己會猜身分）
2. **輸出格式只說「用表格」**：沒寫表格欄位 = 等於沒指定
3. **Few-shot 沒呈現輸入輸出對照**：只給「好範例」沒給對應的 input，AI 學不到「什麼樣的 input → 應該長什麼樣的 output」（建議 2–3 個 input→output 對照）
4. **Few-shot 範例品質差**：範例本身就寫得普通，AI 模仿出來也只是普通
5. **限制沒有可驗證標準**：寫「要簡潔」「要專業」← 沒有可量化的判斷規則。要寫「不超過 120 字」「不用 Sorry for the inconvenience」「risk 等級定義為 X+Y+Z」這種具體規則
6. **沒驗證 prompt 真的能跑**：寫完不丟去 AI 跑就交，常見「結構漂亮但實際輸出爛」

## 變體（assignment-variants · 想再練？）

**變體 1｜System Prompt 改寫（CH1-4 預習）**

把 Prompt 1（月報）拆成兩段：
- **System Prompt**（給 GPTs / Claude Project 的「角色 + 規則」固定指令）：包含角色、評估維度、判定規則、輸出格式 schema
- **User Prompt**（每月跑時貼的「具體本月資料」）：只貼 200 筆 CSV

挑戰：判斷哪些屬於「每次都一樣」（→ System）、哪些屬於「每次不同」（→ User）。

**變體 2｜跨模型語言對比**

把 Prompt 2（客訴回覆）的中文版重寫成英文版 prompt（含英文 input→output Few-shot 範例），分別跑 ChatGPT 和 Claude，對比：
- 哪個更貼近你想要的風格？
- 中文 vs 英文 prompt，哪個讓 AI 更容易學會風格？

挑戰：發現「prompt 語言會影響輸出品質」這個職場實務上重要但常被忽略的細節。
