# 案例操作型講義演練改造規格

> **文件性質**：LLM 執行規格。供 LLM 依循，把「既有講義」的演練段改造到 `office-ai-cases` 水準。
> **基準範本**：`courses/office-ai-cases/`（CH1-1 ~ CH2-2、`assets/datasets/`）。本規格的每條規則都可在該課對照到實例。
> **適用範圍**：**僅限案例操作型講義**。判定見 §1。概念講解型、工具操作型（n8n / gtm 流程串接）**不適用本規格**。
> **套用情境**：改造**既有**講義（已有 HTML）。不涵蓋從零設計新課。
> **製作日期**：2026-05-19
> **修訂**：2026-05-19 增補 §3.3 引導主線模式（Model B），來自 `gen-ai-140h/PRAC3-1` 試點與 Codex 審查回饋。

---

## 1. 適用判定（改造前必跑的 Gate）

開始改造前，先確認目標講義是「案例操作型」。**三條全中才適用本規格**：

1. 演練要求學員**把資料貼進／上傳到 AI 工具**（ChatGPT / Gemini / Copilot / Claude.ai 等），不是操作特定軟體介面。
2. 演練有**可重複的「資料 → prompt → 輸出」動線**，學員產出的是 AI 文字／表格輸出。
3. 輸出**需要被驗證**（數字對不對、有沒有漏看、有沒有捏造）。

**任一條不中 → 停止，回報「此講義非案例操作型，本規格不適用」，不要勉強套用。**

判定範例：
- ✅ 「用 50 條工作日誌跑出週報」「用 200 筆費用 CSV 抓異常」→ 適用
- ❌ 「在 n8n 拉一條 webhook → Sheets 的 workflow」→ 不適用（工具操作型）
- ❌ 「理解 Transformer 注意力機制」→ 不適用（概念講解型）

## 2. 改造總流程（6 步，依序執行）

```
Step 1  診斷現況   → 列出既有演練 vs 目標規格的差距清單
Step 2  確認改造範圍 → 把差距分「結構 / Prompt / 素材 / 驗證」四類，與使用者確認動工範圍
Step 3  選結構模型 → 依 §2.1 診斷選 Model A 或 Model B（見下方說明）
Step 4  改寫 Prompt  → 套 §4 prompt 設計規格
Step 5  製作/補素材  → 套 §5 素材技術規格
Step 6  自檢 + lint  → 跑 §7 DoD checklist + `docs/lint-page.py`
```

- **Step 1、2 完成後必須先回報差距清單與改造範圍**，待使用者確認才進 Step 3。不要診斷完直接動手。
- 改造**沿用目標課程既有的 CSS class 與組件樣式**（如 `office-ai-cases` 的 `.prompt-block` / `.step-card` / `.verify-card`）。不新增未在 `_規範/design-tokens.md` 白名單的組件；目標課若缺對應組件 CSS，回報缺口，不自行發明。
- 改造完成必跑 `python3 docs/lint-page.py <檔案>`，Exit 0 才算放行。

**Step 3 的兩個結構模型**（依 §2.1 診斷結果選一）：
- **Model A — 5 段重構**（§3.1–3.2）：目標演練多數結構項 ❌（無固定結構、無步驟卡、無驗證卡、內容單薄）→ 把演練重構為固定 5 段。
- **Model B — 引導主線疊加**（§3.3）：目標演練結構項多為 ✅、甚至已有延伸案例 / 情境演練 / Quiz / 互動工具，只是「素材可驗證 / answer key」缺、或學員回報「吸收力低、需要更多照著做的引導」→ **不可**用 5 段取代既有內容（會逼掉成熟內容、維護成本翻倍），改用引導主線模式。

### 2.1 Step 1 診斷對標表

逐項比對既有演練，每項標 ✅ 已達標 / ⚠️ 部分 / ❌ 缺：

| 診斷項 | 達標定義（office-ai-cases 基準） |
|--------|-------------------------------|
| 固定段落結構 | 每案例有「場景設定 → 示範操作 → Prompt 拆解 → 上機演練 → 結果驗證+變奏」5 段 |
| 虛構主角 | 場景設定有具名主角（姓名+年齡+職稱）與量化痛點 |
| 講師示範 | 有「預期輸出範例」可對照「成品長什麼樣」 |
| Prompt 變奏 | 至少含 基礎版 / 5 骨架完整版 / 1 個變奏 / 風險檢查版 |
| Prompt 註解 | 每套 prompt 有「骨架」標籤 + 「為何這樣寫」 |
| 通用性 | prompt 不綁單一平台，任一主流 AI 可跑 |
| 上機步驟卡 | 有編號 step-card，學員照走 |
| 結果驗證卡 | 有離場前固定打勾的驗證卡 |
| 素材可驗證 | 資料集內埋有可被算出/抓出的明確 pattern |
| answer key | 素材附標準答案檔供對照 |
| 帶走清單 | 末段列出學員手邊應有的素材/prompt/輸出 |

## 3. 目標結構規格

§3.1–3.2 為 **Model A（固定 5 段重構）**；§3.3 為 **Model B（引導主線疊加）**。依 §2 Step 3 選用。

**Model A** — 每個案例 = 1 個單元頁，固定 5 段 + 4 個輔助段。標準時長 **75 分鐘**。

### 3.1 段落規格表

| # | 段落 | 時長 | 必含元素 |
|---|------|------|---------|
| (01) | 場景設定 | 5 min | 具名主角、量化痛點、本課帶走預告、素材下載連結 |
| (02) | 示範操作 | 15 min | 講師用主版 prompt 跑通、預期輸出範例、點出輸出內的驗證機制 |
| (03) | Prompt 拆解 | 10 min | 多套 prompt（見 §4），每套含骨架標籤 + 為何這樣寫 |
| (04) | 上機演練 | 40 min | 編號 step-card、免費版使用提醒 |
| (05) | 結果驗證 + 變奏 | 5 min | 結果驗證卡、變奏清單、風險提醒 |
| 輔助 | 投影片素材 | — | 若課程有跨案例累積，放本案例貢獻的素材（見 §6）|
| 輔助 | 平台差異備註 | — | 檔案處理 / 常見限制 / 跑不出時，三列 |
| 輔助 | 常見問題 | — | 2-3 條 Q&A，針對本案例特有的失效模式 |
| 輔助 | 本課帶走清單 | — | 列出素材 / prompts / 輸出 / 投影片素材 |

**時長變體**：若一個 75 分鐘案例含多個獨立資料集與任務，可拆成「子場景 A/B/C 各約 25 min + 合併上機 30 min」（對照 `CH1-2`）。子場景內部仍各自走「示範 → prompt → 上機」的微結構。

### 3.2 各段內容規格

**(01) 場景設定**
- 開一個 `intro-band`：標題「{主角}今天的麻煩」，內文描述主角身分與具體困境。
- 主角規格：**姓名 + 年齡 + 職稱 + 所屬產業/部門**。痛點必須**量化**（例：「光整理就花 45 分鐘」）。
- 一段 `body-text` 點出「今天你要跟著{主角}做什麼」。
- 一個 `callout key`「這堂課你會帶走」：列素材檔名、prompt 套數、輸出範例、驗證卡。
- 一段 `body-text` 給**素材檔案下載連結**（`download target="_blank"`）。

**(02) 示範操作**
- `intro-band`「操作步驟」：說明講師會用哪一套 prompt 跑、跑完要觀察什麼。
- 一個 `callout info`：貼出**預期輸出範例**（不是真實 AI 輸出，是「成品該長什麼樣」的參考格式）。
- 一段 `body-text`：明確點出輸出裡的**驗證機制**（例：「本週報引用日誌 50 條」這行的作用）。

**(03) Prompt 拆解** — 見 §4。

**(04) 上機演練**
- `intro-band`「你要做什麼」：說明學員 40 分鐘後會產出什麼。
- `step-list`：每步一張 `step-card`（編號 + 標題 + 描述）。標準步序：
  1. 取得素材（下載連結 + 全選複製說明）
  2. 跑風險檢查 prompt（標「建議但可跳過」）
  3. 跑主版 prompt（標「最重要」，含「若輸出缺驗證行如何追問」）
  4. 跑對比版 prompt（引導學員比較兩版差異）
  5. 跑投影片素材版（若課程有跨案例累積）
  6. 用學員自己的資料試跑（標「鼓勵」）
- 一個 `callout tip`「免費版使用提醒」：各平台訊息上限、context 限制、付費版差異。

**(05) 結果驗證 + 變奏**
- `intro-band`「離場前 30 秒」。
- `verify-card` 結果驗證卡 — 見 §6.1。
- `callout`「本案例可以做的變奏」：對象變奏 / 場景變奏 / 平台 wrapper 變奏。
- `callout`「本案例最常見的風險」：至少含「捏造不存在的事項」+「機密外洩風險」。

### 3.3 Model B — 引導主線疊加（目標演練已成熟時）

**何時用**：§2.1 診斷後，目標演練頁的結構項多數 ✅、甚至已有延伸案例 / 情境演練 / Quiz / 互動工具，只是缺「素材可驗證 / answer key」，或學員回報吸收力低、需要更多照著做的引導。此時用 5 段取代既有內容是**降級**——改用本模式。

**模型**：單頁三層遞進，**不做兩套並列內容**。

```
① 引導主線（置頂、預設可見、低門檻）   ← 新增
② 進階練習池（原既有豐富內容，重貼標籤）  ← 保留，不刪不改結構
③ Quiz / 收尾                          ← 保留
```

**引導主線規格**：
- **置頂、預設完全可見**。禁止收進 `<details>` 折疊——需要支架的學員不會主動展開，把引導藏起來等於觸及不到目標族群。
- 一條主線只跑**一個**案例、一份資料集；時長壓縮版，建議 45 分鐘：場景設定 5 / 看示範 10 / Prompt 拆解 5 / 逐步上機 20 / 結果驗證 5。
- 5 段內容規格同 §3.2，但引導主線只承擔「導航 + 手把手」，**不重寫**既有案例內容——案例細節留在進階練習池。
- 主版 prompt 給**完整可複製版**（低門檻學員貼上即可跑），不要求學員自填骨架。
- 全程沿用目標頁既有 CSS class，不新增組件（對照 PRAC3-1：用既有 `callout` / `steps-list` / `ps-prompt` / `case-check` 即可組出 5 段）。

**進階練習池規格**：
- 原既有的步驟區 / 互動工具 / 延伸案例 / 情境演練 / 修錯練習 / Quiz **全部保留，不刪、不改結構**。
- 在原內容前加一個分隔標題（如「PRACTICE POOL · 進階練習池」），框定為「完成主線後的自選加練區」。
- 主線與進階池是同主題的不同深度，須在文案點出兩者關係，避免學員混淆「哪個才是正式路徑」。

**口徑一致性硬規則**（PRAC3-1 試點 + Codex 審查教訓）：
- 引導主線的 **prompt 規則、預期輸出範例、結果驗證卡** 三者的數字與判定口徑**必須完全一致**。
- 特別注意「待確認 / 邊界案例」：若 prompt 要求某類資料「不合併、標記待確認」，則預期輸出與驗證卡的計數**必須把待確認列獨立計**、不可混入主計數——否則照規則正確操作的學員反而過不了驗證卡。

**素材**：Model B 仍須照 §5 做完整資料集 + answer key + patterns——素材是引導主線與進階練習池**共用的底層 SSOT**，應先獨立完成再寫主線。

## 4. Prompt 設計規格

### 4.1 5 骨架（所有 prompt 的構件）

| 骨架 | 作用 |
|------|------|
| 角色 | 讓 AI 進入特定視角而非任意回答 |
| 資料 | 標明貼入的素材與其結構 |
| 任務 | 要 AI 做什麼，分區塊條列 |
| 輸出 | 格式、字數、語氣的硬約束 |
| 驗證 | 要 AI 自報引用筆數/覆蓋範圍，供快速揪漏看 |

### 4.2 Prompt 變奏矩陣（每案例必備）

每案例至少產出下列 prompt，編號規則 `P{章}-{節}-{序}`（例 `P1-1-2`）：

| 類型 | 骨架組合 | 用途 |
|------|---------|------|
| 基礎版 | 任務 + 輸出 | 給初次使用者，指令少、容易跑出結果 |
| 進階版（5 骨架完整版）| 角色 + 資料 + 任務 + 輸出 + 驗證 | **主版**，課堂主力 |
| 對象變奏 ×1–3 | 視對象調整 | 給主管精簡版 / 給同事協作版 / 給自己備忘版 |
| 風險檢查版 | 任務（先自報不確定性）| 跑主版前先讓 AI 盤點資料品質與幻覺風險 |
| 投影片素材版 | 任務 + 輸出（嚴格禁空話）| 若課程有跨案例累積，產出可貼進簡報的素材 |

### 4.3 每套 prompt 的 HTML 規格

沿用既有 `.prompt-block` 組件，每套必含 4 件：

```html
<div class="prompt-block">
  <div class="prompt-header">
    <span class="prompt-label">P{章}-{節}-{序} · {名稱}</span>
    <span class="prompt-id">骨架：{骨架組合}</span>
  </div>
  <div class="prompt-code" id="p{章}{節}{序}">{prompt 全文，資料區用 [貼上 {素材檔名} 全文] 佔位}<button type="button" class="prompt-copy-btn" onclick="copyPrompt('p{章}{節}{序}',this)">複製</button></div>
  <div class="prompt-why"><span>為何這樣寫：</span>{說明骨架選擇的理由 + 適用情境 + 風險提醒}</div>
</div>
```

- prompt 正文**不得綁定單一平台**（不寫「請用 ChatGPT…」）。平台差異一律收進「平台 wrapper 變奏」與「平台差異備註」段。
- 資料區一律用 `[貼上 {素材檔名} 全文]` 佔位，不把素材內容寫死進 prompt。
- 「為何這樣寫」必須說明**骨架選擇的理由**，不是複述 prompt 在做什麼。

## 5. 素材製作完整技術規格

### 5.1 素材類型與規模

| 類型 | 格式 | 規模建議 | 規模理由 |
|------|------|---------|---------|
| 純文字日誌型 | `.txt` | ≤ 1,500 字（約 50 條） | 須塞得進最低免費版 context（約 8K token），全程貼文字、免上傳 |
| 結構化資料型 | `.csv` | 30–500 筆 | 小檔（≤50 筆）可貼文字；大檔（200–500 筆）走上傳，需在演練註明 |
| 文件型 | `.md` | 1–2 頁 | SOP / 政策 / 會議記錄等供 AI 引用的參考文件 |
| 範本型 | `.md` | 1 頁 | 簡報模板等供學員套用的骨架 |

- 檔名小寫連字號 + 規模後綴：`expense-200.csv`、`work-log-5day.txt`、`ecom-orders-500.csv`。
- 全部放 `courses/{slug}/assets/datasets/`。

### 5.2 刻意埋 Pattern（素材設計核心）

每份資料集**必須刻意埋入 2–4 個可被學員算出／抓出的明確 pattern**，否則「結果驗證」無從驗起。

Pattern 設計四要求：
1. **可計算或可辨識**：能用 Excel 函數、樞紐表或肉眼掃出來。
2. **正解唯一**：埋入時就記下標準答案數字。
3. **對應一個觀察重點**：每個 pattern 要能在演練裡被一個 prompt 觸發。
4. **難度分層**：至少 1 個明顯（建立信心）+ 1 個需要追問才看得出（訓練驗證習慣）。

`office-ai-cases` 已埋的 pattern 範例（改造時照此設計新 pattern）：

| 素材 | 埋入的 pattern | 正解 |
|------|--------------|------|
| 費用 200 筆 | 大額異常、整數異常（99,999 / 49,999 / 999）、重複報帳、週末報帳 | 共 7 筆異常 |
| 業績 50 筆 | 月度斷崖 | 4 月對 3 月跌 -62.7% |
| 電商訂單 500 筆 | ABC 分布、長尾、季節性 | Top 5 佔比 96.5%、母親節檔期高峰 |
| 工作日誌 50 條 | 5 類事件混合（主管交辦/跨部門/客訴/自我學習/臨時插隊）| 引用 50 條、覆蓋 5 個日期 |

### 5.3 Answer Key 規格

凡含「數字 pattern」的資料集，**另存一份標準答案檔**：

- 命名：原檔名加 `-answer` 後綴（例 `expense-200.csv` → `expense-200-answer.csv`）。
- 內容：原始資料 + 一欄標註（哪幾筆是異常、屬哪類異常）。
- 用途：講師對照、L2/L3 驗證、course-validator 真跑驗收。
- answer key **不放進學員下載清單**，僅供講師與驗證使用。

### 5.4 三層驗證包規格

凡「數字結果要交給主管」型案例，附一份驗證包 `.md`（對照 `number-validation-pack.md`）：

| 層級 | 驗證對象 | 工具 | 內容 |
|------|---------|------|------|
| L1 自查 | AI 是否誠實列計算公式 | 反問 prompt | 提供 4 套反問：算法公式 / 來源資料 / 邊界情況 / 不確定處 |
| L2 抽樣 | 隨機抽 3–5 筆是否對 | 計算機 / Excel 函數 | 每案例給「抽查項 + Excel 函數 + 通過標準」表 |
| L3 全量 | 總額 / Top 1 / 比例 | Excel / Sheets 樞紐表 | 每案例給樞紐表 SOP 步驟 |

驗證包另含：
- 「常見『AI 算錯』清單」：錯誤模式 + 警示徵兆 + 修正方式。
- 「主管核可前 N 個檢查問題」：交付前自問的 yes/no 清單。

## 6. 驗證與串連規格

### 6.1 結果驗證卡（每案例 (05) 段固定）

沿用 `.verify-card` 組件，**固定 4 個打勾項**，依案例填具體內容：

| 驗證項 | 含義 |
|--------|------|
| 總筆數 | AI 自報引用筆數 vs 素材實際筆數對得上？ |
| 加總金額 | 關鍵加總數字對得上？（無金額欄的案例改驗「成果件數 vs 肉眼掃描」）|
| 前三大 | AI 列的 Top 3 / 主要成果，最重要的有沒有在裡面、有沒有遺漏？ |
| 異常規則 | AI 的分類/異常判定，學員同意嗎？有沒有重要的被降級？ |

### 6.2 案例間累積串連（選用）

若課程設計成「前面案例的產出在最後一課組裝」：
- 每個前置案例的「投影片素材版」prompt 產出，存進該案例的「投影片素材」輔助段。
- 最後一課（組裝課）的素材 = 前置各案例累積的素材清單。
- 改造時若既有講義無此設計，**不強行加**；如要加須與使用者確認，因為會牽動多個單元頁。

## 8. 邊界與禁止事項

- **禁止揣測素材正解**：埋 pattern 與寫 answer key 時，數字必須真實算過，不得估算或編造。無法確認時回報，不臆造。
- **禁止為了套結構而灌水**：既有講義若某段本來就紮實，保留；本規格是補缺口，不是重寫。演練已成熟卻硬套 5 段＝降級，改走 Model B（§3.3）。
- **禁止把引導主線收進折疊區**（Model B）：引導主線必須置頂、預設可見；`<details>` 折疊區只放選讀 / 進階內容，不放主學習路徑。
- **禁止跨範圍改動**：只改演練段與素材，不順手改該頁的設計樣式、SEO、導覽。
- **概念型/工具操作型一律退回**：見 §1 判定，不適用就停。
