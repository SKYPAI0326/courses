## Part 1：解鎖 AI 工作模式（20 小時）

### 目標
讓學員從「只會聊天」進入「會下指令、會改結果、會寫出可用 Prompt、會配置 AI 工具」的狀態。

| :--- | ---: |
| CH1-1 | 2h |
| CH1-2 | 3h |
| CH1-3 | 2h |
| CH1-4 | 3h |
| PRAC1-1 | 2.5h |
| PRAC1-2 | 2h |
| PRAC1-3 | 2.5h |
| PRAC1-4 | 3h |
| **合計** | **20h** |

---

### CH1-1 AI 不只是聊天工具，而是你的工作介面（2h）
- 重新理解生成式 AI 的定位：文字生成 vs 程式輔助 vs 圖像 vs 資料任務
- ChatGPT、Claude、Gemini 的使用場景差異
- Web、Desktop、API 三種使用型態
- 免費版、付費版、額度與 token 成本基本概念
- 課程工作資料夾建置與檔案命名規則

### CH1-2 黃金提示詞公式：角色、任務、限制、輸出（3h）
- 模糊指令 vs 精確指令的差距
- 四要素結構：角色（who）、任務（what）、限制（constraint）、輸出格式（format）
- 指定輸出格式：表格、JSON、SOP、Email、條列、簡報頁綱
- Few-shot 範例法：為什麼給例子比給說明有效
- Zero-shot vs few-shot 取捨原則
- Chain-of-Thought 基礎：讓 AI 先想再答

### CH1-3 Prompt 除錯：怎麼讓 AI 越改越準（2h）
- 常見問題：太空泛、太長、格式不對、不照角色、不準確
- 多輪修正的節奏：先對，再好，再快
- 要求 AI 自我檢查、重寫、版本比較
- 建立 Prompt 版本意識（v1 → v2 改了什麼、為什麼）

### CH1-4 System Prompt：給工具用的提示詞設計（3h）【v2 新增】
- 一次性對話型 vs 應用型 System Prompt 的根本差異
- System Prompt 結構：Persona、任務邊界、輸出鎖定、拒絕策略
- 怎麼讓 AI 記住格式、角色與限制（跨輪對話穩定性）
- 為自己的工作場景設計一個可重用的 AI 助手配置
- Custom Instructions（ChatGPT）/ Projects（Claude）實際設定

---

### PRAC1-1 Prompt 黃金公式組合台（2.5h）
- 輸入一段模糊需求，套用四要素結構改寫成可執行提示詞
- 練習 5 種輸出格式：表格、JSON、Email、SOP、條列
- 產出：個人 Prompt 基礎模板 20 句

### PRAC1-2 指令品質健診器（2h）
- 檢查現有 10 個提示詞是否缺角色、限制、輸出格式
- 實際比較修改前後的回應品質差異
- 產出：Prompt 除錯紀錄單

### PRAC1-3 第一個工具：會議倒數計時器 / 報價試算器（2.5h）
- 用 AI 直接生成一個可開啟的 HTML 工具（不需要任何寫程式背景）
- 理解「需求 → 提示詞 → 程式碼 → 工具」的完整鏈
- 產出：第一個個人工具頁

### PRAC1-4 個人 AI 助手配置實作（3h）【v2 新增】
- 為自己的工作角色（行政 / 企劃 / 講師 / PM）設計 System Prompt
- 在 ChatGPT Custom Instructions 和 Claude Projects 實際配置
- 測試邊界：嘗試讓 AI 跳出設定，觀察並修正
- 產出：個人 AI 助手 System Prompt v1

### Part 1 帶走產出

- Prompt 基礎模板 20 句

### CH2-1 摘要、分類、改寫、擴寫四大核心任務（2.5h）
- 長文摘要：固定格式、字數限制、多層摘要
- 主題分類與情緒判斷：讓 AI 標記標籤
- 多版本改寫：主管版 / 客戶版 / 社群版 / 同事版
- 條列擴寫成完整段落：從骨架到成品

### CH2-2 文書工作流：Email、公告、週報、會議紀錄（2.5h）
- Email 回信：確認、拒絕、請示、跟進四種類型模板化
- 公告通知的標準結構
- 會議紀錄 → 待辦轉換邏輯
- 週報、月報快速生成框架
- 文風一致化：同一組織所有對外文件風格對齊

### CH2-3 研究整理與簡報生成（2.5h）
- 競品分析：結構化比較表的 Prompt 設計
- 主題研究：多來源資訊整理成統一格式
- 引用與交叉驗證的最低要求（防幻覺策略）
- 從研究結果到簡報頁綱：不只摘要，還要有架構主張

### CH2-4 多模態輸入：圖片、PDF、語音的 AI 處理（2.5h）【v2 新增】
- 把圖片傳給 AI 分析：商品照、截圖、白板照、表格截圖
- PDF 文件分析：結構化摘要、重點萃取、條款整理
- 語音轉文字後再 AI 處理的完整鏈（Whisper / Gemini / MacWhisper）
- 長文脈模型（Gemini 1.5M / Claude 200K）vs RAG 的使用時機判斷

- 產出：可直接使用的簡報大綱模板

### PRAC2-4 文件分析工作流實作（2.5h）【v2 新增】

## Part 3：把資料變成流程（18 小時）

### 目標
從文字型 AI 跨進資料整理與流程設計，開始面對工作場景的複雜度：重複性任務、多步驟流程、自動化判斷。

---

### CH3-1 表格資料清理：CSV / Excel 不再手動修（3h）
- 結構化 vs 非結構化資料的差異
- 欄位清理：命名規則、格式統一、空值處理
- AI 協助分類、標籤化、摘要、命名
- 將輸出結果回填表格的工作流程
- 資料清理的「人工覆核節點」設計

### CH3-2 從單次任務變成工作流（3h）
- 輸入 → AI 處理 → 輸出的標準三段結構
- 哪些任務適合標準化：條件明確、重複頻率高、格式固定
- 哪些任務不適合標準化：需要判斷、情境依賴、品質要求高
- 建立工作流文件：輸入規格、處理邏輯、輸出格式、例外處理

### CH3-3 表單、自動化工具與 Webhook 基礎（2h）
- Google 表單收資料的基本結構
- Make / n8n 的流程邏輯：觸發 → 處理 → 輸出
- Webhook 是什麼、怎麼接資料
- AI 節點在自動化工具中的位置（n8n AI Agent 節點概念）
- 流程失敗的除錯方式

---

### PRAC3-1 混亂資料清理器（2.5h）
- 給定一份格式混亂的 CSV，寫 Prompt 讓 AI 輸出清理後的版本
- 產出：資料清理 SOP + 欄位規格表

### PRAC3-2 客戶意見分類板（2.5h）
- 將一批回饋文字自動分主題與情緒
- 輸出格式：主題標籤、情緒標籤、重點摘要
- 產出：分類標註規則與模板

### PRAC3-3 表單送件 → AI 摘要 → 通知回傳（3h）
- 在 Make 或 n8n 完成最小自動化鏈
- 觸發：Google 表單提交 → AI 摘要 → 寫回試算表或發通知
- 產出：第一條 AI 自動化流程（附流程圖）

### PRAC3-4 工作流文件化：把重複任務變 SOP（2h）【v2 新增】

## Part 4：零代碼工具工坊（20 小時）

### 目標
從需求一句話，到 AI 幫你做出可以直接用的互動工具。這一段的核心精神：你不需要會寫程式，但你要能說清楚你要什麼、能看懂它做了什麼、能要求它改到你要的樣子。

### 單元時數總覽

| 單元 | 時數 |
| :--- | ---: |
| CH4-1 | 3h |
| CH4-2 | 3h |
| CH4-3 | 2h |
| PRAC4-1 | 3h |
| PRAC4-2 | 3h |
| PRAC4-3 | 3h |
| PRAC4-4 | 3h |
| **合計** | **20h** |

---

### CH4-1 讓 AI 生成有輸入、有運算、有輸出的工具（3h）
- 工具頁的基本結構：表單欄位 → 按鈕 → 結果區
- 告訴 AI 你要什麼欄位、什麼運算、什麼輸出格式
- 重設、驗證、錯誤訊息的設計
- 如何要求 AI 做「二次生成」：改欄位、改邏輯、改外觀

### CH4-2 用 UI / UX 要求讓工具變好用（3h）
- 從「能用」到「好用」：欄位排序、標籤說明、預設值
- 錯誤提示與空值提醒的設計
- 卡片式版型、色彩搭配、按鈕大小
- 告訴 AI 你的設計意圖，而不是告訴它怎麼寫程式

### CH4-3 一句話改造介面風格（2h）
- 視覺語言：清爽 / 專業 / 活潑的提示詞寫法
- 資訊階層：哪個最重要、哪個是次要說明
- RWD 概念：在手機上也能用
- 如何把別人的工具介面改成自己的品牌風格

### PRAC4-3 履歷 / 自傳段落整理器（3h）
- 輸入個人素材（學歷、經歷、技能），輸出各種版本段落
- 練習「輸入驗證」：素材不足時給提示，不直接亂生
- 產出：履歷整理工具頁

### PRAC4-4 個人複合工具頁（3h）【v2 新增】
- 整合前三個 PRAC 的經驗，自選需求做一個更複雜的工具

- 可選題：內容標題生成器、會議時間計算器、行銷文案版本機

### CH5-1 看懂 HTML / CSS / JavaScript 的最低必要知識（3h）
- HTML 結構：標題、段落、表單、按鈕、列表、區塊
- CSS：字型、顏色、間距、排版、卡片
- JavaScript：變數、函式、事件、DOM 操作
- 重點：不是要你寫，而是要你能讀懂 AI 產出的前端碼

### CH5-2 與 AI 協作改頁面，而不是盲貼程式碼（3h）
- 如何把頁面需求拆解成 AI 能執行的指令
- 如何要求 AI 只改特定區塊，不動其他部分
- 如何要求 AI 修 bug：「這個按鈕點了沒有反應，原因是什麼，怎麼修？」
- AI 協作開發工具介紹：Cursor / GitHub Copilot / Claude Code 的基本使用場景

### CH5-3 Git 與版本保存（2h）
- 版本控制的核心概念：為什麼不能只存一個檔
- Git 基本操作：init、add、commit、log
- 如何回看修改紀錄，如何回到上一個版本
- 在 GitHub 建立 repo 並推送

### CH5-4 含後端服務的完整部署（2h）【v2 新增】
- 靜態頁面（Vercel / Netlify）vs 含後端服務（Railway / Render）的差異
- 什麼時候需要後端：API 金鑰保護、表單處理、資料庫
- 環境變數（.env）的正確使用：不把金鑰寫進程式碼
- 部署到 Railway / Render 的基本流程

## Part 6：AI 應用開發進階（22 小時）

### 目標
從前端工具跨進 AI App 的核心技術層：API 設計、Embedding 與 RAG、Agent 工具鏈、串流輸出、評估與安全。這一段是課程技術密度最高的部分，不要求全部精通，但要建立完整的概念地圖與操作起點。

### 單元時數總覽

| 單元 | 時數 |
| :--- | ---: |
| CH6-1 | 3h |
| CH6-2 | 3h |
| CH6-3 | 3h |
| CH6-4 | 3h |
| CH6-5 | 2h |
| PRAC6-1 | 2h |
| PRAC6-2 | 2.5h |
| PRAC6-3 | 2h |
| PRAC6-4 | 1.5h |
| **合計** | **22h** |

---

### CH6-1 API、後端代理與錯誤處理（3h）
- API、request、response、headers、JSON 核心概念
- token 成本意識：輸入 token + 輸出 token 的計算方式
- 前端不能直接裸露金鑰的原因與後果
- 後端中介層的角色：接收前端 → 呼叫 AI → 回傳結果
- 基本輸入驗證：空值、長度、格式
- Rate limit 概念與重試策略
- 錯誤碼處理：4xx vs 5xx，使用者體驗設計

### CH6-2 Embedding 與向量搜尋基礎（3h）【v2 新增，原 CH6-2 拆分】
- 語義相似 vs 關鍵字搜尋的根本差異
- Embedding 的直覺理解：把文字變成方向（不需要數學，要有直覺）
- 向量資料庫的角色：Chroma、Supabase pgvector、Pinecone
- 相似度搜尋的基本流程：文字 → embedding → 比較 → 取最近 k 筆
- 長文脈模型（Gemini 1.5M / Claude 200K）何時比 RAG 更簡單

### CH6-3 RAG 實作與品質調優（3h）【v2 大幅擴充】
- RAG 完整流程：文件讀取 → 切塊 → 索引 → 檢索 → 生成
- Chunking 策略：大小、重疊、Markdown-aware 切法
- Metadata 過濾：讓檢索更精準
- 檢索品質評估：相關性、完整性、答案正確性
- Reranking 概念
- 常見失敗模式：幻覺、漏找、找到但沒用到

### CH6-4 Function Calling 與 Agent 工具鏈設計（3h）【v2 大幅擴充】
- 什麼是 Function Calling / Tool Use
- 讓模型輸出結構化參數，供程式接收與執行
- Agent Loop 概念：Observe → Think → Act → Observe（ReAct 思維）
- 工具呼叫的設計原則：工具要小而專、輸入輸出要清楚
- 多工具協調：模型如何選擇呼叫哪個工具
- 常見陷阱：無限迴圈、工具幻覺、工具呼叫失敗的處理
- 實務工具：OpenAI Assistants API、n8n AI Agent 節點

### CH6-5 Streaming、Evals、Guardrails 與成本控管（2h）
- Streaming 回應：為什麼要串流、Server-Sent Events 基本概念
- 在後端代理實作串流轉發、前端顯示逐字輸出效果
- 為什麼 Prompt 要做評估：一個版本變好了不代表另一個案例也變好
- 建立測試案例集：正確性、格式一致性、穩定性
- Guardrails：輸入檢查、輸出限制、違規情境處理
- 成本控管：Prompt 快取、模型分級使用、批次處理
- 上線前最低防護 Checklist

---

### PRAC6-1 API 呼叫測試頁（2h）
- 實際打一次文字生成 API（OpenAI / Anthropic / Gemini）
- 觀察 request 結構、response 結構、token 用量
- 產出：API 呼叫範例檔 + 成本計算練習

### PRAC6-2 最小文件問答器（2.5h）
- 上傳一份文件，做 Chunking，建立可查詢的知識庫
- 實作最基本的 RAG 問答：輸入問題 → 檢索相關段落 → 生成回答
- 工具選項：LlamaIndex 簡單範例、或 n8n 的 vector store 節點
- 產出：簡易文件問答原型

### PRAC6-3 Agent 工具鏈小案例（2h）【v2 新增】
- 設計一個有 2–3 個工具的 Agent：例如「搜尋 → 摘要 → 寫入表格」
- 讓模型自動判斷呼叫哪個工具
- 測試失敗情境：工具不存在、輸出格式錯誤、迴圈發散
- 產出：工具呼叫範例流程 + 失敗處理紀錄

### PRAC6-4 Streaming 回應改造（1.5h）【v2 新增】
- 把 Part 5 的後端代理改成支援串流輸出
- 前端頁面顯示逐字輸出效果
- 產出：Streaming 版 AI 工具頁

### Part 6 帶走產出
- API 呼叫範例檔 + 成本計算練習
- 簡易文件問答原型
- 工具呼叫範例流程
- Streaming 版 AI 工具頁
- 專題安全與驗收 Checklist

## Part 7：專題衝刺與成果發表（20 小時）

### 目標
把前面所有能力收斂成一個完整作品。Part 7 不是「做一個作業」，而是一個結構化的產品開發週期：有定義、有衝刺、有中期審查、有修正、有部署、有發表。

### 單元時數總覽

| 單元 | 時數 |
| :--- | ---: |
| CH7-1 | 2h |
| CH7-2 | 2h |
| CH7-3 | 2h |
| CH7-4 | 2h |
| PRAC7-1 | 4h（開發衝刺第一輪） |
| PRAC7-2 | 3h（中期 Demo + 同儕審查） |
| PRAC7-3 | 3h（開發衝刺第二輪） |
| PRAC7-4 | 1h（部署與驗收） |
| PRAC7-5 | 1h（發表） |
| **合計** | **20h** |

---

### CH7-1 選題與 MVP 拆解（2h）
- 使用者是誰、要解決什麼問題
- 最小可行功能（MVP）是什麼
- 哪些功能「先不做」（明確出局）
- 輸入 / 輸出 / AI 角色的設計決策

可選專題方向：
- AI 文案助手（輸入需求 → 產出多版本文案）
- AI 摘要整理器（上傳文件 → 輸出摘要 + 重點）
- AI 履歷優化工具（輸入素材 → 產出不同版本）
- AI 課程企劃助手（輸入主題 → 產出課程大綱）
- AI 旅遊行程規劃器（輸入條件 → 產出行程與說明）
- 自選：符合「有輸入、有 AI 處理、有有用輸出」即可

### CH7-2 架構設計 Checklist（2h）
- 前端：頁面結構、輸入區、輸出區、錯誤提示
- 後端：需不需要？如果需要，中介層要做什麼
- AI：模型選擇、System Prompt、溫度設定、輸出格式
- 資料：需不需要知識庫或工具呼叫
- 部署：靜態還是含後端，用哪個平台
- 安全：金鑰保護、輸入驗證、使用量控管

### CH7-3 測試與 UX 優化（2h）
- 建立 10 筆測試案例（正常 + 邊界 + 錯誤輸入）
- UX 問題修正清單：載入中、空值提示、結果太長怎麼截
- 失敗時的使用者體驗：有意義的錯誤訊息，不是 undefined
- 如何讓非技術朋友試用並給反饋

### CH7-4 發表敘事與作品集頁建置（2h）
- Demo 怎麼講：問題 → 做法 → 展示 → 限制 → 後續
- 限制怎麼說：誠實比誇大重要，說出來才有改進機會
- 技術選型怎麼說明（給非技術觀眾）
- 作品集頁更新：把這個專題加入 Part 5 建立的展示首頁

---

### PRAC7-1 開發衝刺第一輪（4h）
- 依自己的題目，完成前端頁面 + AI 串接的基本版（v0）
- 目標：能跑通主要使用路徑，不需要完美
- 課堂中可隨時問講師或同學
- 產出：專題 v0（可運作但未優化）

### PRAC7-2 中期 Demo + 同儕審查（3h）
- 每人 5 分鐘 Demo 自己的 v0
- 同儕給三點回饋：一個讚、一個疑惑、一個建議
- 整理改進清單，排列優先順序
- 產出：改進清單（v0 → v1 的差距說明）

### PRAC7-3 開發衝刺第二輪（3h）
- 依改進清單執行 v1 開發
- 重點：UX 問題修正、錯誤處理補齊、成本控管
- 產出：專題 v1（部署前版本）

### PRAC7-4 部署與驗收（1h）
- 部署到 Vercel / Render，確認公開可存取
- 執行 10 筆測試案例，記錄通過率
- 更新作品集展示頁
- 產出：專題正式版 + 測試記錄

### PRAC7-5 最終發表（1h）
- 每人 5 分鐘 Demo：問題 → 做法 → 展示 → 限制 → 後續
- 接收回饋，記錄可延伸的方向
- 產出：發表簡報 + 作品展示網址

### Part 7 帶走產出
- 1 個可展示的小型 AI Web App（公開部署）
- 專題驗收測試記錄
- 發表簡報
- 更新後的個人作品集展示頁
