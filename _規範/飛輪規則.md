# 飛輪規則（錯誤轉規則）

每當 Claude 犯錯（格式不對、命名跑掉、連結失效、邏輯錯誤），立刻將該錯誤轉為一條具體規則加入此文件的對應區塊。規則會因此持續進化，同樣的錯誤不再發生第二次。

## 商業培訓 vs 自學工程師：學員真實訴求驅動設計（2026-05-02 classroom pack 哲學定位）

**最高原則**：商業培訓課程的學員（行銷企劃 / 營運 / 知識工作者 / 主管）**不一定想學原理，他們要的是結果**。把「學員應該想學原理」這個自學工程師教育假設套到商業培訓課程，會導致 funnel rate 崩盤、學員流失、口碑下滑。

### 兩種教育場景對比

| 維度 | 自學工程師教育 | 商業培訓課程 |
|------|---------------|--------------|
| 學員動機 | 想學會自動化 / 想換職涯 | 想解決手上的工作痛點 |
| 時間預算 | 願意花週末研究 | 課堂 3 小時 / 在家 1 小時上限 |
| 容錯度 | 環境卡關自己 Google 解 | 環境卡關 = 放棄整門課 |
| Template 接受度 | 抗拒（想自己理解） | 歡迎（先用後想學） |
| 結業期待 | 能獨立建專案 | 帶走能用的工具 |

對商業培訓場景，**「淪為 template marketplace」不是貶低，是合適的價值定位**——只要原理說明可以查到，「沒心的拿模板，有心的學原理」是 80/20 健康分布。

### 設計準則

1. **能落地是第一優先**：學員下載 zip 雙擊就能跑，30 分鐘內看到成品
2. **原理同步提供但不強制**：每個工作流附「為什麼這樣設」「怎麼改」說明，但學員沒看也能用
3. **進修通道留給有心學員**：Builder Track 評量機制讓學員自己選擇深度
4. **不要罵學員「應該學原理」**：學員是商業培訓參與者不是工程師，套錯期待是課程設計失職

### 對應實作

新課程或既有課程重整時，**檢核這 3 個問題**：

1. 學員下載資源後，**30 分鐘內能看到成品嗎？**（funnel rate 預設 80%+）
2. 卡關時學員能否**不開終端機 / 不打指令 / 不編 yaml** 就解決？（GUI 對話框引導）
3. 想學原理的學員能否**自己查到完整解釋**？（每個 deliverable 附原理文件）

三個答案都「是」才算合格商業培訓課程。任何一個答案「否」就是降低 funnel rate 的設計失職。

### 反例：學員壓測 8+ 連發

n8n 課程「AI 資料工廠」原本講義主流程是「跟著教做的工作流」，funnel rate 估 < 30%。連續 8 個學員壓測卡關（zip 不存在 / compose -f / secure cookie / postgres 密碼 / port 5678 / Cloudflare 綁卡 / 自有網域 / QUIC 被擋）才暴露這個設計失職。

修法：classroom pack 三層架構（Lite Pack 30 分鐘必成 + Google Pack guided + 原理說明文件），詳見 `courses/n8n/_reviews/n8n-classroom-pack-design-2026-05-02.md`。

## 講義主流程必須是 90% 學員可走的最低門檻路徑（2026-05-01 m1-2-tunnel 重整後總結）

當原講義主流程被學員實測撞到多重門檻（綁卡、買網域、ISP 擋協議、Gatekeeper 擋 .command 等），**禁止用補釘方式在原主流程上層層加揭露框 / 替代路徑 / fallback 段落**。要直接重整：把實際可走的最低門檻路徑提升為主流程，原本的高門檻路徑降為進階折疊段落。

### 判斷準則

寫課程主流程前必過一個問題：「**90% 學員實際能跨過這條路嗎？**」

| 答案 | 處理 |
|------|------|
| ✅ 90%+ 能過 | 寫成主流程 |
| ⚠️ 50-90% 能過 | 不要當主流程；寫成「進階」folder |
| ❌ <50% 能過 | 整段拿掉，或標明「本節純概念說明，不需實作」 |

判斷依據：列出每一步的「外部門檻」（要綁卡？要買 X？要會 Y？要在某個 OS 版本？要某種網路環境？）。每個門檻篩掉一批人，乘起來算最終 funnel rate。

### 主流程 + 進階 + 備用三層結構

每個 skill-operation 單元（環境建置、整合服務、API 串接類）的講義都該有三層：

1. **主流程**（90% 學員可走）：步驟最少、門檻最低、能立刻看到成果
2. **進階**（用 `<details>` 折疊）：完整版 / 生產級設定 / 客制化
3. **備用方案**（標明「主流程完全不通才用」）：架構不同的替代工具

### 為何不能補釘疊加

補釘式修法的問題：
- 學員在原主流程踩雷後，要學新概念才看懂揭露框
- 揭露框太多會讓主流程閱讀體驗破碎
- 學員不知道哪條才是「老師預設要走的路」
- 重看講義時看不出主流程是什麼，只看到一堆例外處理

### 修補時機

當同一單元連續累積 3+ 個「主流程踩雷」修補時，**強制觸發整章重整**，不要再加第 4 個揭露框 / 替代段落。把整章改寫成新主流程。

### 對應 m1-2-tunnel.html 案例

原主流程 7 步驟（Cloudflare 帳號 → Zero Trust → cloudflared 安裝 → 連線 → Public Hostname → 取網址 → 測試）。學員實測卡在 step 1（綁卡）+ step 5（自有網域）+ 連線層 QUIC 被擋——funnel rate 估 < 30%。重整後新主流程 3 步驟（cloudflared 安裝 → tunnel-quick 拿 URL → 測 webhook），funnel rate 估 > 85%。完整 Tunnel 降為折疊式「進階：要長期穩定 URL 才看」。

## Cloudflared 預設 HTTP/2 而非 QUIC（2026-05-01 學員 control stream failure 發現）

cloudflared 預設使用 **QUIC（UDP 7844）** 連 Cloudflare 邊緣節點，但**家用 ISP / 公司網路 / VPN 經常擋 UDP 7844**，導致連續 retry loop：

```
ERR failed to serve tunnel connection error="control stream encountered a failure while serving"
ERR Serve tunnel error error="control stream encountered a failure while serving"
INF Retrying connection in up to 4s ... 8s ... 16s ...
```

### 規則

任何 starter-kit 的 cloudflared 啟動腳本（`tunnel-*.command` / `tunnel-*.bat`）必須：

1. **預設加 `--protocol http2`**（不要靠 default auto fallback——auto 在 quic 失敗時 fallback 不可靠）
2. **啟動腳本內建偵測**：30 秒內未取得 `trycloudflare.com` URL，且 logs 含 `failed to serve tunnel` / `Retrying connection` → 跳對話框建議改用 ngrok（架構不同、走 TCP HTTPS 反代，比較不會被擋）
3. **講義必須在 Quick Tunnel 段落明示**：「若 30 秒內看不到 trycloudflare.com URL → 你的網路擋了 cloudflared，改走 ngrok」

### 為何不能讓學員自己加 `--protocol http2`

按「禁止給學員 patch 指令」原則，不能叫學員改腳本參數。預設值就要是「最高機率 work」的設定。HTTP/2 比 QUIC 慢一點點但在 web app 場景無感，**穩定性遠勝**。

## 外部 SaaS 政策變動誠實揭露規則（2026-04-30 學員 Cloudflare 綁卡卡關發現）

**外部 SaaS 的「免費」門檻會隨時間變動**，講義主流程必須誠實揭露當前實際門檻，不能用「個人開發/教學用途在免費方案內可用（依當前規則）」這類模糊話術唬過去。

### 已知必揭露門檻（2026 年現況）

| SaaS | 表面 | 實際門檻 | 講義必寫 |
|------|------|---------|----------|
| Cloudflare Zero Trust | 免費 | **必須綁信用卡** anti-abuse（不扣款，但有人會卻步） | step 1 加 ⚠️ 誠實揭露框 + 指向 ngrok 替代 |
| Make.com | 免費 1,000 ops/月 | AI 模組另計、credit 不是 ops（已在課程處理） | 已修，見 2026-04-28 brief |
| Gemini API | 免費 tier | RPM/TPM/RPD 限制依模型變動，超量會被擋 | 已部分處理，但鎖定明確值仍危險 |
| ngrok | 免費 | 註冊即用不綁卡，但 URL 每次重啟會變 + 有流量上限 | 寫進 fallback 段落 |
| Google Maps API | 免費 $200 | 2025-03 改 SKU quota 制 | 已修 |

### 寫作原則

1. **每個外部 SaaS 註冊步驟前**，必有 ⚠️ 誠實揭露框列出當前門檻（綁卡？驗證手機？需要付款資料？）
2. **fallback 路徑必須單一**——不要列「方案 A / B / C」讓學員猶豫；只給「綁卡走主路 / 不綁卡走唯一替代」
3. **每季回檢**：在 `_change-log.md` 記錄各 SaaS 政策回檢日期，超過 90 天必須再驗證一次

### 對應錯誤訊號

- 學員回報「需要付款方式才能用」「要綁信用卡」「政策變了」→ 第一個動作是更新講義 ⚠️ 揭露框，第二個動作是檢查 fallback 路徑是否仍存在

## 課程修補哲學：學員旁邊沒有 Claude（2026-04-30 學員第四次卡關後總結）

**最高原則**：課程進行時學員旁邊不會有 Claude 提供 patch 指令。troubleshoot 段落、講義錯誤指引、README 排錯內容**禁止**叫學員：

- ❌ 開終端機跑 `sed` / `awk` 一行
- ❌ 用 nano / vim 手動編輯 yaml / .env / .json
- ❌ 跑「兩個指令的組合」處理 race condition
- ❌ 給「方案 A 重下 / 方案 B 手動修」雙路徑（學員會猶豫選哪個）

**唯一允許的指引格式**：

- ✅ 「**重新下載最新 zip 覆蓋整個資料夾**，再雙擊 `start.command`」（單一動作）
- ✅ 「**再雙擊一次 `start.command`**，會跳對話框讓你按一個按鈕」（前提：腳本內建偵測 + GUI 對話框）
- ✅ 「**按下這個按鈕**」（GUI 操作，不下指令）

**設計責任移到 starter-kit 本身**：

學員會碰到的所有可預測錯誤（postgres 密碼、compose 檔名、secure cookie、image tag 撤除、port 衝突、Docker 沒開），**必須由 `start.command` / `start.bat` 主動偵測 + GUI 對話框引導學員一鍵處理**。`docker compose logs` 的關鍵 pattern grep 應內建在啟動腳本裡。

不能修的錯誤（例如 image 撤除）才允許跳「請聯絡講師」對話框，但要清楚講「貼這 3 行 logs 給講師」（不是叫學員自己下指令解）。

**對應錯誤模式**：

每次學員回報新卡關，問自己 3 個問題：

1. 這個錯誤能在 `start.command` 裡 grep logs 偵測嗎？→ 能 → 加偵測 + 對話框
2. 這個錯誤能避免發生（compose 預設正確）嗎？→ 能 → 改 starter-kit 預設值
3. 真的兩者都做不到？→ 才寫到 troubleshoot，但**只寫一條「重下 + 雙擊」單一動作**

### 為何這條優先於「給技術解」

學員的時間很寶貴（課堂 3 小時內）。每多一個「打開終端機 → cd → sed → 等 30 秒 → 看不懂報錯」的迴圈，學員就會放棄一次。把卡關處理成本壓到「下載 + 雙擊」兩個動作以下，學員才能真的學到自動化（M2 之後的內容）而不是把時間都花在環境上。

## n8n localhost 啟動必設 N8N_SECURE_COOKIE=false（2026-04-30 學員第三次卡關發現）

n8n **預設 `N8N_SECURE_COOKIE=true`**——要求只透過 HTTPS 訪問，否則登入頁會擋出「Your n8n server is configured to use a secure cookie, however you are either visiting this via an insecure URL, or using Safari」錯誤。

### 起手式必設（compose.yml 與 .env.example）

任何 n8n self-host starter-kit，**只要預設訪問 URL 是 `http://localhost:...`**，必須在 compose 的 environment 與 .env.example 加：

```yaml
- N8N_SECURE_COOKIE=${N8N_SECURE_COOKIE:-false}
```

```bash
# .env.example
N8N_SECURE_COOKIE=false
```

### 接 HTTPS（Cloudflare Tunnel / Reverse Proxy）後改回 true

當學員把 n8n 接到 Cloudflare Tunnel HTTPS、Caddy / nginx 反向代理時，必須把 `N8N_SECURE_COOKIE` 改回 `true` 提升安全。對應教案章節（如 m1-2-tunnel.html）必須在「接 Tunnel 後設定」段點明這個切換。

### Safari 額外坑

Safari 對 `localhost` cookie 限制比 Chrome / Firefox 嚴格——即使設 `N8N_SECURE_COOKIE=false`，仍可能間歇出現相同錯誤。**教案應建議 localhost 開發階段用 Chrome / Firefox / Edge**，Safari 留給 Tunnel HTTPS 之後使用。

### 對應錯誤訊息

學員回報「Your n8n server is configured to use a secure cookie」→ 第一個假設就是這個。

## Docker Compose 自訂檔名規則（2026-04-30 學員第二次卡關發現）

**`docker compose` 預設只認 `compose.yaml` / `compose.yml` / `docker-compose.yaml` / `docker-compose.yml`**，**不認**任何其他檔名（包括 `n8n-compose.yml`、`xxx-compose.yml`）。如果 starter-kit 用了**自訂檔名**（例如為了多服務區分用 `n8n-compose.yml`），所有呼叫 `docker compose` 的地方**必須加 `-f <檔名>`**。

### 必檢清單

starter-kit 目錄裡所有 `.command` / `.bat` / `.sh` 腳本，凡是含 `docker compose` 字樣的：

```bash
docker compose up -d           # ❌ 找不到 compose 檔
docker compose down            # ❌
docker compose pull            # ❌
docker compose logs n8n        # ❌

docker compose -f n8n-compose.yml up -d    # ✅
docker compose -f n8n-compose.yml down     # ✅
```

`README.md` 與講義頁的「卡關貼這幾行」指令也要一併加 `-f`，否則學員照貼會繼續報相同錯誤。

### 替代方案

若不想每行都加 `-f`：直接把 compose 檔改名為 `compose.yaml`（最新 docker compose 標準預設名）。但要全文搜尋替換所有「`n8n-compose.yml`」字樣（教案 .md / HTML 講義 / FAQ / change-log），跨檔案改動成本反而較高，**建議保留自訂檔名 + 全 script 加 `-f`** 即可。

### 對應錯誤訊息

學員若回報 `no configuration file provided: not found`，第一個假設就是這個——學員跑的是舊版（沒 `-f`）script。

## 試跑包交付完整性規則（2026-04-30 從 n8n 課程實際演練卡關發現）

**「檔案存在 ≠ 學員拿到能跑的包」**——評估「課程可落地實現」必須走完整學員動線，不是只看 `ls` 結果。

### 三層交付驗證（每個 `assets/{slug}-starter-kit/` 都該過）

| 層級 | 檢核問題 | 驗收動作 |
|------|---------|---------|
| **L1 檔案就位** | 9 個檔案、shared/ 子資料夾、README 都在嗎？ | `ls -la assets/{slug}-starter-kit/` |
| **L2 打包可下載** | 有 `.zip` 一鍵下載嗎？瀏覽器點按鈕能拿到完整包？ | 講義頁應有 `<a href="...zip" download>` 主按鈕，**不是 9 個個別檔連結** |
| **L3 解壓可執行** | macOS 解壓後 `.command` 雙擊能跑嗎？或學員需要手動 chmod / 移除 quarantine？ | README 第一次啟動段應寫 `chmod +x` + `xattr -d com.apple.quarantine` 一次性指令 |

### 必修補形式

當 starter-kit 含 `.command` / `.sh` 等執行檔時，講義頁必須包含三件事：

1. **主按鈕**：`<a href="../assets/xxx-starter-kit.zip" download>` 顯眼（建議綠底白字 11px+ padding）
2. **macOS chmod 提示** `<details>`：解壓後若 .command 雙擊報錯，給 `chmod +x` + `xattr -d com.apple.quarantine` 兩行指令解決方案
3. **個別檔備援** `<details>`：zip 損壞時可逐個下載，附「需自建 shared/ 子資料夾」提示

### 評估報告寫法修正

不要寫「✓ 試跑包已具備」這種模糊聲明。改成：

- ✅ **L1 + L2 + L3 都過**：「✓ 試跑包完整交付（檔案 / zip / chmod 提示）」
- ⚠️ **缺 L2 或 L3**：「⚠️ 檔案就位但學員下載動線斷裂（缺 zip 包 / 缺 macOS chmod 指引）」
- ❌ **L1 缺**：「❌ 試跑包尚未建立」

## 複製按鈕定位規則

- **絕對定位的複製按鈕必須同步設定父容器 padding-right**：若 `.step-example-copy`（或同類 copy button）使用 `position:absolute;top:Xpx;right:Ypx`，其父容器必須加 `padding-right:72px`，否則文字會被按鈕覆蓋
- 格式：`padding:14px 72px 14px 16px`（四值寫法，保留左側 16px 不變）
- 建立新頁面時，有 absolute copy button 的地方一律加 `padding-right:72px`

## WordPress 資料庫操作規則

- **禁止**用 Python mysql.connector 讀出 `post_content` 再寫回——會雙重跳脫 `\n` 與 `\u002d`（`--`），導致排版崩潰與 CSS 變數失效
- **必須**用 MySQL `REPLACE()` 直接在資料庫內操作，例如：
  ```sql
  UPDATE wp_posts SET post_content = REPLACE(post_content, '舊值', '新值') WHERE ID=N;
  ```
- 修改後用 `HEX()` 確認關鍵字元（換行應為 `0A`，連字符應為 `2D`）

## 密碼關卡 placeholder 不可帶進 commit（2026-05-09 line-stickers 學員密碼錯誤發現）

`course-web-builder` skill 產出 `index.html` / `module1.html` / `module2.html` 時，gate template 不能用 placeholder 字串當 K（sessionStorage key）和 H（SHA-256 hash）。否則 `inject_gate.py` 會看到頁面已含 `id="_gate"` 直接跳過，placeholder **永遠不會被覆蓋**——學員輸入正確密碼一律「密碼錯誤」。

### 偵測訊號

- HTML 內出現字面字串 `PLACEHOLDER_HASH` / `XXX_HASH` / `TODO_HASH`
- HTML 內出現 sessionStorage key 形如 `{slug}-gate`（`course-web-builder` 預設格式）但 `inject_gate.py` COURSES dict 用的是 `{slug去連字號}_auth`
- `inject_gate.py` 報告「跳過 N 個」但 N 不是 0（正常情況新課程應該全注入）

### 修法（兩條路擇一）

**A. course-web-builder 不放 gate**（建議，更乾淨）：
- 產 `index.html` / `module*.html` 時**不要嵌任何 gate template**
- 等 `course-register` 跑 `inject_gate.py` 統一注入
- 這樣只有一個事實源（COURSES dict）

**B. 若必須先放 placeholder**：
- 強制 K 和 H 都用「會 lint error 的字串」如 `__GATE_KEY_PLACEHOLDER__` / `__GATE_HASH_PLACEHOLDER__`
- 加 lint rule：HTML 內出現 `__GATE_*_PLACEHOLDER__` = BLOCKER
- 強迫 course-register 先替換才能 push

### `course-web-builder` skill 修補

skill 文件加 negative list：「**禁止在產出的 HTML 中嵌入任何 gate template**（包含 `<div id="_gate">` / `<style id="_gs">` / `K='...'` / `H='...'`）。Gate 由 `course-register` + `inject_gate.py` 負責統一注入，web-builder 只負責內容。」

## PR push race 規則（2026-05-09 line-stickers 連續開兩個 fix PR 發現）

push 完 commit 不等於 PR 帶到該 commit。如果 reviewer 在你 push 之前已經按下 merge，**後 push 的 commit 不會自動加進已 merged 的 PR**——需要再開一個新 PR。實務上這發生在「我 commit 後立刻請使用者 merge」的場景下，使用者按得快過 push 完成。

### 偵測訊號

- 你說「已 push 到 PR #X」但 PR view 顯示 mergedAt 時間早於最後 commit 時間
- 使用者回報「merge 後沒生效」「主分支沒看到改動」
- 跑 `git log origin/main` 看不到你最後那個 commit

### 修法（事前防範優先）

**事前**（commit 順序）：

1. **「commit + push 完成 + 確認 origin 收到」是一個原子動作**——push 完跑 `git log origin/{branch} -1` 確認 hash 一致再回報使用者
2. **同一 PR 連續 push 多個 commit 時，明確告訴使用者「等我說 push 完成再 merge」**
3. **修補類 commit 不要與註冊 / 大量內容 commit 放同一 PR**——分開 PR 給使用者更清楚的 review boundary

**事後**（race 已發生）：

1. `gh pr view {N} --json mergedAt,headRefOid` 確認 PR HEAD 是否包含你最後的 commit
2. 不包含 → `git checkout main && git pull` → `git checkout -b fix/{topic}` → `git cherry-pick {missing-commit}` → push → 開新 PR
3. 在新 PR description 標明「PR #N merge 時 race，這是補帶的 commit」

### 偵測腳本：`docs/safe-push.sh`

已實作。`git push` 後跑：

```bash
bash docs/safe-push.sh         # push 並驗證
bash docs/safe-push.sh --no-push  # 只驗證，不推（debug 用）
```

腳本動作：
1. 記錄本地 HEAD
2. `git push`
3. `git fetch` + 比對 remote HEAD
4. 若 hash 不一致 → exit 2 並警告
5. 若有 open PR → 檢查 PR HEAD 是否含最後 commit、提醒使用者 merge 前等 GitHub 同步

**Claude 工作流**：commit 完一次性 push 多個 commit、或修補類 commit 跟內容類 commit 混放時，**push 完都跑這個腳本**再向使用者報告「可以 merge 了」。

- 16:30:37 UTC：fix commit 5f81173 push 到 feature branch
- 4 分鐘的 race，使用者開瀏覽器看到密碼錯誤
- 衍生 PR #76 補 cherry-pick 上去
