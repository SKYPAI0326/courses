**一週文書工作流模板組：4 種文書同時模板化**

## ⏱ 預估時間
核心 25 min + 進階 15 min

## 情境敘述（lede · 約 240 字）

延續上週 — 你是雲流科技的內容運營新人。FlowOps v3.0 上線忙完，產品經理 Daniel 來找你：

「上週你做的跨受眾套件很棒。我看你寫得很快，能不能把我們團隊**每週都要寫的 4 種文書**都模板化？這樣下次你休假其他人也能用，每件文書都能在 5 分鐘內完成。」

具體 4 種：
1. **會議紀錄轉待辦**：每週週會 1 小時錄音 → Asana JSON
2. **週報**：每週 5 個工作項 → 給主管讀的週報 Email
3. **新功能上線公告**：產品上線資訊 → 給全公司 200 人讀的內部公告
4. **客戶 demo 跟進信**：demo 後 5 個 follow-up 點 → 給客戶的 EN 跟進 Email

你今天剛學完**四種 Email 類型 + 會議轉待辦標準流程 + 週報生成器**，這份作業是把這些技巧**模板化打包**，給整個團隊用。

## 本作業拆解

- **核心 25 min（必做）**：4 個文書任務各寫 1 個可複用 prompt 模板（每模板 100–150 字）+ 4 個對應的「素材占位符」標示（哪些是每次都要換的變數）
- **進階 15 min（可選 · 二選一）**：路線 A — 4 模板實跑一次（用下方提供的範例素材跑出 4 份輸出）｜路線 B — 寫一份「**團隊文書 SOP**」（200 字，描述什麼時候用哪個模板、怎麼維護）

## 任務清單（assignment-tasks）

**核心交付（必做）**：

1. **4 個 prompt 模板**：每個 100–150 字，含【角色】【任務】【限制】【輸出格式】4 元素（沿用 CH1-2 學的）
2. **占位符標示**：每模板用 `{{變數名}}` 格式標出「每次要換的部分」（如 `{{逐字稿}}`、`{{本週工作項}}`、`{{客戶名}}`）
3. **使用說明**（每模板 1 句）：「每次怎麼用、多少分鐘可完成」

**進階交付（可選 · 二選一）**：

- **路線 A（實跑驗證）**：用上方 4 個範例素材，把 4 個模板各跑一次（直接貼進 ChatGPT/Claude），把輸出貼回作業
- **路線 B（團隊 SOP）**：寫一份「文書工作流 SOP」200 字 — 描述什麼情境用哪個模板、模板維護由誰負責、什麼時候要更新

---

## 評分標準

**基本要求**：

1. **4 模板都套四元素**（角色/任務/限制/輸出），不能只寫「請幫我寫週報」這種片段
2. **占位符區分清楚**：什麼是「每次都一樣」（→ 寫死在模板）、什麼是「每次要換」（→ `{{...}}`）
3. **限制具體可驗證**：週報限制要寫「不超過 200 字」「分『完成 / 卡住 / 下週』三段」、公告寫「**不用內部術語**」這種
4. **輸出格式具體**：Email 寫「Subject + Body」、週報寫「分段標題」、公告寫「Markdown 結構」

**加分項**：

- ➕ **跨模板共用 system prompt**：發現 4 模板都共享「你是雲流科技內容運營助手」這個身分，可拆出共用 SP（呼應 CH1-4）
- ➕ **多語言版本**：客戶跟進信中英文版各一
- ➕ **失敗模式預測**：每模板加一句「常見失敗：X」（如「會議轉待辦最常編造未說的待辦」）

## 折疊參考解答

### 模板 1｜會議紀錄轉待辦

```
【角色】你是雲流科技 CSM 團隊的會議助理，專做逐字稿轉 Asana 待辦。
【任務】把以下會議逐字稿轉成 Asana JSON 待辦清單。
【限制】
- 待辦 = 動作 + 負責人 + 大概截止，會議明確決議的事項
- 不得編造、不從常識推測
- owner/deadline 沒明說 → 填 null
- 「待議」「未結論」放 pending_decisions 陣列
【輸出格式】JSON: {"tasks": [{"task":"中文","owner":"Name|null","deadline":"YYYY-MM-DD|null","priority":"H|M|L","context":"原話節錄"}], "pending_decisions": [...]}

逐字稿：
{{逐字稿全文}}
```

使用：每週週會結束後 5 分鐘可完成。占位符：`{{逐字稿全文}}`

### 模板 2｜週報

```
【角色】你是雲流科技內容運營專員的個人助手，習慣寫主管喜歡的「事實 + 取捨 + 方向」風格週報。
【任務】根據下方本週工作項，產出給主管讀的週報。
【限制】
- 不超過 200 字
- 分三段：完成（最多 3 項，每項 1 句）/ 卡住（最多 2 項 + 為什麼）/ 下週（3 項）
- 不要寫「努力工作」「持續精進」這種空話
- 卡住的事要附具體請求（要主管支援什麼、何時要回覆）
【輸出格式】Email Body 純文字，無 Subject

本週工作項：
{{本週工作項清單}}
```

使用：每週五下班前 3 分鐘可完成。占位符：`{{本週工作項清單}}`

### 模板 3｜新功能上線公告

```
【角色】你是雲流科技內容運營助手，專責寫給全公司 200 人讀的內部公告。
【任務】把產品資訊改寫成內部公告，目的：讓所有部門知道功能上線 + 怎麼試用 + 出問題找誰。
【限制】
- 不超過 250 字
- 不用內部術語（「multi-tenant」「OAuth flow」這類技術詞要翻譯）
- 必含：上線時間 / 試用方式 / 截止日 / 出問題找誰
- 結尾加 1 句「為什麼這對你重要」（讓非技術同事也想點開看）
【輸出格式】Markdown，含 H2 標題 + 條列重點 + 結語

產品資訊：
{{產品資訊}}
```

使用：每次新功能上線 5 分鐘可完成。占位符：`{{產品資訊}}`

### 模板 4｜客戶 demo 跟進信

```
【角色】你是雲流科技 CSM Lina（語氣：真誠、溫度、不打官腔）。
【任務】寫一封英文客戶跟進信，根據昨天 demo 的重點 + 客戶疑慮。
【限制】
- 不超過 150 字
- 必含：(1) 感謝 + 一句具體 demo 亮點 (2) 針對客戶疑慮的具體回應 (3) 下次會議的明確 CTA
- 不用「Sorry for any inconvenience」「Thank you for your time」這種公版客套
- 若客戶提到競品比較（如 Zapier），用 1-2 句客觀對比，不貶低競品
【輸出格式】Email 草稿，含 Subject + Body

demo 重點：
{{demo 重點與客戶疑慮}}
```

使用：demo 結束 30 分鐘內 3 分鐘可完成。占位符：`{{demo 重點與客戶疑慮}}`

### 加分：跨模板共用 system prompt

把 4 個模板共享的「你是雲流科技內容運營助手」+「不用空話 + 限制具體 + 輸出格式具體」的精神拆成共用 SP（一份 instruction），4 模板都引用。團隊維護成本降低 4x。

### 常見錯誤（自我比對）

1. **限制只寫「要簡潔」**：要寫「不超過 200 字」「不用 X 詞」這種可驗證的
2. **占位符標示不清**：哪些變數要換、哪些是固定，模糊不清會讓團隊用錯
3. **沒寫使用說明**：模板交給同事但他不知道何時用、要花多少時間
4. **公告用了技術術語**：multi-tenant / OAuth → 一般員工看不懂
5. **跟進信寫成 templates**：應該每次基於 demo 細節重寫（不要全部都套同一句客套話）

**變體｜失敗模式 + 改進版**

挑 1 個模板（例：週報），找一個你過去用 AI 寫過但結果很爛的真實場景，套這個模板跑一次，看新模板有沒有解決舊問題。把「v0 prompt（你過去寫的）vs 模板版 prompt」對比結果寫下來。
