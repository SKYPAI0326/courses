## 1. 系統定位

這套系統解決的是「課程內容由單流程一次寫到底 → 無品質閘門 → 不成熟堆積」的問題。

透過**多角色分工 + 10 個檢核點 + 結構化交接**，讓未來新課與既有課改版都有可重複的 SOP，跨課程一致性可控，單次產出的品質可驗證。

系統設計跑過三類型試飛驗證：skill-operation（Make 定時抓網頁）、concept（SEO 搜尋意圖）、programming（Python 讀 Sheets）。三支講義路徑見文末「pilot 實例索引」。

## 3. 工作流（Part 為單位、10 個 Gate）

```
G1  大綱定案 ──→  PM skill 產出（含 Brand Brief）→ 強制檢核題 4 題
 │
 ├─ G2a Part 1 教案審核 ──→ 設計師逐單元產教案（可讀 _sources/ 參考資料）
 ├─ G3a Part 1 講義驗收 ──→ 講義設計師批次產 HTML + 檢核員 Issue List + 你人眼驗
 │
 ├─ G2b Part 2 教案 → G3b Part 2 講義
 ├─ G2c Part 3 教案 → G3c Part 3 講義
 ├─ G2d Part 4 教案 → G3d Part 4 講義
 │
 ├─ G4  全站收尾驗收
 │
 ├─ G4.5 課程註冊 ──→ course-register：密碼 hash、登錄 inject_gate.py、COURSES.md、密碼對照表、批次注入
 │
 └─ G5  運維就緒 ──→ course-ops 跑 SEO + 連結 + sitemap + 效能稽核 → DEPLOY
```

**每個 G 必答強制檢核題**（由總籌透過 AskUserQuestion 提出，不答不放行）：

- **G1**：(1) N 單元裡哪個最沒把握？(2) 受眾畫像對不上？(3) 哪條 Outcome 最浮泛？(4) 品牌調性跟既有課差異？（pilot 可跳）
- **G2-N**：(1) 這 3 單元的商業情境案例最弱是哪個？(2) 動手練習難度排序？(3) 試跑包需求清單漏哪些 credential？
- **G3-N**：(1) 隨機點 1 頁、3 分鐘跟做到第幾步？(2) AI 示意圖/截圖誤導？(3) 3 單元有沒有「明顯 AI 味」？
- **G4**：(1) 哪單元想下架？(2) 最不順銜接？(3) 對標哪門市售課、品質差多少？

通用模板：`_outlines/_style_guide_template.md`

- 老闆（不命名）— 匯報情境

## 7. 已知系統盲點與應對

### 盲點 1：檢核員檢不到視覺品質

HTML 結構合規（all BLOCKER/MAJOR pass）**不等於**排版好看。Pilot A 第一版全部通過 reviewer，但人眼看是「排版亂七八糟」。

**應對**：G3-pilot 人眼驗收是**必要關卡**，不能靠 reviewer 替代。未來若要自動化，需 Chrome MCP + 截圖 + vision 模型（Phase 3 議題）。

### 盲點 2：Subagent 無法使用 `/` slash command

Agent tool 派遣的 subagent context 獨立，**不能叫 `/build-course-page`**。解法：subagent 的 prompt 必須指示它「讀 `~/.claude/skills/build-course-page/SKILL.md` 並自行執行 Step D1-D4」。

### 盲點 3：試跑包定義在三類型間差異大

- skill-operation 的「試跑包」= 可匯入就能跑的 workflow JSON
- concept 的「試跑包」= 能讓學員模擬跑一次完整流程的題庫 + 策略範本
- programming 的「試跑包」= 複製到本機就能跑的程式碼包

PM/設計師在不同類型下產出的試跑包規格需按上面三種範本填，不能混用。

### 盲點 4：Pilot A 第一版產出品質不良的根因

第一版用了 5 種彩色背景 box 堆疊（concepts-strip/case-block/assets-box/quiz-block/hands-on-box）造成視覺噪訊、11 階 font-size、grid auto-fit 產孤兒。根治方式：取法既有 36h 課程的 section-driven 設計語彙（見 §5）。
