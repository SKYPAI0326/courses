# 提示詞用語升級規範（prompt-language-refinement）

**版本**：2026-04-22  
**狀態**：SSOT · Agent 改寫時唯一參考依據  
**適用範圍**：`courses/` 下所有 HTML 檔案的「提示詞範本」內容  

---

## 1. 動機

### 1.1 為什麼「繁體中文」不夠

只要求「繁體中文」時，LLM 常輸出字型正確但詞彙中國用語（軟件 / 視頻 / 激活 / 博客 / 信息 / 項目 / 隊友 / 頁面…）。學員複製 prompt 貼到 ChatGPT / Gemini / Claude 後，結果看起來像繁體卻讀起來不像台灣人說話。

正確寫法是明確指定「**繁體中文台灣用語**」，強迫 LLM 切換到台灣在地語境。simple-ai 已建立此慣例（`courses/simple-ai/CH1-2.html:376`、`CH2-1.html:298`、`prompt-library.html:273`、`CH1-1.html:434`、`prompt-library.html:599`），其他課程沒跟上。

### 1.2 為什麼「專業」「專家」要具象化

`courses/simple-ai/CH2-2.html` 這一整章在教「不要用『優質 / 專業 / 用心』這類泛用形容詞」。但同一個課程網站其他課的提示詞範本卻大量寫「你是專業的 X」「語氣專業」「你是 X 專家」，等於老師在 A 頁教你別寫 Y，卻在 B 頁示範 Y。學員照抄就跟著踩坑。

「專業」對 LLM 幾乎無資訊量——它不知道你要的是「會跑數字的顧問」還是「會寫文案的顧問」，只好給最安全的通用輸出。改成「年資 + 領域 + 專長」才有脈絡可定位。

## 2. 四條替換規則

### 規則 1：繁體中文 → 繁體中文台灣用語

| 原文 | 改為 |
|---|---|
| `繁體中文` / `使用繁體中文` / `用繁體中文` / `全繁體中文` | `繁體中文台灣用語` |
| `以繁體中文回答` / `用繁體中文回應` | `用繁體中文台灣用語回答` / `用繁體中文台灣用語回應` |
| `繁體中文，xx 字以內` | `繁體中文台灣用語，xx 字以內` |
| `繁體中文撰寫` | `繁體中文台灣用語撰寫` |
| `全程使用繁體中文` | `全程使用繁體中文台灣用語` |
| `僅能使用繁體中文` | `僅能使用繁體中文台灣用語` |

### 規則 2：「專業（角色設定）」→ 具象化身份

**原則**：「專業 + 職稱」太空。改成「年資 / 領域 / 專長」三者擇其二具體化，讓 LLM 有脈絡。

**改寫公式**：`你是{年資}的{職稱}，熟悉{領域具體任務}` 或 `你是{具體背景}的{職稱}，專長是{明確輸出}`。

改寫範例（agents 可參考風格但不硬套）：

| 原文 pattern | 建議改寫 |
|---|---|
| 「你是專業的企劃書撰寫助理」 | 「你是跑過 30+ 場活動的企劃執行，熟悉贊助提案、預算估算與時程倒排」 |
| 「你是一位專業的資料分析助手」 | 「你是懂 Excel 樞紐與基本統計的資料助手，專長是把雜亂數字變可讀摘要」 |
| 「你是專業會議書記 / 會議紀錄秘書」 | 「你是熟悉 OKR 與行動項目追蹤的會議記錄秘書」 |
| 「你是一位專業 HR」 | 「你是有 6 年招募經驗的 HRBP，熟悉科技業面試邀約流程」 |
| 「你是一位專業的文書助理」 | 「你是熟悉商務 email 語境的文書助理」 |
| 「你是資深專案負責人」（保留「資深」+補具體） | 「你是管過 5+ 個跨部門專案的 PM，熟悉週報與主管同步節奏」 |

### 規則 3：「專業（語氣/風格）」→ 具體詞彙

**原則**：「專業」當形容詞不傳遞資訊。拆成「語氣 + 選字 + 節奏」三面向具體描述。

| 原文 pattern | 建議改寫 |
|---|---|
| 「語氣：專業」 | 「語氣：正式、動詞開頭、不用敬辭套話」 |
| 「語氣專業有溫度」 | 「語氣正式但不冷，像已合作 3 年的外部顧問」 |
| 「語氣：專業，不誇大」 | 「語氣：正式、用具體數字、不誇大」 |
| 「專業但不冷漠」 | 「正式但有溫度，像有信任基礎的同事」 |
| 「口語但專業」 | 「口語但有據，避免『首先、其次、綜上所述』套話」 |
| 「正式、專業的商業語氣」 | 「正式的商業語氣，用詞精準、不堆疊形容詞」 |
| 「整體保持專業但有人味」 | 「整體正式但不生硬，避免公文語，保留 1-2 句口語」 |
| 「像跟同事說話，口語但專業」 | 「像跟信任的同事說話，口語但用詞精準」 |
| 「簡潔專業」（作為風格名稱/選項） | 「簡潔具體」或「精簡務實」 |
| 「正式專業」（作為風格名稱/選項） | 「正式精準」或「正式、不堆疊形容詞」 |
| 「親切專業」（作為風格名稱/選項） | 「親切直接」或「親切、有據」 |
| 「更專業」（單純作為修飾） | 「更精準」或「更有據」 |

### 規則 4：「專家（角色設定）」→ 具象化身份

**改寫公式**：同規則 2，把「X 專家」改成「做 X 的 Y，熟悉 Z」。

| 原文 pattern | 建議改寫 |
|---|---|
| 「你是 Email 行銷專家」 | 「你是替 DTC 品牌操作 CRM 的 Email 行銷主理人，熟悉開信率與點擊率優化」 |
| 「你是消費者研究專家」 | 「你是做過 20+ 份受眾訪談的品牌研究員，擅長挖掘潛在動機」 |
| 「你是一位資料清理專家」 | 「你是熟悉 Excel 與 CSV 格式規範的資料清理助手」 |
| 「你是一位資訊分類專家」 | 「你是擅長情感分析與主題標記的文本分析助手」 |
| 「你是資深 Railway 部署專家」 | 「你是熟悉 Railway / Render 部署錯誤模式的後端助手」 |
| 「你是一位會議記錄整理專家」 | 「你是熟悉 OKR 與行動項目追蹤的會議記錄秘書」 |
| 「你是一位跨部門溝通專家」 | 「你是做過 5+ 次跨部門衝突協調的 PM，擅長找共識與分歧點」 |
| 「你是一位台灣歷史專家」 | 「你是研讀過清代與日治時期台灣史料的歷史研究助手」 |
| 「你是科普寫作編輯，擅長把學術論文翻譯成非專家能秒懂的三句話」 | 「你是科普寫作編輯，擅長把學術論文翻譯成高中生能秒懂的三句話」（改「非專家」也要一併改） |

## 3. 絕不改的 7 類情境

改寫前**先判斷**是否屬於以下任一類，屬於則跳過。

### 3.1 教學說明文字中的「專業」/「專家」
給人類讀者看的描述、解說、提醒。例：

- 「這不是懷疑 AI，這是**專業**工作習慣」
- 「建議諮詢相關**專業**人士」
- 「展示品牌**專業**度」
- 「E-E-A-T 的 Expertise（**專業**知識）」
- 「**專業**價值主張」「**專業**積累」「**專業**底線」
- 「真正的**專業**就是這一步」

### 3.2 Metadata / og:title / 頁面標題 / 導覽卡
外部 SEO 與導覽用文案。例：

- `<meta name="description" content="...專業...">`
- `<meta property="og:title">`、`<meta name="twitter:title">`
- `<h1 class="lesson-title">`、`<p class="lesson-tagline">`
- `<div class="card-title">`、`<div class="part-title">`
- `<title>Markdown 轉專業報表 · ...</title>`

### 3.3 課程自我介紹的形容
形容課程品質的行銷文案，不是 prompt 內容。

### 3.4 負面示範 / 反例中的「你是專家」
被刻意留下來讓學員看到錯誤 pattern 的。**判斷依據**：看前後文有無「這樣寫不好」「修正為」「反例」「太模糊」「錯誤示範」「請重寫」等標記。已知反例清單見第 4 節。

### 3.5 `simple-ai/CH2-2.html` 的禁用詞教學（整章）
這一整章在教「不要『優質 / 專業 / 用心』這類泛用形容詞」，命中的「專業」全部是負面引用，**整章不動**。相關檔：

- `courses/simple-ai/CH2-2.html`（所有「專業」命中）
- `courses/simple-ai/CH2-1.html` 部分（「溫暖、親切、專業、優質、用心」這類列舉）
- `courses/simple-ai/CH1-1.html:558` 通過條件引用「提供優質專業服務」

判斷原則：若該句是「AI 輸出了這些空泛詞、要改掉」的負面範例，保留。

### 3.6 工具 / 特性 / 比喻描述
- 「長文件**專家**」（simple-ai/CH1-1.html:270，這是描述 NotebookLM 這個工具的特性）
- 「**專業**術語」「**專業**倫理」（名詞片語）
- 「**專業**背書」「**專業**人士」（泛指 domain 人物）
- 「**專業**簡報」「**專業**提案」（描述產出物類型）

### 3.7 工具/介面的實際名稱
- 「**專家**模式」（Google Ads 的介面選項名稱）
- 「繁體中文」作為介面語言選項（`ntub-seo-ga4/m1-2-localwp.html:160-161` 的 Site Language）
- 「支援繁體中文」（描述工具能力，非 prompt 指令）

## 5. UI 選項標籤處理規則（重要）

很多課程用 tone button / select option / JS 物件 key 作為「風格選擇」，這些字串會被 JavaScript 合併進最終產出的 prompt。**這類要改**，且要同步四處：

### 5.1 需要改的四個位置

以 `courses/gen-ai-140h/part2/PRAC2-1.html` 為例：

1. **按鈕/選項顯示文字**：`<button ... onclick="selectTone(this,'正式專業')">正式專業</button>`
2. **JS 變數賦值**：`let currentTone = '正式專業';`
3. **JS 條件判斷 / 物件 key**：`if (tone === '正式專業') { ... }` 或 `tones['正式專業'] = '...'`
4. **HTML data-\* attribute**：`data-tone="正式專業"`

**四處必須同步改**，只改顯示文字會讓 JS 找不到對應 case。

### 5.2 典型命中與改寫

| 原文 | 改為 |
|---|---|
| `'正式專業'` / `"正式專業"` / `>正式專業<` | `'正式精準'` |
| `'簡潔專業'` / `"簡潔專業"` / `>簡潔專業<` | `'簡潔具體'` |
| `'親切專業'` / 類似組合 | `'親切有據'` |
| option `value="簡潔專業"` + 顯示「簡潔專業」 | value 與顯示都改 |

### 5.3 特例：JS 內的描述字串

`friendly: '語氣親切有溫度，使用自然的繁體中文表達...'`（PRAC1-4.html:1356）這類是「風格描述模板」被合併進 prompt，裡面的「繁體中文」要改成「繁體中文台灣用語」。整個字串都視為 prompt 內容處理。

## 7. Agent 執行流程（標準作業）

每個 Agent 拿到自己負責的課程目錄後：

1. **grep 定位**：
   ```bash
   grep -rn "繁體中文\|專業\|專家" courses/<課程>/ --include="*.html"
   ```
2. **逐檔處理**：對每個命中檔
   - 用 Read 讀命中行前後 10 行，判斷脈絡
   - 核對「反例清單」（本檔第 4 節）
   - 依「絕不改情境」（本檔第 3 節）決定是否跳過
   - 要改就用 Edit，依第 2 節規則四條
   - UI 選項標籤依第 5 節，四處同步
3. **每檔改完立即 lint**：
   ```bash
   python3 docs/lint-page.py <file>
   ```
   exit 0 才算完成；BLOCKER 要修到 0。
4. **回報格式**：
