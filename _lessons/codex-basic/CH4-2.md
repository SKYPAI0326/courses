## 教學流程（Teaching Flow）

### 破題 / Hook（2 min）

你已經在 CH4-1 做出一個 HTML 儀表板，追蹤弄一下工作室 6 月月結的進度。

現在陳主管說：「能不能做一頁讓大家都看得到的『部門報帳須知』？讓各部門同事自己查規定、提交報帳資料，我不用一直接電話。」

這件事以前要找人架網站、寫程式、部署主機——你一個人搞定不了。今天 Codex 全程包辦：**規劃 → 製作 → 部署上線 → 自動測試**，你一行程式都不用寫。

不過「上線」是真的對外公開。所以本單元除了看 Codex 怎麼做，還要確認你知道**哪兩個關鍵動作一定要人先確認**，才能讓它跑。

### 概念 / Concepts（3 min）

- **規劃模式（plan mode）** — 啟用後，Codex 不會直接開始做，而是先提問、釐清需求，列出完整執行計畫讓你確認。避免需求還沒說清楚，Codex 就做出一堆用不到的功能，浪費時間和額度。
- **`/pet` 寵物通知** — 在輸入框輸入 `/pet`（Settings > Appearance 可設定），螢幕角落會出現一個小寵物。Codex 有任何進度就跳通知，讓你在它做長任務時去做別的事，不用守在旁邊枯等。
- **Sites plugin 部署（Codex 內建）** — Codex 內建 Sites plugin，由 OpenAI 託管，可直接把做好的網站部署上線，你不用自己搞主機。部署完成後產生 production URL；Sites 支援 access modes（owner/admin、workspace、custom），讓你控制誰看得到。部署前建議先 save version，確認內容無誤後再 deploy。（此功能為**講師示範**，不保證所有學員帳號當下可用；備案：展示本機 HTML 預覽。）
- **Computer Use** — Codex 的進階功能：啟用後，Codex 能像真人一樣操作你的瀏覽器和電腦軟體——移動游標、點按鈕、在欄位裡填字、送出表單。把機械式重複的測試任務交給它，你不用親自動滑鼠。（此功能為**講師示範**，不保證所有學員帳號當下可用。）

### 操作示範 / Demo（10 min・講師示範）

---

**把關紅線 #1（先聲明，後示範）**：透過 Sites plugin 部署後，production URL 就是可存取的正式網址。Sites 支援 access modes 限制存取對象，但分享連結或放寬權限之前，你必須自己先確認：頁面上的文字是否正確、有沒有不該讓目標對象看到的資訊（例如報帳帳號、個人聯絡方式）。Codex 不會主動幫你過濾財務敏感資料，那是你的責任。建議流程：先設受限存取（owner/admin）→ 審內容 → 確認無誤後再決定是否放寬給更多人。

**把關紅線 #2（先聲明，後示範）**：Computer Use 會在你的電腦上真的操作瀏覽器。用之前要確認：測試的對象是你可以亂填的測試環境，而不是正式系統或客戶資料頁面。測試完，去 Gmail 確認收到轉寄信，才算驗證成功。

---

#### 步驟 1：啟用規劃模式，先擬計畫（約 1 min）

講師在 Codex 主頁把「規劃模式」切換到啟用。

把弄一下工作室的 Logo 和正門口相片（已在工作資料夾）拖入。送出提示詞：

> 幫我做一頁「部門報帳須知」小站。頁面需要包含：報帳流程說明、各部門填寫規定，以及一個讓同事提交報帳表單的留言區。

使用規劃模式，Codex 不會馬上動工，而是先問幾個問題，例如：
- 報帳流程要幾個步驟？
- 表單要收哪些欄位（部門、金額、說明、附件）？
- 網頁風格要正式或輕鬆？

講師一一回覆後，Codex 列出完整執行計畫：頁面架構、表單欄位、部署目標。講師確認後請 Codex 正式開始。

> **為什麼先擬計畫**：製作網站要花時間和額度。需求不清楚就動工，做到一半才發現方向錯，重來會更貴。多花一分鐘釐清需求，省掉重做的十分鐘。

#### 步驟 2：製作小站 + 啟用 `/pet` 寵物通知（約 2 min）

Codex 開始製作。因為做網站通常要花較長時間，講師在輸入框輸入：

> /pet

螢幕角落跳出小寵物（在 Settings > Appearance 可設定樣式），Codex 有進度就會主動通知。講師說：「這段時間你可以去做別的事，不用守在這裡。」

過了一段時間（影片中快轉），小寵物跳通知：製作完成。講師回到 Codex 視窗，把 `/pet` 通知關閉。

#### 步驟 3：預覽網站 + 用「註解」功能微調設計（約 2 min）

Codex 提供一個本機預覽網址。講師打開後看到：
- 「部門報帳須知」標題 + 弄一下工作室 Logo
- 報帳流程說明（三步驟：填表 → 上傳憑證 → 送交陳主管）
- 各部門填寫規定（本部門的費用上限與填寫格式）
- 一個表單：欄位含「部門」「姓名」「金額」「說明」「送出」按鈕

講師發現標題的字體大了一點，用**右上角「註解」功能**：
1. 啟用註解模式
2. 把滑鼠移到標題，點一下
3. 在對話框輸入：「標題字體縮小一號，顏色改深咖啡色」
4. 送出

Codex 立即修改程式碼，頁面刷新，標題改好了。

> **這個改法的意義**：你不需要知道 CSS 怎麼寫，用自然語言點哪改哪。頁面改到滿意，再進下一步部署。

#### 步驟 4：用 Sites plugin 部署 + 設定 access mode（約 3 min）

講師先執行 save version（Sites plugin 內），確保有可回溯的快照。

接著確認頁面內容正確、沒有不該讓受限對象以外的人看到的資訊（把關紅線 #1 兌現的動作）。

向 Codex 說：「請透過 Sites plugin 把這個網站部署上線。」

Codex 使用內建 Sites plugin（OpenAI 託管）直接部署，幾分鐘後顯示 production URL，例如：`https://dongyixia-reimbursement.openai.com`

講師隨即在 Sites 設定中把 access mode 設為「owner/admin」（只有自己和管理員看得到），並說明：

> **Sites access modes**：owner/admin（只有自己）→ workspace（同工作區成員）→ custom（自訂名單）→ 公開。陳主管確認內容無誤、才放寬到 workspace 或更廣。分享網址前先確認 access mode 符合目標受眾。

#### 步驟 5：用 Computer Use 自動填表驗證（約 2 min）

講師在 Codex 首頁輸入：

> 請前往剛才部署的 production URL，隨意填入一筆測試資料送出表單（部門填「測試部門」，金額填 0），確認表單送出成功。

Codex 啟用 Computer Use。螢幕自動彈出瀏覽器視窗——學員看到 Codex 操控游標、自己移動到表單欄位、逐一填入內容、點「送出」按鈕，全程不需要講師動手。

表單送出後，講師打開 Gmail 確認是否收到通知信，裡面有剛才填進去的測試資料。

對比給學員看：**你剛才什麼都沒動，但 Codex 真的填了一筆表單。**

> **把關紅線 #2 兌現**：講師說：「你看到 Computer Use 有多省事。但它能操作的不只是測試網站——它可以操作你電腦裡的任何瀏覽器視窗。如果指令寫錯，它可能去點到你不想讓它碰的地方。使用前，確認你的任務範圍是可控的。」

### 觀察單 / Watch（2 min・不繳交）

看完示範，心裡想清楚這三件事（給自己的，不用說出來）：

1. 從「需求描述」到「正式網址」，Codex 做了哪幾個步驟？哪一步是**你做了才讓它繼續**的（不是它自動的）？
2. 如果這頁小站上線後，同事發現有一個欄位說明寫錯——**誰的責任**？Codex 沒有辦法事先知道你的公司規定是否正確。
3. Computer Use 測試完，你怎麼確認它**真的把表單送出去了**，而不是只跑完動作、信沒收到？（答案：去 Gmail 確認收到轉寄信。）

然後想一件事：**你現在有哪一頁資訊想公開給公司內部，讓大家自己查、不用一直來問你？**

### 檢核 / Verification（3 min・口頭 + 全課收束）

學會的標準（不必上機）：

- 你能說出**規劃模式的用途**（先釐清需求再動工，而不是「讓 AI 想更仔細」）。
- 你能說出**部署後分享前要確認什麼**（access mode 是否符合目標受眾、頁面內容有無財務敏感資訊外洩）。
- 你能說出 **Computer Use 的一條邊界**（任務範圍要控在可控環境，不能讓它亂開正式系統）。

## 商業情境案例（Case）

**角色**：你（弄一下工作室 行政會計）

## 動手練習題（Hands-on Exercise）

本單元無上機實作（示範為主，Sites 部署與 Computer Use 需要帳號方案與本機環境）。以「觀察 + 一句話」替代：

**題目**：看完部署示範，想清楚你在弄一下工作室的工作裡，**有沒有一份資訊**（例如：報帳規定、差旅申請流程、供應商聯絡表）如果做成對內小站，可以省掉你每週回答幾次重複的問題。

**預期成果**：一句具體的話。好：「把差旅費報帳的截止日期和附件規定做成一頁，讓業務同事自己查」。不好：「做一個網站」（太籠統）。

**完成標準**（self-check）：
- [ ] 能說出這頁資訊**目前要花你多少時間重複說明**（一週幾次、幾分鐘）
- [ ] 能說出這頁資訊**有沒有不能公開的部分**（若有，上線前要把那段拿掉）
- [ ] 能說出 Computer Use 測試這頁的方式（讓 Codex 填測試資料，去信箱確認收到）

## 常見錯誤 3 條（Common Pitfalls）

1. **錯誤現象**：跳過規劃模式直接叫 Codex 做網站，做完才發現表單欄位少了「部門」，或報帳規定說明和實際的不一樣，要重做。
   **原因**：需求沒有事先釐清，Codex 只能根據你說的做，沒說到的就自己猜或省略。
   **解法**：任何比「單頁文件」更複雜的任務，先啟用規劃模式。花一分鐘讓 Codex 問清楚需求，省掉重做的時間和額度。

2. **錯誤現象**：部署完就把 production URL 丟給同事——結果頁面上的費用上限金額寫的是舊規定，或出現了某個部門的內部分機號碼不應該讓那個對象看到。
   **原因**：Sites 部署後 production URL 立即可存取，access mode 決定誰看得到。Codex 不知道哪些內容對哪些對象有問題。
   **解法**：部署後先設 access mode 為受限（owner/admin）、自己逐段讀過頁面。財務相關數字、人員聯絡資料、內部系統連結確認無誤，再決定放寬 access mode、分享連結。

3. **錯誤現象**：用 Computer Use 測試時，看到它在螢幕上操作就放心了，沒有去收件匣確認是否真的收到通知。結果同事後來填表，通知沒來，才發現表單的後端設定根本沒完成。
   **原因**：Computer Use 只負責執行「填表單、送出」的動作，它不會去你的信箱幫你核實收信結果（除非你下指令叫它去）。
   **解法**：Computer Use 送出表單後，你親自打開信箱確認通知有進來，才算測試通過。自動化的最後一步，還是要人眼確認一次。

## 檢核題 2 條（Quiz）

**Q1（概念驗證）**：啟用「規劃模式」之後，Codex 的第一個動作是什麼？
- [ ] A. 立刻開始撰寫程式碼
- [ ] B. 向你提問、釐清需求，列出執行計畫讓你確認後再動工 ←（正確答案）

**Q2（應用驗證）**：你用 Codex 做好「部門報帳須知」小站並透過 Sites plugin 部署，陳主管說「先不要讓全公司看到，我想自己先看一次」。在陳主管確認之前，你應該怎麼做？
- 預期答案要點：
  1. **部署後立即把 access mode 設為受限**（owner/admin），只有自己能看到 production URL，先把連結分享給陳主管審核——不要在陳主管確認前放寬給更多人。
  2. 特別請陳主管確認：**有沒有財務敏感資料外洩**（費用上限、帳號資訊、人員聯絡方式），Codex 不會主動過濾這些。
  3. 如果陳主管有修改意見，用「註解」功能改完、再 save version，確認後再決定是否把 access mode 放寬給 workspace 或更廣。
