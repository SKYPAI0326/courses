## 教學流程（Teaching Flow · 95 min）

### 1. Decision Checkpoint（3 min）

開場 3 分鐘做架構分流確認。問題：**什麼資料屬於個人層 / 部門層 / 跨來源層？決策邊界是什麼？**

5 部門代表各答 1 句（30 秒一輪）：

- 凱倫（財務）：「我每天記的對帳便箋是個人層、公司會計政策是部門層、年度合併報表流程是跨來源層。」
- 小李（業務）：「我的會議紀錄屬個人層、業務部報價政策是部門層、新品引進要整合採購倉儲算跨來源。」
- 怡君（電商）：「我的競品筆記是個人層、電商平台規範是部門層、跨平台促銷對齊是跨來源。」
- 阿翔（倉儲）：「我的盤點手記是個人層、倉儲 SOP 是部門層、跨倉調撥牽涉採購業務算跨來源。」
- 采蓁（採購）：「我的供應商私下筆記個人層、採購流程部門層、年度供應商評鑑跨採購業務財務算跨來源。」

講師收斂三軸切分：

1. **資料擁有者**：只有你自己 → 個人；同部門共用 → 部門；多部門共用 → 跨來源
2. **共享範圍**：完全私人 → 個人；同部門 share → 部門；跨部門 share → 跨來源
3. **更新頻率**：每天 / 每週 → 個人；每月 / 每季 → 部門；半年以上 → 跨來源

三軸**任一**達標就升級到上一層。

**重要說明**：本課 95 min **個人層佔一半時間深度教**——因為個人層是你回部門「**第一週就要做的版本**」。部門層 / 跨來源層放在後段快速示範，讓你看懂未來怎麼做、但不要求課堂跑完。

### 2. 三層架構引入 + 個人層深度示範（35 min）

#### 2.1 三層架構一頁圖（5 min）

```
┌──────────────────────────────────────────────────┐
│  跨來源層（半年更新 · 多部門 share · 戰略決策用）        │
│  例：新品引進綜合資料 / 年度策略 / 跨部門 KPI 對齊       │
│  source 數：30-50 · 命名：「跨來源-{主題}-{年}」          │
│  ★ 進階情境，CH2-1 會深入；本課快速看過                │
├──────────────────────────────────────────────────┤
│  部門層（每月 / 每季更新 · 同部門 share · 操作 SOP 用）   │
│  例：業務部報價政策 / 採購流程 SOP / 倉儲盤點規範         │
│  source 數：15-25 · 命名：「{部門}-部門共用-v{版本}」     │
│  ★ 進階情境，本課快速示範；要真的建需先過版本管理關卡    │
├──────────────────────────────────────────────────┤
│  個人層（每天 / 每週更新 · 完全私人 · 自己日常工作用）     │
│  例：你的會議紀錄 / 工作日誌 / 個人筆記                   │
│  source 數：5-15 · 命名：「{你的名字}-個人-{季度}」        │
│  ★ 本課重點，深度教 50 min — 你回家第一週要做的版本     │
└──────────────────────────────────────────────────┘
```

三層**不是分類**，是**決策邊界**：

- **個人層**回答「**我**昨天到底承諾了什麼」
- **部門層**回答「**業務部**的標準做法是什麼」
- **跨來源層**回答「**這個跨部門決策**牽涉哪些已知資訊」

> **本課野心邊界**：學員 95 min 課堂能跑出的、且回家能複製的，**只有個人層**。部門層 / 跨來源層需要公司治理配套（版本管理、跨部門權責），不是學員一人下週能落地的事。本課讓你看懂、知道未來怎麼推動，但**不要求課堂跑完三層**。

**第 2 步：3 條上傳規則（學員回家要套用 · 7 min）**

個人層最大風險不是 source 選錯，是「**不確定能不能上就先上**」。下這 3 條規則：

| 規則 | 內容 | 範例 |
|---|---|---|
| 規則 1（**能上**） | 你寫的 + 只你看 + 隨時改 + 不含他人個資 | 自己整理的客戶觀察筆記、自己摸出來的 SOP |
| 規則 2（**不能上**） | 含他人個資 / 含公司機密 / 屬部門共用文件 | 客戶名單 .xlsx、財報、業務 SOP v2026 |
| 規則 3（**不確定**） | 暫存 Google Keep 或本地檔案，**不上 NotebookLM**，下週問主管或資安窗口 | 「老闆轉發給我但他沒明說機不機密的內訓 PDF」 |

3 條具體判斷示範：

| source 候選 | 進個人層？ | 為什麼 |
|---|---|---|
| 小李寫的「A 客戶不爽我們服務」私下觀察 | ✓ 進 | 規則 1：你寫的 + 只你看 + 不含他人個資 |
| 客戶名單 customer-list.xlsx | ✗ 不進 | 規則 2：含他人個資（PII） |
| 公司內部 wiki 上的「業績核算公式」截圖 | ✗ 不進 | 屬部門層（同部門人都該用同一份） |
| 部門經理轉發的「明年加薪規畫」word | ⚠ 暫存 | 規則 3：不確定（轉發人沒明說機不機密），先 Google Keep、下週問 |

<div class="demo-takeaway">個人層 3 條上傳規則 = 能上 / 不能上 / 不確定就先暫存。第三條是 80% 學員會卡的地方，課堂建立習慣。</div>

**第 3 步：命名規則（3 min）**

```
個人層命名：{你的名字}-個人-{季度}
範例：小李-個人-2026Q1
```

為什麼這樣命名：

- 用「名字」開頭 → notebook 列表排序時自己的個人 notebook 連在一起
- 含「個人」字樣 → 一眼分辨層級（避免後續跟部門層搞混）
- 含「季度」 → 每季度建新 notebook、舊的封存，避免 source 累積到 50 上限

**第 4 步：5 題常用查詢題（針對你部門最常被問的事 · 10 min）**

不是隨便問，是針對「**你部門最常被問的事**」設計 5 題。小李是業務副理，他被問的高頻是「進度」「承諾」「客戶狀態」。

**5 題的設計邏輯**：

- **2 題承諾類**（Q1 / Q2）：別人問你「上次說好的呢」，你 30 秒查得到

- **2 題客戶/領域類**（Q4 / Q5）：你日常工作的核心對象，做季度回顧用

### 3. 學員上機 — 個人層完整建置（25 min）

每位學員從 5 部門變體選 1 個，**完整跑完個人層建置**。

#### 3.1 5 部門變體素材

| 部門 | 主角 | 素材包 | source 包含 |
|---|---|---|---|
| 財務 | 凱倫 | `personal-finance-5.zip` | 月底對帳便箋 / 預算追蹤筆記 / 個人結帳 SOP / 異常案例觀察 / 跨部門待簽核紀錄 |
| 業務 | 小李 | `personal-sales-5.zip`（主版） | CH1-1 產出會議紀錄 / 12 月工作日誌 / 個人客戶分級 / 報價追蹤 / 客戶反饋觀察 |
| 電商 | 怡君 | `personal-ecom-5.zip` | 競品筆記 / 上架 SOP / 平台變動觀察 / 流量異常記錄 / 客服高頻問題 |
| 倉儲 | 阿翔 | `personal-warehouse-5.zip` | 盤點手記 / 缺貨追蹤 / 異常處理日誌 / 跨倉調撥筆記 / 揀貨效率筆記 |
| 採購 | 采蓁 | `personal-procurement-5.zip` | 供應商私下觀察 / 議價筆記 / 評鑑追蹤 / 待付款清單 / 品項追蹤 |

#### 3.2 5 步驟（25 min）

1. **（3 min）** 解壓你部門變體素材包、看清楚 5 份各是什麼
2. **（4 min）** 套 3 條上傳規則檢查每份：能上 / 不能上 / 不確定（**不確定的先剔掉**）
3. **（5 min）** 在 NotebookLM 建新 notebook、命名「{你的名字}-個人-2026Q1」、上傳過濾後的 source
4. **（8 min）** 仿小李 5 題範例，**改寫成你部門的 5 題常用查詢題**、跑跑看
5. **（5 min）** 寫 1 題驗證題（故意問「超出個人層」的問題）、跑跑看 + 把 Min 版設計表填掉（見第 6 段）

#### 3.3 講師走場規則

每 5 min 巡一遍，重點抓三件事：

- **「不確定」的 source 有沒有剔掉？**（學員最常忽略規則 3，照樣上）
- **5 題查詢題有沒有對到自己部門的高頻問題？**（不要照抄小李的）
- **驗證題能不能拿到「超出範圍」回應？**（拿到答案就代表 source 越界混進來）

跨來源層 = **為特定跨部門主題量身建** notebook，不是把部門層的 source 全倒進去。

  - 供應商成本表.xlsx
    → 雙方協商：業務只能看「成本區間」非「明確數字」
    → 建一份「成本區間版」進 notebook，明細版不進

  ✗ 不進：
  - 採購部供應商議價策略內訓.pdf（純部門內部）
  - 業務部個別業務員業績排名.xlsx（與新品引進無關）
</div>

<div class="demo-takeaway">跨來源層 = 為特定主題量身選 source + 主決策者主導 + 共享規則寫死。**跨部門權責政治是最大阻力**，CH2-1 會深入練。</div>

### 5. 現實阻力處理段（10 min）

部門層 / 跨來源層回部門落地會撞到 3 大現實阻力。這段給你**對策表**，不期望課堂解決，期望你**識別 + 選對方向**。

#### 5.1 阻力 1：公司不允許上真實資料到免費版（3 min）

**現實**：免費版 NotebookLM 雖然 Google 宣稱不會用 source 訓練模型，但**仍存在 Google 雲端**。多數公司資安政策不允許客戶 PII / 供應商機密 / 財務細節上免費版。

**3 個對策**：

| 對策 | 做法 | 適用情境 |
|---|---|---|
| A · 模擬建 | 用本課提供的虛構素材包**在課堂練熟**，回家**不上真資料、只模擬建 notebook + 查詢題框架** | 公司資安明示禁止 / 你還沒問清楚 |
| B · 公開段 | 用內部 wiki 的**公開段**（FAQ / 公開政策 / 對外資料）建第一個 notebook | 公司有部分文件本來就對外公開 |
| C · 💰 升級 | 升級 💰 NotebookLM Plus（資料隔離）或申請**公司核可的內部 AI 工具** | 必須含敏感資料且公司願意付錢 |

**判斷流程**：先問資安窗口能不能上 → 不行就走 A → 有公開資料就走 B → 真的需要就提 C 給主管。

#### 5.2 阻力 2：SOP / 政策沒有版本管理（3 min）

**現實**：部門層 selection 規則「**已核可 + 半年內**」假設公司 SOP 有正式版本管理（v1.0、v2.1、核可日期、修改人）。**多數中小企業沒有**——SOP 散在 Word / PDF / Notion / Google Drive，沒人知道「哪份是最新核可版」。

**3 個對策**：

| 對策 | 做法 | 適用情境 |
|---|---|---|
| A · LMD 代替 | 用「**最後修改日期 + 修改人**」當代替：上傳前在檔名加 `_LMD-2025-12_張經理` | 公司有檔案但沒版本制度 |
| B · 標警語 | source 描述明標 ⚠「**本 source 版本未經核可，僅供查詢參考，最終以紙本/系統為準**」 | 你不確定哪份才對、但需要先用起來 |
| C · 個人共用層 | 建「**個人共用層**」（自己整理 + 自己標記版本 + 自己負責），不假裝是「部門共用」 | 公司治理短期不會改、但你需要小範圍跟同事共用 |

「個人共用層」概念：name 仍含「個人」，但**邀請 2-3 個同事 share**。本質還是「你負責、你維護」，不要佯裝部門權威。

#### 5.3 阻力 3 預告：跨來源權責政治（4 min）

**現實**：跨來源層卡的不是技術，是**人際 + 權責 + 資訊政治**——

- 採購不想讓業務看到供應商成本表（談判風險）
- 財務不想讓業務看到部門預算明細（部門平衡）
- 業務不想讓倉儲看到客戶議價空間（怕被通報）

**這在 CH2-1 會深入練**。本課先給你 2 個方向：

1. **共享規則先寫死**：每份爭議 source 進 notebook 前，跨部門協商「誰能看 / 看到哪個程度」並寫進 notebook 描述
2. **主決策者主導**：跨來源 notebook 由「最後要靠這個 notebook 做決策的人」召集（例新品引進的主決策者是商品總監），不是「最會用 NotebookLM 的人」

<div class="demo-takeaway">3 大阻力都不是 NotebookLM 教學能解決的，但**識別出阻力 + 選對方向** = 你回部門推動的第一步。回家用 Min 版設計表先跑個人層、別卡在三層全要的迷思。</div>

## 常見錯誤 5 條（Common Pitfalls）

### 1. 把所有東西倒進 1 個 notebook（無分層）

**錯誤現象**：學員一開始想「不分層比較省事」，把個人筆記 + 部門 SOP + 跨部門資料全部丟同一個 notebook。結果問「我上週開過幾次會」答得出來、問「業務部報價政策」也答得出來、但**兩個答案的引用混雜** — 個人筆記裡剛好提到「老闆說折扣超過 20% 要簽」被當成部門政策。

**原因**：NotebookLM 不會主動分層，使用者沒設計就會全混。

**解法**：

1. 開課時老師會撐住「分層不是麻煩、是查得準的前提」這條原則
2. 從個人層開始建（先把 CH1-1 跑出的會議紀錄存起來作第 1 個 source），不要從「公司大全」這種想法開始
3. 三軸切分：擁有者 / 共享範圍 / 更新頻率，**任一**達標就升級
4. 寧可一開始 source 少、慢慢長，不要一開始全倒、後面再分（後面分非常痛）

---

### 2. 把客戶個資 / 供應商成本表丟進免費版 notebook

**錯誤現象**：怡君為了查「VIP 客戶權益」，把含完整客戶名單的 customer-list.xlsx 上傳到 NotebookLM 免費版。後來公司資安稽核發現、被記點。

**原因**：免費版 NotebookLM 雖然 Google 宣稱不會用 source 訓練模型，但**仍存在 Google 雲端**，公司資安政策通常不允許客戶 PII / 供應商機密上免費版。

**解法**：

1. 個人層**規則 2「不能上」**就是擋這種錯（含他人個資 / 含公司機密 / 屬部門共用文件）
2. 不確定就**規則 3 暫存**（Google Keep 或本地檔案、下週問資安窗口）
3. 真的要用就走「阻力 1 對策」：A 模擬建 / B 公開段 / C 升級 💰 NotebookLM Plus
4. 學員回家前先問：「**這份 source 上 Google 雲端可以嗎？**」公司資安沒明示就停下來問，不要當小白鼠

---

### 3. 跨來源層變成「全公司資料垃圾桶」

**錯誤現象**：小李建跨來源 notebook「2026 公司全資料」，把業務 / 採購 / 倉儲 / 財務 / 電商部門 SOP 全部倒進去，想說「以後問什麼都查得到」。結果問「新品引進流程」NotebookLM 引用了 8 份不同部門的 source、答案 800 字夾雜各部門視角，**沒人看得懂**。

**原因**：跨來源層**不是儲存庫**，是**為特定跨部門主題量身建的 notebook**。沒有主題的跨來源 notebook 等於沒有焦點，查詢結果必然散。

**解法**：

1. 一個跨來源 notebook = 一個主題（例：「新品引進」「促銷備貨」「供應商評鑑」），主題寫進命名
2. source 進來前問「對這個主題真的有用嗎？」，不確定就不收
3. 不同主題建不同 notebook，**不要追求「一個 notebook 答所有跨部門問題」**
4. 跨來源層**不是學員一人能建的事**，需要主決策者主導 + 共享規則協商（見阻力 3）

---

### 4. 沒寫驗證題、不知道知識庫設計是否成功

**錯誤現象**：學員建完個人層 notebook、跑了幾題查詢都答得出來，覺得「設計成功」。但下週回家用真實資料時，問「業務部報價政策」結果 NotebookLM 用個人層的會議紀錄答了（因為小李會議裡剛好提到折扣案例），學員以為這就是部門政策、拿去跟同事爭執。

**原因**：沒有設計**驗證題**（故意問「超出本層範圍」的問題測誠實度），無法確認 notebook 會不會「越界」答。

**解法**：

1. 每層 notebook 必寫 1 題驗證題：個人層問部門政策、部門層問個人行程、跨來源層問純部門內部 SOP
2. 驗證題的**正確回應**是 NotebookLM 說「sources 中未明確記載 / 建議查閱另一層」，不是給出答案
3. 若驗證題被「答出來」 → notebook 設計有問題（source selection 過寬、有越界 source 混入），回頭剔
4. Min 版設計表強制留 1 格驗證題、跑出來才能離場

---

### 5. 以為 NotebookLM 能取代公司文件管理系統

**錯誤現象**：學員上完課很興奮、把所有 SOP / 政策 / 客戶資料 / 供應商資料倒進 NotebookLM 想當「公司文件中心」。3 個月後新人問「最新版報價政策在哪」、有人答 NotebookLM、有人答 SharePoint、有人答 Google Drive——**沒人知道哪份才是真相**。

**原因**：NotebookLM 是「**查詢工具**」不是「**文件管理系統**」——

- 沒有權限控制（誰能改誰不能改）
- 沒有版本追蹤（v1.0 → v2.0 → v2.1 改了什麼）
- 沒有修改紀錄（誰在何時改了什麼）
- 沒有審核流程（部門經理核可才能上線）

NotebookLM 是「**讓你快速查公司既有文件管理系統內容**」的工具，**不能取代公司文件管理系統本身**。

**解法**：

1. **公司文件中心仍走原來的位置**（SharePoint / Google Drive / Notion / 紙本檔案室），不要搬到 NotebookLM
2. NotebookLM 的 source 是**從文件中心匯出的「快照」**——查得快但不是真相，真相還在原系統
3. 個人層**規則 2「不能上」**第三條就擋這個錯（「屬部門共用文件」不該進個人層 NotebookLM）
4. 如果你想推動「全公司用 NotebookLM 當文件中心」——**請走 💰 企業版採購 + 資安評估 + 治理導入**，不是用免費版湊
5. 阻力 2「SOP 沒版本管理」如果是公司本來的問題，**不是 NotebookLM 能解決的**；走「個人共用層」對策 C 作短期替代

**Q3（驗證題設計）**：

**情境**：你是阿翔（倉儲），剛建好「阿翔-個人-2026Q1」notebook，含 5 份盤點手記 / 缺貨追蹤 / 異常處理日誌。你要為這個 notebook 設計**1 題驗證題**。

- [ ] A. 「阿翔這週做了幾次盤點？」
- [ ] B. 「倉儲部盤點 SOP 是什麼？」 ←（**正確答案**）
- [ ] C. 「阿翔上個月處理了哪些異常？」

- D 是通用概念、與本 notebook 設計無關

加分：
