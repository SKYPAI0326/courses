## 教學流程（Teaching Flow · 72 min）

### 1. Decision Checkpoint（3 min）

開場 3 分鐘，回到 CH0 埋的鉤——跨來源綜合彙報的責任分工怎麼分？而且回部門以後，**真實狀況跑得起來嗎**？

dialog-box（講師問全班）：
```
今天 3 個人要合作做 1 頁綜合彙報。請答 4 題：
  Q1：誰負責建 3 個獨立的 NotebookLM 知識庫？
  Q2：誰負責把 3 個知識庫取證結果整合成 1 頁彙報？
  Q3：誰負責驗證最後彙報內容對不對？
  Q4：回公司複現時，**3 種狀況一定會撞上**——你怎麼處理？
       (a) 你手上資料含客戶代碼/敏感欄位，公司資安規定不能上傳免費版
       (b) 拿到的同業報價單沒寫版本號，不確定是這個月還是上個月的
       (c) 鄰部門主管不願共享客戶詢問紀錄（怕對方看到議價底牌）
```

expected-output（預期學員答 + 講師收斂）：
> Q1：**領域擁有者建知識庫**——采蓁建採購層 / 小李建業務層 / 阿翔建倉儲層。
> Q2：**彙報人整合**——這次由小李（業務副理）整合，因為新品引進是業務發起。
> Q3：**收件人驗證**——總經理收到後會丟回各領域擁有者再次確認。
> Q4：今天會在情境裡親自處理這 3 種阻力，產出一份「跨來源決策備忘」分 3 層：**可用 / 不可用 / 待主管授權**。回家不是「全做」也不是「不做」，是**明確標出哪段能跑、哪段被擋下、哪段要往上請示**。
>
> 三句收斂：「**建知識庫的人不負責整合；整合的人不負責驗證；驗證的人不負責建知識庫。**」這就是跨來源責任分工。再加一句：「**遇到資料阻力不放棄整個案，改寫成 3 層備忘讓主管看清楚邊界。**」

demo-takeaway：跨來源整合的真實工作不是「3 人圍著資料一起寫」、是「**分工 + 認清阻力 + 把跑不到的部分明確標出**」。今天每個人扮 1 個角色，責任不重疊，現實阻力一個個過。

### 2. 情境設定（10 min · 含 3 阻力內建）

弄一下精選商行決定要引進 1 款新品——從 A 品牌進口運動水壺。這個決策跨 3 個部門，每個部門手上資料各自獨立，**但都帶著真實工作會撞到的阻力**——課堂上不假裝資料乾淨、人人配合。

**今天的任務**：3 人 1 組（采蓁採購 + 小李業務 + 阿翔倉儲），合作產出 **2 份產物**給總經理 + 給自己：
1. **1 頁綜合彙報**（給總經理）：含「**推薦 / 風險 / 不確定性 ⚠**」三段
2. **跨來源決策備忘**（自己內部盤點用）：分 **可用 / 不可用 / 待授權** 三層，明示哪些 source 通過 3 阻力測試

3 部門各自獨立資料 + **內建 3 阻力**：

| 角色 | 部門 | 手上資料 | 知識庫名稱 | 撞到的阻力 |
|------|------|---------|-----------|-----------|
| **采蓁** | 採購 | A 品牌報價單 + 替代供應商 3 家比較 + 6 個月進貨歷史 | 採購層 · A 品牌新品評估 | **阻力 B**：A 品牌報價單**沒有版本號**（不確定是 12 月還是 1 月報價） |
| **小李** | 業務 | 類似品銷售 50 筆 + 客戶詢問 12 筆 + 競品價格表 | 業務層 · 運動水壺市場資料 | **阻力 C**：業務部主管不允許客戶詢問紀錄含名單進跨來源 notebook（怕採購看議價底牌） |
| **阿翔** | 倉儲 | 當前庫存 + 3 個月進出貨流量 + 安全庫存規則 | 倉儲層 · 庫存容量評估 | **阻力 A**：當前庫存資料**含客戶代碼**（公司資安規定禁上傳免費版 NotebookLM） |

**3 種阻力 + 課堂選擇**：

dialog-box（每位學員看自己角色的阻力 + 必做選擇）：
```
阻力 A（阿翔 倉儲）：「當前庫存資料含客戶代碼」
  → 你必須選 1 種處理方式：
     (a) 改用 sample 模擬版（不含客戶代碼）
     (b) 自己手動匿名化原資料（去掉「客戶代碼」欄位後再上傳）
     (c) 不上傳，改用「描述性紀錄」貼進 notebook（如：「當前庫存約 600 件、近 30 天進貨 2 批」）
  → 寫進決策備忘哪一層？選 (a)(b) 進「可用」/ (c) 進「不可用 → 改替代方式」

阻力 B（采蓁 採購）：「A 品牌報價單沒版本號」
  → 你必須選 1 種處理方式：
     (a) 仍引用，但每條取證強制加 ⚠「報價版本未明，建議向 A 品牌再次確認」
     (b) 不引用具體數字、改用區間（如「報價約 NT$ 120 ± 10」）
  → 寫進決策備忘哪一層？兩種都進「可用（含 ⚠ 標記）」，但選 (b) 信心度更低

阻力 C（小李 業務）：「業務主管不允許客戶詢問紀錄含名單進跨部門 notebook」
  → 你必須選 1 種處理方式：
     (a) 用「需求描述」代替「客戶名單 + 報價」（如：「12 筆中 7 筆關注運動水壺功能」，不含客戶名）
     (b) 寫共享規則申請「客戶名隱去、僅留需求類型」，等主管授權
     (c) 不進跨來源 notebook，改用替代資料（如競品價格表替代客戶議價底牌）
  → 寫進決策備忘哪一層？(a)(c) 進「可用」/ (b) 進「待主管授權」
```

expected-output（講師示範 3 阻力的典型決策範例）：
> **阻力 A 阿翔**：選 (b) 手動匿名化——把客戶代碼欄整欄刪掉後再上傳。理由：原資料其他欄位（庫存量、進出貨流量）對決策有用，不能整份放棄。決策備忘記「**可用**：current-stock-anon.csv 已去除客戶代碼欄」。
>
> **阻力 B 采蓁**：選 (a) 引用 + ⚠ 標記。理由：報價數字對毛利估算關鍵，不能改區間；但風險明示讓總經理知道。決策備忘記「**可用（含 ⚠）**：a-brand-quote.csv ⚠ 報價版本未確認，需向 A 品牌覆核」。
>
> **阻力 C 小李**：選 (a) 用需求描述。理由：客戶名單不是決策必須，需求類型反而更直接。決策備忘記「**可用**：customer-inquiry-anonymized（已隱去客戶名）」+ 「**待主管授權**：原始客戶名單版本，若主管同意可後續補上」。

note：3 阻力**不是示範完才講**——它們是情境的一部分。每位學員上機那一刻就要面對自己角色的阻力，跑完工作流時決策備忘就同步產出。彙報強制分 3 段：**推薦**（要不要進貨/多少量/何時上架）、**風險**（3 部門各看到的風險）、**不確定性 ⚠**（3 部門對不上的點 + 3 阻力造成的限制，**不能強行收斂、不能假裝阻力不存在**）。

### 3. NotebookLM 多 source 並行示範（12 min · 含阻力 A 處理示範）

講師現場示範——同時開 3 個 notebook，跟著做，**示範阿翔遇到阻力 A 怎麼處理**。

#### 第 1 步：建 3 個獨立 notebook（3 min）

dialog-box（NotebookLM 操作）：
```
動作 1：開 notebook「採購層 · A 品牌新品評估」上傳 a-brand-quote.csv + alt-suppliers.csv + procurement-history-6m.csv
動作 2：開 notebook「業務層 · 運動水壺市場資料」上傳 similar-sales-50.csv + customer-inquiry-anonymized.txt（已隱去客戶名）+ competitor-prices.csv
動作 3：開 notebook「倉儲層 · 庫存容量評估」上傳 current-stock-anon.csv（已去除客戶代碼）+ flow-3m.csv + safety-stock-rules.md
等：3 個 notebook 都 Ready
```

expected-output：
> 3 個瀏覽器分頁，每個顯示 3 份 sources Ready。**注意**：倉儲與業務的 notebook 上傳的都是**處理過的版本**（阻力 A 與 C 的選擇結果），不是原始檔。

note：**為什麼分 3 個 notebook 不合 1 個**——若合 1 個 notebook，NotebookLM 在跨類別查詢時容易把採購條件跟業務銷售混引用，引用追溯會亂；3 個獨立 notebook 才能做到「**每個取證結果有清楚的來源歸屬**」。同時 3 個獨立 notebook 也比較好處理阻力——資安要求只擋倉儲層那份，採購業務不受影響。

#### 阻力 A 處理示範（2 min · 講師現場操作）

dialog-box（阿翔角色處理流程）：
```
步驟 1：開原始 current-stock.csv 看欄位 → 確認「客戶代碼」是哪幾欄（B 欄、F 欄）
步驟 2：複製整份檔案另存 current-stock-anon.csv → 刪掉 B 欄、F 欄
步驟 3：另開純文字檔記「current-stock.csv 原檔含客戶代碼欄（B、F），已於 current-stock-anon.csv 中刪除」→ 存進決策備忘的「可用」層
步驟 4：上傳 current-stock-anon.csv 到倉儲層 notebook
```

demo-takeaway：阻力處理的關鍵不是「能不能用」、是「**留紀錄**」——刪了什麼、為什麼刪、用什麼替代，全部寫進決策備忘。回家以後主管問起來，這份紀錄就是你的工作軌跡。

#### 第 2 步：每個 notebook 跑 2-3 條取證 query（共 6-9 條）（5 min）

dialog-box（采蓁的採購層 notebook · 跑 3 條取證 · 帶阻力 B 的 ⚠ 標記）：
```
取證 1：A 品牌運動水壺最低訂量？採購單價？付款條件？請附引用。⚠ 報價版本未明，請在回應結尾註明「需向 A 品牌覆核版本」。
取證 2：替代供應商 3 家中，哪家綜合條件最接近 A 品牌？差距在哪？請附引用。
取證 3：過去 6 個月進貨節奏，類似新品（運動類）平均首批訂量是多少？請附引用。
```

dialog-box（小李的業務層 notebook · 跑 3 條取證 · 用阻力 C 處理後資料）：
```
取證 1：過去 50 筆類似品（運動水壺類）銷售中，平均售價？平均月銷量？請附引用。
取證 2：12 筆客戶詢問（已隱去客戶名）中提到「運動水壺」幾筆？關注價格還是功能？請附引用。
取證 3：市場上同類競品的零售價區間是多少？請附引用。
```

dialog-box（阿翔的倉儲層 notebook · 跑 3 條取證 · 用阻力 A 處理後資料）：
```
取證 1：當前倉儲剩餘容量（棧板數）？（資料來源已去除客戶代碼欄，僅保留庫存量欄位）
取證 2：過去 3 個月進出貨流量看，新品進貨「最佳時機」是何時？（避開哪些已排程大批進出貨？）請附引用。
取證 3：以運動水壺類為基準，安全庫存規則建議首批進多少？請附引用。
```

expected-output（6-9 條取證材料各自有引用，每層列代表 2 條）：
> **採購層**：
> · 取證 1：A 品牌最低訂量 500 件、單價 NT$ 120、付款 30 天【引用：a-brand-quote.csv 第 1-5 列】⚠ **報價版本未明，需向 A 品牌覆核**
> · 取證 3：過去 6 個月運動類新品首批平均 400 件【引用：procurement-history-6m.csv 第 11-23 列】
>
> **業務層**：
> · 取證 1：類似品平均售價 NT$ 280、月銷量 80 件【引用：similar-sales-50.csv 摘要】
> · 取證 2：12 筆中 7 筆關注運動水壺、6 筆關注功能【引用：customer-inquiry-anonymized.txt — 已隱去客戶名】
>
> **倉儲層**：
> · 取證 1：當前剩餘 8 棧板（約可容 600 件）【引用：current-stock-anon.csv — 已去除客戶代碼欄】
> · 取證 3：安全庫存規則建議首批 300-500 件【引用：safety-stock-rules.md 規則 3】

note：實際輸出每次略有差異，重點是**每題回應都有明確引用 + 來源歸屬不混雜 + 阻力處理過的資料在引用區明示版本**。

#### 第 3 步：交叉比對示範（2 min）

把 3 個 notebook 的取證結果**並排放**（講師現場切 3 個視窗），逐項做交叉比對。

dialog-box（講師現場帶全班看）：
```
比對 1（採購量 vs 倉儲容量 vs 業務需求）：採購最低訂量 500 / 倉儲可容 600 ✓ / 業務月銷 80 → 500 件約 6 個月銷完
比對 2（採購單價 vs 業務售價 → 毛利）：採購 NT$ 120 進（⚠ 版本未明）/ 業務售價 NT$ 250-350 → 毛利 NT$ 130-230/件（毛利率 52%-66%，受報價版本影響）
比對 3（採購首批 vs 倉儲安全庫存）：採購歷史類似新品首批 400 / 倉儲規則建議 300-500 / 衝突點：A 品牌最低訂量 500 > 倉儲下限 300 → 必須進 500
```

demo-takeaway：交叉比對的目的不是讓 3 來源「對齊」，是讓 3 來源「**互相印證或暴露衝突**」+ **把阻力造成的限制明示**。對齊的部分抓信心；衝突的部分用 ⚠ 標；阻力造成的不確定（如報價版本未明）也用 ⚠ 標。

### 4. Gemini 跨源整合示範（10 min · 含阻力 B/C 標記示範）

把上一段的 6-9 條取證材料（每條都帶引用、含 ⚠ 標記）貼進 Gemini，產出 1 頁綜合彙報。

#### 整合 prompt wrapper（4 min · 講師現場打）

dialog-box（貼到 Gemini · 新對話）：
```
角色：你是業務副理小李的跨來源整合助手。
資料：以下是從 3 個獨立 NotebookLM 知識庫取得的「**有引用事實**」，每條已標來源部門 + 阻力備註：
[採購層取證 1：A 品牌最低訂量 500 件、採購單價 NT$ 120、付款 30 天【引用：a-brand-quote.csv】⚠ 報價版本未明]
[採購層取證 2：替代供應商 B 牌差價 NT$ 8、最低訂量 800 件【引用：alt-suppliers.csv】]
[採購層取證 3：過去 6 個月運動類新品首批平均 400 件【引用：procurement-history-6m.csv】]
[業務層取證 1：類似品平均售價 NT$ 280、月銷量 80 件【引用：similar-sales-50.csv】]
[業務層取證 2：12 筆客戶詢問中 7 筆提運動水壺，6 筆關注功能【引用：customer-inquiry-anonymized.txt — 已隱去客戶名，原始名單未進跨源 notebook】]
[業務層取證 3：市場零售價區間 NT$ 250-350【引用：competitor-prices.csv】]
[倉儲層取證 1：當前剩餘 8 棧板（可容 600 件）【引用：current-stock-anon.csv — 已去除客戶代碼欄】]
[倉儲層取證 2：最佳進貨時機 1/10-1/20【引用：flow-3m.csv】]
[倉儲層取證 3：安全庫存規則建議首批 300-500 件【引用：safety-stock-rules.md】]

任務：產出 1 頁綜合彙報給總經理，**嚴格分 3 段**：
  · 推薦：要不要進貨？多少量？何時上架？對每句明示「資料來自哪個部門」。
  · 風險：3 部門資料各自暴露哪些風險？對每條明示「資料來自哪個部門」。
  · 不確定性 ⚠：3 部門**互相對不上的點** + **3 阻力造成的資料限制**列這。**不要強行收斂**——用 ⚠ 標出，註明「需要 X 部門進一步確認」或「受資料限制」。

最後加 2 段：
  · **3 部門資料互相衝突的點清單**：逐條列：衝突點 X：A 部門說 ___（引用）vs B 部門說 ___（引用）/ 建議澄清方式：___
  · **資料限制清單**：本次彙報受到的 3 阻力是：報價版本未明 / 客戶名單未進跨源 / 庫存客戶代碼已去除。各自影響哪段推薦的信心度？

禁止：(1) 用「綜合判斷」「整體而言」遮蔽矛盾、(2) 把「不確定性」隱藏在「推薦」段裡混過去、(3) 任一段每條未標來源部門 + 引用檔名、(4) **假裝資料限制不存在，把阻力處理後的版本當原版引用**。
```

#### 不確定性與阻力標記示範（4 min · 講師現場看 Gemini 怎麼回）

expected-output（Gemini 整合產出）：
> # 1 月新品引進綜合彙報（A 品牌運動水壺）
>
> **推薦**
> · 建議進貨 **500 件**（受 A 品牌最低訂量限制，採購層）【引用：a-brand-quote.csv】⚠ 報價版本未明，建議覆核後鎖單
> · 建議上架時間 **1/15**（落在倉儲最佳進貨窗口 1/10-1/20，倉儲層）【引用：flow-3m.csv】
> · 預估毛利率 **52%-66%**（採購 NT$ 120 進 ⚠ 待覆核、業務市場售價 NT$ 250-350）【引用：a-brand-quote.csv + competitor-prices.csv】
>
> **風險**
> · 採購層：替代供應商 B 牌最低訂量 800 件 → 若 A 品牌斷貨，備援採購需擴容【引用：alt-suppliers.csv】

> · 倉儲層：500 件接近倉儲建議上限（300-500）→ 若銷量低於預期，去化期可能拉長【引用：safety-stock-rules.md】
>
> **不確定性 ⚠**
> · ⚠ **採購首批 500 件 vs 過去歷史 400 件**：A 品牌最低訂量強制 500，但過去類似新品歷史首批是 400。多進的 100 件去化風險未知。**需要採購（采蓁）+ 業務（小李）共同評估**：是否做促銷加速去化？
> · ⚠ **業務月銷量 80 件 vs 倉儲建議首批 300-500**：若按業務月銷量 80 推算，500 件需 6 個月銷完——與倉儲「安全庫存應 3-4 個月」的隱性假設可能衝突。**需要倉儲（阿翔）+ 業務（小李）對齊**：6 個月去化是否在安全範圍？
> · ⚠ **報價版本未明（阻力 B）**：採購 NT$ 120 取自無版本號報價單，毛利率推算受影響。**需要採購（采蓁）向 A 品牌覆核**最新報價，鎖單前不應對外承諾毛利率。
> · ⚠ **客戶名單版本未進跨源（阻力 C）**：業務層僅引用「需求類型」未引用客戶名單，無法判斷具名客戶是否有大量採購意願。**需要業務（小李）申請主管授權**後可補上具名客戶的議價底牌。
> · ⚠ **庫存客戶代碼已去除（阻力 A）**：倉儲層引用的庫存資料已去除客戶代碼欄，無法回溯「庫存中有多少是特定客戶預留」。**對總量推算無影響**，但若需做客戶層級調撥需另查原檔。
>
> **3 部門資料互相衝突的點清單**
> · 衝突點 1：採購最低訂量 500【a-brand-quote.csv】vs 採購歷史首批 400【procurement-history-6m.csv】→ 請采蓁確認 A 品牌是否可協商首批降到 400
> · 衝突點 2：業務月銷量 80【similar-sales-50.csv】vs 倉儲去化期【safety-stock-rules.md】→ 請小李+阿翔對齊「6 個月去化是否屬安全」
> · 衝突點 3：售價區間 NT$ 250-350【competitor-prices.csv】範圍過大 → 請小李拆解區間（高低端定價邏輯）
>
> **資料限制清單**
> · 阻力 A（庫存客戶代碼）：已去除欄位後上傳，**影響範圍 = 客戶層級調撥分析**（本案不需，可忽略）
> · 阻力 B（報價版本未明）：仍引用但加 ⚠ 標記，**影響範圍 = 毛利率推算信心度**（鎖單前必覆核）
> · 阻力 C（客戶名單未進）：用「需求類型」替代，**影響範圍 = 具名客戶議價底牌**（本案不關鍵，但 Q2 大客戶談判前需主管授權後補）
>
> note：實際輸出每次略有差異，重點是 ⚠ 標記不能少、衝突點不能合併、阻力影響範圍要明示。

#### 不確定性 + 阻力標記為什麼重要（2 min · 講師講）

demo-takeaway：跨來源整合 2 個最大陷阱——(1) Gemini 為了讓報告好看會**強行收斂**衝突點；(2) 學員為了讓報告完整會**假裝阻力不存在**，把處理過的資料當原版引用。⚠ 標記同時抓這 2 件事：對總經理顯示「**這裡 3 部門對不上**」+「**這段資料因 X 阻力做了 Y 處理**」。總經理看到後可以自己判斷該追問誰、要不要授權補充——而不是 AI 替他做了不該做的決定。

## 常見錯誤 5 條（Common Pitfalls · v2.1 加第 5 條）

### 1. 把 3 部門資料合進 1 個 notebook

**錯誤現象**：3 人嫌麻煩，把 9 份資料合上傳到 1 個 notebook，請 NotebookLM「綜合分析」。結果取證時引用混雜——問採購條件，NotebookLM 引用到業務銷售資料；問倉儲容量，引用到採購歷史。

**原因**：NotebookLM 在跨類別資料中**沒有部門概念**——它只認 source 檔名。9 份資料混在一起，跨類別查詢容易抓到不相干的 source。

**解法**：**嚴守 3 個獨立 notebook 分工**——每個 notebook 只放 1 個部門的資料。取證時清楚知道「這條引用來自哪個部門」。整合在 Gemini 那一步才做，不在 NotebookLM 做。

---

### 2. Gemini 強行收斂衝突點

**錯誤現象**：3 部門取證材料貼進 Gemini 後，產出的彙報寫「**綜合判斷建議進貨 400-500 件**」。看起來很合理，但實際上採購最低訂量是 500（不能進 400）、倉儲建議是 300-500（不限 400-500）、業務歷史是首批 400——3 個獨立來源被 AI 包成 1 句模糊範圍。

**原因**：Gemini 預設「報告要好看」——傾向把對不上的數字混成範圍、用「綜合判斷」「整體而言」遮蔽真實衝突。這對總經理是有害的——他看到「400-500」會以為 3 部門都同意，實際是被 AI 騙了。

**解法**：整合 prompt wrapper 必含「**禁用『綜合判斷』『整體而言』等遮蔽詞**」+ 「**對不上的地方一律標 ⚠，不強行收斂**」。產出後讀一遍，看到「綜合判斷」「整體而言」「大致上」這類措辭，**立刻追問 Gemini「把這句拆開，找出背後 3 部門真實衝突」**。

---

### 3. ⚠ 不確定性標記漏抓

**錯誤現象**：3 部門資料看起來「對得上」，學員就跳過 ⚠ 段、只寫「推薦 + 風險」兩段。彙報送給總經理，總經理追問「業務月銷量 80 怎麼來的、信心多高」答不出。

**原因**：學員沒意識到「對得上」跟「沒有不確定性」是兩回事。例如業務月銷量 80 來自 50 筆類似品**樣本估算**，本身就含不確定性；倉儲安全庫存規則來自**過去類別經驗**，新品類別未必適用。這些都是「看似明確、實則需標 ⚠」的點。

**解法**：⚠ 標記**不是只標衝突**，還要標「**樣本不足 / 假設未驗證 / 推論而非事實**」。每條取證問自己：這條是「事實」（檔案裡寫死的數字）還是「推論」（從樣本估出來的）？推論的全部標 ⚠。

---

### 4. 責任分工卡角色重疊

**錯誤現象**：3 人組合 1 張責任分工卡，「整合」跟「驗證」都寫小李——因為小李是彙報人，學員直覺覺得「他做整合也順便驗證」。結果回部門複現時沒人扮「收件人視角」，彙報送上去主管追問才發現漏洞。

**原因**：學員把「整合人 = 驗證人」當理所當然。但**整合人的視角是「3 部門資料怎麼接起來」、驗證人的視角是「收件人看到後會怎麼挑戰」**——這兩個視角必須由不同人扮，才能抓出整合盲點。

**解法**：責任分工卡明確規定**3 角色不重疊**——
- **建知識庫的人** = 領域擁有者（手上有資料的那位）
- **整合的人** = 彙報人（負責對總經理講話的那位）
- **驗證的人** = 收件人代理（用「總經理會怎麼問」的視角審）

若組合裡某個角色沒填、改用「全組共同驗證」——一律退回重填。

---

### 5. 遇到資料阻力就放棄整個跨來源案（v2.1 新）

**錯誤現象**：學員看到自己手上資料「客戶名單不能上傳」「報價單沒版本號」「庫存含敏感欄位」，直接舉手「我這份不能做、跳過跨來源整合」。結果整組 3 人卡住，因為 1 個部門不參與，跨源彙報就少 1 條腿。回部門以後遇到同樣阻力，學員也是直接放棄——「公司資安規定不能上傳，所以這個工作流我用不了」。

**原因**：學員把「**部分資料不能用**」當成「**整個工作流不能跑**」。其實 3 阻力都有應對方式：
- 阻力 A 敏感資料 → 匿名化 / 改用描述性紀錄 / 用 sample 模擬
- 阻力 B 版本不明 → 加 ⚠ 標記 / 改用區間
- 阻力 C 部門不願共享 → 用替代資料 / 申請主管授權 / 改用「需求描述」代替「具名清單」

**只要把不能用的部分明確標出、找替代、必要時申請授權，跨來源工作流仍能跑**——只是邊界更清楚。完全放棄反而錯失整合機會。

**解法**：**強制寫進決策備忘三層**——
- **可用層**：通過 3 阻力測試的 source，可直接進 NotebookLM
- **不可用層**：被擋下的 source，**明示原因**（資安 / 版本 / 共享限制）+ **替代方案**（用什麼補）
- **待主管授權層**：需要往上請示的 source，**明示申請事項**（要請主管同意什麼）+ **何時前**

三層填齊後，總經理（或主管）看到的不是「整個案做不出來」，而是「**這段能跑、這段被擋、這段要您裁決**」——這比放棄整個案有用 10 倍。**真實工作不可能 3 阻力剛好全可處理，「不可用 / 待授權」兩層空白通常代表學員沒認真找阻力。**

## 檢核題 2 條（Quiz）

**Q1（概念驗證）**：跨來源綜合彙報的「不確定性 ⚠」段，**正確做法**是哪一個？

- [ ] A. 把 3 部門對不上的數字算平均值、寫成範圍（例：400-500 件），讓報告好看
- [ ] B. 把 3 部門對不上的點全部隱藏在「綜合判斷」段，避免總經理困擾
- [ ] C. 把 3 部門對不上的點 + 3 阻力造成的限制明示 ⚠、附「需要哪些角色進一步對齊或授權」（**正確答案**）
- [ ] D. 把 3 部門對不上的點都列在風險段、不另開 ⚠ 段

**正確答案**：C。⚠ 段的作用是**讓總經理看到 3 部門真實衝突 + 資料限制邊界、自己判斷該怎麼裁決**——而不是讓 AI 替總經理做決定。把衝突算平均（A）或隱藏（B）是強行收斂；放風險段（D）混淆「風險」（已知不確定的負面情境）跟「不確定性」（資料對不上或被阻力限制）兩種概念。

**Q2（應用驗證）**：

**情境**：你是凱倫（財務專員），組合 B 替代版。3 人組合：你 + 小李業務 + 怡君電商。Q1 業績檢討跨來源整合，**你手上的 finance-budget-q1-original.xlsx 含個別薪資資料**（阻力 A），公司資安規定不能上傳免費版 NotebookLM。Gemini 整合產出寫：

你發現 3 部門資料中：
- 業務 sales-q1.csv 達成 110%（含退貨未沖銷）
- 財務 finance-actual-q1.csv 達成 95%（已沖銷退貨）
- 電商 own-site-q1.csv 達成 120%（含未請款預訂）

且財務的「人力成本」是用**已剔除個別薪資的彙總版**算出來的——但彙報沒提這件事。

請判斷：(1) 這段 Gemini 產出犯了哪 **3 條 Pitfall**？(2) 你該如何要 Gemini 重寫？(3) 決策備忘應該怎麼填？

**預期答案要點**：

- **(1) 犯的 3 條 Pitfall**：
  - **Pitfall 2 強行收斂**——「整體而言 105%」把 3 口徑（110/95/120）強行平均混成 1 個數字，遮蔽口徑差異
  - **Pitfall 3 ⚠ 標記漏抓**——3 部門口徑差異本身就是不確定性，應該標 ⚠ 而不是放推薦段
  - **Pitfall 5（v2.1 新）阻力應對不揭露**——「預算執行率 92%（含人力成本）」沒寫「人力成本來自彙總版（個別薪資已剔除）」，**等於假裝阻力 A 不存在**。經營層看到後若想追溯個別薪資項做不到。

- **(2) 重寫方式**：
  - 追問 Gemini：「**把『整體而言 105%』拆開，分 3 口徑列出，並用 ⚠ 標出口徑差異 + 明示『財務人力成本來自彙總版』**」
  - 期望產出：
    - 推薦段改寫為「Q1 業績 3 口徑差異明顯，**經營層需先裁決用哪個口徑**再判斷 Q2 策略」
    - ⚠ 不確定性段新增：「⚠ Q1 業績 3 口徑：業務 110%（含退貨）/ 財務 95%（沖銷後）/ 電商 120%（含預訂）。**差距 25 個百分點，需經營層裁決標準口徑**。」
    - **資料限制清單新增（v2.1）**：「阻力 A 財務：預算執行 92% 取自個別薪資已剔除的彙總版（finance-budget-q1-anon.xlsx），**個別薪資項追溯需另查原檔（需主管授權）**」

- **(3) 決策備忘填寫**：
  - **可用**：finance-budget-q1-anon.xlsx（已剔除個別薪資）/ finance-actual-q1.csv / finance-recognition-rules.md
  - **不可用**：finance-budget-q1-original.xlsx（含個別薪資、資安規定禁上傳）→ 替代方案：用 anon 版彙總
  - **待主管授權**：若 Q2 經營層需追溯個別薪資項，需申請財務主管授權後使用 original 版

- **加分**：
  - 能說出「**跨來源整合不是讓 3 部門變一致，是呈現 3 部門真實樣貌 + 資料邊界讓決策者裁決**」
  - 能說出「**這次互審反饋寫在彙報背面 + 決策備忘背面，下次回部門複現時主動找『綜合判斷』『整體而言』追問 Gemini + 主動填三層備忘**」
