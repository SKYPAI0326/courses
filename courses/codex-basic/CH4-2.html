部署與 Computer Use

從需求釐清、製作小站、Sites 部署上線，到 Computer Use 自動填表驗證——你全程沒寫一行程式，但每一個「放行」動作都由你決定。

學習成果
能用規劃模式做一頁「部門報帳須知」小站，並透過 Codex 內建 Sites plugin 部署上線
能說出部署前「把關紅線 #1」：設 access mode 受限、自己審內容，確認無誤再放寬
能說出 Computer Use 的邊界與「把關紅線 #2」：任務範圍要可控，送出後親自確認信箱收到通知
(01)

破題：陳主管要一頁讓大家自己查的報帳須知

情境
你已經在 CH4-1 做出一個 HTML 儀表板，追蹤弄一下工作室 6 月月結的進度。現在陳主管說：「能不能做一頁讓大家都看得到的『部門報帳須知』？讓各部門同事自己查規定、提交報帳資料，我不用一直接電話。」

這件事以前要找人架網站、寫程式、部署主機。今天 Codex 全程包辦：規劃 → 製作 → 部署上線 → 自動測試，你一行程式都不用寫。

這個單元，你會看到兩個具體成果：

本單元你會看到的成果
  • 一頁「部門報帳須知」小站，透過 Codex 內建 Sites plugin 部署上線，初始設為受限存取（owner/admin），審完內容才放寬
  • 一筆 Computer Use 自動填入的測試表單，並到 Gmail 確認通知信收到

全程你一行程式都不用寫。看完這段，你也知道這兩個步驟各自哪個動作一定要人先確認才能繼續。

(02)

概念：四個詞，先認識再看示範

概念 01
規劃模式（plan mode）
啟用後，Codex 不會直接開始做，而是先提問、釐清需求，列出完整執行計畫讓你確認。用途：任何比單頁文件複雜的任務，先花一分鐘問清楚，省掉重做的時間和額度。
概念 02
/pet 寵物通知
在輸入框輸入 /pet（Settings > Appearance 可設定樣式），螢幕角落出現一個小寵物。Codex 有進度就跳通知，讓你在它做長任務時去做別的事，不用守在旁邊。
概念 03
Sites plugin 部署（Codex 內建）
Codex 內建 Sites plugin，由 OpenAI 託管，可直接把做好的網站部署上線。部署後產生 production URL；Sites 支援 access modes（owner/admin、workspace、custom）讓你控制誰看得到。部署前先 save version，確認內容無誤後再 deploy。
概念 04
Computer Use
啟用後，Codex 能像真人一樣操作你的瀏覽器和電腦軟體——移動游標、點按鈕、在欄位裡填字、送出表單。把機械式重複的測試任務交給它，你不用親自動滑鼠。
!
Sites plugin 與 Computer Use 為課堂示範功能，不保證所有帳號方案當下可用。如果你的帳號暫時沒有這些功能，不影響學習這套流程的邏輯。
(03)

操作示範：規劃 → 製作 → 部署 → 測試

1
啟用規劃模式，先擬計畫
在 Codex 主頁把「規劃模式」切換到啟用。把弄一下工作室的 Logo 和辦公室相片（已在工作資料夾）拖入，送出提示詞：
送出的提示詞
幫我做一頁「部門報帳須知」小站。頁面需要包含：報帳流程說明、各部門填寫規定，以及一個讓同事提交報帳表單的留言區。
啟用規劃模式後，Codex 不會馬上動工，而是先問幾個問題（例如：報帳流程要幾個步驟？表單要收哪些欄位？網頁風格要正式或輕鬆？）。一一回覆後，Codex 列出完整執行計畫；確認後 Codex 正式開始。
為什麼先擬計畫：製作網站要花時間和額度。需求不清楚就動工，做到一半才發現方向錯，重來會更貴。多花一分鐘釐清需求，省掉重做的十分鐘。
2
製作小站 + 啟用 /pet 寵物通知
Codex 開始製作。因為做網站通常要花較長時間，在輸入框輸入 /pet。螢幕角落跳出小寵物，Codex 有進度就會主動通知。這段時間你可以去做別的事，不用守在這裡。製作完成後小寵物跳通知，回到 Codex 視窗把通知關閉。
3
預覽網站 + 用「註解」功能微調設計
Codex 提供一個本機預覽網址。打開後你會看到：「部門報帳須知」標題 + 弄一下工作室 Logo、報帳流程說明（三步驟：填表 → 上傳憑證 → 送交陳主管）、各部門填寫規定，以及一個表單（欄位含「部門」「姓名」「金額」「說明」「送出」按鈕）。

如果發現標題字體大了一點，用右上角「註解」功能：啟用註解模式 → 把滑鼠移到標題點一下 → 在對話框輸入「標題字體縮小一號，顏色改深咖啡色」→ 送出。Codex 立即修改，頁面刷新，標題改好了。
這個改法的意義：你不需要知道 CSS 怎麼寫，用自然語言點哪改哪。頁面改到滿意，再進下一步部署。
4
用 Sites plugin 部署 + 設定 access mode
先執行 save version（Sites plugin 內），確保有可回溯的快照。確認頁面內容正確、沒有財務敏感資料（報帳帳號、人員聯絡方式、費用上限金額）不該讓目標對象看到，這是部署前必做的把關。送出提示詞：
送出的提示詞
請透過 Sites plugin 把這個網站部署上線，先設定 access mode 為 owner/admin。
Codex 使用內建 Sites plugin 直接部署，幾分鐘後顯示 production URL，例如：https://dongyixia-reimbursement.openai.com

隨即在 Sites 設定中確認 access mode 已設為「owner/admin」（只有自己和管理員看得到）。
Sites access modes 放寬順序：owner/admin（只有自己）→ workspace（同工作區成員）→ custom（自訂名單）→ 公開。陳主管確認內容無誤後，再決定放寬到哪一層。分享網址前先確認 access mode 符合目標受眾。
5
用 Computer Use 自動填表驗證
送出提示詞：
送出的提示詞
請前往剛才部署的 production URL，隨意填入一筆測試資料送出表單（部門填「測試部門」，金額填 0），確認表單送出成功。
Codex 啟用 Computer Use。螢幕自動彈出瀏覽器視窗——你看到 Codex 操控游標、自己移動到表單欄位、逐一填入內容、點「送出」按鈕，全程不需要你動手。表單送出後，打開 Gmail 確認是否收到通知信，裡面有剛才填進去的測試資料。
把關要點：Computer Use 能操作的不只是測試網站，它可以操作你電腦裡的任何瀏覽器視窗。使用前，確認任務範圍是你可以控制的測試環境，而不是正式系統或客戶資料頁面。
(04)

兩條部署把關紅線

看完這段示範，記住這兩個動作——任何時候用 Codex 做部署或 Computer Use 測試，都適用：

!
把關紅線 #1（部署）：透過 Sites plugin 部署後，production URL 就是可存取的正式網址。分享連結前，你必須自己先確認：頁面上的文字是否正確、有沒有財務敏感資料（報帳帳號、人員聯絡方式、費用上限金額）不該讓目標對象看到。Codex 不會主動過濾這些，那是你的責任。建議流程：先設 access mode 為 owner/admin → 自己審完內容 → 確認無誤再決定是否放寬。

把關紅線 #2（Computer Use）：Computer Use 會在你的電腦上真的操作瀏覽器。使用前確認：測試的對象是你可以亂填的測試環境，而不是正式系統或客戶資料頁面。送出後，你要親自打開 Gmail 確認通知信有進來，才算測試通過。
(05)

看完示範，心裡想清楚三件事

給自己的，不用說出來：

自我觀察
  • 從「需求描述」到「正式網址」，Codex 做了哪幾個步驟？哪一步是你確認了才讓它繼續的？
  • 如果這頁小站上線後，同事發現有一個欄位說明寫錯——Codex 沒有辦法事先知道你的公司規定是否正確，誰的責任是什麼？
  • Computer Use 測試完，你怎麼確認它真的把表單送出去了，而不是只跑完動作、信沒收到？

然後想一件事：你現在有哪一頁資訊想公開給公司內部，讓大家自己查、不用一直來問你？

(07)

常見錯誤——三條最容易踩的坑

錯誤 1
跳過規劃模式直接做
現象：直接叫 Codex 做網站，做完才發現表單欄位少了「部門」，或報帳規定說明和實際的不一樣，要重做。
原因：需求沒有事先釐清，Codex 只能根據你說的做，沒說到的就自己猜或省略。
解法：任何比「單頁文件」更複雜的任務，先啟用規劃模式。花一分鐘讓 Codex 問清楚需求，省掉重做的時間和額度。
錯誤 2
部署完就分享連結
現象：部署完就把 production URL 丟給同事——結果頁面上的費用上限金額寫的是舊規定，或出現了某個部門的內部分機號碼不應該讓那個對象看到。
原因：Sites 部署後 production URL 立即可存取，access mode 決定誰看得到。Codex 不知道哪些內容對哪些對象有問題。
解法：部署後先設 access mode 為受限（owner/admin）、自己逐段讀過頁面。財務相關數字、人員聯絡資料、內部系統連結確認無誤，再決定放寬 access mode、分享連結。
錯誤 3
沒確認收件匣就算測試通過
現象：用 Computer Use 測試時，看到它在螢幕上操作就放心了，沒有去收件匣確認是否真的收到通知。結果同事後來填表，通知沒來，才發現表單的後端設定根本沒完成。
原因：Computer Use 只負責執行「填表單、送出」的動作，它不會去你的信箱幫你核實收信結果（除非你下指令叫它去）。
解法：Computer Use 送出表單後，你親自打開信箱確認通知有進來，才算測試通過。自動化的最後一步，還是要人眼確認一次。
(08)

確認你學到了什麼

學會的標準
  • 你能說出規劃模式的用途：先釐清需求再動工，不是讓 AI 想更仔細。
  • 你能說出部署後分享前要確認什麼：access mode 是否符合目標受眾、頁面有無財務敏感資料外洩。
  • 你能說出 Computer Use 的一條把關動作：測試後親自到 Gmail 確認通知信有進來。
Q1：啟用「規劃模式」之後，Codex 的第一個動作是什麼？
查看解析
正確答案是 B。規劃模式的用途是「先釐清需求再動工」——Codex 啟用後不會馬上寫程式，而是先問你幾個問題（表單要幾個欄位？頁面風格要正式還是輕鬆？），列出完整執行計畫確認後才開始。多花一分鐘釐清需求，省掉重做的時間和額度。
Q2：你用 Codex 做好「部門報帳須知」小站並透過 Sites plugin 部署，陳主管說「先不要讓全公司看到，我想自己先看一次」。在陳主管確認之前，你應該怎麼做？
查看解析
正確答案是 C。要點有三：
  • 部署後立即把 access mode 設為受限（owner/admin），只有自己能看到 production URL，先把連結分享給陳主管審核——不要在陳主管確認前放寬給更多人。
  • 特別請陳主管確認：有沒有財務敏感資料外洩（費用上限、帳號資訊、人員聯絡方式），Codex 不會主動過濾這些。
  • 如果陳主管有修改意見，用「註解」功能改完、再 save version，確認後再決定是否把 access mode 放寬給 workspace 或更廣。

全課收束

!
從 CH1-1 到 CH4-2，你走了一條完整的工作流：

看到 Codex 能幫你做什麼（CH1-1）→ 讓它認識你、認識你的工作資料夾（CH1-2、CH1-3）→ 整理雜亂檔案、彙整清理表格（CH2-1、CH2-2）→ 把重複流程封裝成 Skill、用外掛擴充能力（CH3-1、CH3-2）→ 排程讓它自動跑、做儀表板、做小站、部署上線、用 Computer Use 自動測試（CH4-1、CH4-2）。

這些你全程沒有寫過一行程式碼，沒有打開過終端機。

最重要的事是你也記得了：Codex 幫你做事，但責任還是你的——缺資料要停、寄出要你按、部署前要你確認、Computer Use 的範圍要你設定清楚。會用、也守得住底線，才是真的學會。
