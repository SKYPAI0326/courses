MODULE 6 · SECTION 6.3

event_id 去重 + Event Match Quality

Browser + CAPI 雙軌的代價是可能雙算——event_id 讓 Meta 去重；EMQ 6+ 讓廣告效能回歸。

為什麼這很重要 · WHY THIS MATTERS
CAPI 裝好不等於完事——EMQ 分數低代表 Meta 認不出使用者，廣告最佳化沒效。這節把『做對』推到『做好』。
STEP BY STEP · 本節重點
1
1 event_id 生命週期
產生：前端 purchase 時 crypto.randomUUID() → 寫 sessionStorage + push dataLayer。傳到 Browser Pixel：fbq('track', 'Purchase', {}, {eventID: 'xxx'})。傳到 server：寫進訂單 meta（order_item_meta）。CAPI 呼叫時：從 order meta 讀出 → event_id 欄位帶入。四步都對才能去重。
用 order_id 當 event_id 是替代方案但不如 UUID 安全（可被猜測）。
2
2 去重驗證：Events Manager
Events Manager → 選 Pixel → Overview → 找 Purchase 事件 → 看『Deduplication』欄：應該顯示『Server + Browser 事件去重成功，X% 重複』。X% 在 10% 以下健康；超過 30% 代表 event_id 沒對上。
另一個常見錯誤：event_id 有帶但格式不一樣（前端 UUID v4、後端 UUID v5）。統一用一種。
3
3 Event Match Quality（EMQ）分數
0–10 分，衡量 Meta 把事件匹配到 Facebook 使用者的能力。計算方式：傳了多少 user data（em/ph/fn/ln/ge/db/ct/st/zp/country）× hash 正確 × fbc/fbp cookie 有無。目標 ≥ 6.0——低於 4.0 廣告效能會明顯受損。
拆開看每個事件的 EMQ——通常 Purchase 分數最低（需要 match 實際購買者），Lead 次之。ViewContent 反而容易高（匿名瀏覽也算）。
4
4 提升 EMQ 六招
① 補 email（最有效）；② 補 phone；③ 補 first/last name；④ 前端正確放 _fbp cookie；⑤ 從 URL 抓 _fbc 參數（Meta 廣告導流帶）並送出；⑥ 補地區資訊（ct/st/zp/country）。每補一項 EMQ 通常 +0.5 到 +1.0。
Hash 要用 SHA-256，不是 SHA-1 或 MD5。email 要先 trim + 小寫再 hash。phone 要先移除非數字再 hash（E.164 格式）。
5
5 完整 user data payload 範例
user_data: {
  em: sha256(lowercase(trim(email))),
  ph: sha256(only_digits(phone)),
  fn: sha256(lowercase(first_name)),
  ln: sha256(lowercase(last_name)),
  fbc: cookie._fbc,  // 不要 hash
  fbp: cookie._fbp,  // 不要 hash
  client_ip_address: $_SERVER.REMOTE_ADDR,
  client_user_agent: $_SERVER.HTTP_USER_AGENT
}
fbc/fbp 是 Meta 的 cookie 不能 hash（hash 後就對不到 Meta 端紀錄）。email/phone/name 一定要 hash。
6
實戰案例｜EMQ 從 4 拉到 8 的具體手法
某健康食品電商 EMQ 一直卡在 4.2，CAPI 部署完毫無效果。逐一補齊：(1) 加 email hash（+0.8）；(2) 加 phone hash（+0.7）；(3) 帶 fbp/fbc cookie（+1.2）；(4) 帶 client_ip_address + client_user_agent（+0.6）；(5) 加 external_id（你的會員 ID hash）（+0.5）。EMQ 4.2 → 8.0，ROAS 同步 +45%。
email/phone 要 SHA-256 hash、小寫、去空白。沒 hash 直接傳明碼 Meta 會拒收。PHP：hash('sha256', strtolower(trim($email)))。EMQ ≥ 7 是及格線、≥ 8 是優等生。
完成驗證 · VERIFY
Events Manager 顯示 Purchase 事件 EMQ ≥ 6.0、去重率 ≤ 10%，並截圖存證。
常見問題 · TROUBLESHOOT
email 做 Advanced Matching 合法嗎？
Meta TOS 允許，但你的站必須有明確的 privacy policy 告知會做廣告比對。在歐盟需 Consent Mode 管理。
測試流量也有 EMQ 嗎？
有，但通常很低（沒真實 email）。EMQ 要看正式流量才有意義。測試用的就看『事件有進來』即可。
EMQ 一直 3.x 怎麼辦？
先確認 fbp cookie 有送（用 DevTools 看 _fbp cookie 存在）。再看 email/phone 有無 hash（明文會被判無效）。最後查 Hash 用的是 SHA-256。
