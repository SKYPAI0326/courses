MODULE 5 · SECTION 5.3

purchase 去重 × refund × 內部流量排除

資料乾淨的三件事——transaction_id 不重複、退款要送 refund、內部 IP 要排除。這節把電商追蹤的『收尾工作』做完。

為什麼這很重要 · WHY THIS MATTERS
有事件不等於有乾淨資料。這節決定你的 ROAS 報表可不可信。
STEP BY STEP · 本節重點
1
1 transaction_id 去重策略
規則：同一筆訂單只送一次 purchase。保險做法：①用 sessionStorage 記『此 order_id 已送』→ 到 thank-you 頁先檢查；②Server-Side 發送時以 DB 的 order status 判斷（只有 status=completed 時送）。
訂單成立後使用者常常 F5／返回／分享 URL——都可能二次觸發。前端 + 後端 雙重保險。
2
2 refund 事件：讓報表誠實
退款時送 dataLayer.push({event:'refund', ecommerce:{transaction_id, value, items}})，GA4 會從原本 purchase 扣回。沒送 refund＝ROAS 虛增。WooCommerce 有 woocommerce_order_refunded hook 可以自動 push。
部分退款要 push 對應比例的 value 與 items 數量（不是整筆撤銷）。
3
3 內部流量排除：IP 與 debug 參數
GA4 → Admin → Data streams → 資料串流 → Tagging settings → Define internal traffic，加入公司/居家 IP 段。還要去 Data filters 建『Exclude internal traffic』規則並發布。
LocalWP 測試流量一定要排除，否則 ROAS 會被自己污染。通常發現 ROAS 異常高而轉換突然很多→先檢查是不是自己。
4
4 自動化測試資料標記
在 LocalWP 或 staging 的 GTM 容器環境變數設 traffic_type = 'internal'，所有事件都帶這個參數。生產容器不帶。GA4 建立 segment 排除這個。也可用在 Looker Studio filter。
整套測試資料走自己的 GA4 property（測試專用）更乾淨。建議有預算的團隊這麼做。
5
5 上線前 QA checklist
① transaction_id 是否唯一、② value 等於 items 加總 + shipping + tax、③ currency 一致 TWD、④ 退款事件實測一次、⑤ 內部 IP 已排除、⑥ 多裝置（桌機／手機）都能 Fire、⑦ 付款 redirect 回站流程 purchase 能 Fire、⑧ GA4 DebugView + Meta Test Events + Ads Debug 三方都看到。
這張 checklist 每個新站上線都要跑一次。照過 8 項你的電商追蹤穩了。
6
實戰案例｜transaction_id 生成邏輯錯了全盤皆輸
某品牌網站開發者覺得「用 timestamp 當 order id 比較簡單」——結果秒殺活動同秒下了 8 單都用同一個 timestamp，GA4 以為是同一筆訂單，去重後只留 1 筆，丟失 7 筆營收資料。改成 uuid-v4 後恢復。事後靠金流後台補回資料才對帳。
transaction_id 規則：(1) 全站唯一且永久唯一（不是會話唯一）；(2) 跟你的訂單系統 ID 一致，方便對帳；(3) 不要用時間戳、不要用 session_id、不要用 user_id+date。推薦直接用訂單 DB 的 primary key。
完成驗證 · VERIFY
交付一份『我的 WooCommerce 站上線前 QA 報告』，8 項 checklist 逐項打勾/打叉並註明。
常見問題 · TROUBLESHOOT
refund 要不要包含 shipping？
看商業邏輯。大多店家退款不退運費——此時 refund value 不含 shipping。Meta/Ads 歸因要和 GA4 保持一致。
內部 IP 是動態的（員工在外面遠端）怎辦？
用 GA4 traffic_type 參數或 IP + 特殊 cookie（員工瀏覽器預先種植）混合判斷。完美解決不存在，做到『大多時候排除』即可。
transaction_id 洩漏會被攻擊嗎？
通常不會——transaction_id 只是個 ID 不含金額。但若你的 ID 是連續整數（1,2,3…）容易被猜——建議 UUID 或 hash order_id。
