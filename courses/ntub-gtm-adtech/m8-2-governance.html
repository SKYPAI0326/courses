MODULE 8 · SECTION 8.2

GTM 容器治理

版本命名、資料夾結構、工作區協作、Approval 流程——讓多人團隊不在同一個容器打架。

為什麼這很重要 · WHY THIS MATTERS
你會接下來維護這個容器 3 年——這節決定那 3 年是順風還是災難。
STEP BY STEP · 本節重點
1
1 版本命名規範
每次 Publish 要寫清楚：v14 - [2026-04 春季檔] 加 LINE LAP base + TikTok Pixel PlaceAnOrder。包含：版本號、時間、範圍、主要改動。不要只寫『update』。
GTM 版本訊息沒 2000 字限制——寫詳細沒壞處。半年後自己回來看會感謝。
2
2 資料夾結構
Tags 頁用 Folders 分類。建議：01 Base Tags（GA4 Config / Pixel base / LAP base）、02 Ecommerce Events、03 Lead Events、04 Audience Triggers、05 Custom JS、99 Deprecated。
Deprecated 資料夾是『即將刪除的 Tag』暫放——避免直接刪除破壞歷史資料。兩週後確認沒問題再真的刪。
3
3 工作區協作：多人不打架
一人一個 Workspace，改自己的，定期 merge 到 main。Workspace 命名 {人名} - {feature}（例：wilson - add TikTok Pixel）。merge 前先 sync latest workspace。
免費版 3 個 workspace 上限，大團隊要升級 360。或輪流排值日『今天 Wilson 改，明天 Jenny 改』。
4
4 Approval 流程（360 版）
GTM 360 有內建 Tag Approval：改動提交 Merge Request，指定 reviewer，reviewer 通過才能 Publish。免費版無此功能——用『Publish 前必須在 Slack 通知』的流程代替。
Approval 最怕流於形式——設最低檢查清單（改動說明、Preview 截圖、Test Events 結果）reviewer 才能通過。
5
5 上線前 checklist（10 項）
① Preview Mode 跑過 ② 新 Tag 全部命名合規 ③ 對應 Trigger + Variable 沒誤用 ④ 新欄位全部有 Default ⑤ 沒改動未列在 change log 的 Tag ⑥ Consent Settings 設好 ⑦ GA4 DebugView 收到 ⑧ Meta/Ads Test Events 收到 ⑨ Publish message 寫清楚 ⑩ Roll back 計畫知道（版本號）。
印出來貼在桌邊。每次 Publish 前照過一次。
6
實戰案例｜GDPR 罰單的真實案例
2023 某法國電商被 CNIL 罰 €300,000 原因：使用者拒絕 cookie 後，Meta Pixel 仍透過 server-side 送資料。他們以為『server-side 不算 cookie 所以不受 GDPR 規範』——錯。只要能識別個人身份（email/phone/IP），不論 client 或 server 發送都算 personal data processing。正確做法：consent denied 時 CAPI 也要停——或只送不可識別的聚合資料。
不是你有 Cookie Banner 就合規。合規檢查：(1) 拒絕有效（拒絕後 tag 真的不 fire）；(2) 有 consent log（能證明誰在何時同意了什麼）；(3) 使用者能撤回（有入口重新開 banner）；(4) 資料處理條款（DPA）跟 Meta/Google 簽了。
完成驗證 · VERIFY
交付 1 份『你的客戶站治理手冊』，含版本命名、資料夾結構、工作區 SOP、上線 checklist。
常見問題 · TROUBLESHOOT
兩個人同時改同一個 Tag 會怎樣？
GTM 會顯示衝突警告，晚 merge 的人要手動解決。比 Git 簡單（只是欄位層級），但還是會浪費時間。
工作區 3 個不夠用？
升級 360（年費約 $150k USD 起）或輪流排值日。第三招：把關係最淺的改動（例：只改命名）直接在 main 做完 Publish。
改動要不要留備份？
GTM 版本已自動備份（可 rollback）。另外建議匯出 JSON（Admin → Export Container）定期存 GitHub，防止 GTM 帳戶被刪除。
