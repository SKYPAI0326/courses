MODULE 4 · SECTION 4.3

SPA 與 AJAX 追蹤

當網站是 Vue/React SPA 或購物車走 AJAX——傳統 Page View 失準。用 virtual_page_view + history change 解決。

為什麼這很重要 · WHY THIS MATTERS
SPA 網站的老闆常問：『為什麼 GA4 只看到一次造訪？』答案：SPA 真的只換 URL 不換 window。這節把假換頁變成真事件。
STEP BY STEP · 本節重點
1
1 SPA 的問題：換頁不重載
Vue/React/Next.js 切換路由用 history.pushState，URL 變了但 window 沒重建——Page View Trigger 只 Fire 第一次。結果：GA4 看到 1 次 page_view，但使用者其實看了 5 頁。
Next.js 與部分框架有 built-in GA4 plugin 幫你處理——先查是否已有內建方案，有的話別重造輪子。
2
2 方案 A：History Change Trigger + virtual_page_view
GTM → Trigger → History Change → 儲存 Trigger - History Change。建 GA4 Event Tag：event_name = page_view，page_location = {{Page URL}}，page_title = {{Page Title}}。觸發條件綁剛剛的 Trigger。
關鍵：Tag 要同時綁 All Pages（首次載入）+ History Change（後續換頁）兩個 Trigger，首次與後續都要有。
3
3 方案 B：前端主動 push virtual_page_view（推薦）
Vue Router / React Router 的路由變更 hook 裡直接 dataLayer.push：
router.afterEach((to) => {
  dataLayer.push({
    event: 'virtual_page_view',
    page_path: to.path,
    page_title: document.title
  });
});
比方案 A 更可控——可在 push 前加工資料（清參數、補 content group）。
推薦這個方案給有工程資源的團隊。History Change 是『退而求其次』。
4
4 AJAX add_to_cart：不重載頁面的加車
WooCommerce 有 AJAX 加車模式（點加車不跳頁）。用 jQuery(document).on('added_to_cart', function(e, fragments, cart_hash, $button){ dataLayer.push({event:'add_to_cart',...}) }) 監聽內建事件。GTM4WP 已內建這塊。
非 WooCommerce 的站要自己找 AJAX 成功 callback。fetch/axios interceptor 是常見落點。
5
5 驗證 SPA 追蹤：DebugView + 路徑對照
GA4 DebugView 在 SPA 換頁應該看到：首次 page_view（page_location=A）→ 使用者點連結 → 新的 page_view（page_location=B）。Path 是否跟 URL 一致、Title 有沒有更新（SPA 常常 title 沒改）。
SPA 最常見 bug：page_title 永遠是首頁 title——因為 document.title 沒在路由變更時更新。工程要修的是 SPA 本身。
6
實戰案例｜AJAX 加購漏追的連鎖反應
某服飾電商『加入購物車』是 AJAX，沒跳頁。原本沒 push dataLayer，結果：(1) GA4 add_to_cart 事件 0 筆，漏斗斷在 view_item；(2) Meta Pixel AddToCart 沒觸發，廣告優化目標『加購』完全沒資料；(3) 受眾『加購未購買』名單建不出來，再行銷 ROAS 無法衡量。修法是在 AJAX success callback push dataLayer——一行 JS 修復三個下游問題。
AJAX 事件最好在 server 回 200 之後才 push，不要在送出請求時就 push——後端有可能失敗（庫存不足、限購），push 出的事件會是假數據。success callback 是正確時機。
完成驗證 · VERIFY
LocalWP 站（若為 WooCommerce AJAX）或任一 SPA demo 成功送出 AJAX add_to_cart / virtual_page_view 並在 DebugView 看到。
常見問題 · TROUBLESHOOT
Next.js 已經有 @next/third-parties/google 套件，還要自己做 History Change 嗎？
不用。套件已包含路由變更時送 page_view。你只負責『非 page_view 的自訂事件』。
History Change 事件有重複 Fire 的問題嗎？
有機會。某些 SPA 框架會在一次導航內多次 pushState。解法：用前端 throttle / 或用 router hook 方案 B 比較穩。
AJAX add_to_cart 被追了兩次？
常見於既有 WooCommerce 內建 dataLayer 又自己手刻 hook。擇一用即可。GTM4WP 若已開 enhanced ecommerce，不要再寫自己的 jQuery hook。
