PART 1 · 觀念與行銷規劃
1.3

行銷漏斗 → KPI → 事件：一張表把「為什麼埋」講清楚

90% 的埋碼失敗不是因為技術不夠，而是因為沒規劃就動手。這一節我們用業主最常問的問題當起點：「我們的流量到底有沒有在賺錢？」然後倒推回來，什麼該埋、什麼該回報、什麼該進報表。

學完這節，你會
把行銷漏斗四階段轉成可追蹤的 GA4 事件
為每個事件標記「要不要送到廣告像素」與「有沒有轉換價值」
帶走一張可以直接給工程師與老闆看的「埋碼規劃表」
(01)

埋碼的第一個決策不在 GTM

當你打開 GTM，準備新增一個 Tag 的那一刻，事情已經決定完了。真正重要的決策發生在更早 —— 在一張白紙上，問自己三個問題：

◆
這門生意的錢，是怎麼賺到的？（商業模式）
客人從不認識到付錢，中間會做哪些動作？（行銷漏斗）
哪些動作是「領先指標」，值得被即時衡量？（KPI）

如果這三題沒答清楚就埋碼，你會得到一個滿滿 80 個事件的 GA4 帳戶，但老闆一問「所以這個月廣告有沒有賺錢？」，你找不到答案。

技術決定「能不能埋」，
規劃決定「該不該埋」。
(02)

把生意畫成四層漏斗

不管你做的是電商、B2B、實體店、線上課程，顧客行為都能粗分成四階段。這個框架不是新發明，但它是埋碼規劃的最小公約數。

TOFU
認知
「有人來看」。關鍵問題：流量哪裡來？品質如何？
對應事件：page_view、scroll（≥50%）、view_item_list
MOFU
興趣
「開始認真看」。關鍵問題：哪些內容真的吸引人？
對應事件：view_item、video_progress（50%）、file_download、click_external
BOFU
意向
「準備要買/要聯絡」。關鍵問題：哪一步開始流失？
對應事件：add_to_cart、begin_checkout、form_start、click_phone
CONV
轉換
「錢進來 / 名單進來」。關鍵問題：成本多少？值得再投嗎？
對應事件：purchase、generate_lead、sign_up、contact
ℹ
GA4 的 25 個推薦事件幾乎都落在這四層裡。這不是巧合 —— Google 自己就是用這個框架設計事件字典的。用它命名，未來要做轉換標記、受眾、Explorations 都省力。
(03)

案例示範：健身房官網的四層事件對應

來看一個具體例子。假設你接的案子是一間連鎖健身房，主要目標是「官網預約體驗課」。我們把四層漏斗填進實際事件裡：

CASE · 健身房「FitGym 信義店」
商業模式：免費體驗課 → 現場銷售年費會籍

KPI 排序：每月體驗課預約數 > 體驗課完成率 > 年費轉換率 > 網站流量

漏斗階段 使用者動作 GA4 事件 轉換？ 送廣告？
TOFU 看首頁 page_view — Meta: PageView
TOFU 捲到第 50% scroll — —
MOFU 看「課程介紹」頁 view_item — Meta: ViewContent
MOFU 看教練介紹影片 50% video_progress — —
BOFU 打開預約表單 form_start — Meta: InitiateCheckout
BOFU 點擊「撥打電話」 click_phone ✓ 次要 Meta: Contact / Google: Phone
CONV 成功送出預約表單 generate_lead ✓ 主要 Meta: Lead / Google: Conversion
💡
看懂這張表，你會發現：不是每個事件都要送到廣告平台。送太多反而讓 Meta/Google 演算法困惑。原則是：漏斗越深的事件，送得越積極。TOFU 的 scroll 根本不用送，但 BOFU 的 click_phone 一定要送。
(04)

埋碼規劃表：你的工作交付物

這張表不只是給自己看，它是你對業主、工程師、廣告操盤手溝通的「合約」。完整版應該包含以下 8 個欄位：

# 埋碼規劃表欄位（建議用 Notion / Google Sheet） 1. 事件名稱 ← GA4 標準或自訂 (snake_case) 2. 漏斗階段 ← TOFU / MOFU / BOFU / CONV 3. 觸發條件 ← 人話描述（送出表單、點擊 X 按鈕...） 4. CSS Selector ← 或 URL regex，前端工程師看的 5. 參數 (Parameters) ← value, currency, item_id... 6. 是否為轉換 ← 是/否 + 主次標記 7. 送往的平台 ← GA4 / Meta / Google Ads / LINE 8. 驗證方式 ← DebugView / Pixel Helper / Tag Assistant
⚠
新手最常踩的兩個坑：
① 把所有事件都標成「轉換」→ Google Ads 出價會失焦，CPA 爆掉
② 事件命名混用大小寫 / 空格 / 中文 → GA4 報表分組錯亂，三個月後沒人看得懂

下一節（1.4）我們會專門處理命名規範，包含 snake_case、UTM 參數治理、跨專案的統一字典。先把這張表想清楚，埋碼的 80% 工作就完成了 —— 剩下 20% 才是在 GTM 裡點來點去。

好的埋碼規劃師像建築師，
開工前畫好藍圖，工人才不會砌錯牆。
