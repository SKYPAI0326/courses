4.2

Meta CAPI：把漏掉的轉換從伺服器補回來

2021 之後，瀏覽器 Pixel 只能抓到大約 60% 的轉換事件。剩下 40% 被 iOS ATT、廣告封鎖、Safari ITP 吃掉了。CAPI（Conversion API）是 Meta 官方的救命稻草。

學完這節，你會
理解 Pixel 與 CAPI 的分工，以及為什麼要兩者並行
掌握 event_id 去重（dedup）的關鍵設計
了解三種 CAPI 實作路徑：Zapier / 伺服器端 GTM / 直接串 API
(01)

為什麼Pixel 不夠用了

Pixel（瀏覽器端）
·  依賴瀏覽器 cookie / JS ·  會被 iOS ATT 限制 ·  AdBlock 擋得掉 ·  Safari ITP 7 天清 cookie ·  命中率約 50-70%
CAPI（伺服器端）
·  從你的 server 直送 Meta ·  不受瀏覽器限制 ·  AdBlock 擋不到 ·  訂單資料可補首購金額 ·  命中率約 95-99%
◆
最佳實踐是兩者並行（Meta 官方建議），用 event_id 去重。Pixel 抓到的直接用；Pixel 漏掉的，CAPI 會補上。Meta 事件管理工具會顯示「Event Match Quality」分數，目標 > 7.0。
(02)

Dedup 的靈魂：event_id

若同一筆 purchase，Pixel 跟 CAPI 各自送給 Meta，會被算成兩筆轉換 —— ROAS 立刻假。解法：兩邊都帶同一個 event_id，Meta 看到重複就只算一次。

// 1. 結帳完成頁，前端產生唯一 ID（通常就是訂單編號） const eventId = 'TX20260418001'; // 2. Pixel 帶 eventID fbq('track', 'Purchase', { value: 2990, currency: 'TWD' }, { eventID: eventId }); // 3. 同時 push 到 dataLayer，讓伺服器端拿到同一個 ID dataLayer.push({ event: 'purchase', event_id: eventId, value: 2990, currency: 'TWD', user_data: { em: 'hashed_email', ph: 'hashed_phone' } });
(03)

三條實作路徑

  • 入門：Zapier / Make 中轉 — 適合無工程師的電商。WooCommerce 下單 → Webhook → Make → Meta Conversions API。優點免寫 code；缺點每月事件量有上限、延遲 30 秒+
  • 進階：伺服器端 GTM（sGTM） — Google Cloud Run 跑一個 GTM 容器，所有事件先進 sGTM 再分流出去。優點低延遲、完全控制；缺點要會 GCP、每月雲端成本約 $15-40 USD
  • 專業：直接串 CAPI — 後端工程師在訂單成立那一刻直接 POST 到 graph.facebook.com/v18.0/{pixel_id}/events。最穩、最快，但需要工程資源
💡
本課程後續演練採用「Pixel + Zapier Webhook」組合 —— 這是 90% 中小電商的可行解。sGTM 屬於進階選修，有興趣可到 Simo Ahava blog 找教學。
(04)

用戶資訊（user_data）一定要 hash

CAPI 要求你上傳 email / 電話 / 姓名等識別資訊，但必須先 SHA-256 雜湊。這不是為了資安，是 Meta 會用 hash 去跟他們資料庫比對，找出「這個人是哪個 FB 帳號」—— 比對到了才算一筆 match，Event Match Quality 才會高。

// 送進 CAPI 前，email 要先這樣處理 const email = 'Alice@Example.com '; const normalized = email.trim().toLowerCase(); // alice@example.com const hashed = sha256(normalized); // 在後端做
⚠
電話格式：一定要 E.164（+886912345678），去掉空格連字號。姓名：只留英文字母或中文，全小寫。任何一個欄位格式錯誤，Meta 直接丟棄整筆事件，你在事件管理工具的「診斷」頁籤會看到警告。
