MODULE 9 · SECTION 9.3

事件追蹤實作

用 GTM 建立三種常見事件追蹤：按鈕點擊、PDF 下載、表單提交。掌握觸發條件設計與 DataLayer 的基礎用法。

為什麼這很重要 · WHY THIS MATTERS
前兩節建立了 GTM 的概念和容器基礎，這一節是真正的實戰。點擊追蹤、下載追蹤、表單追蹤涵蓋了大多數服務型網站的核心轉換路徑。學完這三種追蹤模式，你就能舉一反三，應對絕大多數業務場景的追蹤需求。
STEP BY STEP · 操作步驟
1
啟用 GTM 的內建變數
在建立觸發條件前，先啟用 GTM 的內建變數，讓你可以在觸發條件中使用「Click URL」、「Click Classes」、「Click Element」等資訊。路徑：GTM 容器 → 變數 → 內建變數 → 設定 → 勾選所有「點擊」類別的變數（Click Element、Click Classes、Click ID、Click Target、Click URL、Click Text）。這是設定點擊追蹤的前提條件。
提示：也建議同時啟用「表單」類別的內建變數（Form Element、Form Classes、Form ID、Form Target、Form URL、Form Text），為後續的表單追蹤做準備。
2
實作一：追蹤特定按鈕點擊
以追蹤「聯絡我們」按鈕點擊為例。首先，在 LocalWP 網站用 Chrome DevTools 確認按鈕的 HTML 屬性（class 名稱或 id）。然後在 GTM 中：新建觸發條件 → 選「點擊 - 所有元素」→ 觸發條件設為「部分點擊」→ 條件設定「Click Classes 包含 contact-btn」。新建標籤 → GA4 事件 → 事件名稱填 click_contact_btn → 觸發條件選剛才建立的觸發條件。
// GTM 觸發條件設定邏輯 觸發條件類型：點擊 - 所有元素 觸發設定：部分點擊 條件 1：Click Classes 包含 contact-btn // 或使用 ID： 條件 1：Click ID 等於 contact-button // 對應標籤 標籤類型：Google Analytics: GA4 事件 設定標籤：（選擇已建立的 GA4 設定標籤） 事件名稱：click_contact_btn
3
實作二：追蹤 PDF 下載
GA4 加強型評估的「檔案下載」功能已能追蹤常見格式（pdf, docx, xlsx 等），但若需要區分不同 PDF 的下載，或記錄特定參數，可以用 GTM 建立更精確的追蹤。觸發條件：點擊 - 僅連結 → 部分點擊 → Click URL 包含 .pdf。
// PDF 下載追蹤設定 觸發條件類型：點擊 - 僅連結 觸發設定：部分點擊 條件 1：Click URL 包含 .pdf // 標籤設定（加入參數記錄檔名） 標籤類型：GA4 事件 事件名稱：download_pdf 事件參數： file_name → {{Click URL}} page_location → {{Page URL}}
與 GA4 加強型評估的差異：加強型評估的 file_download 事件已能自動記錄 file_name 和 file_extension，只有在需要額外自訂參數時才值得用 GTM 另建。
4
實作三：追蹤表單提交
表單追蹤比點擊追蹤複雜，因為需要區分「嘗試提交」與「成功提交」。GTM 提供「表單提交」觸發條件類型，在使用者點擊送出按鈕時觸發（不論是否成功）。若需要只追蹤「成功提交」，需要搭配 WordPress 表單外掛的成功頁面 URL 或成功回調函式。
// 方法 1：追蹤所有表單提交嘗試 觸發條件類型：表單提交 觸發設定：部分表單 條件：Form ID 等於 wpcf7-f1-p2-o1 // 方法 2：追蹤成功後跳轉頁面（更準確） 觸發條件類型：頁面瀏覽 條件：Page Path 等於 /contact/thank-you/ 事件名稱：form_submit_contact
最佳實踐：方法 2（追蹤感謝頁面瀏覽）是最準確的表單提交追蹤方式，因為它確認使用者真正完成了提交流程。若表單沒有跳轉頁面，可以用 Contact Form 7 的 wpcf7submit JavaScript 事件搭配 DataLayer push。
5
在預覽模式驗證三個追蹤標籤
建立完三個標籤後，進入 GTM 預覽模式，在 LocalWP 網站上逐一觸發各個事件：點擊「聯絡我們」按鈕、點擊 PDF 連結、送出聯絡表單。每次操作後，在 GTM Tag Assistant 面板確認對應標籤出現在「已觸發的標籤」清單（而非「未觸發的標籤」）。同時在 GA4 DebugView 確認事件資料和參數是否正確。
事件名稱觸發方式應看到的參數
click_contact_btn點擊聯絡按鈕page_location
download_pdf點擊 PDF 連結file_name、page_location
form_submit_contact造訪感謝頁面page_path
6
發布並標記為轉換
預覽確認無誤後，在 GTM 提交並發布新版本（建議命名「v1.1 新增點擊/下載/表單追蹤」）。回到 GA4 管理員 → 轉換，將 click_contact_btn 和 form_submit_contact 標記為轉換事件。之後在 GA4 的轉換報表中，就能看到透過 GTM 追蹤到的轉換次數，與流量來源交叉分析時，就能評估 SEO 的實際轉換貢獻。
課程重點回顧：SEO 帶流量 → GA4 追蹤行為 → GTM 精確記錄轉換 → 探索報表分析路徑。這三個工具的整合就是 Module 10 的主題。
完成驗證 · VERIFY
GTM 中建立三個事件追蹤標籤（點擊/下載/表單），預覽模式確認全部正常觸發，GA4 DebugView 能看到對應事件與參數，容器已發布新版本，兩個關鍵轉換已在 GA4 中標記。
常見問題 · TROUBLESHOOT
GTM 預覽模式中，標籤顯示為「未觸發」
檢查觸發條件的設定是否與實際元素匹配。用 DevTools 的 Elements 面板確認按鈕的 class 名稱（可能包含多個 class，GTM 的「包含」條件需要完整的 class 名稱）。也可以在 Tag Assistant 點擊「點擊」事件，查看實際傳入的 Click Classes 值是什麼。
表單提交觸發條件設定後，頁面跳轉時 GTM 標籤沒有觸發
若使用「頁面瀏覽」作為感謝頁面追蹤，確認感謝頁面的 URL 路徑與條件設定完全一致（注意結尾的 / 斜線）。若使用「表單提交」觸發條件但沒有跳轉頁面，需要確認「等待標籤觸發完成」選項已勾選，否則頁面可能在標籤傳送前就完成。
GTM 預覽的 Tag Assistant 沒有出現，瀏覽器顯示連線拒絕
GTM Tag Assistant 需要 Chrome 擴充功能支援，確認已安裝「Google Tag Assistant Legacy」或使用 Chrome 的 GTM 內建預覽功能。若 LocalWP 網站使用 HTTPS，需確認憑證被瀏覽器信任（看到 https:// 旁邊是鎖頭圖示而非警告）。
