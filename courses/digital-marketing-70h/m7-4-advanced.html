單元 7-4

進階追蹤：Meta Pixel、電商 Data Layer、GTM 除錯

GTM 不只能安裝 GA4。Meta Pixel 讓你追蹤廣告轉換與建立再行銷受眾；電商 Data Layer 讓產品數據自動傳入分析工具；完整的除錯流程則確保你的追蹤代碼真的在運作。

學習目標 OUTCOMES
能用 GTM 安裝 Meta Pixel 基礎代碼與標準事件（Purchase、Lead、ViewContent）
理解 Data Layer 的結構與 dataLayer.push() 的電商追蹤應用場景
能使用 GTM Preview 模式與 GA4 DebugView 完整驗證追蹤流程
META PIXEL · 安裝與標準事件

用 GTM 安裝 Meta Pixel

Meta Pixel 讓 Facebook/Instagram 廣告能追蹤網站轉換，並建立「曾瀏覽產品」「加入購物車」等再行銷受眾。透過 GTM 安裝，不需動到網站程式碼。

01
取得 Pixel ID
前往 Meta 事件管理工具 → 資料來源 → 連結新資料來源 → 網頁 → 取得 15 位數的 Pixel ID
02
GTM 新增代碼：自訂 HTML
代碼類型選「自訂 HTML」，貼入 Meta 提供的 Pixel 基礎代碼，將 YOUR_PIXEL_ID 替換為實際 ID
觸發條件：All Pages
03
新增標準事件代碼
針對每個轉換動作（購買、填表）新增一個代碼，在 Pixel 基礎代碼下方加入：
fbq('track', 'Purchase', {value: 0.00, currency: 'TWD'});
04
用 Meta Pixel Helper 驗證
安裝 Chrome 擴充套件「Meta Pixel Helper」，瀏覽目標頁面確認 Pixel 燈號為綠色，並確認事件正確觸發

Meta Pixel 標準事件對應不同轉換目標：

Purchase
完成購買，必須傳入 value 和 currency，用於廣告 ROAS 計算
Lead
填表、訂閱電子報等潛在客戶行為，最常用的轉換目標
ViewContent
瀏覽產品頁，建立「對特定商品有興趣」的再行銷受眾
AddToCart
加入購物車，追蹤購物意圖；配合棄購受眾效果最佳
InitiateCheckout
開始結帳，漏斗最後一步，轉換率通常高
Search
站內搜尋，傳入 search_string 了解用戶需求
DATA LAYER · 電商追蹤架構

Data Layer：讓數據主動推給追蹤工具

Data Layer 是頁面上的一個 JavaScript 陣列，網站開發者把產品資訊、交易資料推入其中，GTM 再讀取這些數據傳給 GA4、Pixel 等分析工具。行銷人不需要寫程式，但要能看懂結構並與開發溝通。

💡
概念：把 dataLayer 想成一個傳話站。網站說「有人買了商品 A，金額 NT$1,200」，GTM 聽到後轉告 GA4 和 Meta Pixel，不用每個工具各自撈一次。
dataLayer.push() — 購買事件範例
window.dataLayer = window.dataLayer || []; dataLayer.push({ 'event': 'purchase', 'ecommerce': { 'transaction_id': 'T_12345', 'value': 1200.00, 'currency': 'TWD', 'items': [{ 'item_id': 'SKU_001', 'item_name': '數位行銷課程', 'price': 1200.00, 'quantity': 1 }] } });
電商事件 觸發時機 必要參數
view_item 瀏覽產品詳細頁 item_id、item_name、price
add_to_cart 點擊「加入購物車」 item_id、item_name、price、quantity
begin_checkout 進入結帳流程 value、currency、items 陣列
purchase 訂單確認頁載入 transaction_id、value、currency、items
refund 退款完成 transaction_id、value（負值）
📋
與開發溝通：把上方 dataLayer.push() 範例給開發人員，說明「在這幾個頁面/動作上推入對應的 event」。GTM 的部分由行銷自行設定，開發只需負責推入 dataLayer。
DEBUGGING · GTM 除錯流程

系統性驗證，確保追蹤真的在運作

代碼設定完不等於正確觸發。每次發布前都需要用 Preview 模式驗證，每次上線後再用 GA4 DebugView 確認數據真的到了。

GTM Preview 模式
使用時機：發布前本地驗證
  • 點擊右上角「Preview」，輸入網站網址，開啟除錯視窗
  • 左側顯示頁面上觸發的所有事件，點擊可看到觸發了哪些代碼
  • 確認代碼狀態為「Fired」，若為「Not Fired」代表觸發條件未達成
  • 可在頁面上執行按鈕點擊、表單送出等動作，即時看到事件清單更新
GA4 DebugView
使用時機：驗證數據是否到 GA4
  • GA4 後台 → 管理 → DebugView
  • 在瀏覽器安裝「Google Analytics Debugger」擴充套件並啟用
  • 執行追蹤動作後，數秒內在 DebugView 看到事件出現
  • 點擊事件名稱可展開看事件參數值，確認參數傳遞正確
Meta Pixel Helper
使用時機：驗證 Pixel 事件
  • Chrome 擴充套件，安裝後瀏覽目標頁面自動偵測
  • 綠色圖示表示 Pixel 正常；數字代表偵測到的事件數
  • 點擊擴充套件可看到觸發了哪些事件與參數
  • 紅色警告表示 Pixel 有問題需修復
GTM Assistant（Chrome）
使用時機：快速分析 GTM 設定
  • 可直接在頁面上看到哪些 GTM 代碼已觸發
  • 不需進 GTM 後台，快速掃描頁面的代碼狀態
  • 適合快速確認第三方網站是否安裝了特定代碼

常見問題排查

問題
代碼在 Preview 顯示 Not Fired
解法
確認觸發條件設定是否正確；用 Preview 點擊該元素，在 Variables 頁確認 Click ID / Click Text 的實際值是否與觸發條件填入的值一致
問題
GA4 DebugView 看不到事件
解法
確認 GA4 Debugger 擴充套件已啟用；確認 GA4 設定代碼中的評估 ID 正確；等待 5–10 秒讓事件傳送
問題
表單送出代碼未觸發
解法
確認表單是否用 AJAX 送出（若是，GTM 的 Form Submission 不會觸發）；改為監聽感謝頁面 URL 或用自訂 JavaScript 監聽表單送出事件
問題
同一個事件在 GA4 出現重複
解法
檢查是否有增強型評估和自訂事件同時追蹤同一行為（如外部連結點擊）；在自訂代碼的觸發條件中加入排除條件
