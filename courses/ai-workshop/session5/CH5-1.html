5.1

業務運算與流程工具：把計算邏輯與判斷流程封裝起來

報價、排班、KPI 計算、核取流程確認——這些是職場裡最常見、最適合工具化的場景。這一節帶你認識四種工具類型，並掌握每種類型的指令寫法和除錯技巧。

學完這節，你會
認識四種業務工具類型：計算型、判斷型、流程型、查詢型
掌握每種類型的關鍵指令寫法差異
能快速診斷工具常見問題並用對話修正
(01)

四種業務工具類型

不同類型的工具，指令寫法的重點不同。先認清你要做的是哪一種，再決定怎麼描述需求。

🧮
計算型
輸入數值 → 輸出結果
核心是數學公式或條件計算。重點在把所有計算規則和例外條件都說清楚。
→ 報價計算機、薪資試算、ROI 計算器
🔀
判斷型
輸入條件 → 輸出建議或分類
核心是 if-else 邏輯。重點在把所有情境和對應結果都列出來，不能有遺漏的分支。
→ 客戶分級器、風險評估表、資格審核工具
✅
流程型
一系列步驟的引導與確認
核心是把 SOP 數位化。每個步驟有說明、有核取方塊，完成所有步驟才能繼續。
→ 上架檢核表、客戶 onboarding 流程、活動籌備清單
📋
查詢型
輸入關鍵字 → 搜尋並顯示結果
核心是把靜態資料表變成可搜尋的介面。重點在定義資料結構和搜尋欄位。
→ 產品目錄查詢、FAQ 搜尋器、人員聯絡表
(02)

各類型的關鍵指令寫法

同樣用四維度框架，但每種類型在「處理邏輯」這個維度的寫法重點不同：

計算型工具 · 指令重點
處理邏輯要寫：
· 完整的計算公式（用數學符號或自然語言都行）
· 所有的例外條件（「如果 X 則…，否則…」）
· 邊界值的處理（「最小值為 0，不能出現負數」）

範例：底價 = 人天數 × 日費率。若總金額超過 50,000 則自動打九折。若客戶勾選「政府單位」則不打折。最終金額無條件進位到百位數。
判斷型工具 · 指令重點
處理邏輯要寫：
· 所有判斷條件（用「如果…則…」格式）
· 每個條件對應的輸出或建議
· 沒有符合任何條件時的預設結果

範例：若「購買頻率」≥ 每月一次 且「客單價」≥ 5,000 → 標記為 VIP。若只符合其中一個條件 → 標記為 Regular。兩個都不符合 → 標記為 New。各分級顯示對應的後續行動建議。
流程型工具 · 指令重點
處理邏輯要寫：
· 完整的步驟清單（順序、層次）
· 哪些步驟有子項目
· 完成所有步驟後要顯示什麼（完成畫面、可下載清單等）

範例：共 4 個階段，每階段 3–5 個核取步驟。每個步驟有簡短說明文字。全部打勾後顯示「流程完成」並讓使用者可以複製完成摘要。進度存入 localStorage。
(03)

工具常見問題快速除錯

工具生成後測試發現問題，不要重新問——在同一對話裡告訴 AI 哪裡錯了。以下是最常見的問題類型和對話修正寫法：

問題
計算結果不正確
對話修正寫法
「當我輸入 [X=3, Y=200] 時，結果顯示 600，但正確答案應該是 450。請重新確認計算邏輯並修正。」
問題
某個條件分支沒有觸發
對話修正寫法
「當 [條件 A] 時，應該顯示 [結果 B]，但目前工具沒有反應。請加入這個條件分支。」
問題
頁面顯示空白或出現亂碼
對話修正寫法
「工具用瀏覽器打開後頁面是空白的。請確認 HTML 結構是否完整，並在 <head> 加入 <meta charset="UTF-8">。」
問題
使用者輸入後沒有回應
對話修正寫法
「填入數字後點擊計算按鈕，結果區域沒有任何顯示。請確認按鈕的 onclick 事件和輸出元素的 id 是否正確連結。」
✦
除錯原則：永遠描述「期望結果」和「實際結果」的差距，而不是說「這個工具壞掉了」。越精確的錯誤描述，越快得到正確修正。
(04)

你的工具存在哪裡

HTML 工具建好後，有三種常用的存放和分享方式，選適合你情境的：

三種存放方式
① 本機檔案（最簡單）
存在電腦裡，雙擊開啟。適合只有自己用、不需要分享的工具。

② Google Drive 或 OneDrive 分享連結
上傳 HTML 檔案後，分享連結給同事下載。適合少數人共用、不需要即時更新的工具。

③ GitHub Pages（免費託管）
把 HTML 上傳到 GitHub repository，啟用 Pages 後得到一個永久網址。適合需要讓更多人存取、或需要版本控制的工具。
