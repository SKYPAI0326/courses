(02) Part 1 · 本課招牌單元

三層知識庫設計（NotebookLM）

一個 notebook 放所有東西，查詢結果很亂——問題出在「沒設計」。本單元個人層深度教（50 min），讓你回家第一週就能跑出來；部門層 / 跨來源層快速示範，讓你看懂未來怎麼推動。加一段 3 大現實阻力對策，帶走 Min 版設計表（第一週用）+ Pro 版設計表（3-6 個月期程）。

學習成果 · OUTCOMES
能完成個人層 notebook（1 notebook + 3 上傳規則 + 5 查詢題），回家第一週就能複製
能識別 3 大現實阻力（資安 / 版本管理 / 跨部門政治）並選對對策方向
能產出知識庫設計表 Min 版（4 件套產物 #1），帶走 Pro 版作 3-6 個月期程參考
(01)

Decision Checkpoint：什麼資料屬於哪一層？（3 min）

為什麼這很重要
把所有資料倒進 1 個 notebook，查詢結果一定很亂——個人筆記的引用和部門政策的引用混在一起，你不知道哪個答案「能拿去跟同事講」。分層不是麻煩，是查得準的前提。

5 部門代表各答 1 句（30 秒一輪）：什麼資料屬於哪一層？

凱倫（財務）
「我每天記的對帳便箋是個人層、公司會計政策是部門層、年度合併報表流程是跨來源層。」
小李（業務）
「我的會議紀錄屬個人層、業務部報價政策是部門層、新品引進要整合採購倉儲算跨來源。」
怡君（電商）
「我的競品筆記是個人層、電商平台規範是部門層、跨平台促銷對齊是跨來源。」
阿翔（倉儲）
「我的盤點手記是個人層、倉儲 SOP 是部門層、跨倉調撥牽涉採購業務算跨來源。」
采蓁（採購）
「我的供應商私下筆記個人層、採購流程部門層、年度供應商評鑑跨採購業務財務算跨來源。」

講師收斂三軸切分——三軸任一達標就升級到上一層：

(01)
資料擁有者
只有你自己 → 個人
同部門共用 → 部門
多部門共用 → 跨來源
(02)
共享範圍
完全私人 → 個人
同部門 share → 部門
跨部門 share → 跨來源
(03)
更新頻率
每天 / 每週 → 個人
每月 / 每季 → 部門
半年以上 → 跨來源
→
本課野心邊界：學員 95 min 課堂能跑出的、且回家能複製的，只有個人層。部門層 / 跨來源層需要公司治理配套，不是一人下週能落地的事。本課讓你看懂、知道未來怎麼推動，但不要求課堂跑完三層。
(02)

│  ★ 進階情境，CH2-1 會深入；本課快速看過                │
├──────────────────────────────────────────────────┤
│  部門層（每月/每季更新 · 同部門 share · 操作 SOP 用）    │
│  例：業務部報價政策 / 採購流程 SOP / 倉儲盤點規範         │
│  source 數：15-25 · 命名：「{部門}-部門共用-v{版本}」     │
│  ★ 進階情境，本課快速示範；需先過版本管理關卡            │
├──────────────────────────────────────────────────┤
│  個人層（每天/每週更新 · 完全私人 · 自己日常工作用）       │
│  例：你的會議紀錄 / 工作日誌 / 個人筆記                   │
│  source 數：5-15 · 命名：「{你的名字}-個人-{季度}」        │
│  ★ 本課重點，深度教 50 min — 你回家第一週要做的版本     │
└──────────────────────────────────────────────────┘

個人層回答「我昨天到底承諾了什麼」
部門層回答「業務部的標準做法是什麼」
跨來源層回答「這個跨部門決策牽涉哪些已知資訊」

第 2 步：3 條上傳規則（學員回家要套用 · 7 min）

個人層最大風險不是 source 選錯，是「不確定能不能上就先上」。3 條規則：

規則內容範例
規則 1（能上）你寫的 + 只你看 + 隨時改 + 不含他人個資自己整理的客戶觀察筆記、自己摸出來的 SOP
規則 2（不能上）含他人個資 / 含公司機密 / 屬部門共用文件客戶名單 .xlsx、財報、業務 SOP v2026
規則 3（不確定）暫存 Google Keep 或本地檔案，不上 NotebookLM，下週問主管或資安窗口「老闆轉發但沒明說機不機密的內訓 PDF」

3 條具體判斷示範：

source 候選進個人層？為什麼
小李寫的「A 客戶不爽我們服務」私下觀察✓ 進規則 1：你寫的 + 只你看 + 不含他人個資
客戶名單 customer-list.xlsx✗ 不進規則 2：含他人個資（PII）
公司內部 wiki 上的「業績核算公式」截圖✗ 不進屬部門層（同部門人都該用同一份）
部門經理轉發的「明年加薪規畫」word⚠ 暫存規則 3：不確定（轉發人沒明說機不機密），先 Google Keep、下週問
個人層 3 條上傳規則 = 能上 / 不能上 / 不確定就先暫存。第三條是 80% 學員會卡的地方，課堂建立習慣。

第 3 步：命名規則（3 min）

個人層命名公式
個人層命名：{你的名字}-個人-{季度} 範例：小李-個人-2026Q1 理由： - 用「名字」開頭 → notebook 列表排序時自己的個人 notebook 連在一起 - 含「個人」字樣 → 一眼分辨層級（避免後續跟部門層搞混） - 含「季度」→ 每季度建新 notebook、舊的封存，避免 source 累積到 50 上限

第 5 步：1 題驗證題（5 min）

驗證題 = 故意問一條個人層 source 不該答得出來的問題，看 NotebookLM 會不會誠實說「沒明寫」還是亂編。

小李在 NotebookLM（個人層 · 驗證題）
驗證題：業務部報價政策是什麼？（注意：這應該是部門層的問題）
NotebookLM 大概會回

依目前 sources（個人筆記與會議紀錄），未明確記載業務部報價政策。sop-personal-報價追蹤.md 提及「小李個人跟催節奏」，但這是個人做法非部門政策。建議查閱業務部部門共用 SOP。

(04)

部門層 / 跨來源層快速示範（12 min）

講師 demo 為主，學員看懂、不要求課堂跑完
從個人層升級到部門層，核心多了一件事：核可。跨來源層不是把部門層 source 全倒進去，而是為特定跨部門主題量身建 notebook。

部門層快速示範（POV 小李 → 業務部 · 7 min）

業務部部門層 notebook 大概樣貌（20 份 source）： ✓ 業務 SOP v2026.1.pdf（部門經理 12 月核可） ✓ 客戶分類規則 v2025.3.pdf（去年 6 月核可、仍適用） ✓ 報價政策 2026.pdf（12 月核可） ✓ 折扣審核流程 2026.pdf ✓ 過往成功提案 ×10 份（精選 2025 年成交案） ... 共 20 份 不納入： ✗ 小李個人的會議紀錄（屬個人層） ✗ 兩年前的舊版 SOP（超過半年 + 已淘汰） ✗ 業務 group chat 截圖（未經核可、非正式 source）

(05)

現實阻力處理段（10 min）

識別 + 選對方向，比解決問題更重要
部門層 / 跨來源層回部門落地會撞到 3 大現實阻力。這段給你對策表，不期望課堂解決，期望你識別 + 選對方向。
阻力 1 · 3 min
公司不允許上真實資料到免費版
免費版 NotebookLM 雖 Google 宣稱不訓練模型，但仍存在 Google 雲端。多數公司資安政策不允許客戶 PII / 供應商機密 / 財務細節上免費版。
▶
對策做法適用情境
A · 模擬建用本課提供的虛構素材包在課堂練熟，回家不上真資料、只模擬建 notebook + 查詢題框架公司資安明示禁止 / 你還沒問清楚
B · 公開段用內部 wiki 的公開段（FAQ / 公開政策 / 對外資料）建第一個 notebook公司有部分文件本來就對外公開
C · 💰 升級升級 💰 NotebookLM Plus（資料隔離）或申請公司核可的內部 AI 工具必須含敏感資料且公司願意付錢

判斷流程：先問資安窗口能不能上 → 不行就走 A → 有公開資料就走 B → 真的需要就提 C 給主管。

阻力 2 · 3 min
SOP / 政策沒有版本管理
部門層 selection 規則「已核可 + 半年內」假設公司 SOP 有正式版本管理。多數中小企業沒有——SOP 散在 Word / PDF / Notion / Google Drive，沒人知道哪份是最新核可版。
▶
對策做法適用情境
A · LMD 代替用「最後修改日期 + 修改人」當代替：上傳前在檔名加 _LMD-2025-12_張經理公司有檔案但沒版本制度
B · 標警語source 描述明標 ⚠「本 source 版本未經核可，僅供查詢參考，最終以紙本/系統為準」你不確定哪份才對、但需要先用起來
C · 個人共用層建「個人共用層」（自己整理 + 自己標記版本 + 自己負責），不假裝是「部門共用」公司治理短期不會改、但你需要小範圍跟同事共用

「個人共用層」概念：名稱仍含「個人」，但邀請 2-3 個同事 share。本質還是「你負責、你維護」，不要佯裝部門權威。

阻力 3 預告 · 4 min
跨來源權責政治

(08)

5 個會卡關的地方 + 解法

01 把所有東西倒進 1 個 notebook（無分層）
現象
把個人筆記 + 部門 SOP + 跨部門資料全部丟同一個 notebook。問「我上週開過幾次會」答得出來、問「業務部報價政策」也答得出來——但兩個答案的引用混雜，個人筆記裡剛好提到「老闆說折扣超過 20% 要簽」被當成部門政策。
原因
NotebookLM 不會主動分層，使用者沒設計就會全混。
解法
從個人層開始建（先把 CH1-1 跑出的會議紀錄存起來作第 1 個 source），不要從「公司大全」這種想法開始。三軸切分任一達標就升級。寧可一開始 source 少、慢慢長，不要一開始全倒、後面再分（後面分非常痛）。
02 把客戶個資 / 供應商成本表丟進免費版 notebook
現象
怡君為了查「VIP 客戶權益」，把含完整客戶名單的 customer-list.xlsx 上傳到 NotebookLM 免費版。後來公司資安稽核發現、被記點。
原因
免費版 NotebookLM 雖然 Google 宣稱不會用 source 訓練模型，但仍存在 Google 雲端，公司資安政策通常不允許客戶 PII / 供應商機密上免費版。
解法
個人層規則 2「不能上」就是擋這種錯（含他人個資 / 含公司機密 / 屬部門共用文件）。不確定就規則 3 暫存，下週問資安窗口。真的要用就走阻力 1 三對策：A 模擬建 / B 公開段 / C 升級 💰 NotebookLM Plus。
03 跨來源層變成「全公司資料垃圾桶」
現象
小李建跨來源 notebook「2026 公司全資料」，把業務 / 採購 / 倉儲 / 財務 / 電商部門 SOP 全部倒進去。結果問「新品引進流程」NotebookLM 引用了 8 份不同部門的 source、答案 800 字夾雜各部門視角，沒人看得懂。
原因
跨來源層不是儲存庫，是為特定跨部門主題量身建的 notebook。沒有主題的跨來源 notebook 等於沒有焦點，查詢結果必然散。
解法
一個跨來源 notebook = 一個主題（例：「新品引進」「促銷備貨」「供應商評鑑」），主題寫進命名。source 進來前問「對這個主題真的有用嗎？」，不確定就不收。不同主題建不同 notebook。
04 沒寫驗證題、不知道知識庫設計是否成功
現象
學員建完個人層 notebook、跑了幾題查詢都答得出來，覺得「設計成功」。但下週回家用真實資料時，問「業務部報價政策」結果 NotebookLM 用個人層的會議紀錄答了（小李會議裡剛好提到折扣案例），學員以為這就是部門政策、拿去跟同事爭執。
原因
沒有設計驗證題（故意問「超出本層範圍」的問題測誠實度），無法確認 notebook 會不會「越界」答。
解法
每層 notebook 必寫 1 題驗證題：個人層問部門政策、部門層問個人行程、跨來源層問純部門內部 SOP。驗證題的正確回應是 NotebookLM 說「sources 中未明確記載 / 建議查閱另一層」，不是給出答案。若驗證題被「答出來」→ notebook 設計有問題，回頭剔 source。Min 版設計表強制留 1 格驗證題、跑出來才能離場。
05 以為 NotebookLM 能取代公司文件管理系統
現象
學員上完課很興奮、把所有 SOP / 政策 / 客戶資料 / 供應商資料倒進 NotebookLM 想當「公司文件中心」。3 個月後新人問「最新版報價政策在哪」、有人答 NotebookLM、有人答 SharePoint、有人答 Google Drive——沒人知道哪份才是真相。
原因
NotebookLM 是「查詢工具」不是「文件管理系統」——沒有權限控制、沒有版本追蹤、沒有修改紀錄、沒有審核流程。它讓你快速查公司既有文件管理系統的內容，但不能取代文件管理系統本身。
解法
公司文件中心仍走原來的位置（SharePoint / Google Drive / Notion / 紙本），不要搬到 NotebookLM。NotebookLM 的 source 是從文件中心匯出的「快照」——查得快但不是真相。如果你想推動「全公司用 NotebookLM 當文件中心」，請走 💰 企業版採購 + 資安評估 + 治理導入，不是用免費版湊。阻力 2「SOP 沒版本管理」如果是公司本來的問題，不是 NotebookLM 能解決的。

Q1（規則套用）：以下哪份 source 應該進個人層？

Q2（情境決策）：你是凱倫（財務），剛上完課想建個人層 notebook，但公司資安政策明示「任何含財務數字的檔案不得上 Google 雲端服務」。你會怎麼選對策？請說出選擇 + 理由 + 具體做法。
顯示答案要點
首選對策 A · 模擬建：課堂用 personal-finance-5.zip 虛構素材練熟流程，回家用模擬資料或改過數字的版本建 notebook，跑出 5 題查詢題框架。真實月結 / 預算 / 異常筆不上傳，只在本地用查詢題框架自問自答。

次選對策 B · 公開段：若公司有對外財報 / 對股東公開的資料，可建一個「公開財務 notebook」練手。

不建議對策 C：除非主管願意付 💰 NotebookLM Plus 預算且資安單獨評估通過——財務資料的敏感度通常需要公司核可的內部 AI 工具而非單純 Plus 版。

加分：能對應 Pitfall 第 2 條「把客戶個資丟進免費版」，並說出「不確定就規則 3 暫存、別當小白鼠」。
Q3（驗證題設計）：你是阿翔（倉儲），剛建好「阿翔-個人-2026Q1」notebook，含 5 份盤點手記 / 缺貨追蹤 / 異常處理日誌。你要為這個 notebook 設計 1 題驗證題。以下 4 個候選驗證題，哪個最能驗證 notebook 不會越界答？
顯示答案
正確答案：B。A 和 C 是個人層該答的問題、不是驗證題；D 是通用概念、與本 notebook 設計無關。B 是「部門層的問題」——個人層 notebook 應該誠實說「sources 中未記載部門 SOP、建議查倉儲部門共用 notebook」。如果 B 被「答出來」（notebook 內混進阿翔的部門 SOP 草稿），表示 source selection 過寬，要回頭剔。加分：能說出「驗證題的本質是故意問超出本層範圍的問題」+ 對應 Pitfall 第 4 條。
