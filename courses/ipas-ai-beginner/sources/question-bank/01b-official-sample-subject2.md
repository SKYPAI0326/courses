## S2-Q1【B】No Code / Low Code 主要特色

**題幹**：下列何者最能表達 No Code / Low Code 平台的主要特色？
- (A) 需要撰寫大量程式碼
- (B) ✅ 運用模板快速建立應用程式
- (C) 僅供專業開發人員使用
- (D) 只能製作靜態網站

**詳解**：NC/LC 核心精神是「拖拉模板」，讓非工程師也能組裝應用。A 與「無代碼」精神相反；C 描述顛倒（NC/LC 正是給非開發者用的）；D 過度限縮（NC/LC 也能做動態應用）。

**對應章節**：CH5 No Code / Low Code 概念
**0 基礎註記**：第一次出現要展開「No Code = 零程式碼，靠拖拉組裝；Low Code = 少量程式碼，可加自訂邏輯」。

## S2-Q2【C】推理模型特點（Reasoning Model）

**題幹**：下列何者**不是**推理模型（Reasoning Model）的主要特點？
- (A) 具備多步驟邏輯推理能力
- (B) 具備良好的可解釋性與邏輯一致性
- (C) ✅ 採用強化式學習的訓練方式
- (D) 回應內容結構清晰、推理脈絡完整

**詳解**：注意題幹是「**不是**」。推理模型（如 o1）核心特徵是「先思考再回答」、邏輯一致與多步推理。雖然部分推理模型有用 RLHF/GRPO，但「強化式學習」不是推理模型**核心定義特徵**——傳統 LLM 也會用強化學習。所以 C 為「最不能代表」推理模型的特徵。

**對應章節**：CH6 生成式 AI 工具｜模型選擇
**0 基礎註記**：先解釋「推理模型 = 會分多步思考再給答案的 AI（例：OpenAI o1）」，跟一般 ChatGPT 直接秒答對比。

## S2-Q3【C】Low Code vs No Code 適用情境

**題幹**：下列哪種情況，選擇 Low Code 平台可能比 No Code 平台更為適合？
- (A) 需要非技術人員快速進行開發與應用
- (B) 應用需求簡單，無需自訂功能
- (C) ✅ 需要較複雜的業務邏輯並使用自訂整合功能
- (D) 預算和時間極度有限

**詳解**：LC = Low Code，比 NC 多出「可寫少量程式」，所以遇到複雜業務邏輯時 LC 較適合。A、B、D 都是 NC 的典型場景。

**對應章節**：CH5 NC/LC 工具選擇

## S2-Q4【D】生成式 AI 與 NC/LC 不適用情境

**題幹**：關於生成式 AI 與 No Code / Low Code 平台的應用，下列何者最不適合？
- (A) 自動生成程式碼
- (B) 自動化生成行銷文案
- (C) 快速開發個人化 App
- (D) ✅ 自動化生成法律判決

**詳解**：法律判決涉及司法權、人權與責任歸屬，**不能由 AI 自動產出**（屬於 EU AI Act「禁止」/「高風險」類）。A、B、C 都是合法且常見的 AI 應用情境。

**對應章節**：CH7 AI 倫理與法規（與 CH1 EU AI Act 風險分級呼應）

## S2-Q5【C】No Code vs Low Code 比較

**題幹**：關於 No Code / Low Code 平台，下列敘述何者較正確？
- (A) 兩者完全相同

## S2-Q6【C】Chain-of-Thought 提示語

**題幹**：在使用生成式 AI 模型解決數學或邏輯問題時，若希望引導模型逐步推理以提升答案的準確性與可解釋性，應設計具備思維鏈（Chain-of-Thought）推理特性的提示語。下列哪一組提示語最能有效啟用模型的 Chain-of-Thought 推理能力？
- (A) 請回答這個數學問題，並直接給出答案
- (B) 幫我算出 125 × 12 是多少？
- (C) ✅ 請詳細列出每一個思考步驟，最後再給出答案
- (D) 請用一句話簡要回答問題

**詳解**：CoT 核心是「逐步思考」，C 直接要求列出思考步驟。其他選項都是「直接給答案」的 zero-shot 提示。

**對應章節**：CH6 提示工程（Prompt Engineering）
**0 基礎註記**：把 Chain-of-Thought 翻成「思維鏈」、白話「叫 AI 一步一步想」。

## S2-Q7【D】Low Code 風險（影子 IT）

**題幹**：使用 Low-Code 平台進行開發時，企業應特別留意下列哪一項潛在風險？
- (A) 可能造成企業內部敏感資料的洩露
- (B) 難以進行大規模的應用擴展和維護
- (C) 開發成本將大幅增加
- (D) ✅ 可能有未經 IT 部門管理的應用程式擴散

**詳解**：D 是 LC 最特殊的風險「Shadow IT 影子 IT」——非工程師大量自建應用導致 IT 失控。A、B 不是 LC 獨有；C 是反向（LC 通常降低成本）。

**對應章節**：CH5 NC/LC 風險｜CH7 治理風險
**0 基礎註記**：「影子 IT」要展開白話：「員工偷偷用 LC 平台做出 IT 部門不知道的小工具，導致資料管理失控」。

## S2-Q9【D】個資跨境傳輸限制

**題幹**：某公司擬將客戶個人資料傳輸至國外進行雲端儲存，依據《個人資料保護法》規定，下列哪一種情況下，中央目的事業主管機關有權對該國際資料傳輸行為進行限制？
- (A) 公司規模未達主管機關要求標準，管理能力存疑
- (B) 公司曾發生個人資料外洩事件紀錄，造成用戶信任流失
- (C) 傳輸資料包含當事人已公開資訊，但仍涉及隱私風險
- (D) ✅ 接收國家之個人資料保護法規尚未完備，可能損害當事人權益

**詳解**：依個資法第 21 條，主管機關可限制國際傳輸的法定情境主要是**接收國法規不足以保障當事人**。A、B 屬於企業自身能力問題；C 已公開資訊不適用個資法限制。

**對應章節**：CH7 個資法與跨境傳輸（**0 基礎學員必懂**）

## S2-Q18【A】第三方 AI API 隱私保護

**題幹**：若企業使用第三方 AI API（如 OpenAI、Claude、Gemini 等），協助處理內部資料，為避免敏感資訊遭模型記憶或進一步分析，應優先採取下列哪一種作法？
- (A) ✅ 啟用 Zero-Retention 模式或採用具企業資安保障的 API 服務
- (B) 使用虛擬機建立隔離環境，以模擬匿名資料操作流程
- (C) 將原始數據進行視覺化轉換，以避免直接暴露文本內容
- (D) 將輸入資料轉換為向量嵌入後，再傳輸至語言模型進行處理

**詳解**：Zero-Retention（不保留模式）是直接從資料源頭防止模型記憶的最有效方式。B、D 仍會傳輸原始/可還原資料；C 視覺化不解決資料外傳問題。

**對應章節**：CH7 AI 資安｜API 隱私
**0 基礎註記**：「Zero-Retention = 不保留模式，AI 用完資料就丟掉、不存」是必懂概念。

## S2-Q23【A】生成式 AI 敘述（複選排列）

- A. 生成的內容不會帶有偏見
- B. 具有高度準確性，不會有虛假信息
- C. 生成內容的準確性，建議需要經過人類審核

- E. 生成式 AI 具高度安全性，不會導致數據外流

**選項**：

- (B) A、C、D

## S2-Q24【B】跨國企業財務 AI 效益

- (D) 提高品牌曝光度並增強客戶關係管理

## S2-Q25【C】GPU 利用率低（I/O 瓶頸）

**題幹**：某製造業團隊建立瑕疵偵測模型時，使用單張約 20MB 的高解析度工業影像作為訓練資料，並將所有影像儲存於 NAS（Network Attached Storage）中。訓練時透過 PyTorch 搭配 CNN 模型於 GPU 上進行，工程師觀察到 GPU 使用率僅約 20% 至 30%，整體訓練時間耗費數日。根據上述情境，造成 GPU 利用率低的最可能原因為何？
- (A) 模型架構過於複雜，導致反向傳播時間過長
- (B) 訓練資料品質不穩定，造成模型難以收斂
- (C) ✅ 高解析影像從 NAS 載入產生 I/O（Input/Output）瓶頸，導致 GPU 等待
- (D) 批次大小（Batch Size）設定不當，導致 GPU 長時間閒置

**詳解**：題幹明確指出影像 20MB + NAS 儲存，這是典型的 I/O 瓶頸場景（GPU 比資料載入快太多就閒置）。其他選項雖也會影響 GPU 利用率，但跟題幹情境的最直接因果是 NAS I/O。

**對應章節**：CH7 IT 環境支援｜效能問題排查
**0 基礎註記**：白話「GPU 跑得太快，硬碟餵資料來不及，所以 GPU 閒著等」。

- (B) ✅ 風險轉移

## S2-Q30【D】幻覺現象

**題幹**：使用生成式 AI 協助撰寫一份有關產業碳排政策的分析報告，AI 提供的內容看起來語句通順且用詞專業，但經查證發現其中提到的法規條文與企業名稱並不存在。請問造成此現象最可能的原因為下列何者？
- (A) 模型語料涵蓋不足，導致資料無法完整反映現實
- (B) 提示語過度簡略，導致生成內容缺乏真實依據
- (C) 模型缺乏檢索能力，無法即時查詢正確資訊
- (D) ✅ 模型產生幻覺內容，虛構與真實資訊混雜出現

**詳解**：「看起來真實但實際虛構」就是幻覺（Hallucination）的標準定義。A、B、C 雖會增加幻覺機率，但題幹描述的現象本身就是「幻覺」。

**對應章節**：CH6 生成式 AI 限制｜幻覺
**0 基礎註記**：「幻覺 = AI 一本正經地胡說八道」是必懂概念。

**題幹**：在管理生成式 AI 系統的隱私風險時，下列哪一種技術最能確保數據使用的安全性？
- (A) 強化學習（Reinforcement Learning）
- (B) 深度學習（Deep Learning）

## S2-Q33【A】Temperature 設定

**題幹**：使用生成式 AI 產出需要邏輯清晰、高度準確性與一致性的文本時，下列哪一種溫度（Temperature）設定最能有助於降低回答的不確定性並提升一致性？
- (A) ✅ 將溫度設定在 0.1 至 0.3，使輸出結果趨於穩定可預測
- (B) 將溫度設定在 0.5 至 0.7，以取得適度創意與合理性平衡
- (C) 將溫度設定在 1.0 至 1.2，提高語句的多樣性與發散性
- (D) 維持預設溫度不變，讓模型依照原始設定自由生成

**詳解**：Temperature 越低 → 越穩定可預測；越高 → 越創意發散。要邏輯清晰用低溫。

**對應章節**：CH6 提示工程｜參數設定
**0 基礎註記**：「Temperature = 創意度旋鈕，0 是嚴謹、1 是發散」是必懂。⚠️ 0 基礎友善版可摺疊成補充版（避免初登場就名詞炸彈）。
