資料處理與分析概念

本章與 CH3 並列為題量最大的兩章，共涵蓋 10 道官方樣題。讀完後你會懂得分辨資料種類、清洗問題、ETL 步驟，以及用統計量快速判題。

學習成果

(01)

先把資料想成「整理倉庫」

為什麼這很重要
整個 CH2 是「資料側」的基礎，CH3 才是「模型側」。如果資料本身有問題，再好的模型也沒用。本章六個小節對應「整理倉庫」的六個步驟，只要記住這個比喻，就不會迷路。

在進入正式名詞前，先建立一個直覺：資料處理 = 把雜亂無章的倉庫，整理成可以快速找到東西的貨架。

倉庫整理的階段對應到資料的工作本章小節
看貨物有哪幾種（盒裝 / 散裝 / 液態）認識資料類型（結構化 / 半 / 非）2.1
把破損、過期、重複的貨清掉資料清洗（遺缺、重複、錯誤、離群）2.2
從各倉庫搬到中央倉，重新貼標ETL（萃取 → 轉換 → 載入）2.3
數一下每樣貨平均多少、最多最少集中趨勢與分散度（統計量）2.4
找出哪箱明顯不對勁異常檢測（Z-score、離群值）2.5
描述「我們倉庫到底有多大、變多快」大數據 5V 特性2.6
💡
0 基礎友善承諾：本章不要求你會程式、不要求你會數學，所有公式都用比喻說明。目標只有一個 — 讓你看懂題目選項，選出對的答案。
(02)

2.1 資料的種類與來源

名詞定義
資料（Data）：任何「電腦看得懂、可以存起來、之後可以拿來分析」的東西。文字、數字、圖片、聲音、影片都算。

考試最常用的分類有三種，請記住這張對照表：

類型白話定義工作上看過的例子常見儲存方式
結構化資料
Structured Data		 像 Excel 一樣有清楚的行與列，每個欄位都有明確型態 員工年齡、年資、銷售量、客戶 ID、財務支出 MySQL / PostgreSQL 等關聯式資料庫
半結構化資料
Semi-structured Data		 有標籤、有層次，但欄位不一定每筆都一樣 系統設定檔、API 回傳資料、網頁 XML / JSON / CSV
非結構化資料
Unstructured Data		 沒有固定欄位，就是一坨原始內容 X 光片、監控錄影、客服電話錄音、PDF 文章、郵件內文 檔案系統、物件儲存（S3）
💡
記憶口訣：「Excel 就結構，JSON 半結構，影音就非結構」。
⚠️
容易混淆點：「資料結構分類」（結構化 / 半 / 非）和「資料型態」（數值 / 文字 / 日期 / 布林）是兩件不同的事。前者在講「整體形狀」，後者在講「單一欄位裡裝什麼」。
樣題 S1-Q11（答 D）— 展開查看詳解
下列哪一種資料類型不屬於非結構化資料？
(A) X 光醫學影像
(B) 監控錄影畫面
(C) 客服電話錄音
(D) 組織內部的關係型資料庫記錄 ✓
詳解：A、B、C 都是影像 / 影音檔，沒有固定欄位 → 非結構化。D「關係型資料庫」就是 Excel 那種行列資料 → 結構化。所以答案選 D（本題問哪個「不屬於」非結構化）。
樣題 S1-Q12（答 B）— 展開查看詳解
下列資料型態，何者最常用來儲存員工年齡、員工年資、貨品銷售量等資料？
(A) 文字型（Text）
(B) 數值型（Numeric） ✓
(C) 日期型（Date）
(D) 布林型（Boolean）
詳解：年齡、年資、銷售量都是「可以加減乘除的數字」→ 數值型。注意這題在問型態（資料庫的欄位型態），不是在問結構分類。
(03)

2.2 資料清洗：把髒資料變乾淨

資料清洗是「讓 AI 吃得下去」的關鍵前置步驟。以下是最常考的五種資料品質問題：

問題白話處理方式
遺缺值（Missing Value）該有資料的格子是空的填補（平均 / 中位 / 眾數 / 迴歸預測）或直接刪除
重複值（Duplicate Value）同一筆資料出現兩次以上透過主鍵（如客戶 ID）識別後刪除
錯誤值（Error Value）年齡寫負數、地址有亂碼用規則檢查並修正
離群值（Outlier）數值與其他資料差很遠要視情境 — 可能是錯誤，也可能是真的高價值案例
冗餘資料（Redundant Data）同一個資訊在多個欄位重複（不是整筆重複）設計層面解決，合併欄位或正規化資料表
⚠️ 必辨：冗餘資料 vs 重複資料（S1-Q34 考點）
重複資料（Duplicate）
是什麼整筆紀錄出現兩次（同一客戶、同一日期、同一金額重複）
處理找到重複的列，刪掉其中一筆
記憶點「列重複」
冗餘資料（Redundant）
是什麼同一個東西被存在多個欄位（例如 score1、score2、score3 都在存「成績」）
處理合併欄位、重新設計資料表
記憶點「欄重複」
看到「多個欄位儲存相同資訊」→ 選冗餘（Redundant）
樣題 S1-Q34（答 B）— 展開查看詳解
某份資料中出現多個欄位（如 score1、score2、score3）儲存相同的成績資訊，造成資料結構重複與使用混淆，此情形屬於下列哪一種資料品質問題？
(A) 重複資料（Duplicate Data）
(B) 冗餘資料（Redundant Data） ✓
(C) 格式錯誤資料（Malformed Data）
(D) 缺失資料（Missing Data）
詳解：題目重點在「多個欄位儲存相同資訊」，注意是「欄位重複」不是「資料列重複」。這就是冗餘（Redundant）的定義。
樣題 S1-Q15（答 C）— 離群值情境題
某 AI 團隊在分析一組連續型數據時，發現部分紀錄的數值明顯高於其他資料點。若專案目標是識別高價值客戶的行為模式，下列哪一種處理方式最為合適？
(A) 立即刪除離群值，以避免模型訓練時出現偏差
(B) 視為錯誤值並全部替換為平均值
(C) 保留離群值並標註為高價值異常點，納入後續模型訓練考量 ✓
(D) 將離群值全數轉換為中位數，避免影響平均計算
詳解：題目提示「目標是識別高價值客戶」— 這意味著「明顯高於其他人」的資料點正是我們要找的對象，不能刪。核心原則：業務目標決定處理方式，不是「看到離群值就丟」。
(04)

2.3 ETL 流程：把資料從 A 倉搬到 B 倉

名詞定義
ETL（Extract, Transform, Load）：萃取、轉換、載入。白話 — 把多個地方的資料抓下來、整理一致、放到中央資料庫的標準三步驟流程。
步驟中文做什麼想像場景
Extract萃取從來源（CRM / ERP / Excel / API）把原始資料抓下來從各分店倉庫把貨運出來
Transform轉換統一格式、清洗、去重、欄位對應、單位換算在中央倉重新貼標、整理、合併
Load載入把整理好的資料寫入目標資料庫 / 資料倉儲把整理好的貨上架到中央倉
💡
記法：「同一個東西兩種寫法」這類問題，永遠在 T（轉換）。
樣題 S1-Q18（答 A）— 必考題，展開查看詳解
公司在整合資料時，發現同一客戶的姓名在不同系統中拼寫不一致（例如「陳大文」與「陳大文先生」），導致資料無法正確對應。請問此類資料品質問題應該在 ETL 哪一個流程步驟中進行處理？
(A) 資料轉換（Data Transformation） ✓
(B) 資料擷取（Data Extraction）
(C) 資料載入（Data Loading）
(D) 型態轉換（Type Conversion）
詳解：(B) 擷取只負責把資料搬出來，不管內容對不對。(C) 載入是最後一步，資料應已整理好。(D) 型態轉換是 Transform 的子項，不是 ETL 標準步驟名。(A) 轉換處理「資料一致性」「格式統一」正確答案。

ETL 的「轉換」階段，常會做兩種數值縮放，這兩種方法都屬於「正規化」的範疇：

方法做什麼比喻
Min-Max Scaling把所有數值縮到 [0, 1] 區間把全班身高重新打分，最矮的變 0、最高的變 1，其他依比例插中間
Z-score Standardization讓平均數變 0、標準差變 1每個人比平均高幾個「標準差單位」

2.4 集中趨勢與分散程度

集中趨勢回答的是「這群數字大概在哪裡？」；分散程度回答的是「這群數字散得多開？」兩種概念在考試中常常一起考。

名稱中文白話定義適合用在不適合用在
Mean平均數全部加起來除以個數對稱分布資料受極端值影響大（如收入、房價）
Median中位數排好之後最中間那一個偏態分布、有極端值類別資料
Mode眾數出現最多次的那個類別資料、單一暢銷品分析連續型資料分散時
💡
記法：「有極端值就看中位數」、「類別資料只能看眾數」。

分散程度的常見指標如下：

名稱白話特點
全距（Range）最大值 - 最小值太粗，被極端值影響
四分位距（IQR）Q3 - Q1（資料中間 50% 範圍）抗極端值，常用來找離群值
平均差（Mean Deviation）所有觀察值與平均數絕對差值的平均⚠️ 注意是「絕對差值」
標準差（SD）衡量「平均偏離平均數多遠」最常用、品管核心指標
變異數（Variance）標準差的平方數學上方便運算
⚠️ 勘誤注意：平均差正確定義是「所有觀察值與平均數之間絕對差值的平均」。「絕對」+ 「平均」兩個關鍵字一定要記得，原版學習指引曾少寫「絕對差值」。
樣題 S1-Q16（答 D）— 展開查看詳解
在檢視資料品質時，可參考資料的變異程度及資料的集中趨勢。下列何者不屬於資料集中趨勢衡量的方法？
(A) 平均數（Mean）
(B) 中位數（Median）
(C) 眾數（Mode）
(D) 標準差（Standard Deviation） ✓
詳解：A、B、C 都是「在哪裡」（集中趨勢）。D 標準差是「散得多開」（分散程度），不是集中趨勢。
樣題 S1-Q14（答 B）— 品管標準差題
在品質管理中，若一產品的生產過程中標準差顯著偏大，通常意味著什麼？
(A) 資料點高度集中，產品質量穩定
(B) 生產過程波動大，產品品質不穩定 ✓
(C) 資料無法反映產品實際狀況
(D) 中位數數值高，品質良率較高
詳解：標準差 = 分散程度。標準差大 → 產品的尺寸 / 重量 / 良率忽高忽低 → 製程不穩定。廠長在意的不是「平均做得好不好」，而是「穩不穩」。六標準差（Six Sigma）品管系統就以標準差為核心指標。
(07)

2.5 異常檢測：找出不對勁的資料

名詞定義
Z-score：白話 — 「這個數值離平均幾個標準差？」計算方式：Z = (數值 - 平均) / 標準差。不用記公式，記住用途即可。

判斷規則（業界常用）：

  • |Z| > 3 → 視為極端離群值（出現機率 < 0.3%）
  • |Z| > 2 → 視為輕微離群值（出現機率 < 5%）
  • |Z| ≤ 2 → 視為「正常範圍」
樣題 S1-Q10（答 A）— 必考題，展開查看詳解
在公司財務資料中，發現某筆支出金額達 1,000,000 元，若希望合理判斷該筆資料是否為異常值，下列哪一種處理方式最為合適？
(A) 以 Z-score 方法量化異常程度，判斷是否為極端值 ✓
(B) 以資料中的眾數作為參考基準，判斷其是否異常
(C) 先透過主成分分析（PCA）降低維度，再進行異常值判定
(D) 直接將該筆金額替換為資料中位數，以降低其對分析結果的影響
詳解：(A) Z-score 是量化異常程度的標準工具。(B) 眾數適合類別資料，不是用來判斷數值極端性。(C) PCA 是降維技術，不是異常檢測。(D) 直接替換 = 假設它是錯的，但題目要的是「判斷是否異常」，順序顛倒了。
📝
IQR 法補充：另一種找離群值的方式是四分位距（IQR）。任何小於 Q1 - 1.5 × IQR 或大於 Q3 + 1.5 × IQR 的值視為離群值。IQR 法不假設資料是常態分布，比 Z-score 適用範圍更廣。
初級考試只需記得：「Z-score 找離群值，最常用」。
(08)

2.6 大數據 5V 特性 + PCA 降維

名詞定義
大數據（Big Data）：量太多、變太快、種類太雜，傳統資料庫處理不來的資料。
V中文白話例子
Volume量資料量超大每天產生 PB（10 億 MB）級資料
Velocity速變化速度快股票每秒 tick、IoT 感測器毫秒上傳
Variety多樣結構化 + 半 + 非結構化都有文字 + 圖片 + 影音 + 感測
Veracity真實性資料品質參差社群留言有真有假
Value價值從一堆雜訊中提煉商業價值從點擊資料找出潛在客戶
⚠️
iPAS 學習指引內主要強調 4V（量、速、多樣、價值），出題時兩種都可能出現。早期文獻只講前 3V 或 4V，目前主流講 5V（含 Veracity 與 Value）。
樣題 S1-Q8（答 D）— 展開查看詳解
下列何者非大數據時代資料的特性？
(A) 資料量大
(B) 資料變動速度快
(C) 資料多樣性
(D) 資料存儲位置固定 ✓
詳解：A = Volume、B = Velocity、C = Variety，都是 5V 之一。D「儲存位置固定」反而違背大數據特性 — 大數據通常分散在雲端、地端、各種來源，沒辦法集中固定存放。
名詞定義
PCA（Principal Component Analysis，主成分分析）：「找出資料中最重要的幾個方向，用這幾個新欄位代替原本一大堆欄位」。屬於非監督式學習，是降維最常考的方法。

比喻：原本要描述一個人需要 100 個欄位（身高、體重、年齡、薪水⋯），PCA 會幫你找出「最能區分人與人不同」的 3–5 個綜合指標，用這幾個新指標就能保留 90% 資訊，剩下 95 個欄位可以丟。

樣題 S1-Q27（答 A）— 必考題，展開查看詳解
一位資料分析師希望減少輸入特徵的維度，以提升模型運算效率，並觀察變數間潛在的整體結構關係。若欲保留最大資訊量、同時減少特徵數量，下列哪一項方法最適合？
(A) 套用主成分分析（PCA）以擷取主要變異方向並轉換新變數 ✓
(B) 利用離散化方法將連續變數轉為分類型欄位
(C) 使用標準化方法將所有特徵縮放至相同數值區間
(D) 以 ETL 技術移除空值欄位並改儲存為 JSON 格式
詳解：題目兩個關鍵詞「減少維度」+「保留最大資訊量」→ PCA 的標準定義。(B) 離散化把連續變數變類別，不是降維。(C) 標準化縮到同尺度，不改欄位數。(D) 格式轉換，跟降維無關。
記法：看到「降維 + 保留資訊」就選 PCA。

2.8 資料隱私與安全：AI 不是拿到資料就能用

對應考點
科目一細項 2.3，影響 S2-Q9（個資跨境）/ S2-Q18（Zero-Retention）/ S2-Q26（權限合規）/ S2-Q31（Zero Trust）4 題迂迴考點。

拿到資料不代表可以用。使用資料前，必須先回答三個問題：

  • 這份資料能不能識別到人？
  • 這份資料能不能合法用在 AI 訓練 / 分析？
  • 分享出去後，有沒有辦法降低還原個資的風險？

觀念補充（延伸理解）

這一節是什麼
以下為延伸觀念補充，涵蓋考試常見判斷邏輯與延伸概念。非考試必背範圍，但有助於理解選項背後的設計思路。
資料蒐集佔整個 AI 專案 4–6 成的力
訓練模型時最花時間不是模型本身，是「資料蒐集 + 清洗」這兩件事。資料蒐集要 concern 兩個點：(1) 偏差——例：以前西方人臉孔資料集多、亞洲臉孔少，做出來的情緒辨識在亞洲就特別爛；(2) 代表性——例：銀行升職資料女性少，AI 就會把女性都排除，所以要刻意把女性樣本放多一點、男性壓低一點，達成「資料平衡」。問卷發放也是同樣概念——問男生「這個名牌包吸不吸引你」是無效資料。
標準化的隱藏陷阱：性別代碼
登記性別時若把男生 = 1、女生 = 2，沒宣告型別電腦會視為數值，平均出來變成 1.6、1.7——但「沒有人性別叫 1.6 1.7」。資料的標準化格式不只有日期到秒 / 到分這種格式問題，類別型欄位也要避免被誤算成數值。重複值用 UUID（不可能重複的隨機 ID）防呆；異常值（例：體重 800 公斤）通常直接移除。
個資三層分級實戰
(1) 直接識別 = 姓名 / 身份證 / 手機 / Email / 地址；(2) 間接識別 = 年齡 / 性別 / 職業 / 郵遞區——但「姓名 + 職業」在台灣多數情況就足以鎖定一個人；(3) 敏感個資 = 醫療紀錄、宗教 / 政治傾向、生物特徵。台灣健保資料庫特別龐大（看診頻繁），所以醫療紀錄反向推論身份的風險特別高。考試只要看到「醫療 + 法規 + 釋出資料」幾乎都是綁差分隱私 + 個資保護。
去識別化 vs 假名化 vs 加密 vs 雜湊四層的可逆性
這四個容易混。去識別化（Data De-identification，又稱脫敏） = 不可逆刪除或遮蔽（王曉明 → 使用者 A）；假名化 = 用代碼，有金鑰可以還原；加密 = 用密碼學算法，「複雜到刪掉一個字都解不出來」；雜湊 = 單向轉換、不可還原但可比對。最容易混的是去識別化 vs 假名化——去識別化真的把可識別資訊砍掉、假名化只是換代號。
零信任架構（ZTA）三件事
(1) 身份認證——每次都當外人驗、可能還要 MFA / 二次驗證；(2) 裝置與連線安全——連線裝置本身要安全；(3) 存取權限最小化——學生只能看自己選的課、教授只能看自己開的課。最完整實踐的是金融機構（網銀指紋 / Face ID + 身分證 + 兩種密碼）。零信任就是把「資料最小化」這條原則套到「使用者權限」上。
聯邦式學習的「家庭代工」比喻
每個人在自己家做完一部分零件（本地訓練），最後上繳總工廠拼裝（模型結合）。原始資料「不動」、只動「訓練好的結果」。常見場景：醫院（罕見病多家共訓）、銀行（跨行反洗錢）、跨國資安。314 強調「家庭代工」的合作面，408 強調「梯度 / 權重變化才是傳的東西，加密原始資料也不傳」，415 則從隱私法規角度切入。
Lakehouse 為何取代傳統 ETL
現代資料倉儲傾向「先丟進來再算」（ELT 而不是 ETL），背後的原因有兩個：第一，企業把算力集中到中央電算中心，方便管理，所以開發時只要考量自己的機制跑不跑得動。第二，移動端普及，但移動端的算力一定比本機端或固定桌機差，把運算集中到雲端可以拉齊使用者體驗。這也是為什麼消費模式從買斷硬體變成服務型訂閱的底層原因。
差分隱私 ε 的方向感
差分隱私的 ε（差分預算）有三個必背：(1) ε 越小 → 注入雜訊越多 → 保護程度越高，但資料統計可用性會下降。(2) ε 沒有負數。(3) ε 一般落在 0.1 到 10 之間，不會到 100，且法規沒有強制規定 ε 的具體數值。這是醫療業釋出去識別化資料時最容易考的選項陷阱。
Z-score 公式 + 為何只記這條
老師親口說：「Z-score 是我覺得唯一你們需要記的數學公式」。常態分布 68/95/99% 法則 → Z = (目標值 − 平均) / 標準差。Z 絕對值大於 3 → 非常異常（例：人體溫平均 37.6 度、標準差 1 度，量到 40 度直接觸發警告）。台灣大家熟悉的 PR 值 / 趴數其實同概念。Z-score 在資料清理階段判離群值最常用，比相形圖（box plot）更精準量化。
Min-Max 正規化 vs 標準化（Z-score）的差異
正規化（Min-Max） = 等比例縮放到 0–1 區間，看分佈長相用；標準化（Z-score） = 平均值 0、標準差 1，越接近常態分布越好用，可以直接看誰是離峰值 / 平均 / 中位。穩健縮放（robust scaling） 較少考但會用中位數 + IQR 處理離群值。圖片要降運算量也常做「二階化」——把 RGB 三個數字壓成黑白一個數字（節省約 3 倍算力），因為 CNN 主要抓輪廓，黑白與否影響不大。
向量是上百上千維、不只 3D
高中教三維向量是給人看；電腦做 AI 用的向量上百上千維。貓和狗的向量數值我們不能用「哺乳 / 四腳 / 雜食」這些詞去解讀，電腦只是用距離算相似度。應用：文字向量化（One-hot Encoding 把紅綠藍轉 100/010/001）、影像辨識特徵點、推薦系統、語音音樂分析、SEO（關鍵字向量比對）。
KL 散度為何能偵測資料漂移
資料漂移比的是「機率分布的形狀差異」，不是單點距離。KL 散度與 Wasserstein 距離都是在量化「兩個分布的形狀差多少」。實務上把線上輸入的分布跟訓練時的分布做比較，超過設定的閾值就觸發資料漂移警告。
GDPR 四大資料權利的口語對照
考試時看到「徹底刪除（含備份）」就是被遺忘權；「打包資料到另一家公司」是可攜權，最常見場景是保險業、電信業；「暫停使用但不刪除」是限制處理權；「拒絕用於特定用途」是拒絕權。例子：你把電話號碼給學校，可以拒絕學校把號碼給職涯探訪中心。「歐盟 + 刪除」這組關鍵字幾乎一定綁被遺忘權。被遺忘權的日常實踐：有些信用卡要收年費、有些戶頭要持續交流，使用者長期不用就主動要求關掉刪除。
Human in the loop vs Human on the loop
歐盟對高風險 AI 規範的「人工介入審查權」是 human in the loop——人類必須在關鍵執行節點存在、不能繞過，AI 不能跳過你做事。另一種 human on the loop 是人不用跟著做，但要能看到全局，遇到警報才介入。考試題目給的場景如果寫到「使用者法律權益重大影響」，方向是 in the loop。實務例：電商 AI 助理收到「下架所有紅洋裝」這種大規模操作，正確做法是先產報表給人類審核（in the loop）；自駕車則是 on the loop——不能每動作都問人，否則自駕意義就沒了。
爬蟲 vs robots.txt 的對應
爬蟲只看網頁的「程式碼結構」，每次抓固定結構的資料。網頁右邊那些 HTML 程式碼，使用者看不到的部分，就是爬蟲在讀的。要禁止 AI 爬蟲，業界標準是在網站根目錄放 robots.txt 並宣告禁止特定 AI bot。網頁上放紅色警告圖只給人看、把文字轉 PDF 圖片爬蟲還是抓得到，這兩個都不是真的能擋的方法。
資料譜系（data lineage）的「碳足跡」比喻
要追蹤「從哪一筆資料 → 哪些步驟 → 最後流到哪」這種「資料碳足跡」就是資料譜系，不是 ER 圖、不是平均值 / 標準差統計、不是視覺化儀表板。核心兩記：「資料變成什麼」+「為什麼 / 怎麼變的」。商業效益：(1) 降低合規風險（金融業要證明信用評分模型沒歧視，全靠這套留檔）；(2) 加速 debug；(3) 跨團隊協作——工程師其實「不喜歡看別人寫的、寧可重寫一個新的」，所以記錄越完整越好；(4) 建立顧客信任。
