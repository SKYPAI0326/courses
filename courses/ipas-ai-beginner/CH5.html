No Code / Low Code 概念

「我想做一個內部小工具，到底要學程式、找 IT、還是用拖拉平台？」這章給你決策框架，幫你在 NC、LC、傳統程式三者之間做出正確選擇——也讓你知道 Shadow IT 是什麼、為什麼公司怕它。

學習成果
能說清楚 No Code 與 Low Code 的核心差異（誰能用、適合做什麼）
判斷一個情境該用 NC、LC，還是找工程師寫程式
解釋 Shadow IT 的成因、風險與對策
能正確回答 7 題 iPAS 官方樣題（S2-Q1、Q3、Q4、Q5、Q7、Q10、Q12）
(01)

三種開發方式——用蓋房子類比

為什麼這很重要
這章學完你能回答：Make.com 是 No Code 還是 Low Code？公司 IT 太忙，我可以自己做一個請假申請表單嗎？為什麼老闆說「法律判決書」不要用 NC/LC 做？
No Code（無程式碼）
組合屋
整套現成拼裝，完全不用寫程式。
代表：Wix、Notion、Zapier、Bubble
學習成本：低（半天上手）
適合：非技術人員
Low Code（低程式碼）
半成品裝修
主要拖拉，遇複雜邏輯寫少量程式補強。
代表：n8n、Power Apps、OutSystems
學習成本：中（幾週上手）
適合：有點技術背景的人
Code（傳統程式）
從頭蓋
從畫設計圖、打地基開始蓋。全部用程式，自由度最高。
需要：專業工程師
學習成本：高（數月以上）
🪧
三句話記法：
No Code：完全不寫程式，全靠拖拉組裝。
Low Code：主要拖拉，遇複雜邏輯寫一點程式補強。
Code：全部用程式，自由度最高、開發最久。

S2-Q1 連動：NC/LC 主要特色是什麼？正解：(B) 運用模板快速建立應用程式——NC/LC 的精神就是「拖拉模板」。(A) 需要大量程式碼與精神完全相反；(C) 描述顛倒（NC 正是給非開發者用）；(D) 過度限縮（也能做動態 App、流程自動化）。
(02)

NC vs LC——核心對比與選型判斷

6 維度對比表（必背）

維度No Code（NC）Low Code（LC）
是否需要寫程式完全不需要少量
目標使用者非技術人員（業務、行銷、HR）有技術背景者（IT、資料分析師）
適用情境快速原型、簡單工作流程、內部小工具複雜邏輯、企業整合、需擴展性的應用
學習成本低中
客製彈性受平台模板限制可寫程式突破平台限制
代表工具Zapier、Wix、Notion、Bubblen8n、Power Apps、OutSystems、Mendix

如何選：3 個關鍵提問

問題 1
使用者有技術背景嗎？
沒有 → No Code（讓他們也能上手）
有一些 → Low Code 都可以
問題 2
平台模板能做到嗎？
是 → No Code（不用多此一舉）
否，需要客製邏輯 → Low Code
問題 3
未來規模會長大嗎？
不會 → No Code 夠用
會，要持續長大 → Low Code 留後路

NC vs LC vs Code 三層決策樹

  1. 我完全不會寫程式→No Code
  2. 我會一點程式，需求單純→No Code（CP 值更高）
  3. 我會一點程式，需求有點複雜→Low Code
  4. 我會一點程式，需求超複雜→Low Code 試做 → 不夠再升級到 Code
  5. 全自訂、效能要求極高→找工程師寫 Code
📌
S2-Q3 連動：哪種情況選 Low Code 比 No Code 更適合？正解：(C) 需要較複雜的業務邏輯並使用自訂整合功能——LC 比 NC 多了「能寫少量程式」的自由度。(A)(B)(D) 都是 No Code 的典型場景。

S2-Q5 連動：NC vs LC 描述何者較正確？正解：(C) Low Code 平台更適合開發靈活且可擴展的解決方案。(A) 錯，NC ≠ LC；(B) 錯，LC 仍需少量程式知識；(D) 錯，NC 受平台模板限制，無法無限客製。
(03)

不適用情境與Shadow IT 風險

NC/LC + 生成式 AI「不適合」做什麼

📌
S2-Q4 連動：NC/LC + 生成式 AI 最不適合的是哪一項？正解：(D) 自動化生成法律判決。(A) 自動生成程式碼 ✓；(B) 自動化行銷文案 ✓；(C) 快速開發個人化 App ✓。

為什麼法律判決不能讓 AI 自動產出？

司法權專屬性
判決涉及國家公權力，不能由民間 AI 系統決定
責任歸屬問題
判決出錯誰負責？AI 不能負刑事責任
歐盟 AI 法案
被歸類為「不可接受風險」或「高風險」AI 應用，明確禁止
倫理底線
人權、自由、財產的判定必須有人類最終把關
🪧
白話記法：「能不能換成 AI 自動做」，看這件事「出錯了能不能賠得起」。寫廣告文案出錯，賠個道歉就好；判決出錯，可能毀掉一個人的人生——所以絕不能讓 AI 自動下判決。

看到「法律 / 司法 / 醫療核心診斷 / 性命攸關」這類選項，幾乎都是「不適合 AI 自動化」的正解。

Low Code 特有風險：Shadow IT 影子 IT ⭐⭐⭐

Shadow IT（影子 IT） 考試必懂

定義：員工在 IT 部門不知情的情況下，自己偷偷用 NC/LC 工具做出小工具或應用，導致公司資料管理失控的現象。

為什麼 NC/LC 特別容易產生？正因為 NC/LC 太好上手——業務部、行銷部、HR 都可以自己拉一個系統出來——IT 部門根本不知道公司裡有多少這種「私人系統」。

Shadow IT 帶來的 5 大風險

01 資料外洩：敏感資料流到沒被審核的第三方平台。例：業務部偷偷用 Zapier 把客戶名單同步到自己的 Google Sheet → IT 不知道客戶資料外流。
02 重複建置 / 資源浪費：5 個部門各自做了類似的小工具，誰也不知道彼此存在。
03 資料不一致：每個部門有自己的版本，全公司資料對不起來。
04 法規違反：個資法、GDPR 要求資料處理要可追溯，Shadow IT 違反這個原則。例：行銷部用 Notion 做對外填表頁，沒過資安審核 → 資料蒐集違反個資法。
05 離職交接斷層：自製的工具沒文件、沒交接，員工一走就停擺。

怎麼避免 Shadow IT？

建立內部工具清單
公司明確公告哪些 NC/LC 平台允許使用
市民開發者治理
可以做，但要登記、要審核（設立「市民開發者」認證機制）
資安培訓
教員工「什麼資料不能丟雲端」
定期盤點
IT 主動巡查公司用了哪些 NC/LC 服務
📌
S2-Q7 連動：使用 Low Code 平台時企業應特別留意哪一項潛在風險？正解：(D) 可能有未經 IT 部門管理的應用程式擴散——這就是 Shadow IT 的標準描述。

(A) 資料外洩風險不是 LC 獨有；(B) LC 反而比較容易擴展；(C) LC 通常是降低成本。
⚠️ 這題容易誤選 (A)——記住題目問的是「LC 特別需要留意的風險」，要選 LC 獨有的（影子 IT）。
(04)

NC/LC 在 ETL 中的角色

🪧
ETL 白話複習（CH2 已詳介）：
ETL = Extract（萃取）+ Transform（轉換）+ Load（載入）
白話：把資料從「這裡撈出來」→「整理乾淨」→「送到那裡」。
例：把 ERP 的銷售資料 → 清理日期格式、合併同名客戶 → 倒進 BI 報表系統。
📌
S2-Q12 連動：NC/LC 平台在企業級 ETL 中的主要角色？正解：(B) 取代部分傳統 ETL 解決方案，但可能無法處理過於客製化的邏輯——這是最平衡的描述。(A) 縮小了角色；(C) 過度否定；(D) 完全否定。

答「NC/LC 能做什麼」這類題的訣竅：選「最平衡、有條件」的敘述（「能做⋯但不能⋯」），避開全肯定或全否定的極端選項。

ETL 工具光譜

層級 工具 使用者
傳統 ETL Talend、Informatica、Apache Airflow IT 部門（需懂程式）
Low Code ETL n8n、Microsoft Power Automate、AWS Glue Studio 半技術背景可用
No Code 自動化平台 Zapier、Make.com、IFTTT 業務人員可用

記法：簡單流程 → NC 工具足夠；複雜處理 → LC 加減湊；超大規模或精細邏輯 → 還是要傳統 ETL 工程

6 大應用領域

醫療保健
自動化病歷生成、預約系統、門診流程自動化
製造業
產線監控儀表板、品質回報自動化、SOP 數位化

虛擬客服機器人、自動回應、客訴分類與分派

AI 民主化（AI Democratization）

AI 民主化：把 AI 技術從「少數工程師專利」擴展到「所有人都能用、能做」的趨勢。以前要做 AI 工具，要會寫 Python、會調模型；現在用 NC/LC 平台 + ChatGPT API，行銷專員也能組裝出一個 AI 客服機器人。

市民開發者（Citizen Developer）：不是工程師、但能用 NC/LC 工具自動化日常工作的職場人士。公司裡會用 Zapier 把例行工作自動化的業務、用 Make.com 接 AI 的行銷專員——他們就是「市民開發者」。

🚧
AI 民主化的「反例題」陷阱：
NC/LC 推動 AI 民主化會遇到的挑戰：模型可解釋性不足、資料隱私風險、平台依賴度過高、技能落差。

不是挑戰的選項：「提供昂貴的開發工具」——NC/LC 本來就是為了降低成本而生的。看到「昂貴」「只給專家用」這種跟「民主化精神」相反的選項，先排除。

(07)

觀念補充（延伸理解）

🎙️
以下為延伸觀念補充，涵蓋考試常見判斷邏輯與延伸概念。非考試必背範圍，但有助於理解選項背後的設計思路。
▶ NC / LC 的本質是「平台」這兩個字

不管 No-code（完全不寫程式）還是 Low-code（少量程式碼），它們都是「平台」——沒有平台這東西多數時候是不能動的，這是 NC/LC 的根本特性，也是它最大缺點的根源。

Gemini 內建 Canvas 功能搭配 vibe coding 可以做到中型網站、官網、簡單遊戲的程度，但它跑在 Google 的平台上、Google 一變你的東西就要跟著變。

▶ NC / LC 三大缺點——流量費用 / Token 計費 / 平台依賴

(1) 流量費用：N8N 內容農場真實案例——學員測試期間覺得 OK，試用期一過開出來 6 萬塊台幣 / 月的帳單，「我快吐了」。

(2) Token 計費：現在多數 AI 工具按 Token 計費，指令越複雜消耗越大。

(3) 平台依賴：類比早期 WordPress 一更新就把熟手的網站全打爛——平台一變，你的開發模式就要跟著變。

實務建議：「內部使用完全 OK，但要對外商用、給外部顧客的話不建議」。

▶ 平台鎖定（Vendor Lock-in）三層

(1) 技術鎖定：平台 A 的拉法和 B 不一樣，搬遷要重新學。

(2) 資料鎖定：格式 / 儲存方式不相容，搬遷要重新設計資料庫（搬家就要全部人跟著重做一次）。

(3) 生態鎖定：外掛功能（購物車、定期扣款、刷卡）每家平台接口不同。

比喻：「就像 110V 家電全部要換 220V 插頭」，搬家成本是實際存在的。

▶ API = 餐廳服務生比喻

客人（client）目標是「得到服務」，廚房（server）目標是「提供服務」，服務生就是 API——點餐單從客人傳到廚房、菜從廚房傳到客人。

API 不做運算、只做資料傳遞，「就像插座，電是電，插座是插座，方便你把家電插上去」。

「串接 ChatGPT API」的真實意義：你網頁某個按鈕按下去，把資料用 API 丟給 ChatGPT，ChatGPT 處理完再用 API 回傳到你網頁。

最常見的車牌辨識 API 流程：機子辨識車號 → API 把車號丟給停車場資料庫查詢繳費金額 → API 把金額傳回給機子顯示。

▶ 容器化 = 程式 + 環境一起打包

寫個翻譯程式，丟給人家不能只丟程式碼，還要丟需要的語言文本（例：寫中翻埃及文要連埃及文文本一起丟）。容器化就是把這些必要環境綁在一起，能在手機 / 平板 / 筆電到處跑。

優點：跨平台運行；缺點：需要規劃打包邏輯。容器化算是 NC/LC 概念的延伸——讓「會寫程式但不會佈署環境」的開發者降門檻。

▶ NC / LC 客製化軟體真實案例——餐飲業 POS

餐飲業 POS 系統（iChef / DoDo / 快一點）每家店打卡和出缺席算法都不一樣，沒一個通用系統能完美 fit——這時候 NC/LC 就有優勢，可以完全按自己的規則設計表單請假 / 自動報表 / 公司內部審批這類「內部用」的小工具。

優點：開發效率高 + 圖像化邏輯讓使用者自己也看得懂自己的業務流。

▶ NC / LC 的資安提醒——開源套件集體風險

開源框架 / 套件的資安風險——XAMPP 真實案例：早期很多網站用 XAMPP（Apache + PHP + MySQL 一鍵架站）建置，有人專挑 XAMPP 漏洞攻擊，結果全部使用 XAMPP 的網站連帶遭殃。

意思是：用熱門開源工具便宜方便，但被同樣的零日漏洞集體打趴的風險比較高。NC/LC 平台也有同樣風險——平台被攻破，所有平台用戶都會受影響。

▶ AI 民主化的「爽點 vs 痛點」推廣邏輯

技術推廣四步：雲端運算 + 開源生態 + 介面簡化（NC/LC、vibe coding）+ 教育推廣。最有效的傳播方式是讓人感受爽點——蝦皮、Foodpanda 一開始免費讓你狂用，習慣後加運費還是繼續用；痛點就是「沒這個會麻煩」的感受。

中華電信在大學辦樂齡學堂教長輩 P 圖、生圖也是這條路線——降低資訊落差、讓 NC/LC + AI 真的擴展到弱勢族群。

▶ TensorFlow vs PyTorch 與 NC/LC 的降門檻光譜

開源框架 = 提供免費的「預訓練模型 + 可重用程式碼」。TensorFlow 是 Google 開發、PyTorch 比 TF 新一點且主要用 Python（Python 是現在 AI 模型訓練的主流語言）。

降門檻光譜由低到高：純程式 → 框架（TF / PyTorch）→ Low Code → No Code（圖像化、Scratch 風）→ vibe coding（純自然語言）。同樣是「降門檻」這條主軸的不同點。
