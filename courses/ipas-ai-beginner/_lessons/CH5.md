# CH5 No Code / Low Code 概念（2 小時）

> 本章是「**工具定位章**」。教你怎麼回答這個問題：「我想做一個內部小工具，到底要學程式、找 IT、還是用拖拉平台？」**0 基礎學員特別重要**——因為 No Code / Low Code 平台正是讓「不會寫程式但有想法」的職場人士，能自己做出工具的那條路徑。

---

## 0. 這章學完你能回答的日常問題

- 我聽過 Make.com、n8n、Zapier、Notion，這些到底叫 No Code 還是 Low Code？
- 公司 IT 太忙，我可以自己用 No Code 做一個請假申請表單嗎？這樣安全嗎？
- No Code 跟 Low Code 差在哪？什麼時候該選哪一個？
- 為什麼老闆說「自動寫法律判決書」這種事不要用 NC/LC 做？
- 「影子 IT（Shadow IT）」是什麼意思？為什麼公司怕這個？

---

## 1. 學習目標（Learning Objectives）

研習結束後，學員應能：

1. **分得清** No Code 與 Low Code 的核心差異（誰能用、適合做什麼）。
2. **判得準** 一個情境該用 NC、LC，還是要請工程師寫真正的程式碼。
3. **避得開** NC/LC 不適用的情境（法律判決、需要無限客製、高度安全敏感）。
4. **認得出** 「Shadow IT 影子 IT」這個風險的成因與對策。
5. **講得通** NC/LC 在 ETL、生成式 AI 整合上的角色（取代部分傳統方案，但複雜邏輯仍需傳統工具）。
6. **答得對** 7 題官方樣題（S2-Q1、Q3、Q4、Q5、Q7、Q10、Q12）。

## 2. 0 基礎入口：用「蓋房子」類比 NC / LC / Code

### 2.1 三種開發方式並排對比

| 方式 | 蓋房子比喻 | 網頁比喻 | 開發者 | 學習成本 |
|---|---|---|---|---|
| **No Code（無程式碼）** | 買整套組合屋，現場拼裝 | **Wix / Notion 網站**：拖拉版面就有網站 | **完全不會寫程式的人** | 低（半天上手） |
| **Low Code（低程式碼）** | 買半成品結構件，部分自訂裝修 | **WordPress + 少量自訂 CSS / 程式片段** | **有點技術背景的人** | 中（幾週上手） |
| **Code（傳統程式開發）** | 從畫設計圖、打地基開始蓋 | 從零寫 HTML/CSS/JS/後端程式 | **專業工程師** | 高（數月以上） |

### 2.2 三者最直觀的差別

> 🪧 **三句話記法**：
> - **No Code**：**完全不寫程式**，全靠拖拉組裝。
> - **Low Code**：**主要拖拉**，遇到複雜邏輯**寫一點程式**補強。
> - **Code**：**全部用程式**，自由度最高、開發最久。

### 2.3 你身邊已經在用的 NC / LC 工具

| 工具類型 | No Code 代表 | Low Code 代表 |
|---|---|---|
| **流程自動化** | Zapier、Make.com（簡單流程） | n8n、Microsoft Power Automate |
| **網頁建置** | Wix、Squarespace、Notion 公開頁 | WordPress（可加程式）、Webflow |
| **App 開發** | Glide、Bubble | OutSystems、Mendix、Microsoft PowerApps |
| **資料庫 / 表單** | Airtable、Notion 資料庫 | Google AppSheet、Microsoft Power Apps |
| **AI / 自動化整合** | Make.com（AI 模組）、Coze | n8n（自寫 function）、Dify |

> 🚧 **同一個工具可能跨類**——例如 Make.com 大多時候是 No Code，但它有「Custom App / 寫 JSON」的進階功能就接近 Low Code。**不要花時間糾結每個工具的歸類，記住「能不能寫程式擴充」是分界線就好**。

## 3. No Code 詳解（25 min）

### 3.1 白話定義

> **No Code（無程式碼，無代碼）**：透過**圖形化介面（UI）+ 拖拉操作**就能開發應用，使用者**完全不需要寫程式**。

**核心特徵**：
- 預先做好的「**模板（Template）**」、「**模組（Module）**」、「**節點（Node）**」
- 操作方式：**拖、拉、連、設定參數**
- 適用對象：**非技術人員**——行銷、業務、HR、企劃、營運⋯⋯都能上手

### 3.2 No Code 適合做什麼

| 情境 | 範例 |
|---|---|
| 快速原型 | 一週內把點子做成可給老闆看的 Demo |
| 內部工具 | 自動填表、自動寄信、Slack 通知 |
| 簡單工作流程 | 收到 Google Form 填表 → 自動寄感謝信 + 加進 Notion 資料庫 |
| 部門小工具 | 業務部的客戶名單管理、客服部的 FAQ 機器人 |

### 3.3 No Code 不適合做什麼

| 情境 | 為什麼不行 |
|---|---|
| 高度客製化的計算邏輯 | 模板裡沒有，不能寫程式自己加 |
| 大量資料處理（幾百萬筆） | 平台運算資源有限，會卡或要付高額費用 |
| 嚴格資安合規（金融、醫療核心系統） | 平台是第三方服務，資料控制權有限 |
| 需要無限自由的客製介面 | UI 受限於模板提供的元件 |

> 📌 **S2-Q5 連動 (D) 反例**：考題的錯誤選項說「No Code 平台可以無限客製化」，這是**錯**的——NC 永遠受平台模板限制，這是它的本質。

## 4. Low Code 詳解（25 min）

### 4.1 白話定義

> **Low Code（低程式碼，低代碼）**：以視覺化介面為主，**搭配少量程式碼**擴充功能，讓開發者能在拖拉基礎上實作複雜邏輯。

**核心特徵**：
- 主體還是視覺化拖拉
- **遇到平台不直接支援的邏輯時，可以寫程式片段補強**（JavaScript、Python、SQL 等）
- 適用對象：**有基本技術背景的人**——IT 部門、資料分析師、有寫過 Excel VBA / 簡單腳本的人

### 4.2 Low Code 適合做什麼

| 情境 | 範例 |
|---|---|
| 複雜業務邏輯 | 多條件審批流程、跨系統資料同步 |
| 自訂整合 | 接公司既有 ERP / CRM API |
| 需要擴展性 | 預期未來規模會長大，要保留調整空間 |
| 企業級應用 | 內部審批、業務管理、客服系統 |

> 📌 **S2-Q3 連動**：考題情境「需要較複雜的業務邏輯並使用自訂整合功能」應該選 **Low Code**，因為 LC 比 NC 多了「能寫程式」這個自由度。

> 📌 **S2-Q5 連動**：考題說「Low Code 平台**更適合開發靈活且可擴展的解決方案**」是正解——LC 的賣點就是「可擴展性」。

### 4.3 Low Code 仍然會遇到的限制

- 真正高度客製、超高效能的需求，還是要全程式開發
- 某些底層細節仍受平台框架限制
- 需要工程師維護、不像 NC 完全交給業務人員

## 5. NC vs LC 核心對比表（**必背**）（15 min）

### 5.1 6 維度對比

| 維度 | No Code（NC） | Low Code（LC） |
|---|---|---|
| **是否需要寫程式** | **完全不需要** | **少量** |
| **目標使用者** | 非技術人員（業務、行銷、HR） | 有技術背景的開發者 |
| **適用情境** | 快速原型、簡單工作流程、內部小工具 | 複雜邏輯、企業整合、需擴展性的應用 |
| **學習成本** | 低 | 中 |
| **客製彈性** | 受平台模板限制 | 可寫程式突破平台限制 |
| **代表工具** | Zapier、Wix、Notion、Bubble、Glide | n8n、Power Apps、OutSystems、Mendix |

### 5.2 **如何選**：3 個提問

當你在猶豫該選 NC 還是 LC，問自己這 3 個問題：

1. **「會用我做的人有沒有技術背景？」**
   - 沒有 → No Code（讓他們也能上手）
   - 有一些 → Low Code 都可以

2. **「我要做的事是不是平台模板就能做到？」**
   - 是 → No Code（不用多此一舉）
   - 否，需要客製邏輯 → Low Code

3. **「未來規模會不會變很大？需不需要擴展？」**
   - 不會，做完就放著 → No Code 夠用
   - 會，要持續長大 → Low Code 留後路

### 5.3 NC vs LC vs Code 三層決策樹

| 你的條件 | 建議 |
|---|---|
| 完全不會寫程式 | **No Code** |
| 會一點程式，需求單純 | **No Code**（CP 值更高）|
| 會一點程式，需求有點複雜 | **Low Code** |
| 會一點程式，需求超複雜 | Low Code 試做 → 不夠再升級到 Code |
| 全自訂、效能要求極高 | 找工程師寫 **Code** |

> 📌 **S2-Q1 連動**：考題問「NC/LC 主要特色」，正解是 **(B) 運用模板快速建立應用程式**——NC/LC 共同特徵就是「拖拉模板」。

## 6. 6 大應用領域（10 min）

NC/LC 平台不限於某產業，6 大領域都有典型應用：

| 領域 | NC/LC + 生成式 AI 應用範例 |
|---|---|
| **醫療保健** | 自動化病歷生成、預約系統、門診流程自動化 |
| **製造業** | 產線監控儀表板、品質回報自動化、SOP 數位化 |
| **金融業** | 風險評估流程、客戶 onboarding 自動化、合規報表 |
| **零售業** | 個人化行銷、庫存自動補貨、客戶體驗回饋蒐集 |
| **教育** | 自動教材生成、個人化學習路徑、學員測驗系統 |
| **客戶服務** | 虛擬客服機器人、自動回應、客訴分類與分派 |

### 6.1 應用模式：NC/LC 平台 × 生成式 AI

> 🪧 NC/LC 不只是「拖拉介面」，**現在最熱門的應用是「拖拉 + 接 AI」**：
> - 在 Make.com / n8n 拖一個流程：使用者填表單 → 觸發 ChatGPT API → 回傳客製化郵件 → 寄給客戶
> - 在 Bubble / Glide 做一個 App：使用者上傳照片 → 呼叫 DALL·E API → 生成風格圖
>
> 這類應用**不需要學寫程式**，但能做出有 AI 加持的工具。**S2-Q4 反向考的就是「不適合 NC/LC + 生成式 AI 的場景」**。

## 7. 不適用情境與風險（25 min）

### 7.1 NC/LC + 生成式 AI 「**不適合**」做什麼

> 📌 **S2-Q4 連動**：考題問「NC/LC + 生成式 AI 應用最**不**適合的是哪一項」，正解是 **(D) 自動化生成法律判決**。

**為什麼法律判決不能讓 AI 自動產出？**
- **司法權專屬性**：判決涉及國家公權力，不能由民間 AI 系統決定。
- **責任歸屬問題**：判決出錯誰負責？AI 不能負刑事責任。
- **歐盟 AI 法案（EU AI Act）禁止**：被歸類為「**不可接受風險**」或「**高風險**」AI 應用。
- **倫理底線**：人權、自由、財產的判定必須有人類最終把關。

> 🪧 **白話**：「能不能換成 AI 自動做」要看這件事 **「出錯了能不能賠得起」**。寫廣告文案出錯，賠個道歉就好；判決出錯，可能毀掉一個人的人生——所以絕不能讓 AI 自動下判決。

### 7.2 Low Code 特有風險：Shadow IT 影子 IT ⭐⭐⭐

> **Shadow IT（影子 IT）**：員工在 IT 部門不知情的情況下，**自己偷偷用工具做出小工具或應用**，導致公司資料管理失控的現象。

#### 7.2.1 為什麼 NC/LC 特別容易產生 Shadow IT？

> 🪧 **正因為 NC/LC 太好上手**——業務部、行銷部、HR 都可以自己拉一個系統出來——**IT 部門根本不知道公司裡有多少這種「私人系統」**。

**典型情境**：
- 業務部偷偷用 Zapier 把客戶名單同步到自己的 Google Sheet → IT 不知道客戶資料外流
- 行銷部用 Notion 做一個對外的填表頁，沒過資安審核 → 資料蒐集違反個資法
- HR 部門用 Make.com 自動寄員工生日卡 → 把員工生日資料給了第三方平台

#### 7.2.2 Shadow IT 帶來的風險

1. **資料外洩**：敏感資料流到沒被審核的第三方平台
2. **重複建置 / 資源浪費**：5 個部門各自做了類似的小工具
3. **資料不一致**：每個部門有自己的版本，全公司資料對不起來
4. **法規違反**：個資法、GDPR 要求資料處理要可追溯，Shadow IT 違反這個原則
5. **離職交接斷層**：自製的工具沒文件、沒交接，員工一走就停擺

#### 7.2.3 怎麼避免 Shadow IT？

- **建立內部 NC/LC 平台清單**：哪些工具公司允許用
- **設「市民開發者（Citizen Developer）」治理機制**：可以做，但要登記、要審核
- **資安培訓**：教員工「什麼資料不能丟雲端」
- **定期盤點**：IT 主動巡查公司用了哪些 NC/LC 服務

> 📌 **S2-Q7 連動**：考題問「使用 Low Code 平台時企業應特別留意哪一項潛在風險」，正解是 **(D) 可能有未經 IT 部門管理的應用程式擴散**——這就是 Shadow IT 的標準描述。

### 7.3 其他 NC/LC 4 大挑戰

| 挑戰 | 說明 | 對策 |
|---|---|---|
| **模型準確性與可靠性** | NC/LC + AI 的輸出可能有錯 | 多層次驗證 + 人工審核 |
| **資料隱私與安全性** | 敏感資料跑在第三方雲端 | 加密、匿名化、權限管控、選有合規認證的平台 |
| **道德與倫理風險** | 偏見、歧視、虛假資訊 | 建立倫理準則 + 公平性檢測 |
| **技術整合與平台適配性** | 跨平台串接困難、效能瓶頸 | 選有開放 API 的平台、適時上雲端 |

## 8. NC/LC 在 ETL 中的角色（10 min）

### 8.1 先複習 ETL（白話版）

> **ETL = Extract（萃取）+ Transform（轉換）+ Load（載入）**
>
> 白話：把資料從**這裡撈出來** → **整理乾淨** → **送到那裡**。
>
> 例：把 ERP 的銷售資料 → 清理日期格式、合併同名客戶 → 倒進 BI 報表系統。

（CH2 已經詳細介紹過 ETL 三步驟。）

### 8.2 NC/LC 在 ETL 裡能做什麼？

> 📌 **S2-Q12 連動**：考題問「在企業級數據管道（ETL）中，No Code / Low Code 平台的主要角色為何」，正解是 **(B) 取代部分傳統 ETL 解決方案，但可能無法處理過於客製化的邏輯**。

**白話**：
- ✅ **能取代**：常見的撈資料、合併、寄出去的流程（Make.com、n8n、Zapier 都很擅長）
- ❌ **不能取代**：大批量資料的高效處理、超複雜的業務規則、客製化資料清洗演算法

### 8.3 ETL 工具光譜

| 層級 | 使用者 | 代表工具 |
|---|---|---|
| **傳統 ETL** | IT 部門（需懂程式）| Talend、Informatica、Apache Airflow |
| **Low Code ETL** | 半技術背景 | n8n、Microsoft Power Automate、AWS Glue Studio |
| **No Code 自動化** | 業務人員 | Zapier、Make.com、IFTTT |

> 從上到下：技術門檻遞減、彈性遞減、易用性遞增。

> 🪧 **記法**：**簡單流程 → No Code 工具足夠；複雜處理 → Low Code 加減湊；超大規模或精細邏輯 → 還是要傳統 ETL 工程**。

## 9. NC/LC 與 AI 民主化（10 min）

### 9.1 AI 民主化（AI Democratization）

> **AI 民主化**：把 AI 技術從「**少數工程師專利**」擴展到「**所有人都能用、能做**」的趨勢。
>
> 白話：以前要做一個 AI 工具，要會寫 Python、會調模型；現在用 NC/LC 平台 + ChatGPT API，行銷專員也能組裝出一個 AI 客服機器人。

### 9.2 「市民開發者」（Citizen Developer）

> **市民開發者**：不是工程師、但能用 NC/LC 工具自動化日常工作的職場人士。
>
> 白話：公司裡會用 Zapier 把例行工作自動化的業務、會用 Make.com 接 AI 的行銷專員——他們就是「市民開發者」。

### 9.3 AI 民主化的 5 大正面影響

1. **降低技術門檻**：不會程式也能做出工具
2. **擴大可訪問性**：中小企業、個人創業者都能用得起 AI
3. **推動跨領域創新**：醫師、律師、老師都能組裝自己領域的 AI 應用
4. **強化普及應用**：AI 真正走進日常工作流
5. **啟動全球性影響**：開發中國家也能跟上 AI 浪潮

### 9.4 推動 AI 民主化的挑戰（**反例題**）

NC/LC 推動 AI 民主化會遇到：
- ✅ 真實挑戰：模型可解釋性不足、資料隱私風險、平台依賴度過高、技能落差
- ❌ **不是挑戰**：「提供昂貴的開發工具」——NC/LC 本來就是為了**降低成本**而生的

> 🚧 **這類「找出不是挑戰」的反例題很常考**。看到「**昂貴**」「**只給專家用**」這種跟「民主化精神」相反的選項，先排除。

## 10. 章末練習：7 題官方樣題詳解（30 min）

> 7 題全出自 iPAS 官方樣題科目二（114 年 9 月版）。每題附白話解說，看完就能直覺答題。

### Q1【S2-Q1・B】NC/LC 主要特色

> **題幹**：下列何者最能表達 No Code / Low Code 平台的主要特色？
> (A) 需要撰寫大量程式碼　(B) 運用模板快速建立應用程式　(C) 僅供專業開發人員使用　(D) 只能製作靜態網站

**正解**：**(B) 運用模板快速建立應用程式**

**白話解說**：NC/LC 的精神就是「**拖拉模板**」。
- (A) 跟「無代碼 / 低代碼」精神**完全相反**。
- (C) 描述顛倒——NC 正是給**非開發者**用的。
- (D) 過度限縮——NC/LC 也能做動態網站、App、流程自動化。

---

### Q2【S2-Q3・C】Low Code 比 No Code 更適合的情境

> **題幹**：下列哪種情況，選擇 Low Code 平台可能比 No Code 平台更為適合？
> (A) 需要非技術人員快速進行開發與應用　(B) 應用需求簡單，無需自訂功能　(C) 需要較複雜的業務邏輯並使用自訂整合功能　(D) 預算和時間極度有限

**正解**：**(C) 需要較複雜的業務邏輯並使用自訂整合功能**

**白話解說**：LC 比 NC 多了「能寫少量程式」這個自由度，所以**只要遇到複雜邏輯或自訂整合**，LC 就比 NC 適合。
- (A)(B)(D) 都是 No Code 的典型場景。

---

### Q3【S2-Q4・D】NC/LC + 生成式 AI 最不適合的應用

> **題幹**：關於生成式 AI 與 No Code / Low Code 平台的應用，下列何者最不適合？
> (A) 自動生成程式碼　(B) 自動化生成行銷文案　(C) 快速開發個人化 App　(D) 自動化生成法律判決

**正解**：**(D) 自動化生成法律判決**

**白話解說**：
- (A) 自動生成程式碼 ✓ 合理（GitHub Copilot 等）
- (B) 行銷文案 ✓ 合理（NC + ChatGPT 完美組合）
- (C) 個人化 App ✓ 合理（Bubble、Glide 都做得到）
- (D) **法律判決涉及司法權與責任歸屬，必須由人類法官把關**——不論用什麼工具，都不該由 AI 自動產出。

> 🪧 **記法**：看到「法律 / 司法 / 醫療診斷 / 性命攸關」這類選項，幾乎都是「**不適合 AI 自動化**」的正解。

---

### Q4【S2-Q5・C】NC vs LC 比較

> **題幹**：關於 No Code / Low Code 平台，下列敘述何者較正確？
> (A) 兩者完全相同　(B) Low Code 平台不需要任何程式設計知識　(C) Low Code 平台更適合開發靈活且可擴展的解決方案　(D) No Code 平台可以無限客製化

**正解**：**(C) Low Code 平台更適合開發靈活且可擴展的解決方案**

**白話解說**：
- (A) 錯：NC ≠ LC（差在「能不能寫程式」）。
- (B) 錯：LC 仍需少量程式知識，這就是「Low Code」的字面意思。
- (D) 錯：NC 受平台模板限制，**不可能無限客製化**。
- (C) ✓ LC 的賣點就是「靈活 + 可擴展」。

---

### Q5【S2-Q7・D】Low Code 平台的潛在風險

> **題幹**：使用 Low-Code 平台進行開發時，企業應特別留意下列哪一項潛在風險？
> (A) 可能造成企業內部敏感資料的洩露　(B) 難以進行大規模的應用擴展和維護　(C) 開發成本將大幅增加　(D) 可能有未經 IT 部門管理的應用程式擴散

**正解**：**(D) 可能有未經 IT 部門管理的應用程式擴散**

**白話解說**：(D) 就是「**Shadow IT 影子 IT**」的標準描述——這是 NC/LC **獨有的特殊風險**。
- (A) 資料外洩風險不是 LC 獨有，所有系統都有可能。
- (B) LC 反而**比較容易擴展**（這是它的賣點之一）。
- (C) LC 通常是**降低成本**，不是增加成本。

> 🚧 **這題容易誤選 (A)**——記住題目問的是「**LC 特別需要留意的風險**」，所以要選 LC 獨有的（影子 IT），而不是所有系統都有的（資料外洩）。

---

### Q6【S2-Q10・C】生成式 AI 的基礎技術

> **題幹**：下列哪一項技術是生成式 AI 的基礎？
> (A) 決策樹模型　(B) 聚類演算法　(C) 生成對抗網路　(D) 隨機森林技術

**正解**：**(C) 生成對抗網路（GAN）**

**白話解說**：（CH4 已詳解過）這題在 CH5 出現是因為「NC/LC 整合生成式 AI」是這章的延伸主題。**GAN 是生成式 AI 三大基礎之一**（VAE / GAN / Diffusion）。其餘選項都是鑑別式或非監督式演算法。

---

### Q7【S2-Q12・B】NC/LC 在 ETL 中的角色

> **題幹**：在企業級數據管道（ETL）中，No Code / Low Code 平台的主要角色為何？
> (A) 可作為前端數據可視化工具，協助展示與分析結果
> (B) 取代部分傳統 ETL 解決方案，但可能無法處理過於客製化的邏輯
> (C) 僅適用於小型數據應用，對大型數據處理的效率較低
> (D) 無法應用於任何數據處理場景，因整合難度過高

**正解**：**(B) 取代部分傳統 ETL 解決方案，但可能無法處理過於客製化的邏輯**

**白話解說**：(B) 是**最平衡**的描述——能做大部分 ETL 工作，但複雜邏輯仍要傳統工具。
- (A) 縮小了 NC/LC 的角色（NC/LC 不只能做可視化）。
- (C) 過度否定（NC/LC 也能處理中型資料）。
- (D) 完全否定（NC/LC 早已普遍應用於資料處理）。

> 🪧 **答這類題的訣竅**：選**最平衡**、**有條件**的敘述（「能做⋯但不能⋯」），避開**全肯定**或**全否定**的極端選項。

## 講師口頭補充（多場整合）

### NC / LC 的本質是「平台」這兩個字

不管 No-code（完全不寫程式、用圖像/流程圖/積木拉）還是 Low-code（少量程式碼），它們都是「**平台**」—**沒有平台這東西多數時候是不能動的**，這是 NC/LC 的根本特性，也是它最大缺點的根源。Gemini 內建 Canvas 功能搭配 vibe coding 可以做到中型網站、官網、簡單遊戲的程度，但它跑在 Google 的平台上、Google 一變你的東西就要跟著變。 —— 來源：[315p1]

### NC / LC 客製化軟體的真實案例

餐飲業 POS 系統（iChef / DoDo / 快一點）每家店打卡和出缺席算法都不一樣，沒一個通用系統能完美 fit—這時候 NC/LC 就有優勢，可以完全按自己的規則設計**表單請假 / 自動報表 / 公司內部審批**這類「內部用」的小工具。優點：開發效率高 + 圖像化邏輯讓使用者自己也看得懂自己的業務流。 —— 來源：[315p1]

### NC / LC 三大缺點 ⚠️ 必背：流量、Token、平台依賴

(1) **流量費用**：N8N 內容農場真實案例—學員測試期間覺得 OK，試用期一過開出來 6 萬塊台幣/月帳單，「我快吐了」；(2) **Token 計費**：現在多數 AI 工具計費都是按 Token，指令越複雜消耗越大；(3) **平台依賴**：類比早期 WordPress 一更新就把熟手的網站全打爛—平台一變你的開發模式就要跟著變。**實務建議**：「**內部使用完全 OK，但要對外商用、給外部顧客的話不建議**」。 —— 來源：[315p1]

### 平台鎖定（Vendor Lock-in）三層

(1) **技術鎖定**—平台 A 的拉法和 B 不一樣，搬遷要重新學；(2) **資料鎖定**—格式 / 儲存方式不相容，搬遷要重新設計資料庫（**設計這件事在開發里面很痛苦，移動就要全部人跟著重做一次**）；(3) **生態鎖定**—外掛功能（購物車、定期扣款、刷卡）每家平台接口不同。**比喻**：「就像 110 V 家電全部要換 220 V 插頭」，搬家成本是實際存在的。 —— 來源：[315p1]

### API = 餐廳服務生比喻

客人（client）目標是「得到服務」，廚房（server）目標是「提供服務」，**服務生就是 API**—點餐單從客人傳到廚房、菜從廚房傳到客人。**API 不做運算、只做資料傳遞**，「就像插座，電是電，插座是插座，方便你把家電插上去」。「串接 ChatGPT API」的真實意義 = 你網頁某個按鈕按下去，把資料用 API 丟給 ChatGPT，ChatGPT 處理完再用 API 回傳到你網頁。最常見的車牌辨識 API 流程：機子辨識車號 → API 把車號丟給停車場資料庫查詢繳費金額 → API 把金額傳回給機子顯示。 —— 來源：[315p1]

### 容器化 = 程式 + 環境一起打包

寫個翻譯程式，丟給人家不能只丟程式碼，還要丟需要的語言文本（例：寫中翻埃及文要連埃及文文本一起丟）。容器化就是把這些必要環境綁在一起，能在手機/平板/筆電到處跑。**優點**：跨平台運行；**缺點**：需要規劃打包邏輯。容器化算是 NC/LC 概念的延伸—讓「會寫程式但不會佈署環境」的開發者降門檻。 —— 來源：[315p1]

### AI 民主化的「爽點 vs 痛點」推廣邏輯

技術是雲端運算 + 開源生態 + 介面簡化（NC/LC、vibe coding）+ 教育推廣四步。最有效的傳播方式是讓人感受**爽點**—蝦皮、Foodpanda 一開始免費讓你狂用，習慣後加運費還是繼續用；**痛點**就是「沒這個會麻煩」的感受。中華電信在大學辦樂齡學堂教長輩 P 圖、生圖也是這條路線—降低資訊落差、讓 NC/LC + AI 真的擴展到弱勢族群。 —— 來源：[315p1]

### TensorFlow vs PyTorch 雖是程式框架但跟 NC/LC 屬同一個降門檻光譜

開源框架 = 提供免費的「預訓練模型 + 可重用程式碼」。**TensorFlow** 是 Google 開發、**PyTorch** 比 TF 新一點且主要用 Python（**Python 是現在 AI 模型訓練的主流語言**）。光譜由低到高：純程式 → 框架（TF / PyTorch）→ Low Code → No Code（圖像化、Scratch 風）→ vibe coding（純自然語言）。同樣是「降門檻」這條主軸的不同點。 —— 來源：[315p1]

### NC / LC 的資安提醒

開源框架/套件的資安風險—**XAMPP 真實案例**：早期很多網站用 XAMPP（Apache + PHP + MySQL 一鍵架站）建置，有人專挑 XAMPP 漏洞攻擊，結果全部使用 XAMPP 的網站連帶遭殃。意思是：**用熱門開源工具便宜方便，但被同樣的零日漏洞集體打趴的風險比較高**。NC/LC 平台等級也有同樣風險—平台被攻破，所有平台用戶都會受影響。 —— 來源：[315p1]

## 11. 章末小結

### 11.1 核心觀念回顧

1. **三種開發方式**：No Code（不寫程式）/ Low Code（少寫程式）/ Code（全寫程式）。
2. **NC vs LC 核心差異**：能不能寫程式擴充。
3. **NC 適合**：非技術人員、簡單流程、快速原型。
4. **LC 適合**：有技術背景者、複雜業務邏輯、企業級整合、需擴展性。
5. **不適用 NC/LC + AI 自動化**：法律判決、需要無限客製、極高安全敏感的核心系統。
6. **特殊風險：Shadow IT**——LC 太好上手，員工偷偷做工具導致 IT 失控。
7. **NC/LC 在 ETL**：取代部分傳統方案，但複雜邏輯仍需傳統工具。
8. **AI 民主化 + 市民開發者**：NC/LC 是讓 AI 真正普及到非工程師的關鍵基礎建設。

### 11.2 易錯陷阱清單

| 陷阱 | 正解 |
|---|---|
| 「NC 可以無限客製化」 | **錯**——NC 永遠受平台模板限制 |
| 「LC 不需要任何程式知識」 | **錯**——LC 字面意思就是「低**程式**碼」 |
| 「LC 風險主要是敏感資料洩漏」 | **錯**——LC 獨有風險是 **Shadow IT** |
| 「自動生成法律判決可以用 NC/LC」 | **錯**——司法不能由 AI 自動化 |
| 「NC/LC 完全無法做 ETL」 | **錯**——能取代部分（簡單流程），複雜邏輯才需傳統工具 |
| 「AI 民主化的挑戰是『提供昂貴工具』」 | **錯**——民主化的精神是降低成本，不是增加成本 |

### 11.3 速查卡（一頁濃縮）

**NC vs LC 核心對照**

| 維度 | No Code（NC）| Low Code（LC）|
|---|---|---|
| 定義 | 不寫程式，全靠拖拉 | 主要拖拉 + 少量程式 |
| 代表工具 | Zapier、Wix、Notion、Bubble、Glide、Make.com | n8n、Power Apps、OutSystems、Mendix |
| 使用者 | 非技術人員（業務、行銷、HR）| 有技術背景者（IT 助理、資料分析師）|
| 適用 | 快速原型、簡單流程、內部小工具 | 複雜業務邏輯、企業整合、需擴展性 |
| 風險 / 限制 | 受平台模板限制，無法無限客製 | Shadow IT 影子 IT（員工偷偷做工具）|

**不適合 NC/LC + 生成式 AI 的場景**
- ✗ 自動化法律判決（司法權專屬）
- ✗ 醫療核心診斷（人命攸關）
- ✗ 高度安全敏感的核心系統
- ✗ 需要無限客製、極高效能的場景

**6 大應用領域**：醫療、製造、金融、零售、教育、客服

**6 大選擇評估因素**：目標用戶 → 功能擴展性 → 安全合規 → 成本效益 → 技術支援 → 市場評價（TCO = 總擁有成本，ROI = 投資回報率）

**4 大挑戰**：模型準確性 / 資料隱私 / 道德倫理 / 技術整合

**NC/LC 在 ETL 的角色**：取代部分傳統方案，但複雜邏輯仍需傳統工具

**AI 民主化**：讓非工程師也能做 AI 應用
**市民開發者（Citizen Developer）**：用 NC/LC 自動化日常工作的非工程師
