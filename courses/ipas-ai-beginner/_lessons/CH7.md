## 7.1 AI 專案規劃 6 步驟（章節主軸 ⭐⭐⭐）（45 min）

### 為什麼是「6 步驟」不是「4 階段」？

學習指引官方版寫的是「4 階段：準備 → 設計 → POC → 實施」，但川辰科技實驗室補充版整理成更實用的「**6 步驟**」 — 跟業界 MLOps / 資料科學工作流一致：

```
1. 問題定義（Problem Definition）
   └ 明確要解決什麼商業問題、設定 KPI
2. 資料蒐集與清理（Data Collection & Cleaning）
   └ 找資料、ETL、處理隱私
3. 模型設計與訓練（Model Design & Training）
   └ 選模型、訓練、調參
4. 模型評估與驗證（Evaluation & Validation）
   └ POC、A/B test、偏差檢測
5. 部署與監控（Deployment & Monitoring）
   └ 上線、Data Drift 偵測、自動重訓
6. 持續優化與回饋（Iteration & Improvement）
   └ 收回饋、迭代、AI 價值擴散
```

> 🔧 **「裝潢房子」白話比喻**：
> 1. **問題定義** = 想要什麼風格 + 預算 + 要住幾人
> 2. **資料蒐集** = 量空間、選建材
> 3. **模型訓練** = 真的開始裝潢
> 4. **評估驗證** = 完工檢查、看樣品屋
> 5. **部署監控** = 入住、發現問題隨時修
> 6. **持續優化** = 用幾個月後換家具、優化動線

### ⭐ 第一步永遠是「明確定義導入目標 + KPI」

這是 iPAS 考試**最高頻考點**。所有「AI 導入第一步該做什麼」題目，**正解都是「明確目標 + KPI」**。

| 錯誤起手式 | 為什麼錯 |
|---|---|
| 先選技術供應商 | 連目標都不清楚，怎麼選？ |
| 先蒐集資料 | 不知道要解什麼問題，蒐集到的資料可能完全沒用 |
| 先設計對話範本 | 同上，方向沒定就動工 = 白做 |

### 對應考題詳解 — S2-Q35【D】客服系統第一階段任務

**題幹**：中型零售企業要導入 AI 客服系統，**第一階段任務**為何？

- (A) 設計 AI 對話範本 — 第 3 步驟
- (B) 選 AI 技術供應商 — 第 3 步驟
- (C) 蒐集過往客服對話記錄、清洗 — 第 2 步驟
- (D) ✅ **明確定義導入目標與關鍵績效指標** — 第 1 步驟正解

> 🔧 **記法**：「第一步」+「導入」 → 一律選「目標 + KPI」。

### 6 步驟 vs 錯誤起手式（S2-Q35 必懂）

| 考題選項 / 錯誤起手式 | 對應 6 步驟 | 為什麼不是第一步 |
|---|:---:|---|
| 設計 AI 對話範本 | 第 3 步：模型 / 方案設計 | 目標未定，先設計話術會變成憑感覺做功能 |
| 選 AI 技術供應商 | 第 3 步：模型 / 技術方案選擇 | 需求與 KPI 不清楚，無法判斷供應商是否合適 |
| 蒐集客服紀錄並清洗 | 第 2 步：資料蒐集與清理 | 要先知道要改善什麼 KPI，才知道要收哪些資料 |
| **明確導入目標與 KPI** | **第 1 步：問題定義** ✅ | **正解**。所有資源、資料、模型、驗證都由 KPI 回推 |

**關鍵句**：看到「第一階段任務 / 接下來第一步 / 導入 AI 客服」，**不要被資料、模型、供應商吸走**。第一步永遠是先定義商業問題與 KPI。

### 對應考題詳解 — S2-Q29【D】實施階段順序

**題幹**：在「**實施 / 營運階段**」中，下列步驟正確排序？

- A. AI 價值擴散（從單部門擴散到全企業）
- B. 上線部署
- C. 模型監控與優化

**選項**：
- (A) ACB — 還沒監控就擴散，會把問題複製
- (B) ABC — 還沒監控就擴散，同上
- (C) BAC — 同上
- (D) ✅ **BCA** — 上線（B） → 監控優化（C） → 擴散（A） — 正解

> 🔧 **「先穩自己再幫別人」記法**：上線 → 自己穩 → 才能擴散到其他部門。順序顛倒會把 bug 散播到全企業。

### KPI 量化範例（從模糊改成可量化）

| 場景 | 模糊目標 | 可量化 KPI |
|---|---|---|
| 客服 | 提升客服效率 | 首次回應時間 < 30 秒；人工轉接率 -20%；CSAT ≥ 4.5/5 |
| 行銷 | 提升貼文成效 | 每週產出 10 篇貼文草稿；CTR ≥ 3%；素材製作時間 -50% |
| 內部知識庫 | 讓新人更快上手 | 新人查找 SOP 平均時間 < 2 分鐘；重複詢問主管次數 -30% |
| 業務 | 提升提案品質 | 提案初稿時間從 3 小時降到 45 分鐘；成交率 +10% |
| 製造 | 降低瑕疵 | 瑕疵漏檢率 < 0.1%；停機時間 -20%；人工檢查工時 -40% |

**關鍵句**：KPI 至少要有**數字、方向、時間或門檻**。「變好」不是 KPI；「**從 A 降到 B**」才是 KPI。

## 7.2 IT 環境支援（GPU/TPU/NPU + 雲端三層）（25 min）

### 為什麼這節重要？

**S2-Q19、S2-Q25** 都考 IT 環境。0 基礎學員容易卡這裡，因為硬體名詞密集。

### AI 加速器一句話對照

| 縮寫 | 全名 | 一句話用途 | 何時用 |
|---|---|---|---|
| **CPU** | Central Processing Unit | 通用計算、什麼都能做但 AI 慢 | 一般運算 |
| **GPU** | Graphics Processing Unit | **平行計算強者，AI 訓練主力** | 深度學習訓練 / 推理 |
| **TPU** | Tensor Processing Unit | Google 自研，矩陣運算客製化 | TensorFlow / 大型 LLM |
| **NPU** | Neural Processing Unit | 手機 / 邊緣裝置即時推論 | iPhone Face ID、車載 AI |
| **ASIC** | Application-Specific IC | 特定模型客製化晶片 | 比特幣礦機、特殊應用 |

> 🔧 **口訣**：「**CPU 通用，GPU 並行，TPU 矩陣，NPU 邊緣**」。

### 雲端三層架構：IaaS / PaaS / SaaS（**0 基礎必懂**）

> 🔧 **「從蓋房到租房」白話比喻**：

| 層 | 你管什麼 | 服務商管什麼 | 比喻 | 代表 |
|---|---|---|---|---|
| **IaaS** | 作業系統 + 應用 + 資料 | 機房 + 硬體 + 網路 | **租地基自己蓋房** | AWS EC2、Azure VM |
| **PaaS** | 應用 + 資料 | 作業系統 + 中介軟體 + 機房 | **租毛胚屋自己裝潢** | Google App Engine、Heroku |
| **SaaS** | 只用、輸入資料 | 全部 | **租精裝公寓直接住** | ChatGPT、Notion、Salesforce |

**選擇邏輯**：
- 想完全自主控制 + 有 IT 團隊 → IaaS
- 開發中等規模應用 + 不想管 OS → PaaS
- 只想直接用功能 + 沒 IT → SaaS

### 對應考題詳解 — S2-Q19【C】IT 環境支援

**題幹**：企業要支援生成式 AI 運行，IT 環境最需具備？

- (A) 更多辦公設備 — 跟 AI 無直接關聯
- (B) 精簡內部流程 — 是組織議題，不是 IT 環境
- (C) ✅ **高效能運算資源 + 彈性儲存空間** — 正解
- (D) 增加跨部門交流 — 跟 IT 無關

> 🔧 **記法**：題目問「IT 環境」 → 正解一定談「**運算 + 儲存**」，其他選項都是組織管理 distractor。

### 對應考題詳解 — S2-Q25【C】GPU 利用率低（I/O 瓶頸）

**題幹**：製造業團隊用 PyTorch + CNN 在 GPU 訓練瑕疵偵測，**GPU 使用率僅 20-30%**，訓練數天。最可能原因？

- 影像 20MB / 張，存於 NAS（Network Attached Storage）

**選項**：
- (A) 模型架構複雜 — 會慢但 GPU 利用率應該滿
- (B) 訓練資料品質不穩 — 是收斂問題，不是 GPU 利用率
- (C) ✅ **NAS 載入產生 I/O 瓶頸，GPU 等待** — 正解
- (D) Batch Size 不當 — 雖會影響但不是題幹直接因果

> 🔧 **白話**：GPU 跑得太快，硬碟餵資料來不及，所以 GPU 閒著等。「**NAS + 大檔案 + GPU 利用率低**」 = I/O 瓶頸。

> ⚠️ **0 基礎易踩雷**：題目刻意給「20MB / 張 + NAS」當 hint，要學會抓題幹細節。

### 補充摺疊：AIoT、邊緣運算（想多懂一點？）

> 💡 不在主線測驗，認識即可。

- **AIoT** = AI + IoT（物聯網），裝置會「自己學、自己決策」（例：工廠感測器自動偵測異常並報警）
- **邊緣運算** = 把運算放到「靠近資料的地方」（手機 / 工廠閘道器），不上雲端，省頻寬 + 延遲低
- **NPU** 就是邊緣運算晶片代表

## 7.3 個資法跨境傳輸限制（**0 基礎必懂**）（20 min）

### 為什麼這節重要？

**S2-Q9** 是少數考法律的題目，0 基礎學員最容易答錯（看到法律條文直接懵）。但**只要記住一個關鍵點就會答**。

### 個資法跨境傳輸的核心邏輯

> 🔧 **白話**：你的客戶個資要傳到國外（如美國 AWS、新加坡 Google Cloud），政府會看 **「對方國家的個資法夠不夠保護」** — 不夠就限制你傳。

依《個人資料保護法》第 21 條，**主管機關可限制跨境傳輸**的法定情境主要是：

- ✅ **接收國家的個資保護法規不完備、可能損害當事人權益**（最典型）
- ✅ 涉及國家重大利益
- ✅ 國際條約或協定有特別規定
- ✅ 以迂迴方式傳輸到法規不完備之國家

### 對應考題詳解 — S2-Q9【D】個資跨境傳輸限制

**題幹**：公司擬將客戶個資傳輸至國外雲端儲存，**主管機關有權限制**的情況？

- (A) 公司規模未達標準、管理能力存疑 — 是企業自身能力問題，不是「法定限制理由」
- (B) 公司曾發生個資外洩紀錄 — 同上，企業問題
- (C) 包含當事人已公開資訊但仍涉隱私 — 已公開不適用個資法限制
- (D) ✅ **接收國家個資保護法規尚未完備，可能損害當事人權益** — 正解

> 🔧 **記法**：「政府限制」+「跨境傳輸」 → 正解永遠是「**對方國家法規不完備**」。其他都是企業內部問題。

### 對應法規概覽（**僅認識，不必背條文**）

| 法規 | 區域 | 重點 | 時效備注 |
|---|---|---|---|
| **個資法** | 台灣 | 跨境傳輸限制（第 21 條） | 2025 已修法，2026 個資會籌備處預告施行細則與子法草案；考試以 iPAS 當梯簡章為準 |
| **歐盟 AIA** | 歐盟 | 四級風險分類、高風險強制書面評估（DPIA） | 2026-08-02 起多數規則、Annex III 高風險與透明義務進入適用 / 執法階段 |
| **GDPR** | 歐盟 | 被遺忘權、資料匿名化要求 | 持續有效 |
| **CCPA** | 美國加州 | 消費者隱私法案 | 持續有效 |
| **金管會 AI 指引** | 台灣金融業 | 金融業 AI 風險治理指引 | 2024 指引仍是基礎；2026-05 媒體報導規劃納入可程式化 AI / AI Agent / AI 風險分類，**正式新版尚未公告** |

### 主權 AI（Sovereign AI）一句話

> 🔧 **Sovereign AI** = 國家自己掌握 AI 基礎設施、資料與模型，**不受外部平台控制**。
>
> 這也是 TAIDE / TWCC 的政策出發點：台灣不能完全依賴 OpenAI / Google。

## 7.4 隱私保護：Zero-Retention 與 Zero Trust（25 min）

### Zero-Retention 一句話

> 🔧 **Zero-Retention（不保留模式）** = AI 用完你的資料就**立刻丟掉、不存、不訓練模型**。
>
> 例：OpenAI 企業版 API 預設 Zero-Retention，你公司的內部資料丟進去，OpenAI 不會拿去訓練 GPT。

### 為什麼重要？

公司用 ChatGPT API 處理內部資料，最怕「**我的資料變成下一版 GPT 的訓練料**」 — 競爭對手問問題就可能挖到。Zero-Retention 是最直接的解。

### 對應考題詳解 — S2-Q18【A】第三方 API 隱私保護

**題幹**：企業用第三方 AI API（OpenAI / Claude / Gemini）處理內部資料，**避免敏感資訊遭模型記憶或進一步分析**，應優先採取？

- (A) ✅ **啟用 Zero-Retention 模式或具企業資安保障的 API** — 從源頭防止
- (B) 虛擬機隔離環境 — 仍會傳資料出去
- (C) 視覺化轉換 — 不解決資料外傳
- (D) 向量嵌入後傳輸 — 仍可能被反推

> 🔧 **記法**：「敏感資訊不被模型記憶」 → 正解一定是「Zero-Retention」。

### Zero Trust Architecture（零信任架構）一句話

> 🔧 **Zero Trust Architecture** = 「**永不信任、持續驗證**」的資安架構 — 即使是公司內部員工，每次存取資料都要重新驗證身份。

> 🔧 **「公司 VPN 不再給長期通行證」白話比喻**：
> 傳統：員工進公司一次 VPN 通了，整天都能存取所有資料（**一次信任、終身放行**）
> Zero Trust：每次存取每個系統都要重新驗證 + 檢查設備是否合規（**每次驗證**）

### 對應考題詳解 — S2-Q31【C】Zero Trust Architecture

**題幹**：管理生成式 AI 系統的隱私風險，何種**技術**最能確保資料使用安全性？

- (A) 強化學習 — 是 AI 訓練方法，跟資安無關
- (B) 深度學習 — 是 AI 模型類型，跟資安無關
- (C) ✅ **零信任架構** — 資安架構正解
- (D) 注意力機制 — 是 Transformer 內部機制，跟資安無關

> ⚠️ **0 基礎易踩雷**：A、B、D 都是「AI 技術名詞」，題目問的是「**資安架構**」 → 一律選 Zero Trust。

### 對應考題詳解 — S2-Q26【D】資料安全核心

**題幹**：生成式 AI 導入過程，資料安全與隱私保護**最重要**考量？

- (A) 設定目標優先級 — 是專案管理
- (B) 增強客服回饋 — 是業務功能
- (C) 資料視覺化 — 是分析工具
- (D) ✅ **權限控管與合規要求** — 正解

> 🔧 **記法**：「資料安全 / 隱私」 → 永遠選「**權限 + 合規**」。

### 補充摺疊：差分隱私、同態加密、聯邦學習（想多懂一點？）

> 💡 不必背技術細節，認識用途即可。

- **差分隱私（Differential Privacy）**：在資料中加入隨機雜訊，讓你猜不到「**這筆資料是誰的**」，但統計結果仍正確
- **同態加密（Homomorphic Encryption）**：資料**加密狀態下也能運算**，運算完再解密
- **聯邦學習（Federated Learning）**：資料**不集中**，AI 模型在各裝置本地訓練，只交換模型參數

## 7.5 風險管理四象限（30 min）

### 風險管理四象限速查

> 🔧 **「下雨天怎麼辦」白話比喻**：

| 策略 | 做法 | 比喻 | 適用 |
|---|---|---|---|
| **緩解（Mitigation）** | 採取行動降低發生機率或衝擊 | **帶傘出門** | 風險可控、想自己處理 |
| **轉移（Transfer）** | 把風險丟給第三方 | **買保險 / 外包給防水公司** | 自己處理成本太高 |
| **接受（Acceptance）** | 不做特別處理，準備備案 | **算了、準備毛巾擦頭** | 衝擊小或無法避免 |
| **規避（Avoidance）** | 完全不做這件事 | **乾脆不出門** | 衝擊太大、不可承擔 |

> ⚠️ 注意：學習指引另列了「**溯源**」「**文化**」共 6 大策略，但考試核心是這 4 象限。「溯源」= 追查資料來源、「文化」= 員工教育，可視為「緩解」的子策略。

### 6 大策略 vs 4 象限對齊（學習指引 vs 考題主架構）

| 學習指引 6 大策略 | 對齊 4 象限 | 說明 |
|---|---|---|
| 緩解 | **緩解** | 加護欄、資料清洗、人工審核、偏差檢測，降低發生率或衝擊 |
| 轉移 | **轉移** | 外包、保險、第三方資安服務，把部分責任或處理成本交給外部 |
| 接受 | **接受** | 風險低、成本高於損失時，記錄並準備備案 |
| 規避 | **規避** | 不做高風險用途，或改用不涉及敏感資料的方案 |
| 溯源 | 多半屬**緩解** | 追查資料來源、模型輸出來源，降低錯誤與責任不清 |
| 文化 | 多半屬**緩解** | 教育訓練、使用規範、責任意識，降低人為誤用 |

**關鍵句**：4 象限是**考題主架構**；6 大策略是**實務補充**。考選擇題先用 4 象限判斷——看到「外包」選**轉移**，看到「教育訓練 / 溯源 / 護欄」多半歸**緩解**。

- (A) 緩解 — 自己處理
- (B) ✅ **轉移** — 外包 = 經典風險轉移

- (D) 規避 — 不做

| 風險類型 | 例子 |
|---|---|
| **倫理風險** | AI 生成內容**帶偏見、歧視** |
| **營運風險** | 系統運行中斷導致業務受影響 |
| **成本風險** | 資料儲存成本上升 |

## 7.6 POC 驗證 — 公平性與偏差檢測（25 min）

### POC 一句話

> 🔧 **POC（Proof of Concept，概念驗證）** = 全面導入前的**小規模實驗**，驗證「這個 AI 方案到底行不行」。

### POC 5 大評估方法

| 方法 | 一句話 |
|---|---|
| **驗證基準** | 用標準指標（BLEU、ROUGE、Perplexity）打分 |
| **自動化效能測試** | 大量批次測試，看效能 |
| **主觀人工檢查** | 領域專家 + 用戶回饋 |
| **壓力測試 + 邊界檢查** | 看極端情境下表現 |
| **公平性 / 偏差檢測** | 看不同族群結果是否一致 |
| **A/B 測試 + 線上驗證** | 真實流量比較新舊版本 |

- (B) 對抗性測試 — 查防禦、不查公平

## 7.7 資料漂移 vs 概念漂移（⚠️ 易混淆對比）（20 min）

### 兩者一句話對比

| 名稱 | 英文 | 一句話 | 比喻 |
|---|---|---|---|
| **資料漂移** | Data Drift | **輸入資料分佈**隨時間變化 | 「客戶年齡層變了」 |
| **概念漂移** | Concept Drift | **輸入跟輸出的關係**變了，不是分佈變 | 「客戶喜好變了，AI 學的規則過時了」 |

### 具體例子對比

**情境**：銀行用 AI 預測客戶信用風險。

| 變化 | 屬於 | 為什麼 |
|---|---|---|
| 客戶年齡平均從 35 歲變成 45 歲（**輸入變了**） | **資料漂移** | 是「資料分佈」變化 |
| 月收 5 萬以上才信用好 → 通膨後月收 7 萬以上才信用好（**規則變了**） | **概念漂移** | 是「輸入 → 輸出規則」變化 |
| 銀行新增「信用卡使用率」欄位 | **資料漂移** | 是「資料維度」變化 |
| 疫情後遠距工作族信用評分突然有效（**新模式出現**） | **概念漂移** | 是「規則本身」變化 |

> 🔧 **記法**：「**資料分佈變**」 = 資料漂移；「**規則 / 關係變**」 = 概念漂移。

### 為什麼要監控漂移？

模型上線後**不會永遠準確**。世界在變，模型訓練時的世界 ≠ 現在的世界 → **必須持續監控 + 自動重新訓練**（這是 AI 專案 6 步驟第 5 步「部署與監控」的核心）。

> 💡 雖然目前 35 題官方樣題沒直接考漂移辨識題，但這是模擬題與中級考的常見題型，必懂。

## 7.8 商業評估工具：SWOT 與商業九宮格（25 min）

### 為什麼這節要講？

**S2-Q35** 第一階段任務是「明確目標 + KPI」 — 但要怎麼想出有效的目標？這就需要**商業評估工具**。

### SWOT 分析

> 🔧 **SWOT** = 四個英文字首：

| 內部 / 正面 | 內部 / 負面 |
|---|---|
| **Strengths（優勢）** | **Weaknesses（劣勢）** |
| 公司有什麼強項？ | 公司有什麼弱點？ |

| 外部 / 正面 | 外部 / 負面 |
|---|---|
| **Opportunities（機會）** | **Threats（威脅）** |
| 市場 / 環境給什麼機會？ | 市場 / 環境帶什麼威脅？ |

**用在 AI 導入評估**：

- **S（優勢）**：我們有大量歷史客服對話資料 → 可訓練 AI
- **W（劣勢）**：沒有資料科學團隊 → 要外包或用 SaaS
- **O（機會）**：客戶期待 24/7 服務 → AI 客服正好補位
- **T（威脅）**：AI 客服回答錯誤可能引發法律糾紛 → 要設人類審核機制

### 商業九宮格（Business Model Canvas）

> 🔧 把商業想法畫成一張**九格圖**：

```
[關鍵夥伴]  [關鍵活動]  [價值主張]  [顧客關係]  [客群]
            [關鍵資源]              [通路]
[成本結構]                          [收入來源]
```

**用在 AI 導入評估**：可以把「**這個 AI 應用對應的商業模式**」整張畫出來，看出「成本對誰、收入從哪、價值給誰」 — 比純口頭討論清楚很多。

### 對應考題詳解 — S2-Q23【A】生成式 AI 敘述（複選排列）

**題幹**：關於生成式 AI，下列敘述哪些**正確**？

- A. 生成內容不會帶偏見 ❌（會有偏見，CH7 風險章已講）
- B. 高度準確、不會有虛假 ❌（有幻覺）
- C. 生成內容**準確性建議經人類審核** ✅
- D. 每次生成內容**可能不同** ✅（隨機性）
- E. 高度安全、不會資料外流 ❌（仍有風險）

> 🔧 **記法**：這是「**常見迷思整理題**」 — A、B、E 都是「過度樂觀」迷思，正解是「**人類審核 + 隨機性**」。

## 講師口頭補充（多場整合）

### AI 風險治理四原則 ⚠️ 必背

(1) **整合性**—風險管理要融入企業治理；(2) **比例性**—風險管理強度依 AI 影響力決定（手機全權限 AI 自然是高比例）；(3) **動態性**—AI 模型存活週期通常約一年，要持續監視；(4) **透明性**—以季 / 月 / 年為單位做全面檢視，公開決策過程。 —— 來源：[314p2]

### 「為 AI 而 AI」的真實警告 — 9.2% 真正落地

美國去年數據：真正打算或有導入 AI 的企業 60-70%、**真正落地的只有 9.2%**。多數公司是「老闆要求做 AI」但想不清楚 AI 到底要解決什麼問題，「**AI 的用意是幹嘛的，沒有想清楚就說我要**」。AI 導入四階段必走：(1) 問題定義；(2) 資料盤點；(3) 模型設計；(4) 上線 + 持續追蹤。第一步「問題定義」就是 7.1 節的 KPI，整章主軸都圍繞這個。 —— 來源：[314p2]

### KPI 質性 vs 量性的雙例

以「ChatGPT 教英文」為例—**質性 KPI** = 「降低學生不敢開口的可能性」（一句抽象話）；**量性 KPI** = 「每週讓一個學生練到 100 個新單字」（可量化）。同個專案兩種 KPI 並存才完整。考試與動手題提到 KPI 至少要列出兩種類型。 —— 來源：[314p2]

### ROI vs TCO 的雙線思考

**ROI（投資報酬率）** 關注「丟進去的錢能不能賺回來」；**TCO（持有總成本）** 算開發 + 維運 + 應用 + 合規授權。多數人覺得 ROI 重要但 TCO 也很大—「不同團隊在不同面向會花到的錢不一樣，有的公司專養某類人才這塊就便宜」。**重要實務洞察**：實務接案時最常遇的不是技術成本問題，是**人文治理**—企業內部行政流程或使用工具上「水土不服」、不願嘗試新東西，導致導入效果差。所以 AI 評估要看「能不能整合到既有系統架構」，不只看技術可不可行。 —— 來源：[314p2]

### MLOps 四目標 + 自我學習機制

(1) **自動化**；(2) **可重複**；(3) **持續自我學習**；(4) **可監控 / 回饋**。**核心精神**：人力介入只應該介入在「執行面」，daily 維運要自動化交給機器。鑑別型 AI 遇例外條件要自動存下來，下次回頭學進去—類似漁市場老闆看到新魚不會明天忘。MLOps 工具中**追蹤資料版本**比追蹤模型版本更重要—因為丟什麼資料進去決定模型表現。 —— 來源：[314p2]

### 「考古題 5 年後失效」案例

你今年把哈佛考古題丟進 NotebookLM 通過考試，5 年後學弟用同樣的工具卻沒過—因為**他少了中間 5 年的考題趨勢**，AI 模型對時代變遷敏感、不像簡報是一次性產物。「**AI 專案不會是一次型專案**」，這就是為什麼動態性 + 持續更新是 AI 治理的核心。 —— 來源：[314p2]

### 雲 / 邊 / HPC 三架構的選擇

(1) **雲端運算** = 資料 + 模型放伺服器，主流（GCP/AWS）；(2) **邊緣運算** = 末端裝置運算，延遲低（自駕車、IoT），缺點是受限算力 + 更新困難（虎頭山自駕公車要本地端更新、現場用程式碼控制公車）；(3) **高效能運算（HPC）** = 超級電腦，做物理模擬、數位孿生—物理系教授論文要排隊跑超級電腦兩個月才拿到數據，「資訊科技發展速度比其他領域跳得非常快」。 —— 來源：[315p1]

### AI 民主化的「爽點 vs 痛點」推廣邏輯

技術是雲端運算 + 開源生態 + 介面簡化（NC/LC、vibe coding）+ 教育推廣四步。最有效的傳播方式是讓人感受**爽點**（蝦皮、Foodpanda 一開始免費讓你狂用，習慣後加運費還是繼續用），痛點就是「沒這個會麻煩」的感受。中華電信在大學辦樂齡學堂教長輩 P 圖、生圖也是這條路線—降低偏鄉 / 弱勢族群的資訊落差。 —— 來源：[315p1]

### AI 導入成本佔比口訣

人力成本 ≈ 4-5 成（最大宗）、算力成本（GPU 帳單 / 雲端帳單 / 外包）、授權成本（軟體 / API / 外部模型）、維運成本（伺服器電費 + 重新訓練）。多數軟體專案「人力 + 流量」是兩大頭，AI 把「流量」換成「算力」。**優化策略四方向**：(1) 人力—兩個工程師 + 善用 AI 寫 code 取代 5 個工程師（但「不要變成只會 vibe coding 不會 debug 的學弟妹」）；(2) 算力—模型訓練不要追求完美、學習率適中即可，混合雲；(3) 授權—開源框架便宜但有資安風險（XAMPP 被攻破，所有用 XAMPP 的網站連帶遭殃）；(4) 維運—自動備份 / 歸檔。 —— 來源：[315p1]

### POC vs Pilot 兩階段

**POC（概念驗證）** = 技術可行性，「會不會動」；**Pilot（試點 / 試行）** = 實際價值 + 可用性，「動了但客人喜不喜歡」。例：地獄廚房同桌不同料理時間要同步上菜（龍蝦煮 20 分 + 牛排煮 15 分，5 分後牛排才下鍋）—這就是 Pilot 階段測試出的流程。**所有非 AI 的服務開發也應該有這兩階段**。POC 通常會在小型合作單位、企業概念店做測試（小米的家電生態圈、保全公司中保無限家的智慧家電就是這套）。 —— 來源：[315p1]

### AI 導入成效四類 + 顧客價值的「溫度感」矛盾

(1) **技術績效**（正確率）；(2) **營運效率**（省時間 / 省人力，可量化）；(3) **顧客價值**（質性—溫度感不能丟）；(4) **財務績效**（省錢 / 賺錢，例：印章廠商網頁加 AI 預覽圖既節省設計師溝通也提升轉換）。**重要警告**：「AI 客服太有效率沒溫度反而讓客人有無名火」—顧客價值這塊跟 AI 是矛盾的，「不要那麼像 AI、不要那麼像 AI」是常見要求。**日本是極端例**—一整天可以完全不跟人類交流、不點頭也能完成消費。 —— 來源：[315p1]

### 歐盟 AIA 四級風險清單細節

**禁止使用** = 違反人權的操作（Social Credit、潛意識操作—**演算法社群媒體某方面也碰到這層**）；**高風險** = 信用評分 / 醫療診斷 / 教育評量 / 升職系統—「**台灣繁星玉田系統其實也算高風險，只是它沒那麼 AI、是大數據跑的**」；**有限風險** = 聊天機器人（誤導但不直接決策）；**最低風險** = 過濾器 / 遊戲。**Doom 遊戲是純 AI 即時生成的特別例**—每幀畫面都 AI 生、地圖永遠不一樣。「歐盟 + 高風險 + 強制書面評估」就是 DPIA。 —— 來源：[315p1] [415]

### GDPR 七原則 ⚠️ 必背

(1) **合法透明公平**；(2) **目的限制**（**不能二次運用**—點餐系統的客戶資料不能搬去做旅遊 App，即使可能讓客戶體驗更好）；(3) **資料最小化**（同 ZTA 概念—吃個飯不需要家裡地址）；(4) **準確性**（要可改但**不能讓使用者隨意改生日領禮券**，要客服 + 證件審核）；(5) **保存期限限制**（被遺忘權）；(6) **完整性與保密性**；(7) **責任制**。**七樣缺一不可、每個都很重要**。實踐上會增加 AI 訓練成本（資料治理 + 重訓開銷），所以 GDPR 與 AI 訓練是「天生矛盾」要找平衡。 —— 來源：[315p1]

### AI 著作權四風險

(1) **著作權**—訓練資料來源是否有版權（古阿莫解說電影被告，比電影上映還早出來，但 AI 沒辦法像他那樣辯解「我是融入自己作品」）；(2) **資料隱私**—AI 把訓練時的個人資料當「**前世記憶**」吐出來，「**孟婆湯沒喝乾淨**」是完全沒辯解空間的風險；(3) **授權與歸屬**—使用者用 AI 工具賺很多錢，工具開發商反過來主張分潤；(4) **責任歸屬**—AI 教你做炸藥你照做出事，是你的責任不是 AI 的（「不然化學老師也要被抓」）。 —— 來源：[315p1]

### 三種授權模式 + 暴雪 Dota 故事

(1) **開源許可**—自由使用，附上原作者；(2) **開源衍生注作權（Copyleft）**—生成作品歸你；(3) **自定義限制授權**—**暴雪 Dota 故事**：Dota 是暴雪魔獸自定義地圖，因為條款不明確，原開發者跑去做 Dota2，暴雪後來補上條款「**你做出來的遊戲都算我暴雪的**」，這是限制授權的經典案例。AI 工具的特定風格生成 + 商業條款也常用此模式。 —— 來源：[315p1]

### 風險管理四方向

(1) **風險轉移**—轉嫁第三方（保險、雲端服務商擔保）；(2) **風險接受**—小錯誤忽略，「越描越黑」就不處理（例：模型準確率已 95% 不再為了一點點趴數修）；(3) **風險緩解**—降低發生機率 / 衝擊，AI 護欄就是這層（輸入防護 + 工程優化 + 輸出檢查）；(4) **風險迴避**—直接停止 / 換策略（例：肖像權問題就直接買開源資料）。實務應用上會混合多種策略。 —— 來源：[315p2]

### 各國 AI 法規重點對比

**歐盟** = 透明性 + 可解釋性（嚴格審查制）；**美國** = 市場導向 + 自願性（供需原則 + 資料隱私）；**台灣** = 數據保護 / 個資為主，透明性反而比較少。考試題目寫到「歐盟」幾乎一定綁透明性 / 可解釋性 / DPIA / 高風險審查；寫到「美國」綁市場主導；寫到「台灣」綁個資。 —— 來源：[315p2]

### 系統性防禦五原則

(1) **最小權限原則**—AI 助手只需讀取就只給讀取，不開寫入 / 刪除（「警衛拿到 A 樓鑰匙也進不了 BC 樓」）；(2) **人在迴圈內 / 人在迴圈上**—電商 AI 助理收到「下架所有紅洋裝」要產報表給人類審核（in the loop），不直接執行；自駕車則是 on the loop（不能每動作都問人）；(3) **風險評估**—串金流就要算最壞情況可被刷光多少；(4) **完整 LOG + 即時告警**；(5) **UX 設計**避免使用者輸入不該輸入的東西。 —— 來源：[315p2] [415]

### CH7 各業別 AI 應用（408 整合）

(1) **災害應變** = 影像辨識 + 災情監測（水量 / 降雨量感測 + 即時判斷誰要優先驅離）；(2) **氣象** = 路徑預測（取代專家手算颱風動態）；(3) **無人機** = 拍攝 + 回傳影像為主軸（救災、農業、打仗都同邏輯）；(4) **媒體識讀** = 判別假新聞偏誤—「**現在假新聞大宗都是偏頗角度 + 惡意新聞而不是完全造假**」，講話節目愛引用國外政治名嘴 X 或 FB 發文當作真實新聞來源；(5) **農業六級** = 智慧種植（番茄園 AI 機械手臂自動養殖）+ 供應鏈預測；(6) **漁業 / 食品** = 影像辨識魚種 + 麵包店條碼困境 + 洋芋片工廠 AI 偵測發芽馬鈴薯；(7) **零售業** = 顧客體驗 + 銷售精準度（超商會員 APP 推播）；(8) **物流** = 配送路線優化 + 不在家自動跳下家；(9) **HR** = 自動篩選履歷（薪資 / 報稅 / 門禁是 ERP，跟 HR 無關）；(10) **教育** = 個人化學習 + 學習歷程分析；(11) **醫療** = 影像辨識異常 + 輔助診斷；(12) **金融** = 信用評判 + 詐騙偵測（轉大筆錢時自動鎖；(13) **觀光** = 行程規劃 + 民宿 AI 客服（找充電器在哪這種小問題）；(14) **環境** = 感測器數據預測污染；(15) **交通** = 車流監控 + 號誌控制；(16) **智慧城市** = 整合感測器資料優化能源 / 交通 / 垃圾回收；(17) **文化創意** = **風格轉換**（最不易侵權的方向）+ 內容創作；(18) **公共安全** = 異常行為偵測（突然奔跑 / 倒下）；(19) **能源** = 智慧電網 + 用電預測（台電不像水庫蓄電，用多少發多少）；(20) **客服** = 聊天機器人最直接的 AI 應用，但要注意溫度感矛盾。 —— 來源：[408]

### 銀行信用評分的 XAI 為什麼必綁 SHAP

題目場景如果是「銀行 + 法規審查 + 解釋邊際貢獻度」直接綁 SHAP。實務上銀行業導入 AI 都會被法規要求高度可解釋性，所以這類業務場景必考 SHAP / LIME 二選一。背法：**SHAP 解釋全模型每個特徵的邊際貢獻；LIME 解釋單筆預測的局部行為**。 —— 來源：[415]

### 醫療資料 + 差分隱私是必考組合

醫療機構釋出去識別化病例資料給研究時，怕成員推斷攻擊（membership inference），所以導入差分隱私來控制雜訊量。考試題目寫到「醫療 + 去識別化 + 成員推斷攻擊」幾乎一定考差分隱私的 ε 方向感。 —— 來源：[415]

### 醫院 / 銀行 / 高機敏資料優先聯邦學習

聯邦學習的觸發條件很清楚：「資料不能離開本地網絡」。場景包括：

- 各地醫院共訓練罕見疾病模型（資料受各地法規限制）
- 各家銀行共訓練詐欺偵測（業務機密）
- 跨國資安資料

關鍵記憶點：聯邦學習上傳的是「梯度權重變化」而非「原始資料」、不是「加密原始資料」、不是「完整作業系統印象檔」。其他三個都是干擾項。 —— 來源：[314p1] [408] [415]

### 反事實解釋（Counterfactual）的金融場景

銀行拒絕貸款後告訴客戶：「你的月收入再增加 5000 + 信用卡遲繳次數減少一次就會核准」——這就是反事實解釋。對使用者最直觀、最有具體行動建議，金融業最常用這種解釋方式。 —— 來源：[415]

### 機密運算 / TEE 的「運算中保護」位置

資料三種狀態 + 對應保護：

| 狀態 | 保護機制 |
|---|---|
| 傳輸中 | HTTPS / TLS |
| 儲存中 | 加密儲存 |
| 運算中 | 過去無保護，現在用 TEE / 機密運算 |

題目寫到「雲端最高權限管理員也不能窺探記憶體中原始值」就綁 TEE。這是 2024-2026 年新興的金融 / 醫療雲端合規重點。 —— 來源：[415]

### 高風險 AI 進入歐盟市場前的 DPIA

對使用者權益與自由產生「高風險」的 AI 系統，歐盟 AIA 強制要求書面評估程序——資料保護影響評估（DPIA）。考試出現「歐盟 + 高風險 + 強制 + 書面評估」就是這個。 —— 來源：[415]

### Lakehouse 為何取代傳統資料倉儲

業務面：移動端普及讓本地端算力差異變大，企業傾向把運算集中到雲端，這也是消費模式從買斷硬體變成服務型訂閱的底層原因。技術面：lakehouse 保留了 SQL/ACID 的嚴謹結構化管理能力，又能處理非結構化海量資料，比純資料湖更強在合規查詢、比純資料倉儲更彈性。

導入 AI 時資料側的選型，現在主流是 lakehouse（如 Databricks Delta Lake、Snowflake、Iceberg）而不是傳統 EDW。 —— 來源：[415]

2. **資料蒐集規劃**
   - 要哪些資料？資料在哪？是否涉及個資？
   - 是否有跨境傳輸風險？（呼應 7.3）

3. **IT 環境選擇**（呼應 7.2）
   - GPU / TPU / NPU 怎麼選？
   - IaaS / PaaS / SaaS 怎麼選？
   - 自建 vs 租用 vs 外包？

4. **隱私與資安規劃**（呼應 7.4）
   - 是否使用 Zero-Retention API？
   - 是否導入 Zero Trust Architecture？
   - 法規對應（個資法 / GDPR）？

5. **風險管理四象限**（呼應 7.5）
   - 列出 5 個主要風險
   - 每個風險決定用 緩解 / 轉移 / 接受 / 規避 哪個策略
   - 至少 1 個倫理風險、1 個營運風險

6. **POC 設計**（呼應 7.6）
   - POC 範圍（小規模試點）
   - 5 大評估方法用哪些？
   - 公平性如何驗證？

7. **部署與長期維運**（呼應 7.1 + 7.7）
   - 上線 → 監控 → 擴散順序（B → C → A）
   - 資料漂移 / 概念漂移如何偵測？
   - 多久重新訓練一次？

**驗收標準**：

- [ ] 7 段都有寫
- [ ] KPI 至少 1 個是可量化的（含具體數字）
- [ ] 風險管理至少 5 個風險
- [ ] 風險策略 4 象限至少用到 3 種
- [ ] 隱私規劃有提到「Zero-Retention 或 Zero Trust」其一
- [ ] POC 設計有考慮公平性 / 偏差檢測

**字數建議**：1500 - 2500 字（A4 1.5 - 2 頁）

> 💡 這份報告是**邁向中級的橋梁** — 中級考會考更深的「**模型架構選擇**」「**程式碼閱讀**」，但這份報告的「**評估規劃能力**」是中級的根基。

### 0 基礎特別提醒

1. **AI 專案 6 步驟必反覆強調「第一步永遠是 KPI」**：這是最高頻考點，至少在課堂提 3 次。
2. **「裝潢房子」比喻全章貫穿**：6 步驟用裝潢類比、雲端三層用「蓋房 / 毛胚 / 精裝」、風險策略用「下雨天怎麼辦」 — **比喻一致**幫助 0 基礎建立心智地圖。
3. **個資法 S2-Q9 容易答錯**：正解「對方國法規不完備」是法律精準用語，要強調「不是企業自身能力問題」。
4. **資料漂移 vs 概念漂移用具體例子對比**：純文字定義學員會混淆，必須用銀行信用評分例子拆解。
5. **動手題是「整門課集大成」**：不要當作普通動手題，要花至少 30 分鐘 brief，學員可帶回家完成。

### 特別教學設計：呼應全課程

CH7 是**最終章**（CH8 是中級附錄、CH9 是模擬測驗），必須：
- 章首一句話：「**這章是把 CH1-CH6 串成一條可執行動線**」
- 章末一句話：「**完成這章的動手報告 = 你已具備初級評估規劃師的核心能力**」
- 動手題刻意設計成「**呼應前章學過的所有概念**」 — 讓學員感受「我真的學會了一門完整的學問」
