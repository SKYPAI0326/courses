HYBRID AUTOMATION · 4.2

Make + n8n 雙引擎協作

Make 負責雲端接收與通知，n8n 負責本機執行與文件操作。本節完成整條雙向協作鏈路：任務進來 → 本機處理 → 結果回傳 → Gmail 通知。

CLOUD · Make.com
雲端端
接收 Google Forms / Webhook
HTTP 轉送任務到 n8n
接收 n8n 回傳結果
發送 Gmail / Line 通知
LOCAL · n8n Desktop
本機端
Webhook 接收任務
讀寫本機資料夾
批次文件處理（無次數限制）
HTTP 回傳結果給 Make
STEP BY STEP · 操作步驟
1
確認 4.1 鏈路正常（Make → n8n）
本節建立在 4.1 的基礎上。先確認 4.1 的完整鏈路運作正常：Google Forms 提交後，Make 能成功轉送資料到 n8n Webhook，且 n8n Switch 節點能正確分流。若尚未完成 4.1，請先返回設定。
提示：快速測試方式：直接用 Postman 或瀏覽器呼叫 n8n 的 Webhook URL，傳一筆假資料，確認 n8n 有收到即可繼續。
2
n8n：執行本機操作（讀取資料夾、生成文件）
在 n8n Switch 節點的各個分支中，加入本機操作節點。以「報告」分支為例：用 Code 節點組合文件內容，再用 Write Binary File 儲存到本機，生成 申請人_報告_日期.txt 格式的檔名。
// Code 節點 - 組合報告內容 const applicant = $json.body.applicant; const docType = $json.body.document_type; const dueDate = $json.body.due_date; const ts = new Date().toISOString().slice(0, 10).replace(/-/g, ''); const fileName = `${applicant}_${docType}_${ts}.txt`; const content = `文件類型: ${docType}\n申請人: ${applicant}\n截止日期: ${dueDate}\n\n（本文件由 n8n 自動產生）`; return [{ json: { fileName, content, applicant, docType, dueDate } }];
提示：Write Binary File 前需先用「Convert to File」節點把文字轉成二進位格式，選擇 Encoding 為 UTF-8。
3
n8n：建立結果回傳的 Make Webhook
在 Make.com 建立第二個 Scenario，第一個模組一樣選「Webhooks → Custom webhook」，複製此 Webhook URL。這個 Webhook 專門接收 n8n 完成任務後的回傳結果，準備觸發後續的通知發送。
提示：這是第二個獨立的 Make Scenario——第一個處理「接收表單→轉發」，第二個處理「接收結果→發通知」。兩個 Scenario 分開管理更清晰。
4
n8n：文件完成後 HTTP 回傳結果給 Make
在 Write Binary File 節點之後，新增「HTTP Request」節點。Method 設為 POST，URL 填入步驟 3 的 Make Webhook URL，Body 回傳處理結果（檔名、狀態、申請人）。
// n8n HTTP Request Body 回傳 Make { "status": "success", "fileName": "{{ $json.fileName }}", "applicant": "{{ $json.applicant }}", "docType": "{{ $json.docType }}" }
提示：HTTP Request 節點的 Content-Type 設為 application/json，否則 Make 收到的會是純文字字串無法解析。
5
Make：接收 n8n 結果並發送 Gmail 通知
在第二個 Make Scenario 中，Webhook 之後加入「Gmail → Send an email」模組。To 填申請人信箱，Subject 填「您的文件已處理完成」，Body 填入動態內容引用 n8n 回傳的欄位。
// Gmail 通知模板 Subject: 您的 {{1.docType}} 申請已完成 Body: 您好， 您申請的文件「{{1.fileName}}」已於本機自動完成處理。 狀態：{{1.status}} 申請人：{{1.applicant}} 感謝您使用自動化系統。
提示：確認 Make 的 Gmail 連接已設定好（需 Google 帳號授權）。第一次使用時 Make 會引導你完成 OAuth 授權。
完成驗證 · VERIFY
提交 Google Forms 後，（1）本機對應資料夾出現新文件，（2）Gmail 收到自動通知，郵件內容正確顯示申請人姓名、文件名稱和處理狀態。兩個 Make Scenario 的執行記錄都顯示 Success，代表雙向協作鏈路完全打通。
常見問題 · TROUBLESHOOT
n8n 的 HTTP Request 回傳失敗（Make 沒收到）？
確認 n8n 的 HTTP Request URL 是 Make 的 Webhook URL（不是 n8n 自己的 URL）。Make Webhook 必須是「啟用」狀態，且在等待中（Listening 或 Active Scenario）。檢查 n8n 執行記錄的 HTTP Request 節點有無網路錯誤。
Gmail 模組失敗或通知沒寄出？
在 Make 的 Gmail 模組重新授權（點擊 Reauthorize）。確認 To 欄位填入了正確的 Email 格式。若使用 Gmail 免費帳號，每日發送上限為 500 封，測試階段不會有問題。
💼 實戰案例
情境｜某電商雙引擎月結算：Make 用掉 420 ops（免費內）、n8n 跑了 12,000 次（本機免費）——原本全 Make 方案需每月付 $9 Pro。
做法｜把『本機檔案操作』『批次大量』都丟 n8n；『雲對雲小量』留 Make——每月省 $9 × 12 月 = $108。
踩雷｜Make 回傳 n8n webhook 要設 Timeout 30 秒——n8n 若慢回 Make 會失敗 retry，造成重複觸發。規則：n8n 端先 200 OK 回 Make，再背景處理。
