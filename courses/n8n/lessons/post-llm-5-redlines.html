8 條紅線 — 用 LLM 改 workflow 不能踩

8 頁系列 · 第 5/8 · 核心安全規範

本頁重點
理解每條紅線的具體失效情境，不只知道「不能做」也知道「為什麼」
掌握紅線 2 / 6 的禁止項目清單，在改 workflow 前主動核查
能用紅線 7 / 8 的正確流程，在隔離環境完成驗證再上線
(01)

為什麼有紅線

背景說明
LLM 給的 JSON 看起來合理但容易跑出細節錯誤；以下 8 條是 Codex L3 兩輪審核（call_id 765f5530 / f9c18d2c）採納的強制規範，雷區內容「不可模糊化或淡化」。

這 8 條紅線不是「建議」，是「踩到就出事」的強制邊界。每條都有真實的失效情境：credential 外洩沒得救、production workflow 被未驗證版本接管、destructive 操作寄錯人刪錯檔。逐條讀完，印出貼螢幕邊，每次改 workflow 前掃一眼。

(02)

8 條紅線

紅線 01
不要把 credential 一起貼給 LLM
為什麼

n8n workflow JSON 內可能包含 API key、Gmail OAuth token、Notion integration secret 等敏感欄位。即使你「信任」某 LLM 服務，貼上去就失去控制權。永遠用 n8n UI 的「export without credentials」選項，並在貼給 LLM 前用搜尋確認沒有 apiKey / accessToken / password 字串殘留。

紅線 02
不要讓 LLM 直接產出完整 workflow JSON 當最終答案
為什麼

LLM 從零生成的 workflow JSON 有極高機率出現 typeVersion 過時、節點 ID 衝突、connection 引用錯位、參數 schema 漂移等問題，匯入直接失敗或更糟——匯入成功但跑起來資料形狀不對。永遠以官方 Lite Pack 或 n8n UI 拉出來的範本為基底，LLM 只在限定範圍內改。

特別禁止項目（即使只改一段，這些也不能讓 LLM 自動決定）
  • webhook URL
  • trigger 類型
  • schedule 排程
  • recipient list（收件人 / 通知對象）
  • 任何 delete / update / drop 類資料庫或 API 操作

任一被誤改 = production 直接出事，沒得救。

紅線 03
版本漂移要先用 n8n UI 的自動升級，不要叫 LLM 改 typeVersion
為什麼

n8n 升大版時，節點的 typeVersion 與 parameter schema 可能變動。LLM 不知道你裝的是哪一版，直接改 typeVersion 數字會把問題從「警告」變成「跑時 crash」。先在 n8n UI 開啟 → 它會自動處理可升級的部分 → 跑不通再用 LLM 對照新舊 schema 排查。

紅線 04
節點 deprecation 要查官方 changelog，不要相信 LLM 的記憶
為什麼

LLM 訓練資料有 cutoff，可能不知道某節點已被官方改名／拆分／移除（例：n8n 歷史上有過 Gmail node 拆出 Gmail Trigger 的情況）。遇到「找不到節點」/「節點變灰」時先去 n8n 官方 changelog 與社群論壇查證，再決定是換新節點、還是用 HTTP Request 直接打 API。

紅線 05
不要把 LLM 輸出當答案，要當草稿
為什麼

LLM 給的 JSON 區段有 90% 機率「看起來合理」但「跑起來細節錯」。一定要走完步驟 4「匯入測試」+ 步驟 6 收斂迴圈。沒有跑過真實樣本之前，那只是草稿，不是答案。

紅線 06
保留節點 ID 與 connection 完整性
為什麼

n8n workflow JSON 裡，節點之間靠 id 與 connections 區段串聯。LLM 在區段改寫時容易動到 ID 或漏掉 connection。步驟 3 要求 LLM「只回傳節點區段」、學員在 n8n UI 上做替換就是為了避開這個——不要直接覆蓋整份 JSON 檔。

重要補述
  • 若確實需要新增或刪除節點（不是修改既有節點），connections 區段必須回到 n8n UI 上手動重接，不要叫 LLM 寫 connections JSON
  • UI 上拖線 30 秒就完成，LLM 寫錯一個 ID 你 debug 30 分鐘
  • 第 7 節「錯誤分流 SOP」的 connections 檢查 checklist，只有在你改了整份 JSON 或新增／刪除節點時才需要跑——UI 上區段替換沒動 connections，跳過
紅線 07
改完先在隔離環境驗，不在 production workflow 直接測
為什麼

n8n 匯入流程預設保留原 workflow 的 active 狀態，若改的是已開的 production workflow，匯入瞬間就會用未驗證版本接收真實流量／真實檔案。正確流程如下，任一步跳過 = 你在用真客戶資料當白老鼠。

正確操作流程
  1. 在 n8n UI 點 workflow 右上角「Duplicate」複製一份命名 <原名>-edit
  2. 把 LLM 改的內容套到複本
  3. 確認複本 trigger 是 OFF（或 Manual）
  4. 跑通再複製回原 workflow
紅線 08
destructive 操作必先 dry-run
為什麼

涉及刪資料（Notion delete page / Drive delete file）、寄信（Gmail / Telegram 發給真實收件人）、更新 CRM（HubSpot / Pipedrive update record）的節點，LLM 改錯一個欄位就是真寄錯人、真刪檔。寧願多花 5 分鐘做 dry-run，不要省這步。

正確操作流程
  1. 把 destructive 節點先 disable 或換成 console log 等價假動作
  2. 跑通驗證 input／output 資料形狀
  3. 改寫 recipient 為自己的測試帳號
  4. 用 1-3 筆假資料跑
  5. 全程 OK 才換回真收件人 / 真資料源
(03)

Cheatsheet 提醒

這 8 條建議印成 A4 雙欄一頁貼螢幕邊，每次改 workflow 前快速掃一眼，特別是紅線 1（credential）、紅線 2（禁整份重生）、紅線 7（隔離環境）與紅線 8（dry-run）。

(04)

本頁小結

8 條紅線的核心精神可以用一句話概括：LLM 是草稿機，你是驗收人。LLM 給的 JSON 從不是最終答案，credential 的安全責任、production 環境的保護、destructive 操作的 dry-run，都是你的責任，不能外包給 LLM。
