10 條紅線 — 用 LLM 改 workflow 不能踩

每條紅線：危險動作 + 為什麼 + 安全替代路徑。印出貼螢幕邊，每次改 workflow 前掃一眼。

本頁重點
對每條紅線說出「危險動作 + 安全替代路徑」兩件事（不只知道禁止）
掌握 deprecation 4 步動線、destructive 執行前 3 件事、secret rotate SOP
印出章末對照表 + 動手前 6 條快速檢查貼螢幕邊，每次改前掃 30 秒
🖨
本章設計為列印 cheat sheet，按 ⌘P（Mac）或 Ctrl+P（Windows）列印，A4 直向即可貼螢幕邊。章末對照表 + 6 條快速檢查會自動避免跨頁斷行。
(01)

為什麼有紅線

核心觀念
紅線 ≠ 禁忌。紅線 = 「LLM 在這做容易錯」+「正確路徑你自己（或在 n8n UI 上）做才安全」。每條紅線都有替代動線。

10 條紅線是踩到就出事的強制邊界，不是「建議」。每條都有真實的失效情境：credential 外洩沒得救、production workflow 被未驗證版本接管、destructive 操作刪錯資料。但更重要的是，每條都告訴你「正確要做什麼」。

(02)

10 條紅線

紅線 01
不要把 credential 一起貼給 LLM
危險動作

把含 apiKey / accessToken / password 的 workflow JSON 直接貼進 LLM 對話。

為什麼

貼上去 = 失去控制權。LLM 服務的對話歷史可能被訓練、被洩漏、或在團隊 plan 被同事看到。

安全替代路徑
  1. n8n UI 開啟 workflow → 右上角 ⋮ → Download → 跳出視窗選「Without credentials」（不是 Default）
  2. 下載的 .json 檔，用 VS Code 或記事本打開，Cmd+F 搜尋這 4 個字串都要找不到：apiKey、accessToken、password、token
  3. 如果還是找到 → 該欄位手動改成 "REDACTED" 再貼 LLM
驗收：搜尋 4 個字串都 0 個結果
紅線 02
不要讓 LLM 直接產出完整 workflow JSON
危險動作

對 LLM 說「給我完整 workflow JSON 我直接匯入」。

為什麼

LLM 從記憶生成的 JSON 有 90% 機率出現 typeVersion 過時 / 節點 ID 衝突 / connection 引用錯位 — 匯入直接失敗或更糟（成功但靜默跑錯）。下列 5 項更是 LLM 不能碰的：webhook URL、trigger 類型、schedule 排程、recipient list、delete/update/drop 操作。

安全替代路徑（每種動作各有對應）
  1. 改節點內容：用模板 3 + 在 n8n UI 上區段替換
  2. 改 webhook URL：n8n UI 點 webhook 節點 → 旁邊「Copy Test/Production URL」按鈕 → 自己 copy 自己貼
  3. 改 trigger 類型：n8n UI 手動拖新 trigger 節點 → 接線 → 舊 Manual Trigger disable / 刪除
  4. 改 schedule 排程：Schedule Trigger 節點的 Cron Expression 欄位 → 自己 Google「cron 每天 9 點」貼 expression
  5. 改 recipient list：Gmail 節點 To 欄位 → 自己貼 email，不讓 LLM 從上下文猜
  6. 改 delete/update 操作：先 disable 該節點跑通其他流程，最後人工開啟並走紅線 8 dry-run
紅線 03
不要叫 LLM 改 typeVersion
危險動作

要求 LLM 把節點 typeVersion 從 2 改成 3 來「修升級錯誤」。

為什麼

typeVersion 對應到 n8n 裝的版本，LLM 不知道你裝哪版。猜錯就 crash。

安全替代路徑
  1. n8n UI 開啟舊 workflow → 頂部黃色橫條「This workflow contains nodes that need to be updated」→ 點「Update」按鈕
  2. UI 自動升級到當前裝的版本（比 LLM 準）
  3. 升級後仍跑不過：把「升級前 vs 升級後 parameter」兩段 JSON 貼 LLM，問「哪個欄位格式變了」
驗收：黃色橫條消失，節點不再變灰
紅線 04
節點 deprecation — 不要直接問 LLM，走 4 步動線
危險動作

n8n UI 上某節點變灰跳「This node is deprecated」，直接問 LLM「我要換成什麼節點」。

為什麼

LLM 訓練資料 cutoff 可能比你裝的 n8n 版本舊。LLM 說「換成 X 節點」可能那節點也被 deprecated。

安全替代路徑（5–10 分鐘可走完，4 步順序固定，LLM 是最後 fallback）
1
確認真的是 deprecated（30 秒）
在 n8n UI 點該節點 → 點齒輪 ⚙ → 看 typeVersion 欄位數字 + 看節點顏色是否比旁邊節點深灰。typeVersion 是 1 而其他類似節點是 2/3，加上顏色變灰 → 99% 確認是 deprecated。
↓ 若確認
2
在 n8n UI 內找替代（2 分鐘）
n8n UI 左側 / 右上角 + Add Node 搜尋 box → 輸入舊節點名稱 → 看搜尋結果是否有同名節點後面標 (new) 或 v2/v3 → 直接從搜尋結果拖一個出來。
↓ 若第 2 步沒有 (new)/v2 結果
3
看官方 docs banner（3 分鐘）
到 docs.n8n.io/integrations/builtin/<category>/n8n-nodes-base.<nodename>/，頁面頂端若有黃/紅 banner 寫「This node is deprecated. Use [新節點名] instead.」→ 把新節點名抄回 n8n UI 第 2 步搜尋。
↓ 若第 1–3 步都沒結果才用
4
LLM 候選清單 fallback（5 分鐘）
把第 1–3 步的事實貼 LLM，只要候選節點名 + 每個適合什麼場景一句話，不要 JSON / 不要改 workflow。拿到清單後回 n8n UI + Add Node 搜尋每個候選，看哪個拖出來不變灰能正常設定 → 用那個。
驗收：找到的替代節點在 n8n UI 拖出來不變灰、能正常設定
紅線 05
LLM 輸出是草稿，不是答案 — 必走 Duplicate 驗證動線
危險動作

LLM 給 JSON → 直接在原 workflow 點 Active 開關上線。

為什麼

LLM 給的 JSON 有 90% 機率「看起來合理但跑起來細節錯」。沒有跑過真實樣本，那只是草稿，不是答案。

安全替代路徑（強制動線）
  1. 禁止：直接在原 workflow 點 Active
  2. 必走：Duplicate 成 <原名>-edit → 確認 Active 開關 OFF → 套 LLM 內容 → 跑 1 筆 → 跑 10 筆 → 全量跑通 → 才換回原 workflow
驗收：workflow 列表同時有原版 + -edit 版；只有跑通 10 筆後才合併
紅線 06
保留節點 ID 與 connection 完整性
危險動作

用文字編輯器打開 workflow.json 直接改、或覆蓋整份 JSON 讓 LLM 重寫 connections 區段。

為什麼

節點 ID 對不上 → 匯入後節點變灰 / connections 引用錯 → 連線斷。連 1 個逗號錯整份炸。

安全替代路徑
  1. 不要：用文字編輯器直接改結構
  2. 要：在 n8n UI 上做所有「結構變動」（新增 / 刪除節點 / 重接連線）
  3. 重接連線具體動作：在 n8n UI 上，把上一個節點右側的小圓點 → 按住滑鼠拖到下一個節點左側的小圓點 → 放開 → 出現一條線就接好了
驗收：n8n UI 上所有節點之間都有實線（不是虛線、不是斷線）
紅線 07
改完先在隔離環境驗，不在 production 直接測
危險動作

在 production workflow 上直接改 + 跑 → 用真客戶資料當白老鼠。

為什麼

n8n 匯入流程預設保留 active 狀態，瞬間就接真實流量。

安全替代路徑（self-host 與 n8n Cloud 動線相同）
  1. n8n UI workflow 列表 → 原 workflow 右側三點 → Duplicate
  2. 改名為 <原名>-edit
  3. 點開 -edit 版，右上角 Active 開關確認 OFF（灰色）
  4. 套 LLM 改的內容到 -edit 版
  5. 跑 1 筆 → 跑 10 筆 → 通過後 → 改名為 <原名>-v2 並 Active；原版改名 <原名>-v1-archived 並 OFF

Cloud 用戶注意：Cloud 也有 Duplicate，n8n UI 操作一致。找不到時用「Export without credentials → New workflow → Import」三步替代。

驗收：workflow 列表同時看到原版 + -edit 版；Active 狀態切得乾淨
紅線 08
destructive 操作必先 dry-run
危險動作清單
  • Gmail / Outlook / Telegram / Slack 節點：發訊到真實收件人
  • Notion / HubSpot / Salesforce 節點：delete / update record
  • Google Drive / Dropbox / S3 節點：delete file / overwrite file
  • SQL 節點：DELETE / UPDATE / DROP
為什麼

LLM 改錯一個欄位 = 真寄信 / 真刪檔 / 真改 CRM。沒得救。

執行前 3 件事（至少做 1 件，用來限縮傷害在可恢復範圍）
  • 1
    目標資料先備份：SQL → mysqldump/pg_dump；Google Sheet → File → Make a copy；Notion/HubSpot/Salesforce → Export CSV；Drive/Dropbox/S3 → 複製整個目標資料夾到 <原名>-backup-YYYY-MM-DD
  • 2
    限範圍跑：SQL DELETE 加 LIMIT 10 或 WHERE id IN (1,2,3,4,5)；Notion/HubSpot update 在前一個 Filter 節點加 limit=5；Gmail/Telegram 發訊加 IF 節點只發 5 筆
  • 3
    跑完手動 spot check 5 筆：SQL 開 DB client 查那 5 筆；Sheet 直接打開看；CRM 開 UI 找那 5 筆 record；Drive 用 Finder / 網頁端看檔案；Gmail/Telegram 去自己（或測試帳號）收件夾確認
等價假動作（dry-run 時把 destructive 換成這些）
  • Gmail 發訊：把 To 欄位改成你自己的私人信箱字面值（例 yourself@gmail.com）
  • Notion delete：先換成 Notion read（讀同一個 page，不刪）
  • Drive delete：先換成 Drive list（列出資料夾內容，不刪）
  • SQL DELETE：先換成 SELECT 同樣 WHERE 條件（看會 select 出哪幾筆）
驗收：dry-run 階段你的私人信箱收到 1–3 封測試信，無任何真實收件人收到信；備份/限範圍/spot check 至少做了 1 件
紅線 09
不要把 credentials / .env / webhook secret 貼給 LLM
危險動作
  • 貼 credentials JSON（含 apiKey / OAuth refresh token / service account JSON）
  • 貼 .env 內容（postgres 密碼 / N8N_BASIC_AUTH_PASSWORD / Telegram bot token / Gemini API key）
  • 貼含 secret query param 的 webhook URL（例 https://…/webhook/abc?token=xxx）
為什麼

LLM 對話會被廠商記錄。「不要假設你貼給 LLM 的東西不會出現在訓練語料 / log analyst 螢幕上」是合理 default。一次 paste，這個 secret 必須當作已 leaked、立刻 rotate — 不要心存僥倖。

安全替代路徑
  1. 想讓 LLM 看 workflow 形狀：用 placeholder 替換實值 — "apiKey": "__GEMINI_API_KEY__"、"botToken": "<YOUR_TG_TOKEN>"、"password": "<YOUR_PG_PASSWORD>"
  2. 需要 LLM 解 .env 結構：給 KEY 列表（不貼實值），例：「我有 N8N_BASIC_AUTH_PASSWORD / GEMINI_API_KEY / TG_BOT_TOKEN，請說每個 KEY 應該長什麼樣」
  3. webhook URL 也要去 secret：截掉 ?token=xxx 段，只留 https://<host>/webhook/<id>
  4. 不小心 paste 了：立刻去對應服務 rotate — 見下方 4 種 SOP
4 種 Service Rotate SOP
Google AI (Gemini)
  1. 開 Google AI Studio
  2. 左側 API Key
  3. 點 Regenerate
  4. 更新 n8n credential
Telegram Bot Token
  1. 在 Telegram 找 @BotFather
  2. 傳 /revoke
  3. 選該 bot
  4. 取新 token → 更新 n8n
Postgres
  1. ALTER USER <user>
  2. WITH PASSWORD '<新密碼>'
  3. 更新 n8n credential
  4. 重啟相關服務
n8n Basic Auth
  1. 改 .env 的 N8N_BASIC_AUTH_PASSWORD
  2. 重啟 n8n container
  3. 更新 n8n credential
  4. 通知所有使用者
驗收：貼進 LLM 的內容 Cmd+F 搜尋 apiKey / token / password / secret / AIzaSy / sk- → 0 個結果，或只搜到 placeholder 字串（__X__ / <YOUR_X>）
紅線 10
不要匯入或執行來路不明的 workflow JSON
危險動作

從 Reddit / GitHub gist / Discord 撿到的 workflow JSON 直接匯入並執行。尤其含下列 5 個高風險節點 type 時：

n8n-nodes-base.httpRequest n8n-nodes-base.code n8n-nodes-base.executeCommand n8n-nodes-base.readWriteFile n8n-nodes-base.emailSend

為什麼

n8n workflow 是你 n8n 帳號權限的延伸 — 匯入後你的 credentials 自動是它的 credentials。陌生 workflow JSON 從技術上等同陌生人寄你一個 .exe 叫你雙擊。可能的後果：打外部 server exfiltrate 你的資料、刪你的本地檔案、用你的名義發 email / Telegram。

安全替代路徑
  1. 來源限制：只匯入「課程提供」/「自己親手寫」/「你親自審完每個節點」的 workflow
  2. 匯入前：用 VS Code 打開 JSON，搜尋上方 5 個高風險節點 type，若有 → 逐一看它做什麼（HTTP 打哪 / Code 寫什麼 / 讀寫哪個檔）
  3. 第一次跑：把 Active 開關 OFF + Trigger 換成 Manual Trigger，不讓它一匯入就自動跑
  4. 用測試 credentials 先試一輪：在 n8n 裡開一個 "test" credential（測試 API key、測試 DB），workflow credential 切到測試版跑，行為符合預期才換回正式 credential
驗收：匯入前 5 個高風險節點 type 都看過 + 第一次跑時 Active OFF + Trigger 是 Manual + credential 是測試版
(03)

動手前 6 條快速檢查

每次改 workflow 前掃這 6 條，30 秒確認沒踩雷。

動手前快速檢查 — 每次必掃
  • 我用了 export without credentials 並搜尋 apiKey/accessToken/password/token 都找不到 （紅線 1）
  • 我沒有把 credentials / .env / webhook secret 貼給 LLM （紅線 9 — secret leak 比 prompt 改錯嚴重一級）
  • 我在 -edit 版做事，不在 production （紅線 7）
  • 我請 LLM 給「節點區段」，不是「整份 workflow JSON」 （紅線 2）
  • 我有 destructive 節點時，先 disable 或改 dry-run （紅線 8）
  • 我 10 筆樣本跑通才放全量 （紅線 5）

(05)

常見錯誤 + 怎麼解

錯誤 1
「我看到紅線 4 deprecation，去 Google 半天找不到答案」
商業培訓學員不要花超過 15 分鐘 Google。15 分鐘沒解 → 截圖 + 描述貼課程社群。LLM / Google 不是 unique solution。
錯誤 2
「我做了紅線 7 的 Duplicate，但忘了關 Active 開關，差點上 production」
Duplicate 完成後第一個動作就是檢查 Active 開關。嚴格走「Duplicate → 開關 OFF → 改名」這個固定動線，不省略。
錯誤 3
「Cloud 沒有 Duplicate 按鈕」
Cloud 也有，在 workflow 列表 / 開啟畫面右上角 ⋮。真的找不到 → 「Export without credentials → New workflow → Import」三步替代。
錯誤 4
「dry-run 我把 To 改成自己信箱跑了，結果沒收到信」
(a) Gmail credential 是否還有效？看節點 credential 欄位是否亮綠 (b) 自己的信箱是否在垃圾郵件 (c) 是否真的執行了那個 Gmail 節點（看 n8n UI execution 紀錄）
(06)

本頁小結

10 條紅線的核心精神一句話：LLM 是草稿機，你是驗收人。credential 的安全責任、production 環境的保護、destructive 操作的 dry-run，都是你的責任，不能外包給 LLM。

印出本頁的「動手前 6 條快速檢查」和「業務改動需求對照表」貼螢幕邊，每次改 workflow 前掃 30 秒。下一頁是完整 walkthrough：把 #02 改成處理發票，實際走完 6 步流程。
