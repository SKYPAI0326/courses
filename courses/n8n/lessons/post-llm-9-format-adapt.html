跨格式改寫 — 從 PDF 跨到 Word / Excel / 圖片 / 純文字

結業後第一個業務需求很少還是「PDF 改名」。Word 報價單、Excel 客戶清單、LINE 截圖、業務 CSV 匯出——本章教你把 #03 改造成處理這 5 種格式的具體做法：只動 4 層中的前 3 層，業務規則層永遠不動。

讀完這頁你會能做到 · OUTCOMES
看到任一格式（docx / xlsx / png / csv / txt / md），30 分鐘內把 #03 改成處理該格式，業務規則層 0 動
看到 LLM 回應 30 秒內辨認「純前 3 層改造」還是「擴張到業務規則層」，後者直接拒絕
列印本章 dispatch 對照表貼螢幕邊，每次跨格式改寫前掃 30 秒
(01)

結業後你拿到的不會是 PDF

學員課後問卷最常出現的版本
「我有 50 個 Word 報價單要照 #03 那種方式改名 + 分類」
「主管丟了 200 列的 Excel 客戶清單，要我用 AI 幫每筆寫個 follow-up 摘要」
「同事傳了一資料夾 LINE 截圖（.png）要我整理成檔案名」
「業務每天匯出 .csv 我要做摘要寄給主管」

學員打開 #02 / #03 看——整個 workflow 從 fileSelector *.pdf 到 Extract 節點 operation: pdf 全在處理 PDF。心理卡住的地方通常是三個：

  • 1
    「我要把這份改成 .docx，第一步動哪？」
    不知道「改格式」要動哪幾個節點，感覺要大改。
  • 2
    「我問 ChatGPT 怎麼改，它一口氣把整份 workflow 重寫」
    看不出哪裡是「我要的」哪裡是「它順手加的」，不敢貼進去。
  • 3
    「跑了一張綠燈，但 AI 改名出來變 null_null.png」
    input 是圖片，不知道從哪 debug，跟發票那次症狀一樣但原因不同。

本章給你一個統一的心智模型（4 層分層），以及 5 種格式各自的具體做法。讀完加上第 6 章 walkthrough，你的課後改寫場景有 90% 被涵蓋。

(02)

核心觀念：4 層分層，跨格式只動前 3 層

把 #03 workflow 想成 4 層堆疊。跨格式改寫的所有操作，都只在前 3 層——業務規則層永遠不動。

層 內容 跨格式時 具體到哪個節點 / 欄位
入口層 fileSelector（讀哪些檔） 必動 換副檔名 Read 節點 → File Selector 欄位
Extract / Decode 層 Extract from File operation / Read 直接拿 string 通常動 換 operation 或跳過（disable） Extract from File 節點 → Operation 欄位
lpCall helper 層 retry / 429 / RPM throttle 工具層 只有圖片需動 加 contents 參數（2 行 patch） Code 節點頂部 lpCall 函式定義（line 30-60 區）
業務規則層 prompt 目標 / 改名規則 / Write 路徑 完全不動 Code 節點的 prompt 字串、newFilename 拼接、results.push、Write 節點 fileName
■
新口號：「業務規則層不動」
prompt 目標字串、newFilename 拼接邏輯、Write 節點 fileName 表達式——這 3 件事永遠不動。跨格式只動上面 3 層（最多 3 個欄位；圖片格式多加 2 行 lpCall helper patch）。

LLM 回應裡只要動到這 3 件事其中一個，就是「擴張到業務規則層」——用模板 5 拒絕。

這跟第 5 章紅線 2「不要讓 LLM 直接產出完整 workflow JSON」直接呼應——前 3 層只動 2-3 個欄位（圖片再加 2 行），根本不需要整份重寫。

動線總覽（4 步固定，不論格式）

  • 1
    判斷格式——學員手上的檔案副檔名是什麼？有文字層（docx / xlsx / txt / csv）還是無文字層（png / jpg / 掃描 PDF）？
  • 2
    選 Extract 策略——對照本章第 (08) 節「5 格式 dispatch 對照表」，找到對應策略。
  • 3
    動前 3 層（最多 3 個地方）——入口層 fileSelector 副檔名 + Extract/Decode 層設定 + （只有圖片）lpCall helper 層加 contents 參數。
  • 4
    業務規則層保持原狀——Code 節點裡 prompt 目標字串、改名規則、Write 節點 fileName 表達式全部不動。
△
跟第 6 章的差別：第 6 章 walkthrough 是「同格式（PDF）但業務目標換」，動的是業務規則層。本章是「業務目標一樣但格式換」，動的是前 3 層。兩章合起來涵蓋學員 90% 的課後改寫場景。
(03)

格式 A：.docx — 用「中介格式策略」

n8n 沒有 docx 原生 operation
n8n Extract from File 沒有 docx 專屬 operation（n8n 1.x 確認）。直接用 text operation 會抽到 zip XML 結構不是內文——Word .docx 是 zip + XML 結構，學員拿到 <?xml version="1.0"?>... 開頭的亂碼。
所以 .docx 不能跟 .pdf 一樣「換 Extract operation 就好」，要走中介格式策略。
路線 1 · 推薦給商業學員
手動 batch 轉 PDF 再走 #03
Word / Pages / LibreOffice 內建批次 export PDF：
· Word：檔案 → 另存新檔 → 選 PDF（或 macOS 自動化動作 / Win 巨集 batch 跑）
· Pages（Mac）：批次匯出 → PDF
· LibreOffice：Tools → Macros → 批次 ConvertDocuments

轉完直接用既有 #03。零 workflow 改動。
路線 2 · 進階學員
HTTP Request 接 Microsoft Graph / Google Docs API
不在本章範圍（屬於 #06 webhook + 外部 API 整合教學）。如果學員量大或要全自動化才走這條。

不要走的死路（學員會踩的雷）

❌ 把 #03 fileSelector 改 *.docx + Extract operation 留 text → 會抽到 zip XML 亂碼，AI 拿到 <?xml version="1.0"?>... 亂猜檔名
❌ 試圖叫 LLM 寫 docx parser 進 Code 節點 → mammoth 之類的 npm 模組 n8n 預設沒裝，學員裝不了
❌ 用 Read 直接拿 binary 後 Code 節點手寫 unzip → 走進工程地獄

(04)

格式 B：.xlsx — 多一層：處理多 sheet 邏輯

改造點：fileSelector: "*.pdf" → "*.xlsx"、operation: "pdf" → "xlsx"（這個 enum n8n 真有）。

比 .docx 多的那一層：Excel 檔通常有多個 sheet。n8n Extract from File 的 xlsx operation 預設行為是「讀第一張 sheet」——但學員業務 case 常常不是「永遠取第一張」。

策略 B-1 · 推薦（強制第一次走這條）
取第一張 sheet
適合「.xlsx 永遠單 sheet」或「我手動把要處理的 sheet 移到第一張」的場景。Extract operation 設 xlsx，n8n 預設給第一張。

業務規則層拿到的是第一張 sheet 的 row array——下方 prompt、改名、Write 完全不動。
策略 B-2 · 進階（熟了再升級）
多 sheet 都要
Extract 節點 Options → Sheet → All，output 是「sheet 名稱 → row array」物件。

注意：這會踩到業務規則層——prompt 需要加 sheet 名稱。對非工程師學員，B-1 的成本（手動移 sheet）遠比業務規則層被污染好維護。

(05)

格式 C：.png / .jpg — 最大轉折，要動 lpCall 用 multimodal

■
為什麼圖片格式特殊：前面所有格式（pdf / docx / xlsx / csv / txt / md）都有「文字層」——n8n Extract from File 抽得出文字，Code 節點業務規則層拿到的 input 是字串。

圖片完全相反：沒有文字層。掃描的收據、LINE 截圖、產品照片——Extract from File 出來只有 binary，沒文字。

所以策略要轉折：不再走 Extract from File，改用 LLM multimodal。把圖片 base64 餵進 Gemini 2.5 Flash（lpCall helper），它「真的看圖」回你內容描述。

lpCall 路線選擇

路線 C-1 · 強制走這條
小幅改 lpCall helper（2 行 patch）
在 lpCall 函式內加一段 if (contents) use contents; else use text。

優點：lpCall 從此同時支援 text 和 multimodal，未來其他 workflow 都能用。改動只有 2 行。

為什麼強制走這條：lpCall 加 2 行是「infrastructure 一次性升級」，升級完整個 #03 / #02 / 未來圖片 workflow 都受惠。
路線 C-2 · 不建議
外面構好 contents 直接 httpRequest
放棄 lpCall，在圖片 workflow 直接 this.helpers.httpRequest。

看起來「不動 lpCall」，但失去 lpCall 內建的 retry / 429 / RPM throttle / size check——對每月 200 張掃描收據的學員會撞 RPM 限額。

C-2 看起來簡單，但代價是地雷。

步驟 1：fileSelector 設定（避開 brace expansion 風險）

n8n docs 只保證 * ** ? 三個 wildcard。*.{png,jpg} brace expansion 在不同 fileSelector 實作可能不 match。穩妥做法 2 選 1：

方案 A · 推薦
統一副檔名
把 jpg 全 export 成 png（macOS Preview / Win 內建檢視器都能 export 成 png），fileSelector 統一用 *.png。
方案 B
混合 png/jpg：兩個 Read node
加第二個 Read 節點走 *.jpg，再用 Merge node 合流。Workflow 多 2 個節點但能處理混合。

❌ 不要做的事：直接用 *.{png,jpg} 不測試就交付學員；用 * 然後 Code 節點過濾副檔名（multimodal 對 .pdf / .txt 等非圖片會直接撞 API 錯誤）。

步驟 2：lpCall helper patch（before / after 完整對照，2 行改動）

打開 #03 的 Code 節點，找到 lpCall 函式定義（line 30-60 區）。下方 before/after 只示範改動的兩行段落（// ...省略... 處保持原 jsCode 不動）——學員照位置對齊改即可，不是把整段 paste 取代。

BEFORE（保留 PDF / 文字模式）
async function lpCall({ prompt, gen = {}, _att = 0 }) {
  if (!prompt || typeof prompt !== 'string') return { ok: false, text: '', error: 'empty prompt' };
  try {
    const r = await this.helpers.httpRequest({
      // ...省略...
      body: {
        contents: [{ role: 'user', parts: [{ text: prompt }] }],
        // ...省略...
      },
    });
    // ...
AFTER（同時相容 PDF/文字 + 圖片 multimodal）—— 只動 2 行
async function lpCall({ prompt, contents, gen = {}, _att = 0 }) {
  // 雙路徑：圖片用 contents，文字用 prompt
  if (!prompt && !contents) return { ok: false, text: '', error: 'empty input' };
  try {
    const r = await this.helpers.httpRequest({
      // ...省略...
      body: {
        contents: contents || [{ role: 'user', parts: [{ text: prompt }] }],
        // ...省略...
      },
    });
    // ...

改動只有 2 行（簽名加 contents 參數、body.contents 改條件）。其他 helper 邏輯（retry / 429 / RPM throttle）完全不動。

步驟 3：Code 節點業務段呼叫 multimodal lpCall（完整替換段）

把原本（PDF 路線）的 prompt 構造 + lpCall 呼叫，替換成以下段落：

const imgBase64 = item.binary.data.data;
const imgMime = item.binary.data.mimeType || 'image/png';

if (!lpSizeOK(imgBase64)) {
  results.push({ json: { status: 'failed', originalName, errorMsg: '圖片超過 18MB' } });
  continue;
}

const lpResult = await lpCall.call(this, {
  contents: [{
    role: 'user',
    parts: [
      { text: '分析以下圖片內容，生成語意化檔名（不含副檔名）。格式：YYYYMMDD_主題_關鍵字（繁體中文）。沒日期用 ' + today + '。直接回檔名一行。' },
      { inlineData: { mimeType: imgMime, data: imgBase64 } }
    ]
  }],
  gen: { temperature: 0.3, maxOutputTokens: 200 }
});

// 下方 lpResult.text 解析、檔名清理、results.push 全部不動（業務規則層 0 動）
△
圖片大小注意：lpCall 內建 lpSizeOK()（上限 18MB）。✅ LINE 截圖、手機拍的名片收據（通常 < 2MB）。❌ 動畫 GIF（常 10-30MB）、300dpi 以上全頁掃描（常 15-25MB）、HEIC（iPhone 預設）——這些要學員預先轉格式或降解析度。

9-C：LLM 可能歪掉的 5 種樣子

變體 LLM 回應症狀 學員下一步
A 重寫整份 回 5+ 個節點 JSON，動到 Read / Write / Switch 等不該動的 「請只回 lpCall 函式 patch 的 before/after，其他節點不要動」
B 漏改簽名 給的 contents 構造對，但 lpCall 函式定義沒加 contents 參數 「你忘了在 lpCall 函式簽名加 contents 參數，請補完整 patch」
C 只傳 text 沒傳 inlineData contents 只有 { parts: [{ text: '分析圖片...' }] }，沒給圖片資料 「prompt 沒帶圖片 base64 等於 LLM 看不到圖片，請補 inlineData 段」
D 加 streaming 高玩 加 streamGenerateContent / responseSchema 等本工作流不需要的進階參數 「我們不需要 streaming，請維持原本 generateContent endpoint」
E 自寫 retry/try-catch 在 Code 節點業務規則層加 try/catch 包 lpCall（lpCall 已有 retry） 「lpCall helper 已內建 retry，請移除這層 try-catch」

驗收硬指標（5 條）

  • fileSelector 走方案 A（*.png）或方案 B（兩個 Read node）
    不是 *.{png,jpg} brace expansion。跑 1 張測試圖片有抓到。
  • Extract from File 節點被 disable
    灰色，旁邊有暫停圖示。連線從 Read → Code 直連。
  • lpCall 函式簽名加了 contents 參數
    呼叫端傳 contents 時走 multimodal、傳 prompt 時走原邏輯（兩種都能跑）。
  • 業務規則層保留
    lpCall 的 retry / lpThrottle / lpSizeOK / 檔名清理 + Write 節點都沒動。
  • 真的「看內容」測試通過
    拿一張簡單名片或收據截圖跑 → AI 改名是「能反映內容語意」的合理檔名（例：20260508_名片_張先生、20260508_收據_全聯_385元）——不是檔案 metadata 隨機字串、不是 null_null。
(06)

格式 D/E：.csv / .txt / .md — 最簡單，跳過 Extract

.txt / .md 本身是純文字檔，n8n Read 節點讀進來 binary 解 utf8 就拿到字串——不需要 Extract from File 節點。.csv 有專屬 operation csv，但很多商業 case 只要「整段 CSV 字串塞給 LLM 讓它語意化摘要」——不需要結構化解析，同樣跳過 Extract。

路線 D-1 · 推薦（強制第一次走這條）
純文字 + 跳過 Extract
適合 .txt / .md / .csv「整段當字串餵給 LLM」場景。
· fileSelector：*.txt（單一副檔名）
· Extract from File：disable（保留節點 placeholder）
· 連線：Read → Code 直連
· Code 節點 Extract/Decode 滲入段：binary 解 utf8 拿字串 → 餵 lpCall.prompt
路線 D-2 · 進階
CSV 結構化 + Extract csv operation
適合「.csv 每一 row 各跑一次 LLM」場景。
Extract operation = csv，output 是 row array。
注意：這踩到業務規則層（input 從字串變 row 物件 array，要 loop 每 row 跑 lpCall）。第一次跨 .csv 強制走 D-1。

binary 解 utf8 — 最小 patch（Code 節點 Extract/Decode 滲入段）

原本（PDF 路線）
const pdfText = item.json.text || '';
替換成（純文字路線 D-1）
const rawBinary = item.binary?.data?.data;
const pdfText = rawBinary
  ? Buffer.from(rawBinary, 'base64').toString('utf8')
  : '';
// 注：變數名仍叫 pdfText 是為了業務規則層下方所有引用都不用改
// 雖然語意上現在裝的是 .txt / .md / .csv 內容，
// 但「同一套業務規則處理不同入口格式」就是要讓變數名穩定
brace expansion 同等風險
混合 .txt / .md / .csv 三副檔名一起跑：不要用 *.{txt,md,csv} brace expansion（與 .png/.jpg 同風險）。改用 3 個 Read node 各走一個副檔名 + Merge 合流，或第一次先統一成單一副檔名跑通再擴充。

LLM 最常做的擴張——辨認 + 拒絕

理想回應（9-D） 接受
1. fileSelector：/files/shared/batch-inbox/*.txt
2. Extract from File 節點：建議 disable（右鍵 Deactivate），保留節點 placeholder。連線 Read 直接到 Code。
3. Code 節點最小 patch：
const rawBinary = item.binary?.data?.data;
const pdfText = rawBinary ? Buffer.from(rawBinary, 'base64').toString('utf8') : '';
下方 prompt 構造、lpCall、檔名清理、results.push 全部不動。
歪掉變體（9-D） 拒絕
「為了讓 code 更可讀，我建議：
· pdfText 改名為 fileContent（更通用）
· 把 binary 解 utf8 抽成 helper 函式
· 加一段檢查 binary 編碼（萬一不是 utf8 而是 big5）」
辨認訊號：主動 refactor 變數名、抽 helper、加編碼檢查——全部學員沒要求的。
拒絕：「我故意保留 pdfText 變數名，下方有十幾處引用——改名等於動到業務規則層整段。」

通用驗收 5 條 — 不分格式都要過

    n8n UI 點 Code 節點 → 看 jsCode 的 prompt 字串、改名 template、results.push 結構——跟原 #03 完全一樣（除了圖片或 D-1 純文字路線那 1 行 binary 取值是 Extract/Decode 層滲入，不算業務規則層）
  • fileSelector 對得上格式

  • Extract 節點該 disable 的 disable，該換 operation 的換

  • 跑 1 個測試檔通過 + AI 改名肉眼看合理
    不是 null_null、不是 binary garbage 截字串、檔名語意能反映檔案內容

    少數邊界 case（圖片是純圖無可辨識內容、xlsx 第一張 sheet 真的是空目錄）記備注人工處理，不為 1-2 張改流程（紅線 8 / 第 6 章 walkthrough Step 6）
(10)

常見錯誤 + 怎麼解

錯誤 1
「.docx 我把 fileSelector 改 *.docx + Extract operation 改 text，AI 改名輸出亂猜的字」
症狀
fileSelector + operation 都改了，Extract 沒紅燈，但 AI 改名出來變成 20260508_文件_xml1document 這種看似有抓到內容但跟原 docx 內文無關的字串。
根因
n8n 的 text operation 不會「自動偵測類型抽純文字」——它把檔案 binary 當純文字解。Word .docx 是 zip + XML 結構，text operation 會抽到 zip 內部的 <?xml version="1.0"?>... XML 標籤亂碼。AI 拿到的 input 不是 docx 內文，是 zip XML。
解法
放棄直抽路線，走 9-A 中介格式策略路線 1——Word / Pages / LibreOffice 批次 export 成 PDF，再走既有 #03（fileSelector / Extract operation / 業務規則層全部回到 PDF 路線預設）。
預防
把第 (08) 節 dispatch 對照表貼螢幕邊——.docx 那行明確寫「不能用 Extract」+ 走中介策略。
錯誤 2
「圖片 multimodal 沒看 maxInlineBytes，撞 Gemini 20MB API limit」
症狀
跑了 10 張圖，前 8 張綠燈、第 9 張紅燈，Code 節點錯誤 Request too large 或 Payload exceeds maximum size。
根因
第 9 張可能是 HEIC（iPhone 拍）或 300dpi 全頁掃描，binary > 20 MB。
解法
短期：把那張圖在 macOS Preview 開 → File → Export → 選 JPG 80% 品質，重新放回 inbox。長期：lpCall 內建 lpSizeOK() 預檢已經會把這類撞上限的標 failed——確認 Code 節點呼叫端有用 lpSizeOK(imgBase64) 預檢（看第 (05) 節步驟 3 的範例段落）。
預防
模板 9-C 的 ideal 回應段落有 lpSizeOK 預檢，學員替換 Code 節點時不要把這段砍掉。
錯誤 3
「.xlsx 多 sheet 不知道 LLM 拿到哪張，AI 改名跑出莫名其妙的內容」
症狀
跑了一份 5 sheet 的 .xlsx，AI 改名出來的檔名是 sheet 名「目錄」「索引」這種沒業務意義的字。
根因
n8n Extract from File xlsx operation 預設讀第一張 sheet——學員的 .xlsx 第一張剛好是「目錄」sheet，真正想處理的「客戶清單」在第二張。
解法
（B-1 路線）在 macOS Numbers / Excel 裡把目標 sheet 移到第一張，存檔。或上傳前先存一份只有目標 sheet 的副本（xxx-清單only.xlsx）。
預防
跨 .xlsx 改寫第一次強制走 B-1，把「我要哪張 sheet」這個業務決定離開 workflow。
錯誤 4
「圖片 fileSelector 用了 *.{png,jpg} brace expansion，但跑起來只抓到 .png 不抓 .jpg」
症狀
學員在 batch-inbox 放了 5 張 .png + 5 張 .jpg，跑 Read 節點只跑 5 張（其中一邊副檔名），另一邊全沒進。
根因
n8n docs 只保證 * ** ? 三個 wildcard。*.{png,jpg} brace expansion 在不同 fileSelector 實作不一定 match。
解法
方案 A（推薦）：把 jpg 全 export 成 png，fileSelector 統一用 *.png。
方案 B：workflow 加第二個 Read 節點走 *.jpg，再用 Merge node 合流。
預防
模板 9-C 已強制走方案 A；對照表 .png/.jpg 那行也明確標 ⚠️ 不要用 brace expansion。
