#04 · DAILY AI REPORT · 6 NODES · 進階

每天 8:00 自動讀資料夾、AI 摘要、產 Markdown 日報

這是 #02 / #03 的進階變體：從「處理檔案物理屬性（檔名 / binary）」轉換到「處理檔案語意內容（text）」+ 引入 Schedule Trigger 與 Aggregate 兩個新節點。學完這頁能設計「定時讀多份輸入 → AI 統整 → 結構化輸出」的所有 workflow（早報 / 週報 / 客戶簡報 / 競品分析）。

學習成果
能用 Schedule Trigger + cron expression 設定每天 / 每週 / 每月定時觸發
能用 Aggregate 節點把多個 items 合併成 1 個 array，餵給 AI 做整體摘要
能設計結構化 Markdown 日報 prompt（4 段：摘要 / 待辦 / 決策 / 風險）
(01) WORKFLOW 全貌

節點與資料流

Schedule: 每天 08:00 → Read daily-input → Extract 文字

→ Aggregate 多檔合併 → Code: AI 摘要 + 組 Markdown → Write daily-output

6 節點線性鏈，但比 #02/#03 多一個「Aggregate 多檔合併」節點。為什麼要 aggregate？因為 daily-input/ 內可能有 N 份輸入檔（會議記錄、待辦、便條紙），Read+Extract 後變 N 個 items，Aggregate 把 N 個合併成 1 個 array 一次餵給 AI 做「整體」摘要。如果不 aggregate 直接打 AI，會變成 N 次 API 呼叫各自輸出 N 份小日報。

(02) NODE BREAKDOWN

節點逐一拆解（重點看 1 + 4）

NODE 1 / 6
Schedule: 每天 08:00
scheduleTrigger v1.2cron

關鍵設定：cronExpression: 0 8 * * *（每天早上 8:00）。

cron 5 個欄位：分 時 日 月 週。常用範例：

  • 0 8 * * *：每天早上 8:00
  • 0 18 * * 1-5：週一到週五傍晚 6:00
  • 0 0 * * 0：每週日凌晨 0:00（週報）
  • */5 * * * *：每 5 分鐘（測試用）

注意：Schedule Trigger 必須把 workflow 開 Active才會真的定時跑。沒開 Active 只能手動 Execute。

NODE 2-3 / 6
Read daily-input 所有檔 / Extract 文字

跟 #02 #03 結構類似但抓的是 daily-input/*（不限 .pdf，含 .txt / .md / .docx）。Extract 節點 operation: text（不是 pdf），自動偵測檔案類型抽純文字。

NODE 4 / 6
Aggregate 多檔合併 ← 關鍵節點
aggregate v1

用途：把多個 input items 合併成 1 個 item，把指定欄位變成 array。

關鍵設定：fieldsToAggregate: [fileName, text]

Input：N 個 items 各有 { fileName: "x.md", text: "內容..." }
Output：1 個 item 含 { fileName: ["x.md", "y.md", ...], text: ["內容1", "內容2", ...] }

下游 AI Code 節點只跑一次，把整個 array 當 context 餵給 Gemini，AI 做整體摘要而不是單檔摘要。

NODE 5 / 6
Code: AI 摘要 + 組 Markdown
code v2

結構跟 #02 / #03 類似，差別在 prompt 不要求格式化檔名而是要求 4 段結構化 Markdown：

Prompt 關鍵語句：「請整理成結構化日報（繁體中文 Markdown）。包含：重點摘要 3-5 點、待辦事項（含優先級）、關鍵決策、風險或卡點。日期：YYYY-MM-DD。直接輸出 Markdown，不要 markdown 標記包裹。」

Code 節點 return 一個 item 含 { outputPath, markdown, aiError } + binary 是 Markdown 內容的 base64 編碼（Write 節點寫檔用）。

NODE 6 / 6
Write: daily-output/

fileName: {{ $json.outputPath }} 寫到 shared/daily-output/daily-YYYY-MM-DD.md。

(03) DESIGN RATIONALE

為什麼這樣設計

💭
為什麼 maxOutputTokens 設 2000 而 #02 只設 200？
Markdown 日報含 4 段結構化內容容易 800-1500 tokens。設 2000 留餘裕。同樣設 thinkingBudget: 0 禁 thinking 把整個 quota 給輸出。
⚠️
輸入太少 AI 會幻覺：實測時如果 daily-input/ 內只有 1-2 份簡短測試檔（總字數 < 200），AI 會「合理推斷」造一個虛構故事（虛擬產品名、虛擬待辦）。明天課堂建議用學員真實會議記錄當素材，不要用「test123」這種短測試檔。
(04) 改造方向

5 種改造方向

  1. 改 cron 排程：8:00 換成你的工作節奏（凌晨 6:00、中午 12:00、晚上 23:00）。週報設 0 9 * * 1（每週一早上 9 點）。
  2. 換 prompt 改風格：「給高層看的精煉版（每段 1-2 句）」/「給同事看的詳細版」/「給客戶看的成果報告」。同個架構切換不同對象。
  3. 串 Email 寄日報：在 Write 之後加 Email Send 節點（Gmail / SMTP），每天日報自動寄到自己信箱。
  4. 串 Telegram 推日報：複製 #05 Telegram 節點接在 Code 後面，把 markdown 推到你的 Telegram。
  5. 多語言版本：同個 input 跑 2 次 prompt（中 + 英），輸出兩份 daily-{date}-zh.md / daily-{date}-en.md。
