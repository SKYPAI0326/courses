#09 · GMAIL CATEGORIZE · 8 NODES · 進階 · 需 OAuth 設定

Make 等效的「信箱分類通知」：但跑你本機 + 沒 ops 上限

Make 經典場景（Gmail Trigger → AI 分類 → 多路由通知）的 n8n 對等實作。8 節點：Gmail polling → AI 分 4 類（客訴/詢問/合約/其他）→ Switch 多路由 → 客訴緊急 Telegram、詢問待辦、合約待歸檔。Lite Pack 唯一需要學員自建 Gmail OAuth credential 的 workflow（Lite Pack 無法幫你建，因為要走 Google Cloud Project + OAuth 授權流程半小時）。

學習成果
能在 Google Cloud Console 建 OAuth 2.0 client + 設 redirect URI 給 n8n 用
能用 Gmail Trigger 節點 polling 新信，配合 Switch 節點做多路由分支
能設計「結構化 JSON 輸出 prompt」讓 AI 回穩定可解析的分類結果
(01) 為什麼這個 workflow 重要

Make vs n8n 信箱分類對等度誠實對比

為什麼這頁不藏優劣
「信箱分類通知」是 Make 真的做得很好的場景。如果你只處理 < 300 封信/月、純雲端 + 不在乎本地 AI、信件不含敏感資料 — 繼續用 Make 沒問題。n8n 的優勢只在 3 個情境，下表是不藏的對比。
維度Make（雲端）n8n（本機）
設定難度15 分鐘（Gmail OAuth 一鍵授權）半小時 + 自建 Google Cloud Project
每月處理量免費 333 封信（1000 ops/月 ÷ 3 ops/封）無上限（你電腦電費 + Gemini API 費用）
1000 封/月成本$9 USD/月（Pro 10K ops 方案）~$0（Gemini Flash 免費 tier 通常足夠）
本地 AI 分類不能（必打雲端 OpenAI/Claude）能（換 jsCode 內 URL 為本機 Ollama）
機密信件處理內容必離開本機用本地 LLM 完全不離開本機
附件直接歸檔必經 Google Drive 中轉直接寫本機資料夾（串 #02 / #03）
長期擁有權scenario 是 Make 的workflow 是你的（開源 + 自家硬碟）
當 Make 收掉 / 改價工作流斷不影響
🎯
結論：低量、不在乎機密、純雲端 → Make 簡單。高量（> 1000 封/月）、機密信件、要串本機檔案歸檔 → n8n 是唯一解。本 workflow 做給後者學員。
(02) GMAIL OAUTH 設定（必做、半小時）

前置作業：建 Google Cloud Project + OAuth Client

為什麼比 Telegram 麻煩
Telegram bot token 是純字串貼上就用。Gmail 屬於 Google 帳號級別權限，必須走 OAuth 授權同意流程，需要你建 Google Cloud Project + OAuth client + 加 redirect URI 對應到本機 n8n。第一次設半小時，之後同個 client 可重用。
STEP 1
建 Google Cloud Project
  1. 到 https://console.cloud.google.com 登入你 Gmail
  2. 左上角 Project 下拉 → New Project
  3. 命名 n8n-lite-pack（隨意）→ Create
  4. Project 切到剛建的 n8n-lite-pack
STEP 2
啟用 Gmail API
  1. 左側選單 → APIs & Services → Library
  2. 搜尋 Gmail API → 點進去 → 按 Enable
STEP 3
設定 OAuth consent screen
  1. APIs & Services → OAuth consent screen
  2. User Type 選 External → Create
  3. App name：n8n Lite Pack，User support email：你 Gmail，Developer email：你 Gmail
  4. Scopes：先按 Save and Continue 跳過（OAuth 流程會動態請求）
  5. Test users：加你自己的 Gmail（要授權的那個）
  6. Save → 回 dashboard
STEP 4
建 OAuth 2.0 Client（最關鍵）
  1. APIs & Services → Credentials → Create Credentials → OAuth client ID
  2. Application type 選 Web application
  3. Name 隨意（例如 n8n localhost）
  4. Authorized redirect URIs → + Add URI → 貼這個（必須 100% 一字不差）： 
    http://localhost:5678/rest/oauth2-credential/callback
  5. Create → 跳出對話框含 Client ID + Client secret
  6. 把 client_id + client_secret 複製到記事本
🚨
redirect URI 不能用 https（本機 n8n 沒 SSL）、不能省 port（5678 必須在）、不能換 path（必須 /rest/oauth2-credential/callback）。任何字元錯都會在 Step 5 OAuth 流程跳「redirect_uri_mismatch」錯誤。
STEP 5
在 n8n 內建 Gmail credential
  1. 瀏覽器開 n8n（http://localhost:5678）→ 左側 Credentials → + New
  2. 搜尋 Gmail OAuth2 API → 選
  3. 填 Client ID + Client Secret（從 Step 4 複製來的）
  4. 右下 Connect my account → 跳 Google 授權頁
  5. 選你的 Gmail → 授權同意（會看到「This app isn't verified」警告，因為你是 test user 點 Advanced → Continue）
  6. 授權完跳回 n8n，credential 顯示 「Account connected」
  7. Save credential，命名 My Gmail
STEP 6
把 #09 workflow 內 Gmail Trigger 節點接到剛建的 credential
  1. n8n 進 09 · Gmail 分類通知 workflow
  2. 雙擊 Gmail Trigger: 新信 polling 節點
  3. Credential 下拉選 My Gmail（剛建的）
  4. Save
  5. 右上 Active 開關打開（這個 workflow 預設關，要 active 才會 polling）
(03) WORKFLOW 全貌

節點與資料流

Gmail Trigger（每 N 分鐘 polling） → Code: AI 分類 → Switch: 4 路由

↗ 客訴 → Telegram 緊急
→ 詢問 → Set 待辦
→ 合約 → Set 待歸檔
↘ 其他 → Set 略過

8 節點。Gmail Trigger fan-out N 個 items（每封新信一個），Code 對每封打 AI 分類，Switch 按分類結果送到 4 個輸出。當前 workflow 只對「客訴」推 Telegram，其餘 3 路是 Set 節點當「改造佔位點」（學員自己加 Sheets / Notion / 串 #02 #03 等）。

(04) 重點節點拆解

Code 節點 + Switch 節點是核心

CORE 1 · Code: AI 分類
用結構化 JSON prompt 讓 AI 回穩定可解析格式

關鍵 prompt 設計：

你是公司客服信箱的分類助手。請對下面這封信分類成 4 類之一： - 客訴（angry / 投訴 / 退費 / bug 回報，需立即處理） - 詢問（產品問題 / 報價 / 一般問答） - 合約（合約 / 報價單 / 含 PDF 附件） - 其他（廣告 / 通知 / 系統信 / 訂閱） 回 JSON 格式（不要 markdown 包裹）： {"category": "客訴", "urgency": "high", "reason": "包含投訴用語"} urgency 取值：high / normal / low

為什麼要 4 類不是任意分類：限定 enum 讓 AI 不會發明新類別（如「重要」「待回」這種太模糊的）。下游 Switch 節點才能用固定 4 路精準路由。

fallback 設計：JSON 解析失敗（AI 回了帶解釋的文字 / markdown 包裹）→ 預設 category=「其他」、urgency=「low」。永遠不會炸 workflow。

CORE 2 · Switch: 4 路由
n8n 的 Switch 節點 vs Make 的 Router

Switch v3.2 設定：3 個 rule（客訴/詢問/合約）+ fallbackOutput="extra"（其他）

  • 每個 rule 用 {{ $json.category }} === "客訴" 字串比對
  • fallbackOutput=「extra」表示不匹配前 3 個 rule 的全部走第 4 個 output（其他）
  • renameFallbackOutput 把第 4 個 output 命名為「其他」

vs Make Router：Make Router 一個 input 多個 output，每個 path 是 filter + 後續模組。功能等效但 Make UI 點點選選比較直觀。n8n 的 Switch 寫得清楚但要懂 expression。

(05) DESIGN RATIONALE

設計選擇

💭
為什麼預設 Active = false？
Gmail Trigger 一打開就會立刻 polling 你 Gmail。Lite Pack import 後如果預設 active，credential 還沒設好會立刻噴錯誤。學員設好 credential + 改 polling 頻率合理後再手動 active。
💭
為什麼 polling 預設「每 1 分鐘」？
測試方便（寄一封信 1 分鐘內看到分類結果）。但生產環境會吃 Gmail API 額度（每次 polling = 1 ops），建議改 5-15 分鐘。教學時讓學員體驗即時感，學員自己改。
⚠️
OAuth Test Mode 限制：Step 3 設 OAuth consent screen 為 External 但 publishing status 默認是 Testing。Test mode 只允許 Test Users 列表內的 Gmail 授權（最多 100 個）+ refresh token 7 天過期。要永久使用要 publish app（會跳 verify 流程，個人用無所謂）。
(06) 改造方向

把這個 workflow 改成你業務的 5 種方向

  1. 合約路由串 #02 / #03：合約類信件抽附件 PDF 丟進 ~/Downloads/n8n-starter-kit/shared/pdf-inbox/ → #02 自動 AI 改名 → 你自動拿到歸檔合約。Make 做不到（附件必經 Drive 中轉）。
  2. 詢問路由串 Notion / Sheets：詢問類信件加 row 到 Notion database / Google Sheets 待辦清單，每天早上人工檢視一次（搭 #04 定時 AI 日報摘要 Notion 內容）。
  3. 換本地 LLM：把 Code 節點 jsCode 內 Gemini URL 換成本機 Ollama（例如 http://host.docker.internal:11434/api/generate + Llama 3 model）。**信件內容完全不離開本機**，敏感商務信件適用。
  4. 多 Gmail label 分類：Gmail Trigger 加 filters.labels 設定，只 polling 特定 label（例如 Important）。或不同 label 跑不同 workflow。
  5. 定時轉日報：accumulate 一天的分類結果（寫到本機 JSON 檔），每天 18:00 用 #04 的 Schedule 模式統整成「今日信件分類報告」推 Telegram。
