#01 · WEBHOOK HELLO-WORLD · 2 NODES · 基礎

最簡 webhook 鏈：外部 POST → 接收 → 即時回傳

2 節點、零 AI、零外部依賴的最小 webhook workflow。學會這個就能理解 n8n 三件套：Trigger 節點怎麼接收外部訊號 / 如何用 Expression 引用 Trigger 資料 / 如何用 Respond 回給 caller。所有 webhook-driven workflow（#06 / #07）都建立在這個 pattern 上。

學習成果
能設定 Webhook 節點接收 GET / POST 請求並用 Test URL / Production URL
能用 Expression {{ $json.body.xxx }} 引用 webhook 收到的資料
能用 Set 節點組合即時回應內容（responseMode: lastNode）
(01) WORKFLOW 全貌

節點與資料流

Webhook → Set: 回應內容

這是整個 Lite Pack 最短的 workflow — 只有 2 個節點，而且都是「組態節點」（不需要寫程式）。Webhook 節點負責接收外部 POST/GET，Set 節點組合 JSON 回應，n8n 自動把 Set 的輸出當成 webhook response 回給 caller。

(02) NODE BREAKDOWN

節點逐一拆解

NODE 1 / 2
Webhook（觸發節點）
n8n-nodes-base.webhooktypeVersion 2

用途：對外開一個 URL 接收 HTTP 請求。任何能打 HTTP 的工具（curl / Postman / 別的 SaaS / Make / 你的網站）都能觸發這個 workflow。

關鍵設定：

  • HTTP Method：POST（也可選 GET / PUT / DELETE）
  • Path：hello（決定 URL 後段，本機完整 URL = http://localhost:5678/webhook/hello）
  • Response Mode：Last Node（讓最後一個節點的輸出當作 webhook 回應）

兩種 URL 模式：

  • Test URL（/webhook-test/hello）：開發階段用，按 Execute workflow 後 Webhook 變藍色「Listening」，只接受 1 次請求，收到後 workflow 跑一次然後 Listening 結束
  • Production URL（/webhook/hello）：workflow 必須先 Active 開關打開，才會永久監聽，可接無限次請求
NODE 2 / 2
Set: 回應內容
n8n-nodes-base.settypeVersion 3.4

用途：組合一個 JSON 物件當作 webhook 回應。不需要寫 JavaScript，UI 上點點選選即可。

關鍵設定：在 Assignments 區設 3 個欄位：

  • status = "ok"（固定字串）
  • received_at = {{ $now.toISO() }}（用 n8n 內建 Luxon DateTime 變數取現在時間）
  • echo = {{ $('Webhook').first().json.body }}（把 Webhook 收到的整個 body 原樣 echo 回去）

n8n 自動行為：因為 Webhook 設了 responseMode: lastNode，這個 Set 節點的 JSON 輸出會被當作 webhook response，HTTP 200 回給 caller。

(03) DESIGN RATIONALE

為什麼這樣設計

💭
為什麼用 Set 節點而不是 Code 節點寫 JS？
對「接收 → 組合回應」這種純資料任務，Set 節點 UI 點點選選比寫 JS 直觀很多 + 對下一個學員 fork 改造比較友善。Code 節點留給「真的需要邏輯運算」的場景（API 調用、條件判斷、loop）。
💭
為什麼用 lastNode 模式而不是 onReceived？
n8n Webhook 有 3 種 responseMode：onReceived（立刻回 200，後面節點背景跑）/ lastNode（等所有節點跑完用最後一個節點輸出當回應）/ responseNode（指定某個 Respond to Webhook 節點）。本範例用 lastNode 是因為要回傳 echo 後的結果給 caller 看，需要等 Set 跑完。
⚠️
Test URL 一次性的雷：學員常見錯誤 — 按完 Execute workflow 等很久才打 curl，Webhook 已經 timeout 不在 Listening 了。Test URL 在 Execute 後通常只 listen 約 60 秒，超時就要重按 Execute。Production 模式就沒這限制。
(04) 改造方向

把這個 workflow 改成你業務的 4 種方向

  1. 加身分驗證：Webhook 節點 Authentication 設 Header Auth / Basic Auth / JWT，避免任何人都能打你的 endpoint。學員自己網站對這個 webhook POST 訂單時用 Header Auth 帶 secret。
  2. echo 改成資料驗證：把 Set 節點換成 If 節點，檢查 body 內必填欄位是否齊備（如 email / name）→ 缺失就回 400 錯誤訊息。
  3. 串資料庫：Set 節點後加 PostgreSQL / MySQL / Notion 節點，把 webhook 收到的資料寫進你的資料庫。
  4. 串 Telegram 通知：複製 #05 的 Telegram 節點接在 Set 後面，每次有人打這個 webhook 你手機就收到通知。
