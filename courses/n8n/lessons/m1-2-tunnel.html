SECTION 1.2

Cloudflare Tunnel 設定

為本機 n8n 建立一個公開網址，讓 Make、Google Forms 等雲端服務可以傳送 Webhook 到你的電腦。本節主流程用 Cloudflare Quick Tunnel：不需註冊、不需綁卡、不需自有網域，5 分鐘完成。

⏱主流程 5 分鐘 / 進階 30 分鐘
📋主流程 3 步驟
✅前置：完成 1.1 n8n 安裝
為什麼需要這個步驟

n8n 裝在你的電腦裡，但 Make 或 Google Forms 在網際網路上「看不到」你的電腦——它沒有公開的 IP，也沒有網址。Cloudflare Tunnel 解決這個問題：它幫你的本機 n8n 建立一個公開網址，雲端服務就能透過這個網址把任務傳進來。

本節主流程：用 Cloudflare 官方的 Quick Tunnel（trycloudflare.com）一行指令拿公開 URL，5 分鐘搞定。不需 Cloudflare 帳號、不需綁卡、不需自有網域。代價是 URL 每次重啟會變——對課程練習、課堂演練、開發測試完全夠用。
若需要長期穩定的固定網址（生產環境、客戶整合等），請看本頁底部「進階：完整 Tunnel + 自有網域」折疊段落。

1
安裝 cloudflared 命令列工具

cloudflared 是 Cloudflare 的官方工具，負責把本機連到 Cloudflare 邊緣節點建立 Tunnel。Mac 與 Windows 安裝路徑不同。

# Mac（推薦：Homebrew 安裝。沒裝 Homebrew 先到 brew.sh 裝） brew install cloudflared
🍎 Mac 不熟悉終端機？到 cloudflared GitHub Releases 下載 cloudflared-darwin-amd64.tgz（Intel）或 cloudflared-darwin-arm64.tgz（Apple Silicon），解壓後雙擊安裝。
🪟 Windows：到 cloudflared Releases 下載 cloudflared-windows-amd64.exe，改名為 cloudflared.exe 放到 C:\Windows\System32\（這樣命令列任何位置都能呼叫）。

確認安裝成功：在終端機 / 命令列輸入 cloudflared --version，應該顯示版本號（例如 2026.3.0）。

2
雙擊 tunnel-quick 取得公開 URL

前置：n8n 已啟動（雙擊 start.command 看到「✅ n8n 已啟動」）。在 ~/n8n-starter-kit/ 試跑包資料夾雙擊：

  • Mac：tunnel-quick.command
  • Windows：tunnel-quick.bat

終端機會顯示一行類似：

https://recycling-durham-capacity-patches.trycloudflare.com

請複製這串 URL 保留——下一步會用到。

⚠️ 保持終端機視窗開著，URL 才會持續有效。關閉視窗或按 Ctrl+C → URL 立即失效。每次重新雙擊 URL 都會變。
🍎 macOS 第一次雙擊被擋是正常的：跟你第一次雙擊 start.command 一樣處理——開「系統設定 → 隱私權與安全性」捲到底「安全性」區塊 → 看到被擋的 tunnel-quick.command → 按「強制打開」→ 確認密碼。
🛠 內建防呆：腳本預設用 HTTP/2 協議（避開家用 ISP / 公司網路常擋的 QUIC/UDP 7844）。30 秒沒拿到 URL 會自動跳對話框建議改 ngrok 備用方案。
3
用 Tunnel URL 觸發本機 n8n Webhook

驗證 Tunnel 真的能把外部請求傳到本機 n8n。順序很重要——5 個動作要連續做完，不能停太久：

  1. 瀏覽器打開 http://localhost:5678 進入 n8n 編輯器
  2. 打開 1.1.2 建好的 Webhook workflow
  3. 按畫面正下方紫色「Execute workflow」按鈕（不是節點裡面那個）
  4. Webhook 節點變藍色「Listening for test event」（等候狀態，1 分鐘內有效）
  5. 立刻切到新分頁，貼上你的 Quick Tunnel URL + webhook test path：
https://你的trycloudflare網址.trycloudflare.com/webhook-test/hello

瀏覽器應該顯示：

{"message": "Workflow was started"}
✅ 切回 n8n 編輯器確認三件事：(1) Webhook 節點變綠色；(2) Output 面板出現 headers / query / body 資料結構；(3) cloudflared 終端機視窗多了 INF log 紀錄這次 GET 請求。三件事齊備就完成 1.2 驗收。
⏱ Test mode 是一次性：n8n Webhook 收到一次 test event 後等候狀態自動關閉，要再測就重按「Execute workflow」按鈕。要永久接收請把 workflow 切到「Active」並改用 /webhook/hello（不是 /webhook-test/hello）。
完成驗證 CHECKPOINT
三件事齊備才算 1.2 完成

(1) cloudflared 終端機視窗顯示 trycloudflare.com URL 並持續運作；(2) 用 Tunnel URL 打 webhook test path，瀏覽器顯示 {"message": "Workflow was started"}；(3) 切回 n8n 編輯器，Webhook 節點變綠色 + Output 面板有 headers / query / body 資料結構。

主流程常見問題
tunnel-quick 跑出 control stream encountered a failure while serving 連續 retry
你的網路環境（家用 ISP / 公司網路 / VPN）擋了 cloudflared 預設的 QUIC 協議（UDP 7844）。新版 tunnel-quick.command 已預設改用 HTTP/2，30 秒沒拿到 URL 會自動跳對話框建議改 ngrok。請先重新從 1.1.2 講義頁下載最新試跑包覆蓋整個資料夾，再雙擊 tunnel-quick.command。
瀏覽器打 webhook URL 顯示 {"code": 404, "message": "The requested webhook 'hello' is not registered"}
n8n Webhook 節點沒有處於「Listening for test event」等候狀態。回 n8n 編輯器按下方紫色「Execute workflow」按鈕，立刻（10 秒內）切到瀏覽器分頁打 Tunnel URL。Test webhook 是一次性，每次測都要重按按鈕。
cloudflared 視窗關掉後 URL 失效
這是預期行為——Quick Tunnel 跟 cloudflared 進程綁定，視窗關閉 URL 就失效。下次再雙擊 tunnel-quick.command 會拿到新的不同 URL。如果需要長期穩定 URL，看下方「進階：完整 Tunnel + 自有網域」段落。
tunnel-quick 報錯「找不到 cloudflared 指令」
回 Step 1 確認 cloudflared 安裝成功。終端機跑 cloudflared --version 應該看到版本號。Mac 用 brew 最穩；Windows 把 cloudflared.exe 放到 C:\Windows\System32\ 才能全域呼叫。
⚙️ 進階：完整 Tunnel + 自有網域（要長期穩定 URL 才看）

什麼時候才需要走完整 Tunnel？

  • 需要永久穩定的 URL 給客戶 / 第三方整合
  • 要把 Make scenario 部署成生產環境（不能每天改網址）
  • 需要 Cloudflare Access / Zero Trust 等進階保護

兩層門檻先確認：

  1. 需要綁信用卡：Cloudflare 自 2024 起 Zero Trust 免費方案首次設定要求綁卡作 anti-abuse（不會在免費額度內扣款；50 users / Tunnel 流量無限）
  2. 需要自有網域：Public Hostname 要從你已託管在 Cloudflare 的網域選一個（沒網域則買一個 ~$10/年，例如 Namecheap / Cloudflare Registrar）

完整流程（7 步驟）：

  1. 建立 Cloudflare 帳號 + 綁卡 → 進入 Zero Trust dashboard（one.dash.cloudflare.com）
  2. 左側 Networks → Tunnels → Create a tunnel → 取名（例如 n8n-local）
  3. 跟著頁面提示安裝 cloudflared 並執行 service install 命令（含 Token）。Mac 用 brew install cloudflared；Windows 下載 cloudflared.exe，命令提示字元跑 cloudflared.exe service install <你的token>
  4. 確認 Tunnel 顯示「Connected」（dashboard 連線狀態 + 終端機顯示 Connection registered）
  5. Public Hostnames → Add a public hostname → 選你的網域 → 子網域 n8n-webhook → Service 選 HTTP → URL 填 localhost:5678
  6. 取得永久網址（如 n8n-webhook.your-domain.com）
  7. 用此網址測試 webhook（同主流程 Step 3）
⚠️ 同樣會碰到 QUIC 被擋的問題：cloudflared 的 service install 啟動方式預設仍用 QUIC。若連線失敗，需手動加 --protocol http2 參數，或在 cloudflared config.yml 設定 protocol: http2。

進階子單元：完整 Tunnel 設定後可看 cloudflared 自動啟動（Mac launchd / Win 系統服務）與 多 Hostname 管理。課程主流程跑 Quick Tunnel 不會用到這兩個。

備用方案 · NGROK FALLBACK

若 Quick Tunnel 在你的網路環境完全跑不通（即使 HTTP/2 仍 retry loop），表示 cloudflared outbound 連線被完全擋。改用 ngrok（架構不同，走 TCP HTTPS 反代）通常能繞過。

  1. 前往 ngrok.com 註冊免費帳號（不需綁卡）
  2. 下載 ngrok 並解壓
  3. 終端機跑 ./ngrok http 5678（5678 是 n8n port）
  4. 終端機顯示類似 https://xxxx.ngrok-free.app，把它當主流程 Step 3 的 Tunnel URL 用
⚠️ ngrok free plan 兩個限制：(1) 每次重啟 URL 會變（同 Quick Tunnel）；(2) 第一次用瀏覽器訪問會出現警告頁，加 header ngrok-skip-browser-warning: true 即可跳過。
安全提醒 · TUNNEL SECURITY

把本機 n8n 透過 Tunnel 暴露到外網等於開了一道對外的門，請記住四件事：

  1. 測試期可使用隨機 / 不易猜的 webhook path（例如 UUID），降低被機器人掃到的機率。
  2. 正式使用一定要加驗證——n8n Webhook 節點的 Header Auth / Basic Auth、Cloudflare Access 或 token query string 三選一，切勿讓 webhook 裸奔。
  3. API key、Webhook secret、token 一律用 n8n Credentials 或 .env 管理，不要直接寫在節點的文字欄裡（會被 export 出去）。
  4. 含 credential 的 workflow JSON 不公開分享；分享前用 n8n 內建「Reset Credentials」清除敏感欄位。
💼 實戰案例
情境｜某攝影工作室把 Google Forms 客戶訂單表 → Cloudflare Tunnel → 本機 n8n 自動建專案資料夾 + 下載客戶上傳素材。
做法｜課程演練階段用 Quick Tunnel 驗證流程通暢；正式上線改買網域走完整 Tunnel 拿到永久 URL（一年約 300 元）讓網址不變。
踩雷｜家裡 ISP / 公司網路常擋 cloudflared 預設的 QUIC 協議。新版 starter-kit 預設改 HTTP/2 已解決大部分情境；少數仍擋的環境改用 ngrok 備用方案。
