步驟 4-6 詳解 — 匯入測試、錯誤回報、收斂

LLM 給了你改好的節點區段，接下來是你的主場。步驟 4-6 是流程裡最考驗耐心的段落：每次只跑 1 筆、出錯就精準回報、三輪收斂不過就退回起點。這三步做好，改出來的 workflow 才真的能用。

本頁重點 · OUTCOMES
掌握步驟 4 隔離環境動線（紅線 7），確保改動不碰到 production 流量
能用「三件套」格式（紅燈訊息＋input＋parameter）精準回報錯誤，不再整份貼給 LLM 重寫
懂得在步驟 6 判斷「LLM 越改越複雜」的訊號，並用模板 5 強制收斂
(01)

步驟 4：匯入測試（隔離環境，學員主導）

為什麼這步不讓 LLM 介入
步驟 4 是六步流程唯一一個 LLM 完全不在的步驟。你從步驟 3 拿到 LLM 給的節點區段、在 n8n UI 替換完之後，剩下的事只有一件：用最小樣本跑一次，看對不對。這步不需要 AI，需要你的眼睛。
面向 說明
動作 在 n8n UI 點「Execute Workflow」，或用 trigger 餵一筆樣本資料，跑一次完整流程。
LLM 做什麼 不做。LLM 在這步沒有角色。
你做什麼 用最小樣本（1 個檔案 / 1 列資料 / 1 封信）跑一次。逐一檢查每個節點的 output panel，確認資料形狀符合預期。
驗收訊號 所有節點都綠燈，最後一個節點的 output 結構與步驟 1 便箋上寫的「目標 output」一致。
雷區 用大批量測試（例：直接把 200 張發票全丟進去）——錯誤會被埋在中段，難以追蹤。永遠先用 1 筆。

隔離環境動線（紅線 7）

這是步驟 4 最容易被跳過、後果也最嚴重的一個細節。n8n 匯入流程預設保留原 workflow 的 active 狀態，如果你直接在 production workflow 上替換節點、馬上跑，就是用未驗證版本接收真實流量或真實檔案。

正確的隔離環境動線（紅線 7），依序做這四步，一步不能跳：

a
在 n8n UI 對著原 workflow 點右上角「Duplicate」，複製一份，命名為 <原名>-edit（例：02-pdf-ai-rename-edit）
b
把 LLM 改好的節點區段套到這個複本上，不碰原 workflow
c
確認複本的 trigger 是 OFF 或 Manual——開著的 trigger 會在你不注意時自動執行
d
複本跑通、結果驗收 OK 之後，才把改動複製回原 workflow
任一步跳過的代價
你在用真客戶資料、真收件人、真資料庫當白老鼠。寄錯信、刪錯檔、寫錯記錄——這些操作沒有 Ctrl+Z。隔離環境不是多做一步，是省掉收拾爛攤子的幾個小時。

步驟 4 這步學員完全手動操作，不需要任何 prompt，跳過「模板」直接進 n8n UI 執行。

(02)

步驟 5：錯誤回報（最小可重現訊息）

這步的核心原則
步驟 4 出現紅燈，你要做的不是把整份 workflow 重貼給 LLM。你要做的是「把 LLM 帶到紅燈的那一節點前」，讓它看到同樣的輸入和同樣的錯誤——這叫「最小可重現訊息」。範圍越小，LLM 診斷越準，你修的次數越少。
面向 說明
動作 步驟 4 失敗時，學員組裝「最小可重現訊息」給 LLM，請它指出問題在哪個欄位、最小修改是什麼。
LLM 做什麼 根據錯誤訊息三件套提出診斷，指出問題節點和欄位。若 LLM 回「我不確定」，代表你提供的資訊不夠，要補資料而不是換另一個 LLM。
你做什麼 複製三件事貼給 LLM：(a) 紅燈節點的錯誤訊息全文，(b) 該節點的 input panel 內容或 JSON，(c) 該節點當前的 parameter 設定 JSON（已移除 credential）。
驗收訊號 LLM 能明確指出問題在哪個節點、哪個欄位。如果無法指出，說明回報資訊不足，補充資料後再問一次。
雷區 兩個：(1) 只說「跑不出來」沒有附任何訊息；(2) 把整份 workflow 重貼給 LLM 請它重寫。兩者都讓 LLM 從頭猜，等於退回步驟 3 以前。

模板 4：錯誤回報模板

以下是步驟 5 的標準 prompt。三件套缺一件，LLM 的診斷品質就會明顯下降。

PROMPT TEMPLATE 模板 4 — 搭配步驟 5（錯誤回報）
我是 n8n 非工程師使用者，請以結果為主、不解釋原理。 我把節點 {N} 替換成你上一輪給的內容後，匯入跑不過。 錯誤訊息：{紅燈節點訊息全文} 該節點 input：{input panel 內容/JSON} 該節點當前 parameter：{節點 parameter JSON，已移除 credential} 請只指出：問題在哪個欄位、最小修改是什麼。 不要重寫節點、不要建議換做法。
常見錯誤示範
直接說「你給我的東西跑不過，請幫我重新改一次」——LLM 只會再猜一版，可能和原本一樣或更離題。沒有三件套，LLM 無從診斷，你只是在消耗輪次。

兩條不能踩的雷

不貼 credential
把 parameter JSON 複製出來前，搜尋確認沒有 apiKey、accessToken、password 這類字串殘留。n8n UI 匯出時選「export without credentials」已幫你過濾，但節點 parameter 要手動確認。
不貼整份 workflow
「整份重貼請 LLM 重寫」是紅線 2 的違反，同時也是步驟 5 最常見的跳步。錯誤回報的價值在「縮小範圍到單一節點」——範圍縮小了，修改範圍才能精準，才不會一改又壞掉其他節點。
(03)

步驟 6：收斂（10 筆 → 200 筆）

收斂是六步流程的最後一哩
步驟 3-5 是可以反覆迴圈的：改節點 → 測試失敗 → 精準回報 → 再改。步驟 6 的任務是管控這個迴圈不要無限延伸——最多三輪，三輪不過就退回步驟 1 重看需求。通過之後，從 1 筆小樣本擴大到 10 筆、再到全量。
面向 說明
動作 重複步驟 3-5 直到通過，每次迴圈不超過三輪。三輪不過停下來，退回步驟 1 重看需求，不是繼續跑第四輪。
LLM 做什麼 在步驟 3（限定修改範圍）和步驟 5（錯誤回報）內被呼叫。步驟 6 本身是你的判斷，不是 LLM 的。
你做什麼 每輪結束問自己「我比上一輪更接近目標嗎？」如果答否，停。先跑 10 筆真實樣本，output 全對之後才放全量（例：200 張發票）。
驗收訊號 用真實樣本（≥ 10 筆）跑通，output 結構與步驟 1 便箋一致。通過之後才跑全量。
雷區 「LLM 越改越複雜」——LLM 傾向加防禦邏輯與分支。如果第三次回覆比第一次長 30%，停下來用模板 5 強制收斂。

學員姿態示範：當 LLM 想加 try/catch

收斂過程中很容易遇到這個情境：你在步驟 5 回報了一個錯誤，LLM 給你的不只是修正，還附上 try/catch 防禦邏輯、條件分支、額外的 error handler 節點。這不一定是壞事，但它讓你的 workflow 變複雜，驗收難度也跟著升高。

學員應有的姿態：

LLM 說的
「我幫你加了一個 try/catch，如果 PDF 讀取失敗會自動記錄到 error log，並繼續處理下一筆。」
你的回覆
「謝謝，但我目前只需要修好剛才那個欄位錯誤，不需要加防禦邏輯。請保留最小修改，移除你新增的 try/catch 和 error handler，再給我一次。」

這個「學員拒絕 LLM 的建議」不是在跟 AI 鬥嘴，是在保護你自己：越簡單的 workflow 越好驗收、越好 debug。防禦邏輯留到你對這個 workflow 的基本行為已經完全掌握之後，再加也不遲。

模板 5：收斂止血，當 LLM 越改越複雜時用

這個模板的使用時機是：你發現 LLM 的最新版本比最初版本長了很多，或者多了很多你原本需求裡沒有提到的節點和邏輯。

PROMPT TEMPLATE 模板 5 — 搭配步驟 6（收斂止血）
請用一句話比較你最早給我的版本與現在這版， 哪些是「為了原始需求必要的」、哪些是「你新增的防禦邏輯」？ 請保留必要的、移除新增的，再給我一次。
常見錯誤示範
讓 LLM 一直加：每次報錯就接受 LLM 加的所有防禦措施，到第三輪時 workflow 已經是原本的兩倍大，你完全不知道哪個部分是為了你的需求、哪個部分是 LLM 自己的防禦習慣。這時想退回去很難。
三輪收斂上限的意義
三輪迴圈（步驟 3 → 4 → 5）不過，不是你失敗了，是需求或方向出了問題。退回步驟 1 的便箋，重看「input / output / 保留節點」有沒有哪格寫模糊。通常三輪收不斂，問題都在步驟 1 的便箋沒填清楚。
(04)

收斂後的 Destructive 操作補充（紅線 8）

步驟 6 跑通之後，還有一個容易被忽略的環節：如果你的 workflow 裡有任何「會對真實世界產生不可逆影響」的節點——刪資料（Notion delete / Drive delete）、寄信（Gmail / Telegram 發給真實收件人）、更新記錄（HubSpot / Pipedrive update）——在切換到全量之前，必須先做 dry-run（紅線 8）。

10 筆樣本跑通只代表「資料形狀對了」，不代表「收件人對了」或「刪的是你要刪的那筆記錄」。紅線 8 的正確流程：先把 destructive 節點 disable 或換成 console log 等價假動作，確認 input/output 資料形狀全程正確之後，再把收件人改成自己的測試帳號跑 1-3 筆，最後才換回真收件人 / 真資料源。省掉這 5 分鐘，後果可能要花幾小時甚至幾天補救。
