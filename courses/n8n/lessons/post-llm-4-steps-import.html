步驟 4-6 真詳解 — 匯入測試、錯誤回報、收斂

LLM 給了你改好的節點區段，接下來是你的主場。步驟 4 verify 你的 Duplicate 有沒有做對；步驟 5 學會分辨「精準回答」和「太模糊」；步驟 6 用可數的 proxy 判斷「比上輪更近還是更遠」，每輪有憑有據。

本頁重點 · OUTCOMES
步驟 4：verify Duplicate 三硬指標（節點數 / credentials / Active OFF），確認複本健全再跑
步驟 5：用「三件套」格式回報，並能分辨精準型 vs 太模糊三種變體，知道各自怎麼追問
步驟 6：用可數 proxy 表（紅燈數 / null 數 / 行數 / 動節點數）判斷近遠，全量跑完走 Executions tab 驗收
!
和第 6 章 Phase C walkthrough 的順序不同，請以本章為準。
第 6 章 Phase C 為了示範「學員第一次很可能忘記順序」這個常見錯誤，故意演了「先替換到原版才想到要 Duplicate」的劇情並補救。那是教學示範，不是 best practice。
本章（第 3 + 4 章）教的流程是：先 Duplicate（第 3 章 step 6）→ 替換節點 → 再來本頁 verify。正確順序以本章為準。
(01)

步驟 4：匯入測試（全人工，約 10 分鐘）

Duplicate 已在第 3 章 step 6 做完，本步只 verify
你從步驟 3 拿到 LLM 給的節點區段、在 n8n UI 替換進 -edit 複本之後，下一件事不是直接點 Execute——而是先確認複本本身是健全的。三條硬指標任一不過，退回第 3 章 step 6 重做 Duplicate，不要跳過。

4.1 — verify Duplicate 結果（三硬指標）

在 n8n UI Workflows 列表找到 02-pdf-ai-rename-edit，點開，依序確認：

a
節點數與原版一致
在原版開一個分頁對照數一遍。例如原版 5 個節點，-edit 也要是 5 個。
驗收：兩個分頁節點數字相同
b
credentials 跟著來了
點任一含 credential 的節點（例：HTTP Request、Gemini、Google Drive），看 Credentials 欄位是否還掛著原本那個。n8n Duplicate 預設會繼承；若顯示「Select Credential」表示沒繼承，要手動選回去。
驗收：Credentials 欄位顯示具體帳號名，不是「Select Credential」
c
Active 開關是灰色 OFF
Duplicate 預設關，但仍要肉眼確認。看 workflow 右上角的 Active 開關。
驗收：開關文字顯示「Inactive」或「OFF」，灰色
任一條不過 → 退回
退回第 3 章 step 6 重做 Duplicate，不要在不健全的複本上繼續跑。複本有問題，後面所有步驟的結果都不可信。

4.2 — 準備測試樣本並執行

1
準備 1 張測試 PDF
把 1 張你已經有的真實供應商發票（或任何舊合約 PDF 當煙霧測試）放到 /files/pdf-inbox/。
驗收：n8n UI 上 Read PDF 節點點 Execute Step 能列出這張 PDF
2
點 Execute Workflow
n8n UI 下方紫色「Execute Workflow」按鈕。
驗收：5 個節點全部變綠燈
3
點每個節點看 output panel
點節點 → 右側 panel 上方 4 個 tab：Input / Output / Settings / 其他 → 切到 Output → 看 JSON 樹。
驗收：最後一個節點 (Write) output 裡，newFilename 欄位值符合便箋上寫的命名規則
4
去 Finder 確認
打開 /files/pdf-renamed/。
驗收：看到 1 張改好名的 PDF，檔名是預期格式

步驟 4 全程手動，不需要任何 prompt。跳過「模板」直接進 n8n UI 執行。

(02)

步驟 5：錯誤回報（5–15 分鐘 / 輪）

進場條件：步驟 4 出現紅燈節點，或 output 裡有 null / 空字串
你要做的不是把整份 workflow 重貼給 LLM。你要做的是「把 LLM 帶到紅燈的那一節點前」，讓它看到同樣的輸入和同樣的錯誤。範圍越小，LLM 診斷越準，你修的次數越少。

三件套：怎麼收集 + 怎麼貼

1
複製錯誤訊息全文
點紅燈節點 → 下方面板跳出紅字 → 滑鼠選取整段 → Cmd+C。
驗收：複製到的文字含「Error:」或具體訊息（如「Cannot read property 'X' of undefined」）
2
複製該節點 input panel（SOP-B）
點紅燈節點 → 右側 panel 切到 Input tab → toggle 切到 JSON 模式（不是 Schema / Table）→ 右上角複製按鈕。
若 items > 3 筆：只貼前 2 筆，刪掉 binary 大段 base64，搜尋刪掉 apiKey / accessToken / Bearer。
驗收：你能讀懂這段 JSON 的鍵名 + 範例值
3
複製該節點 parameter（SOP-C）
點節點本身 → Cmd+C（n8n 複製整個節點 JSON）；或點 ⋮ → 「Copy parameters」。
貼之前搜尋 apiKey / accessToken / password / token：全找不到才貼，找到就把 value 改成 "REDACTED"。
驗收：JSON 開頭是 { "parameters": ... 或 { "name": "...", "parameters": ...

模板 4：錯誤回報模板

PROMPT TEMPLATE 模板 4 — 搭配步驟 5（錯誤回報）
我是 n8n 非工程師使用者，請以結果為主、不解釋原理。 我把節點「Code: AI 改名」替換成你上一輪給的內容後，匯入跑不過。 錯誤訊息（紅燈節點底下的紅字全文）： [貼錯誤訊息] 該節點 input（從 Input panel 複製的 JSON）： [貼 input JSON] ← 走 SOP-B（前 2 筆、刪 binary、刪 credential） 該節點當前 parameter（從 Settings 複製的 JSON，已移除 credential）： [貼 parameter JSON] ← 走 SOP-C（搜尋 apiKey 等 4 字串確認都找不到） 請只指出： 1. 問題在哪個欄位（精確到行） 2. 最小修改是什麼（給出具體新值） 不要重寫整個節點、不要建議換做法、不要加 try/catch。
常見錯誤示範
直接說「你給我的東西跑不過，請幫我重新改一次」——LLM 只會再猜一版，可能和原本一樣或更離題。三件套缺一件，LLM 無從診斷，你只是在消耗輪次。

LLM 回應辨認：精準型 vs 太模糊三變體

貼出模板 4 之後，LLM 的回應大致分四種。辨認方法如下：

精準型（過關）— 長這樣
「問題在 parameter 的 jsCode 第 3 行的 prompt 字串。第 3 行目前寫：const prompt = '請從以下發票文字抽取：供應商名稱、稅後金額...'；input panel 顯示 pdfTextPreview 的內容是『賣方：XX 公司，稅後合計：1234 元』。最小修改：第 3 行的『供應商名稱』改成『賣方公司全名』，『稅後金額』改成『稅後合計』。其他不動。」

三要素都有：(a) 指出具體欄位名 (b) 給出具體修改值 (c) 引用 input 內容當證據。
變體 X — 純機率判斷（最常見，要追問）
「這個錯誤訊息看起來是 Gemini API 沒有正確回傳預期格式。建議檢查 Code 節點的 prompt 寫法，可能要重新調整提示詞，或者考慮加上錯誤處理機制讓流程更穩定。」

辨認訊號：出現「可能」「建議」「看起來」「或許」「考慮加上」+ 沒指具體欄位名 + 沒給具體新值。
追問：「請告訴我具體是 prompt 的哪一行需要改、改成什麼新字串。不要建議『調整提示詞』，給我精確的修改。」
變體 Y — 列多個可能根因（要追問）
「這個錯誤可能的原因有以下幾個：1. prompt 不夠精確，2. PDF 解析可能有問題，3. JSON.parse 失敗，4. fallback 邏輯沒處理某些邊界情況。不排除是綜合因素，建議逐一排除。」

辨認訊號：「可能的原因有以下幾個」+ 條列 3 條以上 + 結尾「不排除」「綜合因素」「建議逐一排除」。你不是工程師，沒有時間「逐一排除」。
追問：「請看『錯誤訊息全文 + input panel』兩件事，只挑 1 個最可能的根因告訴我，並指出 parameter 的哪一行要改成什麼。如果你不確定，請告訴我『需要再給你哪個面板才能判斷』，不要列 4 個可能性讓我自己挑。」
i
變體 Z — LLM 反問要更多面板（合理，但回得對）
「我需要再看一下：Read PDF 節點的 output 是否真的有 pdfTextPreview 欄位？不然光看 Code 節點的 input 我無法判斷是上游沒給對還是 Code 節點本身寫錯。」

這是好回應，不亂猜。但你要用「模板 4-延伸」：問 LLM「為什麼需要看這個——一句話」，並只給那 1 個節點對應的面板，不貼整份 workflow。
模板 4-延伸：「你需要看上下游節點才能診斷？請告訴我你想看的是哪個節點的哪個面板，以及為什麼。我只貼那一個節點，不貼整份 workflow。」
(03)

步驟 6：收斂（最多 3 輪 + 全量執行）

你是把關人，每輪結束問自己「比上輪更近還是更遠」
測試-回報-收斂這 3 步是「迴圈閥」：LLM 給的 JSON 是草稿，跑過真實樣本才是答案。每輪結束用下方可數 proxy 表決定「近 / 遠 / 持平」，不靠感覺判斷。

6.1 — 可數 proxy 表：「近」vs「遠」判斷

指標 近（= 進展） 遠（= 停下檢查）
紅燈節點數 比上輪少（例：2 → 1） 出現上輪沒有的新紅燈（問題被搬家，不是進展）
output null / 空字串數 欄位從 null 變成有值（即使值不對也算近） 原本有值的欄位變成 null（退步）
jsCode 行數 持平或變少 比上輪多 30% 以上（LLM 在加防禦邏輯，不是改根因）
LLM 動到的節點數 比上輪少（逼近根因） 比上輪多（範圍蔓延）

紅燈數一樣 + null 欄位一樣 + 行數差不多 → 算「持平」，記為 1 輪但不算進步。持平 2 輪也要停下退回步驟 1 重看便箋。

6.2 — 接受 / 拒絕判斷樹

拒絕
LLM 提案動到「節點數量」（新增 / 刪除節點）→ 立刻拒絕，用模板 5
拒絕
LLM 提案加 try/catch、if 條件、error handler、retry → 拒絕
可接受
LLM 只改原節點裡的「值」（字串、數字、欄位 expression）→ 可接受
可接受
LLM 在現有 Code 節點裡加 console.log 幫你 debug → 可接受（debug 後記得刪）

模板 5：收斂止血，當 LLM 越改越複雜時用

PROMPT TEMPLATE 模板 5 — 搭配步驟 6（收斂止血）
請用一句話比較你最早給我的版本與現在這版： - 哪些是「為了我原始需求必要的」？ - 哪些是「你新增的防禦邏輯」（try/catch、if 條件、error handler、新節點）？ 請保留必要的、移除新增的，再給我一次「只有 Code 節點」的 JSON 區段。
常見錯誤示範
每次報錯就接受 LLM 加的所有防禦措施，到第三輪時 workflow 已經是原本的兩倍大，你完全不知道哪個部分是為了你的需求、哪個部分是 LLM 自己的防禦習慣。這時想退回去很難。

6.3 — 3 輪不收斂的處理

停止 LLM 對話 → 退回步驟 1，看 4 格便箋是否有「目標 input」格寫得太模糊。常見原因：input 那格只寫「發票 PDF」沒寫具體欄位。通常三輪收不斂，問題都在步驟 1 的便箋沒填清楚。

6.4 — 小批量 → 全量驗收動線

1
小批量測試（10 筆）
從 200 張裡隨機挑 10 張不同供應商的發票放 /files/pdf-inbox/，Execute Workflow。
驗收三條：(a) 10 張全綠燈 (b) 10 張的 newFilename 都符合便箋格式 (c) 沒有 null
2
全量執行（200 張）
把 200 張全放 /files/pdf-inbox/，Execute Workflow。
驗收三條：(a) 200 張全綠燈 (b) 抽查 5 張隨機 output 命名正確 (c) Finder pdf-renamed 資料夾數量是 200
3
Executions tab 驗收動線（不等 UI 即時 200 燈）
在 n8n 工作區右側點「Executions」分頁 → 等所有 200 筆狀態 column 變成「Success」（綠 ✓）。若有「Failed」（紅 ✗）或「Cancelled」（灰），把那筆 timestamp 記下，回去看哪個節點掛了。
200 筆會在 Executions 分頁輪流刷新，大約 7-8 分鐘才全跑完，跑完後再看 list，不要等 UI 即時顯示 200 個綠燈。
三輪收斂上限的意義
三輪迴圈（步驟 3 → 4 → 5）不過，不是你失敗了，是需求或方向出了問題。退回步驟 1 的便箋，重看「input / output / 保留節點」有沒有哪格寫模糊。
(04)

收斂後的 Destructive 操作補充（紅線 8）

步驟 6 跑通之後，還有一個容易被忽略的環節：如果你的 workflow 裡有任何「會對真實世界產生不可逆影響」的節點——刪資料（Notion delete / Drive delete）、寄信（Gmail / Telegram 發給真實收件人）、更新記錄（HubSpot / Pipedrive update）——在切換到全量之前，必須先做 dry-run（紅線 8）。

dry-run 的意思：先把寄信節點的收件人改成你自己的測試信箱跑 1 次；如果是刪檔節點，先把它 disable 換成 console.log 假動作。確認 input/output 資料形狀全程正確之後，才換回真收件人 / 真資料源。10 筆樣本跑通只代表「資料形狀對了」，不代表「收件人對了」。省掉這 5 分鐘，後果可能要花幾小時甚至幾天補救。

(05)

常見錯誤 + 怎麼解

錯誤 1
「我點 Execute Workflow 直接在 production workflow 上跑了」
解：立刻關掉 Active 開關。在 Executions tab 看歷史確認 production 有沒有被新版本覆蓋。下次嚴格用 -edit 複本。
錯誤 2
「我貼模板 4，LLM 又給我整份 workflow 重寫」
解：模板 4 結尾「不要重寫整個節點」是關鍵。如果 LLM 還是重寫，加一句：「我只要『修改第 X 行的 Y 字串為 Z』這種句型的回答，不要 JSON」。
錯誤 3
「LLM 第二輪建議我加 try/catch，我拒絕了，第三輪它又建議加 if 條件」
解：用模板 5 強制收斂。這是 LLM 的「helpful 病」。如果第 4 輪還在加，停——換一個新對話 session 重啟。
錯誤 4
「跑了 200 張，發現有 3 張改名失敗（檔名出現 null）」
解：(a) 檢查那 3 張原始 PDF 是不是格式異常（掃描品質差、欄位名不同）(b) 不要改 workflow，改去人工處理那 3 張（80/20 原則）(c) 在 workflow 描述欄記下「3 張邊界 case 已人工處理：[檔名]」。
錯誤 5
「全量跑完，Finder 顯示 195 張不是 200」
解：(a) 打開 n8n UI 的 Executions tab 看「Failed」筆數 (b) 點失敗的 execution 看是哪幾筆 PDF (c) 對應到 pdf-inbox 找原檔，人工處理 / 二次跑。
(06)

你已經走完六步流程的後半段。下一頁是「8 條紅線」單頁聚焦：把整份教材的安全規範濃縮成一張清單，建議動手改 workflow 前先看一遍。特別注意紅線 1、2、7、8。
