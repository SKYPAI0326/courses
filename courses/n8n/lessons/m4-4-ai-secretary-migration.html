MIGRATION REBUILD · 4.4 · 100 min

AI 秘書工作流：從 Make 遷移到 n8n

把 Make 上跑得最熟的 AI 秘書 scenario 在 n8n 完整重建——11 個節點、4 路分發、誠實揭露每一步資料離開或留下的邊界。

⏩ 本節定位：Module 4 整合最終單元。前置請完成 4.1 表單遙控、4.3 AI 計劃生成。本節是「Make 進階使用者轉 n8n」承諾的真正兌現點，**不只是教 n8n，是教你怎麼搬家而且不被舊邏輯誤導**。

心智模型 4 大落差：(1) 資料形狀（n8n 預設多 items 流，要主動 Aggregate）、(2) 分發 vs 分支（Make Router 多 boolean 並行 → n8n 只能 IF×4 並排，不能 Switch）、(3) JSON Parse（Make UDT 友善 GUI vs n8n 要寫 Code 防禦解析）、(4) 錯誤處理（onerror Ignore 不等於 Continue On Fail）。
PHASE A · 跑 Make 原版觀察 + 開新 n8n workflow
A1
在 Make 跑一次原版，做兩件事觀察
打開 Make 上的 02_AI 秘書 進階，按 Run once。觀察兩件事：
// 觀察 1：ops 消耗 History 頁面 → 本次執行 → 總 ops（一般 5-15 ops，看抓到幾封信） // 觀察 2：信件路徑 Gmail Search 模組 → Output 面板 → fullTextBody → 信件已以明文形式經過 Make 的 EU/US 伺服器中轉 → Make 執行 log 保留 1-7 天
重點：後面我們會比對 n8n 版本——信件從 Gmail API 直接到你電腦，不經過任何第三方伺服器中轉。但這不等於「機密留本機」（信件本來就在 Google）。
A2
在 n8n 開新 workflow
n8n 主介面 → 左上角 + Create Workflow → 命名為 M4-4 AI 秘書遷移。藍圖如下：
[Gmail Trigger] → [Aggregate] → [HTTP Gemini] → [Code: Parse JSON] ↓ [Split Out: tasks] → [IF×4 並排路由] ↓ ┌──────────────┬──────────────┬──────────────┬─────────────┐ │ │ │ │ │ [Telegram] [Gmail Send] [Google Docs] [HTTP POST] if to_tg=true if to_email if to_doc 無 filter =true =true
提示：11 個節點，4 路分發。對應 Make 11 步邏輯，但**結構排列不同**——n8n 沒有 Router 模組，要靠 IF 節點並排模擬。
PHASE B · Trigger + Aggregate + Gemini（30 min）
B1
Gmail Trigger node
點 canvas + → 搜 Gmail → 選 Gmail Trigger。設定：
// Gmail Trigger 設定 Event: Message Received Read Status: Unread Limit: 5 Simplify: OFF（重要，要拿 textPlain） Pull Every: 15 minutes
與 Make 差異：Make 的 Gmail Search 是 Action 型；n8n Gmail Trigger 是 Polling Trigger，自帶輪詢。心智模型不同，不要以為要外加 Schedule 節點。
欄位命名：n8n 用 textPlain（不是 Make 的 fullTextBody）。後面 prompt 餵給 Gemini 時要對應改 expression。
B2
Aggregate node：5 items → 1 array
Make ↔ n8n 心智模型最大落差。Make BasicAggregator 顯式打包；n8n 預設不會 aggregate，AI 節點會跑 5 次。要讓 Gemini 一次看到 5 封信，兩種寫法擇一：
// 寫法 1：原生 Aggregate node（GUI 友善） Aggregate: All Item Data (Into a Single List) Put Output in Field: array Include: Selected Fields → id, subject, from, textPlain // 寫法 2：Code node（彈性高，推薦寫過 JS 的學員） const compact = items.map(item => ({ id: item.json.id, subject: item.json.subject, fromEmail: item.json.from?.value?.[0]?.address || item.json.from, fullTextBody: item.json.textPlain || item.json.snippet })); return [{ json: { array: compact } }];
與 Make 差異：Make BasicAggregator 用 GUI 點選欄位，n8n Code 寫一行 items.map。對非工程使用者，原生 Aggregate node 比 Code 友善。
B3
HTTP Request 直打 Gemini（不用 n8n 內建 Gemini node）
刻意不用 n8n 內建 Gemini node：要完全控制 model、response schema、safety、thinking budget，HTTP Request 比 AI Chain 子節點更穩。
// HTTP Request 設定 Method: POST URL: https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key={{ $credentials.geminiApiKey }} Body Type: JSON On Error: Continue On Fail（對應 Make onerror=Ignore） // JSON Body（沿用 Make 原版 4 區 prompt） { "contents": [{ "role": "user", "parts": [{ "text": "<Role>你現在是我的全能數位秘書...</Role>\n\n<Task>分析每封信生成 routing/meta/payloads/action 4 區...</Task>\n\n<Output_Rules>僅輸出純 JSON、嚴禁 ```json``` 標籤、必須以 { 開始 } 結束、繁體中文</Output_Rules>\n\n<Source_Emails>{{ JSON.stringify($json.array) }}</Source_Emails>" }] }] }
與 Make 差異：Make Gemini module 內建 4 區 schema 對應 UDT，n8n HTTP Request 沒有 schema 約束機制——必須在 prompt 裡用 <Output_Rules> 強制 LLM 遵守，並在下一步用 Code node 防禦式解析。n8n 給你更多控制權，代價是你要自己寫驗證。
PHASE C · JSON Parse + Split Out + IF 4 路（30 min）
C1
Code node 解析 AI 字串包 JSON
Gemini 回應路徑：$json.candidates[0].content.parts[0].text。text 是字串，內容才是 JSON 物件。要解析出來：
// Code node（Run Once for All Items） const raw = $input.first().json.candidates?.[0]?.content?.parts?.[0]?.text || ''; let parsed; try { // 容錯：去掉可能的 ```json 包裹 const cleaned = raw.replace(/```json\s*/i, '').replace(/```\s*$/, '').trim(); parsed = JSON.parse(cleaned); } catch (e) { return [{ json: { tasks: [], _error: 'JSON parse failed', _raw: raw } }]; } // 防禦：tasks 必為 array if (!Array.isArray(parsed.tasks)) { return [{ json: { tasks: [], _error: 'tasks not array', _parsed: parsed } }]; } return [{ json: parsed }];
與 Make 差異：Make JSON Parse + UDT 提供 GUI 定義 schema，欄位錯了會 onerror=Ignore 跳過。n8n 沒有對應的 friendly UI，所有錯誤防禦都得寫進 Code node。
C2
Split Out node：攤開 tasks 陣列
對應 Make 的 BasicFeeder。設定 Field to Split Out = tasks。Execute → 5 個 items 出現（每個是 1 個 task 物件）。
對等度 95%：n8n Split Out 與 Make BasicFeeder 心智模型幾乎一致。
C3
IF × 4 並排路由（不是 Switch！）
這裡是 Make ↔ n8n 心智模型最容易出錯處。Make Router 4 路 + 各路 boolean filter 是「4 路同時檢查、各自獨立決定是否觸發」（一個 task 可同時觸發 to_tg 和 to_doc）。n8n Switch 無法做這件事——Switch 是「單值匹配走唯一一條」。
// 正確做法：4 個 IF 節點並排 從 Split Out 拉 4 條線出來，分別接到 4 個 IF 節點： IF #1（to_tg）: {{ $json.routing.to_tg }} is equal to true IF #2（to_email）: {{ $json.routing.to_email }} is equal to true IF #3（to_doc）: {{ $json.routing.to_doc }} is equal to true IF #4（HTTP）: 對應 Make 第 4 路無 filter → 直接跳過 IF，從 Split Out 拉到 HTTP node
絕對不要用 Switch node 來做這件事。Switch 會強制 task 只能進一條分支，與 Make Router 多 boolean 並行邏輯不同。
PHASE D · 4 通道輸出 + Continue On Fail（15 min）
D1
Telegram Send Message
IF #1 的 true 輸出 → 加節點 Telegram → Send a Text Message。Credential 用 BotFather 拿 token。
Chat ID: 1062913956（沿用 Make 原版；正式部署改你自己的） Text: 🚨 【{{ $json.meta.category }}】 (優先度: {{ $json.meta.priority }}) 📝 秘書摘要： {{ $json.payloads.tg_brief }} 🔗 一鍵跳轉： {{ $json.action.source_link }}
D2
Gmail Send Email
IF #2 的 true 輸出 → Gmail → Send a Message。沿用 Phase B 的 Gmail OAuth credential。
To: sky8697@gmail.com Subject: 【秘書通報】{{ $json.meta.category }} - {{ $json.action.sender_info }} Email Type: Text Message: 你好，我是數位秘書。 偵測到一封郵件需要你的關注： 案情摘要：{{ $json.payloads.email_report }} 相關連結：{{ $json.action.source_link }}
D3
Google Docs：Create + Update（拆兩節點）
IF #3 的 true 輸出 → 加 Google Docs → Create a Document，再串第二個 Update a Document：
// Create a Document Title: {{ $json.meta.category }}_{{ $now.toFormat('yyyyMMdd-HHmmss') }} Folder: 「數位秘書」資料夾（用 Tree picker 選） // Update a Document（串接 Create 後） Document URL: {{ $node["Create a Document"].json.documentId }} Action: Insert Text Text: 數位秘書：郵件分析歸檔報告 分析類別：{{ $('Split Out').item.json.meta.category }} 情緒偵測：{{ $('Split Out').item.json.meta.sentiment }} 處理建議：{{ $('Split Out').item.json.payloads.doc_archive }} 原始發件人：{{ $('Split Out').item.json.action.sender_info }} 原始信件連結：{{ $('Split Out').item.json.action.source_link }}
與 Make 差異：Make 的 createADocument 一步到位（建 + 寫入 + 設 folder），n8n 必須拆 Create + Update 兩節點。心智模型複雜度上升，但好處是中間可以插入其他處理（例如 Copy 模板再 Replace）。
D4
HTTP POST ntfy.sh（直接從 Split Out 接，不經 IF）
沿用 Make 原版第 4 路無 filter 的設計。直接從 Split Out 拉一條線（不接 IF #4）→ HTTP Request：
Method: POST URL: https://ntfy.sh/workalarm_for_classtest001 Body Type: Raw Content Type: text/plain Body: 🔴 數位秘書：緊急客訴警報！ 內容摘要：{{ $json.payloads.tg_brief }} Headers: Priority: 5 On Error: Continue On Fail
教學重點：對應原 Make scenario 的「教學遺漏」——第 4 路無 filter 但 body 用 tg_brief 跟 routing.to_tg 沒對齊。我們的決策是「1:1 重建保留原邏輯」，但這在實戰部署時建議加 filter。
D5
啟用 Continue On Fail（取代 onerror Ignore）
n8n 沒有節點層級的 onerror=Ignore，最接近的等效是：
// 個別節點設定 Code（JSON Parse）: Settings → On Error → Continue On Fail HTTP Request（Gemini）: Settings → On Error → Continue On Fail // 全工作流層級 右上 Settings → Error Workflow → 選 error-logger.json（試跑包提供範本）
對等度只 65%：Continue On Fail 與 Make onerror=Ignore 不完全等同——錯誤忽略後 n8n 節點輸出空 array 或 error 物件，後續節點要處理 undefined 的可能性。
⬇ 下載 M4-4 workflow JSON 範本（11 節點）
下載後在 n8n 介面 → Import from File → 選此 JSON。匯入後需重新指定 credentials。

本單元 Phase B-D 跑出來的 n8n 版本並非「機密資料留本機」。誠實列出每個節點的資料離開或留下：
節點資料路徑機密適用？

真正能兌現「機密留本機」的場景：
  1. 本機 PDF/Excel/CSV 清理，不送外部 API（M3 系列已覆蓋）
  2. Ollama / llama.cpp 跑 local LLM（HTTP 打 http://localhost:11434/api/generate）
  3. 本機 OCR + 規則抽取 + embedding 到本機 Qdrant（不依賴雲端 LLM）
  4. 去識別化處理：信件原文留本機跑 local LLM 摘要（去名/去金額/去公司），只把摘要送雲端

你已不只是「會用 n8n」，而是能把 Make 既有 scenario 在 n8n 重建，而且懂得對每一步做威脅模型分析。
下一步的能力延伸：把這個工作流改成「機密 NDA 合約批次」的 local LLM 變體——下個動手題等你。
常見問題 · TROUBLESHOOT
Gemini API 回 400 錯誤 Invalid argument: contents.parts.text exceeded maximum length？
5 封信打包後總 tokens 太大（每封 1500-3000 tokens × 5 可能 15K+）。Gemini 2.5 Flash 雖支援 1M tokens 但有 prompt 長度上限。解法：在 Aggregate 後加 Code node 截斷每封信 fullTextBody 到 2000 字以內（取 snippet + 前 1500 字）；或改 Gmail Trigger limit 從 5 降到 3。
Code node 解析 JSON 報錯 SyntaxError: Unexpected token '\`'？
Gemini 雖然 prompt 禁了 ```json``` 包裹，但仍偶爾不聽話。解法：檢查 Code node 是否完整複製了 cleaned = raw.replace(/```json\s*/i, '').replace(/```\s*$/, '').trim()；若仍報錯，把 _raw 內容貼到 ChatGPT 問「這個 JSON 哪裡壞了」做最後排查。
Telegram / Gmail / Docs 三條 IF 都沒觸發，但 ntfy.sh 一直發？
Gemini 回的 JSON 裡 routing.to_tg 等欄位不是 true/false 而是字串 "true"。IF node 嚴格比對 boolean。解法：在 prompt 加強「所有 boolean 必為 JSON 原生 true/false 不可加引號」；或在 Code node Parse 後加強制轉型 parsed.tasks.forEach(t => Object.keys(t.routing).forEach(k => t.routing[k] = t.routing[k] === true || t.routing[k] === 'true'))。
🎯 動手練習：機密 NDA 合約批次處理
情境｜雯姊接了客戶 NDA 合約批次處理案：60 份合約 PDF，要抽「條款編號 / 簽約方 / 有效期」整理成 Google Sheet。合約原文不能上傳雲端（包括 Gemini API 都不行）。
挑戰｜把本單元的 AI 秘書工作流改寫成「機密合約處理」變體：
  • 第一個節點換成 Read Binary Files（從 /n8n-shared/contracts-inbox/ 讀 60 份 PDF）
  • AI 處理節點換成 HTTP Request 打本機 Ollama（http://localhost:11434/api/generate）而不是 Gemini
  • 輸出節點 Google Sheets Append（可以雲端，因為已是去識別化欄位）
不需要實際跑通，能畫出工作流結構 + 解釋每個節點為何如此選擇即可。判斷重點不是工具，是「送什麼資料、送去哪」。
