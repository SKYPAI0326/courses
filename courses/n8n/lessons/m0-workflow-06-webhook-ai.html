#06 · WEBHOOK AI · 5 NODES · 進階

外部 POST 觸發 AI，同時寫本機檔 + 推 Telegram + 回 200

把 #01（webhook） + #02（AI） + #05（Telegram）整合成一個 workflow，並引入「同節點多輸出」概念 — Code 節點輸出後 fan-out 三條線分別接 Write / Telegram / Respond，三件事**並行執行**而非依序。學完這頁能設計「外部觸發 AI 服務 + 多通道輸出」的所有 workflow。

學習成果
能用 Webhook 節點接收 POST body 內 JSON 欄位（如 question）並餵給 AI
能用 n8n connection 設計「一節點輸出 → 多節點接收」的並行分支
能用 Set 節點當 Respond to Webhook（responseMode: lastNode 機制）
(01) WORKFLOW 全貌

節點與資料流（一節點輸出 fan-out 三條線）

Webhook: ai-ask → Code: AI 生答 + 組 Markdown

↗ Write ai-output/ （寫本機 .md）
→ Telegram 通知 （推手機）
↘ Set: 回 webhook 200 （lastNode 回 caller）

5 節點，但 connection 不是線性 — Code 節點的輸出同時連到 3 個下游節點（n8n 自動並行執行）。這是 n8n 強大但容易被忽略的 pattern：「一份資料同時送多處」（不需要複製資料、不需要分支邏輯）。

(02) NODE BREAKDOWN

節點逐一拆解

NODE 1 / 5
Webhook: ai-ask
webhook v2Path: ai-askResponse: lastNode

用途：接收外部 POST 含 JSON body { question: "..." }。

跟 #01 結構一樣，差別在 path 改 ai-ask、responseMode 一樣設 lastNode。本機完整 URL：http://localhost:5678/webhook/ai-ask（Production）或 /webhook-test/ai-ask（Test）。

NODE 2 / 5
Code: AI 生答 + 組 Markdown
code v2

用途：取 webhook 收到的 question → 打 Gemini → 組 Markdown 內容 + Telegram 訊息文 + 回應 JSON。

關鍵 jsCode 片段：

// 從 webhook 取問題（兩種來源容錯） const question = webhookData.body?.question || webhookData.query?.question || '請告訴我什麼是 n8n？'; // 打 Gemini（同 #02 #03 #04 同樣 thinkingBudget: 0） const response = await this.helpers.httpRequest({...}); const raw = response.candidates?.[0]?.content?.parts?.[0]?.text; // 組 3 個輸出 const markdown = '# AI 回應...'; // 給 Write const tgMsg = stripMD('🤖 AI 回應...' + raw); // 給 Telegram（strip markdown） const outputPath = '/files/shared/ai-output/...'; // 給 Write 路徑 return [{ json: { outputPath, tgMsg, question, answer: raw, aiError }, binary: { data: { data: base64(markdown), fileName: ... } } }];

關鍵設計：return 的單一 item 含 json（給 Telegram + Set 用）+ binary（給 Write 用）。下游 3 個節點各取所需。

NODE 3 / 5
Write: ai-output/

用 Code 節點 return 的 binary.data 寫 Markdown 檔到 shared/ai-output/ai-{時間戳}.md。

NODE 4 / 5
Telegram 通知

同 #05，text: {{ $json.tgMsg }}，Additional Fields 留空（不傳 parse_mode）。

NODE 5 / 5
Set: 回 webhook 200
set v3.4

因為 Webhook 設了 responseMode: lastNode，這個 Set 節點的輸出會被當成 webhook response 回給 caller。

Assignments：status = "ok"、question = {{ ... .question }}、answer = {{ ... .answer }}

caller 用 curl 打進來會收到 {"status":"ok","question":"...","answer":"..."}。

(03) DESIGN RATIONALE

為什麼這樣設計

💭
為什麼三件事並行而不是依序？
Write 寫本機檔 / Telegram 推手機 / Set 回 webhook 三件事互相不依賴，並行執行更快（n8n 自動並行同層下游節點）。如果是線性鏈，前一個失敗會擋住後面。並行的話 Telegram 失敗不會擋住 webhook 回應，學員的 caller 仍能拿到 AI 答案。
💭
為什麼 jsCode 內要 stripMD(tgMsg)？
AI 回應常含 markdown 字元（* _ [），#05 Telegram 節點雖然沒設 parse_mode 應該安全，但雙保險：jsCode 內把要送 Telegram 的訊息 strip markdown 字元。Write 寫本機檔的 markdown 不 strip（讓使用者開檔案看時保留 markdown 格式）。
⚠️
Webhook + curl 學員常踩的雷：Test URL 一次性，按 Execute 後立刻打 curl。curl 多行用 \\ 換行在 zsh paste 時容易斷，建議寫成單行。
(04) 改造方向

5 種改造方向

  1. 接 Make / Zapier：從 Make scenario 打這個 webhook（搭 Cloudflare Quick Tunnel 暴露成公開 URL），把 n8n 變成 Make 的 AI 後端。Make 訊號（Google Form / Email / 排程）→ 觸發 n8n AI workflow。
  2. 支援多種 AI 任務：webhook body 加 task 欄位（如 "summarize" / "translate" / "rewrite"），Code 節點根據 task 切換不同 prompt。
  3. 串資料庫做 history：在 Code 之後加 PostgreSQL 節點記錄每次 question + answer + timestamp，做 AI 對話歷史庫。
  4. 加 LLM Streaming：n8n 暫不支援 SSE / stream，但可改成「webhook 立即回 200 + 後台跑 AI + Webhook callback 回去」的非同步 pattern。
  5. 串自家網站：學員網站表單填問題 → submit POST 到本機 n8n（搭 Cloudflare Tunnel）→ AI 回答 → 顯示在網頁。
