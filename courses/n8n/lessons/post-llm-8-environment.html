環境互鎖 + 真實驗收 — 改 workflow 前先確認在哪跑

每次動 workflow 前，先花 30 秒掃 4 項環境狀態。環境對了，第 7 章的分流才會帶你走對方向。跑完後，不是看 Code 綠燈就算過——Telegram 真收到、Sheet 真寫入、Drive 真產出，才是完整驗收。

本頁學習成果
能在 30 秒內掃完 4 項環境檢查（Local/Cloud、-edit/production、Active 開關、上次 Execution 時間）
理解 Active 開關對非工程師的具體風險，建立「坐下就看 toggle 顏色」習慣
能區分「Code 綠燈」與「真實驗收」的差異，完成下游輸出端的 5 條驗收
(01)

為什麼這章排在第 7 章後——但每次都要先做

章節順序說明
教學順序：先教錯誤分流（第 7 章）讓你能自救，但實機操作時請先做本章 8.X 環境檢查，再進第 7 章分流。

理由：環境錯時，第 7 章分流會帶你走錯方向——你以為是 typeVersion 對不上，實際是你在 Cloud 上，而 LLM 給的是 local 才有的節點；或你以為節點壞了，實際是 production workflow 的 Active 開關被觸發，跑掉的是正式資料。

建議把 8.X 的環境檢查表列印出來，貼在螢幕邊。每次坐下動 workflow 前 30 秒掃過去。

本頁的核心是兩件事：(1) 動手前，確認你在哪個環境；(2) 跑完後，確認下游真的收到你以為送出去的東西。這兩件事都不靠直覺判斷——有具體的「看哪裡、看到什麼字串算對」。

(02)

8.X 環境檢查 4 項 — 每次動 workflow 前 30 秒掃過

使用方式
4 項全部看完才算掃過。順序不重要，重要的是每次都做。下方 anti-pattern 是最常踩的坑，第一次看完就知道為什麼要這樣。
檢查項 在 n8n UI 哪裡看 看到什麼算對 看到什麼要警覺
Local vs Cloud 瀏覽器網址列 OKlocalhost:5678/workflow/...（local） 注意*.app.n8n.cloud/...（Cloud — LLM 建議的節點是否 Cloud 支援？）
-edit vs production 工作區左上角 workflow 名稱 + 瀏覽器 tab title OK名稱結尾有 -edit，例如 02-pdf-ai-rename-edit 警覺無 -edit（是 production）— 立刻 Cmd+Z 退出，不要動
Active 開關 工作區右上角 toggle OK灰色 OFF（schedule / trigger 不會自動跑） 警覺藍色 ON（一動到就可能被 trigger 觸發 — 對 -edit 版應關閉）
上次 Execution 時間 工作區右側 Executions 分頁第一筆 timestamp OK時間跟你預期執行時間一致（手動跑就是剛才） 警覺比預期早 / 你沒跑卻有新 execution = trigger 自己跑了，查 Active 開關
LOCAL VS CLOUD ANTI-PATTERN
跨環境學員（公司 VPS 一個 n8n、家裡 Docker 又一個）：看到 n8n.yourcompany.com 這類自架域名，要當 self-host 處理，不是 Cloud。記下「我這個域名背後實際是 self-host 還是 Cloud」，否則 LLM 對話應對會走錯。
-EDIT VS PRODUCTION ANTI-PATTERN
Duplicate 出 -edit 後，左上角名稱會顯示新名稱，但有些版本 tab title 還停在舊名幾秒鐘才刷新。這時不要急著動，重新整理一次再開始。
EXECUTIONS 時間 ANTI-PATTERN
隔天回來，Executions 第一筆是凌晨某個時間、不是你跑的——表示有 schedule trigger 在自動跑。回頭查 Active 開關 + trigger 節點設定，否則你跑出的數會跟自動跑的混在一起。
(03)

Active 開關 — 4 項中對非工程師最關鍵的一項

重要習慣
Toggle 一點就反。每次坐下來，先花 3 秒看右上角顏色。

灰色 = OFF = 安全。藍色 = ON = 危險（trigger 可能隨時自動跑）。
對 -edit 版 workflow，Active 一定要關；對 production 版，看你需求——但你在改測試的時候，production 最好也是 OFF。
Toggle 狀態對照
灰色 OFF — schedule 不自動跑，手動執行才跑
藍色 ON — trigger 隨時可能觸發，schedule 定時跑

第一次 Duplicate -edit 完，新 workflow 預設 Active OFF——但學員容易在後續操作中不小心點到。每次坐下繼續做之前，30 秒看一眼 toggle 顏色。

△
如果你的 workflow 有 Telegram trigger 或 Webhook，Active ON 就表示外部訊息一進來就會觸發——即使你正在改節點。改節點前先把 Active 關掉，改完測試通過再打開。
(04)

self-host vs Cloud — 4 個關鍵差異 + LLM 對話應對

確認你在哪個環境之後，下面這 4 點對照決定 LLM 給的建議是否能直接套用。

差異點 Self-host n8n Cloud LLM 對話應對
Credential 儲存 你電腦/VPS，自己備份 n8n 雲端，n8n 加密 LLM 給「整份匯入」建議時，Cloud 用戶要追問「credential 部分我要重建嗎」
節點可用性 全部 community node 可用 部分受限 LLM 建議 Read Binary File / Execute Command / LocalAI 等本地節點時，Cloud 用戶說：「我在 Cloud，這節點不能用，請改成 Google Drive Read File 等雲端節點」
檔案路徑 /files/ 等本機絕對路徑 OK 不能直接讀本機 LLM 給的 path（例：/files/pdf-inbox/），Cloud 用戶說：「我在 Cloud，請改成 Google Drive 路徑或 S3 bucket」
Execution 配額 無上限（吃自己資源） 月配額，超用另計費 全量跑前先估算（例：200 張 × 每張 1 次 Gemini = 200 次 execution），看是否超 plan 上限
△
每次 LLM 對話開頭，加一行環境聲明（見第 6 節的 prompt prefix），LLM 就會自動避開你環境不支援的建議。不要每次都等到 LLM 建議了不能用的節點才補說。
(05)

真實驗收 — 不只 Code 綠燈，下游要真的收到

為什麼 Code 綠燈不夠
n8n UI 的綠燈只代表節點執行沒有拋出錯誤。但「節點執行成功」不等於「下游真的收到你以為送出去的東西」。Telegram 顯示 sent 但訊息沒到、Sheet 寫入節點回傳 OK 但實際 row 是空的、Drive 上傳回傳 file ID 但檔案損毀——這些都是 Code 綠燈但驗收失敗的情況。
  • Code 全綠燈：n8n UI 上每個節點的執行結果都沒有紅色錯誤框。
    自評動作：點開每個節點的 Execution 結果，確認沒有 Error 欄位。
    過關：0 個節點顯示紅框。
  • Telegram 真收到（有用 Telegram 輸出端的 workflow）：拿手機打開 Telegram，實際看到機器人送出的訊息。
    自評動作：跑完 30 秒內，打開 Telegram 目標 chat 確認有新訊息，訊息格式跟你設計的一致。
    過關：手機上看到訊息，訊息內文不是亂碼也不是空字串。
  • Sheet 真寫入（有用 Google Sheets 輸出端的 workflow）：打開 Google Sheets 實際看到新 row 被寫入，欄位值跟 input 對應。
    自評動作：跑完後切到瀏覽器開 Sheet，確認最後幾行有新資料，抽一個欄位值比對 input。
    過關：有新 row，至少 1 個欄位值可對應到你跑的 input 資料。
  • Drive 真產出檔（有用 Google Drive 輸出端的 workflow）：打開 Google Drive 目標資料夾，確認有新檔案，檔案可以正常開啟。
    自評動作：跑完後進 Drive 目標資料夾，確認有新檔案，點開確認內容不是空白或損毀。
    過關：Drive 有新檔案，可以正常預覽或開啟。
  • 檔名格式符合便箋設計（針對 #02 或任何有改名邏輯的 workflow）：輸出的 newFilename 符合你便箋上定的格式，0 個 null 欄位。
    自評動作：抽 3 筆 output 的 newFilename，對著便箋格式逐字比對。
    過關：3 筆全部符合格式，0 個 null 值。
(06)

自我驗收 — 5 條可量化清單

使用方式
5 條都給了自評動作（5 分鐘內可做）和過關標準（可量化驗收）。逐條對照，不能做到的回去看對應頁。第 5 條是硬指標，建議實際跑過才算。
  • 能用 4 格便箋拆解一個改造需求（input / output / 保留節點 / workflow 編號），並能對著別人講 1 分鐘不打結。（對應：第 3 頁，步驟 1）
    自評動作：選一個你還沒做的 Lite Pack workflow（例：#04），花 5 分鐘填 4 格便箋，然後對著手機錄音講 1 分鐘。
    過關：4 格全填具體值（input 有 4-6 個欄位名）+ 講話沒打結。
  • 能在不貼 credential 的前提下，把任一 Lite Pack workflow JSON 貼給 LLM 並要求中文流程描述。（對應：第 3 頁，步驟 2）
    自評動作：從 n8n UI export 任一 workflow，用 VS Code 搜尋 4 個字串（apiKey / accessToken / password / token）。
    過關：4 個字串全部 0 個結果。
  • 能下指令讓 LLM 只回傳節點區段、不重寫整份 workflow，並在 n8n UI 上正確完成節點替換。（對應：第 3 頁，步驟 3）
    自評動作：對 LLM 貼模板 3，要求改某個節點，看 LLM 回的 JSON 第二行。
    過關：LLM 回的 JSON 第二行是 "parameters":（不是 "name": 或 "nodes":）。
  • 能用三件套格式回報錯誤——紅燈訊息全文 + 上一節點 input + 當前 parameter。（對應：第 4 頁，步驟 5）
    自評動作：故意製造一個錯誤（把 #02 的 Code 節點 prompt 字串隨便刪掉幾行）→ 跑紅燈 → 用模板 4 回報。
    過關：LLM 給的回應指出「具體哪一行哪個欄位」（不是「建議重新檢查」）。
  • 能把 #02-pdf-ai-rename 改成處理某種非合約類文件（發票 / 收據 / PO / 申請表 4 選 1），跑通 10 筆樣本。硬指標（對應：第 6 頁，完整 Walkthrough）
    自評動作：選一種文件類型，照第 6 章 walkthrough 跑完整六步。
    過關 3 條：(a) 10 張全綠燈 (b) 10 張的 newFilename 全部符合便箋格式 (c) 0 張 null 欄位。強烈建議：跑通後到課程社群貼 Finder 截圖（10 張改名後的 PDF）。
(07)

進階練習 — 不給答案的 2 題（給過關條件）

設計意圖
沒有附範本答案，也沒有附 prompt 模板。你需要回去用前 7 頁學的方法自己解：拆需求、讀 JSON、限定修改範圍、匯入測試、錯誤回報、收斂。每題給「過關 5 條」，你做出來的東西要符合這 5 條才算過。
練習 1
把 #04-daily-ai-report 的輸出端從 Google Sheet 改成 Notion Database
  • 4 格便箋有「目標 Notion Database 的 schema」具體欄位名
  • 用「示範形狀法」：在 n8n UI 拉 Notion 節點 → 隨便填 → 複製 JSON → 貼給 LLM 當形狀範本（不讓 LLM 從記憶生成）
  • 整套流程用 -edit 版做，不動原 #04
  • 跑通 5 筆測試 daily report
  • 在 Notion Database 看到 5 筆有結構的 record（不是 5 條空白 row）
練習 2
把 #11-csv-clean-score 的評分維度從通用版改成你業務的特定維度
  • 4 格便箋有「你業務的特定維度」清單（至少 3 個維度）
  • 對比原 #11 的通用維度，明確標出哪些保留 / 哪些換掉 / 哪些新增
  • 只改 Code 節點裡的 prompt + 評分邏輯，不改 trigger / 不改輸出端
  • 跑通 10 筆真實 CSV 樣本
  • 抽查 3 筆 output 的評分結果，肉眼看分數合不合理
(08)

環境感知 prompt prefix — 每次對話開頭加

把這段加在每次 LLM 對話的最開頭，LLM 就會自動避開你環境不支援的節點，不需要等到踩雷才說。

我用 n8n [Cloud / self-host with Docker / self-host with Cloudflare Tunnel]，n8n 版本 [latest / 1.x.x]。 我的工作流程是 [改造目的]。 [後面接其他 prompt 模板]

LLM 回應範例：

明白，你在 n8n Cloud 上，所以我會避免建議： - Read/Write Files 節點（用 Google Drive 替代） - Execute Command 節點（不可用） - 本機絕對路徑（用 Google Drive folder ID 或 S3 bucket） 進行你的改造需求 [...]
→ 雲端 n8n 決策 FAQ（補充教材，選讀）
(09)

系列結語

8 頁學完，下一步是動手。這套流程的核心不是 prompt 技巧，是「縮小範圍、逐步驗收」的工作紀律——第一次走完 6 步可能要花一個下午，第五次就會變成 30 分鐘的例行作業。改造案例歡迎分享到學員社群，你的實際情境對其他結業學員的參考價值，遠比任何教材範例都高。
