#02 · PDF AI 改名 · 5 NODES · AI

Lite Pack 招牌：丟 PDF 進資料夾，AI 自動取語意檔名

一個跑得通的 AI 自動化要面對的所有真相，這個 workflow 都有：n8n HTTP Request 節點 jsonBody 解析雷、Gemini 2.5 thinking budget 吃掉輸出 token、Code 節點 binary 對齊、AI 回應格式不穩定要 6 層 fallback parser。讀完這頁你會帶走「能在自己業務 AI 工作流頂得住」的設計直覺。

學習成果
能用 Code Node + this.helpers.httpRequest 直接打 Gemini API（取代 HTTP Request 節點）
能設計 multi-layer fallback parser 處理 AI 回應的 6 種變異格式
能診斷 parseMethod / finishReason / aiResponseLength / httpError 4 個指標定位 AI 失敗根因
(01) WORKFLOW 全貌

節點與資料流

Manual Trigger → Read PDF → Extract PDF Text → Code: AI 改名（含 Gemini call） → Write 改名輸出

5 個節點線性鏈。資料流：手動觸發 → 從 shared/pdf-inbox/ 抓 PDF binary → Extract 解析 PDF 文字 → Code 節點打 Gemini 取新檔名 + 6 層 fallback → 寫到 shared/pdf-renamed/。重點全在第 4 個 Code 節點。

(02) NODE BREAKDOWN

節點逐一拆解

NODE 1 / 5
Manual Trigger
manualTrigger v1

最簡觸發節點：點下方紫色 Execute workflow 按鈕就會跑。沒任何設定。改造方向：之後可換成 Schedule Trigger（定時跑）/ Local File Trigger（資料夾有新檔自動跑）/ Webhook（外部 POST 觸發）。

NODE 2 / 5
Read PDF
readWriteFile v1

用途：用 glob pattern 抓資料夾內 PDF 檔。

關鍵設定：fileSelector: /files/shared/pdf-inbox/*.pdf

  • /files/shared/ 是 docker-compose 內 bind mount 把本機 ~/Downloads/n8n-starter-kit/shared/ 對應進 container
  • *.pdf glob 抓所有 .pdf 檔。多份 PDF 的話 Read 會 fan-out 成多個 items（每個檔一個 item）
  • Output 結構：json: { fileName, fileSize, fileExtension, mimeType } + binary: { data: { data: 'base64...', fileName, mimeType } }
NODE 3 / 5
Extract PDF Text
extractFromFile v1operation: pdf

用途：解析 PDF binary 抽出純文字內容。內部用 pdf-parse library。

Output 結構：json: { numpages, info, metadata, text, ... } + binary: 原 PDF 保留

注意（連發 #14 的真相）：Extract success 分支的 json 內沒有 fileName 欄位（被 PDF metadata 覆蓋了）。下游 Code 節點要拿 fileName 必須從 binary.data.fileName 或從 $('Read PDF').all()[i] 反查。

NODE 4 / 5
Code: AI 改名（含 Gemini call）
code v2runOnceForAllItemsv0.9 多重 fallback

用途：對每份 PDF 文字打 Gemini API 取新檔名 + 解析 AI 回應 + 多層 fallback。

關鍵 jsCode 片段（v0.9）：

// 從 Read 用 binary.data.fileName 反查 readItem（v0.9 真相）
const items = $input.all();
const allReadItems = $('Read PDF').all();

for (let i = 0; i < items.length; i++) {
  const binFileName = items[i].binary?.data?.fileName;
  let readItem = allReadItems.find(r => r?.json?.fileName === binFileName)
              || allReadItems[i];

  // 打 Gemini（thinkingBudget: 0 是必須的，2.5 thinking 會吃光 token）
  const response = await this.helpers.httpRequest({
    method: 'POST',
    url: 'https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent',
    headers: { 'x-goog-api-key': GEMINI_API_KEY },
    body: {
      contents: [{ role: 'user', parts: [{ text: prompt }] }],
      generationConfig: {
        temperature: 0.4,
        maxOutputTokens: 200,
        thinkingConfig: { thinkingBudget: 0 }  // ← 這是關鍵
      }
    },
    json: true
  });

  // 6 層 fallback parser（見下表）
  ...
  results.push({ json: {...}, binary: readItem?.binary });
}
return results;

6 層 fallback parser 表：

層判斷結果
ai-plainAI 純文字 + YYYYMMDD_xxx 格式取整段
regex-dateAI 回應內含 YYYYMMDD_xxx 但混雜文字regex 抓出
pdf-titleAI 沒回應 → 從 PDF 文字抓有意義的標題行用 title 命名
fallback-empty真的什麼都沒有 → 用今日日期 + AI無回應 + 原檔名保底命名
http-error: xxxGemini API 直接 HTTP 錯（key / 額度 / network）標 http-error 命名
finish=MAX_TOKENSthinking 吃光 token（v0.6 之前的 bug）標 finish reason 命名
NODE 5 / 5
Write 改名輸出
readWriteFile v1operation: write

關鍵設定：fileName: {{ $json.newPath }} + dataPropertyName: data

從 Code 節點 return 的 json.newPath（如 /files/shared/pdf-renamed/20260506_說明_Adobe.pdf）寫檔，binary 來自 Code 節點 return 的 binary 欄位。

(03) DESIGN RATIONALE

為什麼這樣設計（含踩過的雷）

🚨
連發 #16 的雷：HTTP Request Node 的 jsonBody 解析陷阱
n8n HTTP Request Node 的 jsonBody 欄位用 ={{ JSON.stringify(...) }} Expression 帶動態內容時，n8n 會二次解析這個 JSON 字串導致 invalid syntax 錯誤。Lite Pack 試了 5 種寫法都炸，最終決定完全廢棄 HTTP Request 節點，改用 Code Node 內 this.helpers.httpRequest() 直接打 API。優點：100% 可控、不會被 n8n 二次解析、debug 容易。
🚨
連發 #17 的雷：Gemini 2.5 Flash thinking budget 吃光 maxOutputTokens
Gemini 2.5 Flash 是 thinking model — 會先「想」再輸出。thinking tokens 算進總 token quota。如果 maxOutputTokens 設 100，全被 thinking 吃光，真實輸出只剩 3 個字「202」就被切斷（finishReason: MAX_TOKENS）。修法：強制設 thinkingConfig: { thinkingBudget: 0 } 完全禁用 thinking，整個 quota 給輸出。對「抽取 + 命名」這種純格式化任務不需要推理，禁掉就好。
⚠️
為什麼 6 層 fallback 而不是 try/catch 一層？
AI 回應有 6 種可能變異：純文字格式對 / 純文字混雜文字 / API 直接掛 / 回應為空 / thinking 吃光 / 回應「我不確定」式廢話。每種都有對應的「次優解」（從 PDF 第一行抓標題、用日期 + 原檔名保底）。一層 try/catch 接住所有情況不夠精準 — 學員看 parseMethod 一眼就知道「現在 AI 處於哪個狀態」做下一步診斷。
💭
為什麼 prompt 要求「直接回檔名一行」？
AI 預設會多話（「好的！這是您要的檔名：xxx，希望對您有幫助」）。在 prompt 末段加「直接回檔名一行（不要 markdown，不要引號，不要解釋文字）」能把 90% 的多話 AI 訓練成只回一行格式化字串。剩下 10% 由 fallback 接住。
(04) 改造方向

把這個 workflow 改成你業務的 5 種方向

  1. 換 prompt 改任務：把 prompt 從「取檔名」換成「抽取合約金額 / 抽取發票統編 / 抽取發票日期 / 摘要 PDF 三句話」。同個架構可變身為 N 種 PDF 處理 workflow。
  2. 換 Trigger：Manual Trigger 換成 Local File Trigger，資料夾有新檔自動跑（n8n 學員 #15 的測試）。或換成 Schedule（每天凌晨 4:00 自動跑昨天的 PDF）。
  3. 串資料庫：在 Write 節點之後加 PostgreSQL / Notion 節點，記錄每個 PDF 的「原檔名 + 新檔名 + AI 解析方法 + 處理時間」，做歷史紀錄查詢。
  4. 換 AI：把 jsCode 內 Gemini URL 換成 OpenAI / Anthropic / 你的 local LLM endpoint。Gemini 2.5 Flash 的 thinkingConfig 換成對方對應的 API spec。
  5. 加多語言支援：偵測 PDF 文字是中/英/日，動態切換 prompt 語言。一個 workflow 處理跨語言文件。
