Walkthrough — 把 #02 改成處理發票

完整 LLM 對話劇本，從零跟讀到通。行銷企劃 A 用 ChatGPT 把 #02-pdf-ai-rename 改成處理 200 張供應商發票——含 4 個故意岔路示範你最可能踩的雷，以及「我可以了」拐點的完整示範。

本頁學習成果
能填出 4 格便箋並對人講 1 分鐘不打結
能讓 LLM 只回傳指定節點 JSON 區段，發現「給整份」時知道怎麼追問
能用三件套回報錯誤，並用紅綠對照表判斷 LLM 回應是精準型還是模糊型
(01)

為什麼選 #02-pdf-ai-rename

選案邏輯
這份 walkthrough 示範「安全改 workflow 的六步流程」。選案標準：改造張力高、LLM 介入的價值大、新手第一次走完不會被 typeVersion 雷區打斷信心。
Workflow 改造張力 LLM 介入價值 雷區覆蓋度 結論
#02-pdf-ai-rename 高（改 prompt + 改檔名 template，不換節點類型） 高（學員看 prompt JSON 容易花，LLM 代讀價值大） 中（不太會踩 typeVersion，credential 紅線會踩） 主案例
#11-csv-clean-score 高（改評分維度 + 改 trigger） 高 高（換 trigger 觸發引用斷裂，能完整示範雷區） 次案例（附錄略寫）
#04-daily-ai-report 中（換輸出端） 高（典型 Story B） 中 留作學員自我練習

主案例選 #02 的四個理由：

（a）對應 Story A——行銷企劃 A 接到 200 張發票任務，是結業後最高頻的改造情境，學員看到就有即視感。

（b）不需要換節點類型，整條 workflow 維持 Manual Trigger → Read PDF → Extract PDF Text → Code（AI 改名）→ Write 五節點結構，只需改 Code 節點裡的 prompt 區段與檔名 template，學員第一次走六步流程時不會被 typeVersion 或 connections 雷區打斷。

（c）#02 的核心修改對象是 Code 節點裡的一段 prompt 字串——這是「LLM 幫你改 LLM 看的 prompt」的 meta 操作，新手看一次就記住「LLM 是 JSON 代讀工具」這個核心概念。

（d）改完的驗收訊號清晰——輸出的 PDF 檔名從「YYYYMMDD_合約_XXX.pdf」變成「YYYYMMDD_發票_XXX.pdf」，一眼就看出對不對。

2.4 ChatGPT 回應（sample）

ChatGPT 回應（理想版）
這份 workflow 做什麼： 從 /files/pdf-inbox/ 讀取資料夾內所有 PDF，用 Gemini API 分析每份 PDF 的 文字內容後產出一個語意化檔名（格式 YYYYMMDD_類型_關鍵字），最後把原 PDF 用新檔名移到 /files/pdf-renamed/ 資料夾。整套流程是手動觸發。 節點清單（5 個）： 1. Manual Trigger - 角色：手動觸發點 - 輸出：空訊息（觸發訊號） - 下一個節點吃：觸發訊號 2. Read PDF - 角色：列出 /files/pdf-inbox/ 內所有 .pdf 檔 - 輸出：每張 PDF 的 binary data + 原始檔名 - 下一個節點吃：binary 3. Extract PDF Text - 角色：把 PDF binary 轉成純文字 - 輸出：pdfTextPreview 欄位（純文字） - 下一個節點吃：pdfTextPreview 4. Code: AI 改名（內含 Gemini call） - 角色：呼叫 Gemini API，根據 PDF 文字產出新檔名字串 - 輸出：newFilename 欄位（字串） - 下一個節點吃：newFilename + 原 binary 5. Write - 角色：把原 PDF 用 newFilename 改名後寫到 /files/pdf-renamed/ - 輸出：寫入結果（成功/失敗） - 下一個節點：（流程結束） 資料流向：Manual Trigger → Read PDF → Extract PDF Text → Code → Write

(05)

步驟 3：限定修改範圍

學員 + LLM 協同，明確指定只改哪裡
這一步是六步流程裡最容易犯錯的地方。「叫 LLM 幫你改」不是「貼 JSON 說你要什麼」——而是「告訴 LLM 只動指定區段，其他節點一律不碰，並且只回傳那幾個節點的 JSON」。

3.1 A 決定只動 1 個節點

回到 4 格便箋第 4 格「必保留節點」，看到只有 Code 節點是要改的。其他 4 個節點完全不動。

3.2 A 在 n8n UI 備份 Code 節點

點 Code 節點 → Cmd+C → 在 macOS Notes 開一條新筆記 code-backup-2026-05-07.txt → 貼上。看到內容開頭是 { "parameters": ...，存檔。這是萬一 LLM 給的內容改壞了，可以立刻貼回去的保險。

3.3 A 貼模板 3 給 ChatGPT

A 貼 ChatGPT — 模板 3（限定修改範圍）

基於剛才那份 workflow，只改節點「Code: AI 改名」（其他節點不動）。 我要的修改： - prompt 字串改成抽取賣方公司全名（保留「股份有限公司」等後綴）、 稅後合計（純數字，不含元）、發票日期（YYYYMMDD 格式） - 檔名 template 改成：{YYYYMMDD}_發票_{賣方名}_{稅後合計}.pdf - 類型欄位固定填「發票」，不要讓 AI 自己判斷 - 其他邏輯（fallback、parseMethod、httpError 處理）全部保留 請回傳「只有這個 Code 節點」的 JSON 區段（不要整份 workflow）。 回應格式：直接給 JSON 物件，最外層 { 後面第二行是 "parameters":， 不要 markdown code fence。

ChatGPT 回應（理想版）— 第 2 輪修正
抱歉理解錯。這次只給 Code 節點： { "parameters": { "mode": "runOnceForEachItem", "jsCode": "const text = $input.item.json.pdfTextPreview;\nconst prompt = `請從以下發票文字抽取：賣方公司全名（保留「股份有限公司」後綴）、稅後合計（純數字，不含元字）、發票日期（YYYYMMDD 格式）。回傳 JSON：{seller, totalAmount, invoiceDate}。發票文字：${text}`;\n\ntry {\n const response = await this.helpers.httpRequest({\n method: 'POST',\n url: 'https://generativelanguage.googleapis.com/v1beta/models/gemini-pro:generateContent',\n body: { contents: [{ parts: [{ text: prompt }] }] },\n json: true\n });\n const result = JSON.parse(response.candidates[0].content.parts[0].text);\n const newFilename = `${result.invoiceDate}_發票_${result.seller}_${result.totalAmount}.pdf`;\n return { json: { newFilename, ...result } };\n} catch (e) {\n return { json: { newFilename: `${$input.item.json.fileName}_unparsed.pdf`, error: e.message } };\n}" }, "name": "Code: AI 改名", "type": "n8n-nodes-base.code", "typeVersion": 2, "position": [820, 300] }

現實是 LLM 第二輪修正不會永遠像上面這麼乾淨。你可能會看到下面 3 種「歪掉但常見」的回應，學員必須能辨認後決定下一步：

ChatGPT 回應（歪掉）變體 A：夾雜其他節點說明

辨認訊號 + 學員下一步（變體 A）
辨認訊號：回應字數突然變多、提到 Trigger / Read PDF / Extract PDF Text 等你模板 3 已經寫「不動」的節點。
學員下一步：再貼一次「精簡追問」：「請只回我問的 Code 節點 JSON，不要說明其他節點。」不要試圖從這段 mixed response 裡摘出 Code 節點。
ChatGPT 回應（歪掉）變體 B：漏了欄位
{ "parameters": { "mode": "runOnceForEachItem", "jsCode": "..." }, "name": "Code: AI 改名", "type": "n8n-nodes-base.code" }
辨認訊號 + 學員下一步（變體 B）
辨認訊號：對照 3.7 的完整版，發現少了 typeVersion 與 position 兩行。
學員下一步：不用再叫 LLM 重來，自己手補回去就好（從 3.2 的 backup 抓 "typeVersion": 2 與 "position": [820, 300] 兩行貼回來）。LLM 第 3 輪重做的成本比手補高。
ChatGPT 回應（歪掉）變體 C：先說設計考量

辨認訊號 + 學員下一步（變體 C）
辨認訊號：回應第 1 段是「設計考量／我這樣做是為了⋯⋯」這類說明文，不是直接 JSON。
學員下一步：把這段說明文整段忽略，直接 scroll 找 { "parameters": 開頭那段 code 拿來貼。不需要再追問，這只是 LLM 想多講話而已。
💡
這 3 種變體的共通教學價值：LLM 第 2 輪不會永遠乾淨，但乾淨與否其實不影響學員能不能完成任務——只要你能辨認「這段是垃圾、那段是我要的 Code 節點」，剩下的動作（追問 / 手補 / 直接拿 code 段）都很短。卡學員的不是 LLM 不乾淨，是學員以為「LLM 一回不乾淨就要重新來過」。

6 收斂（10 張測試 + 1 張退款邊界 + LLM 擴張拒絕） 10 分鐘 10-15 分鐘
7 全量 + 驗收（200 張跑 4 分鐘 + 1 張失敗處理 + Finder 確認） 10 分鐘 10-15 分鐘
合計 90 分鐘 90-120 分鐘

容易卡點的 Step 3 與 Step 5 原因見各 step 段首的「學員容易卡」標籤——不是 LLM 對話的問題，是 n8n UI 操作 + JSON / Output panel 閱讀對非工程師是雙重門檻。

A 心算：第 2 次再做這種改造（例：把另一個 workflow 改成處理收據），預估 30-40 分鐘。她在便箋最後寫：

A 自填 — 便箋心得
心得：90-120 分鐘的首次投資，省下 200 張 × 30 秒手動分類 = 100 分鐘（單次）。 但更重要：下個月 200 張、再下個月 200 張⋯⋯一年省 20 小時。
(11)

A 給其他學員的 5 個提醒

章末總結
走完整份 walkthrough 後，A 整理了 5 條她希望開始之前就知道的事。
1
第 1 步別跳：4 格便箋花 15 分鐘，省下後面 60-90 分鐘。跳了步驟 1 直接貼 LLM，LLM 會猜欄位名稱，猜錯了你不知道為什麼 null_null。
2
LLM 第 1 輪很常給整份：別怕，用追問模板要它重給就好。給歪的變體（混入其他節點 / 漏欄位 / 一堆說明文）見步驟 3 的三種辨認法——乾淨與否不影響你能不能完成任務。
3
綠燈不等於對：永遠看 output 的關鍵欄位是不是 null。全綠但 null_null 是「軟錯誤」，進步驟 5 用三件套回報。
4
LLM 不一定每次都建議加 try/catch（ChatGPT 在你明確說「只動 X」時多半收斂；換 Claude / Gemini 或 prompt 鬆了它就會擴張）：只要它建議擴張，就用模板 5 拒絕。保持簡單才好驗收。
5
20 / 80 原則：邊界 case（退款單、掃描圖片）先記備注人工處理，不為 1-2 張改流程。改流程增加複雜度，反而讓下次 debug 更難。

彩蛋：A 的便箋備注怎麼存

走完這份 walkthrough，你手邊會有一份填好的 4 格便箋。這份便箋是這個改造案例的「設計記錄」。三個地方可以存：

（a）n8n workflow 描述欄——最直接，打開 workflow 就看到，不怕遺失。

（b）實體便利貼——貼在螢幕邊框，下個月發票批次跑之前先看一眼。

（c）團隊文檔（Notion / Google Doc）——如果這個流程將來要交給別人，共用文檔比 workflow 描述欄好找。

A 選了（a）+ （b）：workflow 描述欄存技術細節（邊界 case / 失敗案例），便利貼存「上次跑的日期 + 200 張 / 199 張」，讓下次批次跑之前 10 秒就能確認狀態。
