Walkthrough — 把 #02 改成處理發票

實作示範（建構中）— 從合約 PDF 改名版出發，完整走完六步安全改 workflow 流程，最終產出可處理 200 張供應商發票的自動歸檔版本。

本頁學習成果
能填出 4 格便箋（input / output / 保留節點 / 編號），對改造需求做出清晰定義
能讓 LLM 只回傳指定節點的 JSON 區段，不重寫整份 workflow
能用最小可重現訊息格式回報錯誤，讓 LLM 精準定位問題欄位
(01)

為什麼選 #02-pdf-ai-rename

選案邏輯
這份 walkthrough 要示範「安全改 workflow 的六步流程」，選案標準是：改造張力高、LLM 介入的價值大、新手第一次走完不會被 typeVersion 雷區打斷信心。以下是三個候選方案的完整對照。
Workflow 改造張力 LLM 介入價值 雷區覆蓋度 結論
#02-pdf-ai-rename 高（改 prompt + 改檔名 template，不換節點類型） 高（學員看 prompt JSON 容易花，LLM 代讀價值大） 中（不太會踩 typeVersion，credential 紅線會踩） 主案例
#11-csv-clean-score 高（改評分維度 + 改 trigger） 高 高（換 trigger 觸發引用斷裂，能完整示範雷區） 次案例（附錄略寫）
#04-daily-ai-report 中（換輸出端） 高（典型 Story B） 中 留作學員自我練習

主案例選 #02 的四個理由：

（a）對應 Story A——行銷企劃 A 接到 200 張發票任務，是結業後最高頻的改造情境，學員看到就有即視感。

（b）不需要換節點類型，整條 workflow 維持 Manual Trigger → Read PDF → Extract PDF Text → Code（AI 改名）→ Write 五節點結構，只需改 Code 節點裡的 prompt 區段與檔名 template，學員第一次走六步流程時不會被 typeVersion 或 connections 雷區打斷。

（c）#02 的核心修改對象是 Code 節點裡的一段 prompt 字串——這是「LLM 幫你改 LLM 看的 prompt」的 meta 操作，新手看一次就記住「LLM 是 JSON 代讀工具」這個核心概念。

（d）改完的驗收訊號清晰——輸出的 PDF 檔名從「YYYYMMDD_合約_XXX.pdf」變成「YYYYMMDD_發票_XXX.pdf」，一眼就看出對不對，不需要進 n8n UI 比對資料形狀。

(02)

設定情境

學員視角
這份教材的主角是行銷企劃 A——她代表「結業後一到兩週、遇到具體工作任務才回來翻教材」的典型結業學員。

行銷企劃 A 三週前完成了 n8n 課程，用 #02-pdf-ai-rename 跑通了合約 PDF 的自動改名流程，感覺良好地交出成果。

今天早上，主管轉來一封信：「這個月開始，財務要求所有供應商發票都統一歸在 /檔案/2026/發票/{月份}/{供應商名}_{稅後金額}.pdf，現在手邊有一批 200 張 PDF，你來處理一下。」

A 打開 n8n，找到 #02，看了一下 Code 節點裡的 prompt：「請為這份 PDF 文件取一個語意化檔名（繁體中文）。格式：YYYYMMDD_類型_關鍵字……類型 2-4 字（合約 / 發票 / 報告 / 說明……）」她意識到：現在的 prompt 是針對合約設計的，欄位是「甲乙方 + 合約金額 + 簽訂日」，但發票需要抽的是「供應商統編、品項、稅後金額、發票日期」。

她想到課程教的六步流程——第一步不是直接問 LLM，而是先把需求寫清楚。她打開便箋 app，準備填 4 格。

A 知道自己的弱點：上次用 Make 時，她直接跟 AI 說「幫我改這個 workflow 讓它處理發票」，AI 給了一份看起來很長的 JSON，她照貼進去，匯入時 n8n 跳了一個「不支援的格式」錯誤，折騰了一個下午才放棄。這次她決定照著教材走完每一步。

這是這份 walkthrough 的起點：不是「我不會用 n8n」，而是「我會用 n8n，但我不知道 LLM 在這套流程裡該扮演什麼角色」。六步流程就是為了回答這個問題。

(03)

步驟 1：拆需求

學員主導，LLM 不介入
步驟 1 刻意不用 LLM。需求模糊時 LLM 會腦補，越改越偏。逼自己填這 4 格，後面的每個步驟才有參照點。

A 拿起便箋，把四格填完。以下就是她填的內容——這不是空白範本，是真的填了一次：

4 格便箋範本 — 本次改造
原 Workflow 編號
#02-pdf-ai-rename（合約版，v0.6）
目前 prompt 抓：甲乙方、合約金額、簽訂日
目標 Input
供應商發票 PDF（每月 200 張）
來源資料夾：/files/shared/pdf-inbox/
每張 PDF 含：供應商名稱、統編、品項明細、稅後金額、發票日期
目標 Output
改名後 PDF，格式：
YYYYMMDD_發票_{供應商名}_{稅後金額}.pdf
存至：/files/shared/pdf-renamed/
必須保留的節點
Manual Trigger（保留）
Read PDF（保留）
Extract PDF Text（保留）
Write: 改名輸出（保留路徑邏輯）
只改：Code node 裡的 prompt 字串與檔名 template

驗收訊號：A 把這 4 格唸給同事聽，1 分鐘內講完不打結。同事問「供應商名稱要全名還是簡稱？」——A 補記：取 PDF 第一頁的公司名稱全名，不截短。這個細節後來在步驟 3 的 prompt 裡會用到。

⚠
常見跳步錯誤：直接跟 LLM 說「幫我把 #02 改成處理發票」，沒有給 input/output 的具體形狀。LLM 拿不到「供應商統編、品項、稅後金額、發票日期」這些欄位名，會自己猜——猜出來的欄位名如果跟 PDF 裡的欄位對不上，Code 節點的 prompt 就會抓錯資料，改名結果一塌糊塗。
(04)

步驟 2：讀 JSON

LLM 主導翻譯，學員驗收
把 #02 的 JSON（移除 credential 後）貼給 LLM，要它翻成人話：這份 workflow 做了什麼、每個節點的角色與資料流向。LLM 做這件事比學員快 50 倍。

A 在 n8n UI 裡找到 #02，點右上角「...」→「Export」→ 確認選了「Export without credentials」（預設就是這個，但要確認一下）。下載得到 02-pdf-ai-rename.json。

她用文字編輯器打開，先搜尋 apiKey、accessToken、password 三個字串——確認找不到後，才把整份 JSON 貼到 LLM 的對話框。

她使用的 prompt 是教材第 3 頁提供的模板 2：

模板 2 — 步驟 2 讀 JSON
我是 n8n 非工程師使用者，請以結果為主、不解釋原理。
以下是一份 n8n workflow JSON（已移除 credential）。請：
1. 用一段話講它做了什麼。
2. 列出每個節點的角色與資料流向（節點名稱 → 它輸出什麼 → 下一個節點吃什麼）。
不要建議改進、不要寫程式碼。
[貼 JSON]

LLM 應該會回傳 #02 的結構摘要——大致是「讀 PDF 資料夾裡所有 PDF → 用 Gemini API 分析內文並產出語意化檔名 → 把 PDF 用新檔名存到 renamed 資料夾」，並列出 5 個節點的角色與資料流向。下方是這個對話的佔位符，待維運時補入實機截圖。

[需錄製：LLM 對話截圖 — 步驟 2 翻譯回應。內容應包含：(1) 學員貼入 JSON + 模板 2 prompt；(2) LLM 回傳 #02 的流程摘要「Manual Trigger → Read PDF → Extract PDF Text → Code（Gemini 改名）→ Write 改名輸出，5 節點」；(3) 學員在 n8n UI 上對照確認節點順序正確]

對照確認重點：LLM 說的「Code 節點做什麼」是否與 n8n UI 上顯示的一致。如果 LLM 說「HTTP Request 節點打 Gemini API」但 n8n UI 上顯示的是「Code 節點（內含 helpers.httpRequest）」，代表 LLM 讀到的是舊版結構——這個版本（v0.6）已經把 HTTP Request 節點合進 Code 節點了。這種小不一致不影響後續，但要記下來，步驟 3 限定修改範圍時要明確說「我只要改 Code 節點裡的 prompt 字串」。

💡
驗收訊號：A 用自己的話複述了一遍 LLM 的描述，確認能講出「這份 workflow 讀 pdf-inbox 資料夾的 PDF，用 Gemini 分析內文後給出新檔名，存到 pdf-renamed 資料夾」。這句話她 30 秒就說完了，而且準確——步驟 2 完成。
(05)

步驟 3：限定修改範圍

學員 + LLM 協同，明確指定只改哪裡
這一步是六步流程裡最容易犯錯的地方。「叫 LLM 幫你改」不是「貼 JSON 說你要什麼」——而是「告訴 LLM 只動指定區段，其他節點一律不碰，並且只回傳那幾個節點的 JSON」。

A 根據步驟 2 的確認，決定只改一個節點：Code: AI 改名（內含 Gemini call），也就是 id 為 ai-02 的節點。改動範圍如下：

1
備份 Code 節點
在 n8n UI 上選中 Code 節點，點右鍵「Copy」，把節點 JSON 貼到一個臨時 .txt 檔案存起來。這是萬一 LLM 給的內容改壞了，可以立刻貼回去的保險。
2
用模板 3 下指令
基於剛才的對話（LLM 已經讀過 #02 了），用模板 3 明確告訴 LLM：只改 Code 節點裡的 prompt 字串，不動其他任何節點，回傳「只有 Code 節點」的 JSON 區段。
3
確認 LLM 回傳的是區段，不是整份
如果 LLM 回傳的 JSON 開頭是 { "nodes": [，代表它給了整份 workflow——這不是你要的，要求它只給 Code 節點本身。這是這個步驟最常犯的反例，後面有示範。

A 使用的 prompt（模板 3 填入版）：

模板 3 填入 — 步驟 3 限定修改範圍
我是 n8n 非工程師使用者，請以結果為主、不解釋原理。
基於剛才那份 workflow，只改節點「Code: AI 改名（內含 Gemini call）」，不動其他節點。
我要的修改：
- prompt 字串改成抽取供應商名稱（公司全名）、稅後金額（數字含元）、發票日期（YYYYMMDD 格式）
- 檔名 template 改成：YYYYMMDD_發票_{供應商名}_{稅後金額}.pdf
- 類型欄位固定填「發票」，不要讓 AI 自己判斷類型
- 其他邏輯（fallback、parseMethod、httpError 處理）全部保留不動
請回傳「只有這個 Code 節點」的 JSON 區段（不要整份 workflow）。

第一輪的對話佔位符：

[需錄製：LLM 對話截圖 — 步驟 3 第一輪。內容應顯示：(1) A 貼入模板 3 填入版；(2) LLM 回傳 Code 節點 JSON 區段（只有 Code 節點，不是整份 workflow）；(3) 回傳的 prompt 字串已正確改成抽取「供應商名稱、稅後金額、發票日期」，檔名 template 改成 YYYYMMDD_發票_{供應商名}_{稅後金額}.pdf]

但這裡有個常見反例要示範——如果 LLM 第一輪回傳的是整份 workflow 怎麼辦？A 可能會遇到這樣的回應：LLM 貼回一個長 JSON，最外層是 { "nodes": [...], "connections": {...} }，看起來像整份 workflow。這時候要追問：

反例修正追問
你給的 JSON 是整份 workflow 還是只有 Code 節點的區段？
我只要那個 Code 節點本身（一個 JSON 物件，從 { "parameters": 開始），不要整份 workflow。
請重新只給 Code 節點。
[需錄製：LLM 對話截圖 — 步驟 3 修正後。內容應顯示：(1) A 追問「你給的是整份還是區段」；(2) LLM 修正後只回傳 Code 節點 JSON 物件（從 { "parameters": {...} 開始的單個節點物件）；(3) A 確認這是她要的格式]

確認格式正確後，A 回到 n8n UI：在 Code 節點上點右鍵「Paste」，把 LLM 給的內容貼上去替換掉原本的 Code。然後點一下 Code 節點，進入編輯模式，眼掃一遍 prompt 字串——確認「甲乙方、合約金額、簽訂日」這幾個字已經消失，換成了「供應商名稱、稅後金額、發票日期」。

💡
驗收訊號：在 n8n UI 上，Manual Trigger → Read PDF → Extract PDF Text → Code → Write 五個節點還在原位，連線沒斷，其他四個節點的內容原樣。只有 Code 節點的 prompt 字串改了。
(06)

步驟 4：匯入測試

學員主導，用 1 張樣本跑
永遠先用 1 筆測試。用 200 張發票直接測會把錯誤埋在中段，debug 成本乘以 200。

A 先在 n8n UI 右上角點「Duplicate」，複製一份命名 02-pdf-ai-rename-invoice-edit。這是為了確保原本的 #02（合約版）在改造期間不受影響——紅線 7 要求：不在 production workflow 上直接測試。

接著，她在 pdf-inbox 資料夾裡放了一張供應商發票 PDF（是真的發票 PDF，不是合約）。然後在 n8n UI 上打開剛複製的 02-pdf-ai-rename-invoice-edit，確認 trigger 是 Manual（不是 Active），點下紫色「Execute Workflow」按鈕。

第一次執行的動線截圖：

如果所有節點都綠燈，A 會去 pdf-renamed 資料夾確認新檔名格式是否符合「YYYYMMDD_發票_{供應商名}_{稅後金額}.pdf」。如果格式對了，步驟 4 完成，可以跳到步驟 6 收斂。如果有紅燈，進步驟 5。

⚠
這份 walkthrough 故意安排了一個錯誤：Code 節點的 prompt 裡，LLM 用了「甲方」這個欄位名稱的慣性——它改成了「供應商」，但實際 PDF 裡這個欄位的顯示文字是「賣方」（台灣電子發票的法定用語）。這造成 Gemini 在發票裡找不到「供應商」，改名時抽出的金額是 null，最後產出 20260506_發票_null_null.pdf。進步驟 5 示範如何回報這個錯誤。
(07)

步驟 5：錯誤回報

學員 + LLM 協同，最小可重現訊息
步驟 5 的核心原則：不要把整份 workflow 重貼 LLM 說「幫我重改」，要只回報「哪個節點、哪個欄位、出了什麼問題」。縮小範圍才能讓 LLM 給出精準修補，而不是產出一份新的整體草稿。

A 看到輸出的檔名是 20260506_發票_null_null.pdf——null 出現了兩次，代表供應商名稱和稅後金額都沒有被正確抽出來。

她的操作步驟：

1
找到 Code 節點的 output
點 Code 節點，在右側 output panel 找到 json.aiResponsePreview——確認 Gemini 的回應字串長什麼樣，看它有沒有抓到欄位。
2
找 pdfTextPreview
同樣在 output panel 找 json.pdfTextPreview，看 PDF 實際抽出來的文字長什麼樣——這裡會看到是「賣方」而不是「供應商」。
3
用模板 4 組裝錯誤回報
把「節點名稱 + 錯誤訊息 + input 情況 + 當前 parameter」三件套貼給 LLM，不貼整份 workflow，不說「幫我重改」。

A 組裝的錯誤回報（模板 4 填入版）：

模板 4 填入 — 步驟 5 錯誤回報
我是 n8n 非工程師使用者，請以結果為主、不解釋原理。
我把 Code 節點替換成你上一輪給的內容後，跑出來的檔名是 20260506_發票_null_null.pdf。
問題：供應商名稱和稅後金額都是 null。
我看了 pdfTextPreview，PDF 裡的欄位是「賣方」（不是「供應商」）和「稅後合計」（不是「稅後金額」）。
目前 prompt 用的字是「供應商名稱」和「稅後金額」，但發票 PDF 裡對應的中文是「賣方」和「稅後合計」。
請只修改 prompt 裡這兩個欄位的中文名稱，其他不動，再給我 Code 節點區段一次。
[需錄製：LLM 對話截圖 — 步驟 5 錯誤回報與修補。內容應顯示：(1) A 貼入模板 4 填入版（含「賣方」vs「供應商」的欄位名稱差異說明）；(2) LLM 指出問題欄位，給出 prompt 字串修正後的 Code 節點區段；(3) 修正後的 prompt 使用「賣方」和「稅後合計」兩個正確欄位名稱]

A 把修補後的 Code 節點貼回 n8n UI，重跑一次——這次 output 應該是正確的發票公司名稱和金額。如果還是 null，代表回報資訊不夠，她應該繼續補資料（多貼一部分 pdfTextPreview），不是換一個 LLM 重試。

💡
驗收訊號：LLM 能指出「問題在 prompt 裡的欄位名稱與 PDF 中的文字不一致」，並給出只改那兩個欄位名稱的修正版——沒有重寫整個 Code 節點，沒有建議換節點架構。這就是「最小可重現訊息的價值在縮小範圍」的實際效果。
(08)

步驟 6：收斂

迴圈到通過，不超過 3 輪
步驟 3–5 迴圈，每輪問自己「比上一輪更接近目標嗎？」超過 3 輪沒收斂，退回步驟 1 重看需求。

修補欄位名稱後，A 重跑一次單張發票測試——這次輸出了正確格式的檔名。她確認後，把 pdf-inbox 裡的樣本換成 10 張不同供應商的發票（測試集），重新 Execute Workflow，觀察 10 個 output 的 newFilename 是否都符合「YYYYMMDD_發票_{賣方全名}_{稅後合計}.pdf」格式。

9 張正確，1 張出現 20260506_發票_台灣商業發票有限公司_稅後合計_0.pdf——稅後金額變成了 0。A 去翻那張 PDF 的原始文字，發現那張發票的「稅後合計」欄位是「NT$ 0（退款單）」，LLM 把「0」抽出來是正確行為，但金額是 0 代表那是一張退款單，不是一般發票。她決定這個邊界情況不在這次的改造範圍內，記了一筆備注給未來的自己。

此時 LLM 在對話裡提出了一個建議，這個對話很典型，需要示範：

[需錄製：LLM 對話截圖 — 收斂迴圈，含 try/catch 拒絕。內容應顯示：(1) LLM 主動建議「如果稅後合計是 0，可以加一個 try/catch 包住整個 API call，並在 catch 裡加一個 skip 邏輯跳過退款單」；(2) A 回應「我不需要 try/catch，原本的 Code 節點已經有 httpError 處理，退款單是邊界情況先記備注，這次不改」；(3) LLM 確認並收斂，不再追加防禦邏輯]

這段對話展示了「LLM 傾向加防禦邏輯」的本性——每多一輪對話，LLM 就多一點想把方案做得更完整。A 的做法是對的：明確說「不需要 try/catch，保留最小修改」。教材設計提案第 3 節步驟 6 的紅線原文：「LLM 第 3 次回覆比第 1 次長 30%，停下來，要求它保留最小修改、移除新增的防禦邏輯。」

10 張測試通過（退款單邊界情況已記備注，不納入本次改造範圍），A 進入驗收交付步驟。

(09)

驗收交付

真實樣本 → 全量執行
10 筆測試通過後，才用全量樣本跑。這是六步流程的最後一個安全門。

10 張測試通過後，A 把整批 200 張發票 PDF 複製到 pdf-inbox 資料夾，在 n8n UI 上點 Execute Workflow——這次是真正的全量執行。

改造後的 workflow 結構與原 #02 完全相同：Manual Trigger → Read PDF → Extract PDF Text → Code: AI 改名（內含 Gemini call）→ Write: 改名輸出，五個節點，連線完整，其他四個節點未動。唯一的差異是 Code 節點裡的 prompt 字串：原本抓「甲乙方、合約金額、簽訂日」，現在改成抓「賣方（公司全名）、稅後合計、發票日期」，並且把類型欄位固定為「發票」，不讓 AI 自行判斷。

執行完成後，pdf-renamed 資料夾裡應該出現 200 張以「YYYYMMDD_發票_{賣方名}_{金額}.pdf」命名的檔案。A 用 Finder 的排序功能確認：所有檔案都有正確的日期前綴，類型欄位全部是「發票」，公司名稱和金額的組合看起來合理。退款單那一張（稅後合計為 0）也正確改名為 20260506_發票_{賣方名}_0.pdf，沒有跑錯，只是備注裡記了「這張是退款單，金額欄位邏輯正確」。

最後，A 在 n8n UI 把 02-pdf-ai-rename-invoice-edit 這份測試複本，把 workflow 名稱改成 #02-pdf-invoice-rename（v1.0） 並標記為已驗收，原本的 #02-pdf-ai-rename（合約版）保持不動，下次處理合約還是用原版。
