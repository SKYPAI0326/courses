8 頁系列 · 第 1/8 · 開場與設計取捨

用 LLM 協助改寫 n8n Workflow — 結業後 Copilot 操作手冊

你手上有 14 個 Lite Pack workflow，也會用 ChatGPT 或 Claude。這份手冊教你把兩者結合，安全地把現成 workflow 改成自己需要的版本——不是靠 LLM「幫你做自動化」，而是用 LLM 當改 workflow 的協作工具，你來驗收、你來決定。

讀完這頁你會知道 · OUTCOMES
這份手冊教什麼、不教什麼，以及為什麼這樣切

(01)

這份手冊教你做什麼、不教你做什麼

為什麼要把範圍講清楚
很多人拿到這份手冊的第一個念頭是「去跟 LLM 說幫我改 #02 處理發票就好了」。這樣做會拿到一坨匯不進去的 JSON，然後花一個小時 debug 或直接放棄。這份手冊存在的理由，就是讓你不走這條路。

這份手冊的核心是一套六步流程：拆需求 → 讀 JSON → 限定修改範圍 → 匯入測試 → 錯誤回報 → 收斂。流程的每一步都有「LLM 在這做什麼 / 你在這做什麼 / 怎麼知道自己做對了」的明確分工。走完一次之後，你就能把任何一個 Lite Pack workflow 改成自己情境的版本。

教什麼

安全改 workflow 的六步流程
從拆需求到收斂，每一步有明確的學員動作與 LLM 動作，有驗收訊號，有常見雷區提示
每一步對應的 prompt 模板
5 個可直接複製的 prompt，每個 ≤ 8 行，有 placeholder，有「踩雷實例」對照
8 條安全紅線
哪些操作絕對不能交給 LLM 決定，每條一句話加一行原因，可印出貼螢幕旁
完整 walkthrough 案例
把 #02-pdf-ai-rename 改成處理發票，從頭走完六步，含一次故意安排的錯誤回報示範
錯誤分流 SOP
匯入失敗 / 跑不通 / 結果不對，三類錯誤各有 checklist，不用靠感覺猜

不教什麼，以及為什麼

通用 LLM prompt 技巧
few-shot、CoT、role-play 這些技法在這裡沒用——你的目標是把 #02 改成處理發票，不是學 prompt theory。提到 prompt 只在「對應流程某一步、有具體任務」的脈絡
n8n schema 欄位百科
parameters / typeVersion / credentials 各層全解——這會讓非工程師誤以為「要先搞懂底層格式才能改 workflow」，實際上用 LLM 代讀就夠了
手改完整 JSON
這份手冊的核心主張：你不該手改全文。JSON 由 LLM 在限定範圍內改，你在 n8n UI 上做最後替換，靠匯入結果驗收
!
你的勝利條件是「我把 #02-pdf-ai-rename 改成處理發票而且跑通了」，不是「我搞懂了 n8n 的 typeVersion 機制」。這份手冊的成敗用這個指標驗收。
(02)

你會在什麼時候用到這套

真實節奏
結業學員的真實節奏不是「課後立刻動手改」，而是「過一陣子遇到具體痛點才回來找 workflow」。以下 5 個 story 按時間軸排列，看看哪個最像你的情況。
結業 1 週
把 #02-pdf-ai-rename 改成處理發票
行銷企劃結業後回到工作崗位，接到任務「每月 200 張供應商發票要依格式歸檔」。開啟 #02 看到 prompt 是針對合約寫的，不會自己改。需要的是：讓 LLM 把「抽甲乙方+合約金額+簽訂日」改成「抽供應商統編+品項+稅後金額+發票日期」，並把改名 template 換掉。
結業 2 週
把 #04-daily-ai-report 的輸出端從 Google Sheet 換成 Notion
公司剛全面換到 Notion，原 #04 把摘要寫進 Google Sheet。對 LLM 說「把寫 Sheet 的節點換成寫 Notion」，LLM 會需要你提供 Notion 節點的「目標形狀」——這份手冊教你先在 n8n UI 拉一個 Notion 節點、隨便填、複製出來給 LLM 當範本，而不是讓 LLM 從記憶生成（typeVersion 容易錯）。
結業 1 個月
把 #11-csv-clean-score 接在 Google Drive 觸發器上
同事每週丟到共享 Drive 的 CSV 想自動清洗評分，但 #11 是手動上傳觸發。把 trigger 從 Manual 換成 Google Drive Trigger 之後，下游節點引用的 input 路徑也跟著變了——這是 M2 教過的「資料形狀變了，引用會壞」，但學員不會自己抓出有哪些下游引用要跟著改。
結業 1 個月
把 #09-gmail-categorize 改成只看特定客戶來信
信箱雜訊太多，希望 #09 只處理 3 個 VIP 客戶域名的信。這其實是個很簡單的修改（加一個 IF 過濾），但直覺上可能會讓 LLM「重寫整個 workflow」，拿到一份和原 #09 細節漂移的版本。手冊的核心原則之一：最小修改——只插入一個 IF，不重寫全圖。
結業 3 個月
Lite Pack 跨大版本後重跑壞掉
半年後重新啟動 #06-webhook-gemini-file，發現 n8n 已升 1–2 個大版本，部分節點 typeVersion 過時，匯入時跳警告。被「typeVersion mismatch」嚇到，不知道是「不能用了」還是「升一下就好」——手冊教你先用 n8n UI 的「open & save」自動升級，再把 LLM 當 second opinion，不要一開始就讓 LLM 改 typeVersion 數字。
i
以上 5 個 story 的共通模式：學員都不是「想學 n8n」，是「想完成具體任務」。這份手冊的每一節都扣回某個 story，而不是用知識點清單填版面。
