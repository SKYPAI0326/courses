整份丟給 ChatGPT 叫它改 workflow，你會拿到一坨匯不進去的 JSON，然後花兩小時 debug 或放棄。按 6 步驟限定範圍改，首次 40 分鐘、第 3 次 15 分鐘就會內化成例行操作。

(01)

「整份丟 LLM 改」會失敗，原因很具體

學員最常踩的第一個坑
拿到 #02 看到 prompt 是針對合約寫的，第一個念頭幾乎都是「叫 ChatGPT 幫我改成處理發票就好了」。結果拿到一份「看起來能用」但匯入時跳 invalid format 的 JSON，花一小時 debug 之後放棄。

以下是那個直覺反應會對應的 prompt 長什麼樣，以及為什麼它必然失敗：

反例 prompt（不要這樣用）：
「我用 n8n 的 #02 workflow，請幫我改成處理發票，給我完整的 JSON 我直接匯入。」

為什麼這樣會失敗：
1. LLM 不知道你的發票 PDF 長什麼樣（哪些欄位）
2. LLM 不知道你裝的 n8n 是哪一版（節點 typeVersion 會猜錯）
3. LLM 從記憶生成的整份 JSON，節點 ID 對不上、connections 引用斷裂
4. 你拿到一份「看起來能用」但匯入時跳「invalid format」的 JSON
5. 你花 1 小時 debug 後放棄，回到手動分類

這份手冊的核心是一套六步流程：拆需求 → 讀 JSON → 限定修改範圍 → 匯入測試 → 錯誤回報 → 收斂。整套流程裡最容易失手的是步驟 3「限定修改範圍」——你的任務不是「讓 LLM 重寫整個 workflow」，是「讓 LLM 只改你指定的那一段」。這份手冊存在的最大價值就是教你做對這一步。

!
你的勝利條件是「我把 #02-pdf-ai-rename 改成處理發票而且跑通了」，不是「我搞懂了 n8n 的 typeVersion 機制」。時間估算：第 1 次走完六步約 40 分鐘；第 3 次約 15 分鐘，流程會內化成例行操作。
(02)

你會在什麼時候用到這套

真實節奏
結業學員的真實節奏不是「課後立刻動手改」，而是「過一陣子遇到具體痛點才回來找 workflow」。以下 5 個 story 按時間軸排列，選出最像你的情況，對應的章節連結直接跳過去。
結業 1 週
把 #02-pdf-ai-rename 改成處理發票
行銷企劃回到工作崗位，接到任務「每月 200 張供應商發票要依格式歸檔」。開啟 #02 看到 prompt 是針對合約寫的，需要讓 LLM 把「抽甲乙方+合約金額」改成「抽供應商統編+品項+稅後金額+發票日期」，並換掉改名 template。
這個情境 → 第 6 章 walkthrough 從步驟 1 走到驗收
結業 2 週
把 #04-daily-ai-report 的輸出從 Google Sheet 換成 Notion
公司全面換到 Notion，原 #04 把摘要寫進 Google Sheet。這份手冊教你先在 n8n UI 拉一個 Notion 節點、隨便填、複製出來給 LLM 當範本，而不是讓 LLM 從記憶生成（typeVersion 容易錯）。
這個情境 → 第 3 章步驟 2「讀 JSON，找要換的節點」
結業 1 個月
把 #11-csv-clean-score 接在 Google Drive 觸發器上
同事每週丟到共享 Drive 的 CSV 想自動清洗評分，但 #11 是手動上傳觸發。把 trigger 換成 Google Drive Trigger 之後，下游節點引用的 input 路徑也跟著變了——這份手冊教你在步驟 3 先抓出有哪些下游引用要跟著改。
這個情境 → 第 3 章步驟 3「限定修改範圍」
結業 1 個月
把 #09-gmail-categorize 改成只看特定客戶來信
希望 #09 只處理 3 個 VIP 客戶域名的信。這其實只需要插入一個 IF 過濾，但直覺可能讓 LLM「重寫整個 workflow」。手冊的核心原則：最小修改——只插入一個 IF，不重寫全圖。
這個情境 → 第 3 章步驟 1「拆需求，只說要改什麼」
結業 3 個月
Lite Pack 跨大版本後重跑壞掉
半年後重新啟動 #06-webhook-gemini-file，發現 n8n 已升 1–2 個大版本，部分節點 typeVersion 過時，匯入時跳警告。手冊教你先用 n8n UI 的「open & save」自動升級，再把 LLM 當 second opinion，不要一開始就讓 LLM 改 typeVersion 數字。
這個情境 → 第 7 章錯誤分流「typeVersion mismatch」
i
5 個 story 不是要 100% 對上你的場景，是要對上你的「改造類型」：改 prompt / 改節點 / 改 trigger / 加 filter / 版本升級——這 5 種涵蓋 90% 改造需求。先選最接近的一種，再去對應章節找 walkthrough。

(05)

常見誤解

誤解 1：「我先去看第 5 章紅線就好，紅線知道了我自己摸索」
紅線是「不能踩的」，不是「該做的」。光看紅線只知道哪些坑，不知道正確路徑。建議至少先讀第 6 章 walkthrough 看一遍正確路徑長什麼樣，再去翻紅線。
誤解 2：「我的場景很特殊，5 個 timeline story 都不像我的情況」
5 個 story 不是要 100% 對上你的場景，是要對上你的「改造類型」。改 prompt / 改節點 / 改 trigger / 加 filter / 版本升級——這 5 種涵蓋 90% 改造需求。先選最接近的一種再對應章節。
