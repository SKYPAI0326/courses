步驟 1-3 真詳解 — 拆需求、讀 JSON、限定範圍

前 3 步是整套 SOP 的地基。步驟 1 寫不清楚，後面 LLM 會自己補；步驟 2 沒驗收，後面改錯沒人發現；步驟 3 不限範圍，LLM 會重寫整份。地基歪了，後面全白費。每步都有可直接複製的 prompt 模板與具體 LLM 回應範例。

讀完這頁你會能做到 · OUTCOMES
用 4 格便箋獨立拆解一個改造需求，填出含具體欄位名的 input/output，對著別人講 1 分鐘不打結
在不貼 credential 的前提下，把 workflow JSON 貼給 LLM、驗收節點清單 80% 正確率
下指令讓 LLM 只回傳節點區段，判斷回應是「區段」還是「整份」，在 -edit workflow 上完成替換
(01)

步驟 1：拆需求（10-15 分鐘，全人工）

為什麼這一步刻意排除 LLM
LLM 在需求模糊時會自己腦補。你說「幫我改 #02 處理發票」，LLM 不知道你的發票是什麼格式、想改名成什麼樣子、哪些節點一定要保留——它就開始猜，越改越亂。這一步的工作是你把需求說清楚，讓後面的 LLM 對話能有明確的邊界。

打開 4 格便箋範本（紙、Notes、便利貼皆可），依序填完以下 4 格：

  • 1
    填第 1 格「原 workflow 編號」
    動作：寫「#XX-XXX」+ 一行「目前在做什麼」
    • ✅「我要拿 #02 PDF AI rename 改成處理發票」— 直接寫 workflow 編號 + 業務目標一行
    • ✅「複製一份 #02 → 改 prompt 加分類欄位」— 說明改什麼也算
    • ❌「PDF 工作流」— 沒指名是 #02 還是 #06，不算對；請補 workflow 編號
  • 2
    填第 2 格「目標 input」
    動作：寫「文件類型 + 數量 + 來源資料夾 + 4-6 個關鍵欄位名」
    驗收硬指標：填好的 input 必須包含「具體欄位名」（例：「賣方統編」「稅後合計」），不能只寫「發票資訊」
  • 3
    填第 3 格「目標 output」
    動作：寫「成品檔案/紀錄的格式 + 命名規則 + 存放位置」
    驗收硬指標：output 規則要寫到「肉眼比對能驗證」的程度。例：{供應商名}_{發票日期}_{稅後金額}.pdf 完整可變數
  • 4
    填第 4 格「必保留的節點」
    動作：在 n8n UI 看一眼 workflow，把「不應該動」的節點名稱抄下來
    驗收：至少寫出 trigger 名 + 1 個其他節點，避免後續 LLM 重寫整份
  • 5
    自驗：對人講 1 分鐘
    動作：對另一個人/或對著手機錄音講一遍 4 格內容
    • ✅你能在 60 秒內，依序講出「原 workflow 編號 #02 → input 是供應商發票 PDF（200 張/月，4 個欄位：賣方/統編/稅後合計/發票日期）→ output 改名為 {YYYYMMDD}_發票_{賣方名}_{稅後合計}.pdf → 必保留 4 個 trigger/節點」 — 結構完整 + 可量化
    • ✅對方聽完不需要追問，能複述「你要把 200 張供應商發票改名」
    • ❌你講超過 90 秒還沒講完 4 格 → 表示便箋寫太散
    • ❌對方追問「你說的『發票資訊』是什麼意思？」「output 的格式怎麼長的？」→ 便箋第 2/3 格寫太抽象，回去補具體欄位名 / 命名格式

模板 1 的 LLM 回應長什麼樣？
好的 LLM 回應只指出空白，不自己補。例：

「你的需求有 2 個地方還沒講清楚：
1. 『賣方公司全名』格式要不要去除『股份有限公司』後綴？
2. 『稅後合計』如果是 0 元（退款單）要怎麼處理？

你只需要針對指出的空白補完，不需要重新填整份。
常見錯誤示範
直接問：「幫我改 #02，讓它可以處理發票。」
問題：沒有 input 格式（你的發票 PDF 長什麼樣？哪些欄位要抽取？）、沒有 output 形狀（改名規則是什麼？要移到哪個資料夾？）。LLM 會補上自己的假設，結果你拿到的 workflow 是它猜出來的版本，不是你要的。
(02)

步驟 2：讀 JSON（5-10 分鐘，學員 + LLM）

讓 LLM 做它最擅長的事
非工程師看 n8n workflow JSON 通常只看到一大堆 typeVersion、parameters、credentials——不知道哪個節點做什麼、資料往哪裡流。LLM 把 JSON 翻成人話，速度比你快 50 倍，而且能點出資料流向。這一步你只需要驗收它說得對不對。
  • 1
    n8n UI export workflow（不含 credential）
    動作：在 workflow 開啟畫面，點右上角三個點 → Download → 確認跳出視窗的選項是「Without credentials」（如果不是，切換）
    驗收硬指標：下載到的 .json 檔，用 VS Code 或記事本打開，按 Cmd+F 搜尋這 4 個字串：apiKey、accessToken、password、token，4 個全部找不到才算過
  • 2
    打開你慣用的網頁版 LLM（ChatGPT / Claude / Gemini）
  • 3
    貼模板 2 + JSON（見下方 SOP-A 取得整份 JSON 的步驟，再接模板 2）
  • 4
    驗收 LLM 回應
    動作：讀 LLM 回的「節點清單」，到 n8n UI 對照「節點數量是否一致 + 名稱對得上」
    驗收硬指標 (a) 節點數量正確：n8n UI 編輯器畫面數一遍節點圖示有幾個（5 個就 5 個），LLM 回的清單編號也要 5。差 1 個都不行。
    驗收硬指標 (b) 至少 80% 節點角色對得上（5 個裡至少 4 個算對）：
    • ✅LLM 說「Read PDF — 讀 /files/pdf-inbox/ 內所有 PDF，輸出每張的 binary」→ 你在 n8n UI 看到節點名是「Read PDF」+ parameter 的 path 是 /files/pdf-inbox/ → 完全對應 = 算對
    • ✅（容許小落差）LLM 說「Code 節點打 Gemini API」實際你看到 Code 節點裡用 this.helpers.httpRequest 呼叫 Gemini → 核心動作（呼叫 Gemini）對 + 細節有落差 = 算對
    • ❌LLM 說「Read PDF — 從 Google Drive 讀 PDF」實際你看到 path 是 /files/pdf-inbox/（本機路徑）→ 資料來源根本講錯 = 不算對，回頭追問：「Read PDF 節點的 path 寫的是 /files/pdf-inbox/，不是 Google Drive，請重講這個節點的角色」
    • ❌LLM 把「Code: AI 改名」說成「Code: 一般資料處理」→ 節點意圖完全失準 = 不算對，回頭追問

    量化方式：3 個（60%）以下表示 LLM 看 JSON 的能力對你的 workflow 不可靠，換另一家 LLM（或重貼模板 2 + 補一段 workflow 中文描述）

  2. 2.跳出對話框，勾選「Without credentials」（如果預設沒勾，自己勾起來）→ 確認下載
  3. 3.下載到的 .json 檔，用 VS Code 或記事本打開 → Cmd+A 全選 → Cmd+C 複製

  5. 5.token 量參考：典型 5-10 節點 workflow JSON 在 5K-30K tokens 之間，貼 ChatGPT/Claude/Gemini 的網頁版都吃得下；若超過 50K（>40 個節點），分段貼

(03)

步驟 3：限定修改範圍（10-15 分鐘，學員 + LLM 多輪）

最容易跳過、最值得做對的一步
大多數改 workflow 出問題，都是因為讓 LLM 改的範圍太大——它多動了幾個節點、加了你沒要求的防禦邏輯、重寫了整份 JSON 但 typeVersion 對不上你裝的版本。這一步的核心原則是：告訴 LLM 只改哪幾個節點，要它只回傳那幾個節點的 JSON 區段，不是整份。
  • 1
    決定「我要動哪幾個節點」（基於步驟 1 的便箋）
    動作：在 4 格便箋的「目標 output」對照節點清單，圈出 1-3 個要改的節點名
  • 2
    在 n8n UI 備份這幾個節點
    動作：點要改的節點 → Cmd+C → 貼到一個 .txt 檔（命名 XX-backup.txt）
    驗收：.txt 檔有節點 JSON 內容（看到 { "parameters": ...）
  • 3
    貼模板 3 給 LLM（見下方，含具體填法示範）
    注意：模板 3 的 [貼 JSON] 位置對應 SOP-B 取得的節點 JSON
  • 4
    驗收 LLM 回的 JSON
    驗收硬指標 1（區段 vs 整份）：看 LLM 回的第一個非空白字元和第二行：
    • ✅第一行 { 第二行 "parameters": → 單個節點區段（OK）
    • ✅第一行 [ → 多個節點陣列（OK）
    • ❌第一行 { 第二行 "name": → 整份（要追問）

    驗收硬指標 2（節點名沒被改）：LLM 回的節點裡 "name": 那行的值，必須跟你 n8n UI 上看到的節點名一字不差。LLM 偶爾會「順手」改名（例：把「Code」改成「Code: AI 改名」）— 這會讓步驟 5 的回報失準
  • 5
    如果是整份 JSON：用模板 3-追問要它重給（見下方）
!
硬性前置：步驟 3 所有替換動作只發生在 -edit workflow，不動原版

你要替換的 JSON 是 LLM 草稿，跑壞會壞節點 / 連線 / 甚至 n8n 編輯器顯示。在 production workflow 上直接貼，等於拿活的訂單系統當實驗田 — 一旦不小心又把 Active 切回 ON（n8n 介面手滑常事），就真的觸發 200 張 LLM API call 燒錢、或誤改 production 資料庫。

先 Duplicate 再替換 — 這是強制規定，不是建議。
SOP-B · Duplicate -edit workflow（替換前必做）
  1. 1.n8n UI 左側選單 → Workflows → 找到原 #02 → 滑鼠移到該行 → 右側出現 ⋮ 三點按鈕
  2. 2.點 ⋮ → 選「Duplicate」（不是 Open / Share / Delete，看清楚）
  3. 3.n8n 自動產生新檔，預設名稱長這樣：02-pdf-ai-rename Copy
  4. 4.點該新 workflow 的標題 → 進編輯器 → 點頂部標題列 → 改名為 02-pdf-ai-rename-edit（結尾加 -edit 後綴，這是全課統一規約，第 4 章測試時要靠它識別）
  5. 5.確認新 workflow 右上角 Active 開關是「灰色 OFF」（Duplicate 預設會關，但仍要肉眼確認一次）
  • ✅Workflows 列表現在有兩條：原版 02-pdf-ai-rename + 新版 02-pdf-ai-rename-edit
  • ✅你接下來貼 JSON 的瀏覽器分頁，網址列尾巴的 workflow ID 不是原版的 ID
  • ❌如果你發現自己正在原版 workflow 上要按 Paste，Cmd+Z 撤銷 → 退回此步重做
SOP-C · 替換到 -edit workflow 的 n8n UI
  1. 1.確認你在 -edit workflow 編輯器（看頂部標題列）
  2. 2.點要改的節點 → 右鍵 Paste → 貼上 LLM 給的 JSON
  3. 3.驗收：節點還在原位、連線沒斷、節點名相同
  4. 4.再驗一次：頂部標題列必須顯示 02-pdf-ai-rename-edit（不是原版名）

變體 A 過關 — 直接 Paste 到 -edit 
{

第一行 {，第二行 "parameters":，節點名沒被改。直接複製 → Paste 到 -edit workflow 對應節點。

核心原則 — 只回傳節點區段，不重寫整份
n8n workflow JSON 裡，節點之間靠 id 與 connections 區段串聯。LLM 在整份重寫時極容易動到 ID 或漏掉 connection，你匯入後會發現節點全斷線。

要求 LLM「只回傳節點區段」+ 你在 n8n UI 上做替換，就是為了繞開這個坑。整個 connections 結構保持不動，只換節點本身的設定。

若你確實需要新增或刪除節點（不是修改既有節點），connections 必須回到 n8n UI 上手動重接——UI 拖線 30 秒，LLM 寫錯一個 ID 你 debug 30 分鐘。

  • 4 格便箋每格都填了具體值（不是「發票」這種模糊詞）
  • 自驗對人講 1 分鐘沒有打結（或對手機錄音回放確認）

(05)

常見錯誤 + 怎麼解

錯誤 1 · 步驟 1
「填 4 格時，input 那格我寫『發票』就好」
「發票」不夠。LLM 看到「發票」會自己腦補欄位（你以為發票上有「品項」但實際上你的發票沒有品項，只有總額）。最低限度要寫出 4-6 個欄位名 + 該欄位在 PDF 上的中文寫法（「賣方」不是「供應商」）。
錯誤 2 · 步驟 2
「我貼了模板 2，LLM 卻直接開始建議改進」
模板 2 最後一行是「不要建議改進、不要寫程式碼」。如果 LLM 還是建議，追問：「你建議的改進我不需要，請只回答 1 和 2 兩題（描述 + 節點清單）」。
錯誤 3 · 步驟 3
「LLM 給的 JSON 看起來怪怪的，我不確定是區段還是整份」
用驗收硬指標 1 的判斷法（看第二行）。如果還不確定，把 LLM 回應前 5 行貼回去問：「我給你看你回應的前 5 行，這是整份還是單節點區段？」
錯誤 4 · 步驟 3
「LLM 把節點改名了，我不知道」
步驟 3 替換到 n8n UI 後，用 Cmd+F 搜尋你便箋上記的「必保留節點名」——如果搜不到，代表 LLM 改名了，要回頭追問。
