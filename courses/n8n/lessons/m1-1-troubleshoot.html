(01)

分層診斷思維

為什麼這很重要
同一個症狀（localhost:5678 連不上）可能對應不同原因：容器沒起、埠被佔、防火牆擋。先收集症狀，再下結論——這是本單元唯一要建立的思維。

從最外層往內查：使用者瀏覽器 → 網路 → 容器 → Docker Engine → 作業系統。每層都有自己的診斷指令，逐層排除。不要看到錯誤就直接跳到最深層。

!
三行診斷指令——卡住時先跑這三個，把輸出存下來，再決定是自查還是貼給講師。多數情況看 logs 第一行就找得到答案。 
# 看容器是否在跑
docker ps
# 看 n8n 最近的錯誤日誌
docker compose logs n8n --tail 50
# 驗證 compose 設定是否解析正確
docker compose config

8 案詳細排查

案 1 Docker Desktop 沒開 / 啟動中卡住
症狀
跑 start.command / start.bat 跳出對話框「Docker Desktop 未啟動」；或鯨魚圖示顯示 Starting 一直不變綠燈
診斷
看選單列 / 通知列鯨魚圖示狀態：Running（綠）/ Starting（黃）/ Stopped（紅）
點開 Docker Desktop 主視窗看 Dashboard 是否載入
終端機跑 docker info，應看到一大段系統資訊（不是錯誤）
修復
紅燈 → 手動點 Docker Desktop 啟動
黃燈卡 5 分鐘以上 → Docker Desktop 選單「Restart」
還是卡 → Quit Docker Desktop → 重開
仍不行 → Mac 重開機 / Win 重啟並確認 WSL2 是 v2
案 2 5678 埠被佔用
症狀
start 腳本跳錯 Bind for 0.0.0.0:5678 failed: port is already allocated；或 localhost:5678 開出來不是 n8n 而是別的服務
診斷
Mac / Linux： 
lsof -i :5678
Win PowerShell： 
netstat -ano | findstr :5678
會列出佔用 5678 的程式 PID。
修復
解法 A：關掉佔用程式
Mac：kill -9 PID（PID 換成上面查到的數字）
Win：工作管理員 → 詳細資料 → 找到 PID 對應程式右鍵結束工作

解法 B：改 n8n 用其他埠（推薦，不影響其他程式）
編輯 n8n-compose.yml，把 '5678:5678' 改成 '5680:5678'（冒號前面是電腦對外的埠，後面 5678 是容器內 n8n 的埠，不要動）。改完雙擊 stop 再 start，之後改用 http://localhost:5680 連線。
案 3 Apple Silicon platform mismatch 警告
症狀
Apple Silicon Mac 跑容器看到黃色警告 The requested image's platform (linux/amd64) does not match the detected host platform (linux/arm64/v8)
診斷
跑 docker info 看 Architecture 欄位，Apple Silicon 應顯示 aarch64。警告的 image 是 x86 架構，沒有原生 ARM 版本。
修復
本課程的 n8nio/n8n 與 postgres:16 都有原生 ARM 版本，不會出現此警告。
若你跑其他 image 看到此警告，影響不大（Rosetta 自動翻譯，效能稍降約 30%）。
想抑制警告 → 在 compose 服務下加 platform: linux/amd64 明示用 x86，或找原生 ARM 版本的 image。
案 4 Win WSL2 未啟用 / 版本太舊
症狀
Docker Desktop 啟動後一直 Starting 不變綠燈；或安裝時跳錯「WSL2 installation is incomplete」
診斷
PowerShell（系統管理員）跑： 
wsl --status
預期看到「預設版本: 2」。看到「預設版本: 1」或找不到指令就要修。
修復
步驟 1：強制升級 WSL 核心 
wsl --update
步驟 2：設定預設版本為 2 
wsl --set-default-version 2
步驟 3：沒裝過 WSL 時 
wsl --install
完成會要求重啟。

步驟 4：跳「需要啟用虛擬化」→ 工作管理員 → 效能 → CPU 看「虛擬化」是否「已啟用」。沒啟用要進 BIOS 開（重開機按 F2/Del/Esc，找 Intel VT-x / AMD-V / SVM Mode 啟用 → 存檔重啟）
案 5 容器在跑但 localhost:5678 連不上
症狀
docker ps 看到 n8n 容器 status 是 Up，但瀏覽器開 localhost:5678 顯示 ERR_CONNECTION_REFUSED 或一直轉
診斷
# 看是否有 startup error
docker compose logs n8n --tail 50
# 確認兩容器狀態
docker compose ps
確認 postgres 與 n8n 兩個容器都是 Up (healthy)，不是 Restarting 或 Up (unhealthy)。
修復
postgres 顯示 Restarting：通常是密碼設定衝突。停服務 → 刪 volume docker compose down -v（會清資料，請確認）→ 重新 start.command，會用 .env 的密碼重建

n8n 日誌出現 getaddrinfo EAI_AGAIN postgres：postgres 還沒就緒，等 30 秒重整 docker compose restart n8n

兩容器都正常但網頁仍連不上：檢查防火牆是否擋 5678（macOS 系統設定 → 網路 → 防火牆 / Win 防火牆設定）
案 6 macOS start.command 被 Gatekeeper 擋
症狀
雙擊 start.command 跳出「無法打開"start.command"，因為它來自未識別的開發者」
診斷
這是 macOS Gatekeeper 第一次擋未簽署的執行檔，不是檔案壞了。
修復
步驟 1：開「系統設定 → 隱私權與安全性」
步驟 2：捲到「安全性」區塊
步驟 3：看到「start.command 已被阻擋使用」
步驟 4：按「強制打開」→ 輸入密碼確認
步驟 5：再次雙擊 start.command 就會跑

stop.command / update.command 第一次也會被擋，做同樣處理即可。之後同檔不會再擋。
案 7 本機資料夾掛載權限錯誤（macOS）
症狀
Module 3 跑 Watch Folder 觸發失敗；或 n8n 日誌出現 EACCES: permission denied 對 /files/shared 的存取
診斷
在 n8n-starter-kit/shared/ 放一個測試檔，看 n8n 容器能不能讀。
macOS 可能在背景擋住 Docker Desktop 對某些資料夾的存取（Files & Folders 權限）。
修復
步驟 1：開「系統設定 → 隱私權與安全性 → 檔案與資料夾」
步驟 2：找「Docker Desktop」（沒看到的話往下找「完整磁碟存取權」）
步驟 3：確認你的 n8n-starter-kit 所在資料夾的權限有勾選
步驟 4：不確定就把整個「桌面」「文件」「下載」都勾起來
步驟 5：重啟 Docker Desktop

如果問題在 Docker Desktop 設定本身，去 Docker Desktop → Settings → Resources → File sharing 確認 n8n-starter-kit 的父目錄在白名單
案 8 升級後 workflow 不見
症狀
跑 update.command 之後重新登入 n8n，發現之前建的流程全消失，回到「Create your first workflow」狀態
診斷
跑 docker volume ls 確認 n8n-starter-kit_n8n_data 還在。
還在但不見：可能不小心建了新的 Owner Account，n8n 把你導向新帳號的空間。
不在了：之前用過 docker compose down -v（-v 會刪 volumes）。
修復
volume 還在 → 確認登入的 Owner email 跟之前一樣（n8n 1 個 instance 1 個 owner）
volume 不在了 → 救不回來；以後只用 stop.command（背後是 docker compose down，不刪 volume），不要手動跑 down -v

預防：升級用 update.command（拉 image + 重啟，不碰 volume）；停服務用 stop.command（不刪資料）；想全部清掉重來才手動跑 docker compose down -v
(04)

動手練習 — 故意製造埠衝突再修復

情境
阿凱要在自己的電腦上重現案 2 的埠衝突，練習用解法 B（改埠號）完成修復，而不是只看過一次就算。

依以下 8 個步驟操作，完成後把 compose 改回原狀。

1
確認 n8n 已停止
雙擊 stop.command / stop.bat
2
開另一個終端機，佔用 5678
python3 -m http.server 5678
這會在 5678 起一個假伺服器佔用埠
3
雙擊 start.command / start.bat
這時候應該會跳錯
4
確認錯誤訊息
應看到 port is already allocated——拍下截圖備用
5
依案 2 解法 B 修：改 compose 埠號
把 n8n-compose.yml 裡的 '5678:5678' 改成 '5680:5678'
6
雙擊 start，應該成功啟動
等候 30 秒讓容器初始化完成
7
開 http://localhost:5680 確認
應看到 n8n 主介面——拍下截圖
8
收尾：關掉假伺服器，改回原埠
在假伺服器的終端機按 Ctrl+C，把 compose 改回 '5678:5678'
!
預期成果：練習過程中拍下「跳錯訊息」「修改後的 compose 檔」「成功跑起來的 localhost:5680」三張截圖，並寫下「以後遇到埠衝突，我會先做 _____，再 _____」的兩步流程。

(06)

常見錯誤 3 條

錯誤 1
現象：看到錯誤訊息就直接舉手問講師，沒先跑三行診斷
原因：覺得錯誤訊息太長看不懂，習慣性求助
解法：卡住時先跑三行存下輸出，再決定是自查還是問人。多數情況看 logs 第一行就找得到答案
錯誤 2
現象：為了「修錯」直接跑 docker compose down -v，結果把流程全清掉
原因：沒注意 -v 旗標的意義，只想著「重來一次」
解法：案 8 已強調；down 只停容器，down -v 連 volumes 一起刪
錯誤 3
現象：Win 學員跑 wsl --status 顯示一切正常，但 Docker Desktop 還是卡 Starting
原因：WSL 核心版本太舊，--status 沒抓到問題
解法：直接跑 wsl --update 強制升級，多數情況解決；仍不行可能是企業電腦群組原則擋住，需找 IT
(07)

檢核題

Q1（概念驗證）「分層診斷思維」中，看到 ERR_CONNECTION_REFUSED 應該先檢查哪一層？
A. 防火牆設定
B. n8n 設定檔內容
C. Docker 容器是否在跑（用 docker ps）
D. 重新安裝 Docker Desktop
查看解析
從外層症狀切入最內層執行狀態，先確認容器在不在跑。確認容器狀態是成本最低、資訊量最大的第一步，再依結果往其他層追查。
Q2（應用驗證）雯姊跑 update.command 後發現流程全部消失。可能原因是什麼？下次該怎麼做？
A. update.command 本身有 bug，不應使用
B. n8n 版本升級後資料格式不相容，自動清空
C. 之前曾跑過 docker compose down -v，-v 旗標會刪 volumes 連同資料
D. Docker Desktop 版本太舊需升級
查看解析
可能原因：之前曾跑過 docker compose down -v（-v 旗標會刪 volumes 連同資料）。下次預防：只用 stop.command（不加 -v）停服務；想清掉重來才手動跑 down -v。一旦 volumes 被刪資料無法救回；建議重要流程用 n8n 內建 Export 功能定期備份成 JSON。
(08)

三行診斷指令

遇到任何卡關，先跑這三行收集現場資訊。多數情況看 logs 第一行就能找到答案；真的需要請人協助時，把這三行的輸出貼出來，比文字描述快十倍。

!
快速複製版——把游標移到下方程式碼右上角按「複製」，貼到終端機 / PowerShell 直接跑。 
# 看容器是否在跑
docker ps
# 看 n8n 最近的錯誤日誌
docker compose logs n8n --tail 50
# 驗證 compose 設定是否解析正確
