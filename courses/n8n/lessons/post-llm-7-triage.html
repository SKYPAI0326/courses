9 頁系列 · 第 7/9 · 錯誤排查

LLM 給的 workflow 匯不進去？— 3 類錯誤分流 SOP

匯入失敗先別慌，也別重新請 LLM 整份重寫。先用本頁決策樹定位錯誤類別，再照對應 checklist 逐步排查。

本頁學習成果
能看錯誤訊息關鍵字，在 30 秒內判斷屬於哪類分流
能依類別 checklist 完成排查，不走回頭路重貼整份 workflow
知道卡超過 30 分鐘時該做什麼，而不是繼續和 LLM 死磕
(01)

開始前：30 秒環境檢查

進分流前先確認三項
  • 我在 *-edit workflow（不是 production 那份）
  • Active 是 OFF（手動跑，不是 trigger 自動跑）
  • 在 local n8n（如果 LLM 給的含 executeCommand / readWriteFile 等節點）
三項全綠才進下面分流。環境互鎖詳細說明 → 第 8 章：執行環境互鎖

如果環境本身就錯（在 production 動 / Active 開著被 trigger / 在 Cloud 但用 Cloud 不支援節點），錯誤分流會帶你走錯方向，修也修不對。

分流使用方式
找到錯誤訊息 → 複製全文 → 對照下方決策樹 30 秒定位類別 → 跳對應 checklist。每次排查只走一條路，不要同時跑多條。
匯入失敗 / 跑不通 / 結果不對
    │
    ├─ 訊息含 "Could not parse" / "Invalid JSON" / "Unexpected token"
    │  或 "node type not found" / "unknown parameter" / "connections"   → 類別 1：JSON / workflow 格式錯
    │
    ├─ 訊息含 "Credential" / "Credential not found" / "401 Unauthorized"
    │  或 "403 Forbidden" / "invalid_grant" / "invalid_client"
    │  或 "token expired" / "API key" / "OAuth"                         → 類別 2：credential 錯
    │
    └─ 匯入成功但 output 不對 / 紅燈在某節點中段
       或 全綠燈但「資料量縮水 / 寄錯人 / filter 條件錯」               → 類別 3：邏輯錯
△
看不到明確訊息，或訊息是「執行失敗」這種模糊說法？先點開紅燈節點的輸出面板，找到最後一條展開的錯誤文字，再回來對照。
實務比例參考：100 個錯誤大約是類別 1 占 30%、類別 2 占 20%、類別 3 占 50%。遇到「看起來什麼都對就是結果不對」幾乎都是類別 3，不要硬往 1 或 2 找。
(02)

類別 1：JSON / workflow 格式錯

類別 1 JSON / workflow 格式錯（n8n 進不去）
辨識訊號 — 30 秒判斷
n8n 直接拒絕匯入，跳「無效的 workflow」之類視窗；或匯入成功但跳「節點類型不存在」紅字。典型訊息：Error parsing workflow: Unexpected token < in JSON at position 0 / Could not import workflow: Invalid format / node type not found: n8n-nodes-base.googleSheets / unknown parameter "options" for node "Code"
△
類別 1 vs 第 8 章 Cloud 限制：30 秒分辨。unknown parameter 和 node type not found 容易和「Cloud 根本不支援此節點」混淆，動修法前先看下表。
訊號 類別 1（版本 / 欄位差） 第 8 章 Cloud 限制（節點根本不可用）
錯誤訊息 unknown parameter X / parameter Y is required Node type 'n8n-nodes-base.executeCommand' is not available
n8n UI 反應 節點顯示，設定面板有黃色驚嘆號 節點變灰且不能 add，搜尋找不到
解法方向 改 typeVersion / 補欄位 → 走類別 1 動線 走第 8 章「local 才能跑」評估 → 換節點或留 local
5 步 CHECKLIST（依序跑，前一步沒解才跑下一步）
  • 檢查是不是「整份覆蓋」而非「在 UI 替換節點」。你貼到 n8n 的內容，是直接覆蓋 workflow.json 檔，還是用 n8n UI 的「右鍵 Paste 節點」貼？LLM 給的通常是區段 JSON，不是整份 workflow。
    復原：回到 n8n 編輯畫面空白處右鍵 → Paste 節點，不要直接覆蓋整份檔案。
  • 檢查 markdown code fence。你從 LLM 複製 JSON 時，開頭有沒有 ```json 三個反引號？
    復原：把開頭的 ```json 和結尾的 ``` 拿掉，只保留 { 開頭的純 JSON 內容再試一次。
  • 檢查 typeVersion 是不是 LLM 自己編的。（僅在你動了整份 JSON 時才需要。UI 上區段替換、沒動 connections 的話直接跳過。）
    動作：打開你自己之前 export 的同一份 #02 .json 檔（不是「同類」workflow，是這份本身），用 VS Code 搜尋同名 node type（例："type": "n8n-nodes-base.code"），抄它的 typeVersion 數字回填 LLM 給你的 JSON。
  • 用 jsonlint.com 找壞行。複製 LLM 給的 JSON 整段 → 貼到 jsonlint.com → 點 Validate。
    復原：jsonlint 會標哪一行有逗號 / 引號錯，把那行貼給 LLM 用「修單行，不重寫整份」要求修。
  • 節點 deprecation（訊息含 node type not found 時）。依序：google 節點名稱 + n8n → n8n UI 搜尋現有節點 → 課程社群貼截圖 → 不直接信 LLM 給的節點名稱。

如果你動了整份 JSON 或新增/刪除節點，可用下面 prompt 讓 LLM 輸出 connections 核對表：

PROMPT 模板（connections 核對表）

| 來源節點名稱 | 目標節點名稱 | 來源節點是否在 nodes 陣列 | 目標節點是否在 nodes 陣列 |

不要改 JSON、不要建議修法。只輸出表格。

[貼 JSON]

| 來源 | 目標 | 來源在 nodes? | 目標在 nodes? |
|---|---|---|---|
| Manual Trigger | Read PDF | ✓ | ✓ |
| Read PDF | Extract PDF Text | ✓ | ✓ |

5 個節點全顯示且不變灰，重新匯入不再跳錯誤視窗。
(03)

類別 2：credential 錯

類別 2 credential 錯（認證掉了）
辨識訊號 — 30 秒判斷
匯入成功但執行紅燈，訊息含 Credential not found / 401 Unauthorized — Invalid or expired token / 403 Forbidden / invalid_grant: Token has been expired or revoked。節點的 credential 欄位顯示空白或失效。
5 步 CHECKLIST（依序跑）
  • 緊急優先：你是不是不小心把含 credential 的 JSON 貼出去過？先做這條。
    Google credential：myaccount.google.com/permissions → 找 n8n 那個 entry → Remove
    Notion credential：notion.so/profile/integrations → 找你給 n8n 的 integration → Regenerate token
    OpenAI credential：platform.openai.com/api-keys → 找該 key → Revoke
    其他服務：Google「服務名 + revoke API key」找官方撤銷頁
    撤銷後重發新 token → 回 n8n credentials 設定頁更新。撤銷舊 token 會讓所有用同一 credential 的 workflow 全部失效，這是預期行為，正因如此第 5 章紅線 1 嚴格禁止 credential 外洩。
  • 在 n8n UI 找到 Credential 欄重新選。點該節點 → 右側 panel 找 Credential tab（在 Settings tab 旁邊，或同一 tab 內向下捲） → 點下拉選你之前建好的 credential。
    驗收：欄位顯示綠色勾勾或 credential 名稱（不是空白）。
  • 每個有問題的節點都重選一次（即使顯示是對的）。點下拉 → 選同一個 → Save。讓 n8n 重新 bind credential reference。不要試圖在 JSON 裡找 credential 對應位置直接改文字。
  • self-host 額外：OAuth callback URL 是否對齊。self-host 要在每個 OAuth credential 設定裡填 callback URL（例：https://your-domain.com/rest/oauth2-credential/callback）。去 n8n credential 設定 + OAuth 服務 console，確認兩邊一致。
  • Cloud 用戶：self-host 加密的 credential 不能直接搬。Cloud 上要重新建 credential，不能匯入 self-host 加密過的。
節點 credential 欄顯示綠色勾勾，重跑該節點不再出現 401 / 403。
追問 PROMPT 模板
我的 n8n workflow 跑出 [貼錯誤訊息]。我已確認 credential 在 n8n UI 上欄位有顯示，但跑起來還是 401。我的環境是 [self-host / Cloud]，OAuth callback URL 設定是 [貼設定值]。請問還有哪個步驟可能漏了？不要叫我重建整個 workflow。
(04)

類別 3：邏輯錯

類別 3 邏輯錯（最難，最常見）
辨識訊號 — 30 秒判斷
匯入成功、credential 也對，但跑出來的結果不是要的。沒有認證類錯誤，可能長三種樣子：
樣態 1
紅燈卡在中段節點。訊息常見 Cannot read property 'X' of undefined / Item is empty / [Object object] is not iterable。表示某節點試圖讀取一個不存在的欄位。
樣態 2
全綠燈但資料量縮水。200 筆 input 變 50 筆 output — filter 條件錯 / 迴圈邏輯錯 / 上游 split 錯。資料沒有報錯，只是被靜默地過濾掉了。
怎麼發現：在每個節點右側 panel 看「Items」數量（1 of 200 類似顯示），找到哪個節點數字驟降，那個節點的條件就是問題所在。
樣態 3
全綠燈但寄錯 / 存錯。寄信寄到錯收件人 / 檔案存到錯資料夾 / 欄位對應顛倒。需要肉眼對著真實 output（收件人是誰、檔存在哪）比對便箋上寫的目標，哪個欄位的值錯就是哪個欄位的 expression 錯。
5 步 CHECKLIST（依序跑）
  • 用 output panel 比對 dot path。點紅燈節點 → 右側 panel 切到「Input」tab → 看 input 進來時的 JSON 結構。
    怎麼讀：找你以為 LLM 引用的欄位（例：customer.email）。在 input panel 確認是不是真的有 customer 這個 key，且底下有 email。如果 input 裡實際是 contact.mail（不是 customer.email），就是 LLM 假設的欄位路徑和真實資料不符。
  • 用「optional chaining」讓節點暫時不炸紅燈。把 expression {{ $json.customer.email }} 改成 {{ $json.customer?.email }}（在 customer 後面加一個小問號）。
    翻譯：問號的意思是「如果 customer 這欄不存在，整個 expression 就回 undefined，節點走過不炸紅燈」。在 n8n 任何欄位的 expression（不管是 Set 節點還是 Code 節點）都可以加。
    注意：optional chaining 是止血，不是根治
    加了 ?. 讓 workflow 不再炸紅燈，但輸出可能是錯的或缺的（例如 aiResponse?.category 在 LLM 沒回 category 時變 undefined，下游 Sheet 寫入會出現空欄）。跑完不炸不等於跑對。三步根因動線：
    1看 input panel — 是什麼讓欄位 missing？（LLM 回應結構變、prompt 改了沒同步、PDF 解析空白）
    2回第 3 章模板 5 追問 LLM — 「我的 prompt 對 X 種 input 應回 Y 結構，但實際回了 Z（貼 sample），請問是 prompt 缺什麼還是 input 本身就無法生 Y？」
    3決定補 fallback 還是改 prompt — 如果某類 input 真的拿不到 Y（例如圖片化 PDF 抽不到文字），就補 || '未分類' 之類 fallback；如果只是 prompt 沒寫清楚就回去改 prompt。
  • 檢查上游節點是不是換了。你最近換過 trigger（例：Manual → Google Drive Trigger）？換 trigger 後所有下游節點引用的欄位路徑都要重接。在 n8n UI 上每個下游節點的 expression 欄位 hover 上去，確認「左側是哪個 trigger 出來的欄位」 → 重新從新 trigger 的 output 拖。
  • 用模板 4 精準回報（不貼整份）。只貼「紅燈節點 + 上一節點 output + 紅燈訊息」三件套給 LLM，不要貼整份 workflow。
  • LLM 修法看起來要動 5 個以上欄位 → 退回第 1 步。回頭看 4 格便箋，是不是 input 那格寫太模糊，需求需要重拆。
重跑紅燈節點不再報錯，且 output 欄位值和便箋上寫的目標一致。

(05)

卡了 30 分鐘還沒解——逃生口

排查是有邊界的。同一個問題已經跑了 3 輪 checklist，或者和 LLM 來回超過 5 次還是繞圈子，那不是你的問題，也不是 LLM 的問題——是當前資訊量不足以在對話裡收斂。

停損規則
3 輪 checklist + LLM 來回超過 5 次沒解 = 停。不要繼續和 LLM 死磕。每多一輪對話，你提供的上下文越雜，LLM 越容易往錯的方向猜。停下來比繼續試成功率更高。

逃生動線：把下面求救範本填好，貼到課程社群或寄到講師信箱。這份整理對「明天再看的你」也有用——很多問題在睡一覺之後重新看截圖會立刻看出來。
