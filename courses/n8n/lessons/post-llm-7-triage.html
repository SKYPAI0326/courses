8 頁系列 · 第 7/8 · 錯誤排查

LLM 給的 workflow 匯不進去？— 3 類錯誤分流 SOP

匯入失敗先別慌，也別重新請 LLM 整份重寫。先用本頁決策樹定位錯誤類別，再照對應 checklist 逐步排查。

本頁學習成果
能看錯誤訊息關鍵字，在 30 秒內判斷屬於哪類分流
能依類別 checklist 完成排查，不走回頭路重貼整份 workflow
知道卡超過 30 分鐘時該做什麼，而不是繼續和 LLM 死磕
(01)

分流決策樹

使用方式
從「匯入失敗 / 跑不通 / 結果不對」這個起點往下看，對照你看到的錯誤訊息關鍵字，找到對應類別，跳到該類別的 checklist。每次排查只走一條路，不要同時跑多條。
匯入失敗 / 跑不通 / 結果不對
    │
    ├─ 訊息含 "Could not parse" / "Invalid JSON" / "Unexpected token"
    │  或 "node type not found" / "unknown parameter" / "connections"   → 類別 1：JSON / workflow 格式錯
    │
    ├─ 訊息含 "Credential" / "Credential not found" / "401 Unauthorized"
    │  或 "403 Forbidden" / "invalid_grant" / "invalid_client"
    │  或 "token expired" / "API key" / "OAuth"                         → 類別 2：credential 錯
    │
    └─ 匯入成功但 output 不對 / 紅燈在某節點中段
       或 全綠燈但「資料量縮水 / 寄錯人 / filter 條件錯」               → 類別 3：邏輯錯
△
看不到明確訊息，或訊息是「執行失敗」這種模糊說法？先點開紅燈節點的輸出面板，找到最後一條展開的錯誤文字，再回來對照。
(02)

類別 1：JSON / workflow 格式錯

類別 1 JSON / workflow 格式錯（n8n 進不去）
典型特徵
n8n 直接拒絕匯入，跳「無效的 workflow」之類視窗，或匯入成功但跳「節點類型不存在」紅字。
  • 你是不是「整份覆蓋」而非「在 UI 替換節點」？LLM 給的是區段 JSON，不是整份 workflow。回去走步驟 3，在 n8n UI 上逐節點替換，不要直接覆蓋整份檔案。
  • LLM 給的 JSON 開頭是 { 還是 ```json？把 markdown code fence 拿掉，只保留 { 開頭的純 JSON 內容再試一次。
  • 此項僅在你改了整份 JSON 或新增／刪除節點時才需檢查。UI 上區段替換沒動 connections 的話直接跳過，回去檢查節點欄位本身。若確實需要檢查：請 LLM 輸出表格「每個 connection 的來源節點名稱 / 目標節點名稱 / 是否存在於 nodes 陣列」，學員一眼掃完。
  • typeVersion 是不是 LLM 自己編的？怎麼查：打開 Lite Pack zip 裡的 workflows/<同類 workflow>.json，用編輯器搜尋同名 node type（例如 "type": "n8n-nodes-base.googleDrive"），抄它的 typeVersion 數字回填到你的 JSON。
  • 用 jsonlint.com 或 VS Code 開檔看哪一行壞了，把那行貼給 LLM 修，不要重寫整份。
(03)

類別 2：credential 錯

類別 2 credential 錯（認證掉了）
典型特徵
匯入成功但執行紅燈，訊息含 401 / Credential not found。節點的 credential 欄位顯示空白或失效。
  • 你是不是不小心把含 credential 的 JSON 貼出去過？若是，三步動線順序不能跳：
    1. 立刻去 Google / Notion / OpenAI console 撤銷舊 token
    2. 重發新 token
    3. 回 n8n UI 上點該節點 → credential 欄 → 重新選新建的 credential
    這比修 workflow 急，先做這步。
  • LLM 改完的節點，credential reference 是不是空了？在 n8n UI 上點該節點，credential 欄會顯示「未選擇」，重新指定即可。
  • 不要試圖在 JSON 裡用文字找 credential 對應位置。正確做法：匯入後直接回 n8n UI 上每個有問題的節點，credential 欄重新選一次（即使顯示是對的也選一次），讓 n8n 重新綁 credential reference。
  • 是 self-host 還是 Cloud？某些 credential 類型在 self-host 需要額外配 OAuth callback URL，Cloud 版不需要，確認你的環境。
(04)

類別 3：邏輯錯

類別 3 邏輯錯（最難，最常見）
典型特徵
匯入成功、credential 也對，但跑出來的結果不是要的。可能長三種樣子：
樣態 1
紅燈卡在中段節點。訊息常見 Cannot read property 'X' of undefined 或 Item is empty。表示某節點試圖讀取一個不存在的欄位。
樣態 2
全綠燈但資料量縮水。例如預期 200 筆變 50 筆——多半是 filter 條件錯或迴圈邏輯錯。資料沒有報錯，只是被靜默地過濾掉了。
樣態 3
全綠燈但寄錯／存錯。寄信寄到錯的收件人、檔案存到錯的資料夾、欄位對應顛倒——肉眼比對 input／output 才看得出來。
  • 在紅燈節點的上一個節點，看 output panel 的 JSON 結構。是不是和 LLM 預期的資料形狀不同？LLM 假設欄位叫 customer.email，實際可能是 contact.mail。
  • 引用的欄位路徑對不對？在 n8n expression 裡用 ?. 避免欄位不存在時整個 expression 炸掉。範例：{{ $json.customer?.email }} 取代 {{ $json.customer.email }}——前者沒有 customer 欄位會回 undefined（節點走過），後者直接紅燈。
  • 是不是 Trigger 換了之後，下游引用沒跟著改？換 trigger 從 Manual 改成 Google Drive Trigger，下游所有 {{ $json.file }} 引用都要更新，這是最常見的漏改。
  • 用步驟 5 的「錯誤回報模板」貼給 LLM，只貼紅燈那節點 + 上一節點 + 紅燈訊息，不要貼整份 workflow。
  • LLM 的修法看起來要動 5 個以上欄位？回頭看步驟 1，需求是不是模糊了，需要重新拆。
(05)

卡了 30 分鐘還沒解——逃生口

排查是有邊界的。如果你在同一個問題上已經跑了三輪 checklist，或者和 LLM 來回超過 5 次還是繞圈子，那不是你的問題，也不是 LLM 的問題——是當前資訊量不足以在對話裡收斂。

不建議繼續做的事
深夜 1 點繼續和 LLM 死磕。每多一輪對話，你提供的上下文越雜，LLM 越容易往錯的方向猜。停下來比繼續試成功率更高。

正確的逃生動線：把當前 workflow 截圖 + 你已經跑過的 checklist 項目 + 完整錯誤訊息，一次整理好，貼到課程社群或寄到講師信箱。這份整理對於「明天再看」的你自己也有用——很多問題在睡一覺之後重新看截圖會立刻看出來。

△
把「我試過 checklist 哪幾條、結果是什麼」一起說出來，比只貼錯誤截圖省掉 3-5 輪來回確認。講師和社群回應的速度取決於你提供的資訊密度。
