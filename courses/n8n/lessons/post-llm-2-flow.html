6 步流程總覽 — 安全改 workflow 的骨架

8 頁系列 · 第 2/8 · 補充 · POST-COURSE

讀完這頁你會知道 · OUTCOMES
整套六步流程的完整骨架，以及每步「LLM 做什麼 / 你做什麼 / 驗收訊號」的三欄分工
流程的內在邏輯：為什麼這六步順序不能調換，哪幾步之間存在迴圈
哪一頁深入講哪幾步，讓你知道下一步從哪裡跳讀
(01)

為什麼是這 6 步

設計原則
LLM 給的東西看起來合理，但容易跑出細節錯。這六步的設計目的不是「讓 LLM 幫你做更多」，而是「在控制風險的前提下，精準榨取 LLM 的翻譯與局部改寫能力」。

改一個 n8n workflow 最常犯的錯是把所有事情都丟給 LLM：「幫我把 #02 改成處理發票。」LLM 拿不到足夠的脈絡，會自己腦補 input/output 形狀，回傳一份看起來能用但細節全錯的 JSON。結果不是花時間 debug，就是直接放棄。

六步流程的設計邏輯是：讓學員主導邊界、讓 LLM 執行翻譯和局部改寫、讓測試驗收結果。每一步都有明確的角色分工，避免學員在錯誤的步驟把決定權交給 LLM，也避免在正確的步驟重複做 LLM 已經能做的事。

(02)

6 步流程主圖

讀圖方式
步驟 1 到 4 是線性推進。步驟 5 出現錯誤時，只迴圈回步驟 3（不退回步驟 1）。超過三輪無法收斂，才需要退回步驟 1 重新拆需求。步驟 6 是收斂與驗收，不是「再改一輪」的起點。
01
拆需求（4 格便箋）
這一步刻意不用 LLM。需求模糊時 LLM 會自己腦補，越改越亂。
角色動作
LLM 做什麼不做。這步完全由你主導。
你做什麼填一張 4 格便箋：原 workflow 編號、目標 input、目標 output、必保留的節點。
驗收訊號4 格能對著別人講 1 分鐘還不打結。
02
讀 JSON（LLM 翻譯成中文流程）
把 workflow JSON 貼給 LLM，請它用人話描述這份 workflow 做了什麼。移除 credential 是硬規則。
角色動作
LLM 做什麼把 JSON 翻成一段人話描述，列出每個節點的角色與資料流向。
你做什麼用 n8n UI 對照 LLM 的描述，確認它沒有講錯哪個節點的角色。
驗收訊號你能用自己的話複述一遍 LLM 的描述，不用看 JSON。
03
限定修改範圍（只回節點區段）
告訴 LLM「只改節點 X、Y、Z」，要求它只回傳這幾個節點的 JSON 區段，不是整份。
角色動作
LLM 做什麼產出指定節點的 JSON 區段（不是整份 workflow）。
你做什麼先在 n8n UI 選起要改的節點、複製備份。LLM 改完後，在 UI 上替換這幾個節點。
驗收訊號n8n UI 上節點還在原位、連線沒斷、其他節點原樣。
04
匯入測試（隔離環境，1 筆樣本）
先在複本上跑。永遠先用 1 筆最小樣本，不用大批量。
角色動作
LLM 做什麼不做。這步完全由你在 n8n UI 操作。
你做什麼在 n8n UI 把原 workflow 複製成 <原名>-edit，確認 trigger 是 OFF，餵 1 筆樣本跑一次。
驗收訊號所有節點綠燈，output 結構符合步驟 1 便箋裡寫的目標 output。
05
錯誤回報（最小可重現訊息）
步驟 4 失敗時才進入。不要把整份 workflow 重貼，只貼三件事：紅燈訊息、input panel、節點 parameter。
角色動作
LLM 做什麼根據最小可重現訊息提出診斷，指出問題在哪個節點、哪個欄位。
你做什麼複製三件事給 LLM：(a) 紅燈節點的錯誤訊息全文，(b) 該節點的 input panel JSON，(c) 該節點當前的 parameter JSON（移除 credential）。
驗收訊號LLM 能指出問題在哪個欄位。如果 LLM 也不確定，代表回報資訊不夠，補資料，不是換 LLM。
步驟 5 找到問題後，回到步驟 3 做最小修改 → 再跑步驟 4 驗收。每輪問自己「比上一輪更接近目標嗎？」超過 3 輪沒有收斂，退回步驟 1 重新拆需求。
06
收斂（10 筆 → 200 筆）
用真實樣本跑通 10 筆後，才擴大到全量。這一步的重點是「停止繼續迴圈」。
角色動作
LLM 做什麼在步驟 3 / 5 內被呼叫，步驟 6 本身 LLM 不再介入。
你做什麼用 10 筆真實樣本跑通後，確認 output 對，才換回全量（例：200 張發票）。如果 LLM 越改越複雜，要求它「保留最小修改、移除新增的防禦邏輯」。
驗收訊號10 筆樣本全數綠燈，output 與步驟 1 便箋的目標一致。全量跑完無報錯。
i
步驟 4 和步驟 6 都由你在 n8n UI 操作，不需要 LLM。這不是「浪費 LLM」——這是讓結果可驗收的前提。LLM 的輸出永遠是草稿，跑過真實樣本才是答案。

第 5 頁
(04)

為什麼不能跳步

跳步的代價
六步流程每一步都是在控制下一步的風險。跳任何一步，等於拿掉對應的安全網。
跳過哪一步 常見後果
步驟 1
拆需求		 LLM 拿不到 input/output 的具體形狀，會自己腦補方向。結果往往是「改的方向對但細節全錯」，改好了卻不是你要的版本。
步驟 3
限定範圍		 LLM 會把整份 workflow 一起改。覆蓋整份 JSON 時，節點 ID 對不上、connections 斷掉、你的自訂設定全部不見。匯入直接失敗是最好的情況；匯入成功但靜默跑錯是最壞的。
步驟 4
匯入測試		 直接上 production workflow 跑。若 workflow 本來是開著的，改完瞬間就用未驗證版本接真實流量、讀真實檔案。一個欄位對應錯，200 張發票全部存到錯的路徑。
步驟 6
收斂		 用 1 筆樣本跑通就直接上全量。中段有隱藏錯誤時，200 筆全部默默失敗，事後才發現輸出有問題卻不知道從哪開始追查。
*
這六步對有程式背景的人來說看起來很保守。但這份手冊的對象是「不會寫程式、不熟悉 JSON 結構的結業學員」——保守的流程讓第一次走就能成功，成功一次之後就會內化成直覺，後來自然會縮短步驟。
