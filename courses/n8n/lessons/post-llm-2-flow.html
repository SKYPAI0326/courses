六步流程一頁就上手

8 頁系列 · 第 2/8 · 補充 · POST-COURSE

讀完這頁你會知道 · OUTCOMES
6 步流程的真正結構是「2 個迴圈」：動工前 3 步準備 + 動工後 3 步驗收，以及進場與出場條件各是什麼
每步你要做的 30 秒動作、完成後下一步的進場條件，以及步驟 5 失敗後迴圈只回步驟 3（不是步驟 1）
哪一頁深入講哪幾步，讓你知道下一步從哪裡跳讀
(01)

不是「6 件事」，是「2 個迴圈」

核心 mental model
前 3 步（拆需求、讀 JSON、限定範圍）是進入動工前的準備迴圈。後 3 步（測試、回報、收斂）是動工後的驗收迴圈。前 3 步沒做好，後 3 步會無限跑。

進場條件：你已經 export 了 #02 的 JSON，知道你要改成處理發票（或任何新目標），有至少 1 張真實樣本可以測試。

出場條件：pdf-renamed（或你設定的目標資料夾）裡有正確改名的檔案，10 筆樣本全部對，才能擴大到 200 張全量。

動工前 · 準備區塊
01 拆需求（4 格便箋）
02 讀 JSON（LLM 翻成人話）
03 限定修改範圍
做好這 3 步，LLM 才拿得到足夠脈絡。跳任何一步，後面都會補票。
→
動工後 · 驗收區塊
04 匯入測試（隔離，1 筆）
05 錯誤回報（三件套）
06 收斂（10 筆 → 全量）
步驟 5 失敗 → 回步驟 3（最小修改）。超過 3 輪不收斂 → 退回步驟 1。迴圈
(02)

6 步流程：每步 30 秒動作

讀圖方式
每步只看「你做什麼」和「做完才能往下走的進場條件」。細節、prompt 模板、踩雷實例在第 3–4 頁展開。
01
拆需求（4 格便箋）
這步刻意不用 LLM。需求模糊時 LLM 會自己腦補，越改越亂。
你做什麼用紙或便條填 4 格：原 workflow 編號、目標 input（例：發票 PDF）、目標 output（例：改名後存到 invoice-renamed）、必保留的節點（例：Gemini API 節點）。
下一步進場條件4 格能對著另一個人講 1 分鐘不打結，就可以往步驟 2。
02
讀 JSON（LLM 翻成人話）
把 workflow JSON 貼給 LLM，請它描述這份 workflow 做了什麼。移除 credential 欄位（credentials 整個 key）是硬規則。
你做什麼貼 prompt：「請把這份 workflow 翻成人話，列每個節點的輸入 / 輸出」。收到回應後，用 n8n UI 對照確認節點數量對得上。
下一步進場條件你能用 3 句話說出：原 workflow 吃什麼 input、中間做了什麼、產出什麼 output——不用看 JSON 也能說。能說得出就進步驟 3。
SAMPLE — 步驟 2 的 LLM 回應長這樣（#02 PDF AI 改名範例）
這份 workflow 做什麼：
從 pdf-inbox 資料夾讀取所有 PDF，用 Gemini 分析內文後給每張 PDF 一個語意化檔名，
並把檔案改名後存到 pdf-renamed 資料夾。整套流程是手動觸發。

節點清單：
1. Manual Trigger — 手動觸發點，不接受輸入。
2. Read PDF — 讀 /files/pdf-inbox/ 資料夾，輸出每張 PDF 的二進位內容。
3. Extract PDF Text — 把二進位 PDF 轉成純文字。
4. Code: AI 改名 — 用 Gemini API 分析 PDF 文字，產出 newFilename 字串。
5. Write — 把原 PDF 用 newFilename 改名後存到 /files/pdf-renamed/。

資料流向：Manual Trigger → Read PDF → Extract PDF Text → Code → Write
03
限定修改範圍（只回節點區段）
告訴 LLM「只改節點 X、Y」，要求它只回傳這幾個節點的 JSON 區段，不是整份。這步是防止 LLM 亂動你沒叫它碰的地方。
你做什麼貼 prompt：「只改節點 X，回傳區段 JSON，不重寫整份」。在 n8n UI 先選起要改的節點、複製備份。LLM 回傳後，在 UI 替換這幾個節點。
下一步進場條件n8n UI 上節點還在原位、連線沒斷、其他節點原樣，才往步驟 4。
i
「限定範圍」是指哪個節點？ — 對非工程師容易卡的問題，兩個具體例子幫你判斷：

例 1（#02 改成處理發票）：你要改的是「Code: AI 改名」節點裡的 Gemini prompt，讓它分析發票而非通用文件。其他節點（Read PDF、Extract Text、Write）不需要動。指令：「只改節點 Code: AI 改名，回傳這個節點的 JSON 區段」。

例 2（要改存檔路徑）：你要把 pdf-renamed 改成 invoice-renamed。這只在「Write」節點的 fileName 欄位。指令：「只改節點 Write 的 fileName 參數，其他欄位原樣，只回這個節點區段」。

判斷原則：在 n8n UI 點開節點，看哪個節點的哪個欄位需要變——能指名欄位最好，至少要指名節點名稱。
04
匯入測試（隔離環境，1 筆樣本）
先在複本上跑。永遠先用 1 筆最小樣本，不用大批量。這步不需要 LLM。
你做什麼在 n8n UI 把原 workflow 複製成 <原名>-edit，確認 trigger 是 OFF，餵 1 筆樣本跑一次。看每個節點的燈號。
下一步進場條件所有節點綠燈，且 output 結構符合步驟 1 便箋裡寫的目標 output，才往步驟 6 收斂。有紅燈就進步驟 5。
05
錯誤回報（三件套：錯誤 + input + parameter）
步驟 4 出現紅燈才進入。不要重貼整份 workflow，只貼三件事。
你做什麼複製三件事給 LLM：(a) 紅燈節點的錯誤訊息全文，(b) 該節點的 input panel JSON，(c) 該節點當前的 parameter JSON（移除 credentials key）。貼完問：「請只指出錯哪格」。
下一步進場條件LLM 能指出問題在哪個欄位。拿到診斷後，回步驟 3 做最小修改，再跑步驟 4。
步驟 5 找到問題後，回到步驟 3 做最小修改 → 再跑步驟 4 驗收。每輪問自己「比上一輪更接近目標嗎？」
停損訊號：如果你已經跑了 4 輪 LLM 對話還沒收斂，停止往下改。退回步驟 1，看便箋的 4 格哪一格寫得模糊——通常是「目標 output 的格式沒寫清楚」或「限定範圍的節點挑錯了」。
06
收斂（10 筆 → 全量）
步驟 4 綠燈後，用 10 筆真實樣本驗證，才擴大到全量。這步的重點是「停止繼續迴圈」。
你做什麼從你的 200 張裡隨機挑 10 張（或你手上全部，最少 5 張），跑通後確認 output 對，才換回全量。如果 LLM 越改越複雜，告訴它：「請保留最小修改，移除你新增的防禦邏輯」。
出場條件10 筆樣本全數綠燈，output 與步驟 1 便箋的目標一致，全量跑完無報錯。這就是整個流程的出場條件。
i
步驟 4 和步驟 6 都由你在 n8n UI 操作，不需要 LLM。這不是「浪費 LLM」——這是讓結果可驗收的前提。LLM 的輸出永遠是草稿，跑過真實樣本才是答案。
(03)

每步的 1 行 prompt 縮影

完整 prompt 模板在第 3–4 頁。這裡先給風格預覽，讓你知道每步 prompt 的長相。

PROMPT STYLE — 每步縮影
步驟 1（拆需求）
不用 LLM，用便條自己填。
步驟 2（讀 JSON）
「請把這份 workflow 翻成人話，列每節點輸出 / 輸入」
步驟 3（限定範圍）
「只改節點 X，回傳區段 JSON，不重寫整份」
步驟 4（匯入測試）
不用 LLM，在 n8n UI 跑。
步驟 5（錯誤回報）

步驟 6（收斂）

(04)

為什麼不能跳步

跳步的代價
六步流程每一步都是在控制下一步的風險。跳任何一步，等於拿掉對應的安全網。
跳過哪一步 你會看到什麼
步驟 1
拆需求		 LLM 拿不到 input/output 的具體形狀，自己腦補方向。workflow 跑出來但欄位是 null，或者輸出路徑對但檔名格式全錯——因為 LLM 猜了一個「看起來合理」的格式，而不是你要的那個。
步驟 3
限定範圍		 LLM 把整份 workflow 一起改，覆蓋整份 JSON。匯入時最常見的報錯是紅字 invalid JSON format 或節點跳 node type not found（這類錯誤去第 7 章「類別 1」查）。更壞的情況是匯入成功但靜默跑錯，200 張全部存到錯的路徑，事後才發現。
步驟 4
匯入測試		 直接在 production workflow 改，trigger 若是 ON，改完瞬間就用未驗證版本接真實資料。一個欄位對應錯，200 張發票可能全部存到 undefined 路徑或覆蓋原檔。
步驟 6
收斂		 用 1 筆綠燈就直接上 200 張全量。中段有隱藏錯誤時，200 筆全部默默失敗。事後翻 execution log 你會看到每筆都有 Cannot read property of undefined，但不知道從哪一張開始追查。
*
這六步對有程式背景的人看起來保守。但這份手冊的對象是「不會寫程式、不熟悉 JSON 結構的結業學員」——保守的流程讓第一次走就能成功。成功兩三次之後自然會縮短步驟，這是之後的事。
(05)

往哪走 — 下一章為什麼接著這樣排

課程動線
第 2 章給了你整個骨架。下一步是讓每步真正能動手——細節、模板、示範在第 3–4 章。

第 3 章（步驟 1–3 詳解）是你下一頁應該讀的。它會把 4 格便箋的格式給你、教你讀 JSON 的完整 prompt、說明如何用「示範形狀法」讓 LLM 不亂改結構。讀完第 3 章，你就能走完動工前的準備區塊。

如果你急著動手，第 3 章 + 第 4 章合起來就是完整的六步操作指南，可以照著跑第一遍。第 5 章（安全紅線）和第 7 章（錯誤分類手冊）是 reference，遇到問題再翻。
