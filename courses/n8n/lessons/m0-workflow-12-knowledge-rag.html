#12 · KNOWLEDGE BASE Q&A · 10 NODES · 進階

本地知識庫問答：自家 SOP/講義/手冊變成可問答 AI

公司有 SOP、課程講義、合約範本、客服手冊、FAQ 等 PDF/MD/TXT 文件，常見痛點是「資料都在但問問題時找不到」。這個 workflow 是「窮人版 RAG」：對所有文件做 chunk → JSONL 文字索引 → Webhook 問答端點。不需要 vector DB 也能跑（用關鍵字 match 取代 embeddings），但學員學進階版可換成 PostgreSQL pgvector + Gemini embeddings。

學習成果
能設計 chunk 切分策略（500 字 + 50 字重疊保留語境）並寫成 JSONL 索引
能用關鍵字 score + topK 做簡易 retrieval（比 embeddings 快 + 可解釋）
能設計含「來源引用」的 RAG prompt 讓 AI 答覆標 [1] [2] 而非空口無憑
(01) WORKFLOW 全貌

兩條獨立鏈：建索引 + 問答

Part A · 建索引（Manual Trigger）
Manual→Read knowledge-docs/→Extract 文字→Code: 切 chunk + JSONL→Write 索引

Part B · 問答（Webhook）
Webhook: kb-ask→Read 索引檔→Code: 檢索 + AI 生答→Respond

10 節點分兩條獨立鏈。Part A 跑一次（每次知識庫更新跑一次重建索引）。Part B 提供 webhook 端點，外部可 POST 問題拿答案。

(02) 重點節點拆解

兩個 Code 節點是核心

CORE 1 · 切 chunk + JSONL
為什麼 chunk size = 500 字 + 50 字重疊

500 字：適合 Gemini 讀取的單位（不會切太碎導致語境不完整 / 也不會太大讓 AI 抓不到重點）。50 字重疊：避免關鍵字剛好在 chunk 邊界被切斷找不到。

JSONL 格式（一行一 chunk）：

{"source":"員工手冊.pdf","pos":0,"text":"員工請假流程..."} {"source":"員工手冊.pdf","pos":450,"text":"請假分為事假..."} {"source":"客服 SOP.md","pos":0,"text":"客服回應..."}

vs 用 vector DB 的 trade-off：JSONL 直接寫本機檔，無需任何 DB。讀的時候整檔載入記憶體（10 萬 chunk 約 50MB OK）。如果文件量超過 100 萬 chunk 才需要 SQLite + FTS5 或 pgvector。

CORE 2 · 檢索 + AI 生答
關鍵字 score retrieval + 含來源引用 prompt

關鍵字 score：

const qWords = question.split(/[\s，。？！]/).filter(w => w.length >= 2); const scored = chunks.map(c => ({ ...c, score: qWords.reduce((s, w) => s + (c.text.includes(w) ? 1 : 0), 0) })).filter(c => c.score > 0) .sort((a,b) => b.score - a.score) .slice(0, 5);

把問題切詞 → 對每個 chunk 計算「含問題關鍵字的個數」當 score → 取 top 5。比 embeddings 簡單 + 可解釋（學員看到「為什麼這個 chunk 被選上」）。

RAG prompt 含來源引用：

// 把 chunks 編號 [1] [2] 餵給 AI const context = scored.map((c, i) => `[${i+1}] (${c.source})\n${c.text}` ).join('\n\n'); const prompt = `根據下面 context 回答用戶問題。 每段答覆引用來源編號 [1] [2] 等。如果 context 不足以回答，誠實說「資料庫內無此資訊」。 <question>${question}</question> <context>${context}</context>`;

關鍵：要求 AI 標 [1] [2] 引用，避免「空口無憑」幻覺。response 同時含 sources array，學員看答案時能對照原始檔。

(03) 改造方向

5 種改造

  1. 升級成真 vector RAG：把 keyword score 換成 Gemini text-embedding-004 + cosine similarity。能處理「同義詞 / 換句話問」。
  2. 加 Schedule 自動重建：Part A 改 Schedule Trigger 每天凌晨重建索引，自動同步新加的文件。
  3. 串 Telegram bot 對話：Telegram bot 收到訊息 → 打 webhook 到 Part B → 答覆推回 Telegram。變成「公司知識 Telegram 助理」。
  4. 加 metadata 過濾：每個 chunk 加 category / department / level 欄位，問答時依使用者權限只搜對應 chunk（比如業務只看公開 SOP，主管才看薪資制度）。
  5. 換本地 LLM：所有 AI 呼叫換成本地 Ollama → 公司機密 SOP 完全不離開本機。詳見 本地 LLM 銜接補充章節。
