M0 · BONUS · CLOUD vs SELF-HOST

n8n Cloud vs Docker 自架：誠實對比 + 你該選哪條

n8n 有兩條路徑：官方 Cloud（主推、$20-50/月、零維運）vs Docker 自架（本課程選擇、開源免費、但維運自己負責）。本頁不藏優劣、不偏袒任何一方，把 11 個維度真實對比給你看，幫你判斷自家業務適合哪條。看完你會知道：本課程選 self-host 不是因為 Cloud 不好，是因為有 3 件事 Cloud 真的做不到。

學習成果
理解 n8n 兩條路徑各自的成本結構（金錢成本 + 時間成本）與適用場景
能依自家業務 3 個關鍵問題（本機檔案 / 機密資料 / 量級）判斷選哪條
知道兩條路徑切換的可能性（先 Cloud 試水溫、量大後遷 self-host）
(01) 兩條路徑速覽

n8n Cloud 與 Docker 自架

PATH A · OFFICIAL CLOUD
n8n Cloud（官方主推）
怎麼用：到 n8n.io 註冊 → 幾秒鐘可用 → 信用卡綁好計費
收費：Starter $20 USD/月（5K executions）/ Pro $50（10K）/ Enterprise 客製
計費單位：execution（每次 workflow 跑一次 = 1，比 Make 的 ops 寬）
優勢：零維運、自動升級、官方支援、開箱即用
PATH B · DOCKER SELF-HOST（本課程）
Docker 自架（n8n Community Edition）
怎麼用：本機 / NAS / 自家伺服器跑 Docker → 完全免費（Apache 2.0 開源）
收費：$0 軟體費 + 電費 ~$5-15/月 + 你的時間維運
計費單位：無（你電腦能跑多少都行）
優勢：檔案不用上傳就能處理、可選本地 LLM、無 ops 限制
💡
兩種都是 n8n（同一套 workflow 設計、同樣的節點、同樣的 Expression 語法）。學員回家可隨時切換 — 本機跑通的 workflow JSON 可直接 import 到 Cloud；反之亦然（除非用了「本機檔案 / 本地 LLM」這類 Cloud 做不到的功能）。

🚨
self-host 隱私分層（重要釐清）：
Layer 1（預設 Lite Pack 設置） — workflow + 檔案 binary 在你電腦，但呼叫 Gemini API / Telegram 時 prompt 內容 / 訊息會送對應服務。解決「檔案不用上傳中轉」，不解決「資料零外傳」。

Layer 2（完整零外傳） — 所有 AI 換本機 Ollama + 不用 Telegram + 不對外 webhook。真正資料零外傳，但要走 m0-local-llm-guide 補充章節。

本課程 Lite Pack 預設是 Layer 1。對「檔案處理自動化、資料分類整理」這類需求 Layer 1 就夠用（檔案不離開本機，AI 只看 prompt 內容）。要處理合約 / NDA / 員工 PII 才需要 Layer 2。
(03) 為什麼本課程選 self-host

3 件 Cloud 真的做不到的事

不是因為「免費」
誠實說：你付的是電費 + 你的時間維運成本，不是 0 成本。本課程選 Docker 自架的原因是 Cloud 真的做不到下面 3 件事，剛好對應大部分學員（教育工作者 / 顧問 / 行銷企劃 / 課程設計師）的真實業務需求。
  1. 本機檔案不用上傳就能處理（核心優勢）— 課程 14 個 workflow 中 6 個（#02 PDF 改名 / #03 批次 dual-path / #10 客戶資料夾 / #11 CSV 清洗 / #12 知識庫 / #13 營運 CSV）都要讀寫本機資料夾。

    Cloud 必須先上傳 Drive 中轉：跑在 AWS / GCP 上的 n8n 看不到你電腦。要處理本機檔必須走「本機 → 上傳 Drive → Cloud n8n 從 Drive 抓 → 處理 → 結果寫回 Drive → 下載到本機」5 步驟，多 2-3 個 ops 而且失去「本機資料管線」的整潔度。

    self-host 直接讀寫本機資料夾：「PDF 丟資料夾 → n8n 處理 → 結果寫資料夾」3 步驟，檔案 binary 始終在你硬碟。

    注意：「檔案不上傳」 ≠ 「資料零外傳」。AI 處理時，PDF 內容會以 prompt 形式送 Gemini API 給 Google。檔案本身（PDF binary）沒上傳，但內容文字會給 AI 看。
  2. 可選本地 LLM 達成完全零外傳（進階） — 公司合約 / NDA / 員工資料連 prompt 都不該給 Google。本機 Ollama（Llama 3 / Gemma / Qwen）是唯一解。詳見 m0-local-llm-guide。

    Cloud 跑在 AWS 上，你電腦的 Ollama 在 localhost，雲端 n8n 連不到。要用本地 LLM 必須 self-host。

    本課程預設用 Gemini API（簡單 + 質量好），對「不含機密的檔案處理」完全夠用。要處理機密資料才換 Ollama。
  3. 無 execution 上限 — 高量批次（每天處理 1000 + 份文件、5 分鐘 polling 的 monitoring workflow）用 Cloud Starter 5K execution 一週用完。

    但要誠實：高量也要評估你電腦跑得動。一台 Mac M2 跑 1000 次 Gemini API 呼叫沒問題，但跑 100 萬次本地 Llama 3.2 推論會炸 RAM。極高量場景仍可能要租 GPU 雲端。
(04) 學員情境決策表

看你業務情境，5 秒判斷該選哪條

  1. CLOUD 主要做 SaaS 串接（Slack / Notion / Stripe / HubSpot 等），不碰本機檔
    → Cloud 比較省事。每月 $20 換零維運，你 60 分鐘安裝時間用來開發業務 workflow 更划算。
  2. SELF-HOST 要處理本機 PDF / 客戶機密 / 大量批次文件
    → self-host 是唯一解（本課程教這個）。Cloud 完全做不到本機檔讀寫。
  3. 兩者都行 純試用 / 學概念
    → Cloud 免費 trial 14 天最快上手；self-host 本課程 60-90 分鐘也能裝完。看你哪邊學習路徑順。
  4. SELF-HOST 公司資安要求資料不出機房
    → self-host on 公司 NAS / 內網伺服器（本課程教法可直接套用，把 Docker 跑在 NAS 而非個人筆電）。
  5. CLOUD 個人業務小量自動化 + 想省心 + 不在乎月費
    → Cloud Starter $20/月夠用。你的時間比 $20 值錢，外包給 n8n 維運。
  6. SELF-HOST 想學會自動化「全部能力」（含本機檔 / 本地 LLM / ops 監控）
    → 本課程教的這條。學會後上 Cloud 是降級（少了本機檔等），反向不行。
  7. 混搭 公司有純雲端 workflow + 本機批次處理
    → Cloud 跑雲端那部分（節省維運），self-host 跑本機那部分（補上 Cloud 做不到的）。兩邊用 webhook 串接。
  8. SELF-HOST + LAYER 2 處理合約 / NDA / 員工 PII / 商業戰略文件
    → self-host（檔案不離開本機）+ 本地 LLM Ollama（連 prompt 內容也不送雲）。詳見 m0-local-llm-guide。Layer 1（用 Gemini）會把文件內容送 Google，不適合機密場景。
  9. SELF-HOST 處理一般檔案（公開資料 / 內部紀錄 / 客戶報表）+ 想避免 Drive 中轉麻煩
    → self-host + Gemini（Layer 1）就夠用。檔案 binary 始終在你硬碟，AI 看 prompt 內容不算機密外洩。本課程預設這條。
(05) 兩條路徑切換

先 Cloud 試水溫、量大後遷 self-host 可以嗎？

可以，而且是合理路徑。n8n workflow JSON 在 Cloud 跟 self-host 是同一套格式，可直接 export / import 互通。常見遷移流程：

🔄
從 Cloud 遷到 self-host：
1. Cloud n8n 介面 → Workflows → 選要遷的 → Download（拿到 .json）
2. 本機 self-host n8n 介面 → Workflows → Import from File
3. Credentials 在 Cloud 是加密儲存，要在 self-host 重設（這裡不能直接搬）
4. 測試跑通後 Cloud 端關掉

反向（self-host → Cloud）：同樣方式，但要注意本機檔讀寫節點（Read/Write Files）在 Cloud 跑不起來，必須換成 Google Drive / S3 等雲端儲存節點。
📌
務實建議：本課程學完後，如果你發現你的業務根本不需要本機檔 / 本地 LLM / 高量，下個月就可以遷 Cloud 省維運心力。learn → 評估 → 切換 是健康的工程選擇。
總結

不藏優劣的選擇

本課程選 Docker self-host 的核心理由很單純：對 Make 學員，「本機檔案不用上傳就能處理」是真實業務痛點解方。Make + Cloud n8n 都要先把 PDF / CSV / 客戶檔案上傳 Google Drive 中轉，多 2-3 步驟、ops 翻倍、失去本機資料夾整潔度。self-host 直接讀寫本機資料夾，學員的工作節奏不被打斷。

至於「資料隱私」，要誠實分清楚 2 件事：「檔案不上傳」是 Layer 1（Lite Pack 預設達成），「資料零外傳」是 Layer 2（要再走本地 LLM 銜接）。對「客戶 NDA / 員工 PII / 商業戰略」場景才需要 Layer 2；一般檔案處理 Layer 1 已夠用。

如果你看完上面決策表，發現「我業務真的純雲端 + 不碰本機檔 + 在乎維運心力」，那 n8n Cloud 是更合理的選擇 — 不用為了「課程選 self-host」勉強自己學 Docker。課程剩下的內容（14 個 workflow 設計拆解）都是「自動化通用思考方法」，不管你最後用 Cloud 或 self-host，這些 workflow design pattern 都帶走。
