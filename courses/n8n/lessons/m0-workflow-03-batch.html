#03 · BATCH ERROR RECOVERY · 7 NODES · 專家

資料工廠精髓：100 份混合品質 PDF 自動分流，壞檔不會炸整批

本課程最複雜的 workflow，也是 n8n 相對 Make 的核心優勢區。表面看是「批次處理 + 改名」，內裡是 5 次迭代血淚換來的 n8n 對齊真相 — pairedItem bug、binary strip 規則、Code 節點 throw 污染、fileName 反查策略。讀完這頁你會帶走「能設計工程化容錯 workflow」的能力，而不只是「會用 n8n」。

學習成果
能設計 dual-path workflow（success / error 雙分流），用 onError continueErrorOutput 路由壞檔
能診斷 n8n 多 item 場景的 5 大對齊雷（pairedItem 標 0、Extract success 不留 fileName、binary strip、index 對齊飄移、Code throw 污染）
能用 fileName 反查 + items[i] 直接對齊兩條策略保證 binary 對齊 100% 可靠
(01) WORKFLOW 全貌

節點與資料流（dual-path）

Manual Trigger → Read 所有 PDF → Extract PDF Text

↗ success: Code: AI 改名 → Write processed/
↘ error: Code: 失敗處理 → Write failed/

7 個節點 + dual-path 結構。Extract 節點設 onError: continueErrorOutput，PDF 解析失敗的 item 走下方紅線到 fail 路徑。重點不在節點數，而在「**如何讓壞檔不污染整批 + 如何讓兩條路徑的資料對齊**」這兩個對 n8n 來說有 5 個雷的設計挑戰。

(02) NODE BREAKDOWN

節點逐一拆解（重點看第 4 + 第 6）

NODE 1-3 / 7
Manual Trigger / Read 所有 PDF / Extract PDF Text

跟 #02 結構一樣，差別只在資料夾路徑（batch-inbox 而非 pdf-inbox）。Extract 節點關鍵設定：onError: continueErrorOutput — 解析失敗的 item 不會炸整批，會走 error 分支到 Code: 失敗處理路徑。

NODE 4 / 7
Code: AI 改名（含 Gemini call）— success 分支
code v2runOnceForAllItemsv0.9

結構跟 #02 幾乎一樣：jsCode 內部 for loop 處理多份 PDF + 打 Gemini + thinkingBudget 0 + 從 Read 反查 binary。差別在這條路只接合法 PDF（壞檔已被 Extract error 抽走）。

關鍵差異：v0.9 的對齊策略 — const binFileName = items[i].binary?.data?.fileName; const readItem = allReadItems.find(r => r?.json?.fileName === binFileName) || allReadItems[i];

用 fileName 反查 readItem 而不是用 pairedItem.item，避開下方踩過的 pairedItem 標 0 的雷。

NODE 5 / 7
Write: 移到 processed/

fileName: {{ $json.newPath }} 寫到 shared/batch-inbox/processed/。

NODE 6 / 7
Code: 失敗處理路徑 — error 分支
code v2runOnceForAllItemsv0.9

用途：對 Extract error 分支送來的壞檔 item，組合 failedPath 給 Write: failed/ 用，並從 Read 反查 binary（因為 Extract error 路徑會 strip binary）。

關鍵 jsCode：

// fileName 從 input.json 拿（debug 證實 Extract error 保留 fileName）
const originalName = item.json.fileName || 'unknown.pdf';
// binary 從 Read 用 fileName 反查（Extract error strip 了 binary）
const readItem = allReadItems.find(r => r?.json?.fileName === originalName);
results.push({
  json: { status: 'failed', originalName, errorMsg, failedPath: ... },
  binary: readItem?.binary || {}
});
NODE 7 / 7
Write: 移到 failed/

fileName: {{ $json.failedPath }} 寫到 shared/batch-inbox/failed/，保留原檔名 + 原檔內容（從 Read 反查的 binary）。

(03) 5 次迭代真相

v0.6 → v0.9 的血淚史

為什麼這段重要
很多 n8n 教學講「dual-path」一筆帶過。實際在 multi-item + error branch 場景，n8n 有 5 個非顯性的對齊規則，少踩一個整個 workflow 看起來跑了但結果是錯的（檔名對 + 內容錯，最難 debug 的 silent failure）。下表是 5 次迭代發現的真相。
版本真相修法
v0.6Code Node 預設 runOnceForAllItems，多 input 只跑 1 次（4 份 PDF 只處理第 1 份）jsCode 改用 for loop 手動處理 $input.all()
v0.7Extract success 分支的 json 不保留 fileName 欄位，item.json.fileName 是 undefined從 item.binary?.data?.fileName 拿 fileName
v0.8n8n 在 Extract error 分支把 pairedItem.item 錯標成 0（不是真實上游 index）→ readItems[0] 對到第一份 PDF 不是壞檔放棄 pairedItem，用 item.json.fileName 從 Read 反查（fileName-match）
v0.9Code: AI 改名 throw 走 onError continueErrorOutput 會污染 fail 分支的 input（input 變成「Code throw 物件」沒 fileName）→ 4 個合法 PDF 都被誤分到 failCode: AI 改名 拿掉 onError continueErrorOutput，throw 直接讓 workflow 失敗（or 用 try/catch + fallback 命名）。fail 分支只接 Extract error
v0.9Extract error 分支會 strip binary（json.fileName 還在但 binary 沒了）Code: 失敗處理 從 Read 用 fileName 反查 binary
⚠️
silent failure 才是最可怕的：上述 5 個雷大多不是 workflow 直接炸，而是「跑了但結果錯」— 檔名對齊到別人的 binary、success 分支被污染進 fail 分支、4 份 PDF 只處理 1 份。看似 workflow 通過實則資料錯亂。Lite Pack v0.9 在 jsCode 內加 throw new Error('[i=N] No binary. fileName=..., readNames=[...]') 詳細 debug 訊息，當下就能定位是哪個對齊出錯。
(04) 改造方向

把這個 workflow 改成你業務的 5 種方向

  1. 加 summary log：在 Write processed/ 與 Write failed/ 之後加 Merge 節點 + Code 節點，跑完一批寫一份 _summary-{時間戳}.md 列出「N 個成功 / M 個失敗 / 各檔名清單」。
  2. 改 Schedule Trigger：Manual 換成每天凌晨 4:00 自動跑，batch-inbox/ 變「下班後丟一堆 PDF，明早來看歸檔結果」工作流。
  3. 加 retry 機制：失敗 item 不直接進 failed/，先記到 retry queue，3 分鐘後重試一次（API rate limit 自動恢復），3 次都失敗才進 failed/。
  4. 串 Telegram 通知：跑完一批 Telegram 通知「✅ 處理 N 份，X 份成功歸 processed/，Y 份失敗保留 failed/」。
  5. 換 storage：processed/ 從本機資料夾換成 Google Drive / Dropbox / S3，做雲端歸檔。
