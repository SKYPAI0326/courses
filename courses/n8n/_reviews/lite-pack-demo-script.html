明天課堂 Demo 順序與口白速查

依 Codex 第三輪建議：保 3 個核心 workflow（01 / 02 / 05），其他標 Bonus。明天 demo 流程鎖定「學員 30 分鐘內看到自己的工具運作」這個體感成就。所有指令都是預先驗證過的 v0.4 路徑，不要臨場改流程。

PRE-CLASS 開課前 30 分鐘 · 講師端準備
-30 min到場
講師端環境檢查
  • 已跑著的 n8n（瀏覽器開 localhost:5678 有 8 個 workflow + 2 個 credentials）
  • cloudflared 在跑（顯示 trycloudflare.com URL）
  • Docker Desktop 鯨魚圖示綠燈
  • 講師備用 Gemini API key（學員用完額度時轉用）
-15 min
分發 Lite Pack zip 連結 / 投影到大螢幕
  • 學員下載 zip 路徑：~/Downloads/n8n-lite-pack.zip
  • 講師螢幕投影 setup-wizard 的 4 個對話框長相
  • 確認教室網路 cloudflared 通（少數環境會擋 QUIC，HTTP/2 已預設）
0-30 min 第一節：學員下載安裝（含 demo 講師版）
0-5 min開場
講師現場跑 setup-wizard demo（投影）
口白：「我先示範整個流程——下載 zip、雙擊、填 3 個值、30 秒後 8 個 workflow 全部就緒。然後你們照做。」
  • 講師 Mac 雙擊 setup-wizard.command 跑一次
  • 填預備好的 demo Gemini key + Telegram bot token + Chat ID
  • 學員看終端機輸出進度 [1/10] → [10/10]
  • 看到 ✅ 完成對話框 + 自動開 n8n UI 顯示 8 個 workflow
賣點口白：「這就是『一鍵』的真正意義——不是單純執行檔，而是把所有可能踩雷的事都包進去（file access patch、container 重啟、credential 加密、workflow 自動關聯、Telegram + Gemini smoke test）。」
5-25 min學員操作
學員自己跑一次 setup-wizard
前置條件（學員要先有）：
  • Gemini API key：aistudio.google.com/apikey 申請（免費）
  • Telegram bot token：對 @BotFather /newbot
  • Chat ID：api.telegram.org/bot{TOKEN}/getUpdates
常見現場問題：
  • Gatekeeper 擋 → 系統設定 → 隱私權與安全性 → 強制打開
  • 找不到 cloudflared → 跳過（這個 setup-wizard 不需要 cloudflared）
  • Telegram bot 沒回應 → 通常是 chat ID 錯，重貼一次
25-30 min
驗收：所有學員看到 ✅ 完成對話框
檢核全班通過：所有人 Telegram 收到「✅ Lite Pack v0.4 setup-wizard 安裝完成！」訊息。沒收到的學員停下來，講師個別處理。
30-90 min 第二節：3 個核心 Workflow Demo
30-45 min#5 最簡單先過
CORE#5 Telegram 通知測試
目的：驗證每位學員的 Telegram bot credential 真的能用 + 體驗「按 Execute → 手機收到訊息」最直接的 AHA moment。
  • n8n UI 開 05 · Telegram 通知測試
  • 點下方紫色 Execute workflow
  • 3 秒後 Telegram 收到「✅ Lite Pack 通知測試成功」
口白：「這 4 個節點（Manual → Set → Telegram）你之後可以拿來做任何工作流的『成功通知』模組。把 Set 節點的訊息改一下、串到別的 workflow 後面，就是完整的通知系統。」
45-70 min#2 招牌 demo
CORE#2 PDF AI 改名（招牌 demo）
目的：展示「本機檔案 + AI」這個 Lite Pack 招牌——學員看到 PDF 進去 → 系統處理 → 改名輸出。
  1. 講師示範：~/Downloads/n8n-starter-kit/shared/pdf-inbox/ 拖一份 PDF
  2. n8n 開 02 · PDF AI 改名 → Execute workflow
  3. 每個節點變綠（5-15 秒）
  4. 看 pdf-renamed/ 出現新檔名
已知狀況：AI fallback 可能觸發
  • 最佳：20260506_說明_xxx.pdf（AI 真回應）
  • fallback：20260506_文件_xxx.pdf（AI empty，PDF 第一行抽取）
  • 兩種都算 demo 通過——fallback 是「工程化容錯」教學素材
fallback 觸發口白：「AI 沒每次都會回——這就是我們課程教的『6 層 fallback』設計。看 Code 節點 Output 有個 parseMethod 欄位告訴你走哪條。這比『AI 不回應 = 工作流壞掉』好太多。」
70-90 min#1 + 進階概念
CORE#1 Webhook hello-world + 連接外部世界
目的：讓學員理解「webhook = 你的 n8n 對外開的門」。
  1. n8n 開 01 workflow → Execute → 等狀態變藍
  2. 瀏覽器開 http://localhost:5678/webhook-test/lite-hello
  3. 看到 {"message":"hello from lite pack"}
  4. 講師再示範用 cloudflared 拿到 trycloudflare.com URL（如 Phase B 已做）
  5. 學員打公開 URL，全班人都能觸發講師的 n8n（demo 「對外開門」概念）
90-180 min 第三節：Bonus Workflows + 改造練習
90-120 min
BONUS#3 批次處理 / #4 定時日報 / #6 Webhook→Gemini→檔
說明：這些 workflow 已 import 但**未經完整實測**。可以演示但若失敗，講師現場改成「打開 workflow 看設計，當教學素材」，不要硬跑。
  • #3 批次處理錯誤恢復：開 workflow 看「成功 / 失敗雙路徑分流」設計
  • #4 定時日報：開 workflow 看「Schedule + Aggregate + AI」結構
  • #6 Webhook→Gemini→檔：開 workflow 看「同一個 Gemini call 三通道輸出」設計
口白策略：「這 3 個是進階範例。你們今天先學會看懂結構、課後改造成自己的場景就是結業作品。」
120-180 min
改造練習把 #5 改成自己場景
學員任務：把 #5 Telegram 通知改成自己的場景。
  • 場景 A：每日 8:00 自動發「今日待辦」（換 Manual Trigger 為 Schedule Trigger）
  • 場景 B：webhook 觸發 → 收到外部請求發通知（接 #1 邏輯）
  • 場景 C：跑完任何工作流發「完成」通知（其他 workflow 串接）
講師走班協助。完成的學員有「結業作品」可帶走。
PM 下午：原理深入 + Q&A（給有心學員）
PM 1180-240
講原理：n8n 整體架構 + 設計哲學
  • n8n 是什麼：visual programming + Code node 雙模式
  • Trigger / Action / Sub-workflow / Credential 四個核心概念
  • setup-wizard 在做什麼：CLI import / docker exec / file access patch
  • 為何選 Lite Pack 而非 Cloud：本機 = 資料留本機 + 無 ops 限制
PM 2240-360
改造工作坊（學員每人改 1 個 workflow）
每位學員選 1 個既有 workflow，改造成自己的場景，匯出 JSON 帶走。講師逐一檢視。
PM 3360-480
Q&A + 結業
  • 每位學員展示自己的改造作品（5 分鐘 / 人）
  • 下一步建議：n8n Cloud / VPS / 進階課程銜接
  • 群組成立（持續答疑）
TOOLS 講師現場工具箱
現場常用指令（如有學員卡關）
看哪個學員的 setup-wizard 卡哪：tail -f /tmp/n8n-lite-setup-*.log
學員 n8n 沒起來：docker ps --filter "name=n8n"
學員 file access 報錯：grep N8N_RESTRICT ~/Downloads/n8n-starter-kit/n8n-compose.yml
Gemini API 額度用完 → 用講師備用 key 替換 personalization.env
Telegram bot 沒回應 → 看 chat ID 是否正確（純數字）
課堂緊急應變（demo 失敗時）
#02 PDF AI 改名失敗：用 fallback 結果當「工程化容錯教學素材」
#01 Webhook 失敗：直接用 docker exec 看 logs / 改用 n8n UI 內建測試
#05 Telegram 失敗：個別學員問題（chat ID / token）→ 暫時打發給課後處理
全部失敗：直接 demo 成品（投影片或預錄影片），講原理為主
講師心態（最重要）
學員是商業培訓人員，要結果不要原理（Codex 第三輪建議）
明天能贏的版本不是「8 個 workflow 完整」，而是「學員看到自己的工具運作」
fallback 不是失敗，是工程化容錯設計（demo 時要主動講這個）
學員問「為何 X 那樣設」時，能答就答，不能答就說「我們課後群組討論」
