🔴 STEP 0 · 必做（不是選用）
完全歸零本機所有 n8n 相關設定

這次是「本機重佈 + 從零驗證」，先把過去 16 連發累積的所有殘留全清掉，避免新跑的撞 port / 撞 volume / 撞 credential / 撞 cloudflared process。每個指令貼下去執行，不要跳。

會清掉：所有 n8n / postgres container + volume + network、cloudflared 殘留 process、所有 starter-kit / lite-pack 解壓資料夾、所有殘留 zip。不會動到：Docker Desktop 本身、Docker image cache（重灌會直接拿 cache，不重新下載 GB 級 image）、Telegram bot、Gemini API key、Cloudflare 帳號（這些都在雲端）。

PHASE A
n8n 環境建置（重新跑一次）
下載最新 starter-kit zip → 解壓 → 雙擊啟動 → 建 Owner Account。預估 10-15 分鐘。
A1
下載最新 n8n-starter-kit.zip
從課程 1.1.2 講義頁下載（綠色按鈕「⬇ 下載 n8n-starter-kit.zip」）。覆蓋掉舊資料夾，最新版含 9 連發所有修補（auto port 衝突偵測、postgres 密碼自動重置、http2 預設、tunnel-quick.command 等）。
注意
確認下載到 ~/Downloads/n8n-starter-kit.zip，雙擊解壓會在同層產生 ~/Downloads/n8n-starter-kit/。如果之前沒清掉，會解壓成 n8n-starter-kit 2（macOS 防覆蓋）→ 回上一段 Step 0 重做歸零。
A2
macOS 強制打開 4 個 .command
解壓後資料夾內有 start / stop / update / tunnel-quick 共 4 個 .command。每個第一次雙擊都會被 Gatekeeper 擋。**一次處理 4 個比較有效率**：
  1. 分別雙擊 4 個 .command → 都會跳「無法打開」對話框 → 按「取消」
  2. 系統設定 → 隱私權與安全性 → 捲到「安全性」區塊
  3. 會看到「'start.command' 已被阻擋使用」→ 按「強制打開」→ 確認密碼
  4. 對 stop / update / tunnel-quick 重複 step 3
預期看到
4 個 .command 都被加到「允許執行」清單。之後雙擊不會再被擋。
A3
第一次雙擊 start.command
雙擊 ~/Downloads/n8n-starter-kit/start.command。第一次會自動把 .env.example 複製為 .env 並打開 TextEdit 讓你編輯。
改密碼
把 POSTGRES_PASSWORD=please_change_me_to_a_strong_password 那行的值改成自己的強密碼（至少 16 字元、混合英數字符），存檔關閉。
不要再改第二次
Postgres 密碼第一次寫進 volume 之後，**之後改 .env 不會自動同步**，會造成「password authentication failed」（連發 #4 那個坑）。新版 start.command 會自動偵測這個錯誤並跳對話框讓你一鍵重置，但最簡單是「設好就不要再改」。
A4
再次雙擊 start.command
這次會真的拉 Docker image + 啟動 container。首次需 1-5 分鐘下載 postgres / n8n image。終端機會每 10 秒輸出「⏳ 已等候 N 秒...」進度（v0.4 加的防呆）。
預期看到（成功）
終端機顯示 ✅ n8n 已啟動：http://localhost:5678，瀏覽器自動開到 n8n 主介面。
如果跳「Port 5678 已被佔用」對話框
新版自動偵測 port 衝突——按「停掉並重啟」按鈕，會自動清舊 container 並重啟。
如果跳「postgres 密碼認證失敗」對話框

PHASE C
Lite Pack CLI 驗證（核心測試）
驗證「下載安裝後設兩個 key 即用」這條 setup-wizard 自動化路徑能不能 work。預估 10 分鐘。
C1
找 n8n container 名稱
確認當下實際在跑的 container 名（不同次解壓會產生不同名字）。開新終端機視窗跑：
docker ps --filter "name=n8n" --format "{{.Names}}"
預期看到
會列出 1-2 個 container 名稱（n8n + postgres）。記下名稱結尾是 n8n-1 那個（例如 n8n-starter-kit-n8n-1），下面 C3-C6 要用這個。
C2
建立測試 credentials JSON
寫一份「decrypted credentials JSON」到 /tmp/test-cred.json，含 2 個 fake credentials（不會真的能用，純測 import 能不能成功）。
echo '[{"id":"lite-test-gemini","name":"Lite Pack TEST · Gemini","type":"httpHeaderAuth","data":{"name":"x-goog-api-key","value":"AIza-test-key-not-real"}},{"id":"lite-test-telegram","name":"Lite Pack TEST · Telegram","type":"telegramApi","data":{"accessToken":"999999:TEST-ABC","baseUrl":"https://api.telegram.org"}}]' > /tmp/test-cred.json

確認檔案內容（選用）：

cat /tmp/test-cred.json
為什麼用單行 echo 而不是 heredoc
heredoc（cat > ... <<'EOF'）對縮排敏感——終端機自動排版貼上常會在每行前加空格，導致結束符號變 EOF（有空格）對不上 EOF，bash 一直等不到結束會卡在 heredoc> prompt。單行 echo 沒這問題，雖然不易讀但複製貼上 100% 穩。
為什麼要 id 欄位（連發 #11 修補）
n8n 1.x+ 的 credentials_entity 表 id 欄位 NOT NULL 約束，**JSON 內必須自帶 id**（任何 unique 字串都行）。少了會報 null value in column "id" violates not-null constraint。
預期看到
終端機沒任何輸出（成功），檔案已寫到 /tmp/test-cred.json。
C3
設定 container 變數
把 C1 拿到的名字填到下面變數，方便接下來重複使用。把預設值換成你自己 C1 看到的。
CONTAINER=n8n-starter-kit-n8n-1   # ← 改成你 C1 的名字
注意
必須在同一個終端機視窗執行 C3-C6，$CONTAINER 變數只在當下 session 有效。
C4
複製 JSON 進 container
docker cp /tmp/test-cred.json $CONTAINER:/tmp/test-cred.json
預期看到
類似 Successfully copied X.XkB to ...:/tmp/test-cred.json。
C5
執行 n8n import:credentials（核心驗證）
這是最關鍵的一步——驗證 n8n CLI 真的能從 decrypted JSON 自動建 credentials。
docker exec -u node $CONTAINER n8n import:credentials --input=/tmp/test-cred.json
預期看到（成功）
類似這樣的訊息：
Successfully imported 2 credentials.
常見失敗訊息對應
「No user found」/「Owner account not set up」 → A5 沒做完（沒建 owner account）
「No such file or directory」 → C4 docker cp 沒成功，或 $CONTAINER 變數錯
「Invalid credential type」 → n8n 版本與 schema 不相容（少數情況）
失敗請貼錯誤整段給我。
C6
驗證 credentials 真的建好了
docker exec -u node $CONTAINER n8n list:credentials | grep "Lite Pack TEST"
預期看到（兩個 TEST credential 列出來）
Lite Pack TEST · Gemini ...
Lite Pack TEST · Telegram ...

或打開 n8n UI（http://localhost:5678） → 左側選 Credentials → 應該看到剛建的兩個 TEST credential。

grep 找不到 ≠ import 失敗
C5 的 Successfully imported 2 credentials. 才是真正的成功訊號。C6 grep 只是輔助確認，找不到通常是 grep 關鍵字與 credential name 沒對齊（C2 改了 name 但忘了改 C6 grep），不影響後續 Phase D。
🎯 三件事齊備 = 整個 Phase C 通過
(1) C5 顯示 Successfully imported X credentials
(2) C6 grep 列出對應 credential（用對的關鍵字）
(3) n8n UI Credentials 頁看到該 credential
Phase C 通過 = setup-wizard v0.2 路徑可實作，可開始做 7 個剩餘 workflow + 完整 Lite Pack 收尾。
🧹 驗證完選用 · 清理 Phase C TEST credential

D8 · 診斷 AI 是否真的工作（v0.6 走 Code node）
v0.6 已廢 HTTP Gemini node，Gemini 改在 Code: AI 改名（內含 Gemini call） 節點內直接打。診斷時點 Code 節點 看 Output 面板（不是 HTTP node）。

看五個診斷指標（Code 節點 Output 的 json 欄位）：

  1. parseMethod：ai-plain = AI 真的回應 ✅ ｜ regex-date = AI 回但混雜文字（仍算贏）｜ pdf-title 或 fallback-empty = AI 沒回 ❌
  2. finishReason：STOP = 正常 ｜ SAFETY = PDF 觸發內容過濾 ｜ MAX_TOKENS = 回應被截斷
  3. aiResponseLength + aiResponsePreview：AI 回了多少字 + 前 200 字長相
  4. pdfTextLength：PDF 文字真的抽出來了嗎？(0 = ExtractFromFile 失敗)
  5. httpError：如果 helpers.httpRequest throw，這欄會有錯誤訊息（key 錯 / quota 用完 / network 不通）
常見問題對照表
症狀 根因 修法
401/403 + API key not validGemini key 錯或過期重 setup-wizard 改 key
404 + model not foundmodel 名稱錯改 URL 為 gemini-2.0-flash
429 + RESOURCE_EXHAUSTED當日 quota 用完等明天 reset 或申請新 key
200 + finishReason: SAFETYPDF 內容觸發 SAFETY換另一份 PDF 測試
200 + finishReason: MAX_TOKENS回應被截斷調高 maxOutputTokens
aiResponseLength=0 但 status=200Gemini 真的回空換 prompt 或 model
