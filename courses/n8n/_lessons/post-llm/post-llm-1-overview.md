## 重設計內容

## 第 1 章重設計：把「為什麼進來」講清楚

### 1. 學員此時的痛點

學員結業 1-3 週，在工作崗位接到具體任務（例：200 張供應商發票要分類歸檔），打開 #02 看到 prompt 是針對合約寫的，第一個本能就是「叫 ChatGPT 幫我改」。他/她在這頁要解決的問題：

- **我可以直接整份貼給 ChatGPT 叫它改嗎？**（直覺答案：可以；正確答案：不可以，會匯不進去）
- **這份手冊跟我自己上 ChatGPT 摸索差在哪？**
- **我要花多少時間才能改好一個 workflow？我的 200 張發票今天能不能搞定？**

### 2. 核心要傳遞的 1 個觀念

**「整份丟 LLM 改」會浪費你 2 小時 debug；「按 6 步驟限定範圍改」首次 40 分鐘、第 3 次 15 分鐘就會內化成例行操作。**

### 3. 具體 step-by-step（學員讀到這章的動線）

學員在這章不需要動手做事，但讀完後要能做這個決策：

- **看完 1-2 分鐘**：判斷「這手冊跟我有什麼關係」（5 個 timeline story 對照）
- **看完 3 分鐘**：清楚「我的具體任務在第幾頁找到 walkthrough 對照」
- **看完 5 分鐘**：知道下一頁讀什麼（線性 / 跳讀）

### 4. 完整範例 prompt 模板

第 1 章不需要 prompt 模板（純 orientation 章），但要給一個**「整份丟 LLM 的反例 prompt」**讓學員看「為什麼這樣做會失敗」：

```
反例 prompt（不要這樣用）：
「我用 n8n 的 #02 workflow，請幫我改成處理發票，給我完整的 JSON 我直接匯入。」

為什麼這樣會失敗：
1. LLM 不知道你的發票 PDF 長什麼樣（哪些欄位）
2. LLM 不知道你裝的 n8n 是哪一版（節點 typeVersion 會猜錯）
3. LLM 從記憶生成的整份 JSON，節點 ID 對不上、connections 引用斷裂
4. 你拿到一份「看起來能用」但匯入時跳「invalid format」的 JSON
5. 你花 1 小時 debug 後放棄，回到手動分類
```

### 5. 範例 LLM 回應

不適用（第 1 章不執行 prompt）。

### 6. 驗收標準（讀完第 1 章自評過關）

學員讀完第 1 章後應該能回答：

- [ ] 我能說出「這份手冊解決什麼問題」（一句話）
- [ ] 我能說出「我手上的具體任務」對應哪一個 timeline story（5 選 1）
- [ ] 我能決定「我要線性讀完還是直接跳第 6 章 walkthrough」
- [ ] 我知道「整份丟 LLM 改」為什麼會失敗（能說出 1 個原因）

### 7. 常見錯誤 + 怎麼解

**錯誤 1：「我先去看第 5 章紅線就好，紅線知道了我自己摸索」**
- 解：紅線是「不能踩的」，不是「該做的」。光看紅線你只知道哪些坑，不知道正確路徑。建議至少先讀第 6 章 walkthrough 看一遍正確路徑長什麼樣

**錯誤 2：「我覺得我的場景很特殊，5 個 timeline story 都不像」**
- 解：5 個 story 不是要 100% 對上你的場景，是要對上你的「改造類型」。改 prompt（合約變發票）、改節點（Sheet 變 Notion）、改 trigger（手動變 Drive 觸發）、加 filter（過濾客戶）、版本升級——5 種類型涵蓋 90% 改造需求。先選最接近的一種

### 8. 回到課程動線

- **線性讀**：→ 第 2 章 6 步流程總覽（瞭解骨架）
- **跳讀情境 A**（我已經知道要改 #02 處理發票）：→ 直接第 6 章 walkthrough
- **跳讀情境 B**（我已經卡在某個錯誤訊息）：→ 直接第 7 章錯誤分流
