# 步驟 4-6 真詳解 — 匯入測試、錯誤回報、收斂

> **本章重設計教案** · 對應 `courses/n8n/lessons/post-llm-4-steps-import.html`

## 重設計內容

## 第 4 章重設計：步驟 4-6 真詳解

### 1. 學員此時的痛點

學員從第 3 章拿到 LLM 改好的節點區段，貼進 n8n UI。心理狀態：

- **我點 Execute Workflow 會出事嗎？會不會把 200 張發票真的去 LLM 跑掉錢？**
- **跑出紅燈我怎麼跟 LLM 說？我貼整份給它它會幫我改吧？**（直覺答案：對；正確答案：不對，會越改越亂）
- **跑通了 1 筆怎麼判斷可以放 200 張了？**
- **LLM 一直加 try/catch 加新節點，我要不要接受？**

### 2. 核心要傳遞的 1 個觀念

**測試-回報-收斂這 3 步是「迴圈閥」：你是把關人，每輪結束問自己「比上輪更近還是更遠」。LLM 給的 JSON 是草稿，跑過真實樣本才是答案。**

### 3. 具體 step-by-step

#### 步驟 4：匯入測試（10 分鐘，全人工）

> **與第 3 章關係**：Duplicate 動作已經在第 3 章 step 6（硬性前置）做過了。本步驟只 **verify** Duplicate 結果是否健全，不重新建。
>
> **Phase C 對照（教學用）**：第 6 章 Phase C walkthrough 為了示範「學員第一次很可能忘記順序」這個常見錯誤，故意演了「先替換到原版才想到要 Duplicate」的劇情並補救。**那是教學示範錯誤，不是 best practice**。本章（第 3+4 章）教的順序「先 Duplicate 再替換」才是正確流程，請以本章為準。

1. **verify Duplicate 結果**（檢查第 3 章 step 6 的產物，不是重新 Duplicate）
   - 動作：n8n UI Workflows 列表，找到 `02-pdf-ai-rename-edit`，點開
   - **驗收硬指標 (a)**：節點數和原版一致（在原版開一個分頁對照數一遍，例如原版 5 個節點，-edit 也要是 5 個）
   - **驗收硬指標 (b)**：credentials 跟著來了 — 點任一含 credential 的節點（例：HTTP Request、Gemini、Google Drive），看 Credentials 欄位是否還掛著原本那個（n8n Duplicate 預設會繼承；若顯示「Select Credential」表示沒繼承，要手動選回去）
   - **驗收硬指標 (c)**：Active 開關是「灰色 OFF」（Duplicate 預設關，仍要肉眼確認）
   - 如果以上任一條不過 → 退回第 3 章 step 6 重做 Duplicate
2. **確認 trigger 是 Manual / OFF**
   - 動作：打開 -edit 版，看 workflow 右上角的 Active 開關，必須是「灰色關閉」狀態
   - **驗收硬指標**：開關文字顯示「Inactive」或「OFF」
3. **準備 1 張測試 PDF**
   - 動作：把 1 張你已經有的真實供應商發票（或拿手邊 1 張舊合約 PDF 當煙霧測試）放到 `/files/pdf-inbox/`
   - **驗收硬指標**：n8n UI 上 Read PDF 節點點 Execute Step 能列出這張 PDF
4. **點 Execute Workflow**
   - 動作：n8n UI 下方紫色「Execute Workflow」按鈕
   - **驗收硬指標**：5 個節點全部變綠燈
5. **點每個節點看 output panel**
   - 動作：點節點 → 右側 panel 上方有 4 個 tab：Input / Output / Settings / 其他 → 切到 Output → 看 JSON 樹
   - **驗收硬指標**：最後一個節點 (Write) output 裡，找到 `newFilename` 欄位，值符合便箋上寫的命名規則
6. **去 Finder / 檔案總管確認**
   - 動作：打開 `/files/pdf-renamed/`
   - **驗收硬指標**：看到 1 張改好名的 PDF，檔名是預期格式

#### 步驟 5：錯誤回報（5-15 分鐘 / 輪，學員 + LLM）

**進場條件**：步驟 4 出現紅燈節點，或 output 裡有 null / 空字串

1. **找到紅燈節點**
   - 動作：n8n UI 上紅色框框的節點，點它，下方面板會跳出錯誤訊息
2. **複製錯誤訊息全文**
   - 動作：滑鼠選取紅燈節點底下的整段紅字 → Cmd+C
   - **驗收硬指標**：複製到的文字含「Error:」或具體訊息（如「Cannot read property 'X' of undefined」）
3. **複製該節點 input panel**
   - 動作：點該紅燈節點 → 右側 Output tab 切到 Input → 點 JSON 顯示模式（不是 Schema 模式）→ 右上角複製按鈕
4. **複製該節點 parameter**
   - 動作：點該節點旁齒輪圖示 → Settings 看到 JSON → Cmd+A 全選 → Cmd+C
   - **驗收硬指標**：複製到的 JSON 開頭是 `{ "parameters": ...`，**搜尋 apiKey/accessToken 全部找不到**
5. **貼模板 4 給 LLM**（見第 4 節）
6. **驗收 LLM 回應**
   - **驗收硬指標 — 「精準回答」長這樣**：
     - 「問題在 parameter 的 `prompt` 欄位裡，第 3 行的『供應商』應改成『賣方』，因為 input panel 顯示 PDF 文字使用『賣方』」
     - **特徵**：(a) 指出具體欄位名 (b) 給出具體修改值 (c) 引用 input 內容當證據
   - **驗收硬指標 — 「太模糊」長這樣**：
     - 「這個錯誤可能是 prompt 寫法問題，建議重新檢查 Code 節點」
     - **特徵**：「可能」「建議」「重新」 — 沒指具體欄位
   - **遇到太模糊就追問**：「請告訴我具體是哪一行的哪個欄位字串要改成什麼，不要建議『重新檢查』」

#### 步驟 6：收斂（變動，最多 3 輪 + 全量執行）

1. **每輪結束自問「比上輪更近還是更遠」**（用可數的 proxy 取代抽象判斷）
   - **「近」的具體訊號**（任一條成立 = 近）：
     - 紅燈節點數量比上輪少（例：上輪 2 個紅燈，這輪 1 個 → 近）
     - output 裡 null / 空字串的欄位變少（例：上輪 newFilename 是 null，這輪有值即使值不對 → 近）
     - LLM 動到的節點數比上輪少（例：上輪叫你改 3 個節點，這輪只改 1 個 → 近，代表逼近根因）
   - **「遠」的具體訊號**（任一條成立 = 遠 → 停下檢查）：
     - 出現上輪沒有的新紅燈（例：上輪是 Code 節點紅，這輪 Code 綠了但 Write 紅 → 不是進展，是 LLM 把問題搬家）
     - jsCode 行數比上輪多 30% 以上（例：上輪 30 行，這輪 50 行 → LLM 在加防禦邏輯，不是改根因）
     - LLM 提案動到的節點數比上輪多（例：上輪只動 1 個，這輪要動 3 個 → 範圍蔓延）
   - **不確定怎麼算（持平）**：紅燈數一樣 + null 欄位一樣 + 行數差不多 → 算「持平」，記為 1 輪但不算進步；持平 2 輪也要停下退回步驟 1
2. **拒絕 LLM 加防禦邏輯（接受/拒絕判斷樹）**
   - **拒絕**：LLM 提案動到「節點數量」（新增 / 刪除節點）→ 拒絕（用模板 5）
   - **拒絕**：LLM 提案加 try/catch、if 條件、error handler、retry → 拒絕
   - **可接受**：LLM 只改原節點裡的「值」（字串、數字、欄位 expression）→ 可接受
   - **可接受**：LLM 在現有 Code 節點裡加 console.log 幫你 debug 用 → 可接受（debug 後記得刪）
3. **3 輪沒收斂的處理**
   - 動作：停止 LLM 對話 → 退回步驟 1，看 4 格便箋是否有「目標 input」格寫得太模糊
   - **常見原因**：input 那格只寫「發票 PDF」沒寫具體欄位
4. **小批量測試（10 筆）**
   - 動作：從你 200 張裡隨機挑 10 張不同供應商的發票放 `/files/pdf-inbox/`，再次 Execute Workflow
   - **驗收硬指標**：(a) 10 張全綠燈 (b) 10 張的 newFilename 都符合便箋格式 (c) 沒有 null
5. **全量執行**
   - 動作：把 200 張全放 `/files/pdf-inbox/`，Execute Workflow
   - **驗收硬指標**：(a) 200 張全綠燈 (b) 抽查 5 張隨機 output 命名正確 (c) Finder pdf-renamed 資料夾數量是 200
   - **驗收動線**：在 n8n 工作區右側點「Executions」分頁 → 等所有 200 筆狀態 column 變成「Success」（綠 ✓）。如果有 Failed（紅 ✗）或 Cancelled（灰），把那筆 timestamp 記下回去看哪個節點掛了。**不要等 UI 即時 200 個綠燈，那不是 n8n UI 的設計**（200 筆會在 Executions 分頁輪流刷新，要等 7-8 分鐘全跑完再看 list）。

### 4. 完整範例 prompt 模板

#### 【SOP-B】`[貼 input JSON]` 怎麼擷取（給某一個節點吃進來什麼）

**這是模板 4 用的 — 你要讓 LLM 看到「節點當下實際拿到的資料長什麼樣」**

1. 在 n8n UI 點目標節點（紅燈節點）→ 右側 panel 上方 4 個 tab，切到「**Input**」（不是 Output）
2. Input panel 右上角 toggle 切到「**JSON**」（不是 Schema、不是 Table）
3. 看 input items 數量（panel 上方會顯示「N items」）：
   - **≤ 3 筆**：全部複製貼上（panel 右上角有 📋 複製按鈕，或 Cmd+A → Cmd+C）
   - **> 3 筆**：只貼前 2 筆（複製整段後手動刪到只剩前 2 筆即可，n8n 也能用左右箭頭切換 item，截前 2 筆 JSON）
4. **複製前手動清掉這 2 類欄位**（避免 token 浪費 + 隱私洩漏）：
   - **binary 欄位**：如果 item 含 `binary` 物件且裡面是 base64 字串（一長串無意義字元），把整個 `binary: { ... }` 物件刪掉（不影響 LLM 判斷邏輯，因為 LLM 沒辦法 decode base64）
   - **credential / token 字串**：搜尋 `apiKey` / `accessToken` / `Bearer ` / `__GEMINI_API_KEY__`，找到就刪掉那一行（對應第 5 章紅線 9）
5. **驗收**：貼之前自己讀一遍，要看得懂這是該節點吃進來的 JSON 結構（鍵名 + 範例值），看不懂就回 step 1 重做

#### 【SOP-C】`[貼 parameter JSON]` 怎麼擷取（節點本身的設定）

**這是模板 4 用的 — 你要讓 LLM 看到「該節點當前 parameter 是什麼設定」**

1. 在 n8n UI **點該節點本身**（不是它旁邊空白） → 編輯器會跳出該節點的設定 panel
2. **方法 1（最簡單）**：直接在節點上 Cmd+C（n8n 會把整個節點 JSON 包含 parameter + name + type 都複製到剪貼簿）→ Cmd+V 到 LLM 對話框
3. **方法 2（只要 parameter 區段）**：節點 panel 右上 ⋮ → 「Copy parameters」（部分版本叫 Copy node settings） → 取得只有 `parameters: {...}` 那段
4. **複製到的 JSON 開頭應該是** `{ "parameters": ...` 或 `{ "name": "Code: AI 改名", "type": "...", "parameters": ...`
5. **貼之前最後檢查**：搜尋 `apiKey` / `accessToken` / `password` / `token` 4 個字串
   - 全找不到 = OK
   - 任一個搜得到 = 你的 credential 寫死在 parameter 裡（不該這樣，但 LLM 改節點時偶爾會搬進去）→ 手動把那行 value 改成 `"REDACTED"` 再貼
6. **token 量**：單一節點 parameter 通常 200-2000 tokens，3 個節點貼一起也吃得下

#### 【SOP-D】`[貼 output JSON]` 怎麼擷取（節點跑完輸出什麼）

**進階用 — 模板 4 沒用到，但若 LLM 反問「跑出來是什麼」時用**

1. 點該節點 → Output tab → JSON 模式（不是 Schema/Table）
2. 看狀態：
   - 該節點**綠燈但 output 怪**（有 null、空字串、值不對）→ 直接複製當前 output（這是「跑成功但結果不對」的情境，最常見）
   - 該節點**紅燈** → output 那格會顯示 error 物件而不是真正資料；複製這段 error JSON + 同時走 SOP-B 拿 input
3. 同 SOP-B 步驟 3-4 處理 binary 與 credential 清理

錯誤訊息（紅燈節點底下的紅字全文）：
[貼錯誤訊息]

該節點 input（從 Input panel 複製的 JSON）：
[貼 input JSON]    ← 走 SOP-B 擷取（前 2 筆、刪 binary、刪 credential）

該節點當前 parameter（從 Settings 複製的 JSON，已移除 credential）：
[貼 parameter JSON]    ← 走 SOP-C 擷取（搜尋 apiKey 等 4 字串確認都找不到）

#### 模板 4-延伸（LLM 反問要更多資訊時）

```
你需要看上下游節點才能診斷？我可以提供，但請告訴我：
- 你想看的是「Read PDF 節點的 output」還是「下游 Write 節點的 parameter」？
- 為什麼這個錯誤需要看那個節點？（一句話）

我只貼那一個節點，不貼整份 workflow。
```

#### 模板 5（步驟 6 收斂止血，當 LLM 越改越複雜）

```
請用一句話比較你最早給我的版本與現在這版：
- 哪些是「為了我原始需求必要的」？
- 哪些是「你新增的防禦邏輯」（try/catch、if 條件、error handler、新節點）？

請保留必要的、移除新增的，再給我一次「只有 Code 節點」的 JSON 區段。
```

### 5. 範例 LLM 回應

#### 步驟 5 模板 4 sample 回應（「精準」型 — 過關）

```
問題在 parameter 的 jsCode 第 3 行的 prompt 字串。

第 3 行目前寫：
  const prompt = `請從以下發票文字抽取：供應商名稱、稅後金額、發票日期...`;

input panel 顯示 pdfTextPreview 的內容是「賣方：XX 公司，稅後合計：1234 元」
—— 發票上實際的中文是「賣方」（不是「供應商」）和「稅後合計」（不是「稅後金額」）。

最小修改：第 3 行的「供應商名稱」改成「賣方公司全名」，
        「稅後金額」改成「稅後合計」。
其他不動。
```

#### 步驟 5 sample 回應（「太模糊」型 — 要追問）

##### 變體 X：純機率判斷（最常見）

```
這個錯誤訊息看起來是 Gemini API 沒有正確回傳預期格式。
建議檢查 Code 節點的 prompt 寫法，可能要重新調整提示詞，
或者考慮加上錯誤處理機制讓流程更穩定。
```

- **辨認訊號**：句子裡出現「可能」「建議」「看起來」「或許」「考慮加上」 + 沒指出具體欄位名 + 沒給具體新值
- **學員下一步**：追問「請告訴我具體是 prompt 的哪一行需要改、改成什麼新字串。不要建議『調整提示詞』，給我精確的修改。」

##### 變體 Y：列多個可能根因（請學員自己挑）

```
這個錯誤可能的原因有以下幾個：
1. prompt 不夠精確，Gemini 沒抽出正確欄位
2. PDF 解析可能有問題，pdfTextPreview 內容不完整
3. JSON.parse 失敗，可能是回傳格式問題
4. fallback 邏輯沒處理某些邊界情況

不排除是綜合因素。建議逐一排除。
```

- **辨認訊號**：「可能的原因有以下幾個」+ 條列 3 條以上 + 結尾「不排除」「綜合因素」「建議逐一排除」
- **為什麼這對學員不友善**：你不是工程師，沒有時間也沒有能力「逐一排除」
- **學員下一步**：追問「請看『錯誤訊息全文 + input panel』兩件事，**只挑 1 個最可能的根因**告訴我，並指出 parameter 的哪一行要改成什麼。如果你不確定，請告訴我『需要再給你哪個面板才能判斷』，不要列 4 個可能性讓我自己挑。」 → 這句逼 LLM 收斂到單一根因或明確要求補資訊

##### 變體 Z：LLM 反問要更多面板（合理反應，但要回得對）

```
我需要再看一下：
1. Read PDF 節點的 output 是否真的有 pdfTextPreview 欄位？
2. Extract PDF Text 節點的 parameter 設定是什麼？
不然光看 Code 節點的 input 我無法判斷是上游沒給對，還是 Code 節點本身寫錯。
```

- **辨認訊號**：LLM 不亂猜、明確問你「需要看 X 節點的 Y 面板」
- **這算好回應**，但你要按模板 4-延伸只給單一節點面板（不貼整份 workflow）
- **學員下一步**：用模板 4-延伸的格式回，問 LLM「為什麼需要看這個 — 一句話」並只貼那 1 個節點對應的面板

#### 步驟 6 模板 5 sample 回應

```
比較第 1 版和第 3 版：

必要的（保留）：
- prompt 字串的欄位名稱修正（賣方/稅後合計）
- 檔名 template 的格式

新增的防禦邏輯（移除）：
- try/catch 包住整個 API call
- if 條件跳過稅後合計為 0 的單據
- 額外的 error log 寫入

清理後的版本（只有 Code 節點區段）：
[JSON]
```

### 6. 驗收標準

學員整章讀完 + 走完一次過關：

- [ ] 我能在 n8n UI 上 verify -edit workflow（節點數對、credentials 跟上、Active OFF）— Duplicate 動作已在第 3 章 step 6 做過
- [ ] 我能複製錯誤訊息+input+parameter 三件套（並確認 4 個 credential 字串都搜不到）
- [ ] 我能分辨 LLM 回的「精準回答」vs「太模糊」（具體判定例）：
  - ✅ 「精準」= 句子裡有「parameter 的第 X 行」+「具體欄位名」+「具體新值」三要素
  - ❌ 「太模糊」= 出現「可能 / 建議 / 看起來 / 重新檢查 / 調整」這類詞 + 沒指具體行號
  - 我看到「太模糊」會用模板「請告訴我具體是哪一行的哪個欄位字串要改成什麼」追問
- [ ] 我能在 LLM 提案加 try/catch 時用模板 5 拒絕（看到「為了穩定加上 try/catch」「加 if 條件處理空值」這類句子立刻拒）
- [ ] 10 張小批量跑通才放 200 張
- [ ] 全量跑完後 Finder pdf-renamed 資料夾數量正確（200 張 → Finder 顯示 200，差 1 張都要追查）

### 7. 常見錯誤 + 怎麼解

**錯誤 1：「我點 Execute Workflow 直接在 production workflow 上跑了」**
- 解：立刻關掉 Active 開關。檢查 production 是否已被新版本覆蓋（n8n UI 的 Executions tab 看歷史）。下次嚴格用 -edit 複本

**錯誤 2：「我貼模板 4，LLM 又給我整份 workflow 重寫」**
- 解：用模板 4 結尾「不要重寫整個節點」是關鍵。如果 LLM 還是重寫，加一句：「我只要『修改第 X 行的 Y 字串為 Z』這種句型的回答，不要 JSON」

**錯誤 3：「LLM 第二輪建議我加 try/catch，我拒絕了，第三輪它又建議加 if 條件」**
- 解：用模板 5 強制收斂。這是 LLM 的「helpful 病」（總想做更多），明確拒絕後通常會收斂。如果第 4 輪還在加，停 — 換一個新對話 session 重啟

**錯誤 4：「我跑了 200 張，發現有 3 張改名失敗（檔名出現 null）」**
- 解：(a) 檢查那 3 張的原始 PDF 是不是格式異常（例：掃描品質差、欄位名不同） (b) 不要改 workflow，改去人工處理那 3 張（80/20 原則） (c) 在 workflow 描述欄記下「3 張邊界 case 已人工處理：[檔名]」

**錯誤 5：「全量跑完，Finder 顯示 195 張不是 200」**
- 解：(a) 打開 n8n UI 的 Executions tab 看「Failed」筆數 (b) 點失敗的 execution 看是哪幾筆 PDF (c) 對應到 pdf-inbox 找原檔，人工處理 / 二次跑
