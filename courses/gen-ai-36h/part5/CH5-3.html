AI × 自動化：3 種典型工作流

Make / n8n 工具學會了，接下來是「實際要做什麼？」本節教上班族最有價值的 3 種 AI 自動化場景：表單 AI 分類分派、會議錄音自動歸檔、客訴嚴重度升級。每個都給完整 workflow 設計 + AI prompt + 踩坑提醒。

(01) · 通用架構10 分鐘

所有 AI 自動化流程都是這 5 段

不管場景多複雜，把 AI 塞進自動化流程的結構都一樣——先觸發、準備資料、丟給 AI、依結果分流、執行動作。記住這 5 段，你就能設計任何 AI 自動化。

核心觀念 · AI 在自動化裡的角色
AI 不是決策者，是「會判斷的過濾器」。它的工作是：讀一段非結構化文字（email、客訴、會議紀錄），產出結構化標籤（JSON 格式的分類 + 摘要）。後續節點依據這些標籤走不同路徑。關鍵：AI 回傳的格式必須讓後續節點能解析——所以 prompt 裡一定要指定「回傳 JSON」。
真實情境 · 為什麼要用 AI 而不是關鍵字
用 Filter + 關鍵字（如「主旨含報價」）只能處理最簡單的場景。但客戶寫的信千變萬化：有人寫「想了解費用」、有人寫「請問方案」、有人寫「how much」——關鍵字永遠列不完。AI 的優勢是「理解語意」——不管客戶怎麼寫，AI 都能判斷這是「詢價」。AI = 一個比 Filter 聰明 100 倍的分流器。

# 角色 你是客服分類助理。 # 任務 以下是客戶透過官網「聯絡我們」表單送來的訊息。 請分類並產出結構化摘要。 # 表單內容 姓名：{{ name }} Email：{{ email }} 主旨：{{ subject }} 內容：{{ message }} # 分類規則 只能分為以下 4 類之一： - inquiry（詢價 / 想了解方案） - complaint（客訴 / 不滿 / 要退款） - partnership（合作邀約 / 媒體） - other（求職 / 無關 / 垃圾） # 輸出格式（嚴格 JSON） { "category": "inquiry|complaint|partnership|other", "confidence": 0.0-1.0, "summary": "一句話摘要（30字內）", "urgency": "high|medium|low", "suggested_action": "建議的下一步" } # 規則 - 只回傳 JSON，不要加任何說明文字 - confidence 低於 0.7 的設 urgency 為 "medium" 讓人工複查 - complaint 一律 urgency "high"
真實情境 · 軟體公司 PM 阿信的故事
阿信每天花 30 分鐘手動把官網表單訊息分到 4 個 Slack channel。建了這條 workflow 後，表單送出 10 秒內自動分類完畢。一個月後回頭看 Sheet 統計：77% 是 inquiry、15% 是 complaint、8% 其他——這個數據讓行銷部改了官網 FAQ，complaint 降了 40%。自動化不只省時間，還產出數據洞察。

真實情境 · 新創公司 HR 主管 Jenny
Jenny 每天有 3-4 場會議。以前她帶筆電手動記，記完還要整理 30 分鐘。現在她用手機錄音，會後把錄音檔丟到 Drive 的「會議錄音」資料夾——5 分鐘後 AI 摘要出現在 Google Drive，email 也自動寄出。一天省 1.5 小時，一個月省 30+ 小時。

真實情境 · 電商客服主管 David
David 管理 3 個客服人員，每天處理 120+ 封信。過去有一次 severity 5 的客訴（客戶在 PTT 發文）被埋在信堆裡 2 天才發現，差點變公關危機。建了這條 workflow 後，severity 4-5 的信在 30 秒內就通知到他手機。不是「加速處理」，而是「不漏接危機」。
(05) · 設計原則5 分鐘

5 個通用設計原則

原則 1
AI 輸出一定要 JSON
讓後續節點能 parse。prompt 裡明寫「只回傳 JSON，不要加說明文字」。加上範例 JSON 讓 AI 知道格式。
原則 2
類別要互斥、有限
分類不超過 5-7 類。類別名稱用英文（Router 比對方便）。每類定義清楚，不要讓 AI 猶豫。
原則 3
永遠有 fallback 路徑
AI 判斷不出來時（confidence 低、category 是 other），要有一條「人工處理」路徑。不要讓資料消失。
原則 4
全部記到 Sheet
每筆資料不管走哪條路徑，都記錄到 Google Sheet（含 AI 的原始回傳）。日後分析、除錯、改 prompt 都需要。
原則 5
先跑 10 筆測試
上線前手動丟 10 筆不同類型的測試資料。確認每條路徑至少跑過一次。特別測邊界（如同時像 inquiry 又像 complaint 的信）。
(06) · 踩坑博物館5 分鐘

AI 自動化最常踩的 6 個坑

坑 1AI 回傳不是純 JSON
症狀：AI 回覆 ```json {...} ``` 或前面加了「以下是分析結果：」，JSON Parse 失敗。
修：prompt 裡加 「只回傳 JSON，不要用 markdown code block，不要加任何說明文字」。或在 Code 節點做 regex strip。
坑 2分類結果跟路徑對不上
症狀：AI 回傳 "Inquiry"（大寫），但 Router 設的是 "inquiry"（小寫），沒進對的路徑。
修：prompt 裡明確指定大小寫 「category 必須是小寫英文」。或在比對時用 .toLowerCase()。
坑 3AI 偶爾幻覺
症狀：客戶只是問「怎麼修改帳號」，AI 判成 severity 4（嚴重客訴）。
修：加 confidence 欄位。confidence < 0.7 的走人工複查。同時定期回顧 Sheet 裡的分類結果調整 prompt。
坑 4忘記處理 API 錯誤
症狀：OpenAI / Claude API 偶爾 429 / 500，整條 workflow 卡住。
修：Make: 開 Error Handler → Resume → 用備用路徑（如直接通知人工）。n8n: Error Trigger workflow 或 Try/Catch 節點。
坑 5Token 超長文本
症狀：會議逐字稿 20,000 字，超過 API token 限制，回傳 400 錯誤。
修：先用 Code 節點把文本切成 8,000 字一段，分段摘要後再合併。或用 Claude 3.5（200K context）直接吃。
坑 6Whisper 轉出亂碼
症狀：中文會議錄音轉出英文或亂碼。
修：Whisper API 呼叫時加 language: "zh" 參數。錄音品質也很重要——外接麥克風 > 手機麥克風 > 筆電麥克風。
(07) · 課堂動手10 分鐘

Guided Practice：設計你的 AI 工作流

PRACTICE 01 · 改寫 Workflow 01 prompt
把分類改成你公司的實際類別
題目：複製 Workflow 01 的 AI 分類 prompt，把 4 個分類改成你公司/部門實際需要的類別（如：技術支援、帳務問題、退換貨、功能建議）。手動丟 3 筆不同類型的測試文本到 Claude/ChatGPT，確認分類結果正確。
提示：類別名用英文（如 tech_support, billing, return, feature_request）方便後續 Router 比對。每個類別加 1 句定義讓 AI 不會混淆。
PRACTICE 02 · 設計你的場景
用 5 段架構設計一條自己的 AI 工作流
題目：想一個你工作中「每天重複、需要判斷、耗時費力」的流程。用紙筆畫出 5 段架構：① Trigger 是什麼事件 ② 資料長什麼樣 ③ AI 要判斷什麼 ④ 判斷後走哪些路徑 ⑤ 每條路徑做什麼動作。
提示：不需要真的建在 Make/n8n——先用紙筆或 FigJam 畫出流程。PRAC5 會帶你把這個設計實作出來。
常見場景參考
① 應徵信 → AI 篩選 → 合格轉 HR / 不合格自動回覆
② 社群留言 → AI 情緒分析 → 負面即時通知 / 正面收集
③ 內部簽核 → AI 風險評估 → 低風險自動核准 / 高風險轉主管
④ 日報 email → AI 摘要 → 每日彙整寄給老闆
PRACTICE 03 · 邊界測試
故意丟「模糊不清」的輸入給 AI
題目：用 Workflow 01 的 prompt，丟一筆故意模糊的訊息（如「我對你們的服務有些想法，不知道你們有沒有在找合作夥伴，另外費用大概多少」——同時像 inquiry + partnership + other）。觀察 AI 怎麼分、confidence 多少。
提示：這就是為什麼需要 confidence 欄位和 fallback 路徑。模糊輸入是真實世界的常態，不是例外。
