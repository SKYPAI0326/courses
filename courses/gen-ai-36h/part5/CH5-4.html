雙引擎協作：
何時用 Make、何時用 n8n

兩個工具都會了，問題是「實際上哪個放哪邊？」本節給你 6 個選擇準則、3 個實戰場景對照，以及怎麼把 Make 與 n8n 串接成一條完整工作流。

SECTION 02 · 場景對照

6 個常見場景，該用誰？

官網聯絡表單 → AI 分類 → Slack 通知
Make
原因：對外、要 24/7、量不大（一天 10 筆以內）、所有服務都在雲端。雲端自動化最直觀，也不怕你下班電腦關機漏單。
每日從公司內部系統（資料庫 / ERP）抓資料做報表
n8n
原因：機密資料不應送到 Make 雲端；連接內部系統需要在內網；可能要 SQL 查詢。n8n 跑在公司主機上就能直接接。
LINE Bot（用戶訊息 → AI 回覆）
Make
原因：要 24/7 回應（半夜也有人傳訊息）、輕量、用戶數不多時額度夠。若 LINE 用戶破萬，op 量暴增則建議轉 n8n 自架。
每月整理本機 Excel 檔 → 生成 PDF 報表
n8n
原因：本機檔案 Make 讀不到；檔案可能幾十 MB；要做較複雜的運算。n8n 本機跑、Read Binary Files 節點直接讀。
Google Form 送出 → AI 處理 → 同時存 Notion + Sheet + 寄信
Make
原因：全部都是雲端服務、Make 的 3 個模組並行執行很直觀、Router 分流好設定、不需要寫程式。
客戶資料庫批次處理（1,000 筆以上）+ AI 打標籤
n8n
原因：1,000 筆 × 幾個模組 = 幾千 ops，Make 免費版秒爆、付費版也花錢。n8n 本機無限制跑，半夜跑完早上來看結果。
Stripe 付款成功 → 寄發票 → 開通帳號 → 登記 CRM
混合
原因：Stripe webhook 用 Make 接（雲端穩）→ Make 呼叫 n8n webhook（傳訂單資訊）→ n8n 處理內部系統開帳號 → 完成後回 Make 繼續寄發票。
SECTION 03 · 串接架構

讓 Make 與 n8n 互相呼叫

真正進階的用法是「兩者串接」——Make 當前台（接雲端事件、給別人看的）、n8n 當後台（處理本地複雜邏輯）。兩者間用 Webhook 溝通。

付款成功
→
Make

→
n8n
本地 DB 開帳號
→
Make
寄發票 + Slack
# Make 呼叫 n8n 的方式 // Make 裡用 HTTP 模組 Method: POST URL: https://your-cloudflare-tunnel.trycloudflare.com/webhook/xxx Body type: Raw (JSON) Body: { "order_id": "{{ order_id }}", "customer_email": "{{ email }}", "amount": {{ amount }} } // n8n 收到後用 Webhook 節點處理，最後用 Respond to Webhook 回 200
# n8n 呼叫 Make 的方式 // Make 裡開一個「Custom webhook」模組 // Make 會產出一個 URL，例如： https://hook.eu1.make.com/abc123 // n8n 裡用 HTTP Request 節點 Method: POST URL: https://hook.eu1.make.com/abc123 Body: JSON { "status": "done", "result": {{ $json.result }} }
⚡
重要設計：兩邊用同一個「Request ID」串起來，方便除錯時追蹤一筆流程。Make 生成 UUID → 傳給 n8n → n8n 處理完回傳 → Make 記錄。出錯時看 ID 就知道斷在哪。

小心「ops 漏水」：Make 有個常見地雷——polling 型 trigger（每 15 分檢查）就算沒新資料也吃 ops。單一 scenario 每天 96 次 × 30 天 = 2,880 ops。若你啟 5 個這種，免費版直接破表。改用「即時觸發」（Webhook / Instant trigger）可大幅省 ops。
SECTION 05 · 本節行動

課後練習：為你的需求畫雙引擎地圖

列出你目前想到的所有自動化需求（5-10 項），按 Section 1 的 6 個準則逐一歸類。你會得到一張「我的自動化地圖」，清楚知道哪些該放 Make、哪些該放 n8n、哪些需要兩者協作。

挑一個「需要混合」的場景，實際跑一次 Make → n8n 的串接——這是最能展現自動化威力的實戰。
