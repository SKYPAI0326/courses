6.3

GitHub + Railway：備援上架路線

帳號不符合免費 Starter Tier、或需要第 3 個以上的 app 時，把程式碼推上 GitHub、用 Railway 開一條自己的部署線。

學完這節，你會有
判斷自己是不是真的需要這條備援路線的三個依據
從 Sync to GitHub 到 Railway 拿到公開網址的完整流程
四種上架方式的誠實成本比較，知道什麼時候該用哪一種
(01)

情境：這是備援路線，不是必經之路

上一節的 Publish 是內建路線——一個按鈕，AI Studio 幫你把 app 放到公開網址，符合 Starter Tier 資格的話完全免費。但有三種情況，內建路線不夠用：

公司帳號看不到免費層
用公司的 Google Workspace 帳號，因為帳單設定或組織政策，Starter Tier 選項沒有出現在畫面上。
已用完 2 個免費名額
Starter Tier 免費 Publish 名額有限，前面章節已經上架過幾個工具，第 3 個以上的 app 需要別的地方放。
想要版本控制與備份
工具會持續修改，想要有 commit 紀錄可以回溯，或想把程式碼備份在自己的 GitHub 帳號下。
ℹ
符合資格就用內建的就好。如果你的帳號可以看到 Starter Tier、名額也還沒用完，直接照上一節的 Share／Publish 走就夠了。這一節是給前面三種情況的備援方案，不是每個人都需要走完。
(02)

步驟 1：把程式碼推上GitHub

在 AI Studio 的 Build 編輯器裡，設定面板有一個「GitHub」分頁，這就是「Sync to GitHub」功能。第一次點擊會走 GitHub OAuth 授權——同意讓 AI Studio 幫你建立 repo。授權完成後，AI Studio 會為這個 app 自動建立一個專屬的 repo（main 分支），程式碼就在裡面了。

之後每次你在 Build 裡改了東西，回到這個 GitHub 分頁，它會自動草擬一段 commit 訊息、列出這次變更了哪些檔案。確認沒問題後按「Stage and commit all changes」，就會同步推上去。面板上也有解除連結與直接開啟 repo 網址的按鈕。

★
Secrets 不會被推上 GitHub。你在 Secrets 分頁設定的 GEMINI_API_KEY 是安全設計上刻意排除在同步範圍外的。這是好事——代表你的 API key 不會意外出現在公開 repo 裡。但也代表部署到 Railway 之後，你要自己補一次環境變數，下一步會說明。
(03)

步驟 2：Railway 接上GitHub

Railway 是一個能直接讀 GitHub repo、自動建置並跑起來的部署平台。註冊時建議直接用 GitHub 帳號登入——這樣連結的帳號才算「Full Trial」，沒驗證過的帳號是「Limited Trial」，對外連網會受限。

Trial 條件：一次性 $5 credit，30 天內有效，不需要信用卡。限制是 1GB RAM、共享 vCPU、每個專案最多 5 個服務——對一個單機 app 來說綽綽有餘。

01
用 GitHub 帳號登入 railway.app
首次登入會請你授權 Railway 讀取你的 GitHub repo 清單。
02
New Project → Deploy from GitHub repo
從清單裡選出 AI Studio 剛才幫你建的那個 repo。
03
等待自動偵測與建置
Railway 會偵測到這是 React + Vite 專案，自動跑建置流程，不需要手動下指令。
(04)

步驟 3：補上API key

建置完成之後，app 還不會動——因為它需要 GEMINI_API_KEY 才能呼叫 Gemini。到服務的 Variables 分頁，新增一筆 GEMINI_API_KEY，值貼上你在 AI Studio「API keys」頁面產生的那把 key。儲存後 Railway 會自動重新部署一次。

✦
資安鐵則：key 只放在環境變數（Variables）裡，永遠不要把它直接寫進程式碼、更不要 commit 進 GitHub repo。上一步提到 Secrets 本來就不會跟著同步，就是為了避免這件事發生。
(05)

步驟 4：拿到公開網址

最後一步，到服務的 Settings → Networking，點「Generate Domain」，Railway 就會給你一個 *.up.railway.app 的網址。這個網址可以直接分享出去，跟內建 Publish 拿到的網址一樣是公開可存取的。

(06)

結論：能用內建就用內建。Railway 這條路是給名額用完、帳號受限、或需要版本控制的情況，不是預設路線。

(07)

為什麼不能直接用 GitHub Pages

ℹ
GitHub Pages 只能放靜態網頁。AI Studio 產出的 app 是一個 React + TypeScript + Vite 專案（App.tsx、components/、services/geminiService.ts），需要先跑建置流程才能變成可執行的網頁，而且呼叫 Gemini API 的邏輯需要一個能跑後端環境的主機。GitHub Pages 沒有建置流程、也沒有伺服器端運算，放不了這種專案——這正是需要 Railway 這類平台的原因。
