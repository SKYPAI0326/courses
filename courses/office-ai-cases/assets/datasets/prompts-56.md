## 使用說明

- **主課 ①②③④**（P1-x / P2-x）：在任一主流 AI 工具貼上即可跑，不指定平台
- **NotebookLM 專用**（NB-x）：僅 NotebookLM 可用，因為需要 source-grounded 問答功能
- **Bonus**（B1-x / B2-x / B3-x）：跨案例通用的工具型 prompt，搭配主課 prompt 一起用
- 💰 標記 = 需付費版才能完整執行，免費版有限制
- 每個 prompt 用 ` ``` ` 包起來，可直接整段複製貼上

## 案例 ① 週報生成（7 套 · CH1-1）

**素材**：`work-log-5day.txt`（5 天 × 10 條 = 50 條工作日誌）
**案例主角**：小李（業務副理兼財務代理）

---

### P1-1-1 · 基礎版週報

**用途**：把 50 條工作日誌直接變成白話週報，最快速入門用。

**Prompt**：
```
以下是我這週的工作日誌：

[貼上 work-log-5day.txt 全文]

請把這份日誌整理成一份週報，格式如下：
- 本週主要成果（3-5 條）
- 本週雜事（3 條以內）
- 下週待辦（3 條）

每條不超過 30 字，語氣簡潔。
```

**為何這樣寫**：給初次使用 AI 工具的學員，格式清楚、指令少、容易跑出結果，建立信心。

**平台差異備註**：
- 檔案處理：free 版可直接貼文字，不需上傳檔案；50 條日誌約 1,500 字，所有免費版均可處理
- 常見限制：免費版每次對話有訊息上限（ChatGPT free 大約 5 小時 10 則），建議 40 min 以內完成此案例
- 跑不出時：若輸出太籠統，加上 B1-1（防呆追問）要求 AI 引用具體日誌條目

---

### P1-1-2 · 進階版週報（5 骨架完整版）

**用途**：含角色設定 + 5 大骨架，分類更細、可直接交主管的版本。

**Prompt**：
```
【角色】你是一位擅長整理職場成果的商業寫作助理。

【資料】以下是我（小李，業務副理兼財務代理）的一週工作日誌，共 50 條：

[貼上 work-log-5day.txt 全文]

【任務】請整理成一份可以直接寄給主管的週報，分成以下三個區塊：
1. 主要成果：列出 3-5 條有實質進展的事項，每條包含「做了什麼 + 產出或結果」
2. 例行雜事：歸納成 1-2 句，不要一條一條列
3. 下週計畫：列出 3 條最重要的待辦，加上預計完成時間

【輸出格式】純文字，每個區塊有明確標題，整份不超過 200 字，語氣客觀專業。

【驗證要求】請在最後加一行：「本週報引用日誌 X 條，覆蓋日期 YYYY-MM-DD 至 YYYY-MM-DD」
```

**為何這樣寫**：角色設定讓 AI 進入「寫作助理」視角而非「任意回答」模式；驗證要求能立刻揪出 AI 漏看日誌的問題。

**平台差異備註**：
- 檔案處理：貼文字即可；50 條日誌在所有主流免費版 context 內（約 1,500 字遠低於 8K token 下限）
- 常見限制：ChatGPT free 有時會把「下週計畫」寫得太籠統——若這樣，追問 B1-2 要求引用具體日誌項目
- 跑不出時：先跑 P1-1-1 基礎版確認資料可讀，再換本 prompt

---

### P1-1-3 · 變奏版：給主管的精簡版

**用途**：主管看週報時間不超過 1 分鐘，需要「結論先行、數字優先」格式。

**Prompt**：
```
以下是我這週的工作日誌：

[貼上 work-log-5day.txt 全文]

請改寫成「給主管看的精簡週報」，規則：
- 第一行：本週最重要的 1 件成果（一句話，含數字或具體產出）
- 接下來：2-3 條重要事項，每條 20 字以內
- 最後一行：下週最重要的 1 件待辦

整份不超過 100 字。不要開場白，直接進入內容。
```

**為何這樣寫**：強制「不超過 100 字」讓 AI 自動做資訊取捨，訓練學員理解 AI 的優先排序邏輯。

**平台差異備註**：
- 檔案處理：同 P1-1-2，貼文字即可
- 常見限制：AI 有時會超過字數限制——若超過，追問「請嚴格控制在 100 字以內，刪掉最不重要的項目」
- 跑不出時：把字數限制改成「不超過 5 條」，更容易執行

---

### P1-1-4 · 變奏版：給跨部門同事的協作版

**用途**：週報不只給主管，也要讓協作部門知道你做了什麼、需要對方什麼。

**Prompt**：
```
以下是我這週的工作日誌：

[貼上 work-log-5day.txt 全文]

請寫一封「跨部門協作週報」，收件對象是財務部和行政部的同事，格式：
1. 本週我們協作的事項（列出 1-3 條，說明是哪個部門、做了什麼）
2. 目前需要對方協助的事項（1-2 條，說清楚「需要誰、在什麼時間前、做什麼動作」）
3. 下週預計進行的跨部門事項（1 條）

語氣友善、直接，不超過 150 字。
```

**為何這樣寫**：加入「需要誰 + 時間 + 動作」的協助請求結構，讓 AI 輸出能直接用於跨部門溝通，而不是僅做自我彙整。

**平台差異備註**：
- 檔案處理：貼文字即可
- 常見限制：AI 若看不到日誌中有跨部門項目，會自行捏造——先確認日誌確實有跨部門事項（work-log-5day.txt 中有行銷部 / 倉儲 / 會計等協作項目）
- 跑不出時：若 AI 捏造了不在日誌中的協作事項，加 B1-2 追問並要求引用來源

---

### P1-1-5 · 變奏版：自我備忘版（含反思）

**用途**：不給主管、不給同事，純粹給自己做週複盤與下週計畫。

**Prompt**：
```
以下是我這週的工作日誌：

[貼上 work-log-5day.txt 全文]

請幫我做一份「個人週復盤備忘」，包含：
1. 本週做得好的 2 件事（說明為什麼算好）
2. 本週可以做得更好的 1 件事（說明具體改進方向）
3. 下週最重要的 3 件事（依優先序排列）
4. 一句話：本週最大的學習或發現

語氣像在自言自語，不要正式報告口吻，不超過 200 字。
```

**為何這樣寫**：要求 AI「說明為什麼算好」迫使它給出有判斷依據的輸出，而非只是複製日誌。

**平台差異備註**：
- 檔案處理：貼文字即可
- 常見限制：「做得好」「可以更好」需要 AI 有判斷能力，免費版有時給出較空泛的評語——搭配 B1-3 追問不確定點
- 跑不出時：把「說明為什麼算好」改成「舉出日誌中具體哪條事項作為依據」

---

### P1-1-6 · 風險檢查：列出不確定事項

**用途**：寫完週報前，先讓 AI 自報「它不確定什麼」，避免幻覺混入正式報告。

**Prompt**：
```
以下是我這週的工作日誌：

[貼上 work-log-5day.txt 全文]

在整理週報之前，請先做一個「不確定性盤點」：
1. 列出日誌中你**看不懂或資訊不完整**的條目（例：只有時間沒有結果、縮寫不清楚）
2. 列出你在整理週報時需要**假設**才能繼續的地方（例：你怎麼判定某件事是「主要成果」vs「雜事」）
3. 列出日誌中你認為**可能有誤**的資訊（例：時間衝突、事件矛盾）

每條不超過 30 字，不需要幫我解決問題，只需要列出清單。
```

**為何這樣寫**：讓 AI 先自我檢查，比學員事後發現幻覺更有效率；「不需要解決問題」讓輸出保持清單格式，容易快速確認。

**平台差異備註**：
- 檔案處理：貼文字即可
- 常見限制：AI 有時過於自信、回報「沒有不確定事項」——若這樣，改用 B1-3（不確定點追問）強制它說出來
- 跑不出時：分段餵資料，先只貼週一的 10 條，確認 AI 能正確盤點後再貼完整資料

### P1-1-7 · 投影片素材版（給案例 ④ 用）

**用途**：從週報日誌抽出 1 張投影片用的重點摘要素材，給後續 CH2-2 簡報組裝使用。

**Prompt**：
```
以下是我這週的工作日誌：

[貼上 work-log-5day.txt 全文]

請抽出可以放在「本週重點摘要」投影片的內容，格式如下：

投影片標題：[一句話，不超過 20 字]
本週最重要的 3 件事：
- 事項 1：[具體做了什麼 + 成果，不超過 25 字]
- 事項 2：[同上]
- 事項 3：[同上]
下週重點預告（1 條）：[不超過 20 字]

注意：每條必須有「具體成果或數字」，不接受「持續推進中」「積極處理」這類空話。
```

**為何這樣寫**：明確要求「具體成果或數字」並列出禁用空話，訓練 AI 輸出真正能上投影片的素材而非裝飾性文字。

**平台差異備註**：
- 檔案處理：貼文字即可
- 常見限制：AI 有時在「成果」欄位寫「已完成」「正在進行」——若這樣，追問 B1-2 要求引用日誌中具體項目
- 跑不出時：先跑 P1-1-2 取得完整週報，再把週報貼進來，讓 AI 從週報中抽投影片素材（二段式）

## 案例 ② 業務 + 財務雙合（21 套 · CH1-2）

### 子場景 A：行程規劃（7 套）

**素材**：`visit-30.csv`（拜訪 30 筆，欄位：編號 / 日期 / 客戶 / 地區 / 聯絡人 / 職稱 / 拜訪目的 / 狀態 / 下次行動 / 預估成交金額）
**案例主角**：小李（業務副理）

---

### P1-2A-1 · 基礎版行程規劃

**用途**：給 30 筆拜訪記錄，讓 AI 規劃下週行程，最快速入門。

**Prompt**：
```
以下是我的拜訪記錄，共 30 筆：

[貼上 visit-30.csv 全文]

請根據這份資料，規劃我下週（2026-05-18 至 2026-05-22）的拜訪行程，列出每天要拜訪哪些客戶。
每天不超過 3 個拜訪，格式：
- 週一：[客戶名稱]（[地區]）
- 週二：...
（以此類推）

注意：只排已在日期欄顯示為本週的拜訪，不要捏造新行程。
```

**為何這樣寫**：「不要捏造新行程」是關鍵防幻覺指令，讓 AI 只做排程整理而非自行新增拜訪。

**平台差異備註**：
- 檔案處理：30 筆 CSV 約 2,500 字元，可直接貼文字；Gemini、Claude free 版都能處理；ChatGPT free 版直接貼文字無問題，上傳 CSV 功能需看 free 版當前限制
- 常見限制：有些 AI 看到 CSV 格式後會自行推算「下週」的範圍，可能與實際週期不符——prompt 中已明確寫出日期區間
- 跑不出時：把 CSV 表頭單獨貼一行，讓 AI 先確認看得懂欄位，再貼完整資料

---

### P1-2A-2 · 進階版：路線優化 + 優先級（5 骨架）

**用途**：不只排行程，還要考慮路線不走回頭路、優先處理高機會客戶。

**Prompt**：
```
【角色】你是一位有 10 年經驗的業務行程優化顧問，熟悉台灣北中南地理分布。

【資料】以下是業務小李的 30 筆拜訪記錄（CSV 格式）：

[貼上 visit-30.csv 全文]

【任務】規劃 2026-05-18（週一）至 2026-05-22（週五）的最佳拜訪行程，要求：
1. 同一天同地區或鄰近地區的客戶排在一起，減少跨區移動
2. 狀態為「跟進中」的客戶優先排入本週
3. 預估成交金額欄位有值的客戶，依金額大小標記優先級（高 / 中 / 低）

【輸出格式】表格（欄位：日期 / 客戶 / 地區 / 優先級 / 拜訪目的 / 注意事項），每天不超過 3 筆

【驗證要求】請在表格後說明：本週共排入 X 筆拜訪，其中「跟進中」X 筆，跨區移動次數：X 次
```

**為何這樣寫**：驗證要求讓學員能立刻核對 AI 有沒有按規則排，而不是盲目信任輸出結果。

**平台差異備註**：
- 檔案處理：CSV 貼文字；表格輸出在 Markdown 格式下可以直接貼入 Notion / Google Docs
- 常見限制：AI 對地理距離的判斷是基於訓練資料，不是即時路線計算——可接受方向性建議，但實際里程仍需自己確認
- 跑不出時：若 AI 排出的路線不合理，加 B1-1 追問它的排序邏輯，再手動調整

---

### P1-2A-3 · 變奏版：三種排序策略

**用途**：同一份資料，用三種不同優先邏輯排出三種行程，讓學員看到 AI prompt 語句對輸出的影響。

**Prompt（版本 A：依「跟進中」優先）**：
```
以下是我的 30 筆拜訪記錄：

[貼上 visit-30.csv 全文]

請規劃下週（05-18 至 05-22）行程，排序規則：**狀態為「跟進中」的客戶最優先排入**，相同優先級時依日期排序。每天最多 3 筆，輸出格式：日期 / 客戶 / 狀態 / 下次行動。
```

**Prompt（版本 B：依「金額大」優先）**：
```
以下是我的 30 筆拜訪記錄：

[貼上 visit-30.csv 全文]

請規劃下週（05-18 至 05-22）行程，排序規則：**預估成交金額(萬)最大的客戶最優先排入**，沒有金額的客戶排後面。每天最多 3 筆，輸出格式：日期 / 客戶 / 預估成交金額(萬)/ 拜訪目的。
```

**Prompt（版本 C：依地區批次）**：
```
以下是我的 30 筆拜訪記錄：

[貼上 visit-30.csv 全文]

請規劃下週（05-18 至 05-22）行程，排序規則：**同一天只排同一地區的客戶**，優先填滿北部，再填中部，再填南部，每天最多 3 筆。輸出格式：日期 / 地區 / 客戶名稱 / 拜訪目的。
```

**為何這樣寫**：同樣資料、不同排序規則 = 不同行程，讓學員直接看見 prompt 語句如何改變 AI 輸出，是「Prompt 有效性」最直觀的示範。

**平台差異備註**：
- 檔案處理：同 P1-2A-1
- 常見限制：三個版本建議依序在同一對話中跑，比較三種輸出的差異；注意 ChatGPT free 訊息數限制
- 跑不出時：先跑版本 A 確認資料可讀，再換版本 B / C

### P1-2A-4 · 變奏版：30 秒開場白素材

**用途**：為每筆預定拜訪，請 AI 生成個人化的「30 秒開場白」話術素材。

**Prompt**：
```
以下是我下週要拜訪的 5 個客戶（從拜訪記錄中選出）：

| 客戶 | 地區 | 聯絡人 | 職稱 | 拜訪目的 | 狀態 |
|------|------|--------|------|----------|------|
[從完整的 visit-30.csv（30 筆）中挑出最關鍵 5 筆（依預估金額或跟進狀態），貼上資料]

請為每位客戶生成一段「30 秒開場白」話術，要求：
- 提到對方公司名稱和聯絡人姓氏
- 說明拜訪目的
- 針對狀態（例：跟進中 = 承接上次話題；已成交 = 感謝並開啟下一步）調整語氣
- 每段不超過 60 字

格式：[客戶名稱] — [開場白]
```

**為何這樣寫**：讓 AI 根據「狀態」欄位自動調整話術，示範 CSV 欄位資料可以直接成為 prompt 的條件邏輯。

**平台差異備註**：
- 檔案處理：建議只貼 5 筆，不要整份 30 筆——目的是清晰示範，不是批量處理
- 常見限制：AI 可能生成的話術過於「銷售腔」，可加「語氣自然、不要過於推銷」調整
- 跑不出時：把表格改成條列式，有些 AI 對 Markdown 表格的讀取不如條列式穩定

---

### P1-2A-5 · 變奏版：3 個關鍵問題

**用途**：為每筆拜訪生成「3 個最值得問的問題」，提升拜訪品質。

**Prompt**：
```
以下是我下週要拜訪的 5 個客戶資料：

| 客戶 | 拜訪目的 | 狀態 | 下次行動 |
|------|----------|------|----------|
[從完整的 visit-30.csv（30 筆）中挑出最關鍵 5 筆（依預估金額或跟進狀態），貼上資料]

請為每位客戶生成「3 個最值得問的關鍵問題」，要求：
- 問題必須與拜訪目的和客戶狀態相關
- 問題要具體、有指向性，不要泛問（例：「您覺得滿意嗎？」不是好問題）
- 狀態為「跟進中」：問推進決策的問題
- 狀態為「已成交」：問深化合作或交叉銷售的問題

### P1-2C-2 · 進階版：5 骨架 + 明確異常規則（5 骨架）

**用途**：給 AI 明確的異常定義規則，讓偵測結果更一致、更可重複。

**Prompt**：
```
【角色】你是一位擅長財務稽核的分析師，專門偵測費用報帳的異常模式。

【資料】以下是公司 2026-04 至 05 月的費用申請記錄，共 200 筆（CSV 格式）：

[貼上 expense-200.csv 全文]

【任務】依照以下 7 類異常規則偵測費用異常：
1. **大額異常**：同分類費用中，金額超過該分類平均值 3 倍以上（例：差旅類均 8K，某筆 38 萬）
2. **極小金額異常**：單筆金額 < NT$10（疑誤打、缺零）
3. **分類矛盾**：項目描述與分類欄位明顯不符（例：「印表機」分類為「文具」、「計程車」分類為「水電」）
4. **整數異常**：金額為過度整數（如 99,999 / 49,999 / 999）疑湊整逃審核
5. **重複報帳**：同申請人 14 天內同事項 / 同金額 / 同分類二度申請
6. **時間異常**：週末（週六日）或國定假日申請「客戶餐敘 / 員工活動」等業務型項目（請參考備註欄判定）
7. **備註異常**：備註欄含「異常」「重複」「疑」「需確認」等關鍵字、或備註說明與項目矛盾

【輸出格式】異常清單（Markdown 表格）：異常編號 / 異常類型 / 日期 / 申請人 / 金額 / 異常說明 / 規則代號

【驗證要求】請在最後說明：你分析的總筆數、每種異常類型各找到幾筆、你計算「分類平均值」的方式（用了哪些欄位、扣除哪些 outlier）
```

**為何這樣寫**：明確的 7 類規則 + 數值門檻讓 AI 從「模糊判斷」進入「規則執行」模式，輸出更可信。本檔 dataset 埋有 7 類各 1 筆異常（共 7 筆），備註欄已標「異常」幫助驗證。

**平台差異備註**：
- 檔案處理：200 筆建議上傳 CSV；分段貼文字時，驗證要求的「總筆數」更重要
- 常見限制：AI 對「3 倍以上」的計算在文字模式下常出錯——一定要跑 B1-1 或 P1-2C-6 確認計算方式
- 跑不出時：把 7 類規則拆成多個 prompt，每次只偵測 1 種異常類型

### P1-2C-3 · 變奏版：找重複報帳

**用途**：專門偵測同一費用被重複申請的情況（最常見的費用異常）。

**Prompt**：
```
以下是公司的費用申請記錄（200 筆）：

[貼上 expense-200.csv 全文]

請專門找「重複報帳」的嫌疑：定義為「同一申請人，在 15 天內，申請了金額相差不超過 10% 且分類相同的兩筆以上費用」。

請列出所有符合此定義的費用組合，格式：
重複組合 [N]：
- 第 1 筆：[編號] [日期] [申請人] [分類] [金額]
- 第 2 筆：[編號] [日期] [申請人] [分類] [金額]
- 相差金額：X 元（X%）
- 間隔天數：X 天

最後：共找到 X 組疑似重複報帳。
```

**為何這樣寫**：給出精確定義（15 天內 / 金額差 10% 以內 / 同分類）讓偵測邏輯可重現，學員回去可以調整數字適應自己公司的規則。

**平台差異備註**：
- 檔案處理：同 P1-2C-1，200 筆建議分段或上傳
- 常見限制：日期差計算在文字模式下容易出錯——找到結果後一定要手動確認日期差
- 跑不出時：把條件放寬（改成「30 天內」「金額差 20% 以內」），確認 AI 能執行後再收緊

### P1-2C-4 · 變奏版：找超預算費用

**用途**：給定月均費用基準，讓 AI 找出超預算的申請人或部門。

**Prompt**：
```
以下是公司 2026-04 至 05 月費用申請記錄（200 筆）：

[貼上 expense-200.csv 全文]

部門月均費用基準（請用這個做參考）：
- 交通費：以申請人個人加總對比基準
- 餐費：每人每月 3,000 元
- 住宿：每人每月 8,000 元
- 文具 / 耗材：每人每月 2,000 元
- 水電：部門整體每月 15,000 元
- 其他：每人每月 1,000 元

請找出「超過月均基準 1.5 倍以上」的申請人或部門，格式：
[部門 / 申請人] — [分類] — 實際花費 X 元（基準 X 元，超出 X 倍）
```

**為何這樣寫**：學員可以把「月均基準」換成自己公司的實際預算數字，這個 prompt 是最直接可帶走回辦公室用的版本。

**平台差異備註**：
- 檔案處理：同 P1-2C-1
- 常見限制：AI 計算「每人」的費用需要先確定各部門人數，但 CSV 沒有人數欄——AI 可能直接用金額加總對比，而非「每人平均」；若這樣，告知 AI「直接用部門總額對比，不要推算每人平均」
- 跑不出時：只問一個分類（例如只看交通費），確認計算邏輯正確後再擴展到其他分類

### P1-2C-5 · 變奏版：找分類矛盾

**用途**：找出費用項目描述和分類欄位不一致的記錄（常見的人工輸入錯誤）。

**Prompt**：
```
以下是公司的費用申請記錄（200 筆）：

[貼上 expense-200.csv 全文]

請找出「項目描述與分類欄位明顯不符」的記錄，例如：
- 項目是「高鐵票」但分類是「文具」
- 項目是「辦公室清潔」但分類是「水電」
- 項目是「餐廳收據」但分類是「交通」

請列出所有你發現的分類矛盾，格式：
[編號] [申請人] — 項目「[項目名稱]」被分類為「[分類]」— 建議分類：[正確分類]

```

**為何這樣寫**：「可能的輸入錯誤 vs 可能故意錯分」讓 AI 做初步的嚴重程度判斷，幫學員決定哪些要優先追查。

**平台差異備註**：
- 檔案處理：同 P1-2C-1
- 常見限制：AI 對「明顯不符」的判斷有主觀成分——最後那句區分輸入錯誤 vs 故意錯分，AI 通常只能猜測，要告知主管「這是 AI 的初步判斷，需要進一步查詢憑證」
- 跑不出時：只要求找「交通類費用被錯分」這一種，確認 AI 能執行後再擴展

### P1-2C-6 · 風險檢查：列出計算規則和樣本筆數

**用途**：費用異常分析完成後，讓 AI 自報它的判定規則和分析範圍，是財務稽核最重要的可信度驗證。

**Prompt**：
```
你剛才完成了費用異常偵測分析，請做「計算規則透明度報告」：

1. 你實際分析的總筆數（應為 200 筆，若不足請說明原因）
2. 你定義「異常」時使用的每一條規則（盡可能精確，例如「金額 > 分類平均值 × 3」）
3. 你計算分類平均值時，是用全部 200 筆的加總除以筆數，還是有其他計算方式？
4. 你判定「重複報帳」或「分類矛盾」時，是依照規則嚴格執行，還是有用到你自己的「常識判斷」？

這份報告用於稽核底稿，請如實回報，不要簡化。
```

**為何這樣寫**：「用於稽核底稿」讓 AI 切換到「精確報告」模式；問題 4 的「常識判斷」是讓學員理解 AI 並非純規則引擎的關鍵教學點。

**平台差異備註**：
- 檔案處理：接在前面的分析後使用，不需再貼 CSV
- 常見限制：AI 有時無法精確回報「實際分析筆數」——這本身就是一個重要的討論點（AI 的 context window vs 精確計算的差異）
- 跑不出時：B1-1 是精簡版的追問工具，可在任一分析後快速追問

### P1-2C-7 · 投影片素材：異常清單

**Prompt**：
```
以下是剛才費用異常偵測的分析結果：

[貼上 P1-2C-2 或 P1-2C-1 的輸出結果]

請整理成「費用異常清單」投影片素材，格式：

投影片標題：費用異常偵測報告（2026-04）
數字摘要：
- 分析總筆數：200 筆
- 發現異常：X 筆（X% 異常率）
- 最高風險異常：[描述最嚴重的 1 條，含金額]
異常清單（最多 5 條，依嚴重程度排列）：
| 類型 | 編號 | 申請人 | 金額 | 說明 |

**為何這樣寫**：「最多 5 條，依嚴重程度排列」讓投影片保持精簡；「不要寫空話」的建議後續動作讓報告有行動導向。

**平台差異備註**：
- 檔案處理：貼上前步驟分析結果即可，不需再貼原始 CSV
- 常見限制：若前步驟找到超過 5 條異常，AI 選擇哪 5 條可能不是最重要的——可在 prompt 中加上「優先選金額最大或類型最嚴重的」
- 跑不出時：讓 AI 先列出所有異常並標記嚴重程度，再問它選出前 5 條

## 案例 ③ 電商銷售分析（7 套 · CH2-1）

**素材**：`ecom-orders-500.csv`（訂單 500 筆，欄位：訂單號 / 日期 / SKU / 商品名稱 / 數量 / 單價 / 金額 / 客戶區域 / 付款方式 / 配送方式）+ `sku-50.csv`（50 SKU，含 A/B/C 分類）
**案例主角**：怡君（電商營運經理）

---

### P2-1-1 · 基礎版：ABC 分類

**用途**：給 500 筆訂單，讓 AI 做商品 ABC 分類分析，最快速入門。

**Prompt**：
```
以下是我們電商平台的訂單記錄，共 500 筆（CSV 格式）：

[貼上 ecom-orders-500.csv 全文，或前 100 筆示範用]

請做「商品 ABC 分類分析」：
- A 類：佔總銷售額前 80% 的 SKU
- B 類：佔總銷售額 80-95% 的 SKU
- C 類：其他長尾 SKU

輸出格式：
| SKU | 商品名稱 | 總銷售額 | 佔比 | 分類 |
（依銷售額由高到低排列）

最後：A 類共 X 個 SKU，B 類 X 個，C 類 X 個。
```

**為何這樣寫**：ABC 分類規則（80/15/5）直接寫進 prompt，讓 AI 不需猜測，同時讓學員看到如何把商業分析框架內嵌到 prompt 中。

**平台差異備註**：
- 檔案處理：500 筆 CSV 約 35,000 字元，超過多數免費版的安全貼文字範圍——強烈建議直接上傳 CSV 檔案（Gemini free 支援，ChatGPT free 部分支援）；或先貼前 50 筆做示範，再告知學員實際使用時需上傳 💰 ChatGPT Plus / Gemini Advanced：大型 CSV 最佳選擇
- 常見限制：500 筆是此課最大的資料集，免費版最可能在這裡遇到 context 限制——遇到問題先看 B3-1 的「檔案太大」wrapper
- 跑不出時：只貼前 100 筆先確認 ABC 邏輯正確，告知學員結果僅供示範，完整分析需上傳全量資料

### P2-1-2 · 進階版：5 骨架 + ABC + Top SKU + 長尾分布

**用途**：完整的電商商品分析報告，含三個維度，可直接給主管或投資人看。

**Prompt**：
```
【角色】你是一位擅長電商商品分析的數據分析師，熟悉 ABC 分類、長尾理論和季節性分析。

【資料】以下是電商平台 2026 年 4-5 月的訂單記錄，共 500 筆（CSV 格式）：

[貼上 ecom-orders-500.csv 全文或上傳檔案]

另附 SKU 主檔（50 個 SKU 的基本資訊）：

[貼上 sku-50.csv 全文]

【任務】請做以下三個分析：
1. ABC 分類：依銷售額分類（A=前 80%、B=80-95%、C=其他），列出每個 SKU 的分類
2. Top 10 SKU：銷售額 Top 10 的 SKU，含銷售額、數量、平均單價
3. 長尾分析：C 類 SKU 的總數量 vs 總銷售額佔比，評估是否需要清倉

【輸出格式】三個分析各用一個 Markdown 表格，表格後加 1-2 句說明「最值得注意的發現」

【驗證要求】請最後列出：分析總筆數、總銷售額、涵蓋 SKU 數，並說明計算公式
```

**為何這樣寫**：結合訂單資料和 SKU 主檔，示範「多份資料合併分析」的 prompt 寫法，是這個案例的核心教學點。

**平台差異備註**：
- 檔案處理：500 + 50 筆合計資料量大，強烈建議上傳 CSV 而非貼文字 💰 ChatGPT Plus / Gemini Advanced 對多檔案合併分析更穩定
- 常見限制：免費版可能在合併兩份 CSV 時混淆欄位——如果 AI 誤讀，在 prompt 開頭明確說明「這是兩份不同的 CSV，欄位定義如下：...」
- 跑不出時：先只分析 ecom-orders-500.csv，不帶 sku-50.csv，確認 ABC 分類正確後再加入 SKU 主檔

### P2-1-3 · 變奏版：商品矩陣（毛利 × 銷量）

**用途**：用「毛利率 × 銷量」做四象限矩陣，找出明星商品、問題商品、現金牛和瘦狗。

**Prompt**：
```
以下是我們的訂單資料（500 筆）和 SKU 主檔（50 個 SKU，含成本與售價欄位；毛利率需用 (售價-成本)/售價 計算）：

[貼上 ecom-orders-500.csv 全文或結果摘要]
[貼上 sku-50.csv 全文]

請做「商品四象限矩陣分析」，定義如下：
- X 軸：月均銷量（高 = 超過所有 SKU 的中位數，低 = 低於中位數）
- Y 軸：毛利率 = (售價-成本)/售價（高 = 超過 30%，低 = 低於 30%）

四個象限：
- 明星（高銷量 + 高毛利）：重點維護
- 現金牛（高銷量 + 低毛利）：維持銷量、想辦法提毛利
- 問題兒（低銷量 + 高毛利）：試辦促銷、提升曝光
- 瘦狗（低銷量 + 低毛利）：考慮下架或清倉

請列出每個 SKU 的象限位置，並在最後說明：各象限各有幾個 SKU？最值得關注的是哪一個象限和 SKU？
```

**為何這樣寫**：四象限矩陣是電商商品管理的標準框架，prompt 中直接定義各象限讓 AI 不需自行詮釋，結果更可重現。

**平台差異備註**：
- 檔案處理：同 P2-1-2，建議上傳 CSV
- 常見限制：若 sku-50.csv 沒有毛利率欄位，AI 無法計算——先確認 sku-50.csv 的實際欄位，必要時在 prompt 中提供毛利率的計算方式
- 跑不出時：先只做「高銷量 vs 低銷量」的簡單分類，確認銷量計算正確後再加毛利率維度

### P2-1-4 · 變奏版：季節性分析（母親節週）

**用途**：找出母親節（5 月第二週）前後的銷量變化，評估季節性商品策略。

**Prompt**：
```
以下是我們的訂單資料（500 筆，日期範圍 2026-04 至 2026-05）：

[貼上 ecom-orders-500.csv 全文或上傳]

請做「母親節季節性分析」：

1. 請比較以下三個時段的每日平均訂單量和銷售額：
   - 4 月一般週（2026-04-01 至 2026-05-04）
   - 母親節檔期（~2026-05-05 至 2026-05-10、5/10 為母親節）
   - 母親節後觀察期（2026-05-11 至 2026-05-14，資料截止日 5/14）

2. 母親節期間（05-05 至 05-14、資料截止 5/14）銷量明顯提升的前 5 個 SKU 是？

3. 哪些 SKU 在母親節檔期前後的銷量差異最大（漲幅最大的 3 個 + 跌幅最大的 3 個）？

請在表格後說明：你是依據哪些欄位計算「母親節前後」的時段分界。
```

**為何這樣寫**：指定具體日期區間讓 AI 不需猜測「母親節是哪一週」，避免日曆計算錯誤。

**平台差異備註**：
- 檔案處理：同 P2-1-2，500 筆建議上傳 CSV
- 常見限制：AI 對日期計算在文字模式下容易出錯——最後的「你是依據哪些欄位」的追問是關鍵驗證點
- 跑不出時：把三個時段改成「4 月全月 vs 5 月全月」這種更大的區間，確認時段比較邏輯正確後再縮小到週

### P2-1-5 · 變奏版：庫存補貨建議

**用途**：根據 ABC 分析和銷售速度，給出補貨優先序建議。

**Prompt**：
```
以下是我們的訂單資料（500 筆）和 SKU 主檔（50 個 SKU）：

[貼上 ecom-orders-500.csv 全文或摘要]
[貼上 sku-50.csv 全文]

請做「補貨優先序建議」：

假設目前庫存告急、預算有限，請依以下邏輯給出補貨建議：
1. 優先補 A 類 SKU（高銷售額）
2. 次優先補「母親節檔期銷量有明顯上升的 SKU」
3. 暫緩補 C 類 SKU（長尾 + 低週轉）

輸出格式：
| 補貨優先序 | SKU | 商品名稱 | 建議理由 |
（最多列 10 個 SKU）

最後加一行：C 類 SKU 共 X 個，建議下架評估的有 X 個（理由：銷售額佔比低於 X% 或銷量 < N 件）。
```

**為何這樣寫**：把補貨決策邏輯直接寫進 prompt，讓 AI 的輸出有明確依據，學員回去可以把「補貨邏輯」換成自己公司的規則。

**平台差異備註**：
- 檔案處理：同 P2-1-2
- 常見限制：AI 給的「建議理由」可能過於籠統——加 B1-2 追問每個 SKU 引用了哪些具體訂單資料
- 跑不出時：先只列 A 類補貨建議（去掉 B、C 類），確認 A 類識別正確後再擴展

### P2-1-6 · 風險檢查：列出計算公式和假設條件

**用途**：電商分析完成後的標準驗證動作，確保 ABC 分類和數值計算的邏輯透明。

**Prompt**：
```
你剛才完成了電商商品分析，請做「分析透明度報告」：

1. ABC 分類的計算方式：你是用哪個欄位的數值做加總計算？是「金額」還是「數量」？
2. 銷售額加總：請列出計算公式（例：金額欄的所有數字加總，是否排除了退貨或零值？）
3. 資料完整性：你分析的總筆數是多少？是否有跳過任何列（例如空值、格式錯誤）？
4. 你做的假設：在分析中，你在哪些地方做了「資料沒有明確說明但你自行假設」的判斷？

這份報告用於確認分析可信度，請盡量精確，不要說「沒有假設」——所有分析都有隱含假設。
```

**為何這樣寫**：「不要說沒有假設——所有分析都有隱含假設」是讓 AI 誠實說出不確定點的關鍵句，是教學課上最有討論價值的輸出。

**平台差異備註**：
- 檔案處理：接在前面的分析後使用，不需再貼資料
- 常見限制：AI 對「假設條件」的回報品質差異很大——若 AI 說「沒有假設」，追問「你在計算 A 類 SKU 佔比時，分母是所有 SKU 的總銷售額嗎？這不是假設嗎？」
- 跑不出時：B1-1 是更簡短的版本，可快速追問計算公式

### P2-1-7 · 投影片素材：商品矩陣圖

**Prompt**：
```

[貼上 P2-1-2 或 P2-1-3 的輸出結果]

建議圖表：[說明建議用哪種圖表呈現，例：柱狀圖 / 散點圖，並說明 X 軸 Y 軸]

**平台差異備註**：

- 常見限制：AI 給的圖表建議可能過於複雜（例如建議四象限散點圖），免費版 Gemini 的 slide generation 不一定能直接生成——告訴學員「建議圖表」是給人工製作用的，不是 AI 自動生成的
- 跑不出時：把「建議圖表」部分改成「請直接寫出可以貼進 Excel 的資料表（兩欄：X 值 / Y 值）」，降低難度

```

- 跑不出時：去掉「市場機會」要求，只做「內部數據去敏感化 + 推薦方案」兩個改動

## Bonus 防呆追問（4 套 · B1-x）

> 這 4 套 prompt 是通用的「驗證工具」，可以接在任何主課案例的分析後使用。不附平台差異備註。

---

### B1-1 · 計算公式追問

**用途**：追問 AI 用了哪些欄位、哪個公式計算，防止數字錯誤被直接使用。

**Prompt**：
```
請列出你在上面分析中用到的：
1. 具體欄位名稱（例：「金額(萬)」欄、「日期」欄）
2. 每個數值的計算公式（例：「總業績 = 金額(萬)欄全部加總」）
3. 你分析的總取樣行數（不包含表頭）

格式：條列式，每條不超過 30 字。
```

**為何這樣寫**：三個問題覆蓋「欄位 / 公式 / 筆數」三個最常出錯的計算環節，是最精簡有效的驗證工具。

---

### B1-2 · 資料依據追問

**用途**：要求 AI 舉出具體資料行作為結論依據，防止泛化或捏造。

**Prompt**：
```
請說明你的結論依據哪幾筆資料，列出至少 5 筆具體例子（格式：[編號] [日期] [相關欄位值]），並說明這 5 筆如何支持你的結論。

若你的結論是整體統計（例如總金額），請說明你的加總方式，並抽樣列出其中 5 筆的數字。
```

**為何這樣寫**：「列出 5 筆具體例子」讓 AI 從「模糊概括」切換到「具體引用」模式，是防幻覺最有效的追問方式之一。

---

### B1-3 · 不確定點追問

**用途**：讓 AI 主動說出它最不確定的地方，在使用輸出前先知道哪些結論需要特別核對。

**Prompt**：
```
請告訴我這次分析中你最不確定的 3 個點，格式：

不確定點 1：[描述]
不確定的原因：[為什麼不確定，例：資料中這個欄位有空值 / 日期格式不一致]
建議驗證方式：[學員可以怎麼自己核對這個點]

不確定點 2：[同上]
不確定點 3：[同上]

注意：請如實回報，不要說「我對結果很有信心」來跳過這個問題。
```

**為何這樣寫**：最後那句「不要說我很有信心來跳過這個問題」是關鍵——沒有這句，AI 常常說「結果可靠」來敷衍。

---

### B1-4 · 計算結果驗證

**用途**：給 AI 你自己算好的 5 筆數字，讓它驗證自己的計算是否和你一致。

**Prompt**：
```
請驗證以下 5 筆資料的計算結果是否與你的分析一致：

1. [編號 / 名稱]：我算出 [數字]，你算出多少？
2. [編號 / 名稱]：我算出 [數字]，你算出多少？
3. [編號 / 名稱]：我算出 [數字]，你算出多少？
4. [編號 / 名稱]：我算出 [數字]，你算出多少？
5. [編號 / 名稱]：我算出 [數字]，你算出多少？

若你的計算與我不同，請說明差異的原因（哪個欄位、哪個公式）。
若你的計算與我一致，請說明你的計算方式作為確認。

（填寫說明：把上方 5 條替換成你從試算表 SUM/COUNTA 算出的真實數字，讓 AI 交叉比對）
```

**為何這樣寫**：這個 prompt 讓學員扮演「出題者」而不是「被動接收者」，把驗證的主動權交回給學員。

---

## Bonus 對象變奏（4 套 · B2-x）

> 這 4 套是「輸出對象改寫」工具，把任何分析結果改成不同受眾版本。不附平台差異備註。

---

### B2-1 · 給主管的「結論先行」摘要

**用途**：把任何 AI 分析結果壓縮成主管版精簡摘要。

**Prompt**：
```
請把上面的分析結果改寫成「給主管看的精簡摘要」，格式：

結論（第一行，不超過 30 字）：[最重要的 1 件事，含數字]

支撐依據（3 條，每條不超過 20 字）：
1.
2.
3.

需要主管決策的事項（1-2 條）：
-

整份不超過 150 字。
```

**為何這樣寫**：「結論（第一行）」強制 AI 把最重要的事放在最前面，符合主管閱讀習慣，也訓練學員的「結論先行」思維。

---

### B2-2 · 給同事的「協作討論」版

**用途**：把分析結果改寫成適合跨部門協作討論的格式，帶出明確的待澄清問題。

**Prompt**：
```
請把上面的分析結果改寫成「給跨部門同事的協作討論版」，格式：

本次分析重點（3 條，每條不超過 25 字）：
1.
2.
3.

需要大家一起確認的 3 個問題：
1. [具體問題，說明為什麼需要確認]
2.
3.

建議下一步（1 條，說清楚誰要在什麼時間前做什麼）：
```

**為何這樣寫**：「需要大家一起確認的 3 個問題」讓 AI 主動挑出分析中的不確定點，把這些點變成跨部門討論的議題而非隱藏的風險。

---

### B2-3 · 給客戶的「成果展示」版

**用途**：把內部分析結果轉成可對外展示的成果格式，強調效益和數字。

**Prompt**：
```
請把上面的分析結果改寫成「給客戶看的成果展示版」，要求：

1. 只保留對客戶有意義的數字（去掉公司內部人員姓名和機密金額，改用比例或趨勢描述）
2. 把「我們發現的問題」轉成「我們解決了什麼挑戰」的表達方式
3. 加入 1-2 個「你們可以獲得的具體效益」（用數字或百分比表達）

格式：3 段，每段不超過 50 字。

注意：若分析中沒有足夠的效益數字，請說明「此部分需要更多數據支持」，不要自行估算效益。
```

**為何這樣寫**：「不要自行估算效益」是對外溝通最重要的誠信防線，讓 AI 在沒有數字支持時如實說明，而非捏造聽起來好看的數字。

---

### B2-4 · 給自己的「備忘 + 反思」版

**用途**：把分析結果轉成個人學習備忘，含下次改進點，適合每次案例跑完後留存。

**Prompt**：
```
請把上面的分析過程和結果，整理成「個人備忘 + 反思」，格式：

這次我做了什麼：[一句話描述，不超過 30 字]

得到的結論（前 3 重要）：
1.
2.
3.

這次 AI 做得不好的地方（1-2 條）：
-

下次如果再跑這個分析，我會怎麼改 prompt（1 條具體改法）：

保存日期：[今天日期]
```

**為何這樣寫**：「AI 做得不好的地方」讓學員養成批判性評估 AI 輸出的習慣，是整個課程最重要的後設學習行為。

## Bonus 平台 Wrapper（1 套 · B3-x）

> 這套 wrapper 集合 4 種常見的「跑不出來」情況的解法。不附平台差異備註。

---

### B3-1 · 跨平台適配 Wrapper 集（4 種情況）

**用途**：遇到免費版限制跑不出來時，用這 4 種 wrapper 換個方式繼續。

---

**Wrapper 1：檔案太大（context 超出）**
```
上面的資料太大，我先給你前 50 筆，你先分析這 50 筆，輸出格式和規則與前面相同。

[貼上前 50 筆 CSV]

請在分析結束後說明：「以上是前 50 筆的分析結果，若需要完整 [N] 筆的分析，請在有能力處理大量資料的環境（例如 Gemini 上傳 CSV / 試算表函數）中重新進行。」
```

---

**Wrapper 2：Context 不夠（輸出被截斷）**
```
你剛才的回覆似乎被截斷了，請從「[最後一行完整的輸出]」這裡繼續，不要重複前面已經輸出的部分，直接接續。
```

---

**Wrapper 3：輸出格式不對（想要 Markdown 表格但得到條列式）**
```
請把上面的輸出改成 Markdown 表格格式，欄位依序是：[欄位1] / [欄位2] / [欄位3]。
如果某個欄位的資料不存在，請填「—」，不要省略該欄位。
```

---

**Wrapper 4：結果有幻覺（AI 給了資料中不存在的數字）**
```
我發現你在上面的分析中提到「[具體說明 AI 給的疑似錯誤數字或說法]」，但在我的資料中找不到這個數字的依據。

請重新檢查：
1. 這個數字是來自資料中哪一筆（請引用編號或列號）？
2. 還是你推算出來的？若是推算，請說明計算過程。
3. 若你確認找不到依據，請直接說「我無法從給定資料中確認這個數字，建議你用試算表自行計算」。
```

**為何這樣寫**：四種 wrapper 覆蓋了學員在免費版最常遇到的四個狀況，讓課堂上遇到「卡住」的時刻有明確的下一步，而不是放棄。

## NotebookLM 專用（5 套 · NB-x）

> 以下 5 套 prompt 專為 NotebookLM 設計，利用其 source-grounded 問答功能。
> 這是少數需要指定工具的案例——NotebookLM 的「引用來源」功能是不可替代的，其他主流 AI 工具無法在問答中自動顯示原文引用。

**使用前提**：先在 NotebookLM 建立一個新 Notebook，上傳所需文件後，再把以下 prompt 貼到對話框。

---

### NB-1 · 多份文件自動產 FAQ

**用途**：上傳多份公司內部文件（例如：SOP 手冊、公告、會議記錄），讓 NotebookLM 自動產出 FAQ 供同事查詢。

**Prompt**：
```
根據我上傳的文件，請幫我產出「常見問題 FAQ」，要求：

1. 列出 10-15 個這份文件的讀者最可能會問的問題
2. 每個問題附上從文件中引用的具體答案（引用時請標記來源文件和段落）
3. 若同一個問題在多份文件中有不同說法，請明確指出哪份文件說什麼，不要自行合併或取捨

格式：
Q：[問題]
A：[答案]（來源：[文件名稱] [頁碼或段落]）

若某個問題在文件中找不到答案，請說「文件中未提及，建議詢問主管 / 相關部門」，不要自行補充。
```

**為何這樣寫**：「引用時請標記來源文件」是 NotebookLM 的核心優勢，這個 prompt 充分利用這個功能；「找不到答案就說沒有」是防止 AI 超出文件範圍回答的關鍵設計。

---

### NB-2 · 會議紀錄的 Source-Grounded 摘要

**用途**：上傳會議紀錄，讓 NotebookLM 做有來源引用的摘要，避免摘要遺漏或曲解會議結論。

**Prompt**：
```
根據我上傳的會議紀錄，請做一份「有來源引用的會議摘要」，格式：

會議基本資料（日期 / 出席人 / 主要議題）：[從文件引用]

主要決議（3-5 條）：
1. [決議內容]（引用：[原文段落]）
2.
3.

待辦事項（附負責人和期限）：
| 事項 | 負責人 | 期限 | 原文引用 |

尚未有結論的討論點（1-3 條）：
-（說明：為何這點尚未有結論，依據文件中的哪段描述）
```

**為何這樣寫**：「尚未有結論的討論點」讓 NotebookLM 主動標示出會議中未解決的問題，是後續追蹤最重要的資訊。

---

### NB-3 · 技術文件的引用式問答

**用途**：上傳技術說明書、產品手冊或政策文件，做精準的問答並避免幻覺。

**Prompt**：
```
我上傳了一份技術文件 / 政策手冊，請用「引用式問答」模式回答我的問題：

每次我問問題，你的回答格式必須是：
**直接答案**：[一句話答案]
**依據原文**：「[原文引用]」（頁碼 / 段落）
**補充說明**：[若需要解釋原文，加 1-2 句；若不需要則省略]

若我的問題在文件中找不到答案，請直接說「文件未提及」，不要根據你的訓練資料補充答案。

我的第一個問題是：[你的問題]
```

**為何這樣寫**：「不要根據你的訓練資料補充答案」是 NotebookLM 使用中最重要的邊界設定，確保問答結果完全來自上傳文件。

---

### NB-4 · 跨文件 Cross-Reference 分析

**用途**：上傳多份文件後，讓 NotebookLM 找出文件間的矛盾、差異或一致性，適合政策更新後的版本對照。

**Prompt**：
```
我上傳了多份文件（例如：新版和舊版的操作手冊、或不同部門的 SOP）。

請做「跨文件對照分析」，找出：

1. 文件間有矛盾的地方：
   - [矛盾點描述]（文件 A 說：「[原文]」；文件 B 說：「[原文]」）

2. 文件間一致的重要規則（只列最重要的 3-5 條）：
   - [規則]（所有文件都有提到）

3. 只在某份文件中出現、其他文件沒有提到的重要資訊：
   - [資訊]（僅見於：[文件名稱]）

格式要求：每條必須引用原文，不要只給摘要。
```

**為何這樣寫**：三個分類（矛盾 / 一致 / 獨特）讓 NotebookLM 做結構化的文件差異分析，而不是只做單份文件摘要。

---

### NB-5 · 產 Audio Overview（通勤聽用）

**用途**：利用 NotebookLM 的 Audio Overview 功能，把上傳文件轉成對話式音頻，適合通勤學習。

**Prompt**：
```
（注意：這個功能不是在對話框輸入 prompt，而是點擊 NotebookLM 右側面板的「Audio Overview」按鈕直接生成。以下是給學員的操作說明）

操作步驟：
1. 在 NotebookLM Notebook 中上傳你的文件（會議記錄 / 報告 / SOP 等）
2. 點擊右側「Audio Overview」面板
3. 點擊「Generate」按鈕，等待約 2-5 分鐘
4. 生成完成後，可在瀏覽器直接播放，或下載 MP3

自訂重點（optional，在生成前可以設定）：
- 在 Audio Overview 設定中，加入這段文字：
```
請重點說明：[你最想聽到的主題，例：費用異常的處理方式、下一步行動建議]
請略過：[不重要的背景說明、已知的資訊]
語言：繁體中文
```

適用情境：
- 通勤途中聽上週會議記錄的摘要
- 熟悉一份新的政策文件的大意
- 用耳朵「快速掃過」一份長報告

注意：Audio Overview 目前（2026-05）支援繁體中文，約 10-15 分鐘音頻，兩位虛擬主持人對話式呈現。
```

**為何這樣寫**：Audio Overview 是 NotebookLM 最有差異化的功能，但操作方式和其他 AI 工具不同（不是貼 prompt），需要說明清楚操作步驟和適用情境，避免學員期待錯誤。
