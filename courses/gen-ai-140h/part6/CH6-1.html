(01) · API Fundamentals

API 核心概念：
Request、Response 與 Headers

每一次呼叫 AI API，本質上都是一次 HTTP 請求。理解這個請求的結構，你才能看懂錯誤訊息、調整行為、控制成本。

Request（你送出的）
包含 method（POST）、URL（API 端點）、headers（身份驗證、內容格式）、body（你的 prompt 和參數）。
Response（你收到的）
包含 status code（成功/失敗）、headers（速率限制剩餘量）、body（模型的回應 JSON）。
Headers
最重要的兩個：Authorization: Bearer YOUR_KEY（身份驗證）和 Content-Type: application/json（告訴 API 你送的是 JSON）。
Status Code
200 成功、400 你的請求有問題、401 金鑰錯誤、429 超過速率限制、500 伺服器端出錯。
HTTP一次典型的 Anthropic API 請求結構
POST https://api.anthropic.com/v1/messages

// Headers
x-api-key: sk-ant-api03-...
anthropic-version: 2023-06-01
content-type: application/json

// Body（JSON）
{
  "model": "claude-haiku-4-5-20251001",
  "max_tokens": 1024,
  "messages": [
    { "role": "user", "content": "你好，請介紹自己" }
  ]
}
ℹ️
為什麼後端代理是必要的？因為 x-api-key 這個 header 必須放在 request 裡。如果前端直接呼叫 API，任何人打開 DevTools 的 Network tab 就能看到你的金鑰。後端代理的作用是把這個 header 藏在伺服器端。
(02) · Token Cost

Token 成本意識：
每個字都在計費

AI API 不是按「次」計費，而是按「token」計費。Token 是模型處理文字的基本單位——大約 1 個英文單字 = 1.3 個 token，1 個中文字 = 1.5 ~ 2 個 token。

你付費的不只是輸出的回應，輸入的 prompt 也計費。一個包含長 System Prompt 的工具，每次呼叫的成本可能比你想的高三倍。
System Prompt（300 tokens）× $0.0008 / 1K = $0.00024
User Message（50 tokens）× $0.0008 / 1K = $0.00004
Model Response（200 tokens）× $0.004 / 1K = $0.00080
單次呼叫合計≈ $0.00108
1000 次呼叫≈ $1.08

控制 Token 成本的三個方法：

設定 max_tokens 上限
在 API 呼叫時指定 max_tokens，防止模型輸出超長回應。根據你的工具需求設定合理上限，不要用預設值。
選對模型
簡單任務（分類、擷取、格式轉換）用 Haiku。複雜推理才用 Sonnet 或 Opus。成本差距可以到 10 ~ 100 倍。
Prompt Caching
對長且固定的 System Prompt 啟用 cache，後續呼叫的輸入 token 成本降低 90%。高頻工具必用。
回傳 usage 欄位
API response 裡有 usage.input_tokens 和 usage.output_tokens，記錄下來才能知道實際用了多少。

(03) · Rate Limit

Rate Limit 與指數退避重試

每個 AI API 都有速率限制（Rate Limit）：每分鐘最多幾個 request、每分鐘最多幾個 token。超過限制就會收到 429 Too Many Requests。

解法不是「等一秒再試」，而是指數退避（Exponential Backoff）：第一次失敗等 1 秒，第二次等 2 秒，第三次等 4 秒……每次等待時間加倍，避免所有重試同時湧入讓問題更嚴重。

JAVASCRIPT指數退避重試實作
async function callWithRetry(fn, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (err.status === 429 && attempt < maxRetries - 1) {
        const delay = Math.pow(2, attempt) * 1000; // 1s, 2s, 4s
        console.log(`Rate limited, retrying in ${delay}ms...`);
        await new Promise(r => setTimeout(r, delay));
      } else {
        throw err; // 其他錯誤或超過重試次數，直接拋出
      }
    }
  }
}
🔑
Response Headers 裡藏著剩餘配額：Anthropic 的回應 headers 包含 anthropic-ratelimit-requests-remaining 和 anthropic-ratelimit-tokens-remaining。在高流量工具裡可以主動監控這些數值，在接近上限前就減速。

(04) · Error Handling

錯誤碼處理：
4xx vs 5xx，不同的應對策略

API 錯誤分兩大類。4xx 代表你的請求有問題（可以修正）；5xx 代表 API 伺服器端出問題（你只能等待或重試）。錯誤處理的設計原則：4xx 告訴使用者可以做什麼，5xx 告訴使用者你在處理中。

狀態碼原因使用者看到的訊息你的處理方式
400請求格式錯誤（缺少必填欄位、JSON 格式不對）「輸入有問題，請重新確認後送出」檢查你的輸入驗證邏輯
401API 金鑰無效或過期「服務暫時無法使用，請聯繫管理員」（不要暴露金鑰問題）檢查 .env 設定，更新金鑰
413輸入內容太長，超過 context window「輸入文字太長，請縮短後再試（建議 2000 字以內）」在前端加上字數限制
429速率限制「目前使用人數較多，請稍後再試」指數退避重試
500API 伺服器內部錯誤「AI 服務暫時無法使用，請稍後再試」重試 1~2 次後放棄，記錄 log
529API 過載（Anthropic 特有）「AI 服務目前負載較高，請稍後再試」等待 30 秒後重試
💡
UX 原則：永遠給使用者一個「下一步」。不要只顯示「發生錯誤」。是可重試的就說「請稍後再試」；是使用者可以修正的就說「請縮短輸入」；是需要等待的就顯示 loading 狀態，讓使用者知道你沒有卡住。

情境 B 同一個 prompt 打 ChatGPT / Claude / Gemini — 用程式碼做一張對照表
任務背景 + 工具棧比感覺更可靠的是數字
【情境】 你要幫公司挑一個預設模型給內部問答工具用。不要靠「感覺 Claude 比較會寫中文」這種模糊印象，直接用同一段 prompt 丟三個模型，把回覆、token 數、延遲與成本排成一張表。 【工具棧】 - Python 3.10+ - openai、anthropic、google-genai 三個 SDK（pip 安裝） - 三組 API key（環境變數 OPENAI_API_KEY / ANTHROPIC_API_KEY / GOOGLE_API_KEY） - 一個固定 prompt + 固定 temperature 0 【成功條件】 跑完後能輸出一張 3×5 的表：模型｜回覆前 60 字｜prompt_tokens｜completion_tokens｜單次成本（USD）。
程式碼 · 多模型比較腳本python
import os, time from openai import OpenAI from anthropic import Anthropic from google import genai PROMPT = "請用三點說明為什麼 LLM 會產生幻覺，每點不超過 20 字。" def call_openai(): c = OpenAI() t = time.time() r = c.chat.completions.create( model="ChatGPT", messages=[{"role":"user","content":PROMPT}], temperature=0, max_tokens=300, ) return { "model": "ChatGPT", "text": r.choices[0].message.content, "pt": r.usage.prompt_tokens, "ct": r.usage.completion_tokens, "lat": round(time.time()-t, 2), } def call_claude(): c = Anthropic() t = time.time() r = c.messages.create( model="Claude", max_tokens=300, temperature=0, messages=[{"role":"user","content":PROMPT}], ) return { "model": "Claude", "text": r.content[0].text, "pt": r.usage.input_tokens, "ct": r.usage.output_tokens, "lat": round(time.time()-t, 2), } def call_gemini(): c = genai.Client() t = time.time() r = c.models.generate_content( model="Gemini", contents=PROMPT, ) return { "model": "Gemini", "text": r.text, "pt": r.usage_metadata.prompt_token_count, "ct": r.usage_metadata.candidates_token_count, "lat": round(time.time()-t, 2), } rows = [call_openai(), call_claude(), call_gemini()] print(f"{'model':<10}{'pt':>6}{'ct':>6}{'lat':>7} preview") for r in rows: print(f"{r['model']:<10}{r['pt']:>6}{r['ct']:>6}{r['lat']:>7} {r['text'][:60]}")

  • 同題 × 三溫度：拿情境 A 的 curl，把 temperature 依序設 0、0.7、1.2 各打一次，比對輸出風格與 usage tokens。寫一句結論：「哪個 temperature 最適合我的客服工具？」
  • 故意打壞：把 API key 後 3 碼改掉、把 model 名稱寫錯、把 messages 欄位刪掉，分別觀察 401 / 404 / 400 的錯誤訊息差在哪，為將來除錯建索引。
  • 加計成本欄：把情境 B 的腳本加一欄 cost，依三個模型的公開單價把 pt / ct 換算成單次成本，並推算「每天 1000 次要多少錢」。
1API key 最安全的存放方式是？
只要金鑰能被瀏覽器或公開 repo 讀到就算洩漏。private repo 被 clone 過後一樣會外洩；README 更是公開。唯一安全路徑：放伺服器環境變數 + .gitignore + 洩漏時立刻撤銷換新。
2呼叫 API 回 401 Unauthorized，最該先檢查什麼？
401 是身份驗證失敗，與 model / token 無關。排查順序：先 echo 環境變數看有沒有讀到、再看 header 格式（Bearer 前綴要有）、最後登入後台確認 key 沒被撤銷。model 拼錯回的是 404 或 400 不是 401。
3收到 429 Too Many Requests 時，下列哪個處理最妥當？
429 代表你打太快。立即重試會讓問題更糟；換 key 規避限流違反 ToS；直接放棄體驗差。標準做法是指數退避：每次等待時間加倍、限制最大重試次數、超過才回報失敗。
4什麼情境最適合開 streaming 而不是一次性回應？
Streaming 的價值是「感知速度」，使用者有在看畫面才有意義。結構化 JSON 要等全部齊了才能 parse，streaming 反而要多寫拼接；背景批次沒人在等，streaming 只會讓程式變複雜。能省等待焦慮的情境才開。
5估算單次 API 呼叫成本，下列哪個算法正確？
輸入與輸出 token 單價不同（輸出通常是輸入的 3–5 倍），且兩者都會計費。把它們合併算會嚴重低估或高估；主流 API 也不是按時間計費。正確做法是從 usage 欄位分別拿 input / output token，乘以各自單價再加總。
