API 呼叫測試頁

親手送出 API 請求，觀察 token 消耗、比較模型差異、驗證錯誤處理。這是 AI 開發的基本功日記。

任務目標
建立一個可以調整模型、溫度、max_tokens 的 API 測試介面，觀察不同參數組合對輸出的影響，並學會解讀 token 使用量與錯誤回應。
1
理解三個關鍵參數
model 決定能力與成本；temperature（0–1）控制創意度，0 最確定，1 最多變；max_tokens 限制輸出長度，直接影響費用。三個參數的組合決定你拿到什麼。
建議先用 claude-haiku-4-5-20251001：速度最快、成本最低，測試邏輯時不需要用最貴的模型。
2
在下方測試工具送出你的第一個請求
填入 System Prompt 和 User Message，選擇模型與參數，點擊「送出請求」。注意：這是模擬介面，需搭配後端代理才能實際呼叫 API。觀察輸出格式與 token 統計。
3
做三組對比實驗
① 相同 prompt，溫度 0 vs 1：觀察輸出一致性的差異
② 相同 prompt，Haiku vs Sonnet：比較品質與 token 消耗
③ max_tokens 50 vs 500：理解截斷行為對輸出的影響
4
故意觸發一個 4xx 錯誤
把 max_tokens 設成 0 或傳入空的 prompt，觀察錯誤訊息格式。記住：好的錯誤訊息是你的除錯盟友，學會讀懂它比看文件更快。
常見錯誤：invalid_request_error（參數錯誤）、authentication_error（API Key 問題）、overloaded_error（服務忙碌，適合重試）

JAVASCRIPT · 真實呼叫（替換模擬函式）
async function sendRealRequest() {
  const userMsg = document.getElementById('user-input').value.trim();
  if (!userMsg) { alert('請輸入訊息'); return; }

  const res = await fetch('/api/chat', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      model: document.getElementById('model-sel').value,
      max_tokens: +document.getElementById('maxtok-range').value,
      temperature: +document.getElementById('temp-range').value,
      system: document.getElementById('system-input').value,
      messages: [{ role: 'user', content: userMsg }]
    })
  });
  const data = await res.json();
  document.getElementById('result-text').textContent = data.content[0].text;
}

案例 01 · 法律科技
律所建立判決書摘要 API 基線的第一次壓測
難度 ★★☆ 約 45 分鐘

你是中型商務律所新成立的 AI 小組工程師。合夥人要求先做一個「判決書三段摘要」API 原型（事實 / 爭點 / 判決結果），之後會嵌進內部檢索系統。你需要選模型、定 temperature、估成本，給合夥人一份可決策的基線報告。

原始資料｜單筆判決書樣本（節錄）
【案號】臺灣臺北地方法院 113 年度訴字第 01234 號 【當事人】原告 A 股份有限公司 / 被告 B 有限公司 【事實】兩造於 112 年 3 月簽訂 OEM 代工合約，約定被告按月交付 1,000 組電子零件，單價新台幣 580 元，交付期限為次月 15 日前。112 年 9 月起被告連續三個月延遲交付並出現瑕疵品比率達 12%，原告依合約第 7 條終止契約並請求違約金 180 萬元。 【爭點】（1）延遲與瑕疵是否構成重大違約；（2）原告終止權行使是否逾除斥期間；（3）違約金是否過高應酌減。 【判決】被告應給付原告 120 萬元及自起訴狀繕本送達翌日起之法定遲延利息。訴訟費用由被告負擔五分之四。 （全文約 4,800 字，input tokens 約 6,200）
你的任務
  1. 用 API 測試頁跑同一份判決書三次：Haiku temp=0 / Sonnet temp=0 / Sonnet temp=0.7，記下 output tokens、stop_reason、語氣差異
  2. 用四要素寫一版 v1 system prompt（角色＝資深法務助理、任務＝三段摘要、限制＝不得臆測未載事實、輸出＝JSON {facts,issues,ruling}）
  3. 推估 1,000 筆判決書批次跑 Haiku vs Sonnet 的月成本差，寫進給合夥人的決策表
評估指標
事實零臆測
摘要出現的金額、日期、比率必須能在原文對應，不得推論
結構穩定
JSON 三鍵齊全、能 parse，連跑 10 次格式不崩
成本可接受
單筆平均成本算得出來，跟合夥人預算對得上
選型有依據
Haiku 或 Sonnet 的選擇有具體 benchmark 數據，不是感覺
參考 system prompt（建議先自己寫再看）
點擊展開參考答案
你是資深法務助理，協助律師快速理解民事判決書。遵守以下原則： 【任務】讀入單筆判決書全文，輸出三段摘要。 【限制】 - 只能引用原文出現的事實、金額、日期、條號；不得推論或補充原文未載的內容 - 爭點必須列為條列式，至少 2 點、最多 5 點 - 判決結果需包含金額與利息起算日（若有） - 若原文未提及某欄位（例如無違約金爭議），該欄位輸出空陣列或 null，不得捏造 【輸出格式】 嚴格 JSON，不得包裹 markdown code fence： { "facts": "事實段，80-150 字", "issues": ["爭點 1", "爭點 2", ...], "ruling": "判決結果，含金額與利息，50-100 字", "confidence": "high|medium|low（自評摘要可靠度）" } 【溫度建議】temperature 設 0 或 0.1，確保同一份判決書重跑結果穩定。
成本推估｜給合夥人的決策表
假設：單筆 input ≈ 6,200 tokens、output ≈ 450 tokens，月批次 1,000 筆 模型 | 單筆 in+out 成本 | 月成本（1,000 筆） | 事實正確率樣本測試 Claude Haiku 3.5 | 約 US$0.0072 | 約 US$7.2 | 92%（10 筆人工校） Claude Sonnet 4.5 | 約 US$0.0254 | 約 US$25.4 | 99%（10 筆人工校） 決策建議： - 若摘要會直接給客戶看 → Sonnet，每月多 18 美金買 7 個百分點的正確率值得 - 若只是內部檢索索引 → Haiku，並設 confidence=low 時自動 fallback Sonnet * 以上為示意數據，實際跑過再填你的結果

案例 02 · 電商客服
SaaS 客服要把舊便宜模型換成 Claude，先做模型選型 A/B
難度 ★★☆ 約 45 分鐘

你是一家年 GMV 3 億、月客服量 8,000 通的跨境電商內部 AI 工程師。目前客服 bot 用便宜的舊模型跑得便宜但答錯率高。主管問「能不能換掉？小模型或大模型哪個划算？」你需要設計一個最小 A/B 實驗，不是拍腦袋。

原始資料｜過去一週真實客服問題分布（匿名化樣本）
問題類型 | 週出現數 | 樣本問題 查訂單狀態 | 2,100 | 「我訂單 #A10234 到哪了？」 退換貨規則 | 1,450 | 「買錯尺寸 7 天內可以換嗎？要自付運費？」 商品規格比較 | 980 | 「這兩款行動電源差在哪？出國能帶嗎？」 折扣碼與活動 | 760 | 「雙 11 的碼還能用嗎？可以跟會員 9 折疊加？」 物流異常/客訴 | 410 | 「訂單三天沒動態，是不是被弄丟了？」 其他/閒聊 | 2,300 | 「你們今天上班嗎？」 * 前五類佔 70%，屬於高價值可自動回覆區間 * 目前舊模型在「退換貨規則」類答錯率約 18%（客服人員複審判定）
你的任務
  1. 從五大類各挑 4 題組成 20 題 golden set，寫下每題「正確答案應該包含哪些要素」（作為評估依據）
  2. 用 API 測試頁對同一份 system prompt 跑 Haiku / Sonnet 兩個模型，temperature 都設 0.2，記錄回答品質
  3. 產出一份給主管的建議書：要不要換、換成哪個、預期月成本與品質提升
評估指標
Golden set 覆蓋
20 題涵蓋真實高頻問題，不是你憑空想的測試
正確率可量化
每題有 pass / fail 判定依據，不是「看起來還 OK」
成本落地
月預估以真實 8,000 通量計算，含 input/output tokens
風險意識
提到退換貨這類高風險題是否需要人工覆核機制
參考 A/B 報告骨架（建議先自己寫再看）
點擊展開參考答案
【結論先行】建議分層：Haiku 處理查單/折扣碼/閒聊（約 60% 流量），Sonnet 處理退換貨/物流異常（約 10% 流量），其餘 30% 繼續人工。預估月成本 US$180，品質從 82% → 94%。 【數據】 Golden set 20 題 pass 率： - 舊 baseline：13/20 = 65% - Haiku 3.5：16/20 = 80%（退換貨類仍會漏規則） - Sonnet 4.5：19/20 = 95%（唯一一題失分在折扣碼疊加邏輯） 【成本】 假設平均對話 input 800 tokens、output 200 tokens，月 8,000 通： - 全部 Haiku：約 US$72/月 - 全部 Sonnet：約 US$260/月 - 分層（Haiku 60% + Sonnet 10%）：約 US$75/月 - 舊模型：約 US$32/月 【風險】 - 退換貨類必須接 human-in-the-loop，confidence < 0.8 時轉客服 - 折扣碼疊加邏輯建議寫死在 system prompt 當成 constraint 給模型，不讓模型自由推論 - 上線前先影子模式跑 3 天（bot 回答但不發給用戶），人工抽驗 200 題再切流量 【決策建議】 換。多花 43 美金/月買 12 個百分點正確率，以每單客訴潛在損失 NT$500 計算，只要少 3 件客訴就回本。
變體挑戰
A改用 Sonnet + extended thinking（reasoning_effort），看退換貨類正確率能否推到 100%、成本多多少，評估是否值得
B加 prompt caching 把退換貨規則條款 cache 住（長度約 3,000 tokens），算每天 800 通呼叫 Sonnet 時省多少
C設計一個「confidence 自評」機制：要模型每次回答附上 0-1 的 confidence，低於 0.7 自動轉人工，觀察對品質與成本的影響
自我檢核

同儕互評

AI 工程師最容易犯的錯，是「自己做的 RAG 原型自己覺得沒幻覺」。兩兩配對互找漏洞，比自評多抓出 3 倍問題。約 25 分鐘。

同儕互評 01 · RAG 幻覺獵人
RAG 回答對拆——我產 3 題，你找 3 個幻覺點
兩人一組 約 25 分鐘 需一份企業內部 FAQ/規則文件

中型企業導入 RAG 最大的翻車點不是 retrieval miss，而是「retrieval 拿到正確 chunk，但 generation 仍編造未出現在 chunk 的細節」。這個活動讓你用對手視角抓出自己 prompt 或 chunking 的漏洞。

ARAG 產出者
選一份企業文件（HR 政策、產品規格、SOP 任一），用自己的 RAG 原型（或手貼 chunk 進 Claude / GPT）跑 3 個查詢，產出 3 則完整答案（含引用的 chunk 片段）交給 B。題目要選「容易編造」的邊界題，例如數字、日期、例外條款。
B幻覺獵人
拿到 A 的 3 則答案與 chunk，逐句比對「這句話是否能在 chunk 裡找到依據」。每則至少標記 3 個可能幻覺點，分類為：無根據的具體數字 / 過度概化 / 跨 chunk 拼接錯誤 / 改寫失真。寫回 A 並提 1 個修正方向（改 prompt 或改 chunking）。
活動節奏
1A 選文件、設計 3 個邊界題、跑 RAG 產答案· 8 分鐘
2A 把答案 + 原始 chunk 交給 B（不講自己覺得哪裡可疑）· 1 分鐘
3B 逐句比對，標 9 個以上幻覺點並分類· 8 分鐘
4兩人對談：B 講標記理由、A 講原本想法，找出 prompt 或 chunk 切法的真正漏洞· 6 分鐘
5A 改版重跑 1 題驗證修正是否奏效· 2 分鐘
互評指標（1-5 分）
題目挑戰3 題都打在容易編造的邊界（數字、例外、跨段）
引用完整每句主張都能對回 chunk 行號或段落
分類準確4 類幻覺標籤用對，不混淆
修正可行提的修正方向具體到改哪個 prompt 段落或 chunk 大小

(C) · RAG Doc QA Drills

RAG 文件問答系統：兩個情境、一條 pipeline

RAG（Retrieval-Augmented Generation）把內部文件切 chunk、做 embedding、存向量庫，提問時先檢索再餵給 LLM 回答。下面兩個情境貼近真實辦公場景——公司 FAQ 與會議紀錄——都附完整 pipeline 步驟、可貼上跑的 prompt、預期輸出與來源引用格式。每情境約 22 分鐘，兩情境＋延伸＋quiz 共約 58 分鐘，做完你會有一條可直接改吃自家文件的 RAG 雛型。

情境 A · 公司 FAQ RAG 30 篇 FAQ MD 檔 → chunk → embedding → 問答，取代「每次問 HR」的內耗。
資料來源｜資料量｜query 範例FAQ · RAG
【資料來源】公司內部 Notion / Confluence 匯出的 30 篇 FAQ Markdown（HR 政策、IT 帳號、差旅、報銷、合約流程），單篇 400–1,500 字，合計約 28,000 字 / 約 42,000 tokens。 【資料量】 - 30 篇原始檔，切 chunk 後約 180–240 個 chunk（每 chunk 180–220 字，overlap 40 字） - 向量維度 1024（用 voyage-3 或 text-embedding-3-large 皆可） - 向量庫用本機 Chroma 或 sqlite-vec 起手，量小不用雲端 【query 範例】 Q1「新人到職第一週要辦哪些帳號？」（跨 2–3 篇） Q2「出差日支費 500 元包含哪些項目？需要附發票嗎？」（單篇細項） Q3「合約金額超過 30 萬要走誰核可？」（需要抓金額門檻） Q4「公司有沒有寵物假？」（資料沒有，要會拒答） 【成功標準】前 3 題命中正確 FAQ 且 answer 有引用來源；Q4 要答「FAQ 中沒有相關規定」而不是硬掰。
RAG pipeline 步驟 + 關鍵 Prompt
【Pipeline 步驟】 1. Ingest：讀 30 個 .md 檔，保留 filename、section_heading 當 metadata 2. Chunk：按 H2 標題先切大段，每段再按 200 字 + 40 overlap 切小塊，保留 { doc_id, heading, chunk_id } metadata 3. Embed：每個 chunk 丟 embedding model，存向量 + metadata 到 Chroma 4. Retrieve：query embedding → top_k=5 相似度搜尋 → similarity 低於 0.35 的 chunk 直接丟掉 5. Generate：把 top_k chunks 按順序拼進 system prompt，要求答案必附 [doc_id#heading] 【關鍵 System Prompt】 你是公司內部 FAQ 助理。嚴格遵守以下規則： 【規則】 - 只能根據下方【參考資料】回答，不得引用訓練資料或自行推論 - 每句主張後面必附來源標註，格式為 [檔名#段落標題]，多來源用逗號分隔 - 若【參考資料】中沒有答案，或相似度過低，直接回答「FAQ 中沒有相關規定，建議聯絡 HR（hr@company.com）」 - 金額、天數、條款編號一律照抄原文，不得換算、不得四捨五入 - 回答控制在 120 字以內，先給結論再給依據 【參考資料】 {{ retrieved_chunks_with_metadata }} 【使用者問題】 {{ user_question }}
預期 answer + 來源引用格式
Q:「新人到職第一週要辦哪些帳號？」
A: 第一週需辦三組帳號：① Google Workspace（到職當日由 IT 建立）[onboarding.md#帳號開通]；② VPN 與內網 AD（到職第二日至 IT 櫃台領卡）[onboarding.md#帳號開通]；③ 人資系統 HRMS（到職第三日由 HR 發連結自行設密碼）[hr-system.md#新進流程]。超過三日未收到請聯絡 it@company.com。
sources: ["onboarding.md#帳號開通", "hr-system.md#新進流程"]
retrieved_top_k: 5, used: 2, similarity_range: [0.78, 0.62]
判讀重點：答案每句都能對回 [檔名#段落]；Q4「寵物假」應回拒答版本且 retrieved_top_k 最高相似度低於 0.35。若出現沒有 source 標記的句子 = 幻覺，立刻回去改 prompt 加強「每句必附來源」指令。

延伸挑戰（約 12 分鐘）
  • 做一個你公司 20 份文件的 RAG：挑 20 份你真的每天會翻的文件（SOP、規格、合約模板、FAQ），走完 ingest → chunk → embed → retrieve → generate 全流程。
  • 測 10 題準確率：列 10 題同事平常會問的題目（7 題資料內有、3 題資料外），跑完記 pass / fail，目標是資料內題 ≥ 8/7 準確、資料外題 3/3 正確拒答。
  • 加「我不知道」fallback：檢索最高相似度低於 0.35（或你自己調出來的閾值）時，強制模型回「本 RAG 知識庫中未找到相關內容，建議聯絡 XX」——沒有這層 fallback 的 RAG 就是幻覺製造機。
1RAG chunk 大小要怎麼選？
chunk 太大會把不相關資訊一起檢索出來、稀釋 top_k 相關度；chunk 太小會把一個完整論點切碎、檢索到片段但答不出來。實務做法是主體 150–300 字 + overlap 避免切在論點中間，再針對「必須完整」的段（合約條款、會議決議）特別處理整段不切。
2檢索出來的 chunk 相關性很差（similarity 都很低），該怎麼處理？
把爛 chunks 丟給 LLM，LLM 會基於不相關的片段硬湊答案——這就是 RAG 的典型幻覺源頭。正確做法是設 similarity 閾值（每個 embedding 模型要自己跑一輪校正閾值）＋觸發明確的「沒有相關資訊」fallback。加大 top_k 只會讓雜訊更多。
3RAG 系統要怎麼避免 hallucination？
RAG 幻覺的兩大來源：① 檢索對但模型仍加油添醋；② 檢索不到但模型不肯說不知道。兩者都要靠 prompt 硬約束——強制來源引用讓加油添醋的句子無法產生（因為沒來源可標）；強制 fallback 讓「我不知道」變成合法答案。temperature 0 不會解決語意層幻覺。
4embedding 向量該什麼時候重跑？
embedding 是有「版本」的——換模型（text-embedding-3-small → large，或換 voyage-3）等於換向量空間，舊向量跟新 query 不能比。增量更新要補新 chunk 的 embedding，大改 / 模型升級要全量重建。每天跑是浪費成本，不跑又會踩舊向量。
5RAG 系統最常見的失敗模式是哪一類？
RAG 的 debug 要分三段看：retrieval（有沒有拿到對的 chunk）、grounding（拿到後有沒有真的用它）、generation（有沒有適當平衡檢索內容與常識）。只看最終答案錯就歸因於「LLM 笨」會走錯方向——要先看 retrieved chunks，再對照 answer 判斷是哪一段出問題。
