CH 6.2 · 3 小時

Embedding 與向量搜尋基礎

搜尋「貓」找到「寵物」，這件事用關鍵字搜尋做不到，但向量搜尋可以。理解 Embedding 是理解 RAG 的前提，而 RAG 是讓 AI 「讀你的文件」最常用的技術。

學習成果 · OUTCOMES
能解釋語義搜尋與關鍵字搜尋的根本差異
理解 Embedding 的直覺概念：把文字變成方向（不需要數學）
知道向量資料庫的角色和主流選項
能判斷什麼情況用 RAG，什麼情況用長 context 模型更簡單
(01) · Semantic vs Keyword

關鍵字搜尋 vs 語義搜尋

傳統搜尋是字面匹配：你輸入「機器學習」，就只找包含這三個字的文件。如果文件裡寫的是「AI 訓練」，搜尋結果裡不會有它——即使它說的是同一件事。

語義搜尋解決這個問題。它的基礎是 Embedding：把文字轉換成數字向量，語義相近的文字在向量空間裡彼此靠近。

關鍵字搜尋
搜尋「貓」→ 只找到含「貓」字的文件
「寵物」「貓咪」「虎斑」不會出現
優點：速度快、結果可預測
缺點：同義詞、換說法就找不到
語義搜尋（向量搜尋）
搜尋「貓」→ 找到所有語義相近的段落
「寵物」「動物陪伴」「喵星人」都可能出現
優點：理解語義，跨語言也能搜
缺點：需要 Embedding 模型，速度稍慢
(02) · What is Embedding

Embedding 的直覺：
把文字變成方向

Embedding 就是把文字轉換成一串數字（向量），這串數字代表這段文字的「語義位置」。你不需要理解數學，只需要記住一個直覺：語義相近的文字，它們的向量方向也相近。

想像每段文字都是宇宙中的一個星球。Embedding 決定每個星球的座標。語義相似的文字是相鄰星球，搜尋就是找離你最近的那幾顆。

實際操作時，你需要做兩件事：

1
把文件轉成向量（索引階段）
把你的知識庫文件切成段落，用 Embedding 模型（例如 OpenAI text-embedding-3-small 或 Anthropic 的 Voyage）把每個段落轉成向量，存入向量資料庫。這個步驟只需要做一次。
2
把查詢問題也轉成向量（搜尋階段）
使用者輸入問題時，用同一個 Embedding 模型把問題轉成向量，然後在資料庫裡找「向量最近」的幾個段落，把它們一起送給語言模型生成回答。
💡
用來 index 和用來查詢的 Embedding 模型必須一樣。如果你用 text-embedding-3-small 建索引，查詢時也要用 text-embedding-3-small。不同模型的向量空間完全不同，混用結果會很差。
迷你練習 1 約 15 分鐘
跑一次 Embedding 語義相似度實驗

用 OpenAI 或 Voyage 的 Embedding API 把 5 組句子轉成向量，親眼看到「意思近 → 距離近，意思遠 → 距離遠」的對應關係，建立物理直覺。

1準備 5 組句子：其中 2~3 組內部語義相近（例：「我喜歡貓」vs「我家有隻可愛喵星人」），其餘互相無關。
2用 text-embedding-3-small（或同等模型）把每句話轉成向量。
3兩兩計算 cosine similarity，整理成一張對稱矩陣表。
4檢查：語義相近的配對 similarity 是否真的比較高？有沒有反直覺的例子？
範本素材 懶得自己想句子？這組 5 句中英夾雜的測試集，可直接丟 Embedding API 跑相似度實驗。
原始素材 · 5 句測試資料（3 組刻意設計的語義關係）貼進 Prompt 最下方
編號 | 句子 S1 | 我家養了一隻橘色虎斑貓，牠很黏人 S2 | My orange tabby cat loves cuddles and follows me everywhere S3 | 拿鐵需要蒸奶泡，溫度建議控制在 60 到 65 度 S4 | I make lattes every morning with frothed milk at 60–65°C S5 | 今天台北下大雨，通勤族記得帶傘 預期關係： - S1 與 S2：同義不同語言（中英）→ similarity 應 > 0.7 - S3 與 S4：同義不同語言（中英）→ similarity 應 > 0.7 - S1 與 S3：不同主題（貓 vs 咖啡）→ similarity 應 < 0.4 - S5 與其他：完全無關 → similarity 應 < 0.3
範例 Prompt · Embedding 實驗腳本產生器可直接貼 ChatGPT / Claude
你是一位 Python ML 工程師。請幫我產出一段最精簡的 Embedding 相似度實驗腳本。 需求： 1. 使用 OpenAI text-embedding-3-small（若沒有 API key 改用 sentence-transformers 的 paraphrase-multilingual-MiniLM-L12-v2） 2. 把下方 5 句話全部 embed 成向量 3. 計算 5×5 cosine similarity 矩陣，用 numpy 或 sklearn 4. 以 pandas DataFrame 印出矩陣（index / columns 為 S1~S5） 5. 額外印出「預期關係 vs 實際 similarity」的驗證結果：每條預期是否成立（True/False + 實際分數） 輸出規範： - 不超過 40 行 - 每段關鍵處加一行中文註解 - 結尾附一段 5 行內的「觀察指引」：如果 S1-S2 相似度偏低可能是什麼原因？ 句子： [貼上上方原始素材]
預期輸出片段（供對照）
# 1. 讀 API key 並初始化 client
client = OpenAI()
# 2. 批次取得 embeddings
vecs = [client.embeddings.create(model='text-embedding-3-small', input=s).data[0].embedding for s in sentences]
# 3. cosine similarity 矩陣
sim = cosine_similarity(np.array(vecs))
print(pd.DataFrame(sim, index=['S1','S2','S3','S4','S5'], columns=['S1','S2','S3','S4','S5']).round(3))
# 典型結果：sim[S1,S2] ≈ 0.72，sim[S3,S4] ≈ 0.78，sim[S1,S3] ≈ 0.18
# 觀察指引：若 S1-S2 偏低（< 0.5），可能是模型跨語言能力不足，建議改用 text-embedding-3-large 或 voyage-multilingual-2
成果產出
5×5 similarity 矩陣截圖 + 一段觀察筆記（「最高 0.XX 是哪兩句，最低 0.XX 是哪兩句，有沒有超乎預期」）。

(03) · Vector Databases

向量資料庫：
存哪裡、用哪個

向量資料庫專門用來儲存和查詢高維向量。你不需要了解它的內部實作，只需要知道：它很快（幾毫秒內在百萬向量裡找到最近 k 個）、它通常也支援 metadata 過濾（下節課會用到）。

工具適用情境起步難度備注
Chroma本機開發、小型原型低（pip install）完全本機，無需帳號
Supabase pgvector已用 Supabase 的專案低（SQL 擴充）關聯資料庫 + 向量合一
Pinecone需要雲端託管、高可用中（需要帳號）有免費方案，最常被提到
Qdrant自建或雲端都需要中開源，效能好，支援 Rust
ℹ️
初學者建議：從 Chroma 開始。它不需要任何帳號設定，一行 pip install 就能用，適合理解概念和建立原型。等你的應用需要雲端部署了，再考慮遷移到 Supabase 或 Pinecone。
迷你練習 2 約 20 分鐘
用 Chroma 建一個最小向量資料庫

用 Chroma 在本機建一個存有 10 段文字的向量資料庫，丟 3 個查詢進去，觀察 top-3 返回結果是否合理。

1pip install chromadb，建立一個 persistent client 與一個 collection。
2準備 10 段主題混合的短文（例：3 段講咖啡、3 段講狗、4 段講其他），用 add() 塞進 collection。
3寫 3 個查詢（例：「拿鐵怎麼泡」「我想養寵物」「無關問題」），各取 top-3 結果。
4評估：相關查詢有沒有抓到正確主題？無關查詢的 top-1 distance 是否明顯較高？
範本素材 10 段短文 + 3 個查詢 + 3 個誘餌查詢，直接餵 Chroma 跑 top-3 retrieval。
原始素材 · 10 段主題混合短文 + 6 組查詢（約 420 字）貼進 Prompt 最下方
【Collection 要 add 的 10 段文字】 D1 拿鐵咖啡的做法：用義式濃縮搭配蒸奶泡，蒸奶溫度建議 60–65°C D2 手沖咖啡需要注意水溫，淺焙豆用 92°C，深焙豆用 88°C D3 摩卡咖啡加了巧克力醬，熱量比拿鐵高 D4 養狗前要評估活動量，柴犬每天需要 1 小時散步 D5 柯基犬的短腿容易造成椎間盤問題，體重控制是關鍵 D6 貓咪普遍比狗獨立，但橘貓特別黏人 D7 台北捷運板南線尖峰時段人潮集中在市政府站到國父紀念館 D8 台北松山機場停車費每小時 60 元，首次半小時免費 D9 Python 的 list comprehension 可以取代簡單的 for loop D10 JavaScript 的 async/await 本質是 Promise 的語法糖 【6 組查詢】 Q1 拿鐵怎麼泡（預期命中 D1、可能帶到 D2 D3） Q2 我想養寵物但是家裡不大（預期命中 D4 D5 D6） Q3 async 跟 promise 有什麼關係（預期命中 D10） Q4 量子電腦是什麼（誘餌題，distance 應偏高） Q5 機場停車多少錢（預期命中 D8） Q6 今天午餐吃什麼（誘餌題，無相關資料）
範例 Prompt · Chroma 最小腳本產生器可直接貼 ChatGPT / Claude
你是一位熟悉 LangChain / Chroma 的 Python 工程師。請產出一段最小可執行腳本，完成以下任務。 需求： 1. 使用 chromadb.PersistentClient，路徑 ./chroma_store 2. 建立（或取得）名為 demo 的 collection，採用 Chroma 預設 embedding function（all-MiniLM-L6-v2） 3. 把下方 10 段文字用 collection.add() 塞入，ids 為 D1~D10 4. 對下方 6 組查詢各跑一次 collection.query(n_results=3)，印出： - 查詢編號 + 查詢文字 - top-3 的 id、distance（四捨五入 3 位）、前 30 字預覽 5. 最後印出一張表：Q1~Q6 vs top-1 distance，並標記「命中」(< 0.4)、「勉強相關」(0.4–0.7)、「應視為無答案」(> 0.7) 輸出規範： - 單檔 Python，含 import - 不超過 50 行 - 結尾附一段 3 行內的判斷指引：「若 Q4 Q6 的 top-1 distance < 0.5 代表什麼？」 資料： [貼上上方原始素材]
預期輸出片段（供對照）
import chromadb
client = chromadb.PersistentClient(path='./chroma_store')
col = client.get_or_create_collection('demo')
col.add(ids=[f'D{i+1}' for i in range(10)], documents=docs)
for qi, q in enumerate(queries, 1):
  r = col.query(query_texts=[q], n_results=3)
  print(f'Q{qi} {q}', list(zip(r["ids"][0], [round(d,3) for d in r["distances"][0]])))
典型結果：
Q1 top-1 D1 dist 0.21（命中）
Q3 top-1 D10 dist 0.28（命中）
Q4 top-1 D9 dist 0.82（應視為無答案）
Q6 top-1 D3 dist 0.91（誘餌題正確被識別）
判斷：若 Q4 Q6 dist < 0.5，代表 embedding 空間對這類查詢區分度不夠，需要更高品質模型或加 threshold 過濾。

(04) · RAG vs Long Context

RAG 還是長 Context？
選對工具省一半力氣

Claude 有 200K token 的 context window，Gemini 更高達 1.5M。這讓一個問題變得有意義：我到底需要建 RAG 系統，還是直接把所有文件塞進 context 就好？

決策指南 · WHEN TO USE WHAT
文件 < 50 頁
直接塞進 context。省時省力，不需要建索引。
文件 50–500 頁
視情況。如果呼叫頻率高，長 context 成本貴；如果只是偶爾查詢，context 更簡單。
文件 > 500 頁
RAG。長 context 成本太高，而且真的超過 context window 了。
文件持續更新
RAG（有 incremental indexing）。長 context 每次都要重新載入所有內容。
需要引用來源
RAG。可以在回答裡附上「來自第幾頁第幾段」的 metadata。
🔑
RAG 的代價是複雜度。你多了 Embedding 步驟、向量資料庫、Chunking 策略、Retrieval 品質調優。如果長 context 能解決問題，就不要加這些複雜度。
