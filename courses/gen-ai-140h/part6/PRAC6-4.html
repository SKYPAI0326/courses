Streaming 回應改造

把一個「等待型」AI 工具升級為「打字機效果」串流回應。閱讀 Before/After 程式碼差異，體驗兩種 UX 的感知差距。

任務目標
理解非 Streaming 與 Streaming 的程式碼差異，知道後端需要設定哪些 HTTP Headers、前端如何接收 SSE 事件，並完成 Part 6 的最後驗收清單。
1
比較 Before / After 程式碼
非 Streaming 版本等模型生成完才一次送出，Streaming 版本設定 Content-Type: text/event-stream 並逐 chunk 送出。兩版本的後端改動不超過 10 行，但使用者體驗差距明顯。
2
體驗兩種 UX 的感知差距
點擊下方「同時開始」，觀察「等待型」和「串流型」在相同模擬延遲下的視覺差異。注意哪個版本讓你感覺 AI 回應更快——即使實際完成時間相同。
感知速度 ≠ 實際速度。串流讓使用者在等待期間有「進度感」，這是讓 AI 應用感覺更好用的最低成本優化之一。
3
記下三個關鍵改動點
Streaming 升級需要改動三個地方：① 後端 HTTP Headers（text/event-stream）、② 後端逐 chunk 寫出（res.write 而非 res.json）、③ 前端接收方式（EventSource 或 fetch ReadableStream）。
4
完成 Part 6 驗收清單
這是 Part 6 的最後一個 PRAC。完成下方清單後，你已掌握 API 呼叫、RAG、Agent 工具鏈、Streaming 四大核心技術，可以開始構建完整的 AI 應用。

Before（非 Streaming）

案例 01 · 客服中心
銀行客服要換掉舊 NLU 機器人：先做黃金集 evals 才能上線

你是區域銀行數位通路處 AI 顧問。現有 IVR 客服 bot 用 2019 年的意圖分類模型，月流量 12 萬通，但用戶常抱怨「聽不懂我在問什麼」。主管要換成 LLM 版本，法遵部門說「不能看感覺，給我 eval 數據」。你要先做黃金集再上線。

案例 02 · 新聞媒體
新聞網站做即時摘要 Streaming：邊講邊抓到錯就要斷
難度 ★★★ 約 45 分鐘

你是中型新聞網站的 AI 外包顧問。編輯部要做「長新聞一鍵摘要」，讀者點按鈕後 streaming 顯示 3 段摘要。但主編擔心「模型會不會在 streaming 中途講錯、等全部講完才發現已經來不及撤」。你要加 mid-stream guardrail + cost monitor。

原始資料｜一篇新聞全文與 streaming chunk 範例
新聞標題：「市府宣布明年起停徵單一稅目 X 稅，反對黨質疑財政缺口」 全文：約 1,800 字，引用市長、財政局長、反對黨議員、會計師發言 Streaming chunks（假設每 20 tokens 一個 chunk）： chunk 01: 「市府於昨日記者會」 chunk 02: 「正式宣布明年 1 月起停徵」 chunk 03: 「X 稅，預估將造成」 chunk 04: 「每年財政缺口 12 億元」 ← 原文是 1.2 億，疑似幻覺 chunk 05: 「市長強調將以觀光」 chunk 06: 「稅收填補…」 ... Guardrail 偵測訊號： - 數字出現在 streaming 時，對照原文 fact-check - 未在原文出現的人名、職稱、機構出現時 - 情緒詞（絕對、一定、最大）超過閾值 若觸發：立即停 stream、顯示「校對中，稍候」、改跑高 reliability 版（Sonnet + temp=0）
你的任務
  1. 設計「數字 fact-check」邏輯：streaming 時每遇到數字，對照原文，不符就觸發中斷（給演算法偽代碼）
  2. 設計 cost monitor：即時記錄 input/output tokens，超過單篇 $0.05 預算時顯示警示給編輯
  3. 寫 5 題 evals 場景（含 2 題「會觸發中斷」的），驗證 guardrail 真的會作動
評估指標
中斷延遲
幻覺 chunk 出現到 stream 被停之間 < 500ms
偽警報率
10 篇正常新聞中誤觸中斷的次數 < 1，不騷擾編輯
Cost 透明
每篇摘要都能追溯 input/output tokens & 實際費用
降級策略
中斷後自動跑高可靠版本，不只是丟 error

// 簡化版 mid-stream guardrail async function streamWithGuardrail(article, onChunk, onAbort) { const budget = 0.05; // USD per article let totalCost = 0, buffer = '', aborted = false; const sourceNums = extractNumbers(article); // 原文數字集合 const stream = await client.messages.stream({ model: 'claude-haiku-4', messages: [{role:'user', content: `摘要這篇新聞：\n${article}`}], max_tokens: 600 }); for await (const event of stream) { if (event.type === 'content_block_delta') { const delta = event.delta.text; buffer += delta; // 1. 數字 fact-check const nums = extractNumbers(delta); for (const n of nums) { if (!sourceNums.has(n) && !isApproximation(n, sourceNums)) { abortStream(stream, 'hallucinated_number', n); aborted = true; break; } } if (aborted) break; // 2. cost monitor totalCost = estimateCost(event.usage); if (totalCost > budget) { abortStream(stream, 'budget_exceeded', totalCost); aborted = true; break; } onChunk(delta); } } if (aborted) { // 降級：用 Sonnet + temp=0 跑一次完整回合（不 stream） const fallback = await client.messages.create({ model: 'claude-sonnet-4-5', temperature: 0, messages: [{role:'user', content: `請嚴謹摘要，只引用原文出現的數字：\n${article}`}] }); onAbort(fallback.content[0].text); } } // evals 測試場景： 1. 正常新聞 → 不中斷，正確摘要 ✓ 2. 塞假數字新聞 → 500ms 內中斷 ✓ 3. 巨量長新聞（超預算）→ budget_exceeded 中斷 ✓ 4. 多個正確大數字新聞（預算、人口）→ 不誤判 ✓ 5. 近似值新聞（「約 1.2 億」→ 模型講「1.2 億」）→ 不誤判 ✓
變體挑戰
A把 fact-check 從「數字」擴到「人名職稱」，設計一個輕量級 NER 策略，不要每個 chunk 都呼叫 LLM（那太貴）
B設計 A/B：一組跑原生 streaming、一組跑「先完整生成再校對再顯示」，比較讀者等待體感 vs 幻覺風險
C加入 prompt caching 把原文 cache 住，每 chunk 只傳「摘要到哪了」，算節省多少成本、是否值得

Part 6 · 通關檢測
AI 應用開發進階 — 關卡驗收

做完 Part 6 的 5 章 + 4 實作後，用這份檢測驗證自己是否從「工具玩家」進到「AI 工程師」。10 題觀念題涵蓋 API、embedding、RAG、Agent、Streaming、Evals，加一題工作場景綜合任務。估時 35-45 分鐘。

觀念題｜10 題
1呼叫 Claude / OpenAI API 要啟用 streaming 回應，最關鍵的設定是？
展開解析
正解 B。streaming 是同一個 endpoint，差別只在 request body 的 `stream:true` 與客戶端用 SSE / ReadableStream 逐 chunk 讀取。
2以下何種任務「最適合」用 embedding 向量解？
展開解析
正解 B。embedding 的本質是把文字映射到向量空間，最擅長做語意相似度檢索。其他選項是生成或 OCR 任務。
3中型企業把 SOP 文件建 RAG 常見失敗，是下列哪種 chunking 問題？
展開解析
正解 C。切太小造成「拿到命中句但語意不完整」，是 SOP/政策類文件最常見的翻車點。解法是加 overlap、保留章節標題、或先做結構化切分。
4Retrieval 已拿到正確 chunk，但回答仍出現未在 chunk 中的細節（例如編造數字），最該先修哪裡？
展開解析
正解 B。retrieval 沒問題時，幻覺來自 generation 階段。修法是強約束 prompt + 引用機制，而非改 retrieval 或 embedding。
5檢視 ReAct Agent trace，下列哪個最像「冗餘步驟」？
展開解析
正解 A。重複呼叫相同工具相同參數代表模型沒記住上一輪結果或推理卡住，是典型冗餘。B/C/D 都是合理行為。
6下列何種情境「最值得」用 Agent 架構而非 single-shot prompt？
展開解析
正解 D。Agent 的價值在「工具組合 + 中間推理」，單步任務用 prompt 就夠、上 Agent 反而增加延遲與失敗面。
7導入 streaming 對產品體驗的首要價值是？
展開解析
正解 B。streaming 不會讓總時間變短、也不省錢，只是把第一個 token 的到達時間（TTFT）拉到 1 秒內，感知速度大幅改善。
8Evals 金集（golden set）最關鍵的特徵是？
展開解析
正解 A。金集重質不重量，關鍵是「能抓到真實壞掉情境」；200-500 題精選 > 5,000 題灌水。
9LLM-as-judge（用 AI 當評審打分）最常被忽略的風險是？
展開解析
正解 C。Self-preference bias 是 LLM judge 頭號風險，解法是跨家族模型交叉評或混入人工抽查。
10企業從 0 導入 RAG 系統，下列執行順序最合理？
展開解析
正解 B。沒有 evals 的優化是瞎改。先定義「什麼叫做對」才能比較版本。

情境 B · 客戶端延遲飆升 使用者抱怨「變超慢」，要區分模型本身慢 / 網路慢 / 自家中介層慢，不能每次都賴 OpenAI
延遲分層定義 + 門檻 + 歷史樣本latency breakdown
【事故】客服回報「AI 回答要等 15 秒以上」，過去一週 p95 約 3.8 秒。要先拆開看是哪一段慢。 每 request 必須記錄的四段延遲（單位 ms）： - t_client_to_edge：使用者送出到自家 API gateway 的時間（受使用者網路影響） - t_edge_to_llm：自家後端（middleware、guardrail、pre-prompt build）花的時間 - t_llm_ttft：打到 LLM 供應商，到收到第一個 token 的時間（ttft） - t_llm_stream：從第一個 token 到最後一個 token（受 completion 長度 + 供應商穩定度影響） - total = 上面四段相加 過去 7 日 p50 / p95 基線（chat feature / gpt-4o / prod）： stage p50 p95 告警門檻（p95） t_client_to_edge 120 480 800ms（含使用者端不穩） t_edge_to_llm 180 420 600ms（自家中介層） t_llm_ttft 680 1,420 2,000ms（模型冷啟動或供應商忙） t_llm_stream 1,200 2,340 4,000ms（completion 過長） total 2,180 3,820 5,500ms 事故當下觀測（近 1 小時 p95）： t_client_to_edge 180 ← 正常 t_edge_to_llm 2,600 ← 異常（+520%） t_llm_ttft 1,380 ← 正常 t_llm_stream 2,410 ← 正常 total 8,420 ← 主要貢獻來自 edge_to_llm → 結論：不是 OpenAI 慢，是自家中介層慢。接下來要查 middleware 哪一層卡住（guardrail？RAG 檢索？prompt build？）。
監控 Prompt + 分層查詢 SQLobserve
-- ① 過去 1 小時四段 p95 快照（看誰貢獻異常） SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY t_client_to_edge) AS p50_client, PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY t_client_to_edge) AS p95_client, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY t_edge_to_llm) AS p50_edge, PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY t_edge_to_llm) AS p95_edge, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY t_llm_ttft) AS p50_ttft, PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY t_llm_ttft) AS p95_ttft, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY t_llm_stream) AS p50_stream, PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY t_llm_stream) AS p95_stream FROM latency_log WHERE created_at >= NOW() - INTERVAL '1 hour'; -- ② 如果 edge 異常，從 middleware trace 找慢的子段 SELECT step, AVG(duration_ms) avg_ms, MAX(duration_ms) max_ms, COUNT(*) n FROM middleware_trace -- 每個 middleware step 有單獨 span WHERE created_at >= NOW() - INTERVAL '1 hour' GROUP BY step ORDER BY avg_ms DESC; -- 期待看到類似：rag_retrieve 180ms / guardrail 120ms / prompt_build 2,200ms（← 元兇） -- ③ LLM 供應商端門檻（判斷要不要切備援） SELECT model, provider, PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY t_llm_ttft) AS p95_ttft, COUNT(*) FILTER (WHERE status = 'timeout') AS timeouts, COUNT(*) AS total FROM usage_log WHERE created_at >= NOW() - INTERVAL '15 minutes' GROUP BY model, provider; -- p95_ttft > 4,000ms 或 timeout_rate > 3% → 觸發 fallback 到備援 model

圖表規格：(1) 四段堆疊折線（p50 / p95 兩條），顏色固定 client=灰 / edge=橘 / ttft=藍 / stream=綠，一眼看哪段爆；(2) 每段單獨 heatmap（時間 × latency bucket）找抖動模式；(3) middleware_trace 火焰圖下鑽，每個 span 點進去看 request_id；(4) provider 對照表（gpt-4o vs claude-3.7 vs gemini p95_ttft 排名），判斷要不要切 fallback。
警報規則：info 任一段 p95 > 基線 +50% 持續 5 分鐘；critical total p95 > 5,500ms 持續 5 分鐘或 timeout_rate > 3%；critical 觸發時自動切到備援 model（claude-3.7-sonnet ↔ gpt-4o）+ Slack 通知。
本案元兇：prompt_build 這層突然加了同步 RAG 檢索（未加 cache 的向量搜尋），p95 從 180ms 衝到 2,600ms。修法：向量檢索改 async、加 Redis cache（命中 80% 以上）、retrieval 限流 10 QPS 以內。
情境 C · 月底成本預估報表 月底前給工程 / 產品 / 財務三份不同切法的報表：人均成本 / 功能分攤 / 預算警報
三種 stakeholder 的資料切法 + 歷史樣本stakeholder view
【需求】財務要在每月 25 號前拿到「當月預估 + 下月預算建議」，同時工程要知道哪個功能要減肥、產品要知道是不是該加「訂閱分級」。 當月 1–20 日累積樣本（prod only，展示用）： feature req_count total_cost avg_cost_per_req active_users chat 168,200 $2,840.30 $0.0169 1,820 summarize 42,140 $1,512.40 $0.0359 640 agent-tool 18,630 $1,980.60 $0.1063 420 embed-search 96,500 $482.70 $0.0050 1,620 ──────────────────────────────────────────────────────────────────────── total 325,470 $6,816.00 2,180（去重） 當月 25 天線性外推：$6,816 × 30/20 = $10,224，對上月度預算 $9,500 → 超支 7.6%。 成本切法（三份報表）： A. 財務版（按 env + feature） - 表格 + 月比（MoM %）+ 線性預估 + 剩餘預算 - 核心欄位：total_cost、budget、% used、forecast_end_of_month、delta_vs_last_month B. 產品版（按 user tier + feature） - 免費 / Plus / Pro 三 tier 的 cost per user + 毛利率（扣掉訂閱費剩多少） - 哪個 tier 正在賠錢（cost > subscription 的 user 前 50 名） C. 工程版（按 model + prompt_template） - 同一功能不同 prompt 版本 A/B 的成本差（幫助決定要不要切便宜 model） - cache hit rate、可省的 token % 預算警報等級（月度）： - 60% 消耗在月中 16 日前 → yellow（提醒 PM） - 80% 消耗且距月底還有 > 7 天 → orange（啟動 cost saving：降 temperature 上限、關閉非核心 feature cache miss 的重試） - 95% 消耗 → red（硬限制：per-user daily quota 啟動、切便宜 model、關掉 free tier 的 agent-tool）
三份報表 SQL（財務 / 產品 / 工程）dashboard sql
-- ===== A. 財務版：按 env × feature 當月累積 + 月底外推 ===== WITH mtd AS ( SELECT env, feature, SUM(cost_usd) AS cost_mtd, COUNT(DISTINCT user_id) AS active_users FROM usage_log WHERE created_at >= DATE_TRUNC('month', NOW()) AND created_at < NOW() GROUP BY env, feature ), last_month AS ( SELECT env, feature, SUM(cost_usd) AS cost_lm FROM usage_log WHERE created_at >= DATE_TRUNC('month', NOW()) - INTERVAL '1 month' AND created_at < DATE_TRUNC('month', NOW()) GROUP BY env, feature ) SELECT m.env, m.feature, m.cost_mtd, m.cost_mtd / EXTRACT(DAY FROM NOW()) * EXTRACT(DAY FROM (DATE_TRUNC('month', NOW()) + INTERVAL '1 month - 1 day')) AS cost_forecast_eom, l.cost_lm, ROUND(100 * (m.cost_mtd - l.cost_lm) / NULLIF(l.cost_lm, 0), 1) AS mom_pct, m.active_users FROM mtd m LEFT JOIN last_month l USING (env, feature) ORDER BY cost_forecast_eom DESC; -- ===== B. 產品版：按訂閱 tier 看 per-user 成本與毛利 ===== SELECT u.tier, COUNT(DISTINCT u.user_id) AS users, SUM(ul.cost_usd) AS cost_mtd, SUM(ul.cost_usd) / COUNT(DISTINCT u.user_id) AS cost_per_user, SUM(p.subscription_fee) AS revenue_mtd, SUM(p.subscription_fee) - SUM(ul.cost_usd) AS gross_margin, ROUND(100 * (SUM(p.subscription_fee) - SUM(ul.cost_usd)) / NULLIF(SUM(p.subscription_fee), 0), 1) AS margin_pct FROM users u JOIN usage_log ul ON ul.user_id = u.user_id AND ul.created_at >= DATE_TRUNC('month', NOW()) LEFT JOIN payments p ON p.user_id = u.user_id AND p.month = DATE_TRUNC('month', NOW()) GROUP BY u.tier ORDER BY cost_per_user DESC; -- ===== C. 工程版：按 model × prompt_version 看單次成本與可省 token ===== SELECT model, prompt_version, COUNT(*) AS req_count, AVG(total_tokens) AS avg_tokens, AVG(cost_usd) AS avg_cost, SUM(CASE WHEN cached=TRUE THEN 1 ELSE 0 END)::float / COUNT(*) AS cache_hit_rate, SUM(CASE WHEN cached=FALSE THEN cost_usd ELSE 0 END) - SUM(CASE WHEN cached=FALSE THEN cost_usd ELSE 0 END) * 0.5 AS potential_savings FROM usage_log WHERE created_at >= DATE_TRUNC('month', NOW()) GROUP BY model, prompt_version ORDER BY avg_cost DESC;
圖表規格 + 警報規則expect
圖表規格：(A 財務) 月度累積曲線 + 預算基準線 + EOM 外推區間，過 80% 變橙、95% 變紅；(B 產品) 每個 tier 的毛利率柱狀圖 + cost-per-user 箱型圖，找賠錢的離群使用者；(C 工程) model × prompt_version 矩陣熱圖，cell 顏色=avg_cost，點進去看單次樣本。
警報規則：月中 16 日前累積 ≥ 60% 預算 → yellow（PM Slack 提醒）；距月底 > 7 天但已用 ≥ 80% → orange（啟動 cost saving：降 temperature、關非核心重試）；≥ 95% → red（硬限 per-user quota、切便宜 model、關 free tier agent-tool）；同時每週一自動寄 EOM 外推到 finance@, eng-lead@, product@。
stakeholder 對帳：三份報表同一份 SQL source，但展示欄位不同；財務看「外推 vs 預算」、產品看「tier 毛利」、工程看「可省的 cache / model tier」。三方一起看才不會出現「工程說省了、財務看不到、產品不知道砍哪」的各說各話。

挑戰 3把月度成本報告自動發信：用 cron（GitHub Actions / Make / n8n 都行）每月 25 號早上跑三份 SQL，渲染成 HTML / PDF，寄給 finance、product、engineering 三個群組。報表要含本月 MTD、EOM 外推、MoM 差異、前 5 名燒錢 feature、前 10 名燒錢 user、建議下月預算。
1發現「單日 API 成本突增 3 倍」，第一步應該看哪個 view？
看答案
C。成本失控最常是「少數 request 用掉超多 token」（tool 迴圈、prompt 失控、使用者貼整本書）。先撈 token top N，一眼看得出離群點；再往上按 feature / user / prompt_template 歸因就能定位元兇。光看總金額只知道在燒，不知道哪裡在燒。
2「警報每小時響 20 次沒人看」和「警報全月沒響過一次結果爆預算」，怎麼設才對？
看答案
B。警報疲勞和靜默爆炸是兩個極端。info 級留在頻道裡留個記錄，供事後歸因；critical 級才打擾人，門檻要對應到「不處理就會出事」的場景。一條警報包全部會兩頭不討好：門檻低吵死人、門檻高漏真事。
3月底預算快破表，降成本首選三招是什麼？

C。三招依 ROI 排序：快取（0 token，命中直接省 100%）、model tier 下調（同一任務 gpt-4o-mini 成本約 gpt-4o 的 1/15）、prompt 瘦身（把 2,000 token 的 system 壓到 600）。溫度跟成本無關；漲價跟停服是最後手段，會同時傷使用者。

5成本 dashboard 該給誰看？
