PART 6 · AI 應用開發進階
PRAC 6.4 · 1.5 小時

Streaming 回應改造

把一個「等待型」AI 工具升級為「打字機效果」串流回應。閱讀 Before/After 程式碼差異，體驗兩種 UX 的感知差距。

任務目標
理解非 Streaming 與 Streaming 的程式碼差異，知道後端需要設定哪些 HTTP Headers、前端如何接收 SSE 事件，並完成 Part 6 的最後驗收清單。最低完成線是：看到逐段輸出、說出 streaming 流程、交出一張 Part 6 核心技術回顧卡。
GUIDED PATH · 先完成最小成功畫面
先看到逐段輸出，再補上失敗回復
主線 20 分鐘 先體驗再改造

第一輪只比較等待型與串流型。成功標準是串流欄位會逐段長出文字，而不是等全部完成才一次出現；若串流失敗，也知道可以回退成一般非串流回覆。環境跑不起來時，至少交出 Before / After 差異截圖或程式碼標註。

最小成功畫面
操作：點擊「同時開始」或頁面中的 streaming demo 按鈕。 畫面至少要出現： 1. 等待型欄位先維持等待狀態 2. 串流型欄位逐段顯示回覆文字 3. 游標或狀態文字顯示回覆仍在進行 4. 完成後能說明若 stream 中斷，應回復到非 streaming 結果或顯示重試訊息
結果驗證卡
成功畫面出現
串流欄位會一段一段增加文字，完成前不是空白等待。
輸入可替換
換成其他 prompt 時，仍能逐段顯示對應內容。
錯誤看得見
stream 斷線或後端失敗時，畫面有錯誤或回復狀態，不會停在半截文字。
一句話說明流程
能說出「後端分段送資料，前端逐段讀取並追加到畫面」。
老師檢查證據
請截圖保留這些欄位： 1. 等待型欄位：demo-before、status-before，需看得到等待中與完成後文字。 2. 串流型欄位：demo-after、status-after，需看得到逐段累積、游標 / loading 狀態，以及完成後完整文字。 3. 若有真實 API：Network 截圖需看得到 status 200、Content-Type: text/event-stream、多段 data: chunks、完成訊號與 usage / token 或後端 log。
最低完成線與卡住排查
Level 1 / Level 2 完成線
Level 1 模擬成功：本頁 demo 逐段輸出，能交等待型 / 串流型前後截圖、status-before / status-after、loading 或游標、完成後文字。 Level 2 真實成功：真實 /api/stream 回 HTTP 200，Content-Type 是 text/event-stream，Network 可看到多段 data: chunks，最後有完成訊號，並保留 usage / token 或後端 log。Level 1 模擬畫面不可當成 Level 2 真 API 成功。
失敗替代交付
若 SSE 失敗，交 3 件保底成果： 1. 錯誤 status 截圖（401 / 429 / 500 / CORS / network error 任一）與原因判斷。 2. fallback 非串流 response 截圖，證明使用者仍能拿到完整回答或友善錯誤。 3. 預期 chunk 假資料：至少 3 段 data: 片段、[DONE] 完成訊號、你預期畫面如何逐段追加。
1
比較 Before / After 程式碼
非 Streaming 版本等模型生成完才一次送出，Streaming 版本設定 Content-Type: text/event-stream 並逐 chunk 送出。兩版本的後端改動不超過 10 行，但使用者體驗差距明顯。
2
體驗兩種 UX 的感知差距
點擊下方「同時開始」，觀察「等待型」和「串流型」在相同模擬延遲下的視覺差異。注意哪個版本讓你感覺 AI 回應更快——即使實際完成時間相同。
感知速度 ≠ 實際速度。串流讓使用者在等待期間有「進度感」，這是讓 AI 應用感覺更好用的最低成本優化之一。
3
記下三個關鍵改動點
Streaming 升級需要改動三個地方：① 後端 HTTP Headers（text/event-stream）、② 後端逐 chunk 寫出（res.write 而非 res.json）、③ 前端接收方式（EventSource 或 fetch ReadableStream）。
4
完成 Part 6 驗收清單
這是 Part 6 的最後一個 PRAC。完成下方清單後，你已掌握 API 呼叫、RAG、Agent 工具鏈、Streaming 四大核心技術，可以開始構建完整的 AI 應用。
程式碼差異：Before → After

後端 Express 路由的改造：

Before（非 Streaming）
app.post('/api/ask', async (req, res) => { const msg = await client.messages.create({ model: 'claude-haiku-4-5-20251001', max_tokens: 1024, messages: [{ role: 'user', content: req.body.prompt }] }); res.json({ result: msg.content[0].text }); });
After（Streaming）
app.post('/api/stream', async (req, res) => { res.setHeader('Content-Type', 'text/event-stream'); res.setHeader('Cache-Control', 'no-cache'); res.setHeader('Connection', 'keep-alive'); const stream = await client.messages.stream({ model: 'claude-haiku-4-5-20251001', max_tokens: 1024, messages: [{ role: 'user', content: req.body.prompt }] }); for await (const chunk of stream) { if (chunk.type === 'content_block_delta') res.write(`data: ${chunk.delta.text}\n\n`); } res.write('data: [DONE]\n\n'); res.end(); });

前端接收方式（EventSource）：

// 前端：用 fetch 接收 Streaming（支援 POST） const res = await fetch('/api/stream', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ prompt: userInput }) }); const reader = res.body.getReader(); const decoder = new TextDecoder(); let output = ''; while (true) { const { done, value } = await reader.read(); if (done) break; const text = decoder.decode(value); text.split('\n').forEach(line => { if (line.startsWith('data: ') && line !== 'data: [DONE]') { output += line.slice(6); outputEl.textContent = output; // 即時更新 DOM } }); }

延伸案例

Streaming 改造練完後，用兩個上線情境走 eval 與成本控管：黃金集替換舊系統、串流 + guardrail 中斷。重點是「敢替換舊版」「敢在模型講錯時拉斷」。每個案例約 45 分鐘。

案例 01 · 客服中心
銀行客服要換掉舊 NLU 機器人：先做黃金集 evals 才能上線
難度 ★★★ 約 45 分鐘

你是區域銀行數位通路處 AI 顧問。現有 IVR 客服 bot 用 2019 年的意圖分類模型，月流量 12 萬通，但用戶常抱怨「聽不懂我在問什麼」。主管要換成 LLM 版本，法遵部門說「不能看感覺，給我 eval 數據」。你要先做黃金集再上線。

原始資料｜黃金集抽樣（50 題中節錄 6 題）
ID 類別 用戶問題 期望行為 G-01 卡片掛失 我皮夾掉了信用卡不見了怎麼辦 轉卡片掛失專線 & 確認最後使用紀錄 G-02 餘額查詢 我戶頭剩多少 身份驗證 → 回餘額 G-03 轉帳問題 昨天轉給我媽的錢一直沒到 查交易紀錄 & 判斷狀態 → 指引後續 G-04 利率諮詢 現在定存一年利率多少 回最新公告利率 & 比較方案 G-05 投訴 你們那個分行的櫃員超兇我要投訴 情緒安撫 → 轉客訴專員 & 留案號 G-06 邊界/非業務 今天天氣真好對吧 禮貌引導回金融業務 & 不閒聊 評估欄位： - intent_correct：意圖分類是否正確 - action_correct：採取的動作是否正確（例：掛失類有無轉專線） - tone：語氣是否合適（投訴類不能冷冰冰） - safety：有沒有洩漏個資或給錯金融建議 舊 bot baseline（50 題人工評）： intent 71%、action 65%、tone 不評估、safety 100%
你的任務
  1. 從 6 大類（卡片 / 餘額 / 轉帳 / 利率 / 投訴 / 邊界）擴充到 30 題 golden set，每類至少 1 題邊界問題（例如「我要取消昨天的轉帳」這種舊 bot 一定失敗的）
  2. 跑 Sonnet 4.5 + Haiku 3.5 兩組評估，用 LLM-as-judge 自動評分 + 10 題人工抽驗對齊
  3. 寫法遵可讀的 eval 報告：表格 + 風險清單 + 上線決策
評估指標
Intent 提升幅度
新 bot vs 舊 71% 的百分點差，含信心區間
Safety 零退步
Safety 不得低於舊 bot 100%（這條是法遵死線）
Judge 對齊
LLM-as-judge 分數與人工分數一致率 > 85%
決策可行
報告最後明確寫「上線 / 不上線 / 分流 X% 上線」

案例 02 · 新聞媒體
新聞網站做即時摘要 Streaming：邊講邊抓到錯就要斷
難度 ★★★ 約 45 分鐘

你是中型新聞網站的 AI 外包顧問。編輯部要做「長新聞一鍵摘要」，讀者點按鈕後 streaming 顯示 3 段摘要。但主編擔心「模型會不會在 streaming 中途講錯、等全部講完才發現已經來不及撤」。你要加 mid-stream guardrail + cost monitor。

原始資料｜一篇新聞全文與 streaming chunk 範例
新聞標題：「市府宣布明年起停徵單一稅目 X 稅，反對黨質疑財政缺口」 全文：約 1,800 字，引用市長、財政局長、反對黨議員、會計師發言 Streaming chunks（假設每 20 tokens 一個 chunk）： chunk 01: 「市府於昨日記者會」 chunk 02: 「正式宣布明年 1 月起停徵」 chunk 03: 「X 稅，預估將造成」 chunk 04: 「每年財政缺口 12 億元」 ← 原文是 1.2 億，疑似幻覺 chunk 05: 「市長強調將以觀光」 chunk 06: 「稅收填補…」 ... Guardrail 偵測訊號： - 數字出現在 streaming 時，對照原文 fact-check - 未在原文出現的人名、職稱、機構出現時 - 情緒詞（絕對、一定、最大）超過閾值 若觸發：立即停 stream、顯示「校對中，稍候」、改跑高 reliability 版（Sonnet + temp=0）
你的任務
  1. 設計「數字 fact-check」邏輯：streaming 時每遇到數字，對照原文，不符就觸發中斷（給演算法偽代碼）
  2. 設計 cost monitor：即時記錄 input/output tokens，超過單篇 $0.05 預算時顯示警示給編輯
  3. 寫 5 題 evals 場景（含 2 題「會觸發中斷」的），驗證 guardrail 真的會作動
評估指標
中斷延遲
幻覺 chunk 出現到 stream 被停之間 < 500ms
偽警報率
10 篇正常新聞中誤觸中斷的次數 < 1，不騷擾編輯
Cost 透明
每篇摘要都能追溯 input/output tokens & 實際費用
降級策略
中斷後自動跑高可靠版本，不只是丟 error

// 簡化版 mid-stream guardrail async function streamWithGuardrail(article, onChunk, onAbort) { const budget = 0.05; // USD per article let totalCost = 0, buffer = '', aborted = false; const sourceNums = extractNumbers(article); // 原文數字集合 const stream = await client.messages.stream({ model: 'claude-haiku-4', messages: [{role:'user', content: `摘要這篇新聞：\n${article}`}], max_tokens: 600 }); for await (const event of stream) { if (event.type === 'content_block_delta') { const delta = event.delta.text; buffer += delta; // 1. 數字 fact-check const nums = extractNumbers(delta); for (const n of nums) { if (!sourceNums.has(n) && !isApproximation(n, sourceNums)) { abortStream(stream, 'hallucinated_number', n); aborted = true; break; } } if (aborted) break; // 2. cost monitor totalCost = estimateCost(event.usage); if (totalCost > budget) { abortStream(stream, 'budget_exceeded', totalCost); aborted = true; break; } onChunk(delta); } } if (aborted) { // 降級：用 Sonnet + temp=0 跑一次完整回合（不 stream） const fallback = await client.messages.create({ model: 'claude-sonnet-4-5', temperature: 0, messages: [{role:'user', content: `請嚴謹摘要，只引用原文出現的數字：\n${article}`}] }); onAbort(fallback.content[0].text); } } // evals 測試場景： 1. 正常新聞 → 不中斷，正確摘要 ✓ 2. 塞假數字新聞 → 500ms 內中斷 ✓ 3. 巨量長新聞（超預算）→ budget_exceeded 中斷 ✓ 4. 多個正確大數字新聞（預算、人口）→ 不誤判 ✓ 5. 近似值新聞（「約 1.2 億」→ 模型講「1.2 億」）→ 不誤判 ✓
變體挑戰
A把 fact-check 從「數字」擴到「人名職稱」，設計一個輕量級 NER 策略，不要每個 chunk 都呼叫 LLM（那太貴）
B設計 A/B：一組跑原生 streaming、一組跑「先完整生成再校對再顯示」，比較讀者等待體感 vs 幻覺風險
C加入 prompt caching 把原文 cache 住，每 chunk 只傳「摘要到哪了」，算節省多少成本、是否值得
