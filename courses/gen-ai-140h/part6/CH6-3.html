Part 6 · AI 應用開發進階
CH 6.3 · 3 小時

RAG 實作與品質調優

知道 Embedding 是什麼之後，下一步是把它串成完整的問答系統。RAG（Retrieval-Augmented Generation）讓 AI 能夠「讀你的文件回答問題」，而不是靠訓練資料猜答案。

學習成果 · OUTCOMES
能描述 RAG 的完整流程：文件讀取→切塊→索引→檢索→生成
理解 Chunking 策略的取捨：大 chunk 保留 context，小 chunk 檢索精準
知道 Metadata 過濾和 Reranking 如何提升回答品質
能辨識 RAG 的三種常見失敗模式並知道如何修正
(01) · RAG Pipeline

RAG 完整流程：
五個必經環節

RAG 分兩個大階段：索引（離線，只做一次）和查詢（線上，每次回答）。理解這個區分，你才知道哪裡出問題要去哪裡修。

01
文件讀取
PDF、Word、網頁轉成純文字
→
02
切塊
Chunking：切成適當大小的段落
→
03
Embedding
每個 chunk 轉成向量，存入資料庫
→
04
檢索
問題轉向量，找最近的 k 個 chunk
→
05
生成
把問題 + 檢索結果送給 LLM 生成答案
💡
01–03 是離線步驟（Indexing）：文件更新才需要重跑。04–05 是線上步驟（Querying）：每次使用者提問都會執行。效能瓶頸通常在 04（向量搜尋速度）和 05（LLM 回應時間）。
(02) · Chunking Strategy

Chunking 策略：
切多大才對？

Chunking 是 RAG 品質最關鍵的決策之一。切太小：每個 chunk 缺乏上下文，AI 找到了但看不懂。切太大：向量無法精確代表語義，找到的東西不夠相關。

固定大小切塊（最常用的起點）
設定 chunk_size=512（tokens），overlap=50（重疊 tokens 保留跨塊語義）。適合大多數文件，起步先用這個，再根據品質調整。
依語意邊界切塊（段落、標題）
按照文件的自然結構切——在換行、標題、段落符號處切割。好處是每個 chunk 語義完整；缺點是 chunk 大小不均。
Markdown-aware 切法（結構化文件）
針對 Markdown 文件，按 ##、### 標題層級切塊，把每個小節當一個 chunk，同時保留父標題作為 metadata。知識庫、技術文件特別適合。
⚠️
重疊（overlap）很重要。如果 chunk A 在第 500 個 token 結束，chunk B 從第 450 個 token 開始，它們共享 50 個 token 的內容。這讓一個跨越切割點的概念不會因為切塊而丟失語義。
迷你練習 1 約 20 分鐘
對同一份文件比較三種 Chunking 策略

拿一份你熟悉的技術文件或 SOP，分別用「固定 256 / 固定 512 / 依標題切」三種方式切 chunk，觀察 retrieval 品質的差異。

1準備一份 2000~5000 字的文件，內容要有明確章節結構。
2用 LangChain 或自己寫 Python 切出三個版本：chunk_size 256+overlap 30、chunk_size 512+overlap 50、Markdown 標題切。
3擬 5 個真實查詢（使用者真的會問的問題），對三個版本各跑一次 top-3 retrieval。
4用「找到 / 找對一半 / 找錯」三檔評分，算出每個策略的命中率，選出當前最適合的方案。
範本素材 手邊沒有 SOP 文件？用這份模擬版「員工請假流程 SOP」當實驗對象，已附 5 組查詢。

(03) · Quality Tuning

品質調優：
Metadata 過濾與 Reranking

基本 RAG「找最近的 k 個向量」只是起點。當你的知識庫大了、複雜了，需要兩個進階技術來提升回答品質。

Metadata 過濾
在每個 chunk 上附加 metadata（文件名稱、日期、章節、作者）。查詢時先用 metadata 過濾（「只找 2024 年以後的文件」），再做向量相似度搜尋。這大幅縮小搜尋範圍，同時讓回答可以引用來源。
Reranking（二次排序）
向量搜尋找到 top-20 個 chunk，再用一個小型 cross-encoder 模型對這 20 個做精細排序，取 top-3 送給 LLM。第一階段用向量搜尋（快但粗糙），第二階段用 reranker（慢但精準）。
🔑
實務優先順序：先把基本 RAG 跑起來，再觀察失敗案例。如果是「找到了不相關的內容」→ 加 Metadata 過濾。如果是「找到的內容相關但順序不對」→ 加 Reranking。不要一開始就全部加上去。
(04) · Failure Patterns

常見失敗模式診斷

RAG 出問題通常表現為三種症狀。知道症狀對應的根本原因，才能對症下藥。

幻覺
AI 回答了，但內容不在文件裡
原因：檢索失敗（沒找到相關 chunk），但 AI 仍根據訓練資料「補腦」生成答案。
修正：在 System Prompt 加上「如果在提供的文件中找不到答案，明確說『文件中沒有此資訊』，不要自行猜測」。
漏找
文件裡有答案，但 AI 說找不到
原因：Chunking 把關鍵資訊切斷了，或查詢向量與相關 chunk 的向量距離太遠（換了說法）。
修正：增加 top-k 數量（從 3 改到 10）、加大 overlap、或對查詢做 query expansion（用 LLM 把問題改寫成多種說法再分別搜尋）。
找到但沒用
正確的 chunk 被檢索到，但 AI 回答時忽略了
原因：LLM 的 context 裡有太多無關 chunk，稀釋了關鍵資訊的影響力（Lost in the Middle 問題）。
修正：減少 top-k 數量、使用 Reranking 確保最相關的 chunk 排在前面、或在 prompt 中明確指示「以下是最相關的文件片段，請優先參考」。
迷你練習 2 約 20 分鐘
做一次 RAG 失敗案例診斷

對你現有（或這節新建的）RAG 原型故意問 10 個刁鑽問題，把每個失敗案例歸類到「幻覺 / 漏找 / 找到但沒用」，並開出修正處方。

1準備 10 個問題，至少 3 個是文件裡沒答案的誘餌題、3 個是需要換說法才找得到的隱藏題。
2逐題跑 RAG，同時記錄 retrieved chunks 與最終回答。
3對每個失敗案例貼標籤：幻覺 / 漏找 / 找到但沒用，並寫一句原因。
4依標籤對應本節的修正處方（system prompt 約束 / top-k 提高 / reranking / metadata 過濾），選一項下週實驗。
範本素材 沒有現成 RAG 原型？給你一段模擬的「3 個 chunk + 3 個使用者問題 + 3 個實際回答」，讓 AI 幫你標失敗類型與處方。
原始素材 · 模擬 RAG 執行記錄（3 組失敗案例）貼進 Prompt 最下方
【知識庫現有 chunks（僅列出本次 retrieved 的）】 C1 [product-spec.md §2.3] 旗艦款 X1 保固 2 年，標配充電器一顆；非原廠配件不在保固範圍。 C2 [faq.md §退貨] 消費者依消保法享有 7 天鑑賞期，鑑賞期非試用期，商品須保持全新。 C3 [shipping.md §5] 離島訂單加收 250 元運費，寄送時效為本島 +2 工作日。 【Case 1】 Q: 「X1 這款用久了電池衰退明顯，有沒有保固延長方案？」 Retrieved top-3: C1, (C2), (C3) ← 括號代表低相關 A: 「X1 提供 3 年延長保固，費用 $1,200，可直接在官網加購。」 （實際知識庫完全沒有延長保固方案的資料） 【Case 2】 Q: 「我買了東西還沒拆封，幾天內可以退？」 Retrieved top-3: C1, C3, (low-scored chunk about warranty) ← C2 未進入 top-3 A: 「抱歉，目前我找不到這方面的資訊，建議您聯絡客服。」 （知識庫其實有 C2 涵蓋答案，但 retriever 漏抓） 【Case 3】 Q: 「我住澎湖，這台會加收運費嗎？多久到？」 Retrieved top-3: C1, C2, C3 ← C3 在第 3 位 A: 「本商品享有 7 天鑑賞期，保固期為 2 年，若您有任何問題請聯絡客服。」 （C3 有答案但被前面兩個弱相關 chunk 稀釋，模型忽略）
範例 Prompt · RAG 失敗案例診斷器可直接貼 ChatGPT / Claude
你是一位 RAG 系統除錯顧問。請針對下方 3 個案例做失敗類型診斷與處方建議。 對每個 Case 輸出： 1. 失敗類型標籤（擇一）： - [HALLUCINATE] 幻覺：檢索失敗 + 模型補腦捏造 - [MISS] 漏找：知識庫有答案但沒被 retrieved - [LOST] 找到但沒用：正確 chunk 被檢索到卻被稀釋忽略 2. 根因分析（不超過 30 字） 3. 建議處方（擇一或多）： - [PROMPT] 在 system prompt 加「找不到就說不知道」 - [TOPK] 提高 top-k 從 3 到 10 - [EXPAND] query expansion，用 LLM 改寫成多種說法 - [METAFILTER] 加 metadata 過濾（例：doc_type、section） - [RERANK] 加 cross-encoder reranker 4. 為何不選其他處方（1 行） 最後輸出一張「3 案例 × 類型 × 處方」對照表，並建議「下一步先實驗哪一個處方、為什麼」。 資料： [貼上上方原始素材]
預期輸出片段（供對照）
Case 1: [HALLUCINATE] 根因：retrieved 相關度都低，模型卻生成了具體數字。處方：[PROMPT] 強制「找不到就說 找不到」，避免模型捏造。不選 RERANK 因問題不在排序而在模型覆寫。
Case 2: [MISS] 根因：查詢「還沒拆封幾天內可以退」與 C2 的「鑑賞期 7 天」表層詞彙差很多。處方：[EXPAND] + [TOPK]。不選 RERANK 因 C2 根本沒進 top-3。
Case 3: [LOST] 根因：C3 在第 3 位被前面弱相關 chunk 稀釋。處方：[RERANK] 把 C3 提到第 1 位。不選 EXPAND 因檢索已經命中。
| Case | 類型 | 處方 |
| 1 | HALLUCINATE | PROMPT |
| 2 | MISS | EXPAND + TOPK |
| 3 | LOST | RERANK |
下一步：先做 [PROMPT] 約束，因為零成本、擋住最危險的幻覺類型。
成果產出
一份 10 題診斷表（欄位：問題｜retrieved 摘要｜回答｜失敗類型｜處方），加一句「下一步我要先試 ___」。
