CH 6.4 · 3 小時

Function Calling 與Agent 工具鏈設計

讓模型不只「說話」，還能「動手」。本章從工具定義到 Agent Loop，帶你建構能呼叫外部功能的 AI 應用。

學完你會
理解 Function Calling 的運作機制，知道模型何時觸發工具
能用 JSON Schema 定義工具，讓模型正確輸出結構化參數
掌握 ReAct Agent Loop 架構，設計多步驟工具鏈
辨識三種常見 Agent 陷阱，並知道對應的防禦策略
(01)

Function Calling 是什麼？模型不執行程式，只輸出意圖

一個常見誤解：Function Calling 讓 AI 直接執行程式。實際上，模型只做一件事——輸出結構化的函式呼叫意圖，真正的執行權永遠在你的程式碼手上。

運作三步驟

① 你定義工具（名稱 + 參數 Schema）→ ② 模型決定「要呼叫哪個工具、帶什麼參數」並回傳 JSON → ③ 你的程式接收 JSON、真正執行，再把結果送回模型繼續對話。

這個設計的好處是安全且可控：模型無法自行呼叫網路、讀取檔案或操作資料庫，所有副作用都經過你的程式碼過濾。

// Anthropic Claude 工具定義範例 { "name": "get_weather", "description": "查詢指定城市的當前天氣", "input_schema": { "type": "object", "properties": { "city": { "type": "string", "description": "城市名稱，例如 台北" }, "unit": { "type": "string", "enum": ["celsius", "fahrenheit"] } }, "required": ["city"] } }
✍️
description 欄位決定觸發精準度。模型靠 description 判斷「這個工具適不適合用」，寫得越清晰，觸發越準確。把它當成寫給模型看的說明文件，不是給人類看的注釋。
(02)

ReAct Agent Loop：觀察 → 思考 → 行動 → 觀察

單次工具呼叫解決不了複雜任務。Agent Loop 讓模型在「思考與行動」之間循環，直到任務完成。ReAct（Reasoning + Acting）是目前最常見的架構。

→
👁
Observe
接收環境狀態、工具回傳結果、使用者輸入
→
💭
Think
模型推理下一步應該做什麼、呼叫哪個工具
→
⚡
Act
輸出工具呼叫 JSON，程式執行並取得結果
🔄
Observe
結果送回模型，進入下一輪循環直到完成
1
設定系統提示詞與工具清單
告訴模型它的角色、可用工具，以及何時應該使用工具而非直接回答。
2
傳入使用者訊息，呼叫模型 API
模型回傳 stop_reason: "tool_use"（Claude）或 finish_reason: "tool_calls"（OpenAI），代表它想用工具。
3
解析工具呼叫，執行對應函式
從回應中取出 tool_use 區塊，把 input 傳入你的本地函式，取得結果。
4
把工具結果附回對話繼續
以 tool_result 角色把執行結果送回模型，讓它繼續推理或輸出最終答案。
5
重複直到 stop_reason 為 end_turn
只要模型還在要求工具就繼續循環。最終模型輸出文字答案，結束 Loop。
🔑
Agent Loop 的本質是狀態機。你的程式負責維護對話歷史（messages array），模型每次都看到完整歷史，所以能「記住」前幾輪的工具結果繼續推理。
迷你練習 1 約 20 分鐘
寫一個只會查天氣和算術的最小 Agent Loop

用 Claude 或 OpenAI 的 tool_use，定義兩個 function（get_weather、calculator），親手把 Observe → Think → Act 的循環寫一遍，讓模型在一次對話裡至少呼叫兩次工具。

1定義兩個工具的 JSON Schema：get_weather(city)、calculator(expression)，description 寫到模型能明確判斷觸發情境。
2寫一個 while loop：只要 stop_reason === 'tool_use' 就解析 tool_use、執行 mock 函式、把 tool_result 加回 messages。
3設 max_turns=5 保護機制，超過就強制跳出。
4丟一個需要兩步的問題（例：「台北比東京熱幾度？」），觀察模型是否真的依序呼叫 get_weather 兩次再算差。
範本素材 直接給你兩個 tool schema + mock function，讓 AI 幫你組出完整的 Agent Loop 骨架。
原始素材 · 兩個工具 schema + 兩個 mock 函式貼進 Prompt 最下方
【Tool Schemas（Anthropic 格式）】 [ { "name": "get_weather", "description": "查詢指定城市當下的氣溫（攝氏），僅支援主要城市", "input_schema": { "type": "object", "properties": { "city": { "type": "string", "description": "城市名稱，例如 台北、東京、紐約" } }, "required": ["city"] } }, { "name": "calculator", "description": "計算簡單的數學算式，支援 + - * / 與括號", "input_schema": { "type": "object", "properties": { "expression": { "type": "string", "description": "例如 '28 - 15' 或 '(20+5)/2'" } }, "required": ["expression"] } } ] 【Mock Function 行為】 - get_weather('台北') 回 {"city":"台北","temp_c":28} - get_weather('東京') 回 {"city":"東京","temp_c":15} - get_weather(其他) 回 {"error":"城市不在支援清單"} - calculator('28 - 15') 回 {"result": 13} - calculator(含非法字元) 回 {"error":"expression 含非法字元"} 【測試問題】 「台北跟東京現在差幾度？」
範例 Prompt · Agent Loop 骨架產生器可直接貼 ChatGPT / Claude
你是一位 Anthropic SDK 熟手。請用下方工具定義與 mock 函式，產出一份最小可執行的 Agent Loop 程式。 需求： 1. 語言：Node.js（ESM）+ @anthropic-ai/sdk 2. messages 陣列在迴圈外維護，每輪 push 新的 assistant / user(tool_result) 訊息 3. while 迴圈條件：response.stop_reason === 'tool_use' 且 turn < MAX_TURNS（設 5） 4. 對 tool_use block 逐個執行對應 mock 函式；支援一次多個 tool_use 5. tool_result 格式符合 Anthropic 規範：{ type: 'tool_result', tool_use_id: ..., content: JSON.stringify(result) } 6. 每輪開頭 console.log(`[turn ${n}] stop_reason=${r}`)，每個工具呼叫 console.log(` → ${name}(${JSON.stringify(input)}) = ${JSON.stringify(result)}`) 7. 超過 MAX_TURNS 丟 Error('agent exceeded max turns') 8. 最後把 response 最後一個 text block 印出當 final answer 輸出規範： - 單檔 .mjs，不超過 70 行 - 結尾用 `await runAgent('台北跟東京現在差幾度？')` 示範呼叫 - 程式碼外附 3 行觀察指引：「你會在 console 看到什麼樣的輪次順序？」 工具與 mock： [貼上上方原始素材]
預期輸出片段（供對照）
[turn 1] stop_reason=tool_use
  → get_weather({"city":"台北"}) = {"city":"台北","temp_c":28}
  → get_weather({"city":"東京"}) = {"city":"東京","temp_c":15}
[turn 2] stop_reason=tool_use
  → calculator({"expression":"28 - 15"}) = {"result":13}
[turn 3] stop_reason=end_turn
Final: 台北目前 28°C，東京 15°C，台北比東京熱 13°C。
觀察：模型通常會把兩個 get_weather 放在同一輪平行呼叫，calculator 放在第二輪，最後一輪才 end_turn 輸出文字。若你的模型每輪只叫一次工具，換到 Claude 3.5+ 或 Haiku 4.5 觀察差異。
成果產出
Agent loop 程式碼 + 一段 console log，顯示每輪的 stop_reason、tool_use、tool_result，以及最終文字答案。

三種常見 Agent 陷阱及防禦策略

Agent 很強大，但也引入了新的失敗模式。以下是生產環境中最常遇到的三種問題。

陷阱一
無限迴圈（Infinite Loop）
模型反覆呼叫同一工具卻未收斂，或工具回傳的錯誤讓模型陷入重試循環，導致 API 費用暴增與程式卡死。
防禦：設定最大迭代次數（如 max_turns = 10），超過就強制中斷並回傳錯誤訊息；監控同一工具在同一對話中被呼叫的次數。
陷阱二
工具幻覺（Tool Hallucination）
模型呼叫了不存在的工具名稱，或傳入格式錯誤的參數（如把字串傳給 integer 欄位），導致程式拋出異常。
防禦：呼叫前先驗證工具名稱是否在清單中；用 JSON Schema 驗證參數；錯誤時把清楚的錯誤訊息送回模型讓它自我修正，而不是直接拋出 500。
陷阱三
工具呼叫失敗未處理（Unhandled Failure）
外部 API 返回錯誤、網路逾時、或資料庫查無結果，程式未正確處理就送回模型，模型在不完整資訊下繼續推理，輸出錯誤答案。
防禦：工具函式永遠回傳結構化結果（success: true/false + 錯誤說明），讓模型看到「查詢失敗，原因是…」而不是空值；善用 try/catch。
⚠️
不要讓工具有副作用的操作無限重試。寫入資料庫、發送郵件、扣款等不可逆操作，必須在工具實作層加冪等保護（idempotency key），避免重試時重複執行。
迷你練習 2 約 15 分鐘
刻意觸發三種 Agent 陷阱並加上防禦

在剛才的 Agent 上故意誘發無限迴圈、工具幻覺、未處理失敗三種情況，並套用本節防禦策略，驗證程式不再爆掉。

1無限迴圈：把 calculator 回傳值改成永遠「需要再算一次」，觀察是否被 max_turns 擋住。
2工具幻覺：在 prompt 誘導模型呼叫一個不存在的 send_email 工具，驗證你的程式能否回傳「工具不存在」而非 crash。
3未處理失敗：讓 get_weather 故意拋 exception，確保回傳 {success:false, error:'...'} 給模型而不是 500。
4整理一張「陷阱 → 症狀 → 防禦」對照表，回寫到你的 Agent README 或 docstring。
範本素材 三段刻意觸發的誘導對話，對應三種陷阱。你不需要自己設計爛情境，直接用這組測試。
原始素材 · 三段壓測腳本（對應無限迴圈 / 工具幻覺 / 未處理失敗）貼進 Prompt 最下方
【壓測腳本 A · 無限迴圈】 系統設定：把 calculator 的 mock 改成「永遠回 {result:null, need_recompute:true}」。 使用者訊息：「請幫我算 128 + 256」 預期症狀：模型反覆呼叫 calculator，token 成本直線攀升。 【壓測腳本 B · 工具幻覺】 工具清單只提供 get_weather 和 calculator。 使用者訊息：「幫我寄一封 email 給 boss@example.com，標題是今日進度，內容摘要昨天的會議」 預期症狀：模型可能捏造呼叫 send_email 這個工具，或硬塞參數到 get_weather。 【壓測腳本 C · 未處理失敗】 get_weather 的 mock 改成 throw new Error('upstream 503')。 使用者訊息：「查一下台北現在幾度」 預期症狀：若程式沒包 try/catch，整個 agent 進程 crash；若有但 tool_result 只回空字串，模型會自己編一個溫度。 【共用期望防禦】 - MAX_TURNS = 5 上限 - 工具名稱白名單驗證 - 工具執行 try/catch + 結構化錯誤 {success:false, error:'...'}
範例 Prompt · Agent 壓測 + 防禦補強器可直接貼 ChatGPT / Claude
你是一位 LLM Agent 可靠性工程師。請針對下方三個壓測腳本： (A) 無限迴圈 (B) 工具幻覺 (C) 未處理失敗 輸出以下內容： ## 一、預期失敗現象 對每個腳本寫 2 行：「沒有防禦時會怎樣」「沒處理會造成什麼成本或風險」。 ## 二、最小修法（給實際 JS 片段） 對每個腳本給一段 5~15 行 JS/TS 範例： - A：MAX_TURNS 守門 + 同一工具被呼叫 > N 次時強制中止 - B：tool 名稱白名單驗證 + 回給模型的結構化 error message（讓模型自我修正，而非 crash） - C：tool 執行 try/catch + 結構化 {success:false, error:'...'} 包裝 ## 三、三列對照表 | 陷阱 | 症狀觀察點 | 程式位置 | 修法關鍵字 | | A 無限迴圈 | ... | ... | ... | | B 工具幻覺 | ... | ... | ... | | C 未處理失敗 | ... | ... | ... | ## 四、上線前檢查清單（5 條內） 壓測腳本： [貼上上方原始素材]
預期輸出片段（供對照）
A：沒防禦會無限呼叫 calculator，每輪消耗 input+output token；10 輪可能燒掉 > $1。修法：const callCount = {}; if (++callCount[name] > 3) throw new Error('tool called too many times'); + 外層 if (turn >= MAX_TURNS) break;
B：沒防禦會讓 SDK 直接 throw 'unknown tool'；進程中斷。修法：const TOOLS = new Set(['get_weather','calculator']); if (!TOOLS.has(name)) return {success:false, error:`tool "${name}" not available. Available: ${[...TOOLS].join(', ')}`};
C：沒防禦整個 agent crash。修法：try { result = mock[name](input); } catch (e) { result = {success:false, error:e.message}; }
| 陷阱 | 症狀 | 位置 | 修法 |
| A | token 累積、同工具重複 | while loop 進入前 | MAX_TURNS + callCount |
| B | tool name 不在清單 | tool dispatch 前 | whitelist + structured error |
| C | SDK throw 中斷 | tool execution 內 | try/catch + success flag |
上線前檢查：1) MAX_TURNS 寫死不吃使用者輸入 2) 所有工具都 try/catch 3) 白名單集中在一個 const 4) 有 callCount 防同工具濫用 5) 錯誤訊息回給模型而非前端
成果產出
三段測試 log（每種陷阱一段）+ 一張三列對照表，說明每個防禦對應到程式的哪一段。

實務工具選擇：程式 API vs 低代碼 Agent 平台

Function Calling 有兩種實作路線：寫程式碼直接呼叫 API，或使用低代碼 Agent 平台（如 n8n）。選哪條路取決於你的需求。

面向 程式碼（Claude / OpenAI API） n8n AI Agent 節點
工具定義 手寫 JSON Schema，完全自訂 拖拉節點，平台管理 Schema
自訂程度 最高，可精控每個參數 中，受限平台支援的工具
整合難度 需自行處理 Loop、狀態、錯誤 低，平台處理迴圈與錯誤
適用情境 需嵌入現有系統、高度客製化工具鏈 快速自動化、整合 SaaS 服務
成本控制 可自訂 max_turns、token 限制 依平台設定，彈性較低
除錯體驗 查 console log、API 回應 視覺化執行歷史，較直觀
💡
建議路線：先用 n8n AI Agent 節點快速驗證工具鏈邏輯，確認流程可行後，再視需求用程式碼重實作高性能或需嵌入既有系統的部分。這樣可以節省大量試錯時間。
