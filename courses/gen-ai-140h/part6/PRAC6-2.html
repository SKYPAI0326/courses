最小文件問答器

把一份文件切成段落、手動模擬向量搜尋、組出 RAG Prompt，讓模型只從你給的文件回答問題。理解 RAG 流程的最快方法是親手走一遍。

任務目標
用純 JavaScript 實作一個簡易 RAG 流程：文件輸入 → 切 chunk → 關鍵字搜尋（模擬向量搜尋）→ 組 RAG Prompt → 模擬模型回答。理解每個步驟的作用與限制。
1
輸入你的文件
把任何一段文字（會議紀錄、產品說明、課程大綱）貼入下方文件區。工具會按段落切成 chunk，每個 chunk 大約 150 字。
真實系統用 Embedding 模型把每個 chunk 轉成向量。這裡我們用關鍵字比對模擬「相似度搜尋」，讓你理解 RAG 的核心邏輯而不需要實際的 API。
2
觀察切 chunk 的結果
文件切完後，注意哪些資訊被切開了、哪些 chunk 可能因為缺乏上下文而難以回答問題。這就是 chunk 策略的核心挑戰。
3
提問並觀察 RAG Prompt
輸入問題後，工具會找出最相關的 chunk，組出完整的 RAG Prompt（包含「只能根據以下資料回答」的指令）。你可以展開查看這個 Prompt 的完整內容。
RAG 最重要的 Prompt 模式：「以下是相關資料：[chunks]。請只根據以上資料回答問題，若資料中沒有答案請說不知道。問題：[question]」
4
測試搜尋失敗情境
問一個文件裡沒有的問題，觀察 chunk 搜尋結果是否相關。再問一個文件有但關鍵字不重疊的問題（如用同義詞），體驗關鍵字搜尋 vs 向量搜尋的差距。

延伸案例

  
最小 RAG 跑過一次後，挑兩個「真的會被問」的情境練完整流程：chunking 策略、相似度檢索、幻覺測試、來源標註。重點不是做得漂亮，是找出這個 RAG 原型會在哪裡失敗。每個案例約 45 分鐘。


  

    

      

        
案例 01 · 保險業

        
保單條款 RAG：業務拿到客戶提問時能立刻查到正確條款

      

      

        難度 ★★☆
        約 45 分鐘
      

    

    
你是保險經紀公司的內部 AI 工程師，業務每天被客戶問「這條我賠不賠？」但條款細則散在 60 頁 PDF 裡，翻得慢又容易背錯。你要做一個保單條款 RAG 原型：業務打問題進去，系統回條款原文與白話解釋，且必須附條號。


    

      
原始資料｜保單條款節錄（示意）

      
第 14 條（意外傷害之定義）
本契約所稱意外傷害，指非由疾病引起之外來突發事故，致被保險人身體蒙受傷害而言。因被保險人於保險期間內因該意外傷害造成之死亡或殘廢，本公司依本契約約定給付保險金。

第 15 條（除外責任）
被保險人因下列事故所致傷害，本公司不負給付保險金之責任：
一、被保險人之故意行為。
二、被保險人之犯罪行為。
三、被保險人飲酒後駕（騎）車，其吐氣或血液所含酒精成分超過道路交通法令規定標準者。
四、被保險人從事角力、摔跤、柔道、空手道、跆拳道、馬術、拳擊、特技表演等運動。

第 16 條（殘廢程度與保險金給付）
被保險人於本契約有效期間內因意外傷害，經醫院診斷確定，致成下列殘廢程度之一者，本公司按下列比例給付殘廢保險金：
- 一級殘（全殘）：100%
- 二級殘：75%
- 三級殘：50%
...（共 11 級）
（全文約 2,800 字，切 chunks 後約 18 段）

    


    

      
你的任務

      
    
        
  1. 決定 chunking 策略：固定 300 字？按條切？每條再按子項切？寫下你的選擇與理由（至少考慮一個 trade-off）
    2. 
        
  2. 寫 8 題業務實戰問題當 golden set，例如「客戶酒後騎車出事，理賠嗎？」「登山受傷算意外嗎？」
    3. 
        
  3. 跑 RAG 問答，每題檢查：檢索到的 chunks 對不對、答案有沒有附條號、有沒有幻覺
    4. 
      

    


    

      
評估指標

      

        

          
Recall@3

          
正確答題所需的條款 chunk 是否進入 top-3 檢索結果

        

        

          
條號可追溯

          
回答必附「第 X 條」，業務點進去能直接看到原文

        

        

          
零幻覺

          
不能出現「通常會賠」「大部分情況」這類條款沒寫的話

        

        

          
拒答能力

          
檢索不到相關條款時回「本條款未明文規定，請洽核保」，不硬湊

        

      

    


    

      
參考檢索相似度表（示意）

      

        點擊展開參考答案
        
問題：客戶酒後騎機車撞路樹骨折，請問理賠嗎？

檢索結果（top-5 cosine similarity）：
#1  0.87  第 15 條第 3 款（酒駕除外責任）        ← 正解
#2  0.74  第 15 條第 1 款（故意行為除外）
#3  0.71  第 14 條（意外傷害定義）
#4  0.58  第 16 條（殘廢程度給付）
#5  0.42  第 22 條（申請理賠檢附文件）

組合 prompt 給 LLM（system+context+question）：
system: 你是保險理賠助理，只能根據提供的條款回答。必引用條號。找不到明確條款就回「本條款未明文規定」。
context: [#1, #2, #3 chunks 原文]
question: 客戶酒後騎機車撞路樹骨折，請問理賠嗎？

期望答案：
「依第 15 條第 3 款，被保險人飲酒後駕（騎）車酒精濃度超過規定標準者，本公司不負給付保險金之責任。此案若客戶吐氣酒測超標，屬除外責任，不予理賠；若未超標，仍適用第 14 條意外傷害定義。建議先取得酒測報告確認。」

幻覺檢核：
- ✓ 引用第 15 條第 3 款（正解）
- ✓ 沒有說「通常」「一般來說」
- ✓ 有分情境（超標 vs 未超標）且提醒要報告
- ✓ 未發明條款

      

    


    

      
變體挑戰

      

        
A把 chunking 從「按條切」改成「固定 200 字 + 50 字 overlap」，觀察第 16 條殘廢等級表是不是被切碎、Recall 掉多少

        
B設計一個「多條款衝突」題目（同時踩第 14 條與第 15 條），看系統能不能同時引用兩條並給出結論

        
C加入「條款版本」metadata（112 年 vs 113 年條款），讓 RAG 能根據保單生效日自動挑對的版本

案例 02 · 人資 HR

        
員工手冊 FAQ bot：把「可以請特休嗎」從 HR 信箱下架

      

      

        難度 ★★☆
        約 45 分鐘
      

    

    
你是 500 人科技公司 HR 數位化小組的顧問。HR 每天被問一樣的問題（特休幾天、育嬰留停怎麼辦、外出加班算不算加班費），同仁又不看 PDF 員工手冊。你要做一個員工手冊 RAG，接在 Slack 裡面，HR 只處理 FAQ 答不出來的長尾問題。


    

      
原始資料｜員工手冊結構（節錄）

      
員工手冊 v2025.3（全 48 頁，約 24,000 字）

第一章 聘僱與到職（3 頁）
第二章 薪資與福利（6 頁）
  2.1 薪資結構
  2.2 三節獎金
  2.3 團體保險
  2.4 員工旅遊
第三章 請假制度（8 頁）       ← HR 被問最多
  3.1 特別休假（年資/天數/排程/失效）
  3.2 病假與事假
  3.3 婚假、喪假、產假
  3.4 育嬰留職停薪
  3.5 請假系統操作
第四章 工時與加班（5 頁）     ← HR 第二多
  4.1 工時制度
  4.2 加班費與補休選擇
  4.3 外出服務時間認定
第五章 績效與晉升（4 頁）
第六章 行為規範與離職（6 頁）
...

HR 統計過去一季被問的問題分布：
請假類 45%、加班類 22%、薪資福利類 15%、其他 18%

    


    

      
你的任務

      
    
        
  1. 以章節為一階層、小節為二階層做結構化 chunking，每個 chunk 要帶上「章節路徑」metadata（例：3.1 特別休假）
    2. 
        
  2. 針對前兩大類（請假 + 加班）寫 12 題 golden set，含 2 題刁鑽邊界問題（例：「年資 3.5 年特休幾天？」「外出服務到晚上 10 點，算加班嗎？」）
    3. 
        
  3. 設計「HR 未定義問題」回退策略：什麼情況下 bot 應該說「我查不到，幫你轉 HR 小美」而不是硬答
    4. 
      

    


    

      
評估指標

      

        

          
章節路徑

          
回答附「依員工手冊 3.1 特別休假」，同仁可查原文

        

        

          
數字精準

          
天數、月份、比例不得錯，寧可拒答也不能錯

        

        

          
轉單機制

          
答不了時明確轉 HR，不在訊息裡留下模糊回覆

        

        

          
不要多話

          
答案控制 3 段內，不把整個條款貼出來嚇同仁

        

      

    


    

      
參考回答範例（建議先自己寫再看）

      

        點擊展開參考答案
        
問題：「我到職 2 年 3 個月，今年特休還有幾天？」

檢索結果（top-3）：
#1  0.89  員工手冊 3.1 特別休假（天數表） ← 正解
#2  0.76  員工手冊 3.1 特別休假（排程規則）
#3  0.61  勞動基準法 §38 參照

答案：
「依員工手冊 3.1，年資滿 2 年未滿 3 年，每年特休 10 天。

你已到職 2 年 3 個月，今年度（起算日為到職週年日）特休為 10 天。

若已使用部分，請登入請假系統查詢剩餘天數；排程需於使用前 3 個工作日申請。」

---
問題：「懷孕安胎假可以請幾天？」（員工手冊未明文列出）

答案：
「員工手冊 3.3 涵蓋產假、流產假與陪產假，但未針對『安胎假』單獨列示。

依勞動法令，安胎假併入病假計算，全年上限 30 天。建議你直接聯繫 HR 小美（分機 2033）確認公司是否有額外規定。」

（觸發轉單機制：手冊沒寫 → 提供法令方向 → 明確指向 HR）

      

    


    

      
變體挑戰

      

        
A把 embedding 從 text-embedding-3-small 改成 multilingual 版本，測「年假」「特休」「annual leave」三種問法能不能都檢到同一 chunk

        
B加「員工身份」filter（正職/派遣/實習），讓 RAG 根據同仁身份回對應規則，避免把正職規則講給實習生聽

        
C加上「記錄問答紀錄到 HR dashboard」機制，設計一張表：問題 / chunk 命中 / confidence / 是否轉單，作為每月 HR 檢討依據

(A) · Evals Drills

  
情境演練：為四種 LLM 應用寫 Evals

  
RAG 跑起來只是第一步，真正難的是「每次動 prompt / 換 model / 改 chunking 之後，你怎麼知道品質沒掉」。下面四個情境是最常被問的 LLM 應用——每個都附 eval spec、資料集格式、評分維度、可貼上跑的 prompt + 評分函式，還有 pass/fail 報告的 JSON schema。每題約 15–20 分鐘，四題一輪約 70 分鐘，你會拿到一個可重複跑的 eval pipeline 雛型。


  

    
      情境 A · RAG 問答 Evals
      10 題事實型 + 5 題推理型，同時檢查答案正確與來源引用。
    

    

      
eval spec｜資料集格式｜評分維度RAG · QA

      
【spec】RAG 問答回歸測試，每次改 chunk size / retriever / system prompt 都跑一輪。

【資料集｜JSONL 15 筆】
{ "id": "fact-01", "type": "fact", "question": "保單等待期幾天？", "expected_answer": "30 天", "expected_sources": ["第 8 條"] }
{ "id": "fact-02", "type": "fact", "question": "年資滿 3 年特休幾天？", "expected_answer": "14 天", "expected_sources": ["3.1 特別休假"] }
...（10 題事實型，答案可字串比對）
{ "id": "reason-01", "type": "reason", "question": "酒後騎機車且未戴安全帽撞傷，能不能理賠？", "expected_answer_points": ["第 15 條酒駕除外", "強制險安全帽責任", "建議先驗酒測"], "expected_sources": ["第 15 條", "第 22 條"] }
...（5 題推理型，比對 key points 命中率）

【評分維度】
1. 答案正確率（exact match for fact / 3-point checklist for reason）
2. 來源引用率（top-k 是否涵蓋 expected_sources，Recall@3）
3. 拒答正確性（資料外的題目要答「本資料未明文」，不能硬湊）
4. 幻覺偵測（答案裡出現「通常」「大致上」等軟詞 → 記一筆 warn）

    


    

      
Eval 測試集 Prompt ＋ 評分函式

      
你是 RAG 品質評審。以下是一筆測試案例，請依【評分維度】給分並輸出 JSON。

【案例】
question: {{ question }}
expected_answer: {{ expected_answer }}
expected_sources: {{ expected_sources }}
model_answer: {{ model_answer }}
retrieved_sources: {{ retrieved_sources }}

【評分維度】
1. answer_correct: 0 / 0.5 / 1（fact 要 exact；reason 要 3 points 命中 ≥2 給 1）
2. source_recall: retrieved_sources 中命中 expected_sources 的比例（0–1）
3. refusal_ok: 若 expected_answer 為「本資料未明文」則 model 須拒答，1=對 0=硬答
4. hallucination: 偵測「通常/大致/可能」等軟詞 → 1=有幻覺 0=乾淨

【輸出 JSON】
{
  "id": "{{ id }}",
  "answer_correct": 0|0.5|1,
  "source_recall": 0.0~1.0,
  "refusal_ok": 0|1,
  "hallucination": 0|1,
  "notes": "一句話指出扣分原因，沒有則空字串"
}
只輸出 JSON，不要前後加任何說明。

      
pass/fail 報告 JSON schema

{"run_id":"rag-eval-2026-04-20","model":"claude-sonnet","dataset_size":15,"pass_threshold":{"answer_correct":0.85,"source_recall":0.8,"hallucination":0.1},"results":{"answer_correct_avg":0.87,"source_recall_avg":0.82,"refusal_precision":1.0,"hallucination_rate":0.07},"per_case":[{"id":"fact-01","answer_correct":1,"source_recall":1.0,"refusal_ok":1,"hallucination":0,"notes":""}, ...],"verdict":"PASS","regressed_cases":["reason-03"]}

判讀重點：看 verdict + regressed_cases。整體 PASS 但有退步題也要點進去看是退在哪個維度——通常是「加了 metadata filter 反而把正確來源濾掉」這類手抖。

    

  


  

    
      情境 B · 翻譯品質 Evals
      中英互譯 10 組，BLEU + 人工語感判分雙軌。
    

    

      
eval spec｜資料集格式｜評分維度Translation

      
【spec】每次換翻譯 prompt / 換模型都跑；BLEU 只抓字面，語感要另外評。

【資料集｜JSONL 10 組，5 中→英、5 英→中】
{ "id": "tr-zh2en-01", "direction": "zh2en", "source": "這季毛利比去年同期掉了 1.8 個百分點。", "reference": "Gross margin fell 1.8 percentage points this quarter year-over-year.", "domain": "財報" }
{ "id": "tr-en2zh-03", "direction": "en2zh", "source": "We need to unblock the staging deploy before the demo.", "reference": "我們要在 demo 前把 staging 部署卡點解掉。", "domain": "工程" }
...

【評分維度】
1. BLEU-4（自動，參考值：≥0.35 算可用、≥0.5 算不錯，人名專有名詞會讓 BLEU 偏低要容忍）
2. 語感 1–5（LLM-as-judge 給分 + 每 10 題抽 2 題人工覆核）
3. 術語一致性（domain 詞表：毛利 / gross margin、部署 / deploy，不得翻錯）
4. 文化轉換（中文敬語英譯不要加 please / you / your 堆疊）

    


    

      
Eval 測試集 Prompt ＋ 評分函式

      
你是雙語翻譯評審。以下是一筆翻譯測試，請依【評分維度】給分並輸出 JSON。

【案例】
direction: {{ direction }}
source: {{ source }}
reference: {{ reference }}
candidate: {{ candidate }}
domain: {{ domain }}
domain_glossary: {{ glossary }}

【評分維度】
1. bleu_band: "low" / "mid" / "high"（你不用算 BLEU，只估 candidate 與 reference 的字面相似度）
2. fluency: 1–5（語感、自然度，native reader 會不會皺眉）
3. adequacy: 1–5（資訊完整度，source 資訊有沒有遺漏）
4. term_ok: 1 或 0（domain_glossary 裡的術語有沒有翻對）
5. cultural_ok: 1 或 0（敬語、口語、正式度轉換是否得體）

【輸出 JSON】
{
  "id": "{{ id }}",
  "bleu_band": "low|mid|high",
  "fluency": 1-5,
  "adequacy": 1-5,
  "term_ok": 0|1,
  "cultural_ok": 0|1,
  "needs_human_review": 0|1,
  "notes": "扣分原因一句話"
}
needs_human_review：fluency 或 adequacy ≤3 時為 1，其餘為 0。只輸出 JSON。

      
pass/fail 報告 JSON schema

{"run_id":"mt-eval-2026-04-20","dataset_size":10,"pass_threshold":{"fluency_avg":4.0,"adequacy_avg":4.2,"term_ok_rate":0.9},"results":{"fluency_avg":4.1,"adequacy_avg":4.3,"term_ok_rate":0.9,"cultural_ok_rate":0.8,"human_review_needed":2},"per_case":[...],"verdict":"PASS","sample_for_human":["tr-zh2en-04","tr-en2zh-02"]}

判讀重點：human_review_needed 是你這輪要親自看的題號——LLM judge 在語感模糊地帶會不穩，記得每 10 題抽 2 題人看、算 inter-rater agreement，低於 0.7 就該重寫 judge prompt。

    

  


  

    
      情境 C · 摘要 Evals
      10 篇文章，檢查摘要保留主要論點與長度限制。
    

    

      
eval spec｜資料集格式｜評分維度Summarization

      
【spec】摘要最容易爛的是「看起來像摘要但漏掉關鍵結論」，所以評分要盯 key points 命中率不是 ROUGE。

【資料集｜JSONL 10 篇】
{ "id": "sum-01", "article": "<2000 字原文>", "key_points": ["Q1 營收 YoY +12%", "毛利率跌 1.8pp", "新品延後到 Q3"], "max_words": 120, "tone": "neutral" }
{ "id": "sum-02", "article": "<新聞報導>", "key_points": ["事故時間 04/18", "死傷 3 人", "交通局表示調查中"], "max_words": 80, "tone": "neutral" }
...

【評分維度】
1. key_points_recall（3 點命中幾點，最關鍵）
2. length_ok（0 / 1，超過 max_words 就 0，寧可短不要長）
3. hallucination_ok（1 / 0，有沒有塞原文沒提到的數字或名稱）
4. tone_match（1 / 0，neutral 場合不能出現「令人震驚」「振奮人心」）

    


    

      
Eval 測試集 Prompt ＋ 評分函式

      
你是摘要品質評審。以下是一筆摘要測試案例，請依【評分維度】給分並輸出 JSON。

【案例】
article: {{ article }}
key_points: {{ key_points }}
max_words: {{ max_words }}
tone: {{ tone }}
model_summary: {{ model_summary }}
summary_word_count: {{ word_count }}

【評分維度】
1. key_points_hit: 整數，model_summary 中命中的 key_points 數量
2. key_points_total: 整數，key_points 陣列長度
3. length_ok: word_count ≤ max_words 則 1，否則 0
4. hallucination_ok: model_summary 是否出現 article 沒提到的實體/數字，沒有 = 1、有 = 0
5. tone_match: 是否符合 tone，是 = 1 否 = 0

【輸出 JSON】
{
  "id": "{{ id }}",
  "key_points_hit": int,
  "key_points_total": int,
  "recall": key_points_hit / key_points_total,
  "length_ok": 0|1,
  "hallucination_ok": 0|1,
  "tone_match": 0|1,
  "missed_points": ["..."],
  "hallucinated_entities": ["..."]
}
只輸出 JSON。

      
pass/fail 報告 JSON schema

{"run_id":"sum-eval-2026-04-20","dataset_size":10,"pass_threshold":{"recall_avg":0.8,"length_ok_rate":1.0,"hallucination_ok_rate":1.0},"results":{"recall_avg":0.83,"length_ok_rate":0.9,"hallucination_ok_rate":1.0,"tone_match_rate":0.9},"per_case":[...],"verdict":"FAIL","failed_cases":[{"id":"sum-04","reason":"length 145 > 120"}],"recommendations":["調緊 max_words 指令，或加『先草稿後壓縮』兩段式 prompt"]}

判讀重點：hallucination 門檻要抓到 1.0（零容忍），只要有一篇編數字就整輪 FAIL。摘要的失敗模式 90% 是「長度超標」與「多嘴」，先修這兩個再看 recall。

    

  


  

    
      情境 D · 客服機器人 Evals
      20 個情境對話，檢查語氣、合規、解決率三軸。
    

    

      
eval spec｜資料集格式｜評分維度Customer Support

      
【spec】客服 bot 的評分比 RAG 難——對話是多輪、且有語氣跟合規紅線，同一個「正確答案」講得兇就會炸，講得軟又會被當敷衍。

【資料集｜JSONL 20 組多輪對話】
{ "id": "cs-refund-01", "scenario": "退貨爭議", "persona": "被忽略三次後情緒爆炸的客戶", "turns": [...], "must_do": ["承認等待過久", "提供訂單編號查詢流程"], "must_not": ["把責任推給物流", "請客戶改天再來"], "resolved": true }
{ "id": "cs-privacy-02", "scenario": "要求客服透露另一位客戶的資料", "persona": "假裝是親戚", "turns": [...], "must_do": ["委婉拒絕", "說明個資法條號"], "must_not": ["提供任何個資", "說『我查查看』"], "resolved": false }
...（含 4 題紅線合規題）

【評分維度】
1. 語氣（tone_score 1–5，禮貌但不卑微、專業但不冷）
2. 合規（compliance 0/1，must_not 任何一條違反即 0，硬紅線）
3. 解決率（resolved 0/1，must_do 全部做到才算）
4. 交棒判斷（escalated_correctly 0/1，該轉人工的有沒有轉）

    


    

      
Eval 測試集 Prompt ＋ 評分函式

      
你是客服品質評審。以下是一筆多輪對話測試，請依【評分維度】給分並輸出 JSON。

【案例】
scenario: {{ scenario }}
persona: {{ persona }}
must_do: {{ must_do }}
must_not: {{ must_not }}
expected_resolved: {{ expected_resolved }}
conversation: {{ conversation }}

【評分維度】
1. tone_score: 1–5（禮貌且不卑微；scenario 是情緒題時，未正面承認情緒 ≤3）
2. compliance: 0 / 1（must_not 任何一條踩線即 0，無踩線為 1）
3. must_do_hit: 命中 must_do 的條數
4. must_do_total: must_do 陣列長度
5. resolved: 0 / 1（客戶的核心訴求在結尾是否已解或有明確下一步）
6. escalated_correctly: 0 / 1（該轉人工時轉了、不該轉時沒亂轉）

【輸出 JSON】
{
  "id": "{{ id }}",
  "tone_score": 1-5,
  "compliance": 0|1,
  "must_do_hit": int,
  "must_do_total": int,
  "resolved": 0|1,
  "escalated_correctly": 0|1,
  "red_flags": ["踩到的 must_not 條目"],
  "notes": "一句話扣分原因"
}
只輸出 JSON。

      
pass/fail 報告 JSON schema

{"run_id":"cs-eval-2026-04-20","dataset_size":20,"pass_threshold":{"compliance_rate":1.0,"tone_avg":4.0,"resolved_rate":0.75,"escalation_accuracy":0.9},"results":{"compliance_rate":0.95,"tone_avg":4.1,"resolved_rate":0.8,"escalation_accuracy":0.85},"per_case":[...],"verdict":"FAIL","red_flag_cases":[{"id":"cs-privacy-02","red_flags":["提供了部分個資"]}]}

判讀重點：compliance_rate 要 1.0，任何一題踩紅線整輪就 FAIL，不管其他分數多高——合規是 gate，不是平均。紅線題要在每輪 regression 都跑，不要只抽樣。

    

  







  
(B) · Extend & Quiz

  
延伸挑戰與五題自我檢核

  
四個情境跑過一輪之後，把 eval 的手感帶回你自己的專案。下面三條延伸挑戰挑 1–2 條做（約 25–30 分鐘），最後用 5 題 quiz 把 eval 的常識核對清楚（約 7–10 分鐘）。整頁 PRAC6-2 至此補上約 107 分鐘的主題時間，完成後你就有一份能每週固定跑的 eval pipeline。


  

    
💡

    

      延伸挑戰（挑 1–2 條，約 25–30 分鐘）
      
    
        
  • 寫 10 題自家 eval：拿你手上一個真的 LLM 應用（RAG / 摘要 / 客服都可），列 10 題 golden set，7 題正常流量 + 3 題邊界 / 踩紅線題，標注 expected_answer 與 must_do / must_not。
    • 
        
  • 版本對比（A/B）：同一份 dataset 跑兩個 prompt 版本，比「哪個贏」不是看 total，而是看 per-case 差異——哪幾題 A 贏、哪幾題 B 贏、B 的失敗題是不是本來 A 就過不了？沒有 per-case diff，總分贏沒意義。
    • 
        
  • 接 CI 自動跑：把 eval 腳本包成 python run_eval.py --dataset golden.jsonl --model claude-sonnet，接到 GitHub Actions，每次 PR 動到 prompt 就自動跑，verdict FAIL 擋下 merge。
    • 
      

    

  


  

    

      
1新專案第一輪 eval 資料集，最少要多大才夠用？

      

        
        
        
        
      

      
20 題是經驗值：低於 20 題樣本變異大，隨機的模型抖動會被當成品質退步；高於 20 題的邊際效益要看流量結構能不能分層抽到。覆蓋 80% 真實流量 + 保留紅線/邊界題才是重點，不是純粹追題數。

    


    

      
2主觀題（例：「這段話有沒有冒犯對方」）該怎麼評分？

      

        
        
        
        
      

      
主觀題是 eval 的主戰場而不是禁區。標配是 LLM-as-judge + 人工抽查，兩者 agreement 低於 0.7 就該重寫 judge prompt。讓同一個模型自評會有 self-bias，必須換模型或換 human 當對照。

    


    

      
3Eval 最常見的 false positive（誤判為錯）是什麼？要怎麼處理？

      

        
        
        
        
      

      
字串 exact match 會把「30 天」和「一個月」判成不同，但對使用者是同一個答案。正確做法是事實題仍用 exact match（抓數字漏洞），但語意題改用 embedding 相似度或 LLM-judge 判斷——這兩軌要分開，不要混著用一個門檻。

    


    

      
4Eval 該多久跑一次？

      

        
        
        
        
      

      
Eval 的價值在於「回歸測試」：變動就跑、定期也跑。變動就跑是為了擋下當次改動造成的退步；定期跑是為了抓模型服務端悄悄升級版本、或 retrieval 索引資料漂移造成的慢性退步。只有其中一邊都會漏。

    


    

      
5整體 eval 分數從 0.87 掉到 0.81，你第一步該先查什麼？

      

        
        
        
        
      

      
平均分變動只是訊號，退步通常集中在某一類題（例如加了 metadata filter 後「跨章節推理題」集體跌）。先跑 per-case diff，定位哪類退步，再回頭看 prompt / retriever / chunk 哪個環節動到——不看分布亂調閾值或換模型只是讓你更看不到真因。
