Streaming 與使用者體驗

使用者按下送出後最怕一片空白。Streaming 用 Server-Sent Events 把回應逐字推送，第一個字一秒內就出現，感知速度立刻不同。這一章把一次性回傳改造成會打字的即時介面。

學完你會
實作 SSE Streaming，讓使用者看到逐字回應而非空白等待
用 fetch + ReadableStream 在前端做出會打字的聊天介面
用 AbortController 實作可取消的串流，中斷時保留已收片段
用 Time to First Token（TTFT）衡量串流體驗好不好
GUIDED PATH · 照著做

先判讀一份 stream log

這段只做一件事：拿一份模擬的 SSE 串流 log，判讀三件事——文字有沒有逐段累積、按 Cancel 後有沒有停止、出錯時有沒有顯示給使用者。看懂這三件事，就抓到 Streaming 體驗的核心。

最低完成線：用下方 log 答出三個判讀問題（Level 1 模擬成功）；有後端環境的學員再到本章實戰演練真跑 /api/stream（Level 2）。

最小可見結果 約 10 分鐘
判讀一份 SSE 串流 log

不用寫程式，先讀懂一份串流回應長什麼樣、發生事件時畫面該怎麼反應。

模擬 stream log
00:00 status=pending
00:01 data: {"delta":"你好"}
00:02 data: {"delta":"，這是逐段"}
00:03 cancel clicked → request aborted
error case: data: {"error":"429 rate_limit_error"}
1逐段累積：哪幾行代表文字一段段長出來，而不是一次回完？
2可取消：哪一行代表使用者中斷了請求？中斷後畫面該保留已收到的片段還是清空？
3錯誤顯示：error case 那行該怎麼呈現給使用者，而不是讓畫面卡在打字中？
成果產出
用一句話分別回答三個判讀問題；做到這裡就算本頁最低完成線。

錯誤排查卡
streaming 卡住先檢查 server 是否 flush、前端是否正確讀 stream、client 斷線是否終止上游請求；畫面沒有逐段長出來多半是前端等整包收完才 render，改成邊收邊 append；按 Cancel 沒反應，檢查有沒有用 AbortController 真的 abort fetch。
(01)

Streaming：別讓使用者盯著空白等

預設的 API 呼叫像「按下送出後變成黑盒子」——使用者只看到一段沉默，等到結果才會出來。Streaming 透過 Server-Sent Events（SSE）把 token 逐個推送，從第一個字就有反饋，感知速度立刻提升。

非 Streaming（預設）
  • 等待 3–15 秒，畫面空白
  • 突然出現完整文字
  • 使用者不確定 AI 是否在處理
  • 適合批次處理、背景任務
Streaming（推薦 UX）
  • 首個 token 在 < 1 秒內出現
  • 逐字顯示，像打字機效果
  • 使用者感覺 AI 正在「思考回答」
  • 適合聊天介面、即時問答
// Express 後端：開啟 Streaming 模式 app.post('/api/stream', async (req, res) => { res.setHeader('Content-Type', 'text/event-stream'); res.setHeader('Cache-Control', 'no-cache'); res.setHeader('Connection', 'keep-alive'); const stream = await client.messages.stream({ model: 'claude-haiku-4-5-20251001', max_tokens: 1024, messages: [{ role: 'user', content: req.body.prompt }] }); for await (const chunk of stream) { if (chunk.type === 'content_block_delta') { res.write(`data: ${chunk.delta.text}\n\n`); } } res.write('data: [DONE]\n\n'); res.end(); });
💡
前端用 EventSource 接收。建立 new EventSource('/api/stream?prompt=...')，在 onmessage 事件中把每個 chunk 附加到 DOM，就能實現打字機效果。注意 EventSource 只支援 GET，若需 POST 改用 fetch 搭配 ReadableStream。
(02)

實戰演練：把 Streaming 變成會打字的聊天介面

理論講完了，現在補一段完整可跑的路徑：先用假 stream log 判讀畫面是否逐段累積、能否取消、錯誤如何顯示；有環境者再把後端用 Server-Sent Events 把 AI 回覆一段段推給前端。模擬判讀是 Level 1，真跑 /api/stream 是 Level 2。

Level 1 模擬成功 · 假 stream log
用下列 log 判斷三件事：文字是否逐段累積、Cancel 後是否停止、錯誤是否顯示給使用者。
00:00 status=pending
00:01 data: {"delta":"你好"}
00:02 data: {"delta":"，這是逐段"}
00:03 cancel clicked → request aborted
error case: data: {"error":"429 rate_limit_error"}
Level 2 真實成功與失敗範例
真跑 /api/stream 時，Network 截圖需看到 Status 200、Content-Type: text/event-stream、TTFT、分段 data: chunks，Cancel 後 request 變 canceled。失敗保底交 401 invalid api key、429 rate limit，或 stream 中途 {"error":"upstream disconnected"} 的截圖與一句排查下一步。
情境 A · Level 2 真實串流 Server-Sent Events 把 AI 回覆串流給前端 — 後端 POST + 前端 fetch ReadableStream
任務背景 + 工具棧從一次性回傳升級成串流
【情境】 你在 CH6-2 的 /api/chat 現在是等 AI 全部回完才一次把文字吐出來，長回答時使用者要空等好幾秒。這次改用 SSE：後端邊收到 token 邊推給前端，前端邊收邊印。 【工具棧】 - Node.js 20+ / Express 後端（承接 CH6-2 的 server.js） - OpenAI SDK 的 stream: true 模式 - 前端用 fetch + ReadableStream 讀（不用 EventSource，因為要 POST） - 一個簡單的 HTML 頁面當前端測試台 【成功條件】 - DevTools Network 看到 /api/stream 是 event-stream、狀態 200、回應持續 ping - 前端畫面會一段段累積文字，不是等 5 秒後一次跳出
程式碼 · 後端 SSE + 前端 fetch 串流javascript
// server.js — 新增 /api/stream app.post('/api/stream', async (req, res) => { const { prompt } = req.body || {}; res.setHeader('Content-Type', 'text/event-stream'); res.setHeader('Cache-Control', 'no-cache, no-transform'); res.setHeader('Connection', 'keep-alive'); res.flushHeaders(); try { const stream = await client.chat.completions.create({ model: 'ChatGPT', messages: [{ role: 'user', content: prompt }], stream: true, }); for await (const chunk of stream) { const delta = chunk.choices[0]?.delta?.content || ''; if (delta) res.write(`data: ${JSON.stringify({ delta })}\n\n`); } res.write('data: [DONE]\n\n'); } catch (err) { res.write(`data: ${JSON.stringify({ error: err.message })}\n\n`); } finally { res.end(); } }); // index.html — 前端最小版（
