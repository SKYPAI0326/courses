PART 6 · AI 應用開發進階
CH 6.5 · 2 小時

Streaming、Evals與成本控管

讓回應體驗更流暢、品質有數據依據、帳單在你掌控之中。這三件事決定一個 AI 應用能不能長期運營。

學完你會
實作 SSE Streaming，讓使用者看到逐字回應而非等待
理解 Evals 評估框架，用數據判斷模型品質而非憑感覺
掌握五種成本控管策略，讓 API 費用可預測
知道何時加 Guardrails，以及如何不過度限制模型能力
(01)

Streaming：別讓使用者盯著空白等

預設的 API 呼叫是「等全部生成完才回傳」，對使用者而言就是一段沉默。Streaming 透過 Server-Sent Events（SSE）把 token 逐個推送，大幅提升感知速度。

非 Streaming（預設）
  • 等待 3–15 秒，畫面空白
  • 突然出現完整文字
  • 使用者不確定 AI 是否在處理
  • 適合批次處理、背景任務
Streaming（推薦 UX）
  • 首個 token 在 < 1 秒內出現
  • 逐字顯示，像打字機效果
  • 使用者感覺 AI 正在「思考回答」
  • 適合聊天介面、即時問答
// Express 後端：開啟 Streaming 模式 app.post('/api/stream', async (req, res) => { res.setHeader('Content-Type', 'text/event-stream'); res.setHeader('Cache-Control', 'no-cache'); res.setHeader('Connection', 'keep-alive'); const stream = await client.messages.stream({ model: 'claude-haiku-4-5-20251001', max_tokens: 1024, messages: [{ role: 'user', content: req.body.prompt }] }); for await (const chunk of stream) { if (chunk.type === 'content_block_delta') { res.write(`data: ${chunk.delta.text}\n\n`); } } res.write('data: [DONE]\n\n'); res.end(); });
💡
前端用 EventSource 接收。建立 new EventSource('/api/stream?prompt=...')，在 onmessage 事件中把每個 chunk 附加到 DOM，就能實現打字機效果。注意 EventSource 只支援 GET，若需 POST 改用 fetch 搭配 ReadableStream。
(02)

Evals：用數據評估品質，不靠感覺

改了 Prompt 以後，真的變好了嗎？沒有評估框架的話，你只能憑主觀印象猜測。Evals 讓你用可重現的方式量化模型輸出品質。

評估可分為四個層次，由易到難：

Level 1 · 最快
精確匹配（Exact Match）
回答必須完全等於預期答案。適合分類任務、是/否問題、固定格式輸出（JSON、代碼）。
Level 2 · 常用
語義相似度（Semantic Similarity）
用 Embedding 計算回答與標準答案的餘弦相似度，容許同義表達。適合問答、摘要。
Level 3 · 更準
LLM 作為評審（LLM-as-Judge）
讓另一個模型（通常更強）評分。提供評分 Rubric，讓評審模型給出 1–5 分與理由。
Level 4 · 最貴
人工評估（Human Eval）
由領域專家手動評分。最準但最慢最貴，通常只用於驗收關鍵里程碑或校準 LLM-Judge。
🔑
實務建議：先建黃金測試集（Golden Set）。收集 20–50 個代表性的「輸入 → 預期輸出」配對，每次改動 Prompt 或換模型都跑一次 eval，確保不退步。這是 AI 應用品質管理的最小可行實踐。
迷你練習 1 約 20 分鐘
為你的工具建一個 20 題的黃金測試集

針對你在 Part 5 做的工具，收集 20 筆「輸入 → 預期輸出」，用 Level 1（精確匹配）+ Level 2（語義相似度）跑一次 eval，算出 baseline 分數。

1挑出你工具最核心的 20 種使用情境，每筆寫下 input 與「理想 output」，存成 CSV 或 JSON。
2寫一個 eval 腳本：對每筆跑一次 API，比對結果，分類任務用 exact match，文字任務用 embedding cosine 相似度（閾值 0.8）。
3印出通過率、失敗案例清單、平均相似度分數。
4把這個分數記下來當 baseline，下次改 prompt 或換模型時重跑比對。
範本素材 沒有自己的工具？用這份「發票分類助理」的情境與 6 題種子測試，讓 AI 幫你擴展到 20 題並產出 eval 腳本。
原始素材 · 工具背景 + 6 題種子測試貼進 Prompt 最下方
【工具背景】 名稱：發票分類助理 任務：接收一句使用者描述的消費，輸出 JSON {category: "...", amount: ...} category 限定六類：餐飲、交通、差旅、辦公用品、雲服務、其他 若模型無法判斷，category 應填 "其他" 【6 題種子測試（input → expected）】 T1 input: "今天午餐便當 95" expected: {"category":"餐飲","amount":95} T2 input: "搭高鐵去台中來回 2400" expected: {"category":"交通","amount":2400} T3 input: "買了 5 包影印紙 450 元" expected: {"category":"辦公用品","amount":450} T4 input: "Claude Pro 月費 20 鎂換成台幣大約 640" expected: {"category":"雲服務","amount":640} T5 input: "東京出差住宿三晚共 12000" expected: {"category":"差旅","amount":12000} T6 input: "送給同事的生日卡片 150" expected: {"category":"其他","amount":150} 【模型設定】 - 模型：claude-haiku-4-5-20251001 - max_tokens: 200 - system prompt: "你是發票分類助理。根據使用者輸入輸出 JSON。category 只能是：餐飲/交通/差旅/辦公用品/雲服務/其他。若無法判斷 category 填其他。"
範例 Prompt · 擴展黃金集 + Eval 腳本產生器可直接貼 ChatGPT / Claude
你是一位 LLM 應用的品質工程師。請根據下方工具背景與 6 題種子測試，產出完整的 20 題黃金集與 eval 腳本。 任務一：擴展到 20 題 - 每個 category 至少 2 題 - 至少 4 題是邊界情境（金額混寫法、英文混中、模糊描述、可能誤判） - 至少 2 題是「理應填『其他』」的陷阱題 - 輸出 JSON 陣列，欄位 id, input, expected 任務二：eval.py 腳本 - 讀 golden_set.json - 對每題呼叫 Claude API（模型與設定如素材），解析回傳 JSON - 評分： - Level 1 精確匹配：category 與 expected.category 相同 → 1 分 - Level 2 金額容差：amount 與 expected.amount 絕對誤差 < 5 → 1 分 - 最終通過：兩者同時符合 - 輸出： - 通過率 X/20 - 失敗案例前 3 筆（含 expected vs actual） - 各 category 的通過率 任務三：結尾建議 - 1 句：「下次 prompt 迭代後，什麼分數視為 regression？」 素材： [貼上上方原始素材]
預期輸出片段（供對照）
golden_set.json: [{"id":1,"input":"今天午餐便當 95","expected":{"category":"餐飲","amount":95}}, ... 20 筆]
eval.py 片段：
for item in golden: actual = json.loads(call_claude(item['input'])); cat_ok = actual['category'] == item['expected']['category']; amt_ok = abs(actual['amount']-item['expected']['amount'])<5; results.append(cat_ok and amt_ok)

通過率 17/20 (85%)
失敗案例：T12 'Notion 年繳 9600 台幣' actual=其他/9600, expected=雲服務/9600（category 錯）
各 category 通過率：餐飲 3/3 交通 3/3 差旅 2/3 辦公 3/3 雲服務 2/3 其他 4/5
regression 門檻：通過率低於 80% 視為退步，立即 rollback 改動。
成果產出
golden_set.json（或 .csv）+ eval.py 腳本 + 一張 baseline 結果表（通過率 / 平均相似度 / 失敗案例前 3 筆）。

(03)

成本控管：五個策略讓帳單可預測

API 費用按 token 計算，input token + output token 各有單價。應用上線後，成本很容易在不知不覺中放大。以下五個策略可系統性降低費用。

策略做法節省潛力
選對模型 Haiku 處理簡單分類/摘要，Sonnet 處理複雜推理，Opus 只用在最關鍵任務 60–90% ↓
Prompt 快取 把不變的系統提示詞標記為 cache_control，重複呼叫時 input cost 降 90%（Claude） 40–70% ↓
限制 max_tokens 依任務設定合理上限，分類任務設 50、摘要設 300，避免模型輸出冗長回答 20–50% ↓
Server-side 快取 對相同或高度相似的輸入快取回應（Redis、記憶體），直接略過 API 呼叫 視命中率
設每日預算上限 在 API 後台設 Usage Limit，超過後自動停止，防止 bug 或濫用造成帳單爆炸 風險控管
⚠️
帳單爆炸最常見原因：Agent Loop 沒有 max_turns 限制。一個失控的 Agent 可以在幾分鐘內消耗數百美元的 token。上線前一定要確認 Loop 有退出條件，並在 API 後台設好預算警示。
迷你練習 2 約 15 分鐘
為你的工具做一次成本瘦身

從五種成本策略中挑兩種套用到你的工具，跑前後對照實驗，實際測量 token 與金額差異。

1跑 10 次現有版本，記錄 usage.input_tokens、output_tokens 平均值，換算單次成本。
2選兩個策略套用：例如「Sonnet → Haiku」+「max_tokens 降到實際需求」，或「開 prompt caching」+「server-side cache」。
3再跑 10 次，記錄新版本的 token 與成本，並跑 golden set 確認品質沒退步。
4算出節省百分比，並在 API 後台設一個「每日 $X」的 usage limit 作為最後保險。
範本素材 一組前版 usage 統計 + 工具特性，讓 AI 幫你挑出兩個最該先動的成本槓桿。

【工具基本資料】 名稱：內部 SOP 查詢助理 模型：claude-sonnet-4-5-20250929 max_tokens: 2048 每天呼叫次數：約 300 次 是否已開 prompt caching：否 是否有 server-side cache：否 【System Prompt 特徵】 - 長度：約 2400 字（含 8 段 SOP 摘要、4 個拒答規則、3 個輸出格式範例） - 每次呼叫內容完全一致 【典型呼叫 10 次平均 usage】 input_tokens: 3800 output_tokens: 520 單次成本（Sonnet in $3/M + out $15/M）：≈ $0.0192 每日成本：300 × 0.0192 = $5.76 月成本：≈ $172.8 【品質 baseline】 golden set 20 題通過率：18/20 (90%) 【已知現象】 - 約 70% 的查詢其實只需要簡單的關鍵字比對 + 模板回覆 - output 常常超過 300 tokens 但實際有用資訊不到一半 - 使用者問題中約 15% 是同一天問過的 FAQ

(04)

Guardrails：保護應用，但不過度限制

Guardrails 是在模型輸入或輸出上加的防護層，用來攔截有害內容、防止越獄、確保輸出格式合規。但加太多會讓應用變得不好用。

1
輸入過濾（Input Guard）
在送入模型前先掃描使用者輸入。可用關鍵字黑名單、另一個輕量模型分類，或 Anthropic Constitutional AI 等機制。攔截明顯惡意請求，減少不必要的 API 呼叫。
2
系統提示詞約束（System Prompt Constraint）
在 system prompt 中明確定義「這個 AI 只做什麼、不做什麼」。比硬編碼過濾更靈活，也讓模型理解邊界而不只是被阻擋。例如：你只回答與台灣稅務相關的問題，其他問題請禮貌拒絕。
3
輸出驗證（Output Validation）
對模型輸出做格式或內容驗證，確保符合預期結構（如要求 JSON 就驗證 JSON 格式）。若驗證失敗，可重試或回傳降級結果，而非直接崩潰。
4
速率限制（Rate Limiting）
對每個使用者設定每分鐘/每日呼叫上限，防止單一使用者濫用或惡意探測。搭配 API Gateway 或自行用 Redis 實作計數器。
💡
Guardrails 的黃金原則：少即是多。過度防護會讓正常請求也被攔截，使用者體驗下降，你反而要花更多時間處理誤判申訴。先從系統提示詞約束開始，只在有明確需要時才加額外層。
