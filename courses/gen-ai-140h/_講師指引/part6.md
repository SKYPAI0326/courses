- Part 6 是全課程「從工具玩家跨到 AI 工程師」的分水嶺，學員會第一次真的寫 while loop、估 token 帳單、看 trace 找錯。務必把每個失敗案例當教材，不要幫學員把錯改掉。

# CH6-1 · API、後端代理與錯誤處理

## MP `ch6-1-mp1`｜估算你手上一個工具的單次 API 成本

- **預估時間**：10 分鐘（網頁標示 15，實際課堂跑 10 即可；剩餘 5 分鐘併入 MP2）
- **帶領要點**：
  - 開場先示範打開 Anthropic / OpenAI 定價頁，讓學員看到「input 和 output 單價不同、且差 3-5 倍」。
  - 強調「中文 token 數 ≈ 字數 × 1.5」是粗估；正式要用 tokenizer。
  - 學員最常卡在「prompt caching 能不能開」—請直接回答「system prompt 不變、呼叫量 > 每天 50 次，就開」。
- **常見卡點**：
  - 把 input 和 output token 合起來算，導致成本低估 2-3 倍。
  - 只算單次不換算月成本，沒有尺度感。
  - 用「感覺不貴」而不是具體美金。
- **示範答案語氣**：
  > 「你這個 system prompt 有 800 字，估 1,200 tokens。Sonnet 4.5 input 是 $3 per 1M tokens，光 system 每次就 $0.0036。一天 500 次、一個月 15,000 次，光 system 就 $54。如果把它 cache 住，會降到 $5.4。你看這就是為什麼要算帳。」
- **延伸話題**：Prompt caching 的 TTL（Anthropic 5 分鐘 vs 1 小時）對批次任務的影響、token 計費與 character 計費的差異（Gemini 新版）。

## MP `ch6-1-mp2`｜為後端代理加一層指數退避重試

- **預估時間**：10 分鐘
- **帶領要點**：
  - 先畫三條時間軸：固定重試 / 線性退避 / 指數退避，讓學員看到「為什麼等一秒是壞主意」。
  - 逼學員「真的觸發 429」—示範用 bash 連續發 20 個 request，看 console 真的印出等待訊息。
  - 提醒 4xx（除 429）不要重試，這是新手最常寫錯的地方。
- **常見卡點**：
  - 把所有錯誤都包進 retry（包含 401 金鑰錯誤），結果一直重試打爆 log。
  - 沒有 `max_retries`，無限迴圈。
  - 只在前端 catch，後端沒 retry 架構。
- **示範答案語氣**：
  > 「退避不是為了你，是為了整個 API 的健康。如果所有 client 都 1 秒重試，你就是 DDOS 共犯。指數退避的 1、2、4 秒數字看起來隨便，但它讓重試流量像波浪錯開，不是海嘯。」
- **延伸話題**：Jitter（加隨機擾動防重試同步）、Circuit Breaker Pattern、Anthropic response header 的剩餘配額怎麼用來主動降速。

# CH6-2 · Embedding 與向量搜尋基礎

## MP `ch6-2-mp1`｜Embedding 語義相似度實驗

- **預估時間**：10 分鐘
- **帶領要點**：
  - 強制學員先「用紙筆預測」每對句子的 similarity 高低，再跑 API 比對。預測錯誤的地方就是 embedding 直覺最值錢的一刻。
  - 準備一組講師示範句子（例：「貓咪」vs「喵星人」vs「老虎」vs「電腦」），親手跑 matrix 給全班看。
  - 特別強調「0.7 就算高相似」—很多學員誤以為要 > 0.95。
- **常見卡點**：
  - 用不同模型生的向量互比，distance 全亂。
  - 以為絕對數值重要（跨模型不可比，同模型內才可比）。
  - 沒有對角線 self-similarity = 1 的驗證。
- **示範答案語氣**：
  > 「你看 matrix 對角線都是 1，這是 sanity check。『我喜歡貓』跟『我家有喵星人』similarity 0.82，跟『我家路由器壞了』只有 0.31，差距這麼大，說明模型真的『看懂同義』。這就是為什麼 RAG 敢用它做檢索。」
- **延伸話題**：多語 embedding（multilingual-e5、voyage-multilingual）在中英混用文件的優勢、embedding 維度壓縮（Matryoshka）降本。

## MP `ch6-2-mp2`｜用 Chroma 建最小向量資料庫

- **預估時間**：10 分鐘
- **帶領要點**：
  - 開場先講「你不需要懂 HNSW 演算法」—這是讓學員別卡在內部實作的心理門檻。
  - Chroma `pip install chromadb` 安裝失敗是常見問題，預備一份 Colab notebook 作為 fallback。
  - 提醒 `distance` 與 `similarity` 方向相反（distance 小 = 相似），學員常看錯。
- **常見卡點**：
  - persistent client 與 in-memory client 搞混，重開後資料消失。
  - Collection 已存在又 `create_collection` 報錯，要改用 `get_or_create`。
  - 查不到結果以為程式壞了，其實是測試資料太少（10 段）+ 無關查詢本來就 distance 大。
- **示範答案語氣**：
  > 「Chroma 對你來說就是一個『能塞進文字、能拿出相似文字』的黑盒子。你真正要學的是『丟什麼進去、拿出來怎麼判斷夠不夠好』，內部 index 演算法等你規模到百萬向量再研究。」
- **延伸話題**：Supabase pgvector 在已有 Postgres 的專案是首選、Pinecone 的 Serverless 定價適合不穩定流量、Qdrant 自建部署在金融/醫療的資料主權需求。

# CH6-3 · RAG 實作與品質調優

## MP `ch6-3-mp1`｜三種 Chunking 策略比較

- **預估時間**：10 分鐘（標 20，實務課堂跑 10-12 即可）
- **帶領要點**：
  - 一定要用「同一份文件、同一組查詢、只差 chunking」的 A/B 對比，不要讓學員每個版本換材料。
  - 提前準備一份 2,000-5,000 字有明確標題結構的 SOP 或政策文件，避免學員挑到純散文。
  - 評分直接用「找到 / 半對 / 找錯」三檔，不要陷入 0-1 連續分數。
- **常見卡點**：
  - overlap 忘了設，chunk boundary 跨段資訊全漏。
  - 三個版本用不同 embedding 模型，變項沒控制。
  - 查詢太廣（「請告訴我這份文件」）無法測出 chunking 差異。
- **示範答案語氣**：
  > 「這份 SOP 按標題切 top-3 命中率 80%，固定 256 只有 50%，為什麼？因為標題切保留了『這段在第幾章』的脈絡；256 把『例外條款』從標題切斷了。你之後換文件時，看這份文件有沒有明顯結構，有就走標題切，沒有就固定切。」
- **延伸話題**：Contextual Retrieval（Anthropic 2024 方法：每個 chunk 前加 LLM 生成的段落摘要）、Semantic Chunking（依 embedding 距離自動切）、Hierarchical chunks（粗/細雙層索引）。

## MP `ch6-3-mp2`｜RAG 失敗案例診斷

- **預估時間**：10 分鐘
- **帶領要點**：
  - 這題是整 Part 6 最有實戰價值的練習，請多給 5 分鐘；若時間緊，砍 MP1 不砍這題。
  - 強制要求學員「把 10 題失敗都跑一次」，不能用想的。真跑才會看到 retrieved chunks 跟預期不同。
  - 三個失敗類別（幻覺 / 漏找 / 找到但沒用）對應的修法完全不同，這是本題核心。
- **常見卡點**：
  - 把「漏找」跟「找到但沒用」混為一談—差別是 retrieved chunks 裡有沒有正解。
  - 只看最終回答不看中間 retrieval，無法分診。
  - 一次加全部修法（metadata + rerank + prompt 約束），測不出哪個有效。
- **示範答案語氣**：
  > 「你這題是『找到但沒用』—retrieval top-3 裡第 2 個就是正解，但 LLM 選了第 1 個不相關的生成。這是 Lost in the Middle 問題。先試 rerank 把正解推到 top-1，不要去改 chunking，那會改壞其他題。」
- **延伸話題**：Query expansion（用 LLM 把問題改寫成多版本）、HyDE（先讓 LLM 生假答案再拿去檢索）、BM25 + vector 的 hybrid search。

# CH6-4 · Function Calling 與 Agent 工具鏈

## MP `ch6-4-mp1`｜最小 Agent Loop（查天氣 + 算術）

- **預估時間**：10 分鐘
- **帶領要點**：
  - 開場畫一張 messages array 的變化圖，讓學員看到每輪多一個 assistant+tool_result pair，這是 state 所在。
  - 強調 `stop_reason === 'tool_use'` 是迴圈條件—這是最多新手寫錯的地方（用 try/catch 或固定次數）。
  - 「台北比東京熱幾度」這題刻意設計成要呼叫 get_weather 兩次，觀察模型會不會自己做這個判斷。
- **常見卡點**：
  - tool_result 沒把 `tool_use_id` 帶回去，API 報錯。
  - max_turns 忘了設，觸發無限迴圈燒 token。
  - tool description 寫太模糊，模型該呼叫時不呼叫或亂呼叫。
- **示範答案語氣**：
  > 「Agent 不是模型變聰明了，是你幫它寫了一個 while loop。模型每輪只做一個決定：要不要用工具、用哪個。剩下的執行、記帳、退出條件，全是你的程式在管。這件事看懂了，Agent 就不神秘了。」
- **延伸話題**：Claude 的 `tool_choice: required` 與 `any` 的差異、Parallel tool use（一輪呼叫多個工具）、Model Context Protocol（MCP）讓工具可跨 client 共用。

## MP `ch6-4-mp2`｜刻意觸發三種 Agent 陷阱

- **預估時間**：10 分鐘
- **帶領要點**：
  - 「刻意搞壞」是這題的靈魂—學員不敢故意讓程式爆掉，講師要示範「來，我把 calculator 改成永遠回 ERROR」。
  - 陷阱二（工具幻覺）最值得講：很多初學者不知道模型會呼叫不存在的工具。
  - 結束時問學員：「你 Part 5 的工具有哪個陷阱沒防？」這是引導遷移的關鍵提問。
- **常見卡點**：
  - 防禦寫在錯的層級（在 prompt 防無限迴圈）。max_turns 要寫在程式，不是 prompt。
  - structured error 沒回給模型，只寫到 log，模型看不到修正機會。
  - 忘了冪等保護，重試時對外部 API 重複扣款/發信。
- **示範答案語氣**：
  > 「這三個陷阱防禦的層級不一樣：max_turns 是『程式絕對不讓迴圈超過 10 輪』，schema 驗證是『呼叫前擋下格式錯誤』，structured error 是『讓模型看到失敗並自己修』。三者同時存在，才叫做有防禦。少一個，線上就會出事。」
- **延伸話題**：冪等性（idempotency key）在金流/郵件工具的實作、Agent observability 工具（LangSmith、Braintrust）、多 Agent 協作的 orchestrator 設計。

# CH6-5 · Streaming、Evals 與成本控管

## MP `ch6-5-mp1`｜建 20 題黃金測試集

- **預估時間**：10 分鐘
- **帶領要點**：
  - 學員最容易把金集寫成「同一題的 10 個變體」，要示範「5 種情境各 4 題」的結構。
  - Level 1（exact match）適合分類任務、Level 2（embedding similarity）適合文字任務，兩者都要跑過。
  - baseline 分數「一定要寫下來」，否則下次優化無從比對。
- **常見卡點**：
  - 理想 output 寫得太死（「必須一字不差」），結果模型答對意思但被判失敗。
  - embedding 相似度閾值 0.8 憑感覺，沒校準。
  - 只跑一次就認定 baseline，沒考慮 temperature 造成的浮動。
- **示範答案語氣**：
  > 「20 題金集的重點不是『多』，是『涵蓋真實壞掉的情境』。你客服 bot 如果退換貨類錯很多，你的金集裡退換貨題就該佔 30%，不是平均分配。金集是產品品質的鏡子，它長什麼樣，你的優化方向就往哪個方向去。」
- **延伸話題**：LLM-as-judge 的偏誤（self-preference、position bias）、A/B eval 的 CI 計算、持續 eval pipeline 接 GitHub Actions。

## MP `ch6-5-mp2`｜成本瘦身實驗

- **預估時間**：10 分鐘
- **帶領要點**：
  - 強制「前後都跑 golden set」—只看成本不看品質的優化叫做破壞，不叫優化。
  - 五個策略請學員只挑 2 個，多了講不清楚哪個有效。最常見組合是「換 Haiku」+「開 caching」。
  - 最後一步「設每日預算上限」是硬性要求，不是可選項。
- **常見卡點**：
  - 換模型後品質退步超過 5% 還不回退，執念於省錢。
  - Prompt caching 寫錯位置（放在使用者訊息），完全沒命中。
  - Server-side cache 的 key 設計太嚴（要完全一樣才命中），命中率幾乎 0。
- **示範答案語氣**：
  > 「成本省 30%、品質掉 2%，這叫優化；省 50%、品質掉 15%，這叫退步。永遠先鎖品質門檻（例如金集分數不得低於 X），再動成本。不是反過來。」
- **延伸話題**：Anthropic prompt caching 5m / 1h TTL 成本差異、模型級聯（cascade：先 Haiku，信心不足才升 Sonnet）、每日 budget alert 接 Slack/Email。

# PRAC6-1 · API 呼叫測試頁

## 案例 `prac6-1-c1`｜律所判決書摘要 API 基線

- **預估時間**：15 分鐘（快速過流程，不要學員真的跑 1,000 筆）
- **帶領要點**：
  - 重點在「給合夥人看」的決策表長相—這是工程師最缺的輸出能力。
  - temperature 0 vs 0.7 的差異學員要親眼看到，才懂為什麼法務任務不能用高溫。
  - 推成本時用真實美金、真實流量，不要算完說「看起來不貴」就帶過。
- **常見卡點**：
  - 用 temperature=0 以為就不會幻覺—其實 0 只是穩定，不等於正確。
  - 沒把 `confidence` 欄位加進 JSON schema，導致無法自動 fallback。
  - 月成本算漏 output tokens 或重跑 retries 的成本。
- **示範答案語氣**：
  > 「決策表給合夥人看，他只在乎三欄：每月要多少錢、錯誤率多少、錯了會怎樣。Haiku 月 7 塊但錯誤率 8%，Sonnet 月 25 塊但錯誤率 1%。結論是如果摘要會給客戶看就上 Sonnet，內部索引就 Haiku。這一句話，他 30 秒就能決策。」
- **延伸話題**：Extended Thinking 在法律推理題的表現、判決書 OCR 的 prompt caching 實作、給客戶看的輸出須通過法務覆核的流程設計。

## 案例 `prac6-1-c2`｜電商客服 A/B（舊模型換 Claude）

- **預估時間**：15 分鐘
- **帶領要點**：
  - golden set 從真實客服紀錄抽 20 題是關鍵—不是憑空想的。
  - 「分層策略」（高頻用 Haiku、高風險用 Sonnet）是正確答案的核心，直接指出來。
  - 人工覆核機制不能省，示範 confidence < 0.8 轉人工的 flow。
- **常見卡點**：
  - 沒分題型就跑整體分數，掩蓋退換貨類大幅退步的問題。
  - 把舊模型月成本當 benchmark，忘了算改版後的「客訴減少 × 單件成本」ROI。
  - 影子模式沒跑就直接切流量。
- **示範答案語氣**：
  > 「主管想聽的不是『換不換』，是『要花多少、省多少、風險在哪、怎麼擋』。你這份 A/B 能給他分層建議 + 影子模式 + human-in-the-loop，他會直接簽。主管最怕拿到『我覺得 Sonnet 比較好』這種沒數據的結論。」
- **延伸話題**：客服轉人工的 SLA 設計、confidence 自評的 prompt engineering、客服 log 用來持續擴充 golden set 的 pipeline。

# PRAC6-2 · 最小文件問答器

## 案例 `prac6-2-c1`｜保單條款 RAG

- **預估時間**：15-18 分鐘
- **帶領要點**：
  - 強調「條號引用」是這案例的命脈—業務拿到沒條號的答案比沒答案還慘（會背錯去跟客戶講）。
  - chunking「按條切」vs「固定切」的 trade-off 要直接示範：按條切當條款長度差很多時會有大 chunk 撐爆 context。
  - 拒答能力比答對率更重要—「本條款未明文規定」是合格答案。
- **常見卡點**：
  - 條號跟著 chunk 一起切掉，retrieval 命中但引用不出條號。
  - 金額/日期幻覺—「通常會賠」這種話一出現就是大問題。
  - 多條款衝突題（同時踩第 14、15 條）只引一條，漏了另一條。
- **示範答案語氣**：
  > 「保險 RAG 跟一般 RAG 差在哪？差在『答錯會賠錢』。所以你這版 system prompt 要寫死兩件事：必引條號、找不到就說沒規定。其他優化都是其次。記住：在金融/法律領域，拒答是美德，不是失敗。」
- **延伸話題**：條款版本 metadata（新舊版切換）、多語條款（繁/簡/英）的 embedding 選型、業務 app 整合時的「點條號跳原文」UI。

## 案例 `prac6-2-c2`｜員工手冊 FAQ bot

- **預估時間**：15 分鐘
- **帶領要點**：
  - 章節路徑 metadata 是這題的核心—示範 chunk 上帶 `{chapter: "3.1", path: "請假制度 > 特別休假"}`。
  - 「轉單機制」的觸發條件要寫具體（手冊未明文 + 涉及個人權益 + confidence 低於 X），不要寫「答不出來就轉」。
  - 「答案控制 3 段內」是現場最容易被忽略的需求—同仁看到 500 字回覆會直接放棄。
- **常見卡點**：
  - metadata 沒帶章節，答案寫「依手冊規定」卻說不出是哪章。
  - 邊界題（安胎假這種手冊沒寫的）硬答，錯給法令。
  - 年資計算題沒區分「到職週年 vs 曆年」，數字錯。
- **示範答案語氣**：
  > 「HR bot 的成功指標不是『能答幾題』，是『能讓 HR 每週少花幾小時』。如果你的 bot 連『安胎假幾天』這種手冊沒寫的題都硬答，HR 反而要花更多時間去收拾。寧可多轉單，不要錯答。」
- **延伸話題**：多身份（正職/派遣/實習）的 metadata filter、中英混用員工手冊的 multilingual embedding、HR dashboard 的問答紀錄分析 loop。

# PRAC6-3 · Agent 工具鏈小案例

## 案例 `prac6-3-c1`｜3C 電商商品推薦 Agent

- **預估時間**：18-20 分鐘（工具設計要時間）
- **帶領要點**：
  - 先讓學員「手畫 ReAct trace」再寫 schema—不畫流程就寫工具的學員，schema 永遠漏欄位。
  - `description` 欄位的品質決定 Agent 好壞—示範 bad/good description 的對比（例：「搜尋商品」vs「依據 category、預算、規格搜尋符合條件的商品；客戶提到預算或用途時使用」）。
  - Guardrail（max_turns、成本上限）一定要設，這是工程師級別的要求。
- **常見卡點**：
  - tool schema 沒寫 enum，category 傳進「3C」「筆電」「電腦」三種版本。
  - 沒驗 `is_in_stock`，推了缺貨商品。
  - max_turns 設太低（3），實際需要 5-6 輪才收斂。
- **示範答案語氣**：
  > 「這個 Agent 看起來花俏，但它的價值只有一個：把客戶模糊的一句話拆成多個具體查詢再組合。你看 Turn 1 是搜尋、Turn 2-3 是比規格、Turn 4 是配件、Turn 5 是算折扣、Turn 6 是建車。少了任何一步都會斷。」
- **延伸話題**：Tool result caching 避免重複呼叫、需求變更的 context 管理（全清 vs 接續）、A/B 測試 Agent 版本 vs 傳統 faceted search 的轉換率。

## 案例 `prac6-3-c2`｜醫師助理病歷摘要

- **預估時間**：20 分鐘（這題最嚴肅，不要趕）
- **帶領要點**：
  - 開場直接講：「這 Agent 的正確率不是 90% 就好，要接近 100%，因為錯了醫師會誤判」。這個語境壓力要傳遞。
  - 「絕對不做的三件事」是本案例靈魂—不給處方建議、不編造欄位、不主動聯絡病患。
  - 資料來源可驗是醫療 Agent 的最基本要求，每個數字必須標記來自哪次檢驗。
- **常見卡點**：
  - 想讓 Agent「體貼」加一句「建議改用 SGLT2i」—這是越權，必扣分。
  - 檢驗值空值用 0 或 N/A 填，應寫「未取得」。
  - 幻覺風險還寫成警示紅字，應降級為「請醫師確認」。
- **示範答案語氣**：
  > 「醫療 Agent 的角色是『資訊整合者』，不是『決策建議者』。你把最近三次 HbA1c 排好、把交互作用查好、把 guideline 貼好，醫師用兩眼掃過 30 秒做判斷，這就成功了。任何『建議』字眼出現就是失敗，因為那是醫師的 scope。」
- **延伸話題**：HIPAA/個資法合規的資料處理、醫療 LLM 的 red teaming 流程、確定性高的工具（SQL 查庫）優先於模型推理的設計哲學。

# PRAC6-4 · Streaming 回應改造

## 案例 `prac6-4-c1`｜銀行客服黃金集 evals

- **預估時間**：18-20 分鐘
- **帶領要點**：
  - 法遵的角度比工程角度重要：Safety 100% 是死線，其他都可談判。
  - LLM-as-judge 與人工對齊率是關鍵驗證—沒對齊 85% 以上，judge 分數等於沒跑。
  - 「分流 20% 影子模式 2 週」是標準答案，直接帶入，不要讓學員自己摸。
- **常見卡點**：
  - 只看 intent 不看 action—意圖分對了但動作錯（掛失類沒轉專線）。
  - LLM judge 用同家族模型（Sonnet 評 Sonnet），self-preference bias 全跑出來。
  - 紅隊測試（敵意 prompt）漏了，上線後被社工攻擊。
- **示範答案語氣**：
  > 「法遵部門拿到你的報告，他只看兩件事：Safety 有沒有退步、上線計畫有沒有煞車機制。你的報告給 Safety 100%、影子模式 2 週、每日抽驗 50 通、不達標不切流量—這份報告他會簽。給他『Sonnet 比 Haiku 好』的一句話結論，他會退件。」
- **延伸話題**：金融業特有的 prompt injection 風險、多輪對話的 session eval（不是單題）、退步偵測（drift detection）的自動化 alert。

## 案例 `prac6-4-c2`｜新聞即時摘要 Streaming Guardrail

- **預估時間**：20 分鐘（演算法要時間）
- **帶領要點**：
  - Mid-stream 中斷是進階技術，示範「為什麼等摘要完再校對來不及」（讀者已經看到錯數字了）。
  - 數字 fact-check 的演算法邏輯：從原文抽所有數字建 set，streaming 每 chunk 查是否在 set 內。
  - 降級策略（中斷後跑 Sonnet + temp=0 非 stream 版）是關鍵—不能只丟 error。
- **常見卡點**：
  - 把近似值（「約 1.2 億」vs 原文「1.2 億」）誤判為幻覺。
  - fact-check 延遲太高，幻覺字已經顯示給讀者才中斷。
  - 對每個 chunk 都呼叫 LLM 驗證，成本爆炸。
- **示範答案語氣**：
  > 「Streaming guardrail 的核心矛盾是『速度 vs 正確性』。你擋得太兇，讀者一直看到『校對中』；擋得太鬆，錯數字已經上螢幕。500ms 延遲、偽警報 < 10%、降級有版本，這三個指標同時達標才算過關。」
- **延伸話題**：NER 輕量方案（regex + lookup）vs LLM fact-check 的取捨、prompt caching 在 streaming 場景的實作、A/B 測試「streaming + guardrail」vs「非 stream 完整校對」的讀者留存。

# 同儕互評 `prac6-1-p1`｜RAG 幻覺獵人

- **預估時間**：25 分鐘（網頁標 25，足夠）
- **帶領要點**：
  - 開場先講一個血淋淋例子：「某銀行 RAG 跟客戶說『你的保單包含海外醫療』，條款根本沒這條，後來真的去理賠被拒」。壓力上來學員才會認真找漏洞。
  - B 角色要當「刁鑽對手」，不是「禮貌同學」—講師要鼓勵學員真的吐槽。
  - 最後改版重跑 1 題是硬性要求，否則只是理論練習。

## 配對策略

- **技術落差大時**：高階 + 中階配對，讓中階扮演 B（幻覺獵人），逼他學批判視角；高階扮演 A（被挑錯），練被 review 的肚量。
- **技術落差小時**：兩人都跑兩輪，互換角色。
- **人數奇數時**：講師加入湊對，不要讓學員落單觀戰（觀戰學不到）。
- **關係緊繃時**：主動拆開熟識兩人組，混進不熟的人—同儕互評最忌 echo chamber。

## 時間管理

- 8 分鐘 A 產答案時，講師巡場檢查「題目有沒有挑邊界」—看到挑太簡單的立刻介入要求重挑。
- 8 分鐘 B 標註時，講師不介入，讓他獨立思考。
- 6 分鐘對談是全活動最關鍵，講師要「聽對談」而非「維持秩序」，聽到好的判斷當場點名表揚。
- 2 分鐘改版是硬計時—時間到立即結束，即使還沒跑完。

## 冷場處理

- B 標不到 9 個幻覺點：提示「找無根據數字先」「找過度概化詞（通常、大部分、一般）」。
- A 不服氣 B 的標記：介入說「先別辯，把 chunk 原文讀第二次再辯」。
- 兩人沉默：丟一題「如果 A 問的是『公司特休幾天』，chunk 裡寫的是『年資 X 年給 Y 天』，AI 說『全體員工每年 14 天』—這算幻覺嗎？」引爆討論。
- 提早結束：要求他們用剩下時間設計「如果這個 RAG 上線，你會監控哪三個指標」。

## 示範答案語氣

> 「你這句『大部分情況下都會理賠』就是典型過度概化—chunk 原文根本沒說『大部分』。這是模型在填空。修法不用改 chunking，改 system prompt 加『不得使用通常、大部分、一般等模糊詞』，幻覺率會直接掉。」

## 延伸話題

- 幻覺檢測的自動化（每則回答跑一次 fact-check prompt）、上線後的幻覺日誌收集與 golden set 擴充、跨文件衝突時的引用策略。

# Part 通關檢測 `part6`

## 觀念題｜10 題（10-15 分鐘）

- **預估時間**：15 分鐘（含講解）
- **帶領要點**：
  - 建議分批講解，不要 10 題一次解完。2-3 題一組，間隔討論。
  - 每題展開解析後，加一句「你在哪個 MP / 案例遇過這情境？」觸發學員回想。
  - 錯題不要當場糾正，先問「你選這個的理由」—通常可以聽出觀念盲點。

### 講解節奏建議

| 題組 | 主題 | 節奏 |
|------|------|------|
| Q1, Q7 | Streaming 基礎 | 2 分鐘組講完，示範 SSE 實際封包 |
| Q2, Q3, Q4, Q10 | Embedding / RAG | 5 分鐘講透，這是本 Part 最重的觀念 |
| Q5, Q6 | Agent | 3 分鐘，重點在「何時該用、何時不該用」 |
| Q8, Q9 | Evals | 3 分鐘，LLM-as-judge 的 bias 是亮點 |

## 綜合任務｜1 題（25 分鐘）

- **預估時間**：25-30 分鐘（自己寫 20 分鐘 + 同儕對讀 5-10 分鐘）
- **帶領要點**：
  - 場景選擇讓學員自由（法務審約 / 客服知識庫 / 產品規格），但要求「選一個你自己會用到的」。
  - 提案格式是 markdown 1-2 頁，不是文件堆疊—訓練學員把事情講清楚。
  - 交換給同儕讀，如果對方讀完要追問超過 3 個問題，提案就是沒寫清楚，要回去補。

### 綜合任務評分準則

| 向度 | 0 分 | 5 分 | 10 分 |
|------|------|------|------|
| **chunking 有理由** | 寫「用固定 512」沒說為什麼 | 有提 chunk size 與 overlap 但沒說 trade-off | 明確寫「因為文件結構 X，所以選 Y，trade-off 是 Z」 |
| **金集具代表性** | 題目 < 20 或全同類 | 20 題覆蓋多類但邊界題 < 3 題 | 20 題含邊界、高風險、日常，每題有 pass 條件 |
| **失敗分類可行動** | 只寫「會幻覺」 | 分 3-4 類但修法模糊 | 4 類以上失敗，每類對應具體修法（prompt/chunk/embedding/rerank） |
| **通得過同儕審閱** | 同儕追問 > 5 個 | 同儕追問 2-3 個 | 同儕讀完能直接實作，追問 < 2 個 |

- **及格線**：4 向度總分 ≥ 30/40（相當於每項 7.5 分以上）
- **本卷合格門檻**：觀念題 70 分 + 綜合任務 30 分 = 100 分（滿分 140）。70% 及格。
- **不及格處置**：不允許跳過，回去補 PRAC6-1 或 PRAC6-2 其中一題案例，重寫提案。

### 常見卡點

- 場景選太大（「全公司知識管理」）：要求縮到「法務審 NDA」「客服退換貨」這種單一 use case。
- 金集只有 happy path：講師追問「客戶故意套話你擋不擋？手冊沒寫你答不答？」引導補邊界題。
- 失敗分類寫成「回答很糟」：要求分診到「retrieval 錯 / generation 錯 / prompt 錯」層級。

### 示範答案語氣

> 「你這份提案的強項是 chunking 理由清楚、金集覆蓋夠。弱項是失敗分類只寫了『幻覺』一類，實際 RAG 失敗有 4 種模式，你只抓到 1 種。回去參考 CH6-3 的『幻覺 / 漏找 / 找到但沒用』再補一類『跨文件衝突』就完整了。」

### 延伸話題

- 這份提案能否上真的 RAG 專案？（引導學員思考從「作業」到「產品」的距離）
- 上線後 3 個月要監控什麼指標？（Recall@3、拒答率、LLM judge 分數、人工抽驗一致率）
- 第二版優化的優先順序（先抓量最多的失敗類型，不要先做最酷的優化）
