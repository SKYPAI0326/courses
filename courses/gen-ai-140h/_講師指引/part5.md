## CH5-1 看懂 HTML / CSS / JavaScript 的最低必要知識

### MP `ch5-1-mp1` · 用 DevTools 解剖一個你常用的網頁
- **預估時間**：8 分鐘
- **帶領要點**：先全班一起對 Google 首頁做 1 次示範，再各自練自選頁面。重點不是記住標籤名，是建立「右鍵檢查 → 認出區塊」的反射動作。
- **常見卡點**：
  - 學員打開 Inspect 後盯著整顆 DOM 樹發呆 → 教他用「選取元素箭頭（Cmd+Shift+C / Ctrl+Shift+C）」點頁面，讓 DOM 自動滾到該節點。
  - 把 class 和 tag 搞混 → 講「tag 是積木的類型，class 是你給這塊積木貼的標籤」。
- **示範答案語氣**（可朗讀）：
  > 「我打開 Google 搜尋結果頁，右鍵點搜尋框選『檢查』。Styles 旁邊是 Elements，我看到搜尋框是一個 `<textarea>`，外面包一個 `<form>` 和 `<div class="RNNXgb">`。所以搜尋框這一整塊在 HTML 結構上是『form 裡放 textarea』，class 命名看起來是亂碼，這是大廠編譯過的結果，不用記。」
- **延伸話題**：可連到「為什麼大公司的 class 都是亂碼」講 CSS Modules / utility class 的演化；或示範 Reader Mode 把 HTML 結構講光。

### MP `ch5-1-mp2` · 用 DevTools 即時改外觀，學 CSS 屬性對應
- **預估時間**：10 分鐘
- **帶領要點**：強調「改的是記憶體中的副本，不會動到原始網站」，鼓勵大膽改。要求至少改 3 個屬性：`color`、`font-size`、`background-color`。
- **常見卡點**：
  - 學員改了沒反應 → 通常是被更高優先序覆蓋，教他看 Styles 面板的刪除線。
  - 刷新頁面後改動消失就慌了 → 強調這是 DevTools 的 feature，不是 bug。
- **示範答案語氣**：
  > 「我把某新聞網主標題的顏色改成紅色，字體調 32px，背景加淡黃色。我發現我原本寫的 `color: red` 被下面一條 `color: #222 !important` 劃掉了，原來 `!important` 是最大招，平常寫 CSS 不要亂用。」
- **延伸話題**：CSS specificity（特異性）速記法、`:hover` 狀態怎麼在 DevTools 鎖住來調樣式（Force state）。

### MP `ch5-1-mp3` · 請 AI 產一段 JS，然後用白話翻譯它
- **預估時間**：10 分鐘
- **帶領要點**：這題的目的不是寫程式，是訓練「逐行讀 AI 給的程式」。要求每個學員必須用中文寫出每一行的功能，不能抄註解。
- **常見卡點**：
  - 學員直接把 AI 的英文註解複製貼上 → 要求用自己的話寫，而且不超過 15 字一行。
  - 遇到 `addEventListener`、箭頭函式 `=>` 卡住 → 用「這行是『當 X 發生時做 Y』」的句型帶過。
- **示範答案語氣**：
  > 「這行 `document.getElementById('btn').addEventListener('click', ...)` 翻成白話是：『抓畫面上 id 叫 btn 的那個元素，當有人點它的時候，跑括號裡這段。』大括號裡面那段就是『當下要做什麼』。」
- **延伸話題**：示範同一個需求讓 AI 產兩次會有不同寫法，引導學員選「自己看得懂的那版」而不是「看起來最炫的那版」。

## CH5-2 與 AI 協作改頁面，而不是盲貼程式碼

### MP `ch5-2-mp1` · 把一句模糊需求改寫成 AI 能執行的指令
- **預估時間**：8 分鐘
- **帶領要點**：這是 CH5-2 的核心方法論「位置＋意圖＋限制」的第一次練習，要求學員當場把「美化一下頁面」改寫成具體指令。可以用白板收集前後對照。
- **常見卡點**：
  - 寫出來還是「把按鈕弄得漂亮一點」 → 反問「漂亮的標準是什麼？大小？顏色？間距？」
  - 忘了寫限制 → 提示「AI 常會順手改你沒要它動的區塊，所以要明確說『不要動 header』」。
- **示範答案語氣**：
  > 「原本我會說『幫我把送出按鈕美化一下』，改寫後變成：『請修改 `.submit-btn` 這個 class，把背景改成 #b5703a，滑鼠移上去時稍微變暗 10%，其他地方不要動，特別是 header 的 logo 樣式保持不變。』這樣 AI 就不會順手改到整個頁面。」
- **延伸話題**：這個三段式結構其實就是軟體工程裡 user story 的「在 X 情境下，我要 Y，為了 Z」的翻版。

### MP `ch5-2-mp2` · 寫一份「會讓 AI 找到問題」的 bug 報告
- **預估時間**：10 分鐘
- **帶領要點**：要求學員至少包含 4 項：重現步驟、預期結果、實際結果、瀏覽器+版本。強調「AI 只能看你給的文字，你給的資訊品質 = 它能給的答案品質」。
- **常見卡點**：
  - 只寫「按鈕壞了」 → 用「想像你是打 1999，你只說『我家壞了』會被掛電話」比喻。
  - 貼 Console 錯誤訊息時把前後文砍掉 → 要求整段貼（從紅字那行往上 3 行、往下 2 行都要）。
- **示範答案語氣**：
  > 「重現步驟：1. 開啟 BMI 工具頁。2. 身高輸入 170、體重 65。3. 點『計算』按鈕。預期結果：顯示 BMI 值。實際結果：按鈕沒反應，Console 出現 `Uncaught TypeError: Cannot read properties of null (reading 'value')`。環境：Chrome 126 / macOS 14。」
- **延伸話題**：這份格式直接拿去發 GitHub Issue 或跟後端同事溝通都通用。

### MP `ch5-2-mp3` · 為自己選一套「主力 + 次要」AI 編碼工具組合
- **預估時間**：8 分鐘
- **帶領要點**：不是叫學員買工具，是讓他們對比用過的幾個（ChatGPT、Claude、Cursor、Copilot、Gemini）寫出各自的擅長/不擅長。這是 metacognition 練習。
- **常見卡點**：
  - 只用過一個工具就下結論 → 現場讓他們各挑一個還沒試過的，當堂跑 1 題相同需求對比。
  - 把「我習慣用」誤認為「它最好」 → 提醒這是偏好不是評價。
- **示範答案語氣**：
  > 「我的主力選 Claude，因為改中大型 HTML 時它比較不會亂重構；次要用 ChatGPT，因為需要快速查 API 語法或找 snippet 時它回答比較短。我不選 Cursor 當主力是因為它在 inline 補全會打斷我思考節奏，我還在適應。」
- **延伸話題**：把這份筆記貼在工作桌上，3 個月後再回來看會不會變——工具版本更新很快，半年後答案可能全翻。

## CH5-3 Git 與版本保存

### MP `ch5-3-mp1` · 把一個現有資料夾變成 Git 倉庫，做出第一個 commit
- **預估時間**：10 分鐘
- **帶領要點**：一定要現場實機跑，不是看投影片。命令順序：`git init` → `git add .` → `git commit -m "初次提交：OOO 專案"`。之後要求跑 `git log` 看到那行訊息。
- **常見卡點**：
  - 學員忘了先設 user.name / user.email → Mac 會擋 commit，給一條萬用設定：`git config --global user.name "你的英文名"`。
  - 用 `git add .` 把 node_modules / 截圖 / 影片全都加進去 → 趁此講 `.gitignore` 基本模板。
- **示範答案語氣**：
  > 「我把桌面上那個之前做的 BMI 工具資料夾打開 Terminal，先跑 `git init`，會看到 `Initialized empty Git repository`。接著 `git add .`，再 `git commit -m "初次提交：BMI 單頁工具"`，最後 `git log` 看到我的 commit 出現在歷史第一行，代表這個資料夾現在有 Git 在守著了。」
- **延伸話題**：`.gitignore` 是所有人第二天就該學的，否則很快會 commit 到 .DS_Store 或 API key。

### MP `ch5-3-mp2` · 刻意改壞一個檔案，再用 Git 救回來
- **預估時間**：8 分鐘
- **帶領要點**：這題的教學價值是讓學員體驗「Git 是救命而不是麻煩」。指令：`git diff` 確認改壞內容 → `git checkout -- <file>` 或 `git restore <file>` 還原。
- **常見卡點**：
  - 改檔後先 `git add` 了才要還原 → 要多跑一步 `git restore --staged <file>`，順序反了會卡住。
  - 誤以為還原後改壞內容還能救 → 強調「沒 commit 的還原就是徹底消失」，這是為什麼要勤 commit。
- **示範答案語氣**：
  > 「我故意把 index.html 第一個 `<h1>` 整行刪掉存檔，瀏覽器確認真的不見了。接著跑 `git diff` Git 把我刪掉的那行用紅色顯示出來。我打 `git restore index.html`，回到瀏覽器刷新，標題回來了。原來 Git 是我的 ctrl+Z 但不會因為關電腦就消失。」
- **延伸話題**：延伸到 `git stash` 用在「改到一半但老闆叫你切回 main」的場景。

### MP `ch5-3-mp3` · 把本地 repo 推到 GitHub，拿到公開網址
- **預估時間**：10 分鐘
- **帶領要點**：最容易卡的單元，要全班同步跑。流程：GitHub 網頁建 repo → 複製 remote URL → `git remote add origin <url>` → `git push -u origin main` → Settings 啟用 Pages。
- **常見卡點**：
  - HTTPS 推上去要輸入密碼卻跳 Personal Access Token 錯誤 → 改用 SSH 或 GitHub CLI `gh auth login`。
  - 預設分支是 master 不是 main → 示範一次改名 `git branch -M main`。
  - Pages 啟用後立刻開網址 404 → 告訴他們通常 1–3 分鐘才生效。
- **示範答案語氣**：
  > 「我在 GitHub 點 New repository，取名 `bmi-tool`，不要勾初始化 README（會衝突）。複製 SSH URL，回 Terminal 跑 `git remote add origin git@github.com:你的帳號/bmi-tool.git`，再 `git push -u origin main`。推完到 Settings → Pages 選 main 分支 / root 路徑，等 2 分鐘後打開 `https://你的帳號.github.io/bmi-tool/`，看到剛剛寫的首頁。」
- **延伸話題**：帶到 Vercel / Netlify 的自動部署（push 即上線），以及 custom domain 的概念。

## CH5-4 含後端服務的完整部署

### MP `ch5-4-mp1` · 判斷你手上 3 個點子「需不需要後端」
- **預估時間**：5 分鐘
- **帶領要點**：強化「做技術決策前先分類」的習慣。3 個判準：要存資料嗎？要用 API key 嗎？要跨使用者嗎？任一個 yes 就需要後端。
- **常見卡點**：
  - 學員把「要儲存到 localStorage」也算成需要後端 → localStorage 是前端儲存，不算。
  - 把「會呼叫 Gemini API」誤判為純前端能做 → 這題是 mp2 的鋪墊，強調有 API key 就需要代理。
- **示範答案語氣**：
  > 「我列出 3 個點子：A. 個人 BMI 工具，不存資料不呼叫 API → 純前端。B. 部落格留言板，要存留言、跨使用者 → 需要後端。C. AI 翻譯小工具，要呼叫 OpenAI API → 需要後端代理（金鑰不能放前端）。」
- **延伸話題**：Serverless Function（Vercel Edge / Cloudflare Workers）是「需要一點點後端」的折衷解，比自架伺服器輕。

### MP `ch5-4-mp2` · 寫一份「我的 API 金鑰 .gitignore 檢查表」
- **預估時間**：8 分鐘
- **帶領要點**：學員要產一份自己的 checklist，不是抄模板。要項：`.env` / `config.local.*` / `*.pem` / 截圖資料夾 / 個人 credentials 檔名。
- **常見卡點**：
  - 學員覺得「我 repo 是 private 所以沒差」 → 舉例 GitHub 曾經 public/private 設定誤切換、員工離職帳號外流的案例。
  - `.gitignore` 加了但之前已經 commit 過 → 要多一步 `git rm --cached <file>` 才會真的停追蹤。
- **示範答案語氣**：
  > 「我的檢查表五條：1. `.env` 2. `.env.local` 3. `credentials.json` 4. `*.pem` 5. `/screenshots/`。檢查方法：每次 commit 前先 `git status` 掃一眼，有任何可疑檔名立刻停下來處理，寧可多花 30 秒也不要 key 外洩。」
- **延伸話題**：git-secrets 或 trufflehog 這類自動掃描工具，適合團隊做 pre-commit hook。

### MP `ch5-4-mp3` · 把一個最小 Node 後端部署到 Railway 拿到公開網址
- **預估時間**：10 分鐘
- **帶領要點**：課堂時間有限，建議提前把一個最小專案（Express + 一支 `/hello` 路由）放在 GitHub template repo，讓學員 fork 後直接 Railway 連進去。
- **常見卡點**：
  - Railway 預設 port 不是 3000 → 提醒用 `process.env.PORT` 而不是 hardcode。
  - 忘記把 `start` script 寫在 `package.json` → Railway 會起不來，Logs 面板看錯誤訊息。
  - 免費額度跑到限制 → 提前告知 Railway 現在是信用卡驗證 + 有使用上限，準備 backup（Render / Fly.io）。
- **示範答案語氣**：
  > 「我把 template repo fork 過來，Railway 點 New Project → Deploy from GitHub → 選這個 repo。不用改任何設定，它偵測到 Node 自動跑 `npm install` 跟 `npm start`。約 90 秒後拿到 `https://xxxxx.up.railway.app`，開啟 `/hello` 看到 JSON 回應，代表從本機程式碼到公開可訪問後端，我真的自己走完一遍了。」
- **延伸話題**：環境變數怎麼在 Railway 面板設、部署失敗怎麼看 Logs、自訂網域要怎麼接。

## PRAC5-1 單頁工具 v2 改版任務

### Case `prac5-1-c1` · 接手三個月前離職同事留下的單頁 BMI 工具
- **預估時間**：20 分鐘
- **帶領要點**：這是 Part 5 的第一個案例，要帶出「讀陌生碼 → 列 v2 目標 → 動手」的三步法。強調先讀懂再改，而不是邊讀邊改。
- **常見卡點**：
  - 學員直接跳下去改 CSS → 要求先寫一份「v2 要改的 3 條清單」才能動。
  - 把整個 HTML 貼給 AI 要它重寫 → 這是反例，示範用位置＋意圖＋限制的指令替代。
- **示範答案語氣**：
  > 「我先花 5 分鐘讀一遍檔案，看 head 的 title、body 的結構，發現是單一 form + script。v2 目標我列 3 條：（1）按鈕配色跟品牌對齊。（2）結果加一段 BMI 區間說明。（3）在手機上字級放大。接著一次改一條，每改完 commit 一次，不要全部改完才一口氣 commit。」
- **延伸話題**：真實接案場景 80% 都是接別人的 code，這個題型比「從零開寫」更貼近工作實況。

### Peer `prac5-1-p1` · 我寫的 commit，你當 PR reviewer 挑毛病
- **預估時間**：30 分鐘（5+8+5+10+2，嚴格按節奏跑）
- **帶領要點**：全 Part 5 唯一的同儕互評，份量重但效果最好。先全班對齊規則再開始。
- **配對策略**：
  - **程度配對**：依 PRAC5-1 完成度分三級，中級配中級、高級配初級（高級當示範者），不要把兩個初級放一組會互相卡住。
  - **背景混搭**：有轉職學員就搭現職工程師；都是轉職則按 commit 數量排序兩兩配（commit 多的 vs 少的）。
  - **人數不成雙**：三人一組，A→B→C→A 輪一圈，每段時間乘 1.5 倍。
- **時間管理**：
  - 講師必須**全程拿計時器**按節奏走，不允許「再 3 分鐘」。
  - Step 3 的「A 只能聽不能辯解」是最難守的，見到 A 開口立刻提醒「現在是吸收階段」。
  - 最後 2 分鐘的「當場改進」是重點，不要省略 —— 這是整個活動的閉環。
- **冷場處理**：
  - B 寫不出建議 → 指定他用三指標（可讀性 / 必要性 / 可維護）逐項打 1–5 分，分數低的那項就是建議切入點。
  - A 覺得被挑得沒面子 → 強調「被挑毛病的是 commit 不是你」，這是所有工程團隊的日常。
  - 兩人彼此熟到不敢講真話 → 拋出一個示範評語：「這個 commit 混了 3 個不相干的改動，下次請拆開。」讓尷尬破冰。
- **常見卡點**：
  - 學員的 commit 太小沒東西可 review → 要求挑「最有代表性的一個 feature commit」，而不是 `fix typo`。
  - Diff 工具不熟 → 現場教 `git show <commit>` 或 GitHub 網頁 Compare 視圖。
- **示範答案語氣**（B 當評審時）：
  > 「可讀性 4 分，你的 class 命名 `btn-red` 直接寫死顏色，未來改色要全檔搜尋，建議改 `btn-primary`。必要性 5 分，這個 commit 就是為了主 CTA 一致性，目標清楚。可維護 3 分，你把 hover 樣式直接 inline 寫在 HTML 的 style 屬性，之後其他按鈕要跟進會抄得很辛苦，建議抽到 CSS 的 `:hover`。替代方案：把顏色變數抽到 `:root` 的 `--c-primary`，之後全站統一。」
- **延伸話題**：GitHub 真實 PR 的 review comment 怎麼寫（建設性而非攻擊性）、Conventional Commits（`feat:` / `fix:` / `refactor:`）。

## PRAC5-2 靜態頁面部署到 Vercel / Netlify

### Case `prac5-2-c1` · 手沖咖啡小店要把菜單頁推上 GitHub Pages 公開
- **預估時間**：20 分鐘
- **帶領要點**：這題重點不是技術，是「決策」：為什麼選 GitHub Pages（免費 + 方便）vs Vercel（Edge 快 + 自動 preview）？引導學員自己推導。
- **常見卡點**：
  - 只試一種平台就收尾 → 要求至少推兩個平台拿到兩個網址，才能比較。
  - 部署完發現圖片路徑壞了 → 幾乎都是絕對路徑寫死或大小寫錯誤。
- **示範答案語氣**：
  > 「我先推 GitHub Pages 當備份站，因為它跟 repo 綁在一起永遠不會失聯；再推 Vercel 當主力，因為每次 push 會自動出 preview URL，老闆審稿時我直接丟 preview 網址給他。」
- **延伸話題**：custom domain 綁定、什麼情況會需要 CDN（大量圖片或影片）。

## PRAC5-3 個人作品展示首頁

### Case `prac5-3-c1` · 轉職前端的第 90 天，作品集首頁要投 3 家公司
- **預估時間**：20 分鐘
- **帶領要點**：很多學員會「設計導向」做到天亮，引導他們用 60/20/20 原則：60% 作品內容、20% 自我介紹、20% 聯繫方式。不要追求動畫特效。
- **常見卡點**：
  - 作品只有 1–2 個硬要做 portfolio → 建議改成「單作品深入頁」，講清楚一個專案比列 5 個半成品值錢。
  - 堆 CSS 動畫堆到讀取慢 → 用 Lighthouse 打一下效能分數，學員會被分數刺激收斂。
- **示範答案語氣**：
  > 「我只有 2 個作品，所以首頁上半做一個大大的自我介紹 + 1 句話定位『我擅長把設計稿變成可部署的單頁工具』，下半放兩張作品卡片，點進去是詳細頁。這樣比硬湊 5 個空殼作品更像一個真的想被錄取的人。」
- **延伸話題**：HR 看作品集的平均時間是 30 秒，首屏訊息密度是生死線。

## PRAC5-4 含後端代理的完整部署

### Case `prac5-4-c1` · 客戶要整合 OpenAI，但 key 放前端第三天就被偷了
- **預估時間**：20 分鐘
- **帶領要點**：這是 Part 5 最貼近真實事故的案例。一定要講「為什麼 key 外洩這麼快」—— 爬蟲 + GitHub 全站索引 + Telegram 機器人秒檢測。
- **常見卡點**：
  - 學員以為「混淆一下就好」 → 示範把 minify 過的 JS 用 Sources 面板 Pretty Print 按鈕還原，混淆無意義。
  - 代理層做了但忘了加 rate limit → 延伸講「外洩的不只是 key，還可能是你的代理 endpoint 被濫用」。
- **示範答案語氣**：
  > 「正確做法：前端不存任何 key，只呼叫自己的 `/api/proxy-openai` endpoint。代理層讀 `process.env.OPENAI_API_KEY`（Railway 面板設）再轉發。額外加：代理層做 IP rate limit 每分鐘 10 次、檢查 Origin header 只允許自己網域、Logs 送告警通知。客戶看到這三招就知道我有做功課。」
- **延伸話題**：OWASP Top 10、API key rotation（定期換金鑰）、secret scanning 工具。

## PRAC5-5 專案進度紅綠燈看板

### Case `prac5-5-c1` · 週一早會要對齊三條產品線進度，會議時間只剩 15 分鐘
- **預估時間**：15 分鐘
- **帶領要點**：這個案例的教學目的是「規約化狀態」。為什麼只用紅/黃/綠三色？因為多一個色就多一個討論，15 分鐘開不完。
- **常見卡點**：
  - 學員加了「淺綠」「深黃」搞出 5 種顏色 → 退回去，三色是鐵律。
  - 卡片描述寫整段文字 → 改成「進度數字 + 一句話 blocker」。
- **示範答案語氣**：
  > 「產品線 A 綠燈、進度 80%、無 blocker。產品線 B 黃燈、60%、設計稿還卡客戶回覆。產品線 C 紅燈、30%、後端 API 規格要重談。早會每條 3 分鐘，紅燈的那條深入談解法，綠燈的 pass，這樣 15 分鐘絕對夠。」
- **延伸話題**：RAG status（Red/Amber/Green）是 PM 界通用語言，這題其實在教管理方法論。

## PRAC5-6 面試評價橫向對比表

### Case `prac5-6-c1` · 三位前端 Junior 候選人，下週要決定誰發 offer
- **預估時間**：20 分鐘
- **帶領要點**：這題的核心是「定義評分維度」比「做表格」重要。花 10 分鐘定義 4–6 個維度（技術基礎、溝通、學習力、文化契合、作品深度），剩下 10 分鐘填表並做推薦決策。
- **常見卡點**：
  - 維度加到 10 個以上 → 大於 6 個會失焦，強制收斂。
  - 「主觀感覺」寫進表格 → 改成「舉證式評分」，每個分數要能講一個面試中的具體片段支撐。
- **示範答案語氣**：
  > 「三位候選人用 5 個維度打分（1–5），A 技術 4 溝通 3 學習力 5 文化 4 作品 3 = 19。B 全 3 = 15。C 技術 5 溝通 2 學習力 2 文化 3 作品 5 = 17。我會發 offer 給 A，因為她溝通雖然不是最強但學習力是三位中最高，長期來看成長曲線最陡。C 技術天花板高但溝通分太低會拖累團隊。」
- **延伸話題**：面試官評分偏誤（月暈效應、對比效應）、structured interview 為什麼比 free chat 準。

## PRAC5-7 站立會議發言抽籤器

### Case `prac5-7-c1` · 每週站立會議 Tim 都第一個發言，其他人變成點頭附議
- **預估時間**：15 分鐘
- **帶領要點**：這是「用工具解社會問題」的範例。學員常見誤區是把重點放在「抽籤演算法多炫」，引導回到「為什麼要抽籤 = 打破發言順序慣性」。
- **常見卡點**：
  - 學員做出「完全隨機」版本，結果 Tim 連續被抽到三週 → 要求加入「上週發過的這週降低機率」的機制。
  - 工具做好沒人用 → 討論「工具導入」的社會面：主管要第一個用、第一週要刻意誇獎用工具後的改善。
- **示範答案語氣**：
  > 「我的抽籤器有三條規則：（1）上週最後發言的這週第一個被抽中機率降半。（2）一直被點頭附議的人（由會議後回饋表決定）下週優先抽到。（3）主管永遠在名單最後一個以免搶話。不是純隨機，是『帶偏好的隨機』，目的是讓發言量平均而不是純公平。」
- **延伸話題**：心理安全感（psychological safety）、Google 亞里士多德專案、會議設計。

## PRAC5-8 Markdown 轉專業報表樣式

### Case `prac5-8-c1` · 接了品牌顧問案，工作坊會議記錄要交給甲方行銷總監
- **預估時間**：20 分鐘
- **帶領要點**：這題考「同一份內容，不同受眾就有不同排版」。工作坊即席筆記給自己是一回事，交給客戶又是另一回事。
- **常見卡點**：
  - 把 Markdown 原始碼直接丟給客戶 → 提醒客戶不會安裝 Markdown viewer，要轉成 PDF / 美化 HTML。
  - 加太多裝飾（陰影、漸層）→ 商業顧問案要「乾淨專業」，不是「視覺設計比賽」。
- **示範答案語氣**：
  > 「我把 Markdown 轉成一份 A4 直式的 HTML 報告，只用三種字級（H1 用 Serif 粗體、H2 分隔線、內文 1rem）。每節結尾加一個『決議 / 待辦 / 風險』的三欄小表。封面頁放專案名 + 日期 + 顧問署名，客戶打開 3 秒就知道這是正式文件不是隨手筆記。」
- **延伸話題**：Paged.js（用 CSS 做分頁列印）、Typora / Pandoc 的匯出選項。

## PRAC5-9 活動倒數與進度壓力牆

### Case `prac5-9-c1` · 品牌線下 workshop 剩 14 天開放報名，團隊 3 人，每天都有事
- **預估時間**：20 分鐘
- **帶領要點**：這題是「時間壓力可視化」練習。關鍵概念：人對「還剩 14 天」沒感覺，但對「倒數牆上的紅條」有感覺。
- **常見卡點**：
  - 學員把所有任務都標紅色 → 紅色必須保留給「今天不做明天就爆」，否則失去預警意義。
  - 倒數牆放在 Notion 沒人點開 → 要求放在團隊每天都會經過的地方（Slack pinned、公司大螢幕、Telegram bot 每日推）。
- **示範答案語氣**：
  > 「我做的壓力牆有三個元素：（1）頂部大字 D-14 每天自動減一。（2）中間三欄任務清單（報名設定 / 內容錄製 / 宣傳貼文），每條標期限。（3）底部進度條，綠=已完成、黃=進行中、紅=逾期。團隊早會前大家打開這頁看一眼就知道今天該衝哪一塊。」
- **延伸話題**：Parkinson's Law（工作會膨脹到填滿所有可用時間）、可視化管理（Lean / Kanban 的源頭）。

## PRAC5-10 文字敏感詞過濾器

### Case `prac5-10-c1` · 金融科技品牌的 IG 小編，每篇貼文發出前都要掃合規
- **預估時間**：20 分鐘
- **帶領要點**：合規敏感詞過濾是真實業務，但這題更深的教學點是「工具只是輔助，最後還是要人判斷」。講明這類工具的定位。
- **常見卡點**：
  - 用 hard-code 的正規表達式寫死清單 → 討論「清單維護責任在誰」，通常是法遵部門。
  - 過度信任工具，工具沒抓到就真的發 → 強調這是「預警」不是「保險」。
- **示範答案語氣**：
  > 「我做的過濾器分三層：（1）紅色詞（保證報酬、無風險）直接擋下不給發。（2）黃色詞（高收益、穩賺）跳警示視窗要二次確認。（3）綠色通過。關鍵是清單檔案跟工具分離，法遵同事可以自己改清單不用找工程師。工具不替他們決定合規，只替他們把關機率。」
- **延伸話題**：金管會廣告規範、content moderation 產業、False Positive vs False Negative 的代價不對稱。

## PRAC5-11 團隊排班熱點圖

### Case `prac5-11-c1` · 5 人手搖飲店下週排班，兩位有硬限制、一位下週大夜
- **預估時間**：20 分鐘
- **帶領要點**：這題是 Part 5 最接近「真實排程問題」的案例。學員會想寫排程演算法，要拉回來：先手動排一輪，知道難在哪再考慮工具。
- **常見卡點**：
  - 沒把硬限制跟軟偏好分開 → 一定要兩層結構，硬限制不能破、軟偏好能滿足最好。
  - 熱點圖顏色太花 → 只用 2–3 色（滿/缺/爆），多了難解讀。
- **示範答案語氣**：
  > 「我的排班表橫軸一週七天、縱軸早中晚三班，每格顯示『需求人數 / 實排人數』。硬限制先鎖：小 A 週二不能上、小 B 只能週末、小 C 下週大夜全週只能上晚班。填完硬限制後空格才開放軟偏好抽籤。最後熱點圖把人力缺口用紅色格標出來，店長一眼看到週四中午只有 1 人（需求 2 人），就要主動調班或徵臨時工。」
- **延伸話題**：OR-Tools 或 Gurobi（工業級排程求解器）、勞基法對連續工時的限制。

## PRAC5-12 個人化 Kanban 任務板

### Case `prac5-12-c1` · 同時跑 4 個案子，早上打開電腦不知道該做哪個
- **預估時間**：15 分鐘
- **帶領要點**：這題是所有自由演練裡「個人生產力」取向最強的。不要變成 Notion 比拼，重點是「每天只有一張 doing 卡片」的規則。
- **常見卡點**：
  - todo 塞 50 張卡片壓力更大 → 強制「todo 最多 10 張，多的進 backlog」。
  - doing 同時 3 張 → 違反 Kanban 的 WIP 限制，一張一張做完才能領下一張。
- **示範答案語氣**：
  > 「我的個人 Kanban 規則：todo 上限 10 張、doing 上限 1 張、done 每週五清空到歷史檔。早上打開電腦第一件事：看 doing 那張卡，沒了就從 todo 領一張拖進 doing。不允許『等一下再決定今天做什麼』，決策一次做完。」
- **延伸話題**：Kanban 的 WIP limit 來源（Toyota 豐田生產系統）、Cal Newport 的 Deep Work、Eisenhower Matrix 四象限。

### Quiz `part5` · 前端與部署實戰 — 關卡驗收
- **預估時間**：40 分鐘（觀念題 10 題 × 1.5 分鐘 ＝ 15 分鐘 + 綜合任務 20 分鐘 + 全班檢討 5 分鐘）
- **組成**：10 題觀念選擇題（每題 10 分，滿分 100）+ 1 題綜合任務（自評 0/5/10，30 分以上算過關）
- **講解節奏**：
  - **先答後講**：全班 15 分鐘獨立作答，禁止開手冊。按計時器到就停筆。
  - **逐題公布**：每題花 1 分鐘公布正解 + 解析，不要學員自己展開 `<details>` 自由看——會跳著看失焦。
  - **錯最多那題放大講**：快速舉手統計「A/B/C/D 各有多少人」，錯最多的那題花 3 分鐘深講。Part 5 經驗上最常錯是第 6 題（`git revert` vs `force push`）跟第 8 題（前端金鑰安全）。
  - **節奏口訣**：答（15分）→ 對（10分）→ 深講（5分）→ 進綜合任務（20分）。
- **綜合任務評分準則**：
  - **任務**：挑一個自做單頁工具，走完 `git init` → GitHub Pages 公開 → 刻意改壞 → `git revert` 回滾 → 附 commit 列表 + 截圖。
  - **10 分（優秀）**：四項驗收全達成，且 commit message 清楚、revert 後線上版本確實回到正確版、有無痕視窗驗證截圖。過程中展現對「為什麼用 revert 不用 force push」的說明。
  - **5 分（及格）**：拿到公開網址 + 至少一次 revert，但 commit message 只有 `update` `fix`，或回滾方法用了 `git reset --hard` + `push -f`（錯誤但達成目的）。
  - **0 分（未過）**：網址打不開、或沒完成 revert、或整段流程是別人代做。
  - **評分重點在過程不是結果**：學員如果做到「公開網址能打開」但不能解釋 revert 跟 reset 差異，應該給 5 分不是 10 分。這題考的是 Git 工作流心智模型。
- **常見卡點**：
  - 第 5 題漏掉 `git remote add origin` 這步驟 → 帶回 CH5-3-mp3 再示範一次完整流程。
  - 第 8 題覺得「就算 key 外洩我改掉就好」→ 講真實案例：OpenAI key 外洩 10 分鐘內會被爬蟲找到，帳單可能跑到五位數美金。
  - 綜合任務卡在「不敢 push 到真的 GitHub」→ 鼓勵建一個測試用 repo，失敗就刪，上線不可怕。
- **示範答案語氣**（綜合任務總結時）：
  > 「這個任務要你證明三件事：你能把本機程式碼送上公開網址、你的 commit 歷史是可讀的故事、你遇到錯版時用的是保留歷史的 revert 而非覆蓋歷史的 force push。拿到 10 分的同學，下次接案時客戶問『你們怎麼處理版本』，你可以直接把今天的 repo URL 丟過去，那就是最強的作品集。」
- **延伸話題**：真實團隊的 PR workflow（branch → PR → review → merge）、GitHub Actions 自動部署、semantic versioning。

## 講師備忘

- **全 Part 5 建議總時數**：Part 5 有 4 章 CH + 12 自由演練 PRAC，若全程帶完所有 MP + Case + Peer + Quiz，組件本身約 360–450 分鐘；加上講解、示範、Q&A，實際教學時數抓 20 小時（對應大綱配置）。
- **優先級**：若時間壓縮，MP 可指派課後自學但必須交成果筆記；Case 必跑（是 Part 核心）；Peer 不可省（是 Part 5 唯一的同儕互評）；Quiz 必跑（是 Part 通關依據）。
- **最常見學員瓶頸**：CH5-3 Git 同步 push 卡關、CH5-4 後端部署卡 Railway 設定、PRAC5-4 金鑰安全概念。這三關一定要預留緩衝時間。
