# gen-ai-140h 學員互動性審核報告

> 給 Codex 的提示：本報告盤點「弄一下工作室」140h 職訓班 39 個已上線 HTML 單元的學員互動性現況，分級指出缺口並提出可落到 HTML 元件層級的改造方案。希望你檢視：(a) 互動性面向定義是否完備；(b) 缺口分級是否合理；(c) 改造方案在 140h 班型下是否實際可行；(d) 工時估算是否現實；(e) 有沒有我漏掉的關鍵互動類型（例如「自我元認知」「跨單元復用」這類）。本報告**不**寫程式碼，只寫設計規格與優先序。

---

## 0. 審核背景

- **課程定位**：140h 職訓班，56 單元（CH ×27 + PRAC ×29），目前 Part 1–5 上線 39 頁，Part 6/7 待建。
- **學員背景（依大綱）**：「有基礎 AI 工具使用經驗，希望系統化提升、具備開發與部署能力的職場工作者」。
- **教學形式假設**：職訓班 = 面授 + 講義 + 同儕協作。本網頁是「講義」，**不是**自學平台 — 它的互動性必須與「課堂節奏」搭配，而非完全自助。
- **既有設計語彙**：無印良品風（米白底 #f5f3ee、Shippori Mincho 標題、禁漸層）。互動元件必須在這個視覺約束下設計，不能引入花俏的 SaaS 風 UI。

**重要前提**：本審核**僅看互動性**，不審內容正確性、不審 SEO、不審視覺合規。內容/合規由 `course-reviewer` 與 `lint-page.py` 負責。

## 1. 互動性五面向定義

把「學員互動」拆成五個關係維度。每一面向都列出在課堂講義頁可實踐的元件型態。

| # | 面向 | 定義 | 在 HTML 講義頁的可實踐元件 |
|---|------|------|-----------------------------|
| I-1 | **與內容（Content）** | 學員主動處理教材本身：填空、選擇、排序、找錯、改寫 | inline `<textarea>`、checkbox、radio、`<details>` 解答揭露、reorderable list |
| I-2 | **與工具（Tool）** | 學員操作 AI 工具/外部工具並把產出帶回 | 內嵌 builder（多 textarea + 組合輸出）、複製按鈕、deeplink 到 ChatGPT/Claude、API embed |
| I-3 | **與同儕（Peer）** | 學員之間的互動：交換作品、互評、分組討論、講解給對方聽 | 「找一位 buddy」段落、可分享的作品摘要卡、互評 rubric checklist |
| I-4 | **與講師（Teacher）** | 學員 ↔ 講師/助教：提問、上交檢核、批改、解鎖下一段 | 「上交給講師」標記點、QR/解鎖碼、講師簽核 checkbox（區別於自評） |
| I-5 | **與真實任務（Real Task）** | 把練習題延伸成可帶回工作場域的真實 deliverable | 「換成你公司／你部門」改寫題、作品集累積、結訓統合作品 |
| I-6 | **與學習資產（Continuity / Reuse）** | 學員的單元產出能跨單元被引用、改寫、累積 — 不是單頁互動，是課程級資產流動 | 跨頁 artifact 引用（CH3 引 PRAC1-1 prompt）、結訓統合 portfolio、可命名的 reusable templates |

**判準**：一個成熟的 140h 工作坊講義頁，**每個 CH/PRAC 至少出現 3 個面向**；一份完整的 Part **必須涵蓋全部 6 個面向**。

> Codex L3 補強（2026-04-27 審核採納）：原 5 面向版漏掉「學習資產跨單元復用」這個課程級維度。單頁可以做得很完整，但若學員 56 單元做完的東西彼此孤立、沒有引用鏈，就只是 56 個獨立小作業。I-6 是**作品集 schema 的設計層**，不是元件層。

## 2. 各 Part 現況盤點

### 2.1 量化掃描總表（Part 1–5，39 頁）

掃描方法：grep 各互動關鍵詞與標籤的出現次數，作為**訊號強度的代理指標**（不等於品質）。

| 訊號 | CH 頁均值 | PRAC 頁均值 | 解讀 |
|------|-----------|-------------|------|
| `<textarea>` 個數 | 0 | 1.4 | CH 完全沒有讓學員寫字的欄位（除自評 checkbox 外） |
| `<input>` 個數（含密碼+checkbox） | 11 | 24 | CH 的 input 主要是 9 個自評 checkbox + 1 個密碼 input；PRAC 是工具輸入 |
| 「討論／同學／小組／互評」字串 | 0.9 | 4.1 | CH 幾乎不觸發同儕互動；PRAC 的「討論」多是內容裡的字眼（如「會議討論」），**非教學設計層的同儕觸發** |
| 「講師／老師／助教」字串 | 0.4 | 1.7 | 全課 39 頁總計講師觸發點 < 30 處，平均每頁不到 1 處 |
| 「進階／基礎／快速路徑／挑戰題」 | 0.3 | 7.8 | CH 幾無分流；PRAC 的「進階」多為描述語，非結構性分流 |
| 主角名（阿凱/雯姊/老闆/弄一下） | 4.1 | 5.6 | Style Guide 主角貫穿一致，✅ |
| 動作動詞（複製/下載/上傳/貼上/匯出） | 8.7 | 14.2 | 動作型互動充足（多為「複製」），但**缺「上傳」與「下載作品」** |

### 2.2 共通設計模式（已確立的好習慣）

從抽樣 CH1-2 / CH5-2 / PRAC1-1 / PRAC2-2 / PRAC4-1 等代表頁面確認以下模式已穩定：

- **CH 頁 Mini Practice 結構**：標題 → 約 X 分鐘 → 4 步驟 → 範本素材（`<details class="prac-sample" open>`）→ 應產出 → 自我檢核（3 個 checkbox）
- **CH 頁三段檢核**：每章 3 個 section 末段都有「自我檢核 · CHECKLIST」共 9 個 checkbox
- **PRAC 頁互動工具**：多 textarea 輸入 + format buttons + 組合輸出 + 複製按鈕 + 範本卡片 + 進度追蹤（5 種格式打勾）
- **延伸案例段（PRAC 末段）**：每個 PRAC 附 2–3 個 30–45 分鐘的真實情境案例
- **主角一致性**：阿凱（行銷專員）、雯姊（業務主管）、老闆、弄一下行銷工作室貫穿全課
- **複製按鈕**：所有 sample prompt 都有複製到剪貼簿功能

### 2.3 各 Part 抽樣觀察

#### Part 1（解鎖 AI 工作模式）
- CH1-2 是 Mini Practice 結構的標準樣板，做得相當完整
- PRAC1-1 是高互動度工具（textarea + 組合 + 範本 + 進度），是全課程的互動性高水位
- **缺**：CH 反思題沒有 textarea；PRAC 沒有「我的作品」累積機制

#### Part 2（建立文字與知識產能）
- PRAC2-2 範本素材豐富（5 種會議逐字稿範本可一鍵載入），互動深度高
- 「討論」字眼最密集，但**全部是「會議討論」「逐字稿討論」這類內容詞**，**不是教學設計層的同儕觸發**
- **缺**：缺講師簽核點；PRAC 的延伸案例沒有 rubric 結構

#### Part 3（把資料變成流程）
- 概念抽象度上升（webhook、自動化），但 CH3-3 的「反思」次數高 (7) 卻無 textarea，**反思淪為單向陳述**
- PRAC3-1 ~ PRAC3-3 互動工具完整
- **缺**：學員「自己的工作流」沒有持久化儲存空間

#### Part 4（零代碼工具工坊）
- PRAC4-3 互動深度高（複製動作 24 次），有 contenteditable 出現的少數 PRAC
- **缺**：「工具部署到 GitHub Pages / Vercel」是 CH4-3 的學習目標，但沒有「上傳成功 → 分享連結到課堂」的觸發點

#### Part 5（前端與部署實戰）
- **重要結構性問題**：`courses/gen-ai-140h/CLAUDE.md` 已標註 `part5/` 放了 12 個「自由演練工具頁」（PRAC5-1 ~ PRAC5-12），**尚未對應 index.html**。學員從總覽頁看不到這 12 頁。
- PRAC5-5 / PRAC5-6 出現 contenteditable，是全課唯二有此元件的頁面
- CH5-2「AI 協作開發」是課程核心，但「反思」字眼出現 11 次卻無 textarea，這個落差最明顯
- **缺**：Part 5 的「個人作品頁」「互動工具頁」「API 串接展示頁」應該天然導向作品集，但目前是分散獨立頁

#### Part 6 / Part 7（待建）
- 大綱已定，未上線。**Part 6（API/後端/Agent/Streaming）天然會涉及 evals、guardrails、cost dashboard，這幾個主題本身高度互動**。
- **建議**：Part 6/7 建立前先把本報告 Phase 1 的元件做出來，新建單元直接套，不要建完才回頭補。

## 3. 互動性缺口分級（BLOCKER / ERROR / WARN）

> 分級判準：BLOCKER = 不修會讓 140h 班型的核心教學節奏無法成立；ERROR = 不修讓學習效果顯著打折；WARN = 不修也能上，但會錯過提升機會。

### BLOCKER（必修，2 條）

#### B-1 · 缺乏同儕互動觸發點
**證據**：CH 頁的「同儕／互評／互相」訊號平均 0；PRAC 頁的「討論」多為內容詞，非教學觸發。全課程沒有一處明確設計「跟同學交換」「請組員看一下」「兩兩互評」。
**班型佐證**：`_講師指引/part1.md` 第 3 行明寫「對應 140h 課程……對應 180h 實體班的帶領節奏」、Part 5 講師指引第 370 行寫「加上講解、示範、Q&A」— 此為**面授職訓班**確認，B-1 列 BLOCKER 成立。
**影響**：面授班沒有同儕互動 = 學員從早到晚 8 小時面對講義獨自填表，與在家自學 Coursera 無異。
**對應面向**：I-3

#### B-3 · Part 5 PRAC5-5 ~ PRAC5-12 未進 index.html
**證據**：`courses/gen-ai-140h/CLAUDE.md` 第 38–39 行已標註：「⚠ 目前放的是 12 個『自由演練』工具頁，尚未對應 index.html」。
**影響**：12 個練習頁學員看不到。**這是檔案存在但流量為零的死頁**，且大綱與 index 不一致 = 課程結構斷裂。
**對應面向**：I-1（內容存取本身的互動）

### ERROR（強烈建議，5 條）

#### E-1 · CH 頁「反思題」沒有留答區
**證據**：CH 頁「反思／想想／思考」字眼平均 6.7 次/頁，但全部是被動陳述（「想想看你的工作有沒有……」），**沒有 textarea**。
**影響**：反思淪為單向訊息，學員看過就忘。教學設計上的「elaborative interrogation」（自問自答）效果為零。
**對應面向**：I-1

#### E-2 · 缺「作品集」累積機制
**證據**：56 單元產出的所有作品（prompt、工作流、工具、頁面）目前散在各 PRAC 頁的 textarea，沒有集中存放、沒有 export。
**影響**：學員結訓那天，沒有東西可以**帶走**或**展示**。140h 結訓 = 應該有一份「我做過的作品集」，目前缺。
**對應面向**：I-5, I-6

#### E-3 · 缺 AI 結果回收機制（修訂）
**證據**：所有 sample prompt 都靠「複製到 ChatGPT/Claude」實踐 — 這在職訓班反而貼近真實工作流，**不是失敗模式**。問題在「貼出去之後」：學員把結果貼回頁面、跟其他版本比較、寫反思、存檔的循環沒有被設計。
**影響**：複製→跑→看→關掉，循環沒有閉環。學員產出的東西沒有被結構化收回頁面，導致 I-2 / I-5 / I-6 都流失。
**對應面向**：I-2, I-5, I-6
**Codex 修訂**：原文寫「需 embed iframe」過度，已改為「需回收機制」（textarea 貼回+多版本比較+rubric 評分+存檔）。

#### E-4 · 講師觸發點極稀少
**證據**：全 39 頁「講師／老師／助教」字串總計 < 30 次，平均每頁 < 1。沒有任何「上交給講師」「請講師檢核」的結構性元件。
**影響**：講師在課堂上沒有節奏錨點，無法檢驗學員進度，**講義無法支援講師教學流程**。
**對應面向**：I-4

#### E-8 · 自評檢核 / 工具進度不持久化（從 B-2 降級）
**證據**：CH 頁的 9 個自評 checkbox、PRAC 頁的進度追蹤（如 PRAC1-1 的 5 個格式打勾）都沒有 localStorage 儲存。重整即失。
**影響**：學員無法看到「我累積學了多少」「哪些章還沒完成」，UX 差但不阻擋課堂節奏成立。
**對應面向**：I-1, I-5
**Codex 修訂**：原列 BLOCKER 過嚴 — checkbox 重整消失是 UX 問題，不是「課堂節奏無法成立」的決定性條件。降為 ERROR。

### WARN（可選優化，5 條）

- **W-1 · 雙路徑分流不嚴格**：「進階／基礎」多為描述語（「進階技巧」「基礎做法」），無頁面層分流或測驗導引。
- **W-2 · 延伸案例缺 rubric**：PRAC 末段的「延伸案例」自評是模糊提示（「自己評估指標」），沒有結構化評分表。
- **W-3 · CH 頁缺前後連結**：每章沒有「上一節學了什麼」「下一節要準備什麼」的回顧/預告 callout。
- **W-4 · 缺「我的學習進度」總覽頁**：index.html 是課程地圖，但沒有 student-facing 的進度儀表板。
- **W-5 · PRAC 工具沒有 export**：學員做出來的 prompt/工具/工作流，沒有 .md / .json 一鍵下載。

## 4. 改善方案（HTML 元件層級規格）

每個方案都給 (a) 元件名 (b) 對應缺口 (c) HTML 結構草案 (d) 持久化策略 (e) 在哪些頁面套用。

### 4.1 InlineReflection 元件
- **對應**：E-1
- **目標**：把 CH 頁的反思題從「被動陳述」變「主動留答」
- **HTML 草案**：

  ```html
  <div class="inline-reflection" data-rk="ch1-2-r1">
    <div class="ir-eyebrow">REFLECTION · 你的答案</div>
    <div class="ir-question">想想你本週用過、結果不如預期的一個 Prompt，最弱的是哪個要素？</div>
    <textarea class="ir-input" placeholder="寫下你的觀察，60 字以內…"></textarea>
    <div class="ir-foot">
      <span class="ir-saved" hidden>已儲存到本機</span>
      <button class="ir-export">匯出我的反思 .md</button>
    </div>
  </div>
  ```

- **持久化**：localStorage key = `gen140_reflect_{rk}`，autosave on blur
- **匯出**：把全課所有 reflection 收集打包成一份 `我的反思紀錄_{date}.md`
- **套用**：18 個 CH，每章末段 1–2 個。預估 27 個 reflection 點。

### 4.2 PortfolioCollector 元件
- **對應**：E-2, B-2
- **目標**：56 單元產出集中到「我的作品集」分頁
- **HTML 草案**：

  ```html
  <!-- 在每個 PRAC 工具的最後加 -->
  <div class="artifact-save" data-ak="prac1-1-prompt-builder">
    <button class="btn-save-artifact">儲存到我的作品集</button>
    <span class="save-hint">儲存後可在「我的作品集」頁面看到</span>
  </div>
  ```

- **持久化**：localStorage key = `gen140_artifact_{ak}` + `gen140_artifact_index`
- **總覽頁**：新建 `my-portfolio.html`，列出所有已儲存作品 + 一鍵匯出 zip
- **套用**：21 個 PRAC，每頁至少 1 個 artifact key

### 4.3 PeerHandoff 元件
- **對應**：B-1
- **目標**：在 CH 末段與 PRAC 末段強制觸發同儕互動
- **HTML 草案**：

  ```html
  <div class="peer-handoff">
    <div class="ph-eyebrow">PEER · 找一位 buddy</div>
    <div class="ph-task">把你剛才寫的 v2 Prompt 給隔壁同學看 30 秒，請對方說一句「最有用的限制」。</div>
    <div class="ph-time">3 分鐘</div>
    <label class="ph-check"><input type="checkbox" data-pk="ch1-2-peer-1"><span>我已經跟同學交換過</span></label>
  </div>
  ```

- **設計準則**：每個 peer 任務有**明確時間限制（30 秒 ~ 5 分鐘）**和**單一可說出口的提示語**，不要寫「跟同學討論」這種模糊指令
- **套用**：39 頁全部，每頁至少 1 個 peer handoff，預估 50 個觸發點

### 4.4 InstructorCheck 元件（Codex 瘦身版）
- **對應**：E-4
- **目標**：每章 1–2 個結構性「上交給講師」標記點
- **HTML 草案**（移除解鎖碼，改純 checkpoint + 自勾 + 講師回饋連結）：

  ```html
  <div class="instructor-check">
    <div class="ic-eyebrow">CHECKPOINT · 講師檢核</div>
    <div class="ic-desc">把你的 v2 Prompt + 跑出來的結果截圖貼到課堂協作板。</div>
    <div class="ic-action">
      <a class="ic-link" href="#" data-board-link>前往課堂協作板（講師指引提供連結）</a>
      <textarea class="ic-feedback" placeholder="（選填）講師回饋摘要、或我自己想記下的修改方向"></textarea>
      <label class="ic-check"><input type="checkbox" data-ck="ch1-2-ic"><span>已上交給講師</span></label>
    </div>
  </div>
  ```

- **設計原則**：節奏錨點不需偽解鎖。學員自勾 + 寫一句回饋 + 連結到協作板已足。
- **套用**：18 個 CH，每章 1 個 checkpoint，PRAC 不需要

## 9. 追加審核：時數浮報問題

> 此節為使用者追加要求（2026-04-27 補加）。問題起點：「PRAC1-4 標 3h，內容無法支撐」。
> 審視範圍延伸至：(a) 時數標示一致性 (b) 內容支撐度 vs 標示時數 (c) 網頁／大綱／講師指引三方時數對齊狀況。

掃描全 39 頁的 hero-meta 區，時數標示**完全不一致**：

### 9.3 對使用者點名 PRAC1-4 的回應

**結論：使用者直覺正確，但問題不是字數，是「動手結構」。**

PRAC1-4「個人 AI 助手配置實作」標 3h，純文字角度估算 2.7h（差距 10%，**字數能撐**）。但檢視內容結構：
- **3 個 case-study 都是「讀+貼到 ChatGPT」模式**：學員讀情境、複製範本、貼到外部工具、回來看結果。這是**斷裂的實作流**，每次切分頁就斷一次，3h 內反覆切會耗散注意力。
- **只有 1 個 textarea**：頁內可以實際填寫的欄位只有 1 個，遠不及 PRAC1-1 的 4 個。學員「動手寫」的密度低。
- **缺結構化擴展引導**：3 個 case 做完後，沒有「換成你公司的版本」「跟 buddy 交換」這類延伸引導，預設學員會自我擴展撐滿 3h，但講義沒指引方向。

**比對組 PRAC1-1（標 2.5h）**：4 個 textarea + 範本載入 + 進度追蹤 — **頁內可以反覆操作的密度高**，2.5h 學員會自然填滿。

**判定**：PRAC1-4 不是「字數浮報」，是「**動手結構密度浮報**」— 讀完是夠的，但「實作 3h」的承諾兌現不了。

### 9.4 真正嚴重的浮報：Part 5 PRAC5-1~4

| 檔案 | 標示 | 實估 | 浮報 |
|------|------|------|------|
| PRAC5-1 個人作品頁 | 2.5h | ~0.7h | **72%** |
| PRAC5-2 互動工具頁 | 2.5h | ~0.7h | **72%** |
| PRAC5-3 API 串接展示頁 | 2.5h | ~0.9h | **64%** |
| PRAC5-4 完整專案部署 | 2.5h | ~0.7h | **72%** |

四頁標示總時數 10h，講義純內容只能撐 ~3h。

**辯護理由（部分成立）**：這四頁是「部署實作」，學員要實際去 GitHub / Vercel 操作，操作時間 1–1.5h 是合理 buffer。但即便加 buffer，每頁約 1.5–2h 仍撐不到 2.5h。

**核心問題**：講義指引密度極低 — 5 步驟 + 1 case + 0 範本 + 0–1 textarea。學員 30 分鐘讀完後，剩下 2 小時不知道做什麼。

### 9.5 講師指引 vs 網頁標示 vs 大綱（三方對齊）

`_講師指引/part1.md` 第 25–35 行有「180h 時數分配建議」表，列出 **140h 班**對照欄：

| 單元 | 講師指引 140h 欄 | 網頁實際標示 | 差距 |
|------|-----------------|-------------|------|
| CH1-1 | 2h | 2h | ✓ |
| CH1-2 | 2h | **3h** | +1h ✗ |
| CH1-3 | 2h | 2h | ✓ |
| CH1-4 | 2h | **3h** | +1h ✗ |
| PRAC1-1 | 2.5h | 2.5h | ✓ |
| PRAC1-2 | 2.5h | **2h** | -0.5h ✗ |
| PRAC1-3 | 3h | **2.5h** | -0.5h ✗ |

**8 個單元中 5 個不一致**。網頁總和 = 講師指引總和 = 20h（對齊大綱），但**內部分配方式不同**：
- 網頁版：CH 偏重（CH1-2, CH1-4 各 3h）
- 講師指引版：PRAC 偏重（PRAC1-3 = 3h, PRAC1-4 = 4h）

**反諷**：使用者直覺認為「PRAC1-4 標 3h 撐不到」，但**講師指引的官方標準是 4h**。網頁可能不是浮報，是**漏標 1 小時**。但無論如何，網頁與講師指引必須統一。

另一個結構性線索：Part 1 講師指引第 3 行寫「**本指引對應 140h 課程**……**對應 180h 實體班的帶領節奏**」— 也就是這份指引是 180h 班遷移而來，140h 是壓縮版。Part 5 講師指引第 370 行則寫「**組件本身約 360–450 分鐘；加上講解、示範、Q&A，實際教學時數抓 20 小時**」— **承認講義本身只能撐 6–7.5h，剩下 12.5–14h 靠講師講解吃掉**。

### 9.6 時數浮報缺口分級

#### BLOCKER（必修）
- **B-4 · 時數標示缺漏**：Part 2 PRAC1-3、Part 3 PRAC（4 頁）、Part 4 全部（7 頁）、Part 5 PRAC5-5~12（8 頁）共 20 頁無時數標示。學員無法估時程。

#### ERROR（強烈建議）

### 9.6.1 Codex 補充：4 種額外的時數浮報模式

> Codex L3 審核（2026-04-27）指出原報告漏掉的浮報模式。這四條與 §9.6 並列，需要在內容補強階段一併考量。

- **F-1 · 講師示範灌水**：頁面內容不足，但靠講師 live demo 撐時數。範例：CH3-3「webhook 基礎」實際讀完 1h，靠講師現場示範 Make.com 撐 2h。**改善**：講義須明標「此段含講師示範 X 分鐘」，並提供示範腳本給講師。
- **F-2 · 外部帳號／環境等待時間灌水**：註冊、權限、部署冷卻不應算學習時數。範例：PRAC5-2 標 2.5h 但其中 30 分鐘是等 Vercel 部署。**改善**：把「等待時間」獨立標示，不計入「學習時數」。
- **F-3 · 重複範本換皮**：多 case 但同一操作路徑，不能線性累加時數。範例：PRAC1-4 三個 case 都是「讀情境 → 套四要素 prompt → 貼到 ChatGPT」同一路徑，第二、三個 case 學員會加速。**改善**：第二個 case 之後改變動作（debug 模式 / 比較模式 / 改寫模式），不要單純換皮。
- **F-4 · 成功路徑單一**：沒有錯誤處理、debug、變體任務時，實作時數通常被高估。範例：所有 PRAC 都假設學員一次成功，沒有「prompt 跑壞了怎麼修」的支線。**改善**：對應 §4.11 DebugLoop 元件 — 每個 PRAC 加一個「壞掉版」要學員修。
