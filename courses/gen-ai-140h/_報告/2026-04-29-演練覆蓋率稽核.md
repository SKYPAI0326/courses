## 檢核 Rubric

每檔逐項評，4 項齊備=PASS，缺 ① = FAIL，缺 ②③④ 任一 = PARTIAL。

| 維度 | 標準 |
|------|------|
| ① 可複製 prompt | 至少 1 段標示貼到 ChatGPT/Gemini/Claude 的提示詞，包在 `<pre>/<code>` 或有 copy 按鈕 |

## Codex L3 第二意見（call_id: 6a45efc6）

### Codex 判讀
1. **Rubric 合理**：4 維度直擊「可貼、會做、能驗收、對齊主題」最小閉環。複製按鈕不必獨立判準。**建議加輔助項：「prompt 自包含」**（離開本頁也能跑），否則 prompt 容易只是頁內 context 延伸。
2. **判讀盲點**：63/64 PASS 偏寬鬆。建議深審 Part 3、5、6（概念/策略/流程型 prompt 易看似完整但貼出去只吐泛用摘要）。

## 修補優先序

### 🔴 P0（必修，剛性 FAIL）

**PRAC7-1.html — 專題開發 v0 衝刺**
- 缺口：JS「專題定義器」表單代替演練，無可貼 LLM prompt
- 補法（採納 Codex 建議）：保留現有 JS 工具，於工具旁補一個 ps-prompt block，內容為「ChatGPT 可貼專題釐清提示詞」：將表單輸出的 spec 餵給 LLM 做需求釐清、功能拆解、驗收標準產出
- 範本（建議寫法）：
  ```
  你是產品需求顧問。下方是我用「專題定義器」產出的初版 spec：
  【貼上 spec JSON】

  請幫我做三件事：
  1. 找出 spec 中模糊或互相矛盾的需求（列 3-5 點）
  2. 把功能拆成 MVP 必做 / 可砍 / 後續迭代三級
  3. 為 MVP 必做的每個功能寫 1 行驗收條件（給定 X，當 Y，則 Z）
  ```
- 配套：補 ps-expect（預期 LLM 回覆結構）+ checkbox 自我檢核

### 🟡 P1（建議修，PARTIAL）

**PRAC5-2.html — 靜態部署實戰**
- 缺口：主流程 0 個 LLM prompt，學員只練「照 SOP 點按鈕部署」沒練「用 LLM 除錯」
- 補法：在「部署檢查清單」與「Deliverable」之間插入 1 個 ps-prompt block，主題為「部署 troubleshoot」：
  - 角色：你是部署 troubleshooter，熟悉 Vercel/Netlify/GitHub Pages
  - 情境槽：平台名稱、repo 結構、build log error 樣本、空白頁/404 路徑問題
  - 輸出格式：原因 → 修法 → 驗證指令
- 配套：ps-expect 給 1 個診斷回覆樣本
