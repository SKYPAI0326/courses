解法參考（先寫自己的版本再展開）
原版的問題： 1. 「節省時間」太籠統——AI 不知道是節省「回覆時間」「查找時間」還是「分類整理時間」，會給你一個全功能大雜燴方案 2. 「好用、穩定、讓同事願意用」是使用者體驗原則，不是功能規格，AI 無法據此設計工具 3. 沒有說明現有流程、痛點來源，AI 只能猜想客服部門的工作內容 參考改法： 你是 AI 工具顧問（角色）。 我們的客服部門每天要處理 100–150 封客戶郵件，其中約 60% 是「詢問退換貨政策」「查訂單狀態」「回報送貨延誤」等重複性問題，目前全靠人工逐封回覆，平均回覆時間 4 小時（任務背景，具體情境）。 請設計一個「客服郵件初篩與草稿輔助工具」，目標是把重複性問題的回覆時間縮短到 30 分鐘內（任務，目標量化）。 工具需涵蓋： 1. 自動分類：把進來的郵件分類為「退換貨」「訂單查詢」「送貨問題」「其他」 2. 草稿生成：前 3 類各有 1 個客製化回覆草稿，客服人員只需確認 + 微調 3. 不可自動發送：草稿需人工確認後才能寄出 限制：不改現有郵件系統，工具以 Google Apps Script 或 Make.com 實作。 請給出：工具架構圖（文字版）、各步驟使用的技術/服務、預估每封郵件節省的時間。 關鍵改動：把「節省時間」→ 具體說明是什麼類型的重複工作佔多少比例；把「好用穩定」→ 明確的功能邊界（初篩+草稿，不自動發送）；加技術限制（Apps Script 或 Make.com）；要求輸出架構圖+時間估算（可驗證方案可行性）。

案例 01 · 社區組織
社區志工協會的「半日快閃工具日」規劃台
難度 ★★☆ 約 50 分鐘

你在社區志工協會當理事，協會 30 多位志工年齡從 40 到 70 歲，平常幫里長、弱勢家庭辦活動。大家一直說「流程好煩可不可以電腦化」但沒人會寫程式。你想辦半天快閃工具日：每人用 AI 做出一個真正能用的小工具帶回去給協會用。需要一個主持現場的線上工具。

原始資料｜活動設計需求
活動規格： - 3 小時現場，12 位志工分 4 組（3 人 / 組） - 每組現場抽一個「協會日常痛點」當題目 - 限定用 Gemini 或 ChatGPT 網頁版，不寫原生程式 - 產出：一個可打開即用的單頁 HTML 可能的痛點池（抽題用）： - 便當訂購統計（每週三集體訂便當，統計誰吃素/蛋奶素/葷） - 志工值班表自動換班（有人臨時請假要找替補） - 物資捐贈入庫清單（收到物資記錄類別數量捐贈人） - 長輩關懷電訪紀錄（志工打電話給長輩後記錄狀況） - 活動報名截止表（多場活動並行各自有人數上限） - 每月例會議程自動統整（誰要報告什麼） 現場規則： - 20 分鐘一輪，共 4 輪 - 每輪結束 5 分鐘全組上台演示給其他組看 - 最後各組選一個「最想帶回協會用」的成品投票
你的任務
  1. 寫 Prompt 產出「現場主持台」單頁 HTML：4 組組別卡片，各組抽題按鈕、計時器、當輪題目顯示區
  2. 抽題邏輯：從題目池隨機抽且同組四輪不重複，四組間同輪可重可不重
  3. 計時器 20 分鐘倒數，剩 5 分鐘變黃、剩 1 分鐘變紅，時間到全螢幕跳「請停手展示」
評估指標
抽題不重
同一組四輪不會抽到同題；題目池足夠大時確實隨機
時間警示
5 分鐘/1 分鐘/結束 三階段視覺提示明顯，老年志工也看得清楚
投影友善
投到大螢幕時字夠大（可 1080p 遠距讀取）、對比足夠
投票簡單
最後投票區有一鍵按鈕，不需輸入帳號

案例 02 · 學生社團
大學程式社「24 小時極限挑戰」主題生成器
難度 ★★★ 約 55 分鐘

你是大學程式社社長，社團每學期辦一次 24 小時程式馬拉松。最大煩惱是題目不夠多樣，常跟去年或校外比賽撞題。想做一個「題目生成器」，社員輸入幾個關鍵字（技術棧 / 社會議題 / 趣味主題）工具自動組合成新題目，並附評分標準與預期產出。

原始資料｜主題生成架構
輸入元素： - 技術棧選擇（可複選）：前端 / 後端 / AI/ML / 區塊鏈 / IoT / 資料視覺化 / 遊戲 - 社會議題選擇（可複選）：教育 / 醫療 / 環境 / 交通 / 食農 / 勞權 / 高齡化 / 無家者 - 趣味限定（可複選）：限時 / 合作互動 / 極簡 UI / Voice only / 離線可用 / 多語言 - 難度（單選）：初階 / 中階 / 進階 輸出結構： 1. 題目敘述（100 字內，一段可直接念出來的題目） 2. 情境背景（為什麼這題目值得做，使用者是誰） 3. 最低可交付（MVP 必須有什麼） 4. 加分項（做得到就加分，不強求） 5. 評分標準（4 項，各 25 分） 6. 禁止事項（避免參賽者走偏） 工具需要： - 產出後可「重新洗牌」保留關鍵字換新題目 - 儲存收藏最多 5 題，正式比賽當天再從中選 - 一鍵產出比賽用 PDF（含所有細節）
你的任務
  1. 寫 Prompt 產出「題目生成器」單頁 HTML：輸入三組關鍵字 + 難度 + 產出按鈕
  2. 輸出區顯示完整結構，下方有「換一題」「加入收藏」「匯出 PDF」三按鈕
  3. 收藏區用 localStorage，關閉瀏覽器重開仍在；超過 5 題時最舊的被擠掉
評估指標
組合有邏輯
「AI + 環境 + 離線可用」會產出像樣的題目，不是四不像
評分可衡量
四項評分標準具體可驗證，不是「創新性」這種模糊詞
難度對齊
初階題目用 HTML/CSS 即可完成，進階題目會用到 API 或資料處理
收藏持久
加到收藏後關閉再開，5 題依然在且順序正確
參考 Prompt（建議先自己寫再看）
點擊展開參考答案
【角色】你是熟悉大學程式社文化的前端工程師，了解黑客松題目要「限定夠明確但不扼殺創意」才會好玩。 【任務】根據下方主題結構，產出「24 小時黑客松題目生成器」單頁 HTML。 【輸入與輸出】 （貼上原始資料） 【限制】 - 單一 HTML 檔 - 上區三個多選 tag 群組（技術棧/社會議題/趣味限定），一個難度 radio - 「生成題目」按鈕：呼叫本地 JS 組合邏輯（題目內容在 Prompt 中先給出 10 組範例結構讓 AI 一次生成，後續用 template 混合） - 輸出卡片顯示 6 個區塊（題目/情境/MVP/加分/評分/禁止） - 「換一題」保留當前選項重新隨機組合 - 「加入收藏」把當前題目存入 localStorage 陣列，最多 5 題先進先出 - 「匯出 PDF」用 window.print + @media print 樣式（A4 直式單頁） - 收藏區浮動在側邊，點擊展開列表可刪除單項 - 配色深藍 + 米白，正經有工程感 - 禁用情境模糊詞，每個評分標準必須以可驗證動詞開頭（「可以輸出...」「可以在 X 秒內完成...」） 【輸出格式】 直接輸出完整 HTML。

正解 B。RWD 要給具體斷點+具體行為規則，AI 才知道怎麼改。A/C 太空泛，D 引入不必要的依賴還可能被禁用。
4同事說「我每週都要手動整理 8 個業務的業績 Excel」，下一步應該？
展開解析
正解 B。痛點規劃要先搞清楚「輸入/頻率/現況耗時/輸出對象」，才知道該做什麼工具、值不值得做。
520 分鐘黑客松時間到，工具還有 bug，最佳策略是？
展開解析
正解 B。限時挑戰的重點不是完美，是「可檢視的進度」。誠實標記 bug + 下一步，比裝懂有價值。
6AI 生成的工具在 Chrome 可用，在手機 Safari 按鈕點不到，最可能原因？
展開解析
正解 B。行動版常見 bug 是小螢幕重疊，把問題截圖丟回 AI 並指定「手機 375px 按鈕可點」是最快解法。
7評黑客松作品，以下哪一組指標最能抓到「實用性」？
展開解析
正解 B。實用性 = 真需求 × 可上手 × 可交付 × 可驗證效益。A/C/D 都是過程指標不是成果指標。
8一個工具要給不太會用電腦的倉管大姐用，UI 設計優先要做？
展開解析
正解 B。目標使用者不熟電腦時，「預設值 + 人話錯誤訊息 + 大按鈕」是三個最能降門檻的做法。
9痛點規劃時，應該優先做工具的痛點是？
展開解析
正解 B。工具化 ROI = 頻率 × 單次耗時 × 規則穩定度。B 三者都高。A 頻率低，C 規則不穩，D 原本就快不值得做。
10工具做好交給同事，他說「我不敢用，怕弄壞」。最有效的解法？
展開解析
正解 B。採用阻力通常是「我不知道按下去會怎樣」。可回復操作 + 資料安心說明 + 示範影片，三件事一起做就能把門檻壓下來。
綜合任務｜1 題（45 分鐘）
任務
在 45 分鐘內，把你工作中一個重複的痛點做成 AI 生成的單頁工具，並附上 3 個改進版本（v1 原始生成 / v2 UI 改進 / v3 加功能或 RWD 修正）。每個版本寫一行「這次改了什麼、為什麼改」。
驗收指標

手機可用：v3 在手機 375px 寬度下不破版

(04) · CASE · 情境演練

四個真實需求，把完整流程跑一次

這一單元的重點不是做出什麼厲害的工具，是把「訪談 → 規格 → AI 生成 → 測試 → 部署」這套流程在自己手上跑過一遍。下面 4 個情境刻意挑了身邊會遇到的角色：親戚、社團、公司同事、老師。每一題都有完整的對話素材、可直接複製的訪談 Prompt、以及最小規格範本。選一個最貼近你最近會被拜託的情境開始，做完再換另一個。

情境 A · 親戚拜託 需求模糊、使用者完全不懂技術，最容易被牽著走的那種
對話素材 · LINE 訊息原文你的起點
大表姊（45 歲，家庭主婦兼做直銷）： 「欸我聽說你會用 AI 弄網頁喔？拜託你幫我做一個記帳 App 啦。我現在都用紙本記，超麻煩。要可以記收入支出，分類要清楚，最好還可以看圖表。對了我手機不是很好別太吃記憶體。什麼時候可以好？我下週就要用欸。」 使用者畫像： - 45 歲，Android 中階機（6GB RAM），不會裝 App 只會點連結 - 每天記 3-5 筆，金額小（百元等級） - 重點想看「這個月花在小孩身上多少」、「直銷收入有沒有比去年好」 - 她覺得 Excel「很複雜，看不懂」
訪談 Prompt · 從模糊需求挖出真規格直接貼 Gemini / ChatGPT
你是一位需求訪談顧問，專門協助非技術使用者把模糊想法拆成可做的規格。 【背景】 使用者是一位 45 歲家庭主婦（兼直銷），想要一個「記帳 App」。她只會點連結、不會裝 App，每天記 3-5 筆小額開銷，想看「花在小孩身上」和「直銷收入對比」兩個視角。 【任務】 根據上方背景，幫我生成一份結構化訪談問卷，用來釐清最小規格。問題要： 1. 分三組：使用情境、必要功能、不做的事 2. 每組 3-5 題，題目要具體可回答（不要問「你希望怎樣」這種空題） 3. 每題附一個「她可能怎麼回」的預想答案，幫我預判 【輸出格式】 - Markdown 表格：組別 / 問題 / 預想答案 - 最後一段寫「問完之後，我應該可以把規格壓在哪幾項」
工具最小規格（訪談後收斂）：單頁 HTML，大按鈕輸入金額與分類（8 個固定分類寫死，不讓她自己新增）、一個月曆檢視、兩個圓餅圖（本月支出分類 / 收入 vs 支出）、localStorage 存資料、分享連結就能用、不做登入。

情境 B · 社團活動 報名表 + 費用試算，含早鳥折扣與多人優惠的組合邏輯
對話素材 · 社團幹部需求說明你的起點
社團副會長（23 歲，工學院大四）： 「我們 5 月要辦一個兩天一夜的畢業旅行，大概 40-60 人會來。想做一個報名表，填一填就能看到自己要付多少錢。規則有點複雜： - 5/1 前報是早鳥 1800，之後就 2200 - 3 人一房比 2 人一房便宜 200 - 社員再折 100，眷屬（不是社員）不折 - 素食餐點加 80，海鮮過敏不加價但要特別標註 - 活動保險是強制的每人 60 元 想讓同學們自己填，最後自動加總，然後把這筆收費明細匯出給財務同學。最好還能看到目前有幾個人報了、住房配對如何。」 時程：4/20 就要上線開放填寫。
訪談 Prompt · 把口語規則整理成條件邏輯直接貼 Gemini / ChatGPT
你是一位擅長把口語需求轉成邏輯規格的產品分析師。 【背景】 社團幹部要做畢業旅行報名表 + 費用試算，規則是： - 基礎費：5/1 前 1800，之後 2200 - 3 人房每人再折 200（2 人房不折） - 社員折 100（眷屬不折） - 素食餐加 80（海鮮過敏僅標註不加價） - 保險強制 60 - 需要：每人看到自己金額、財務匯出明細、即時報名人數與住房配對狀態 【任務】 1. 把上面規則整理成一張「條件-動作」表，欄位是：條件 / 動作（金額加減）/ 備註 2. 列出一個計算函式的偽代碼（pseudocode），顯示加總邏輯 3. 點出 3 個「規則可能互衝或講不清的地方」，請幹部回去確認再做 【輸出格式】 - Markdown 表格 + pseudocode 區塊 + 確認清單
工具最小規格：單頁 HTML + Google Sheets 當後端（Apps Script 寫入）。表單欄位：姓名 / 社員身份 / 房型偏好 / 同房指定 / 餐食 / 報名日期。即時顯示金額拆解（基礎費 / 早鳥折扣 / 房型差 / 身份折 / 餐食差 / 保險 = 總額）。幹部後台頁顯示目前人數 + 住房配對缺口。
初版 Prompt 重點：費用計算邏輯用「條件陣列」寫，每條規則單獨列可讀；日期切點用伺服器端判斷（避免本機時區問題）；送出後一律顯示金額明細給使用者，不只給總數。
情境 C · 公司內部工具 5-10 人每週用一次，不值得上大系統，但每次都煩
對話素材 · 行政同事抱怨你的起點
行政同事（31 歲，辦公室行政）： 「我們部門 8 個人，每週五中午會一起叫外送。每次都我在收訂單、算錢、轉帳，真的超累。現在流程是： 1. LINE 群貼這週的菜單連結（通常 3-4 家輪流） 2. 大家各自回『我要 XXX 加珍珠不要冰』 3. 我一個一個抄到 Excel 4. 算每個人要付多少（外送費平均分） 5. 貼出總表讓大家轉帳給我 6. 我下單 最花時間的是第 3 和第 4 步。想要一個工具讓每個人自己點，最後自動算帳。但我們不太想用 Surveycake 或 Notion，因為同事不會註冊帳號。」
訪談 Prompt · 找出「值不值得做」的門檻直接貼 Gemini / ChatGPT
你是一位內部工具 ROI 顧問。 【背景】 部門 8 人，每週一次叫外送。目前由行政一人收單、抄表、算錢。她想做一個自助點單工具。 【任務】 1. 幫我估算現況：每週耗時多少、一年總工時、換算時薪 200 的成本 2. 列出「工具做出來後能省下哪幾步、還剩哪幾步」的對照表 3. 給出一個「最小可行工具」的功能清單，剔除所有一年用不到 3 次的功能 4. 點出 2 個「如果只用 Google Forms + Sheets 公式也能做，就不必寫工具」的情境 【輸出格式】 - ROI 試算（條列） - Before/After 對照表 - MVP 功能清單（≤ 5 項） - 用現成工具就夠的情境（2 點）
工具最小規格：單頁 HTML + 同一個連結每週沿用，菜單 JSON 每週更新一次（行政貼上 JSON 即可）。8 個人點選後按「送出」寫入共用 Sheet（用 Apps Script），工具即時顯示「誰點了什麼 / 各自金額 / 總額 / 外送費分攤」。不做登入，用姓名下拉選（8 個寫死）。
初版 Prompt 重點：功能只做 1 件事（點餐 + 算錢），不加歷史紀錄、不加評價、不加訂金。菜單支援「基礎品項 + 客製化備註」兩欄就好。每週結束自動封存當週資料，行政只要換菜單 JSON。
情境 D · 老師需求 輸入章節就自動出 5 題選擇題，還要能列印成考卷
對話素材 · 國中老師需求你的起點
國中理化老師（38 歲，任教 12 年）： 「每週小考我都要出 5 題選擇題，課本一章要出三次（前測、隨堂、複習）。最花時間的不是寫題目，是把題幹和四個選項排整齊、答案寫在另一頁、然後印成 A4 兩面。 想要一個工具：我貼上這章課本的重點摘要（我自己整理的 Word 檔，約 800 字），工具幫我出 5 題選擇題，包括： - 題幹 - 四個選項（一個正解 + 三個合理的錯誤選項） - 答案與簡短解析 - 可以直接列印成 A4 考卷格式（正面題目，背面答案） 難度我要能調（基礎 / 中等 / 挑戰），出完我會再人工微調 1-2 題再用。」
訪談 Prompt · 把「AI 出題品質」變成可驗收規格直接貼 Gemini / ChatGPT
你是一位教育測驗設計顧問，擅長把老師的口語要求轉成 AI 可驗收的出題規格。 【背景】 國中理化老師要做「章節生成 5 題選擇題」出題器，輸入是她自己整理的重點摘要（約 800 字），輸出要能列印成 A4 正反面考卷。她能接受人工微調 1-2 題。 【任務】 1. 列出「一題好選擇題」的 6 項驗收標準（題幹清楚、選項等長、錯誤選項合理、不能互相排除、唯一正解、難度一致） 2. 針對難度分級（基礎 / 中等 / 挑戰），各寫一段「生成時要遵守的規則」 3. 列出 3 個「AI 常見的出題爛招」，在 Prompt 裡明文禁止 4. 給出可貼進工具的「出題 Prompt 骨架」，含變數 {章節摘要} 和 {難度} 【輸出格式】 - 驗收標準清單 - 難度分級規則（3 段） - 禁用清單 - Prompt 骨架（可直接複製）
工具最小規格：單頁 HTML。上半部：貼上摘要 textarea + 難度 radio + 題數預設 5 題（可改 3/5/10）+ 生成按鈕。下半部：題目列表可編輯（點擊直接改題幹或選項），右上角「列印」按鈕套 print-only CSS 正面題目、背面答案與解析。
初版 Prompt 重點：出題 Prompt 裡明列禁用「以下何者正確」這種無效題幹、禁用「以上皆是 / 以上皆非」當選項、禁止四個選項長度差太多；生成後每題附「出題理由」一句話讓老師判斷要不要留；列印樣式用 @media print 寫死 A4 尺寸與字級。
(05) · EXTEND · 延伸挑戰 + Quiz

做完 demo 只是起點，真的上線才算走完

上面 4 個情境你只要挑一個做到「Gemini 幫你生出第一版」其實不難，難的是後面三件事：敢不敢真的讓人用、能不能收到回饋再改一次、有沒有留下一份別人也看得懂的使用說明。這三件事才是這個單元真正要你練的。做完再用下面 5 題檢核自己是不是把重點抓對。

🌐
挑戰 01 · 真的上線，不只 demo。把你做出來的工具推到 GitHub Pages（純靜態單頁最快）或 Vercel（需要後端或環境變數時選這個）。拿到一個可以貼在 LINE 的公開網址，傳給情境裡那個親戚 / 社團 / 同事 / 老師。沒有公開網址，就等於沒做。
🔁
挑戰 02 · 收集回饋，迭代一次。工具交出去後 3-7 天，主動問使用者三個問題：哪裡卡住了、最常用哪個功能、最想要加什麼。挑一個最多人提的問題，改掉它，推 v2 上線。沒有 v2 的工具只是練習；有 v2 才叫做產品思維。
📘
挑戰 03 · 寫 README 教別人用。在同一份工具的公開網址旁邊放一份 README（可以直接是 GitHub Pages 的 index 下方或一份 Google Doc），內容只有三件事：3 張操作截圖、3 步驟文字說明、3 個常見問題 + 答案。寫 README 的過程會逼你發現工具哪裡還不夠直覺。
1做需求訪談時，以下哪一項最容易漏問、卻決定工具值不值得做？
展開解析
正解 C。使用情境與頻率決定工具的 ROI 與介面強度——一週一次的和每天十次的做法完全不同。A 是表面美觀、B 是加分資訊、D 在單頁工具裡通常不是主要變數。情境與頻率沒問清楚，後面的規格都是空轉。
2第一版規格該做到多詳細？
展開解析
正解 B。MVP 的重點是「砍」不是「加」。第一版塞越多功能，AI 做出來越容易亂、使用者看到越害怕、你自己改起來越痛苦。保留核心、砍掉一半以上，留空間給真實回饋引導 v2。
3AI 生出來的工具第一次打開不能跑，最該先查哪裡？
展開解析
正解 A。Console 是所有 JS 錯誤第一手訊息，紅字通常直接指出哪一行哪個變數有問題，複製錯誤訊息貼回 AI 修，幾乎都能一次修好。B/C/D 在沒看錯誤訊息前都是碰運氣，等於在黑箱裡亂試。
4工具做好要部署給使用者點開就用，首選方案是？
展開解析
正解 D。GitHub Pages 免費、永久公開網址、只要 push 就更新，是純靜態工具的首選。需要 API key 保護、函式執行、環境變數時才升級到 Vercel。A/B/C 都會在使用者更換裝置或離開你身邊時失效，等於沒部署。
5給非技術使用者的 README / 使用說明該包含什麼？
展開解析
正解 B。非技術使用者看到程式碼會關掉、看到 30 分鐘影片會拖進收藏列從此不看。3 截圖 + 3 步驟 + 3 QA 是門檻最低、最能被完整讀完的格式。A 是給工程師的、C 等於沒寫、D 太長會被略過。
