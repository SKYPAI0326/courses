用 UI / UX 要求讓工具變好用

3 小時 · Part 4 · 零代碼工具工坊

「能用」和「好用」之間的差距，不是靠運氣，而是靠提問方式。這節課教你用語言描述設計意圖，讓 AI 幫你做出從排版、視覺到互動體驗都真正好用的工具。

學習成果
能辨識工具的常見 UX 問題並用正確語言要求 AI 修正
能描述視覺設計意圖（色彩、排版、卡片樣式）而不需要知道 CSS 語法
能在不同情境（行動裝置/桌面）下驗收工具的使用體驗
RECAP · 上一節帶走的
AI 能從需求描述生出可執行的 HTML 工具頁面——不需要你從零寫程式，只需要把需求說清楚。
PREVIEW · 這節解決什麼
「能用」和「好用」差的不是運氣，是提問方式。這節教你描述設計意圖，讓 AI 生出從排版到互動體驗都真正好用的工具。
(01) · Usability Gap

從「能用」到「好用」：五個最常見的差距

AI 第一稿幾乎都是「能用」的，但是否「好用」要靠你的驗收眼光和要求能力。

1
欄位沒有說明：使用者不知道要填什麼
如何要求 AI 修正：「在每個輸入框下方加一行灰色說明文字，例如『請填入稅前月薪，如：32000』」
2
按鈕不明確：不知道按下去會發生什麼
如何要求 AI 修正：「按鈕文字改成描述動作的動詞，如『計算加班費』而非『確認』」
3
輸出沒有層次：結果都是同樣大小
如何要求 AI 修正：「最重要的數字用大字（1.5rem），次要說明用小字（0.85rem），用不同顏色區分」
4
錯誤訊息不清楚：「發生錯誤」沒有指引
如何要求 AI 修正：「錯誤訊息改成具體可操作的說明，如『月薪不能為 0，請重新輸入』，顯示在輸入框正下方」
5
空狀態沒處理：未輸入時輸出區空白
如何要求 AI 修正：「在用戶尚未按下計算前，輸出區顯示淡灰色提示文字：『計算結果將顯示在這裡』」
迷你練習 1 約 10 分鐘
幫你 CH4-1 做出來的工具做一次 UX 稽核

打開你在 CH4-1 練習 3 生成的工具，用本節的「五個差距」清單逐項檢查，把發現的問題寫成給 AI 的修正指令。

1對照五個 UX 差距，逐項在紙上打勾：欄位說明／按鈕動詞／輸出層次／錯誤訊息／空狀態。
2選「差距最明顯」的 2 個問題，每個寫一條修正指令（只說要的效果，不寫 CSS）。
3把兩條指令合併送給 AI 跑一次修正，打開新檔案比對。

## 結果卡片層次
- 主要數字：每月成本（TWD）1,152 元 — 大字（約 1.45rem）、深色加粗、置頂
- 次要並列：每月成本（USD）36.00 ｜ 每月觸發 900 次 — 較小字（約 .85rem）、灰色、同一列
- 主次之間用細分隔線或約 20px 留白區隔，不再是三個數字同樣大小擠成一團

## 錯誤訊息具體化
- 每日次數空：欄位正下方「請先填寫每日觸發次數（例如 30）」
- 每日次數 -5 或 20000：「請輸入 0 到 10000 之間的整數」
- 匯率填 100：「匯率不在合理範圍，請確認（例如 32）」
- 所有錯誤訊息紅字 #b5703a，顯示於對應欄位正下方，不使用 alert

## 其他保持不動
主色霧藍、版型、驗證邏輯、試算公式、清除按鈕，一切與上一版完全一致

對照重點
判斷學員版本是否：(1) 主次層次靠「字級+顏色」兩個維度區分，而不是只改字級，因為顏色才是掃視時的第一優先；(2) 錯誤訊息每條都含具體範例值（如「例如 30」「例如 32」），讓使用者直接知道怎麼填，不是只說「格式錯誤」；(3) 負值與超上限走同一條訊息「0 到 10000 之間的整數」，避免訊息碎裂；(4) 學員若順便改了主色或版型，表示沒讀到題目「只改這兩項」的限制。

(02) · Visual Language

說設計意圖，不說 CSS：給 AI 可執行的視覺描述

你不需要知道 CSS 語法就能描述你想要的視覺效果。

CSS 技術語言（看不懂）
border-radius: 8px;
box-shadow: 0 2px 12px rgba(0,0,0,.08);
font-size: .85rem;
color: #8a8278;
line-height: 1.7;
background: linear-gradient(135deg, ...);
設計意圖語言（你能說）
「卡片要有圓角，加上輕微的陰影讓它看起來像浮在頁面上」
「說明文字比正文小一點，顏色淡一點（灰色），行距寬鬆一點」
「背景使用純色，不要漸層，要簡潔」
◆
溝通設計的三個關鍵字：大小（比正文大/小）、顏色（強調/輔助/警告）、間距（緊湊/寬鬆）。掌握這三個維度你就能引導 AI 做出你要的視覺。
(03) · Layout and Card Style

版型、卡片、色彩：讓工具像個作品而不是作業

常見的版型模式、卡片設計和色彩使用方式。

一欄式版型
輸入在上，輸出在下，最適合行動裝置優先的工具。
告訴 AI：「版型採單欄，輸入區在上，結果區在下，中間有分隔線」
左右分欄
輸入在左（40%），輸出在右（60%），桌面體驗更好。
告訴 AI：「桌面版左右分欄，手機版上下排列（RWD）」
卡片式設計
用白色卡片 + 淺灰背景，讓工具有層次感。
告訴 AI：「整個工具包在一個有白色背景的卡片裡，卡片外是米色背景」
主色與強調色
一個工具最多用 2 個顏色，強調色用在按鈕和重要數字。
告訴 AI：「主色用深藍（#1e2d40），按鈕和重要數字用橘色（#e8753c）」
◎
視覺最簡單的改進：讓工具的背景和卡片有顏色差距。全白的頁面看起來空洞，加個淡灰（#f5f3ee）背景就立刻有層次感了。
迷你練習 2 約 10 分鐘
用「設計意圖語言」一次升級工具的視覺

不碰 CSS，只用大小／顏色／間距三個關鍵字，幫工具寫一段視覺升級指令，讓它看起來像作品不是作業。

1選一種版型（單欄 / 左右分欄 / 卡片式）並用一句話描述為什麼選它。
2指定主色 1 個 + 強調色 1 個（最多 2 色），說明強調色用在哪（按鈕？重要數字？）。
3用「大小／顏色／間距」三個維度描述卡片、說明文字、按鈕各要什麼感覺。

- 單欄版型置中，最大寬度 780px；上半輸入區、下半結果區，中間一條淡色分隔線

- 霧藍（主）：用在主按鈕底色、結果卡片標題底線、重要數字底下的細線
- 琥珀黃（強調）：只出現在「每月成本（TWD）1,152」那組主要數字，其他地方找不到

## 大小／間距
- 主要數字比正文大很多（一眼就看到）
- 欄位說明文字比正文小一級、灰色、行距較寬
- 主按鈕全寬、上下內距較厚、霧藍底白字
- 卡片內部上下留白寬鬆有呼吸感，不擠

## RWD

對照重點
判斷學員版本是否：(1) 琥珀黃只出現在 TWD 主數字，若在按鈕或標題也用到就違反「強調色唯一」；(2) 輸入卡用純白而非霧藍淡色，避免與主按鈕搶視覺；(3) 主按鈕必須全寬且厚度明顯高於次要按鈕，否則會看不出主次；(4) 學員版本若回覆中出現 CSS 屬性名（flex、padding、border-radius），表示 AI 沒聽懂「用意圖語言」的要求，該退回再提醒一次。

(04) · Intent not Implementation

告訴 AI 你的設計意圖，不是告訴它怎麼寫程式

最常見的錯誤是試圖告訴 AI 技術實作細節（你其實不確定對不對），而不是告訴它你想達成的效果。

1
錯誤做法 vs 正確做法
錯：「把 div 的 display 改成 flex，然後 justify-content 設成 space-between」（你不確定這對不對）
對：「讓標題和右側的數字在同一行，一個靠左一個靠右」
2
錯誤做法 vs 正確做法
錯：「加一個 setTimeout 延遲 300ms 後顯示結果」（你其實只是想要動畫）
對：「結果出現時加一個淡入動畫，讓它不那麼突然」
3
錯誤做法 vs 正確做法
錯：「在 @media 576px 的 breakpoint 改版型」（你不確定這個數字）
對：「在手機上（螢幕寬度小於 600px 時）改成單欄版型，輸入在上輸出在下」
ℹ
記住：你是建築師，AI 是施工方。建築師說「我要一個落地窗讓光線進來」，不是「你先挖 3 公分深的溝再放 C 型鋼架⋯⋯」
迷你練習 3 約 8 分鐘
把「技術指令」翻譯成「設計意圖」

下列三個指令都寫了技術細節，請逐一改寫成只描述意圖的版本；然後手機開一次你的工具，看哪條最該補。

1改寫題 A：「把 container 的 display 改成 grid，template-columns 設 1fr 1fr」。
2改寫題 B：「加一個 @keyframes fadeIn，animation-duration 400ms」。
3用手機（或把瀏覽器拉窄到 <600px）開一次你的工具，記一條最需要加的 RWD 修正（用意圖語言描述）。

## 整合後的意圖指令（示範）
這是對上一版的追加調整，以下三項請一起修正，其他保持不動：
- 桌面版（螢幕寬 ≥768px）的輸入欄位改為左右並列，每欄各佔一半寬；手機寬度下自動回到上下堆疊。
- 結果卡片出現時改為溫和淡入，大約半秒完成，不要瞬間跳出。
- 手機（螢幕寬 <600px）下：結果卡片三個數字改為上下排列各一行；主按鈕與結果區之間多一段呼吸留白；欄位標籤字級比目前放大約一級，戶外光線下也讀得清。

## 檢查點
通篇文字不出現 display、grid、template-columns、@keyframes、animation-duration、flex、@media、breakpoint 等 CSS 技術字
「螢幕寬 <600px」「大約半秒」「放大約一級」等描述使用**使用者感受到的結果**而非實作細節

對照重點
判斷學員改寫版本是否：(1) 三條合成一段整合指令而非三條各自獨立，因為 AI 收到整段比三次追加更穩；(2) 用「螢幕寬 <600px」描述斷點，而非「breakpoint」或「@media」；(3) 用「溫和淡入約半秒」描述動畫，而非「animation-duration 500ms」；(4) RWD 那條基於學員實測的具體觀察（三數字擠在一行、按鈕太擠、標籤太小），不是複製題目原文——這是檢驗學員有沒有真的拿手機開過一次工具的關鍵。
成果產出
三條改寫後的意圖指令（A / B / RWD），每條 20–40 字，不含 CSS 屬性名。
自我檢核
SOLO · 自我演練
把工具截圖丟進新開的 LLM 對話，請它「扮一位第一次看到這頁的陌生使用者」挑出三個「第一眼會搞錯」的元素，並對照你的 UX 稽核表標出你漏掉哪幾項，把回饋當成下一輪修工具的依據。
3 分鐘
進階選配 · 不計入課程時數（給想加練的學員）
把 LLM 對話截圖存到學習資料夾，或貼到工作群組／社群求 review。這些痕跡在轉職投履歷時可作為作品集素材。
CHECKPOINT · 講師檢核
把你的工具 v1 → v2 對照（說出 3 個 UI 改進點）+ LLM 扮陌生客戶試用得到的回饋整理貼到協作板。
