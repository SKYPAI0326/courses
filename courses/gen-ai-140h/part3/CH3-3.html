表單、自動化工具與 Webhook 基礎

自動化不是黑魔法，而是把你每次手動做的步驟告訴一個工具，讓它幫你重複執行。這節課拆解 Make 和 n8n 的核心邏輯，以及讓流程自動觸發的關鍵概念：Webhook。

PREVIEW · 這節解決什麼
自動化不是黑魔法，就是把手動步驟告訴工具讓它重複執行。這節拆解 Make / n8n 核心邏輯與讓流程自動觸發的 Webhook 概念。
(01) · Form Data

Google 表單的資料結構：自動化的起點

Google 表單是自動化流程最常見的觸發起點——免費、熟悉、不需要寫程式。理解表單怎麼存資料，才能知道自動化工具怎麼接它。

結構 01
表單 → 試算表
每次提交在 Google Sheets 新增一列，欄位名稱就是表單的題目。這是自動化工具讀取資料的標準入口。
結構 02
時間戳記
每筆紀錄都有自動生成的提交時間，是觸發自動化的判斷依據——「新資料出現」就從這裡判斷。
結構 03
欄位格式
短答案 → 文字；選擇題 → 固定選項值（注意 AI 清理時的一致性）；評分 → 數字。格式決定後續處理方式。
結構 04
觸發時機
「收到新提交」是最常用的觸發條件；也可以用「試算表有新列」觸發，彈性更高。
◆
選擇自動化入口的原則：用你最常用、最熟悉的格式當輸入。Google Sheets 是絕大多數零代碼自動化的最佳起點。
(02) · Make and n8n

Make / n8n 的流程邏輯：三步驟就能理解

Make 和 n8n 這類工具的核心邏輯非常一致——無論介面怎麼設計，所有流程都可以用同一個框架理解。掌握這個框架，任何自動化工具都能快速上手。

觸發 · TRIGGER
什麼事件讓流程開始？
每條流程都從一個觸發事件開始，沒有觸發就不會執行。
Google 表單新提交 / 定時排程 / Webhook 收到資料 / Email 收到信
處理 · ACTION
要做什麼動作？
觸發後執行的一個或多個動作，可以串接、可以加條件判斷。
呼叫 AI API / 讀取試算表 / 發送 Email / 格式轉換 / 條件判斷
輸出 · OUTPUT
結果放哪？
處理完成後，把結果送到目的地或觸發下一個動作。
寫回試算表 / 發送 Slack / 寄 Email / 觸發另一條流程
Make.com 特色
適合初學者的視覺化工具
視覺化模組連接，直覺易懂
適合無程式基礎的初學者
有豐富的 app 連接器（1000+）
付費版功能較完整
n8n 特色
開源彈性高的自架工具
開源可自架，完全掌控資料
程式碼節點彈性高
可免費自架，無限執行次數
社群模板豐富
✦
初學者建議先用 Make.com，因為視覺化拖拉比較直觀。有 IT 基礎或想自架的人考慮 n8n。兩者的流程邏輯幾乎一樣，學會一個就很容易轉換。

(03) · Webhook

Webhook：讓外部事件直接通知你的流程

Webhook 是自動化工具裡最重要、也最容易讓人困惑的概念。先從一個類比開始理解它：

Webhook 就像快遞的「到貨簡訊」。你不需要一直打電話問「快遞到了嗎」——而是告訴快遞公司一個手機號碼（Webhook URL），包裹一到就自動傳訊息給你。Webhook URL 就是你的手機號碼，外部事件發生就是包裹到達。

概念 01
Webhook URL
Make / n8n 自動生成的唯一網址，是「接收資料」的入口。複製這個網址，貼到觸發來源。
概念 02
誰來發送
任何能發送 HTTP 請求的系統——表單、網站、另一個自動化工具、你自己的程式都可以。
概念 03
資料格式
通常是 JSON，包含事件的詳細資訊（觸發時間、資料內容等）。Make / n8n 會自動解析。
概念 04
觸發即執行
Webhook 收到資料的瞬間，流程就啟動，不需要排程。即時性是 Webhook 最大的優點。
ℹ
在 Make / n8n 設定的第一步，就是複製 Webhook URL 貼到觸發來源（如 Google 表單的回覆通知、付款系統等）。之後這條路就會自動通了。
迷你練習 2 約 15 分鐘
建立你的第一個 Webhook 並送出測試資料

實際在 Make 或 n8n 建立一個 Webhook 觸發節點，用瀏覽器或 curl 送一筆測試資料，親眼看到資料進流程的那一刻，就不再怕 Webhook。

1在 Make 建立新 Scenario（或 n8n 新 workflow），第一個節點選 Webhook / Custom Webhook，複製生成的 URL。
2按下「Run once / Listen for test event」讓它等待接收。
3把 Webhook URL 後面加 ?name=test&msg=hello 貼到瀏覽器網址列按 Enter（或用 curl 發 POST）。
4回到 Make / n8n，確認節點顯示收到資料，截圖保存欄位對應結構。
範本素材 第一次碰 Webhook 不知道要送什麼資料？用下方某客服表單的 payload 範例送到你的 Webhook URL，感受 JSON 欄位怎麼對應到 Make / n8n 的動態欄位。
原始素材 · 客服表單提交 payload（約 220 字）送進你的 Webhook URL
## 測試用 JSON payload（POST body，Content-Type: application/json） { "submission_id": "FM-20260420-0031", "submitted_at": "2026-04-20T14:32:11+08:00", "customer": { "name": "某客戶", "email": "customer@example.com", "phone": "0912345678" }, "issue": { "category": "product_defect", "priority": "high", "content": "上週購買的產品使用三天後出現異常，希望安排更換或退款。" }, "meta": { "source": "web_form", "user_agent": "Mozilla/5.0" } } ## 或用 GET 方式測試（瀏覽器直接貼） 在你的 Webhook URL 後加：?submission_id=FM-20260420-0031&name=某客戶&priority=high&category=product_defect
範例 Prompt · Webhook 測試與欄位對應 checklist 生成器可直接貼 ChatGPT / Claude
你是一位自動化工程師。我要在 Make / n8n 建立一個 Webhook 接收下方 payload。請幫我產出「測試與欄位對應 checklist」，讓我能一步步驗證 Webhook 設定正確。 ## 輸出要求 1. 給兩種測試方式（GET 網址列 / POST curl 指令），curl 請給可直接貼終端機的完整指令（URL 用 [YOUR_WEBHOOK_URL] 占位） 2. 列出 Make / n8n 接收後會看到的欄位路徑（dot notation，例如 customer.name） 3. 給 5 條驗證 checklist（打勾式），每條要具體可執行，不要寫「確認正常」這類模糊敘述 4. 最後補一行「下一步」：建議接哪個節點處理這份 payload ## 規則 - GET 測試的 query string 需 URL-encode 中文 - curl 指令需包含 Content-Type header - 欄位路徑請含 meta 與 customer 的巢狀結構 ## payload 如下： [貼上上方 JSON]
預期輸出片段（供對照）
curl -X POST [YOUR_WEBHOOK_URL] -H "Content-Type: application/json" -d '{"submission_id":"FM-20260420-0031",...}'
欄位路徑：submission_id / customer.name / customer.email / issue.priority / issue.content / meta.source
Checklist：□ Webhook 顯示「Accepted」狀態碼 200 □ customer.name 欄位有值（非 undefined）□ issue.priority 值為 high/medium/low 其中之一 □ submitted_at 能解析為有效 ISO 8601 □ meta.source 等於 "web_form"
下一步：接 Router 節點依 issue.priority 分流，high 走即時通知，其他走排程處理
成果產出
一個可用的 Webhook URL + 一張收到測試資料的截圖 + 能說出資料欄位結構（name / msg 在 JSON 的哪個位置）。

(EXT) · Webhook Drill

兩個情境演練：把 Webhook 從「好像懂」變成「接得起來」

Webhook 不是讀懂概念就會用，要看過一條完整鏈才知道每段該填什麼。這節用兩條真實世界常見的鏈——Typeform 報名通知、Google Form 建 Notion 卡——把 payload 結構、錯誤碼、retry 邏輯一次攤給你看。

情境 A Typeform 收到線上報名 → Webhook 進 Make → 推一條訊息到 Slack 的 #signups 頻道。
情境說明 · Typeform → Webhook → Slack任務背景
## 任務背景 你辦一場週末實體工作坊，用 Typeform 收報名。過去每次有人報名都要自己登後台看，常常晚看半天。目標：每有新報名，3 秒內推一則 Slack 訊息到 #signups 頻道，讓行政組立刻處理候補與付款提醒。 ## 輸入資料 - Typeform 表單欄位：姓名、Email、電話、是否曾上過課、預計到場人數、備註 - Typeform Webhook 設定頁（Connect → Webhooks → Add webhook） - 你在 Make / n8n 建立的 Webhook URL - Slack #signups 頻道 Incoming Webhook URL ## 期望產出 Slack 訊息格式固定：一行標題「🆕 新報名：某姓名（n 人）」+ 四行欄位摘要 + 一顆「👀 我來處理」按鈕（Slack block kit）。錯誤時推到 #ops-alerts 頻道。
流程步驟 · Typeform → Webhook → Slack 完整鏈可貼進 Make / n8n 當配置備忘
## 第 1 步：在 Make 建立 Custom Webhook 觸發 - 新 Scenario → Webhook → Custom Webhook → Add → 取名 typeform-signup - 複製產生的 URL，格式類似 https://hook.eu2.make.com/abc123xyz - Data structure：留空（第一次跑會自動推斷） ## 第 2 步：Typeform 端綁定 Webhook - Typeform 後台 → Connect → Webhooks → Add a webhook - Endpoint：貼上第 1 步的 URL - 點 Test → Typeform 會送一筆 mock payload 過來 - Save and activate ## 第 3 步：在 Make 內查看 payload 結構 預期 payload（POST body, Content-Type: application/json）： { "event_id": "01FYZ...", "event_type": "form_response", "form_response": { "form_id": "abcd1234", "submitted_at": "2026-04-20T09:12:00Z", "answers": [ {"field": {"ref": "name"}, "type": "text", "text": "某學員"}, {"field": {"ref": "email"}, "type": "email", "email": "x@example.com"}, {"field": {"ref": "headcount"}, "type": "number", "number": 2} ] } } ## 第 4 步：加 Slack 發訊節點 - Action：Slack → Create a Message - Channel：#signups - Text：🆕 新報名：{{form_response.answers[0].text}}（{{form_response.answers[2].number}} 人） - Blocks：用 block kit 貼一顆 button，value 帶 event_id ## 第 5 步：加錯誤路由（Router → Error handler） - 右鍵 Slack 節點 → Add error handler → Resume / Rollback - 錯誤路徑接第二個 Slack 節點 → 發到 #ops-alerts，文字包含 error.message 與 event_id ## 第 6 步：啟用排程 + 測試 - Scheduling 設 On（Immediately as data arrives） - 實際送一筆 Typeform 報名，確認 #signups 與 payload 時間差 < 5 秒
預期輸出格式（Slack 訊息供對照）
🆕 新報名：某學員（2 人）
Email · x@example.com
電話 · 0912-xxx-xxx
是否曾上過課 · 是
備註 · 需要素食便當
[👀 我來處理] 按鈕（value=01FYZ...）
情境 B Google Form 送出 → AI 判斷需求類型 → 自動在 Notion 對應 Database 建卡 + 指派負責人。
情境說明 · Google Form → AI 分派 → Notion 建卡任務背景
## 任務背景 你是行銷組長，對外開一張 Google Form 當「需求收件箱」——合作提案、採訪邀約、客訴、一般詢問都從這進。過去每天要手動分類再丟到 Notion 不同 Database，每天耗 40 分。目標：送出 → 30 秒內 AI 判斷類別 → 自動進 Notion 正確資料庫 + 指派對應負責人。 ## 輸入資料 - Google Form 欄位：姓名、單位、聯絡方式、需求類型（下拉選）、內容描述 - Notion 4 個 Database：合作提案 / 媒體採訪 / 客訴處理 / 一般詢問，各有不同欄位結構 - 負責人對照表：合作→Amy、採訪→Ben、客訴→Cindy、一般→Dan ## 期望產出 送出表單 30 秒內：Notion 對應 Database 多一張卡、Title 為「YYYYMMDD 單位名稱 需求摘要」、指派欄位自動填入負責人、Status=New。若 AI 判斷信心度 < 80% 則建到「待分類」資料庫並 Slack 通知組長手動分。
流程步驟 · Google Form → AI → Notion 完整鏈可貼進 Make / n8n 當配置備忘
## 第 1 步：Google Form 綁試算表，Make 監聽新列 - Form 回應連結到 Google Sheet「需求收件箱」分頁 - Make 建 Scenario → Google Sheets → Watch New Rows → 選該 Sheet - 排程：Every 1 min（Google Sheet 不直接發 Webhook，用輪詢） ## 第 2 步：OpenAI 分類節點 - Action：OpenAI → Create a Completion（Chat） - Temperature：0.1（分類要穩定） - Prompt： 你是需求分流員。請把下方訊息分到 [合作提案/媒體採訪/客訴處理/一般詢問] 四類其一， 並給信心度 0–100。禁止回覆其他類別。 輸出 JSON：{"category":"...","confidence":85,"summary":"12 字內摘要"} 訊息：{{內容描述}} - Parse JSON：在下一步用 JSON → Parse JSON ## 第 3 步：Router 分支 - 分支 1：confidence >= 80 → 走對應 Notion Database - 分支 2：confidence < 80 → 走「待分類」Database + Slack 通知 ## 第 4 步：Notion Create a Database Item - 4 個分支各接一個 Notion 節點，資料庫 ID 寫死 - Title：{{now | formatDate "YYYYMMDD"}} {{單位}} {{AI 摘要}} - Assignee：照對照表（Amy/Ben/Cindy/Dan 的 Notion user ID） - Status：New ## 第 5 步：錯誤處理與 retry - Notion 節點右鍵 → Add error handler → 類型選 Break - Break 設定 Retry 3 次、每次間隔 60 秒（Notion 偶爾 429 rate limit） - 3 次都失敗 → Slack 通知 + 把原始列標「需人工」寫回 Sheet ## 第 6 步：401 錯誤排查 - 若 Notion 回 401：Integration 沒被邀進 Database（Notion Database → Share → 邀請你的 Integration） - 若 OpenAI 回 401：API key 過期或 Make 連接器要重新授權 - 每次 401 都在錯誤處理節點記到 Sheet「錯誤紀錄」分頁，週末盤點

延伸挑戰 · 架一條真實 Webhook 鏈
1. 挑你工作中一個「收到訊息就要手動做事」的情境（新報名、新客訴、新需求），用今天學的 Webhook 概念架出第一條真實鏈，24 小時內送出第一筆測試資料。
2. 故意把 Webhook URL 改錯一個字元、把 API key 拿掉、把欄位格式塞錯資料，看看會跳什麼錯誤碼（401 / 404 / 500），把錯誤處理節點加進去。
3. 壓力測試：一次送 20 筆測試資料進 Webhook，看 Make / n8n 有沒有掉單、retry 有沒有生效、錯誤 log 有沒有留下來。
1Webhook 最核心的定位是？
正解 C。Webhook 本質是「事件發生 → 主動 POST 一包資料到某 URL」，對照組是「輪詢（polling）—— 我每分鐘問你一次有沒有新東西」。輪詢費流量又慢，Webhook 省資源又即時。
2GET 與 POST 在 Webhook 測試時的差別？
正解 B。GET 瀏覽器網址列就能測（?name=xx&msg=yy），但中文要 URL-encode、長度有限。POST 能塞 JSON 結構、欄位巢狀、資料量大，是所有真實 SaaS Webhook 的標準做法。
3收到 Webhook 的 JSON payload 後，第一件事應該做什麼？
正解 A。Webhook 到的資料往往是巢狀結構（customer.name、issue.priority、meta.source），Make / n8n 要看懂路徑才能在下游動態引用。不先看結構就拖元件，填錯欄位後流程跑一半掉資料，debug 會非常痛苦。
4Webhook 回傳 401 時，最可能的原因與處理？
正解 D。401 Unauthorized 就是「我不認識你 / 你沒權限」。常見原因：API key 輪替後沒更新、Notion Integration 沒被加到該 Database、OAuth token 過期。對照：404 是網址錯（找不到路徑）、500 是伺服器自己炸。
5下游節點偶爾失敗，retry 機制的關鍵設計原則？
正解 B。無限 retry 在對方 429（rate limit）時只會把自己封更久；不 retry 等於一小波網路抖動就掉單。實戰模板是「有限次數 + 指數退避 + 失敗通知 + 原始資料留底」，每個元素少一個都會在凌晨炸出問題。
(04) · Debugging

流程失敗時：三種常見原因與除錯方式

自動化流程第一次不成功很正常。好消息是失敗的模式通常就那幾種，學會識別就能快速修復。不要一看到失敗就全部重來——先定位問題在哪一段。

1
觸發沒有收到
Webhook URL 貼錯、表單設定連結錯誤 → 在 Make / n8n 用「手動觸發」或「Send Test」確認是否收到資料。先確認資料有進來，再看後面的步驟。
2
AI 節點失敗
API 金鑰錯誤、請求格式不對、回應解析失敗 → 先在 API 測試工具確認呼叫成功，再回來接流程。不要在流程裡 debug API，太費時。
3
輸出寫入失敗
試算表欄位名稱不一致、權限未開放 → 確認試算表已授權工具存取，欄位對應用「動態欄位」而非手動輸入名稱。
✦
除錯最重要的心態：一次只改一個地方，確認後再改下一個。同時改多個地方不知道是哪個修好了，下次遇到同樣問題還是不會。
SOLO · 自我演練
請 LLM 扮演「值班半夜接到告警電話的維運工程師」，針對「觸發 → AI 處理 → 通知」三節點刁鑽追問「這一步壞了我從哪裡知道？我要怎麼救回來？AI 回傳格式不對誰要負責？」把 AI 點出的最脆弱環節與補救方式補進設計。
