STEPS · 任務步驟

依序完成這五個步驟

1
建立 Google 表單
設計一個情境表單（如「課後回饋」或「需求申請」），連結到 Google Sheets。確保 Sheets 有一個「回應」工作表，作為後續觸發的資料來源。
2
在 Make/n8n 建立流程
新增「Google Sheets → 觸發」節點，設定監聽新列。Make 使用「Watch Rows」，n8n 使用「Google Sheets Trigger」。
3
加入 AI 摘要節點
呼叫 Claude 或 GPT，輸入表單回答，輸出一段結構化摘要。System Prompt 要明確說明摘要格式（例如：主題、關鍵點、建議行動）。
4
設定輸出目的地
將摘要寫回 Sheets 的新欄位，同時發 Email 或 Slack 通知。兩個輸出可以並行，用「Router」或「Branch」節點分叉。
5
執行測試
實際提交表單一次，確認整條鏈路跑通。在 Make/n8n 的執行紀錄中確認每個節點都成功，沒有錯誤訊息。

解法參考（先寫自己的版本再展開）
原版的問題： 1. 沒有檢查表單內容是否有效——若必填欄位空白，AI 仍會嘗試生成摘要，產出無意義結果 2. OpenAI API 呼叫沒有超時設定與錯誤處理——API 掛掉時整條流程靜默失敗，負責人不知道發生了什麼事 3. Line Notify 發送失敗沒有備援——摘要已生成但通知沒到，負責人毫不知情 參考改法： 觸發：Google 表單收到新填寫 ↓ 前置檢查：確認「姓名」「問題描述」兩個必填欄位不為空 → 若為空：Line Notify 發送「⚠ 收到空白表單，請手動確認：[表單連結]」，終止流程 ↓ 步驟 1：送給 OpenAI 生成摘要（逾時設 20 秒，重試 1 次） → 若仍失敗：Line Notify 發送「⚠ AI 摘要生成失敗，原始內容：[填寫人姓名] 於 [時間] 填寫」，跳到步驟 2b ↓ 步驟 2a（成功路徑）：Line Notify 傳送 AI 摘要給負責人 → 若 Line Notify 失敗：改寄 Email 給負責人（含摘要全文） 步驟 2b（失敗路徑）：Line Notify 傳送原始表單摘要（非 AI）給負責人 ↓ 步驟 3（所有路徑）：把執行結果（成功/失敗原因）寫入 Google Sheets 紀錄 關鍵改動：前置檢查（避免空白輸入跑完整流程）；API 失敗加重試+通知（讓負責人至少知道發生了什麼）；通知失敗加 Email 備援；所有路徑最後都寫執行紀錄（下次除錯有據可查）。

案例 01 · 活動企劃

難度 ★★★ 約 50 分鐘

你承辦一場 80 人付費讀書會，報名頁用 Google Form。過去三次活動報名後都要人工回信、加 Google Calendar、Line 通知主辦群組，累計花 5 小時。這次要在 Make 或 n8n 做成自動鏈，從填表到三方通知全自動。

原始資料｜Google Form Webhook payload
{ "event_id": "evt_2026_spring_bookclub", "form_id": "form_signup_042", "submitted_at": "2026-04-18T14:32:10+08:00", "answers": { "name": "林小華", "email": "xiaohua.lin@example.com", "phone": "0912-345-678", "ticket_type": "早鳥單人票", "price_ntd": 880, "diet": "素食", "first_time": true, "referrer": "朋友介紹", "notes": "我是老師 能否提早 20 分入場拍照" }, "payment_status": "pending" }
你的任務
  1. 在 Make/n8n 設計流程：Webhook 接收 → AI 產生個人化確認信 → 寄 Gmail → 建立 Google Calendar 事件（含報名者資訊）→ Line Notify 主辦群組
  2. 設計 AI 產生確認信的 Prompt，根據 ticket_type、diet、first_time、notes 調整語氣與內容
  3. 支付未完成（payment_status = pending）要多寄一封溫和的付款提醒，24 小時後自動發送

個人化
素食、首次、備註這些欄位反映在信件內容，不是罐頭信
通知訊號清
主辦 Line 通知包含票種、餐點需求、特殊備註，一眼看完
容錯

案例 02 · 補教業
家長 LINE 訊息 → AI 分類 → Notion 工單
難度 ★★★ 約 45 分鐘

你是國中補習班行政，家長 LINE 官方帳號每天進 40–60 則訊息，從請假到問學費、問成績、抱怨老師。以前用人工一則則回，漏掉一則就會收到客訴。這次要讓 AI 分類後丟進 Notion Database，明確該誰處理、截止日是什麼。

原始資料｜LINE Webhook 範例
{ "source": {"type":"user","userId":"U3a4b...c92","displayName":"張小華媽媽","bind_student":"張小偉_國二B班"}, "message": {"type":"text","text":"老師 小偉今天發燒 38 度請假一天 作業可以晚交嗎","timestamp":"2026-04-18T08:12:00+08:00"} } { "source": {"type":"user","userId":"U8b9c...f01","displayName":"李爸爸","bind_student":"李小美_國三A班"}, "message": {"type":"text","text":"下次繳費是不是這週三 我要線上刷卡還是 ATM 現在是不是有分期","timestamp":"2026-04-18T09:34:00+08:00"} } { "source": {"type":"user","userId":"U1d2e...a55","displayName":"陳媽媽","bind_student":"陳小偉_國一C班"}, "message": {"type":"text","text":"那個王老師上課一直滑手機 我女兒說根本沒在教 這個月要解約","timestamp":"2026-04-18T10:01:00+08:00"} }

案例 03 · 電商小團隊
訂單異常 → AI 診斷 → Email + 內部通知
難度 ★★☆ 約 40 分鐘

你是七人電商團隊的營運，接手後發現每天有 10–15 張訂單卡在異常狀態（付款成功但出貨延遲、地址不全、超商取貨碼失效）。過去客服要人工看 Shopify 後台判斷，現在想讓系統每 30 分鐘掃一次、AI 判斷原因、自動寄信 + 丟 Slack。

原始資料｜異常訂單 JSON
[ {"order_id":"SO-25018","customer":"林小明","status":"paid_awaiting_ship","last_update_hours":52, "ship_method":"宅配","address":"台北市大安區信義路三段","item_count":2,"sku":["SKU-A12","SKU-B08"], "warehouse_note":"SKU-B08 補貨中"}, {"order_id":"SO-25023","customer":"陳小美","status":"paid_address_invalid","last_update_hours":18, "ship_method":"宅配","address":"新北市林口區文化三路","item_count":1, "warehouse_note":"地址無門牌號碼"}, {"order_id":"SO-25031","customer":"張大偉","status":"711_code_expired","last_update_hours":72, "ship_method":"7-11 取貨","address":"-","item_count":3, "warehouse_note":"三次通知未取"} ]
你的任務
  1. AI 判斷異常根因（缺貨/地址錯/客人未取），給客服建議的行動腳本
  2. 根據嚴重度分 P1/P2/P3：P1 立刻打電話；P2 發簡訊+Email；P3 只發 Email

(A) · Automation Drills

情境演練：三條真實觸發鏈

工具產生器只是把規格書寫出來，真實自動化要面對的是：Webhook payload 欄位缺、AI 節點偶發噴錯、通知頻道一多就開始亂炸。下面三個情境提供觸發事件、輸入資料、期望通知三段，附上 n8n 流程節點列表讓你對著做。做完一輪約 60 分鐘，把規則帶回公司手上的表單試跑一次才是真的學會。

情境 A · 新客戶聯絡表單 Webhook 接入 → AI 判斷意圖（詢價／合作／客訴／其他）→ 分派到對應 Slack 頻道。
觸發事件｜官網「聯絡我們」送出後打 Webhook
POST /webhook/contact-form Content-Type: application/json { "submitted_at": "2026-04-20T10:14:22+08:00", "name": "王大明", "company": "大明國際股份有限公司", "email": "daming.wang@example.com", "phone": "02-2345-6789", "message": "你好 看到你們網站上的企業顧問方案 想約一個 30 分鐘的線上簡報 我們公司 80 人規模 主要想解決業務流程自動化 下週三四下午比較方便" }
n8n 流程步驟
1. Webhook (trigger) └─ Path: /contact-form └─ Response: immediately (204) 2. Set (normalize payload) └─ 抽出 name / company / message 欄位 └─ 補 submitted_at 若缺 3. OpenAI / Anthropic (classify intent) └─ System: 你是客服意圖分類助手 └─ 分類：詢價｜合作提案｜客訴｜徵才｜其他 └─ 輸出 JSON: {"intent":"...", "urgency":"high|mid|low", "summary":"30字內"} 4. Switch (route by intent) ├─ 詢價 → Slack #sales ├─ 合作提案 → Slack #biz-dev ├─ 客訴 → Slack #customer-success (tag @主管) ├─ 徵才 → Slack #hr └─ 其他 → Slack #inbox-triage 5. Slack (send message) └─ Template: {{intent}} | {{company}} / {{name}} {{summary}} ↪ 回信按鈕 (deep link to Gmail compose) 6. Error Trigger (fallback) └─ 若 AI 節點失敗 → 直接丟 #inbox-triage 附原始 payload
期望通知（Slack #sales 收到）
[詢價｜urgency: mid] 大明國際 / 王大明
80 人企業詢問顧問方案 想約下週三四下午線上簡報
欄位檢核：intent 正確（不是分到「其他」）／company 帶出來／urgency 不是亂標 high 炸業務／summary 壓在 30 字內。
常見漏點：phone 沒規範格式（有人填 09xx-xxx-xxx 有人填 +886）要在 Set 節點先 normalize；message 是空字串時要 skip 不要送 AI 浪費 token。

延伸挑戰（挑 1–2 項做，約 30 分鐘）
  • 🎯 接你公司真實表單：把上面任一條鏈的第一步 Webhook／Sheets 觸發，改成接你公司實際在用的聯絡表單、請假單或客服管道，跑 5 筆真實資料，看哪個欄位 AI 摘不乾淨、哪個分派規則漏掉真實情境。改規則、再跑一次，直到通知訊息值得你下週一早上看到不覺得是雜訊。
  • 🎯 加重試與錯誤通報：AI 節點包 Retry（最多 3 次，exponential backoff），第 3 次還失敗就寫進一個 failure log Sheets，每天早上 9 點發一次摘要給你自己（「昨天 5 筆成功 1 筆失敗，失敗原因：AI 回非 JSON」）。沒有錯誤通報的自動化，等你哪天發現時已經漏了兩週訊息。
  • 🎯 監控儀表板：把每條鏈的執行紀錄（觸發時間、AI 分類結果、通知送到哪、耗時）寫進一個 Sheets 當 log，用 Looker Studio 或 Google Sheets 內建圖表做成每日看板：今日觸發數／平均處理秒數／AI 分類分布／失敗率。看三個禮拜的趨勢，你會知道哪條鏈值得繼續投資、哪條該砍掉。
1設計一條「表單 → AI → 通知」自動化時，首要考慮的是？
自動化的第一性問題是「當它壞了，你會不會知道」。成功路徑誰都會寫，真正決定這條鏈能不能上線的是失敗路徑——AI 節點噴錯、Webhook 斷線、下游通知服務掛掉，每一個都要預先有 fallback。價格、UI、端到端串聯都是細節；沒有錯誤路徑的自動化等於埋了一顆不知道何時爆的雷。
2AI 節點在生產環境偶發噴錯（回傳非 JSON、timeout、API 限流）時，最穩的 fallback 是？
AI 會壞是常態不是例外。正確的 fallback 公式是「告警 + 保全原始資料」：第一時間讓值班人知道這筆需要人工看，同時把原始 payload 存進一個 failure log，不要因為 AI 壞掉就把整個觸發事件丟掉。無限 retry 會在限流時變成遞迴炸彈；靜默跳過會產生資料黑洞（你永遠不知道漏了什麼）；切換 provider 是進階作法但要先有監控才知道何時該切，不是第一線的 fallback。
3通知回傳的速度該設多快？
通知速度是業務決策不是技術決策。客訴慢 10 分鐘可能流失客戶，請假單慢 10 分鐘完全沒影響，內部報表一天出一次就夠。通通即時會讓主管 Slack 被非急件洗版，久了就不看；統一批次會讓真急件被淹沒。正確做法是每條鏈自己設 SLA（例：P1 客訴 1 分鐘內、請假 5 分鐘內、日報 隔日 9:00），並在 Switch 節點按優先級分流通知管道。
4表單資料含個資（email、電話、姓名、身分證），丟進 AI 摘要前最該做的是？
個資最小化是資料隱私的第一原則，不管你用哪家 AI provider。AI 真正需要的是「理解文義」，不是「知道這人姓名電話」——脫敏後 AI 摘要品質幾乎不變，風險卻降一個量級。正確做法是在 Set 節點先跑一個 regex／欄位對照：電話 09XX-XXX-XXX、email xxx@domain、身分證 X123***789，並把原始欄位留在自家 Sheets／DB，只把去識別化後的內容送 AI。NDA 是法律保護不是技術保護；改欄位名完全沒用；免費 API 通常隱私保護更差。
5流程跑不通時，除錯首要工具是？
執行紀錄是自動化除錯的黃金標準。n8n 每次執行會留下每個節點的輸入 JSON、輸出 JSON、耗時、錯誤訊息——打開一條紀錄從上往下看，第一個輸出不符合你預期的節點就是問題起點。重建流程是把已知 bug 換成未知 bug；問 AI 它不知道你的 context；關掉流程是放棄而不是除錯。真正進階的團隊還會把 execution log 寫進 Sheets／DB 做統計，找出「平均每週失敗率 3% 的節點」主動優化，而不是等客戶抱怨才發現。
