STEPS · 任務步驟

依序完成這五個步驟

1
建立 Google 表單
設計一個情境表單（如「課後回饋」或「需求申請」），連結到 Google Sheets。確保 Sheets 有一個「回應」工作表，作為後續觸發的資料來源。
2
在 Make/n8n 建立流程
新增「Google Sheets → 觸發」節點，設定監聽新列。Make 使用「Watch Rows」，n8n 使用「Google Sheets Trigger」。
3
加入 AI 摘要節點
呼叫 Claude 或 GPT，輸入表單回答，輸出一段結構化摘要。System Prompt 要明確說明摘要格式（例如：主題、關鍵點、建議行動）。
4
設定輸出目的地
將摘要寫回 Sheets 的新欄位，同時發 Email 或 Slack 通知。兩個輸出可以並行，用「Router」或「Branch」節點分叉。
5
執行測試
實際提交表單一次，確認整條鏈路跑通。在 Make/n8n 的執行紀錄中確認每個節點都成功，沒有錯誤訊息。

延伸案例

第一條自動化鏈跑通後，用三個真實場景設計完整的觸發→處理→通知流程。先畫流程圖、再實作 Webhook payload、最後驗證通知內容可讀性。每個案例 40–50 分鐘。

案例 01 · 活動企劃
報名表 → 確認信 → 加行事曆 → 通知主辦
難度 ★★★ 約 50 分鐘

你承辦一場 80 人付費讀書會，報名頁用 Google Form。過去三次活動報名後都要人工回信、加 Google Calendar、Line 通知主辦群組，累計花 5 小時。這次要在 Make 或 n8n 做成自動鏈，從填表到三方通知全自動。

原始資料｜Google Form Webhook payload
{ "event_id": "evt_2026_spring_bookclub", "form_id": "form_signup_042", "submitted_at": "2026-04-18T14:32:10+08:00", "answers": { "name": "林小華", "email": "xiaohua.lin@example.com", "phone": "0912-345-678", "ticket_type": "早鳥單人票", "price_ntd": 880, "diet": "素食", "first_time": true, "referrer": "朋友介紹", "notes": "我是老師 能否提早 20 分入場拍照" }, "payment_status": "pending" }
你的任務
  1. 在 Make/n8n 設計流程：Webhook 接收 → AI 產生個人化確認信 → 寄 Gmail → 建立 Google Calendar 事件（含報名者資訊）→ Line Notify 主辦群組
  2. 設計 AI 產生確認信的 Prompt，根據 ticket_type、diet、first_time、notes 調整語氣與內容
  3. 支付未完成（payment_status = pending）要多寄一封溫和的付款提醒，24 小時後自動發送
評估指標
流程完整
四個動作都跑到（信、日曆、Line、付款提醒排程）
個人化
素食、首次、備註這些欄位反映在信件內容，不是罐頭信
通知訊號清
主辦 Line 通知包含票種、餐點需求、特殊備註，一眼看完
容錯
欄位空值不會炸鏈，有設 fallback（例如 diet 沒填就跳過素食段）
參考 Prompt（AI 產生確認信）
點擊展開參考答案
【角色】你是讀書會主辦人的寫作助理，語氣親切、不套公式。 【任務】根據下方報名資料，產生一封繁體中文的個人化報名確認信。 【輸入】 - 姓名：{{name}} - 票種：{{ticket_type}}（金額 NT${{price_ntd}}） - 飲食需求：{{diet}} - 是否首次參加：{{first_time}} - 備註：{{notes}} - 付款狀態：{{payment_status}} 【限制】 - 主旨 15 字內，帶報名者姓名 - 正文 150–220 字，包含：感謝報名、活動時間地點確認、特殊需求處理說明、付款資訊（若 pending）、下一步 - 首次參加者多加一段「當天流程說明」 - 有備註的要明確回應（不是「已收到備註」了事） - 不用「親愛的」「敬祝安康」這類套話 【輸出格式】 主旨：... 正文：...
變體挑戰
A加一個「人工審核」關卡：VIP 票種或提出特殊要求的，改寄給主辦審核不直接發信
B活動前 48 小時自動再寄一封提醒信，內容根據天氣 API 決定要不要加「記得帶傘」
C取消報名 webhook 串 Stripe 退款流程，自動寄退款確認並從 Calendar 移除

案例 02 · 補教業
家長 LINE 訊息 → AI 分類 → Notion 工單
難度 ★★★ 約 45 分鐘

你是國中補習班行政，家長 LINE 官方帳號每天進 40–60 則訊息，從請假到問學費、問成績、抱怨老師。以前用人工一則則回，漏掉一則就會收到客訴。這次要讓 AI 分類後丟進 Notion Database，明確該誰處理、截止日是什麼。

原始資料｜LINE Webhook 範例
{ "source": {"type":"user","userId":"U3a4b...c92","displayName":"張小華媽媽","bind_student":"張小偉_國二B班"}, "message": {"type":"text","text":"老師 小偉今天發燒 38 度請假一天 作業可以晚交嗎","timestamp":"2026-04-18T08:12:00+08:00"} } { "source": {"type":"user","userId":"U8b9c...f01","displayName":"李爸爸","bind_student":"李小美_國三A班"}, "message": {"type":"text","text":"下次繳費是不是這週三 我要線上刷卡還是 ATM 現在是不是有分期","timestamp":"2026-04-18T09:34:00+08:00"} } { "source": {"type":"user","userId":"U1d2e...a55","displayName":"陳媽媽","bind_student":"陳小偉_國一C班"}, "message": {"type":"text","text":"那個王老師上課一直滑手機 我女兒說根本沒在教 這個月要解約","timestamp":"2026-04-18T10:01:00+08:00"} }
你的任務
  1. AI 分類訊息到「請假 / 繳費問題 / 課程詢問 / 抱怨申訴 / 其他」五類
  2. 建 Notion 工單：標題 = 學生名 + 訊息摘要；責任人 = 班導/財務/主任（依分類自動指派）
  3. 抱怨申訴類額外 Slack 通知主任，24 小時內要回覆家長（AI 擬一份初步回覆草稿）
評估指標
分類正確
繳費問題不會被誤歸成課程詢問
指派合理
請假 → 班導；繳費 → 財務；申訴 → 主任，不會全部丟給櫃台
資料完整
工單包含學生班級、家長稱謂、原訊息全文、建議回覆
升級機制
申訴類真的有觸發 Slack 告警，不是只進工單
參考 Prompt（分類 + 工單生成）
點擊展開參考答案
【角色】你是國中補習班行政主任，熟悉家長溝通分流。 【任務】讀取 LINE 訊息，輸出 JSON 給 Notion API 建工單。 【分類規則】 - 請假：出現發燒、請假、不能來、生病等 - 繳費問題：費用、刷卡、分期、發票、ATM - 課程詢問：時間、教室、教材、成績、考試 - 抱怨申訴：投訴老師、要求退費、要解約、服務問題 - 其他：以上都不是 【指派邏輯】 - 請假 → 該生班導 - 繳費問題 → 財務「林姐」 - 課程詢問 → 教務組 - 抱怨申訴 → 主任 + Slack 告警 - 其他 → 櫃台 【輸出 JSON】 { "category": "...", "owner": "...", "title": "{學生名}_{類別}_{20字內摘要}", "priority": "高|中|低", "due_hours": 24 | 48 | 72, "parent_name": "...", "student": "...", "original_message": "原訊息全文", "draft_reply": "給責任人參考的回覆草稿，不超過 80 字" } 【限制】 - priority：申訴→高、請假/繳費→中、其他→低 - draft_reply 要有學生名、稱謂、具體下一步，不能空泛
變體挑戰
A加情緒分析：訊息帶怒氣 / 威脅退費的 priority 自動升級為高
B每週五自動跑一次工單統計報表，Slack 發給主任看本週類別分布
C把 draft_reply 串回 LINE，責任人點「已確認」按鈕後才真的送出

案例 03 · 電商小團隊
訂單異常 → AI 診斷 → Email + 內部通知
難度 ★★☆ 約 40 分鐘

你是七人電商團隊的營運，接手後發現每天有 10–15 張訂單卡在異常狀態（付款成功但出貨延遲、地址不全、超商取貨碼失效）。過去客服要人工看 Shopify 後台判斷，現在想讓系統每 30 分鐘掃一次、AI 判斷原因、自動寄信 + 丟 Slack。

原始資料｜異常訂單 JSON
[ {"order_id":"SO-25018","customer":"林小明","status":"paid_awaiting_ship","last_update_hours":52, "ship_method":"宅配","address":"台北市大安區信義路三段","item_count":2,"sku":["SKU-A12","SKU-B08"], "warehouse_note":"SKU-B08 補貨中"}, {"order_id":"SO-25023","customer":"陳小美","status":"paid_address_invalid","last_update_hours":18, "ship_method":"宅配","address":"新北市林口區文化三路","item_count":1, "warehouse_note":"地址無門牌號碼"}, {"order_id":"SO-25031","customer":"張大偉","status":"711_code_expired","last_update_hours":72, "ship_method":"7-11 取貨","address":"-","item_count":3, "warehouse_note":"三次通知未取"} ]
你的任務
  1. AI 判斷異常根因（缺貨/地址錯/客人未取），給客服建議的行動腳本
  2. 根據嚴重度分 P1/P2/P3：P1 立刻打電話；P2 發簡訊+Email；P3 只發 Email
  3. 內部 Slack 頻道每天早上 9:00 摘要昨天所有異常（類別、筆數、未解決的優先順序）
評估指標
根因判斷
status + warehouse_note 一起看，不只看 status
行動腳本
客服拿到就能直接用，不用再自己判斷下一步
分級正確
超過 48 小時 + 缺貨 = P1；地址錯 18 小時 = P2
摘要可讀
早會 Slack 訊息一眼看出優先、不需要點進 Dashboard
參考 Prompt（異常診斷）
點擊展開參考答案
【角色】你是電商營運分析師，熟悉台灣物流狀態碼與客服處理流程。 【任務】對每張異常訂單輸出診斷、分級、行動腳本。 【規則】 - 根因判斷：綜合 status、warehouse_note、last_update_hours 三個欄位 - 分級 P1：last_update_hours ≥ 48 且非客戶端問題（缺貨、系統卡單） - 分級 P2：客戶端需介入（地址錯、資料不全）且 < 48 小時 - 分級 P3：客戶自身未動作（未取貨、未回覆），但仍在規則內 【輸出 JSON】 { "order_id": "...", "root_cause": "一句話說明，最多 25 字", "priority": "P1|P2|P3", "action_for_cs": "客服下一步，動詞開頭，最多 40 字", "customer_email_draft": "寄給客人的郵件內容，含主旨與正文，繁體中文，不套公式", "internal_note": "給倉庫/主管看的內部備註，最多 30 字" } 【限制】 - 缺貨類的 customer_email_draft 要提供兩個選項（等待 or 改其他 SKU），不要只道歉 - 地址錯的 email 要直接請客人回覆正確地址，附上原址讓他對
變體挑戰
A加一層「重複異常」偵測：同一客人一個月內 2 次以上，升級 P1 並給行銷做關懷
BAI 多產一份「倉管回饋」：哪些 SKU 最常缺貨，建議採購優先補哪些
C把 Slack 早安摘要改成 Notion 每日日報，含前一天處理完成率折線圖
