個人化 Kanban 任務板

最輕量的個人任務管理。新增卡片、在三個欄位間移動、關閉瀏覽器再打開資料還在——沒有多餘功能，只有你需要的。

PRACTICE GOALS
學會用 localStorage 做完整的資料持久化
練習動態新增/刪除 DOM 元素的互動設計
產出一個可替代便利貼的數位看板工具
TOOL · 互動工具

新增任務、移動卡片、關掉瀏覽器資料還在

在任何欄位的底部輸入任務內容，按「新增」後卡片出現在該欄位。卡片下方有左移、右移按鈕可在三個欄位間流動，右上角的 ✕ 可刪除。所有資料存入 localStorage，關掉瀏覽器再開也不會消失。

你是一位擅長製作生產力工具的前端工程師。 請製作一個「個人 Kanban 任務板」的單頁 HTML 工具，功能如下： 【功能需求】 - 三個並排欄位：待辦 / 進行中 / 完成 - 每個欄位底部有文字輸入框和「新增」按鈕，可新增任務卡片 - 每張卡片有兩個移動按鈕（← 向左移 / → 向右移）和一個刪除按鈕 - 卡片頂部顯示欄位任務數量計數器 - 所有資料存入 localStorage，重新整理瀏覽器後仍然保留 【設計要求】 - 背景 #f5f3ee，三欄使用淺灰背景（#edeae3），卡片白色 - 待辦計數標籤用霧藍色，進行中用芥末黃，完成用鼠尾草綠 - 字體 Google Fonts（Shippori Mincho + Noto Sans TC） - 圓角 6px，無陰影漸層，輸入框聚焦時邊框變橘色 請生成完整的單一 HTML 檔案。

延伸案例

Kanban 工具把任務分為幾欄拖來拖去很爽，但真實自由工作者常見問題是「四個案子同時跑」。重點不是欄位設計得多漂亮，而是讓自己一眼看出「今天該做什麼不該做什麼」。

案例 12 · 自由工作者
同時跑 4 個案子，早上打開電腦不知道該做哪個
難度 ★★☆ 約 40 分鐘

你是自由接案工程師，四月手上同時四個案子：一家品牌官網改版（大案、進度落後）、一個 Chrome 擴充功能（小案、新加）、本課程作業（自我成長類）、朋友的健身工作室 WordPress 維護（小額月費、每月固定 2 小時）。每天早上打開電腦第一件事是發呆 30 分鐘才動手。你想用 Kanban 板解決這個問題。

Part 5 · 通關檢測
前端與部署實戰 — 關卡驗收

做完 Part 5 的 4 章 + 12 個自由演練，用這份檢測確認你真的能獨立從本機寫到上線。先自己作答，再展開解析對照。最後完成綜合任務，把 HTML / DevTools / Git / 部署串成一條完整動作。估時 40 分鐘。

觀念題｜10 題
1讀懂一份陌生 HTML 檔，最有效率的切入順序是？
展開解析
正解 B。先 meta 建立情境，再抓 body 主要 section／class 命名的骨架，最後才看 script 互動。從「結構→互動」而非逐行。
2想改一個按鈕的顏色，但不確定 class 名稱，在 DevTools 最快的作法是？
展開解析
正解 B。Inspect 箭頭（Cmd+Shift+C）是 DevTools 第一招，直接鎖定元素＋顯示所有命中樣式的優先順序。
3CSS 樣式「被覆蓋」時，Styles 面板會顯示什麼線索？
展開解析
正解 B。刪除線代表「這條屬性被後面更高優先序的選擇器壓掉」，排除 CSS 問題的第一個視覺線索。
4要把程式碼交給 AI 修一個按鈕樣式，最精準的指令做法是？
展開解析
正解 B。這是 CH5-2 的核心三要素——位置（selector）＋意圖（視覺目標）＋限制（不能動的範圍），避免 AI 亂重構。
5本機要做「從 0 到 GitHub Pages 公開」，正確順序是？
展開解析
正解 B。標準動線：本機先 init／commit，再跟遠端 repo 連線 push，最後到 Pages 設定啟用。
6不小心把錯誤版本 push 到 main，最安全的「回到上一版」做法是？
展開解析
正解 C。revert 保留歷史只追加反向 commit，是唯一不傷協作者工作樹的回滾法。force push 會覆寫他人歷史。
7靜態頁面上線後，「改了檔但網站沒變」最常見的原因？
展開解析
正解 B。先檢查 commit 有沒有推上遠端，再用無痕視窗／強制重整（Cmd+Shift+R）排除快取。
8為什麼純前端頁面要呼叫需要金鑰的 API（例如 OpenAI）時，需要一層 Node.js 代理？
展開解析
正解 B。前端原始碼是公開的，任何金鑰寫進去都會被翻出來盜用／帳單爆掉。代理層的作用是把金鑰藏在伺服器環境變數。
9DevTools Console 看到紅字 Uncaught ReferenceError: foo is not defined，最可能的原因？
展開解析
正解 B。ReferenceError 幾乎都是拼錯、或 script 載入順序不對導致變數還沒宣告就被呼叫。
10Git 提交訊息寫「update」「fix bug」是常見壞習慣，好的 commit message 應該？
展開解析
正解 B。commit message 是給三個月後的自己看的線索，動詞開頭＋一句話說清楚動機，勝過十行散文。
綜合任務｜1 題（20 分鐘）
任務
挑一個自己做好的單頁工具，完成一次從本機 git init 到 GitHub Pages 公開網址的完整推送，並刻意製造一次「改壞」→ 用 git revert 回到上一版，附上回滾紀錄（commit 列表 + 網址變化截圖）。
驗收指標
公開可訪問：拿到 https://<user>.github.io/<repo>/ 的網址，他人也打得開
commit 可讀：至少 3 個 commit，訊息說清楚做了什麼
回滾留痕：用 revert 而非 force push，歷史保留「錯→正」兩筆紀錄
DevTools 驗證：用無痕視窗確認線上版與本機一致，不是快取假象
