照著做主線 · STEP 1 / 5 · 場景設定（5 分鐘）

建宏的麻煩

建宏，41 歲，補習班行政組長。他在前面的課程做了一個「家長問答小幫手」——家長輸入問題，工具呼叫 AI 回覆課程與報名相關說明。問題是：呼叫 AI 的 API 金鑰直接寫在前端的 JavaScript 裡。他想把工具掛到補習班官網給家長用，但只要任何人按右鍵開 DevTools，幾秒就能看到金鑰，拿去亂用、帳單算補習班的。他需要的，是把「呼叫 AI」這件事搬到一個藏在伺服器端的後端代理，金鑰只放在後端、永遠不進前端，再把整個應用部署上線。

今天你就是建宏。你要拿一個「前端 + 後端代理」的小專案，照著下面的指令走一次，把它完整部署到雲端，拿到公開網址，而且確認金鑰不外洩。

動手前先準備好三樣東西：（1）一個免費 GitHub 帳號——到 github.com 註冊，部署平台要靠它讀程式碼。（2）一個免費 Railway 帳號——到 railway.app 用 GitHub 帳號登入即可。（3）一組 AI API 金鑰（OpenAI 或 Anthropic 都行）——這是要被「藏起來」的那個東西，先準備好、先別貼進任何檔案。

照著做主線 · STEP 3 / 5 · 照著做步驟清單（10 分鐘）

逐步操作指令（直接照打，不用改）

下面分三段：第一段在本機把專案準備好（含後端代理與 .gitignore），第二段推上 GitHub，第三段在 Railway 設定金鑰並部署。指令都可以直接照打——只有 ~/Desktop/qa-tool（換成你專案的實際路徑）和 你的帳號（換成你的 GitHub 帳號）兩處要換成你自己的。

JAVASCRIPT 第一段之一 · 後端代理 server.js（放進專案資料夾） 
// server.js — 一個只做「代理」的最小後端：
// 前端打 /api/ask，後端拿環境變數裡的金鑰去呼叫 AI，再把結果回傳。
const express = require("express");
const app = express();
app.use(express.json());
app.use(express.static("public"));  // 前端檔案放 public/

app.post("/api/ask", async (req, res) => {
  const r = await fetch("https://api.openai.com/v1/chat/completions", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      // 金鑰只從環境變數讀，絕不寫死在程式碼裡
      "Authorization": `Bearer ${process.env.OPENAI_API_KEY}`
    },
    body: JSON.stringify({
      model: "gpt-4o-mini",
      messages: [{ role: "user", content: req.body.prompt }]
    })
  });
  res.json(await r.json());
});

app.listen(process.env.PORT || 3000);
BASH 第一段之二 · 在本機準備專案 
# 1. 打開終端機，切換到你的專案資料夾
cd ~/Desktop/qa-tool

# 2. 初始化 npm 並安裝後端用到的套件
npm init -y
npm install express

# 3. 把前端檔案（index.html 等）移進 public/ 資料夾
mkdir public && mv index.html public/

# 4. 建立 .gitignore，確保金鑰與 node_modules 不會被上傳
echo "node_modules/" > .gitignore
echo ".env" >> .gitignore

# 5. 在 package.json 的 "scripts" 加一行啟動指令：
#    "start": "node server.js"  ← 部署平台靠這行啟動後端

# 6. 本機先測一次：設臨時環境變數再啟動
OPENAI_API_KEY=你的金鑰 npm start
# 瀏覽器開 http://localhost:3000 能用，就代表後端沒問題

本機測通後，停掉伺服器（終端機按 Ctrl + C），接著把專案推上 GitHub。先到 github.com 登入，點右上角「+」→「New repository」，repo 名稱填 qa-tool，不要勾「Add a README」，按「Create repository」。

BASH 第二段 · 推上 GitHub 
# 7. 把專案存進 Git 並提交
git init
git add .
git commit -m "含後端代理的家長問答小幫手"

# 8. 連到 GitHub repo 並推上去（網址換成你自己的）
git remote add origin https://github.com/你的帳號/qa-tool.git
git branch -M main
git push -u origin main

推完後到 GitHub repo 頁面確認：看得到 server.js、package.json、public/，但看不到 .env 也看不到 node_modules/（被 .gitignore 擋掉了）。確認後做第三段——在 Railway 部署：

9
建立 Railway 專案
登入 railway.app，點「New Project」→「Deploy from GitHub repo」，選剛剛推上去的 qa-tool。Railway 會自動讀到 npm start 並開始建置。
10
設定環境變數（這步是金鑰保護的關鍵）
在專案的「Variables」分頁，新增一筆：名稱 OPENAI_API_KEY、值貼上你的金鑰。金鑰只存在這裡——不在程式碼、不在 GitHub。存檔後 Railway 會自動重新部署。
11
產生公開網址
到「Settings」→「Networking」→「Generate Domain」，Railway 會給你一個 .up.railway.app 結尾的網址。等建置狀態變成綠色的「Success」，就可以打開這個網址了。

任務說明

這次任務是把 CH5-4 的知識完整實作一遍。你可以選擇用 Node.js（Express）或 Python（Flask）——兩種語言的後端架構在 CH5-4 都有說明，選你比較熟悉的那個。

1
建立專案目錄，初始化後端

三段交付清單（每 30 分鐘一段）

第 1 段（30 min）：列出你想做的工具需求
定義你這個完整專案的功能與邊界：
  1. 這個工具要解決什麼問題？目標使用者是誰？
  2. 使用者輸入什麼，得到什麼輸出？
  3. MVP 版本只做哪幾個功能（不超過 3 個）？
  4. 會用到哪個 AI API？（OpenAI / Anthropic / Gemini）
第 2 段（30 min）：拆成前端 + 後端工作
把需求拆解成前端（HTML/JS）和後端（API proxy）各自要做什麼：
  1. 前端：使用者介面（輸入框、按鈕、結果顯示區）
  2. 後端：接收前端請求、帶 API key 呼叫 AI、回傳結果（key 不暴露在前端）
  3. API endpoint 設計：路徑、request body 結構、response 結構
第 3 段（30 min）：部署兩端 + 測試
把前後端都部署上線，驗證完整流程跑通：
  1. 後端先部署，取得 API URL（如 https://your-app.railway.app/api/chat）
  2. 前端的 fetch 指向後端 URL，部署前端
  3. 在瀏覽器的 DevTools Network tab 確認：前端只向你的後端發 request，沒有直接打 AI API
  4. 測試正常流程（有輸入、有輸出）和邊界情況（空輸入、AI 回傳空）

API key 直接寫在程式碼裡——只要 git push 或分享給任何人，key 就外洩了；即使之後移除，git 歷史記錄裡仍然留著 2. 沒有使用 .env 檔案，無法在不同環境（本機開發 / Vercel 生產）用不同的 key 3. 如果環境變數沒設定，process.env.OPENAI_API_KEY 會是 undefined，程式會帶著 undefined 去呼叫 API 並收到 401，錯誤訊息不明確 修正版： 步驟 1：建立 .env 檔（不要 commit 進 git） OPENAI_API_KEY=sk-abc123xyz456 步驟 2：確認 .gitignore 有排除 .env .env .env.local node_modules/ 步驟 3：server.js 改成讀取環境變數 // server.js require('dotenv').config(); // 本機開發用，Vercel 自動注入不需要 const express = require('express'); const app = express(); // 啟動時就檢查，而不是等到被呼叫才發現 const API_KEY = process.env.OPENAI_API_KEY; if (!API_KEY) { console.error('錯誤：OPENAI_API_KEY 環境變數未設定'); process.exit(1); // 明確報錯，不讓程式繼續跑 } app.post('/api/chat', async (req, res) => { const response = await fetch('https://api.openai.com/v1/chat/completions', { headers: { 'Authorization': `Bearer ${API_KEY}` }, }); res.json(await response.json()); }); app.listen(3000); 步驟 4：Vercel 上設定環境變數 Vercel Dashboard → 你的專案 → Settings → Environment Variables → 新增 OPENAI_API_KEY 關鍵改動：API key 移到 .env（不進 git）；加 .gitignore 排除 .env；啟動時檢查環境變數存在（process.exit(1) 讓錯誤明確而不是靜默帶 undefined 呼叫 API）；Vercel 上透過 Dashboard 設定（不寫死在程式碼）。

案例 04 · 接案後端
客戶要整合 OpenAI，但 key 放前端第三天就被偷了
難度 ★★★ 約 60 分鐘

你接的客戶做了一個 AI 食譜生成器，上線第三天收到 OpenAI 帳單警告：單日用量從平常 $2 暴增到 $180。打開前端 F12 一看，key 直接 hardcode 在 fetch 呼叫裡。客戶很慌，要你在 24 小時內改用後端代理，key 撤換後重發，額度控制在每 IP 每天 20 次以內。

原始資料｜現有前端程式碼（問題版）
// app.js（目前上線版本） const OPENAI_KEY = 'sk-proj-AbC1234xyz...'; async function generateRecipe(ingredients) { const res = await fetch('https://api.openai.com/v1/chat/completions', { method: 'POST', headers: { 'Authorization': 'Bearer ' + OPENAI_KEY, 'Content-Type': 'application/json' }, body: JSON.stringify({ model: 'gpt-4o-mini', messages: [{role:'user', content:`用這些食材寫食譜：${ingredients}`}] }) }); const data = await res.json(); return data.choices[0].message.content; } OpenAI 帳單警告（節錄）： 2026-04-15 單日用量：$180.43 USD 2026-04-14 單日用量：$2.18 USD 2026-04-13 單日用量：$1.92 USD
你的任務
  1. 寫一個最小可跑的 Node.js 後端代理（express + fetch，一個 /api/recipe 端點）
  2. 加上每 IP 每天 20 次的速率限制（可用 Map 內存版先做，說清楚正式環境該換什麼）
  3. 改前端 app.js，只呼叫自家 /api/recipe，不再碰 OpenAI key
評估指標
key 不外洩
後端從 process.env 讀 key，前端完全看不到 sk- 開頭字串
額度可控
每 IP 每天 20 次限制寫死，超過回 429 並給清楚錯誤訊息
前端改動小
只改 fetch URL 與 body，不要重寫整個 generateRecipe 函式
正式環境提醒
說明內存版速率限制的缺點，建議正式用 Redis 或平台原生方案
參考 Prompt（建議先自己寫再看）
點擊展開參考答案
【角色】你是接案的後端，熟悉 Node.js + express，處理過 API key 外洩事故，會用最少的程式碼解決問題。 【任務】根據下方前端問題碼與帳單警告，產出： 1. server.js 最小完整版（express + /api/recipe + 速率限制，可直接 node server.js 跑起來） 2. 速率限制用 Map 內存版，加註釋說明為什麼不能 production 用 3. 前端 app.js 需要改的那幾行（用 diff 風格標 - / +） 4. 部署說明：OPENAI_KEY 放哪（Vercel / Railway / Render 三選一的設定位置），key 撤換流程 【限制】 - 不要引入 rate-limit 套件，用手寫 Map 版（教學目的讓學員看清原理） - 前端只能改兩處：fetch URL、body，不動其他邏輯 - 速率限制超過時回 HTTP 429 + JSON {error:'...'}，不要只回純文字 - 不要寫「非常抱歉給你帶來不便」這類客套話 【輸出格式】 分四段：server.js 完整程式碼、速率限制註釋、app.js diff、部署與撤 key 說明。
