含後端代理的完整部署

把 CH5-4 學到的後端架構實際跑一遍：建立代理伺服器、設定環境變數、推到 GitHub、部署到 Railway 或 Render。完成後你的 AI 工具就有了可公開的網址，且金鑰安全保存在雲端。

產出目標 · DELIVERABLE
本機可運行的後端伺服器（Node.js 或 Python）
含後端的工具已部署到 Railway 或 Render
公開網址可正常呼叫 AI API，且金鑰不暴露在前端
Task Overview

任務說明

這次任務是把 CH5-4 的知識完整實作一遍。你可以選擇用 Node.js（Express）或 Python（Flask）——兩種語言的後端架構在 CH5-4 都有說明，選你比較熟悉的那個。

如果你在 Part 4 已經有一個直接在前端呼叫 AI API 的工具，這次就把它「升級」成後端代理版本。如果沒有，建立一個新的最小後端工具也完全可以。

💡
從 CH5-4 的程式碼開始：不需要從零開始寫。複製 CH5-4 頁面裡的 server.js 範例作為起點，修改 prompt 和回應格式就好。
1
建立專案目錄，初始化後端
在桌面或任意位置建立一個新資料夾，例如 my-ai-backend。在終端機進入該目錄，執行 npm init -y 初始化 Node.js 專案。
2
安裝套件，複製後端程式碼
執行 npm install express @anthropic-ai/sdk dotenv。建立 server.js，貼入 CH5-4 的程式碼，根據你的工具調整 prompt 邏輯。前端 HTML 放在 public/ 目錄。
3
建立 .env 和 .gitignore
建立 .env 填入金鑰，建立 .gitignore 加入 .env 和 node_modules/。建立 .env.example 作為範本（可進 git）。
4
本機測試
執行 node server.js，開啟 http://localhost:3000，確認工具能正常呼叫 AI 並顯示結果。
5
推送到 GitHub，部署到 Railway / Render
確認 .env 不在 git 追蹤中後，git init、git add .、git commit、git push。在 Railway 或 Render 連接 repo，在環境變數頁面填入 API 金鑰，等待部署完成。

三段交付清單（每 30 分鐘一段）

第 1 段（30 min）：列出你想做的工具需求
定義你這個完整專案的功能與邊界：
  1. 這個工具要解決什麼問題？目標使用者是誰？
  2. 使用者輸入什麼，得到什麼輸出？
  3. MVP 版本只做哪幾個功能（不超過 3 個）？
  4. 會用到哪個 AI API？（OpenAI / Anthropic / Gemini）
第 2 段（30 min）：拆成前端 + 後端工作
把需求拆解成前端（HTML/JS）和後端（API proxy）各自要做什麼：
  1. 前端：使用者介面（輸入框、按鈕、結果顯示區）
  2. 後端：接收前端請求、帶 API key 呼叫 AI、回傳結果（key 不暴露在前端）
  3. API endpoint 設計：路徑、request body 結構、response 結構
第 3 段（30 min）：部署兩端 + 測試
把前後端都部署上線，驗證完整流程跑通：
  1. 後端先部署，取得 API URL（如 https://your-app.railway.app/api/chat）
  2. 前端的 fetch 指向後端 URL，部署前端
  3. 在瀏覽器的 DevTools Network tab 確認：前端只向你的後端發 request，沒有直接打 AI API
  4. 測試正常流程（有輸入、有輸出）和邊界情況（空輸入、AI 回傳空）

EVIDENCE · 證據提交
DEBUG · 修好這段環境變數讀取有問題的 Node.js 程式碼
// server.js
const express = require('express');
const app = express();

const API_KEY = "sk-abc123xyz456";  // 直接寫在程式碼裡

app.post('/api/chat', async (req, res) => {
  const response = await fetch('https://api.openai.com/v1/chat/completions', {
    headers: { 'Authorization': `Bearer ${API_KEY}` },
    // ...
  });
  res.json(await response.json());
});

app.listen(3000);
解法參考（先寫自己的版本再展開）
原版的問題： 1. API key 直接寫在程式碼裡——只要 git push 或分享給任何人，key 就外洩了；即使之後移除，git 歷史記錄裡仍然留著 2. 沒有使用 .env 檔案，無法在不同環境（本機開發 / Vercel 生產）用不同的 key 3. 如果環境變數沒設定，process.env.OPENAI_API_KEY 會是 undefined，程式會帶著 undefined 去呼叫 API 並收到 401，錯誤訊息不明確 修正版： 步驟 1：建立 .env 檔（不要 commit 進 git） OPENAI_API_KEY=sk-abc123xyz456 步驟 2：確認 .gitignore 有排除 .env .env .env.local node_modules/ 步驟 3：server.js 改成讀取環境變數 // server.js require('dotenv').config(); // 本機開發用，Vercel 自動注入不需要 const express = require('express'); const app = express(); // 啟動時就檢查，而不是等到被呼叫才發現 const API_KEY = process.env.OPENAI_API_KEY; if (!API_KEY) { console.error('錯誤：OPENAI_API_KEY 環境變數未設定'); process.exit(1); // 明確報錯，不讓程式繼續跑 } app.post('/api/chat', async (req, res) => { const response = await fetch('https://api.openai.com/v1/chat/completions', { headers: { 'Authorization': `Bearer ${API_KEY}` }, }); res.json(await response.json()); }); app.listen(3000); 步驟 4：Vercel 上設定環境變數 Vercel Dashboard → 你的專案 → Settings → Environment Variables → 新增 OPENAI_API_KEY 關鍵改動：API key 移到 .env（不進 git）；加 .gitignore 排除 .env；啟動時檢查環境變數存在（process.exit(1) 讓錯誤明確而不是靜默帶 undefined 呼叫 API）；Vercel 上透過 Dashboard 設定（不寫死在程式碼）。

延伸案例

後端代理的概念與 express 範例跑過一次之後，拿一個真實會有 API key 被偷的情境走一遍。重點是「為什麼不能直接放前端」以及「後端代理的最小完整程式碼」，不是寫花俏的中介層。

案例 04 · 接案後端
客戶要整合 OpenAI，但 key 放前端第三天就被偷了
難度 ★★★ 約 60 分鐘

你接的客戶做了一個 AI 食譜生成器，上線第三天收到 OpenAI 帳單警告：單日用量從平常 $2 暴增到 $180。打開前端 F12 一看，key 直接 hardcode 在 fetch 呼叫裡。客戶很慌，要你在 24 小時內改用後端代理，key 撤換後重發，額度控制在每 IP 每天 20 次以內。

原始資料｜現有前端程式碼（問題版）
// app.js（目前上線版本） const OPENAI_KEY = 'sk-proj-AbC1234xyz...'; async function generateRecipe(ingredients) { const res = await fetch('https://api.openai.com/v1/chat/completions', { method: 'POST', headers: { 'Authorization': 'Bearer ' + OPENAI_KEY, 'Content-Type': 'application/json' }, body: JSON.stringify({ model: 'gpt-4o-mini', messages: [{role:'user', content:`用這些食材寫食譜：${ingredients}`}] }) }); const data = await res.json(); return data.choices[0].message.content; } OpenAI 帳單警告（節錄）： 2026-04-15 單日用量：$180.43 USD 2026-04-14 單日用量：$2.18 USD 2026-04-13 單日用量：$1.92 USD
你的任務
  1. 寫一個最小可跑的 Node.js 後端代理（express + fetch，一個 /api/recipe 端點）
  2. 加上每 IP 每天 20 次的速率限制（可用 Map 內存版先做，說清楚正式環境該換什麼）
  3. 改前端 app.js，只呼叫自家 /api/recipe，不再碰 OpenAI key
評估指標
key 不外洩
後端從 process.env 讀 key，前端完全看不到 sk- 開頭字串
額度可控
每 IP 每天 20 次限制寫死，超過回 429 並給清楚錯誤訊息
前端改動小
只改 fetch URL 與 body，不要重寫整個 generateRecipe 函式
正式環境提醒
說明內存版速率限制的缺點，建議正式用 Redis 或平台原生方案
參考 Prompt（建議先自己寫再看）
點擊展開參考答案
【角色】你是接案的後端，熟悉 Node.js + express，處理過 API key 外洩事故，會用最少的程式碼解決問題。 【任務】根據下方前端問題碼與帳單警告，產出： 1. server.js 最小完整版（express + /api/recipe + 速率限制，可直接 node server.js 跑起來） 2. 速率限制用 Map 內存版，加註釋說明為什麼不能 production 用 3. 前端 app.js 需要改的那幾行（用 diff 風格標 - / +） 4. 部署說明：OPENAI_KEY 放哪（Vercel / Railway / Render 三選一的設定位置），key 撤換流程 【限制】 - 不要引入 rate-limit 套件，用手寫 Map 版（教學目的讓學員看清原理） - 前端只能改兩處：fetch URL、body，不動其他邏輯 - 速率限制超過時回 HTTP 429 + JSON {error:'...'}，不要只回純文字 - 不要寫「非常抱歉給你帶來不便」這類客套話 【輸出格式】 分四段：server.js 完整程式碼、速率限制註釋、app.js diff、部署與撤 key 說明。
變體挑戰
A把內存速率限制換成用 cookie + 前端記憶體，討論這比後端限制「弱」在哪
B加一個「今日剩餘次數」回傳給前端顯示，需同時回傳在成功與 429 情境
C把 OpenAI 換成 Gemini API，寫清楚同一個 proxy 模板要改哪幾行就能遷移
