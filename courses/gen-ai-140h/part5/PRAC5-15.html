(PRAC 5.15)

自貼 API KEY 的 AI 工具（自用 demo 版）

90 分鐘做一個只有你自己能用的 AI 工具，並學會「自用 vs 可分享」這條最重要的安全邊界。

學習成果
用提示詞讓 AI 產出含 API KEY 輸入框的單檔 HTML 工具
在 Google AI Studio 申請 KEY、設 referrer 限制、部署到 GitHub Pages
建立「KEY 不進 repo」習慣，並清楚知道「自用 demo 不能分享」的邊界
(01)

為什麼要自己做 AI 工具？小陳的故事

本單元定位
跟 PRAC5-3「API 串接展示頁」不同：PRAC5-3 練的是「fetch 怎麼跑、回應怎麼處理」（機制）；本單元練的是安全意識為主軸、技術為輔——KEY 怎麼安全地進入網頁、為什麼不能寫死、為什麼做完不能分享。

行政小陳這週做了個人單頁、做了同事姓名遊戲，信心爆棚。她想到一個自己每天都會用的東西：「我每次寫 ChatGPT 提示詞都覺得太粗，想有個工具幫我改得更具體。」朋友推薦了幾個 SaaS，每月 $10 美金，要訂閱、要登入。她突然想：「這種小工具，我自己用 AI 做一個不就好了？」

她貼提示詞給 Claude → Claude 給她一個單檔 HTML、頁面有輸入框可以貼她的 Gemini API KEY → 部署到 GitHub Pages → 上班打開、貼 KEY、貼粗略提示詞、按優化 → 看到 AI 改寫後更具體的版本。問題來了：她想分享給組長用。這時候紅燈響起來。

🔴
「這個工具能不能分享給別人？」答案是：不能，至少這個版本不行。這 90 分鐘不只教你做出工具，更要教你「為什麼不能分享」的安全意識。
(02)

自用 vs 可分享：三種 AI 工具部署形態

本頁最重要的觀念
以下三種形態決定你的工具安不安全、能不能分享。本單元做的是 B。讀完再往下。
形態 誰來付 AI 帳單？ 安全狀態 適合什麼？
A. KEY 寫死在程式碼裡 你（但任何人都能看到你的 KEY） 🔴 超不安全 永遠不要做
B. 使用者前端貼 KEY 訪客自己付（貼自己的 KEY） 🟡 只有自用安全 自用 demo（本單元）
C. KEY 藏在後端 proxy 你（訪客看不到 KEY） 🟢 安全 可分享給朋友（PRAC5-17）

為什麼 A 永遠不做？把 KEY 寫進 HTML/JS → 推到 GitHub Pages → 任何人按右鍵「檢視原始碼」都能看到你的 KEY → 24 小時內 Google AI Studio 額度被刷爆、信用卡被刷錢。這條死路有人走過，你別重蹈覆轍。

為什麼 B 只有自用安全？你的工具有個輸入框讓訪客貼 KEY → 訪客把自己的 KEY 貼進去、KEY 在他的瀏覽器 sessionStorage 裡 → 你之後改頁面 JS（哪怕只是 bug）→ 訪客的 KEY 可能洩漏 → 訪客不會檢查你的程式碼有沒有 bug、他信任你 → 出事你扛責任。

🔴
今天你學的是 B。完成的工具自己用 OK，不要分享連結讓別人貼 KEY。要做可分享版請等 PRAC5-17（Phase 2，Vercel proxy 把 KEY 藏在後端）。
(03)

兩個範例提示詞，任挑一個跑完整流程

兩張提示詞卡任挑一個。建議現場示範用範例 A（提示詞優化器），最貼近日常工作場景。

範例 A
AI 提示詞優化器
★ 建議現場用
請用單一 HTML 檔做一個 AI 提示詞優化器。要求： - 頁面上方放一個 <input type="password"> 讓使用者貼 Gemini API KEY - KEY 用 sessionStorage 暫存（關掉分頁就消失），絕對不要寫進程式碼 - 下方一個 textarea 讓使用者貼粗糙的提示詞 - 一個「優化」按鈕，按下去呼叫 Gemini API（model: gemini-2.0-flash） - 把粗糙提示詞改寫得更具體（指定角色、任務、限制、輸出格式） - 結果顯示在下方另一個 textarea，可一鍵複製 - 頁面醒目位置加紅字警告「此頁僅供你自己使用，請勿分享連結讓他人貼 KEY」 - 介面米白底、簡潔現代 - API 錯誤要顯示具體訊息（不要只說「壞了」） 請直接給我完整 HTML。
小提示：部分 AI 會先說一句「我看懂了，這是...」再給程式碼，這也 OK。
範例 B
翻譯小幫手
請用單一 HTML 檔做一個翻譯小幫手。要求： - 頁面上方 <input type="password"> 讓使用者貼 Gemini API KEY - KEY 用 sessionStorage 暫存、絕對不寫進程式碼 - 兩個下拉選單：來源語言、目標語言（中英日韓泰越六選一） - 一個 textarea 讓使用者貼要翻譯的文字 - 「翻譯」按鈕、呼叫 Gemini API（model: gemini-2.0-flash） - 結果顯示在下方、可一鍵複製 - 加紅字警告「此頁僅供你自己使用，請勿分享連結讓他人貼 KEY」 - 介面米白底、簡潔現代 請直接給我完整 HTML。
適合需要日常多語言翻譯的學員選用。
(04)

AI 對話完整示範（講師現場跑範例 A）

講師現場跑一次範例 A，用 Claude Artifacts（最穩、產出單檔最乾淨）。

🔴
示範前再次確認邊界：等下 Claude 產出的 HTML 是「使用者貼 KEY」的 B 形態——你自己用 OK，不能分享連結讓他人貼 KEY。
教師示範 · 範例 A 完整對話流程
第 1 步：複製範例 A 提示詞，貼進 Claude Artifacts
直接複製上方「範例 A」的完整提示詞文字，貼到 Claude 的輸入框送出。
第 2 步：Claude 回覆完整 HTML
Claude Artifacts 回覆
Claude 大概會回

好的，這是一個讓你自己貼 KEY、優化提示詞、且明確標示為自用 demo 的單檔 HTML 工具：

實際輸出每次略有差異，重點是 HTML 包含：密碼輸入框 + sessionStorage 暫存 + 優化按鈕 + 紅字警告。

Claude 給出的 HTML 結構關鍵觀察：

<!-- KEY 用 password 輸入框 + sessionStorage 暫存，不寫死 --> <div class="warning"> ⚠️ 此頁僅供你自己使用，請勿分享連結讓他人貼 KEY </div> <input type="password" id="apiKey" placeholder="AIza..."> <script> // 從 sessionStorage 還原 KEY（同分頁內） document.getElementById('apiKey').value = sessionStorage.getItem('GEMINI_KEY') || ''; document.getElementById('apiKey') .addEventListener('input', e => sessionStorage.setItem('GEMINI_KEY', e.target.value)); async function optimize() { const key = document.getElementById('apiKey').value.trim(); const url = `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=${key}`; // ... fetch 呼叫 Gemini API ... } </script>
💡
三個關鍵觀察：(1) KEY 用 type="password"（輸入時看不到）(2) sessionStorage.setItem 暫存、關掉分頁就消失 (3) 程式碼裡沒有任何寫死的 KEY——你 fork 到 GitHub 是空 KEY 樣本
第 3 步：儲存 HTML、準備部署
把 Claude 回給你的完整 HTML 複製，貼到記事本存成 index.html，或直接在 github.dev 新建檔案貼上。
安全的 HTML AI 工具 = 密碼輸入框 + sessionStorage + 程式碼零 KEY + 頁面紅字警告，四者缺一都不算完整。
(05)

Google AI Studio 申請 KEY（必教，80% 學員第一次做）

申請流程 · 三步完成
步驟 1 — 開 Google AI Studio 並登入
開 aistudio.google.com → 用你的 Google 帳號登入
第一次登入會問「Choose a country」→ 選台灣 → 勾兩個 checkbox 同意服務條款 → Continue
步驟 2 — 申請 API KEY
左側選單點 Get API key（鑰匙圖示）→ 右側點藍色按鈕 Create API key
跳出對話框 → Create API key in new project → 選 My First Project → Create

畫面出現 AIza... 開頭的長字串。點右邊「複製」圖示 → KEY 進剪貼簿

重要：KEY 只顯示一次。現在立刻開記事本貼上、或用 1Password 存起來。等下要用。
步驟 3 — 設 referrer 限制（強烈建議做）
在 API key 列表找到剛建的 KEY → 點 Edit API key

設定頁操作：
1. API restrictions → 選 Restrict key → 下拉選 Generative Language API
2. Application restrictions → 選 HTTP referrers (web sites)
3. Add an item → 填 https://{你的帳號}.github.io/*（注意末尾 /* 不能漏）
4. Save

設完後：這把 KEY 只能在你的 GitHub Pages 網域用，別人拿去自己的網站會被 Google 擋。
申請 KEY 後立刻存到安全地方 + 設 referrer 限制 = 5 分鐘換 10 倍安全。
🔴
referrer 最常錯的點：只寫 https://你的帳號.github.io 沒有 /* → GitHub Pages 上跑不動，報 API_KEY_HTTP_REFERRER_BLOCKED。一定要 /*。

(07)

卡關地圖：5 個常見錯誤

第 1 個（KEY 寫死）和第 4 個（想分享）是最高危，視覺最醒目。

🔴 錯誤 1
KEY 寫死
症狀：在 github.dev 編輯 index.html，把 KEY 直接寫進 JS：const key = "AIza..."
修法：立刻刪掉 → commit push → 回 Google AI Studio 刪除這把 KEY（舊的視為已洩漏）→ 重新申請。
鐵律：每次按 Commit 之前，先按 Cmd+F 搜 AIza——不應該出現在程式碼任何地方。
錯誤 2
referrer 漏 /*
症狀：部署後貼 KEY 按優化 → 看到 API_KEY_HTTP_REFERRER_BLOCKED
原因：referrer 設成 https://chen.github.io 沒有 /*
修法：回 Google Cloud Console → referrer 改成 https://chen.github.io/* → Save → 等 5 分鐘
錯誤 3
KEY 額度用完
症狀：跑幾次後報 Quota exceeded
修法：等 24 小時配額重置（免費版每天 1500 次請求）。學員自用日常絕對夠。
🔴 錯誤 4
想分享連結
症狀：學員做完超興奮、想丟群組分享
講師現場攔截：「慢著！你做的是自用 demo，不可以分享連結讓別人貼 KEY。要分享給朋友用怎麼辦？等 Phase 2 上線的 PRAC5-17，教你用 Vercel proxy 把 KEY 藏在後端、訪客不用自備 KEY——那才是可以分享的版本。」
錯誤 5
iOS Safari
症狀：Mac 正常、iPhone Safari 按優化沒反應
修法：回 AI 說「請確保程式碼在 iOS Safari 16+ 能跑：(1) fetch 加 mode: 'cors' (2) 複製改用 navigator.clipboard.writeText()」

完成後：你會做的 + 你不應該做的

完成本單元你具備的能力：

✅用提示詞讓 AI 產出含 API KEY 輸入框的工具
✅在 Google AI Studio 申請 KEY、設 referrer 限制
✅部署成自用 demo、能呼叫真實 AI API
✅「KEY 不進 repo」習慣（commit 前先搜 AIza）

你不應該做的四件事：

❌
分享這個版本的連結給朋友 — 你的工具是 B 形態（使用者自貼 KEY），分享後對方的 KEY 安全由你負責。要分享請等 PRAC5-17 Phase 2。
❌
把 KEY 寫死進程式碼 — 永遠不要。任何人看原始碼都能拿走你的 KEY，24 小時內額度歸零。
❌
把 KEY 截圖傳給同事「你用這把」 — KEY 是你的，別人用會吃你的配額，出事你扛。
❌
不設 referrer 限制 — 5 分鐘換 10 倍安全，沒理由不做。
💡
你的下一步：每天上班開 https://你的帳號.github.io/prompt-optimizer/，把工作要寫的提示詞先過一輪優化，產能直接拉高。等 Phase 2 的 PRAC5-17 上線後，做 Vercel proxy 版，才是你能丟群組炫耀的版本。

(11)

觀念確認

Q1. 你把自己申請的 Gemini API KEY 寫進 HTML 的 JavaScript 常數（const key = "AIza..."），然後部署到 GitHub Pages。以下哪項描述正確？
A. 只要頁面沒公開，KEY 不會被看到
B. GitHub Pages 會自動加密程式碼，KEY 安全
C. 任何人用瀏覽器「檢視原始碼」都能看到這把 KEY，極度危險
D. 只有自己知道網址才能看到 KEY
看解答

答案是 C。GitHub Pages 上的所有 HTML/JS 都是公開明文，「檢視原始碼」一秒可見。KEY 一旦被看到，任何人都能用你的配額，24 小時內帳單爆掉。KEY 永遠不寫進程式碼。

Q2. 你用「B 形態：使用者前端貼 KEY」做完了提示詞優化器，想分享連結給組長用。以下哪項說法最正確？
A. 只要告訴組長這是自用版就可以分享
B. 組長的 KEY 在他自己的瀏覽器，完全安全
C. 不應分享；若你之後改 JS 不小心洩漏組長的 KEY，責任在你。要分享請做 C 形態（PRAC5-17）
D. 可以分享，但要提醒組長不要分享給別人
看解答

答案是 C。B 形態的風險在於：你未來任何一次 JS 修改都可能引入 bug，組長的 KEY 可能因此洩漏，但組長不會去看你的程式碼。出事你扛責任。真正可以分享的版本是 C 形態（Vercel proxy，KEY 藏在後端），PRAC5-17 Phase 2 會教。
