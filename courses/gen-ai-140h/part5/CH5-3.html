Git 與版本保存：讓每一步都可以反悔

你有多少次改了頁面，然後想回到昨天的版本，卻發現完全回不去？Git 解決的就是這個問題——不是讓你不犯錯，而是讓每一個錯都有後路。

學習成果 · OUTCOMES
理解版本控制的核心思維：為什麼存檔不夠，需要「快照」
能在本機執行 Git 的基本操作：init、add、commit、log
知道如何用 git log 查看修改紀錄，並用 git checkout 回到特定版本
能在 GitHub 建立 repo 並將本機程式碼推送上去
(01) · Why Git

版本控制的核心概念：
為什麼存檔不夠？

你可能已經有「存檔」的習慣。每次修改前複製一份，取名 index_備份.html，或是 index_0414最終版.html。這個習慣說明你已經在思考版本——只是用了一個很脆弱的方式。

問題來了：改了五次之後，你有五個備份檔。你知道第三個版本的按鈕顏色是什麼嗎？你知道從第二個到第三個，改了哪些地方嗎？你能在十秒內回到「兩天前按鈕還沒壞掉的那個狀態」嗎？

Git 不是「備份工具」，是時間機器。每次 commit，你拍下一張快照——帶著時間戳記、作者、你自己寫的說明。
手動備份的問題
檔名越來越混亂，不知道哪個才是「正確版」
無法追蹤「這次改了什麼」，只知道有新版本
回溯需要手動比對兩個檔案的差異
多人協作時，不同人的修改容易互相覆蓋
Git 的解法
只有一個工作目錄，版本紀錄在 .git 資料夾裡
每個 commit 都有說明，記錄「為什麼改」
一行指令回到任何歷史版本
多人協作時自動追蹤誰改了什麼，衝突可手動解決

Git 有三個概念區域，是理解所有指令的基礎：

工作區
Working Directory
你正在編輯的檔案
→ git add →
暫存區
Staging Area
準備要提交的修改
→ git commit →
版本庫
Repository
永久保存的快照紀錄
💡
暫存區（Staging Area）的用途：讓你在「commit 之前」先決定這次要包含哪些修改。你可以改了五個檔案，但只把其中三個加入這次的 commit——這讓每個 commit 的說明能更精確。
(02) · Basic Commands

第一次使用 Git：
init、add、commit 三步驟

打開終端機，切換到你的專案目錄。以下三個指令就是你 90% 的日常 Git 操作——先學熟這三個，其他的慢慢補。

01
初始化 Git 追蹤（只需做一次）
在你的專案目錄下執行 git init。這會建立一個隱藏的 .git 資料夾，從此這個目錄的所有修改都能被追蹤。
02
把修改加入暫存區
執行 git add 檔案名稱，或用 git add . 一次加入所有修改。這是告訴 Git：「這些東西我要納入下一次的快照。」
03
提交快照並寫說明
執行 git commit -m "說明文字"。說明文字很重要——未來你需要回溯時，靠的就是這段文字判斷每個版本做了什麼。

完整的第一次操作流程如下：

BASH 第一次初始化並提交 
# 進入你的專案目錄
cd ~/Desktop/my-tool

# 初始化 Git（只做一次）
git init

# 查看目前哪些檔案有修改（養成習慣）
git status

# 把所有檔案加入暫存區
git add .

# 提交第一個快照
git commit -m "初始版本：工具頁面基本架構完成"

之後每次改完，重複 git add . 和 git commit -m "說明" 兩個步驟即可。以下是完整的常用指令速查表：

指令 用途 什麼時候用
git init 在目前目錄建立 Git 版本庫 專案第一次設定
git status 查看哪些檔案有修改、哪些在暫存區 add 前養成習慣先確認
git add . 把所有修改加入暫存區 要提交前
git add 檔案 只把特定檔案加入暫存區 只想提交部分修改時
git commit -m "說明" 提交暫存區的內容成為一個版本快照 每次完成一個小段落就提交
git log --oneline 查看簡短的版本歷史 需要找回特定版本時
⚠️
commit 說明寫什麼？寫「改了什麼」沒那麼重要——Git 自己記得你改了哪些行。寫「為什麼改」才是關鍵。
差別：update button vs 修正送出按鈕在 iOS Safari 無法點擊的問題。三個月後翻紀錄，前者毫無幫助。

(03) · Time Travel

查看與回溯：
log、diff 與 checkout

Git 最有價值的能力不是「存檔」，而是讓你在任何時候回到任何版本。掌握這三個指令，你就擁有了真正的反悔能力。

BASH 查看版本歷史 
# 顯示完整的 commit 歷史
git log

# 每個 commit 只顯示一行（ID + 說明）
git log --oneline

# 輸出範例：
# a3f8c12 修正 iOS Safari 送出按鈕無法點擊
# 7d2e901 新增聯絡表單，調整版面
# 4b1a034 完成首頁 Hero 區塊設計
# c9e2f57 初始版本：工具頁面基本架構完成

每個 commit 都有一個獨特的 ID（上面的 a3f8c12 這類的短碼）。你可以用這個 ID 做三件事：

查
查看兩個版本之間的差異
git diff 4b1a034 a3f8c12 — 會顯示從第一個 commit 到第二個 commit，每一行程式碼的新增（綠色 +）與刪除（紅色 -）。
回
回到某個舊版本來查看
git checkout 4b1a034 — 你的工作目錄會暫時變成那個版本的狀態。看完後用 git checkout main 回到現在。
還
正式回到舊版本（產生一個新 commit）
git revert a3f8c12 — 這會「反轉」指定的 commit，把它的修改撤銷，同時產生一筆新的紀錄。這是最安全的回溯方式，不會破壞歷史。
🔑
revert vs reset：初學者請優先使用 git revert——它在現有歷史上疊加「撤銷」，不破壞任何既有紀錄。git reset --hard 雖然「更徹底」，但會直接刪除歷史，操作錯誤的話真的回不來。

實際情境：你在 AI 協助下改了頁面，改到一半發現整個壞掉，但又不確定是哪步出錯。這時候的標準流程：

BASH 找出問題版本並回溯 
# 第一步：查看最近幾個 commit
git log --oneline

# b2c4d8e 用 AI 調整整體排版（壞了）
# a3f8c12 修正送出按鈕
# 7d2e901 新增聯絡表單

# 第二步：回到「新增聯絡表單」這個版本查看
git checkout 7d2e901

# 確認是這個版本還好的？ 執行撤銷
git checkout main  # 先回來
git revert b2c4d8e  # 撤銷那個壞掉的 commit
BEST PRACTICE · 提交頻率
不要等「全部完成」才提交。每完成一個小功能就 commit——改好了一個按鈕、加完了一段文字、測試通過了一個功能。commit 越小越具體，回溯就越精準。「改了一百個地方」的 commit 是 Git 最難利用的一種。

(04) · GitHub

推送到 GitHub：
讓程式碼存在雲端

到目前為止，所有的 Git 歷史只存在你的電腦裡。如果電腦壞了、或你想在另一台電腦繼續工作，你需要一個遠端 repo——GitHub 就是最常用的服務。

GitHub 上的 repo 是你本機 Git 版本庫的「雲端鏡像」。你用 git push 把本機的 commit 上傳，用 git pull 把雲端的更新同步下來。

1
在 GitHub 建立新 repo
前往 github.com，點右上角的 + → New repository。填入名稱（如 my-tool），選 Public 或 Private，不要勾選 Initialize with README（因為你本機已有檔案了）。
2
連接本機與遠端 repo
GitHub 建完後會給你一段指令。在終端機執行：git remote add origin https://github.com/你的帳號/my-tool.git
3
第一次推送
執行 git push -u origin main。-u 參數讓往後可以直接用 git push 而不需每次指定目標。
4
之後的日常操作
改了東西 → git add . → git commit -m "說明" → git push。三個步驟，15 秒，程式碼就安全存在雲端了。
BASH 完整的初始推送流程 
# 確認本機已有 commit（沒有的話先 git add . && git commit）
git log --oneline

# 連接遠端（只做一次）
git remote add origin https://github.com/yourname/my-tool.git

# 重新命名預設分支（如果需要）
git branch -M main

# 第一次推送（-u 設定追蹤關係）
git push -u origin main

# 往後的日常推送
git add .
git commit -m "修正首頁 RWD 在手機橫向的顯示問題"
git push
ℹ️
GitHub 推送需要驗證：第一次推送時，GitHub 可能要求你輸入帳號密碼或設定 Personal Access Token（PAT）。2021 年後 GitHub 已停用密碼驗證——建議在 Settings → Developer settings → Personal access tokens 產生一個 token，用 token 代替密碼。

把程式碼推上 GitHub 之後，你同時得到了兩個額外好處：第一，下一節課（CH5-4）我們會從 GitHub 直接連接到部署平台，讓你的頁面有公開網址。第二，GitHub 本身就是你的作品集——面試、接案、展示，都可以直接分享 repo 連結。
