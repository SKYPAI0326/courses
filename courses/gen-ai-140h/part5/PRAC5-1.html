單頁工具 v2 改版任務

拿出 Part 4 做的工具頁，用 CH5-2 學到的「位置＋意圖＋限制」三要素，真正改一次版。不靠猜測，只靠精準的 AI 指令。

產出目標 · DELIVERABLE
改版後的個人工具頁 v2（HTML 檔案，可在瀏覽器開啟）
至少完成 3 項具體修改，每項都能說出「改了什麼、為什麼改」
每項修改使用精準指令格式，不使用模糊描述
Task Overview

任務說明

你在 Part 4 建立了至少一個工具頁（PRAC4-1 ~ PRAC4-4）。這次的任務是選一個最想改進的工具，用 AI 協助完成一次有紀錄的改版。改版不是「重做」——重點是練習用精準語言描述修改需求，讓每次 AI 協作都有明確指令，而不是反覆試誤。

💡
沒有「正確答案」：改什麼、改多少，完全根據你的工具和偏好。重點是指令的品質，不是改版的範圍。
1
選定要改版的工具
從 PRAC4-1 ~ PRAC4-4 中選一個。建議選你最常用、或覺得「有點不順手」的那個——改版動機越具體，指令越容易寫。
2
列出 3 個想改的地方
不要貪多。列出 3 個具體的不滿意點，例如：「輸入框太小」「送出按鈕顏色不明顯」「結果區塊字體太小」。
3
用下方工具把每個想法轉成 AI 指令
填入位置、意圖、限制三個要素，生成格式化指令。每次只生成一個指令，執行確認後再進行下一個。
4
執行修改，在瀏覽器確認
把指令貼給 AI，執行修改後立刻在瀏覽器開啟確認。沒問題才進行下一個，別一次跑完全部。
5
存檔，命名為 v2
完成 3 項修改後，把工具頁另存，檔名加上 -v2（例如 my-tool-v2.html）。

DEBUG · 修好這個引用路徑錯誤的 HTML 片段
<!-- 在 courses/gen-ai-140h/part5/my-tool.html 裡 -->
<link rel="stylesheet" href="/assets/style.css">
<script src="/assets/app.js"></script>
<img src="/images/logo.png" alt="logo">
解法參考（先寫自己的版本再展開）
原版的問題： 1. 以 `/assets/` 開頭的是「根相對路徑」——在本機用瀏覽器直接開 HTML 檔時，根目錄是你的磁碟根目錄而非專案根目錄，/assets/ 會找不到檔案 2. 若專案部署在子目錄（如 GitHub Pages 的 /username/repo/），/assets/ 也會指向錯誤位置 3. img 的路徑同樣有這個問題，圖片會破圖 正確做法（相對路徑）： <!-- 從 courses/gen-ai-140h/part5/my-tool.html 往上 3 層到專案根目錄 --> <link rel="stylesheet" href="../../../assets/style.css"> <script src="../../../assets/app.js"></script> <img src="../../../images/logo.png" alt="logo"> 或者，若資源和 HTML 放在同一個 part5/ 資料夾： <link rel="stylesheet" href="_assets/style.css"> 驗證方式：在終端機用 python3 -m http.server 8080 啟動本機伺服器，用 http://localhost:8080 開頁面——如果這樣可以載入，但直接開 file:// 不行，表示你用了根相對路徑，需改成相對路徑。 關鍵改動：把 /assets/（根相對）→ ../../../assets/（相對路徑）；相對路徑無論在本機、GitHub Pages、Vercel 都能正確解析，因為它是相對於 HTML 檔案自己的位置而非伺服器根目錄。

延伸案例

三要素改版指令工具熟了之後，拿一個真實接手的舊專案完整做一次。先讀情境，寫 v1 改版指令，再用評估指標檢核 AI 回傳的 diff 是否符合「只改指定位置、不重構其他」。

案例 01 · 接案維運
接手三個月前離職同事留下的單頁 BMI 工具
難度 ★★☆ 約 40 分鐘

你是接案的前端，小型健身房客戶把三個月前離職同事留下的 BMI 工具頁轉交給你維護。客戶提出三個小改動：單位從 kg/cm 加 lb/inch 切換、結果區加體脂率預估、顏色調成品牌色 #5a7a5a。原始碼你只拿到這份單檔 HTML，不能重構架構（老闆說「別動壞它」）。

原始資料｜現有 HTML 片段
<!-- 原始 index.html（節錄）--> <div class="card"> <h1>BMI 計算機</h1> <input id="h" placeholder="身高 (cm)"> <input id="w" placeholder="體重 (kg)"> <button onclick="calc()">計算</button> <div id="out"></div> </div> <script> function calc(){ const h=parseFloat(h.value)/100, w=parseFloat(w.value); const bmi=(w/(h*h)).toFixed(1); out.textContent='BMI: '+bmi; } </script>
你的任務
  1. 用「位置＋意圖＋限制」三要素各寫一條改版指令（三個需求＝三條指令）
  2. 丟給 AI 跑一次，檢查回傳是否只動了指定區塊
  3. 若 AI 順便重構了計算函式，改寫限制條件重跑
評估指標
位置精確
指令明確到行號、id 或函式名，AI 不用猜要改哪裡
意圖清楚
說清楚要達到什麼效果，不只是「加一個功能」
限制完整
寫出「不動計算邏輯」「只回傳 diff」等邊界
diff 乾淨
AI 回傳的修改範圍與指令一致，沒有多餘重構
參考 Prompt（建議先自己寫再看）
點擊展開參考答案
【位置】現有 index.html 第 4–5 行的兩個 input 區塊之前。 【意圖】在身高、體重輸入框上方，新增一組「單位切換」的 radio 按鈕，預設選 kg/cm，可切到 lb/inch；切換時改變下方 input 的 placeholder 與 calc() 換算。 【限制】 - 只修改 input 區塊與 calc() 函式內換算部分，HTML 其他區塊、CSS、id 命名都不要動 - 不要把 calc() 重構成新架構（例如 class 或 module），保留原本 function 寫法 - 只回傳需要改的那幾行，不要貼整份 HTML 【輸出格式】 先列出要新增的 radio HTML，再列 calc() 修改後的完整函式，最後用一句話說明切換邏輯。
變體挑戰
A第二條指令（加體脂率預估）改用「位置＝calc() 函式結尾」寫一條，限制「不新增 function」
B第三條指令（改品牌色）改用「位置＝style 區塊的 .card」寫一條，限制「只改顏色變數，不動 padding / border-radius」
C把三條指令合併成一條大指令，觀察 AI 是否比較容易搞混位置——理解為什麼「一次只改一處」比較穩
自我檢核
RUBRIC · 自評檢核（4 點全勾才算過關）

同儕互評

自己改完 v2 很容易「越看越順眼」，卻漏掉別人一眼就看到的問題。兩兩配對，用 PR reviewer 的角度拆解彼此的 commit。

同儕互評 01 · Code Review 對拆
我寫的 commit，你當 PR reviewer 挑毛病
兩人一組 約 30 分鐘 需雙方完成 v2 改版

接案時沒有團隊幫你 review，容易長出壞習慣（過度複製貼上、內聯樣式亂噴、變數命名隨便）。這個活動讓你短暫戴上 reviewer 的帽子，用 3 個指標檢視別人的 commit，也被別人挑出你習以為常的壞習慣。

A作者（Author）
把你 PRAC5-1 改版完的工具頁，挑一個最有代表性的 feature commit（例如：「調整送出按鈕為主色並加 hover」），整理成 diff 或直接貼出修改前後片段給 B。說明一句「為什麼這樣改」就好，不要過度辯護。
B評審（Reviewer）
假裝自己是接手這份專案的下一位開發者。讀 A 的 diff，用下方 3 指標逐項寫評語，並指出至少 1 個「如果是我會這樣改」的替代方案。不能只說「看起來不錯」——沒具體建議等於沒 review。
活動節奏
1A 整理 1 個 commit 的 diff 給 B，附一句改動說明· 5 分鐘
2B 獨立讀 diff，寫下 3 指標評語＋ 1 個替代方案· 8 分鐘
3B 向 A 口頭說明評語，A 只能聽不能辯解· 5 分鐘
4角色互換再跑一次（A ↔ B）· 10 分鐘
5各自挑一個最受用的建議，當場改進 commit· 2 分鐘
互評指標（1-5 分）
可讀性命名清楚、格式一致、沒有無關改動混進來
必要性改動解決了真問題，不是「順手調一下」
可維護下一個接手的人能看懂，不會踩到隱形副作用
