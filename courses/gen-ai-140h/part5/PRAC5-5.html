專案紅綠燈週報工具

不只是點燈玩具——每列要有 owner、due date、燈號理由、下一步行動，整理完一鍵匯出 Markdown 週報，直接貼給主管。

PRACTICE GOALS
練習為決策工具設計資料模型：從點燈玩具升級到含 owner / due / 理由 / 行動的週報
用 localStorage 讓工具有真正的持久化，重新整理也不丟資料
產出一份可直接貼到主管 Slack 或 Email 的 Markdown 週報
TOOL · 互動工具

把每個專案的「燈號 + Owner + Due + 理由 + 行動」填齊，一鍵匯出 Markdown 週報

點燈循環切換綠 → 黃 → 紅。紅黃燈如果沒填「理由」或「下一步」，這列會自動標記「缺週報依據」——主管追問會卡。資料自動存到 localStorage，重新整理也不會消失。

→ 複製這段 Prompt 貼入 Gemini
你是專為週報設計的前端工程師。 請製作「專案紅綠燈週報工具」單頁 HTML，學員可用它對主管交付一份 Markdown 週報。 【資料模型】 每個專案物件： - name 專案名（必填） - status 燈號：green / yellow / red（必填，預設 green） - owner 負責人（必填） - dueDate 截止日（選填，YYYY-MM-DD） - reason 燈號理由（紅黃必填） - nextAction 下一步行動（紅黃必填） 【功能需求】 - 表格 row：燈號 / 專案 / Owner / Due / 理由 / 下一步 / 刪除 - 點燈循環切換（綠 → 黃 → 紅） - 紅黃燈如 reason 或 nextAction 為空，列底顯示「缺週報依據」warning - 載入範例按鈕：填入三條產品線（App push 紅 / 會員改版 黃 / B 端儀表板 綠） - 匯出週報按鈕：產 Markdown 到 textarea - 複製週報按鈕 - 清空重來：二次確認 - localStorage 持久化 【匯出 Markdown 格式】 # 本週專案紅綠燈週報 ## 摘要 紅燈 N｜黃燈 N｜綠燈 N 本週最需關注：(最緊急那條的專案 + 理由) | 燈號 | 專案 | Owner | Due | 理由 | 下一步 | |---|---|---|---|---|---| | 🔴 | ... | ... | ... | ... | ... | 【設計】 - 背景 #f5f3ee，卡片白色 - 燈號顏色：綠 #5a7a5a / 黃 #c9963a / 紅 #b5703a - 字體 Google Fonts Shippori Mincho（標題）+ Noto Sans TC（內文） - 整體簡潔，能直接在週會螢幕展示 請生成完整單一 HTML 檔，所有功能寫在 inline JS 中。
CHALLENGE · 進階挑戰

完成基本練習後，試試這些變化

PRACTICE VARIATIONS
01
加歷史快照：按一個「存檔本週」按鈕保存當前狀態到 localStorage，下週載入時可看到「上週紅燈這週變綠燈」的變化軌跡。
02
加標籤分類：每個專案多一個 tag 欄位（行銷 / 工程 / 法務...），匯出 Markdown 時自動依 tag 分群顯示。
03
加風險量化：紅燈再分「延一週 / 延一個月 / 延無期」三檔，匯出時依嚴重度自動排序，最該關注的放最上面。

延伸案例

紅綠燈看板工具做過一次之後，帶回真實工作中的專案狀態來用。重點是「哪些欄位值得上看板、哪些是雜訊」，不要把所有 Notion 欄位都搬過來。

案例 05 · 內部 PM
週一早會要對齊三條產品線進度，會議時間只剩 15 分鐘
難度 ★★☆ 約 35 分鐘

你是台灣一家 40 人 SaaS 公司的 PM，負責三條產品線：會員系統改版、B 端儀表板、App push 通知。每週一早會老闆只給 15 分鐘對齊，他不想聽細節，只想知道「哪條卡了、卡在誰」。你要用紅綠燈看板把三條線呈現在一頁內。

原始資料｜本週狀態筆記
【會員系統改版】 - 里程碑：4/25 上線 - 目前：後端 API 已完成，前端 UI 改版 70% - 卡點：設計師上週離職，新人還沒補齊，4/22 的設計 QA 可能延 - 擁有者：工程 Max、設計 Vacant 【B 端儀表板】 - 里程碑：4/30 Beta - 目前：資料表串接完，圖表還在改 - 卡點：無 - 擁有者：工程 Luna 【App push 通知】 - 里程碑：4/22 上線 - 目前：Apple 審核被退件一次，理由「推送同意書文案不完整」 - 卡點：法務還沒回覆新文案 - 擁有者：工程 Ken、法務 Ivy
你的任務
  1. 判斷三條產品線各是紅 / 黃 / 綠燈（並寫一行理由，要經得起老闆追問）
  2. 每條線寫一行「本週行動」（誰在什麼時間前做什麼）
  3. 最後用一句話做 15 分鐘的開場，告訴老闆「這 3 條裡最該關心哪一條」
評估指標
燈號有據
紅/黃/綠理由可追問，不是「感覺有點卡」這種模糊判斷
行動具體
寫明「誰 + 做什麼 + 哪一天前」，不用「持續追蹤」這種廢話
優先級清楚
開場一句話明確指出最該關注的一條，不是三條都提
不報細節
API 進度百分比、哪個 component 改了，都不該出現在這一頁
參考 Prompt（建議先自己寫再看）
點擊展開參考答案
【角色】你是資深 PM，習慣用紅綠燈看板跟高層對齊，重視「一頁可看懂、經得起追問」。 【任務】根據下方三條產品線筆記，產出： 1. 每條線的燈號（紅/黃/綠）+ 一行理由（≤30 字） 2. 每條線的本週行動（誰 + 做什麼 + 哪一天前），一行內完成 3. 開場 15 秒一句話：最該關心哪一條、為什麼 【限制】 - 燈號判斷標準：綠=里程碑安全、黃=有風險需人介入、紅=里程碑已確定延或被退 - 行動要有 owner 名字，「持續關注」「再看看」不算具體 - 開場不准提「整體進度良好」「大家辛苦了」 - 不要把任何技術細節搬進來（API、component 名稱都不寫） 【輸出格式】 | 產品線 | 燈號 | 理由 | 本週行動 | 表格三行，最後一句話開場。
變體挑戰
A把 App push（紅）換成 B 端儀表板突然出狀況的情境，重做一次判斷練習切換
B老闆改說他有 30 分鐘，加一個「下週 30 秒前瞻」欄位，要說出下週最大風險
C把這份看板改寫成 Slack 貼文格式（純文字 + emoji 紅黃綠），給團隊當週報同步
自我檢核
RUBRIC · 自評檢核（4 點全勾才算過關）
SOLO · 自我演練
把進度看板打開放在手邊；開新對話請 LLM 扮「每週要看你回報的 PM」，連續追問：① 哪個燈號的判斷標準最模糊（紅/黃/綠）？② 同一個任務，你和其他 RD 會打出不同顏色嗎？舉例。③ 如果這個燈號要寫進 SOP，紅黃綠各對應幾天延遲？——你必須回貼具體標準（不能說「快沒時間就紅」）。最後寫下修正後的燈號標準（一段條列）。
12 分鐘（演練 8 分 + 燈號標準 4 分）
進階選配 · 不計入課程時數（給想加練的學員）
把 LLM 對話截圖存到學習資料夾，或貼到工作群組／社群求 review。這些痕跡在轉職投履歷時可作為作品集素材。
