自由演練 5

專案進度「紅綠燈」看板

開會報告不需要複雜的甘特圖。用紅黃綠三色即時傳達每個專案的健康度，點一下就切換，5 秒完成週報更新。

PRACTICE GOALS
練習狀態循環切換（click to cycle）的互動邏輯
理解如何動態計算並顯示統計數字
產出一個可直接用於週報的專案狀態看板
TOOL · 互動工具

點擊圓點切換狀態，即時更新統計

點擊左側彩色圓點可循環切換「進行順利 → 注意風險 → 需要協助」三種狀態。直接點擊專案名稱可編輯文字。用「+ 新增專案」按鈕加入更多項目，底部統計列自動更新。

project-traffic-light.html
HOW TO REPLICATE · 如何自己做出這個工具

用這段 Prompt，讓 Gemini 幫你做一模一樣的工具

1
打開 Gemini，切換到「Canvas」模式
進入 gemini.google.com，點右上角切換到 Canvas 模式，這樣 AI 生成的 HTML 會直接可以預覽。
2
貼入以下指令
把下方指令完整複製，貼入 Gemini 的輸入框，直接送出。
3
等待生成，預覽並存檔
Gemini 會生成一個完整的 HTML 工具。點「預覽」確認功能正常後，點「下載」或「複製」儲存到你的電腦。
→ 複製這段 Prompt 貼入 Gemini
你是一位擅長製作視覺化管理工具的前端工程師。 請製作一個「專案進度紅綠燈看板」的單頁 HTML 工具，功能如下： 【功能需求】 - 清單顯示多個專案，每個專案名稱旁有一個彩色圓點（紅/黃/綠） - 點擊圓點可循環切換狀態：綠（順利）→ 黃（注意）→ 紅（需協助） - 雙擊或點擊專案名稱可以直接編輯 - 有「新增專案」按鈕，可加入空白列 - 頁面底部顯示三種狀態的統計數量 【設計要求】 - 背景 #f5f3ee，卡片白色，強調色 #b5703a - 綠色用 #5a7a5a，黃色用 #c9963a，紅色用 #b5703a（配合整體色系） - 圓點切換時有輕微縮放動畫 - 字體 Google Fonts（Shippori Mincho + Noto Sans TC） - 整體簡潔，適合週報時對著螢幕展示 請生成完整的單一 HTML 檔案。
CHALLENGE · 進階挑戰

完成基本練習後，試試這些變化

PRACTICE VARIATIONS
01
加入 localStorage 儲存功能，讓每次重新整理後專案狀態不會消失。
02
在每個專案列新增「備註」展開欄位——點擊箭頭展開後可以輸入一行說明文字。
03
加入「匯出為文字」按鈕，將所有專案狀態轉換為 Markdown 格式（可貼入週報訊息）。

延伸案例

紅綠燈看板工具做過一次之後，帶回真實工作中的專案狀態來用。重點是「哪些欄位值得上看板、哪些是雜訊」，不要把所有 Notion 欄位都搬過來。

案例 05 · 內部 PM
週一早會要對齊三條產品線進度，會議時間只剩 15 分鐘
難度 ★★☆ 約 35 分鐘

你是台灣一家 40 人 SaaS 公司的 PM，負責三條產品線：會員系統改版、B 端儀表板、App push 通知。每週一早會老闆只給 15 分鐘對齊，他不想聽細節，只想知道「哪條卡了、卡在誰」。你要用紅綠燈看板把三條線呈現在一頁內。

原始資料｜本週狀態筆記
【會員系統改版】 - 里程碑：4/25 上線 - 目前：後端 API 已完成，前端 UI 改版 70% - 卡點：設計師上週離職，新人還沒補齊，4/22 的設計 QA 可能延 - 擁有者：工程 Max、設計 Vacant 【B 端儀表板】 - 里程碑：4/30 Beta - 目前：資料表串接完，圖表還在改 - 卡點：無 - 擁有者：工程 Luna 【App push 通知】 - 里程碑：4/22 上線 - 目前：Apple 審核被退件一次，理由「推送同意書文案不完整」 - 卡點：法務還沒回覆新文案 - 擁有者：工程 Ken、法務 Ivy
你的任務
  1. 判斷三條產品線各是紅 / 黃 / 綠燈（並寫一行理由，要經得起老闆追問）
  2. 每條線寫一行「本週行動」（誰在什麼時間前做什麼）
  3. 最後用一句話做 15 分鐘的開場，告訴老闆「這 3 條裡最該關心哪一條」
評估指標
燈號有據
紅/黃/綠理由可追問，不是「感覺有點卡」這種模糊判斷
行動具體
寫明「誰 + 做什麼 + 哪一天前」，不用「持續追蹤」這種廢話
優先級清楚
開場一句話明確指出最該關注的一條，不是三條都提
不報細節
API 進度百分比、哪個 component 改了，都不該出現在這一頁
參考 Prompt（建議先自己寫再看）
點擊展開參考答案
【角色】你是資深 PM，習慣用紅綠燈看板跟高層對齊，重視「一頁可看懂、經得起追問」。 【任務】根據下方三條產品線筆記，產出： 1. 每條線的燈號（紅/黃/綠）+ 一行理由（≤30 字） 2. 每條線的本週行動（誰 + 做什麼 + 哪一天前），一行內完成 3. 開場 15 秒一句話：最該關心哪一條、為什麼 【限制】 - 燈號判斷標準：綠=里程碑安全、黃=有風險需人介入、紅=里程碑已確定延或被退 - 行動要有 owner 名字，「持續關注」「再看看」不算具體 - 開場不准提「整體進度良好」「大家辛苦了」 - 不要把任何技術細節搬進來（API、component 名稱都不寫） 【輸出格式】 | 產品線 | 燈號 | 理由 | 本週行動 | 表格三行，最後一句話開場。
變體挑戰
A把 App push（紅）換成 B 端儀表板突然出狀況的情境，重做一次判斷練習切換
B老闆改說他有 30 分鐘，加一個「下週 30 秒前瞻」欄位，要說出下週最大風險
C把這份看板改寫成 Slack 貼文格式（純文字 + emoji 紅黃綠），給團隊當週報同步
