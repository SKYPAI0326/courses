PART 7 · 專題衝刺與成果發表
PRAC 7.1 · 4 小時

開發衝刺第一輪

4 小時，目標一個：讓你的專題主要使用路徑可以跑通。不需要完美，需要可以讓人用。

任務目標
完成專題 v0：前端頁面架好、AI 串接跑通、核心輸入→輸出路徑可以演示。用下方「專題定義器」確認你的設計方向後，開始開發。
1
用專題定義器確認方向（15 分鐘）
填完下方工具的所有欄位，輸出會是你的開發規格書。如果填寫過程中發現哪個欄位填不出來，代表那個部分還沒想清楚——先停下來想，不要帶著模糊的設計開始寫程式。
2
先建 HTML 骨架（45 分鐘）
用 Part 5 學到的技術建前端：topbar、輸入區、輸出區、按鈕。不要美化，先讓結構跑通。可以直接複製 PRAC5-1 的頁面結構修改。
快速起步：打開 Claude Code，說「幫我建一個 AI 工具頁面，輸入區是 [你的欄位]，輸出區顯示 AI 回應，有 Loading 狀態和基本錯誤處理」
3
串接 AI（90 分鐘）
依你的架構決策選路線：靜態前端直呼（快，不安全，適合個人用）或後端代理（安全，較複雜）。先把核心路徑跑通，之後再加 System Prompt 優化。
4
跑三個測試案例（30 分鐘）
正常輸入、邊界輸入、空白輸入。把結果截圖存起來，PRAC7-4 需要。
5
Push 到 GitHub（15 分鐘）
就算程式碼不完整也 push。git init → git add . → git commit -m "v0 first working version" → git push。確認 .env 在 .gitignore 裡。
專題定義器 — v0 規格書生成
完成進度：0 / 5
第一輪衝刺驗收

延伸案例

v0 規格書練熟後，挑一個貼近你轉職處境的情境完整做一次。每個案例 30–45 分鐘，先讀情境與原始資料、寫 v1 產出、對照參考版，再用評估指標檢視。

案例 01 · 副業轉正職
用 90 秒 Pitch 爭取 CEO 立項你的 AI 副業工具
難度 ★★☆ 約 40 分鐘

你在一家 80 人的 SaaS 公司做產品經理，週末用 Claude API 寫了一個「客服回覆草稿生成器」，內部 3 位同事試用都說好用。下週一早會 CEO 開放 90 秒內部提案時段，你想爭取轉為正式立項（專屬工時 + 2 萬元 API 預算）。你要一份能當場念出來的 pitch 腳本。

原始資料｜副業專案現況
工具名稱：ReplyDraft 技術棧：靜態前端 + Claude Haiku API 目前數據：3 位同事試用 2 週，回覆平均撰寫時間從 8 分鐘降到 2 分鐘 每月 API 成本估算：1,200 元（200 張工單） 問題：同事用個人 API key 不可持續、需要公司整合客服系統 Zendesk CEO 已知痛點：客服部門上季離職率 25%，主管抱怨回覆品質參差
你的任務
  1. 寫一份 90 秒 pitch 腳本（約 270 字），結構：痛點 → 現況證據 → 提案 → 要什麼
  2. 搭配一頁投影片大綱（標題 + 3 個重點 bullet + 要求數字）
  3. 預想 CEO 會問的 2 個質疑點，各寫 30 字回應
評估指標
痛點先行
前 15 秒要讓 CEO 感到這跟他的 KPI 有關，不是你想玩技術
數字撐腰
至少兩個具體數字：節省時間、API 成本、或離職率
要求明確
講清楚要多少工時、多少預算、多久後可驗收
不吹牛
不講「革命」「顛覆」，只講可驗證的事實與合理推論
參考 Pitch 腳本（建議先自己寫再看）
點擊展開參考答案
【0–15 秒 痛點】 上季客服離職率 25%，主管在回報裡寫「人員撐不住工單量」。我聽到同事抱怨最多的一句話是：「回覆一張單要八分鐘，下班前根本清不完」。 【15–45 秒 證據】 過去兩週我寫了一個叫 ReplyDraft 的小工具，接 Claude API 自動生草稿。三位客服同事試用，平均撰寫時間從 8 分鐘降到 2 分鐘，正確率同事自評 4.2/5。每月 API 成本大約 1,200 元、處理 200 張單。 【45–75 秒 提案】 我想把這個工具正式接 Zendesk，讓整個客服部門都能用。預估正式版每月成本 8,000 元、每月可省下相當於 1 位兼職客服的工時。 【75–90 秒 要求】 我需要三件事：每週 8 小時專屬工時、2 萬元 API 與 Zendesk 整合預算、兩個月後用「平均回覆時間」做驗收。請讓我試看看。
變體挑戰
A改成給直屬主管（不是 CEO）的 3 分鐘版本，重點變成主管 KPI 而非公司策略
BCEO 當場回「聽起來小工具而已，為何不買現成 SaaS？」你要 60 秒回應
C把 pitch 改寫成 150 字 Slack 訊息版，附一張截圖的提案
自我檢核
案例 02 · 內部立項
寫一份「自動化週報 AI」v0 規格書說服主管批 3000 元 API 預算
難度 ★★☆ 約 35 分鐘

你是行銷企劃，每週一要彙整 4 個廣告平台（Google Ads、Meta、LINE、TikTok）的上週數據寫成中文週報。現況每週花 6 小時。你想做一個「貼 CSV 進去 AI 自動生成週報文字」的工具，主管說先看規格書再決定要不要批預算。

原始資料｜現況週報工作流
每週一 10:00 收集 4 平台 CSV 報表 11:00 Excel 做樞紐：點擊 / 曝光 / 花費 / CTR / CPA 13:00 撰寫中文週報（約 800 字，含亮點、警訊、下週建議） 15:00 發給主管 主管常問但週報沒寫的： - 為什麼 CPA 變化？ - 跟上上週比呢？ - 哪個素材要換？ 主管的底線：要可以交給新人照做、不能亂講數字、3000 元/月以下
你的任務
  1. 寫一份一頁 v0 規格書：一句話說明、輸入欄位、System Prompt、模型/Temperature、架構、先不做
  2. 明確寫出 MVP 邊界（先做 1 個平台還是 4 個？先生敘述還是也給建議？）
  3. 估算每月 API 成本並證明落在 3000 元內
評估指標
一句話清楚
使用者 + 問題 + 解法三要素齊全，主管讀一遍就懂
MVP 夠小
兩週內一個人能做完，邊界明確寫出「先不做」清單
成本可算
有 token 數估算，不是憑感覺寫「大約 3000 元」
驗收可測
寫清楚「怎樣算成功」：省幾小時、字數範圍、同事評分
參考規格書（建議先自己寫再看）
點擊展開參考答案
【一句話說明】 給每週要寫廣告週報的行銷企劃，不用手寫 800 字，貼 CSV 就能生成含亮點與警訊的中文週報初稿，作者再修 30 分鐘就能交。 【輸入欄位】 - Google Ads CSV（必填） - 上週 vs 上上週對比旗標（可選，預設開） - 週報語氣（下拉：正式／內部聊天） 【System Prompt】 你是資深數位廣告分析師，熟悉中文週報寫法。根據使用者貼的 CSV 數據，產出 600–800 字週報，結構：本週表現、週對週變化、三個亮點、兩個警訊、下週建議。嚴禁編造數字、只用輸入資料裡的數字。 【模型 / Temperature】 claude-haiku-4-5-20251001 / 0.3 【架構】 靜態前端 + Anthropic SDK（直接呼叫，不需後端） 【先不做】 - Meta/LINE/TikTok（先只做 Google Ads，成功再擴充） - 自動抓 API（先人工貼 CSV） - 素材層級建議（先只做帳戶層） - 圖表生成 【成本估算】 每份 CSV ≈ 3000 tokens 輸入 + 1200 tokens 輸出，Haiku 定價換算每份約 5 元。每月 4 份 × 4 人 = 64 元。含測試與重跑抓 300 元。遠低於 3000 元上限。 【驗收標準】 - 寫作時間從 6 小時降到 1.5 小時 - 主管抽查 4 份都沒有數字錯誤 - 新人 1 次 onboarding 就能用
變體挑戰
A主管說「我不懂技術，用故事講」——把規格書改寫成 200 字敘事版給主管看
B如果 Claude API 被公司資安封鎖，改用本地 Ollama + Gemma 的版本，重寫架構與成本段
C加入「失敗回退」策略：AI 輸出的數字跟 CSV 對不上時怎麼辦
自我檢核

同儕互評

專題衝刺階段最怕的是「自我感覺良好」——以為 pitch 清楚、以為 demo 沒 bug。找同組夥伴兩兩配對，一位扮演「台上」、一位扮演「評審」，用共同指標互評，把盲點提前逼出來。兩個活動各約 25–30 分鐘，最好連做。

同儕互評 01 · Pitch 互評
90 秒 Pitch 對拆——我講的，你聽懂了嗎
兩人一組 約 25 分鐘 需計時器

轉職面試、內部立項、接案提案都要在 90 秒內讓對方決定「繼續聽」還是「下一位」。自己念稿覺得流暢，別人聽完可能只抓到三個字。這個活動用互評把「我想表達的 vs 對方聽到的」落差逼出來。

A提案者
準備一份 90 秒 pitch（可用 PRAC7-1 案例 01 的腳本或自己專題的版本）。計時念給 B 聽，禁止念完補充、禁止看著 B 的表情改講法。
B評審
全程不打斷。聽完立刻（不看 A 的稿）寫下「我聽到什麼問題、什麼解法、他要我給什麼」。再用下方 4 指標評 1–5 分，並寫 2 個具體改進建議。
活動節奏
1各自準備/複習 pitch 稿· 3 分鐘
2A 對 B 念 90 秒 pitch（B 不打斷、不看稿）· 2 分鐘
3B 獨立寫下「我聽到什麼」＋4 指標評分＋2 改進建議· 5 分鐘
4對照：A 講原本想傳達什麼、B 講實際接收到什麼，標落差點· 5 分鐘
5角色互換，重來一次（B 當提案者、A 當評審）· 10 分鐘
互評指標（1-5 分）
清楚問題 → 解法 → 要求三段結構聽得出來
動機前 15 秒就讓我覺得「這跟我有關」
可行技術與成本講得像能跑，不是空話
差異化說得出為什麼不是用 ChatGPT 就好
活動檢核
同儕互評 02 · Demo 排演
5 分鐘 Demo 排演——角色扮演技術評審
兩人一組 約 30 分鐘 需開電腦實際 demo

最終 demo 不是「能跑」就夠，是「有人刻意挑毛病時還撐得住」。找搭檔實際對著你的專題演一次，對方扮演難搞的技術評審，在 5 分鐘內把你問到冒汗，這樣真正上場時反而會顯得冷靜。

ADemo 者
打開你的專題（本地或部署版皆可），對 B 做 5 分鐘完整 demo：問題 → 做法 → 實際操作 → 限制 → 後續。全程禁止說「這個還沒做好」、「等一下通常不會這樣」。
B技術評審
角色扮演一位刁鑽的技術評審（或你未來想應徵公司的工程主管）。Demo 中看到任何「怪怪的」都記下來，結束後挑出 3 個最嚴重的 bug／設計缺陷，並問 A 3 個犀利問題（技術、商業、法律各至少 1 個）。
活動節奏
1A 打開 demo 環境，確認可以跑· 2 分鐘
2A 做 5 分鐘 demo，B 全程筆記不打斷· 5 分鐘
3B 挑 3 個最嚴重的毛病＋問 3 個犀利問題，A 當場回答（不準說「回去想想」）· 8 分鐘
4兩人一起整理：哪幾題是真的要回去修、哪幾題是講法沒準備好· 5 分鐘
5角色互換，B 做 demo、A 當評審· 10 分鐘
評審四指標（1-5 分）
跑通demo 全程沒卡死、沒需要「換個輸入試試」
節奏5 分鐘內講完 5 段，沒超時也沒空白
誠實限制段有講真正的弱點，不是偽謙虛
抗問被問時不慌、答不出來敢說「不確定」
