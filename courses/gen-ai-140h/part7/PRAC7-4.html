部署與驗收

把 v1 部署到公開網址，完成 10 筆測試案例，更新作品集。這一小時結束，你有了一個真實存在於網路上的 AI 應用。

任務目標
部署到 Vercel / Netlify / Render，取得公開 URL，執行驗收測試並記錄通過率，更新作品集展示頁。
1
部署（20 分鐘）
靜態頁面：連接 GitHub repo 到 Vercel 或 Netlify，設定環境變數，點部署。
含後端：連接到 Render 或 Railway，設定環境變數，觀察 build log 確認成功。
最常見的部署失敗原因：環境變數名稱打錯（記得是 ANTHROPIC_API_KEY，不是 API_KEY 或其他變形）；或 build command 設定錯誤。
2
驗收測試（25 分鐘）
在部署後的正式網址上執行測試，不是 localhost。用下方 Checklist 完成 10 筆驗收，記錄通過率。
3
更新作品集（15 分鐘）
打開 PRAC5-3 建的作品展示首頁，加入這個專題：名稱 + 一句話說明 + 公開網址 + GitHub 連結。

延伸案例

部署與驗收不只是「推上 Vercel 就好」，還要應付真實用戶的誤用、安全攻擊、交付文件。挑一個貼近你處境的情境完整做一次。每個案例 30–45 分鐘。

案例 01 · 部署前驗收
Vercel 上線前的 12 項驗收清單（含限流與 API key 保護）
難度 ★★☆ 約 40 分鐘

你幫一家 20 人的茶飲加盟總部做了「新店選址評分 AI」，明天要上線給加盟顧問內部用。你很擔心：API key 外洩會被盜刷、沒限流有人連按 50 下、URL 外流到 PTT 被亂玩。上線前要一份 12 項驗收清單，老闆會陪你一起跑一次。

原始資料｜專案技術現況
前端：靜態 HTML 單頁（呼叫 Claude API） 部署：Vercel 免費方案 網域：score.xxxxxx-tea.com（已買好） 使用者：8 位加盟顧問（無需登入，用 iPad） 使用頻率：平均每人每天 5–10 次 API 預算：每月 2,500 元 上限（老闆底線） 既有 issue： - API key 目前還寫在前端 JS - 沒有設任何 rate limit - console 會印出 debug log - 沒有 404 頁面
你的任務
  1. 寫 12 項驗收清單，分 4 類：安全（4）、效能（3）、UX（3）、監控（2）
  2. 每項寫「怎麼驗」——不是「確認 X」，是「打開 Y 看 Z 是否 = W」
  3. 決定 1 項可以「上線後 24h 內補」，9 項必須上線前過，2 項可跳過並解釋
評估指標
API key 不外洩
明確寫「打開 DevTools → Sources → 搜 'sk-' 應找不到」
限流可驗
寫「連按 20 次應看到 429 或友善提示」，不是「有設限流」
成本可控
至少 1 項驗「如果有人濫用會被擋下」，關聯 2500 元預算
取捨合理
跳過的 2 項有充分理由，不是「時間不夠」

案例 02 · 客戶交付 SOP
交付給非技術客戶的驗收 SOP + 3 分鐘教學錄影逐字稿
難度 ★★☆ 約 40 分鐘

你是自由接案者，幫一位牙醫診所老闆做了「AI 預約確認簡訊生成器」，費用 2.5 萬元已收訂金 1 萬。這週要交付，但老闆 55 歲不懂 HTML、看到 code 就頭痛。你要一份 SOP 讓老闆驗收、加一支 3 分鐘錄影讓她能教給櫃檯小姐。

原始資料｜交付情境
專案：預約確認簡訊生成器 使用者：牙醫診所櫃檯（1 位主櫃 + 1 位兼職） 使用流程：輸入病患姓名、約診時間、治療項目 → 生成中文簡訊（含醒酸注意、遲到規則） 網址：sms.xxxxxx-dental.com（已設好） 帳號：無需登入 已付：訂金 1 萬，驗收通過付尾款 1.5 萬 老闆的擔心：「我是不是要每年付你維護費？」「會不會被駭？」 老闆會 LINE 你但不會 GitHub 交付需要： - 驗收書（老闆簽字用） - 使用手冊（1 頁給櫃檯看） - 3 分鐘錄影（老闆能轉給新進櫃檯）
你的任務
  1. 寫驗收 SOP（老闆簽字用）：6 項可勾選的驗收條件，每項 20 字內
  2. 寫 3 分鐘教學錄影逐字稿（約 360 字）：開場 → 實操 → 常見問題
  3. 寫給老闆的 FAQ 3 題：維護費、安全、找人改的方法
評估指標
不用技術詞
不出現 API、HTML、部署、deploy、cache 等詞
可勾選驗收
6 項都是「做了 X → 看到 Y」，老闆可自己驗，不用你在旁邊
錄影可朗讀
逐字稿實際念出來 180–200 秒，留 30 秒緩衝
FAQ 誠實
維護費說清楚「哪些免費、哪些另計」，不含糊

4 種情境，把你專題做成能被看懂的一頁

部署完有網址還不夠。真正的關卡是：訪客停留 3 秒到 3 分鐘之間，你能不能把「這是什麼、誰用、為什麼值得打開」講清楚。下面 4 個情境對應 4 種訪客與 4 種目的。挑貼近你處境的那個，把「頁面結構 Prompt」複製貼進 AI 工具產骨架，再往裡塞真實內容。

情境 A 作品集首頁｜3 個專題 × 3 秒看懂你做什麼
情境 A｜作品集首頁
● 目標受眾：潛在雇主、接案客戶、面試官、社群同業 ● 停留時間：3–15 秒（手機滑到，沒興趣就走） ● 期望行動：點進其中 1 個專題詳細頁，或把網址加到書籤 ● 限制：3 個專題同時出現，不能讓訪客選擇障礙；自我介紹 1 句話，不要寫履歷式自述

## 角色 你是作品集 landing page 的資訊架構師，擅長讓訪客 3 秒看懂製作者在做什麼。 ## 任務 為以下作品集首頁產出完整頁面結構（HTML 區塊順序 + 每區要放什麼內容）。 ## 作品集資料 - 製作者身份：[填你的定位，例：「用 AI 解決中小企業行政工作的獨立接案者」] - 3 個代表作品：[填名稱 + 一句話結果，例：「診所簡訊生成器 — 櫃檯每天省 40 分鐘」] - 聯絡方式：[填 Email / LinkedIn / 可選] ## 輸出要求 列出 6 個區塊的順序、每區的內容重點、每區應佔螢幕多少高度： 1. Hero（占 60% 首屏）：一行定位句 + 一句證明價值的數字 2. 3 個作品卡（並排或上下）：每卡 = 專題名 + 一句話結果 + 代表圖 + 連結 3. 簡短自述（約 80 字）：你用什麼工具、解什麼問題、服務過誰 4. 聯絡 CTA：給動詞，不要「歡迎來信」 5. 頁尾：GitHub / LinkedIn 圖標列 6. 排除項：不放學歷、不放證照清單、不放「關於我」長文 ## 限制 - 訪客滑 3 秒就要看到你是誰、做什麼；滑 10 秒要看到 3 個作品；滑 30 秒要找到聯絡方式 - 3 個作品卡的「一句話結果」必須量化（時間、金額、使用次數），禁止「體驗良好」「介面漂亮」 - 整頁禁止出現「歡迎來到我的作品集」「感謝蒞臨」

## 角色 你是寫給技術社群看的工程 blog / case study 結構師，目標讓讀者學到東西，而不是炫耀。 ## 任務 為以下專題的「開發歷程分享頁」產出區塊順序與每區內容重點。 ## 專題資料 - 專題名稱：[填名稱] - 核心技術棧：[填實際用到的 3–5 個工具 / 框架 / API] - 最關鍵的 1 個技術決策：[例：「為什麼用 edge function 而不是傳統 server」] - 最痛的 1 個踩坑：[例：「Vercel 冷啟動讓 Demo 第一次 timeout」] - 耗時：[從 idea 到 v1 多久] ## 輸出要求 列出 6 個區塊，每區說明內容重點： 1. 一句話專題定位（不是 hero，是技術讀者要看的那種） 2. 架構圖（ASCII 或簡單方塊圖）：資料流 + 哪塊跑在哪 3. 3 個關鍵技術決策：每個用「為什麼選它 / 排除了什麼 / 代價是什麼」三段式 4. 踩坑紀錄：2–3 個具體故事（時間 + 現象 + 排查 + 結論），每則 100–150 字 5. 量化結果：成本、延遲、可用性數字（有就放，沒有就跳過） 6. 下一步與開源連結：GitHub repo + issue 歡迎 PR ## 限制 - 技術決策段禁止只寫「我選了 X」——必須三段式（為什麼、排除了什麼、代價） - 踩坑紀錄禁止事後諸葛口吻（「我一開始就知道 Vercel 會冷啟動」）；要寫現場如何發現、如何排查 - 避免行銷語言（「強大的」「創新的」「無縫整合」），改用具體數字 - 架構圖可以是 ASCII art，不用非得 Figma 畫漂亮
區塊順序 + 每區重點
一句定位 → 架構圖（資料流）→ 3 個技術決策（為什麼/排除什麼/代價）→ 2–3 則踩坑故事（現場感）→ 量化結果 → 開源連結。技術決策與踩坑段要誠實，禁事後諸葛。
情境 D 商業提案版｜for 客戶 / 投資人：規模 + 差異 + 成果
情境 D｜商業提案版
● 目標受眾：潛在客戶、投資人、想找外包的中小企業主 ● 停留時間：2–5 分鐘（會看數字、比較差異） ● 期望行動：預約 30 分鐘通話、下載方案書、直接 LINE 聯絡 ● 限制：第一屏就要有「問題規模有多大」的數字；解法段要明確與市面上現有工具的差異，不能含糊
頁面結構 Prompt
## 角色 你是 B2B 商業提案 landing page 的結構策略師，讀者會用「值不值得預約通話」的角度看。 ## 任務 為以下專題產出商業提案版 landing page 的區塊順序與每區內容重點。 ## 專題資料 - 解決的問題：[填 1 句話] - 目標客戶規模：[填具體企業規模 / 行業，例：「10–50 人牙醫診所」] - 問題的量化規模：[填：問題每年造成多少損失 / 花費多少時間，例：「每位櫃檯每天花 40 分鐘寫簡訊」] - 你的解法差異：[填：與現有 X 工具的差異，例：「比 Y 工具便宜 60%、客製中文提醒語、無需整合」] - 已有成果：[填：試用客戶數、節省時間、成本差、續用率等，有數字就放] ## 輸出要求 列出 6 個區塊： 1. Hero：問題句 + 量化規模（「X 行業每年浪費 Y 在 Z 上」）+ 主 CTA（預約通話） 2. 問題三連擊：3 個具體痛點場景 + 每個附一個數字 3. 解法差異矩陣：3×3 表格（你 vs 工具 A vs 工具 B），欄位：價格 / 客製度 / 整合成本 4. 量化成果：已試用客戶的數字（節省時間、成本、使用頻率），或 pilot 數據 5. 定價 + ROI 計算：月費 + 回收期（例：「月費 3,500，省下 1 位兼職櫃檯半職費用」） 6. CTA + 信任元素：預約通話按鈕 + 公司名 / 聯絡人照片 / 回覆時間承諾 ## 限制 - 第一屏的量化規模必須是客戶能驗證的數字（市場報告 / 行業平均 / 實測），禁止捏造 - 解法差異矩陣要列出「競爭對手」，不能寫「市面上沒有」——那代表你沒做功課 - 定價要給具體數字，禁止「聯絡我們報價」（那會直接流失 70% 訪客） - CTA 文案：「預約 30 分鐘通話（免費）」比「立即聯絡」有效 3 倍
區塊順序 + 每區重點
Hero（問題 + 規模數字 + CTA）→ 3 痛點場景 → 差異矩陣（你 vs 競品）→ 量化成果 → 定價 + ROI 計算 → CTA + 信任元素。定價要有具體數字、差異段要列競品。
CHALLENGE & QUIZ · 延伸挑戰 + 通關檢測

把頁面真的上線，找人測，再自測你抓到要領沒

看完 4 個情境、產完結構還不夠。landing page 的回饋不是「設計得好漂亮」，是「訪客 3 秒後還在、5 秒後點 CTA」。下面 3 個挑戰會逼你真的上線、真的找人測、真的把 SEO / OG 補齊。跑完再做 5 題自測。

●
延伸挑戰（預計 45 分鐘）
真的上線挑情境 A–D 其中一個，把你專題的 landing page 真的做出來並 deploy 到 Vercel / Netlify 拿到公開網址。驗收：手機打開 3 秒內能看到 Hero、5 秒內找到 CTA。不要停在「我草稿寫好了」的階段——沒上線就等於沒做。
5 秒測試把網址丟給 3 位朋友（最好一個技術、一個非技術、一個目標受眾），請他們看 5 秒後關掉，再問：「這是做什麼的？給誰用？你知道下一步要做什麼嗎？」3 個人答案一致才算過關。不一致的地方就是你 Hero 沒寫清楚。
SEO + OG 補齊加上 meta description（80–120 字）、og:title、og:description、og:image（1200×630）。用 metatags.io 或 opengraph.xyz 預覽實際分享到 LINE / Facebook / Twitter 的樣子。驗收：貼到 LINE 有縮圖、有標題、有描述，不是光禿禿的網址。
1landing page 第一屏（Hero）最該放什麼？
看答案
C。第一屏的任務是讓訪客在 3 秒內回答「這解決我什麼問題」。A/B/D 都把訪客擺第二位，只有 C 直接對準訪客的痛點與行動。「問題 + 解法 + CTA」是 landing page 黃金結構。
2技術棧（用了什麼框架、API）該放在哪裡？
看答案
C。一般訪客不在乎你用什麼技術，只在乎能不能解決問題。放頁尾是給技術讀者看的彩蛋；放第一屏反而讓非技術訪客感覺「這東西跟我沒關係」。例外：情境 C（開發歷程分享）給技術社群看時才上頁面。
3產品 Demo 影片該多長？
看答案
B。landing page 的 Demo 影片要在 1 分鐘內讓訪客看到核心功能。太長（A）訪客滑走；太短（C/D）看不出價值。30–90 秒且前 10 秒就切入產品操作，是被測過最穩的區間。完整介紹留給產品頁或 YouTube。
4CTA 按鈕文案該怎麼寫？
看答案
C。好的 CTA 回答訪客心裡的「點下去我要付出什麼」。動詞開頭（「預約」「試用」「下載」）+ 明確成本（時間、金額、步驟數）能降低點擊門檻。A/B/D 全都沒說「點下去會發生什麼」，訪客會因不確定而不點。
5沒有 budget 做設計、自己又不會美術時怎麼辦？
看答案
B。沒設計資源時，極簡排版反而最安全——乾淨的字體層級 + 1 個強視覺焦點（真實產品截圖最有效）勝過花俏但不協調的拼湊。A 會讓頁面雜亂、C 讓訪客分心離題、D 有法律與品牌識別雙重風險。
