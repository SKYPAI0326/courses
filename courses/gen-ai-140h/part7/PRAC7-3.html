PART 7 · 專題衝刺與成果發表
PRAC 7.3 · 3 小時

開發衝刺第二輪

依改進清單執行 v1：UX 修好、錯誤處理補齊、成本控管加上。v1 是你準備部署的版本。

任務目標
從 PRAC7-2 的改進清單中，選出「高優先」項目完成。這一輪重點是三個面向：UX 問題修正、錯誤處理補齊、成本控管。結束前確認 v1 可以準備部署。
1
從改進清單選出 3–5 個高優先項目（15 分鐘）
打開 PRAC7-2 的改進清單，選 3–5 個在這 3 小時內可以做完的項目。不要試圖全部做完，這一輪目標是讓工具「夠好」，不是「完美」。
時間分配建議：UX 修正 90 分鐘 → 錯誤處理 45 分鐘 → 成本控管 30 分鐘 → 最終測試 15 分鐘
2
UX 修正（90 分鐘）
依下方三個方向，修正你的工具中最影響使用體驗的問題。每個改動都要在瀏覽器中確認。
3
錯誤處理補齊（45 分鐘）
把所有可能崩潰的地方加上防護。目標：不管使用者怎麼操作，頁面都不應該顯示技術錯誤訊息或 undefined。
4
成本控管確認（30 分鐘）
確認 max_tokens 設定合理、API 後台有預算上限、輸入驗證阻擋空白和超長輸入。
5
最終測試 + commit（15 分鐘）
跑完 PRAC7-1 設計的 10 個測試案例，全部通過才算 v1 完成。git commit -m "v1 ready for deployment"
UX 修正三個優先方向
  • Loading 狀態：點擊送出後按鈕改為「處理中…」並禁用，收到回應後恢復
  • 複製功能：輸出區旁加「複製全文」按鈕（navigator.clipboard.writeText()）
  • 說明文字：input placeholder 和頁面標題要讓第一次看到的人秒懂這個工具做什麼
錯誤處理必須覆蓋的情況
  • 空白輸入：if (!input.trim()) { showError('請輸入內容'); return; }
  • API 錯誤轉友善訊息：429 → 「請稍後再試」、500 → 「系統暫時忙碌」
  • 輸出安全取值：response?.content?.[0]?.text ?? '無法取得回應'
  • 網路離線偵測：if (!navigator.onLine) { showError('請確認網路連線'); return; }
成本控管部署前必確認
  • 輸入上限：超過 3000 字截斷並提示（避免 token 暴衝）
  • max_tokens 設合理值（創意類 800 / 分析類 500 / 摘要類 300）
  • API 後台設每月預算上限（Anthropic Console → Billing → Usage Limits）
完成進度：0 / 5
第二輪衝刺驗收

延伸案例

第二輪衝刺最關鍵的是「取捨」——哪些要做、哪些要放掉、bug 要補到多嚴。挑一個貼近你處境的情境，把優先級與回歸測試設計完整做一次。每個案例 30–45 分鐘。

案例 01 · 技術債排序
從 8 個 issue 決定做哪 6 個、cut 哪 2 個
難度 ★★★ 約 45 分鐘

你是客服組長，兼差做一個「工單分類 AI 助手」給自家組員用。第一輪衝刺剩 6 個工作天，但同學 demo 後回報的 issue 有 8 個。你要決定 6 個做、2 個砍，並寫理由給指導老師。

原始資料｜8 個 issue 清單
#1 貼長工單（>2000 字）會逾時，前端沒有錯誤提示 #2 AI 分類結果常把「退款申請」誤判成「查詢進度」 #3 手機版排版跑掉，按鈕重疊 #4 沒有歷史記錄，組員每次要重貼 #5 API key 寫死在 client JS，同學說有安全風險 #6 載入時白屏 3 秒沒 loading indicator #7 想加「一鍵轉 Zendesk」按鈕（老闆喜歡） #8 Prompt 沒有 few-shot 範例，新類別辨識率低 你的條件：剩 6 天，每天可工作 3 小時，總共 18 小時 指導老師要求：必須寫「做與不做」的理由，不能只列清單
你的任務
  1. 為每個 issue 估工時（0.5h / 1h / 2h / 4h 四擋），總和應超過 18h
  2. 用「嚴重度 × 使用頻率 × 修補成本」排序，決定 6 做 2 砍
  3. 寫 2 個被砍項的理由各 30 字，能直接貼給指導老師
評估指標
時間吻合
6 個做的項目工時加總 ≤18h，含 20% buffer
安全優先
#5（API key 安全）不能被 cut，就算老師沒特別叮嚀
砍得乾脆
被砍的理由要寫「為什麼後期再補可接受」，不能只寫「時間不夠」
使用者視角
排序有考慮「組員天天遇到的」vs「偶爾才發生的」
參考決策（建議先自己排再看）
點擊展開參考答案
【工時估算】 #1 長工單逾時：2h（前端加字數檢查 + 切分邏輯） #2 退款誤判：4h（加 few-shot + 2 輪評測） #3 手機排版：2h #4 歷史記錄：4h（需 localStorage + UI） #5 API key 安全：2h（移到環境變數 + 後端代理） #6 Loading indicator：0.5h #7 一鍵轉 Zendesk：4h（API 整合） #8 Prompt few-shot：合併到 #2 做（算 0h） 總和：18.5h 【做的 6 項（優先順序）】 #5 → #2+#8 → #1 → #6 → #3 → #4 安全 > 核心準確率 > 穩定性 > UX 小坑 > 手機相容 > 便利 【砍的 2 項 + 理由】 #7 一鍵轉 Zendesk（cut 理由） 4 小時工時大，且老闆的「喜歡」不代表組員日常會用；中期 demo 先用手動複製貼上也能過關，第二期再做。 （#8 沒列獨立工時，已合併入 #2） 實際上被正式 cut 的第 2 項： #3 手機排版（cut 理由） 問過 3 位組員都在桌機操作，手機使用率 <5%。先加一行提示文字「建議桌機使用」，2 週後有真實需求再排工。

案例 02 · 回歸測試
bug 修完後的 10 題回歸測試 checklist
難度 ★★☆ 約 35 分鐘

你是獨立開發者，做了一個「發票條碼 OCR + 自動記帳」網頁工具給小吃店老闆用。上週改了 3 個 bug（上傳大圖 crash、金額抓錯小數點、重複發票沒檢測），今天要交付。你怕這次改又壞了別的地方，想寫一份 10 題的回歸測試 checklist，15 分鐘跑完。

原始資料｜已知修過的 bug 與主要功能
已修 3 個 bug： B1 上傳 >5MB 圖片直接 crash（已加壓縮） B2 金額「NT$1,234.50」被抓成 123450（小數點錯位） B3 同一張發票貼兩次沒提示（已加 hash 檢查） 主要功能（務必不能壞）： F1 拍照 → 自動辨識金額 + 日期 + 店名 F2 手動修正辨識錯的欄位 F3 存入瀏覽器 localStorage F4 匯出 CSV 給會計 F5 統計本月支出 測試環境： - Chrome 桌機（老闆用的） - iOS Safari（老闆老婆用的） - 測試發票：15 張（含 3 張破損、2 張手寫）
你的任務
  1. 寫 10 題測試 checklist：3 題針對修過的 bug、5 題針對主功能、2 題邊界情況
  2. 每題寫「怎麼做」+「預期結果」，可以讓非工程師（你老婆、朋友）照著跑
  3. 標示 3 題必過紅線（失敗就不能交付）
評估指標
可執行
每題有明確步驟與預期結果，不是「檢查 X 是否正常」
覆蓋修過的
3 個 bug 都有對應的驗證題
覆蓋沒改的
5 個主功能都有快速驗證，避免「改 A 壞 B」
紅線清楚
3 題紅線標示明顯，失敗立即 block 交付

四個情境 × 迭代日誌：把每次改動都版本化

v1 準備部署不是終點，是迭代日誌的起點。沒寫日誌的優化，下一週就忘了「為什麼那樣改」；寫了日誌的優化，半年後還能回頭找根因。下面四個情境各自對應一種迭代面向：Prompt、UI、成本、功能。挑最像你處境的一組，把「日誌模板 Prompt」貼進 AI 裡，讓它幫你把第一版日誌寫出來——你只要填欄位值。建議每組 15 分鐘。

情境 A · Prompt 迭代日誌 v1 → v2 → v3 每版改什麼 + Eval 分數變化｜約 12 分鐘
日誌模板欄位 · 紀錄頻率 · 受眾情境 A
【日誌模板欄位】 - 版本號（v1 / v2 / v3...） - 日期 - 改了什麼（prompt diff，新增/刪除/改寫哪段） - 為什麼改（觀察到什麼 bug 或客訴） - Eval 分數（黃金集通過率 before → after） - 失敗案例樣本 1–2 筆（連結或原文） - 下一版預計動什麼 【紀錄頻率】每次 prompt 真的進 main 分支就寫一條；實驗分支不進日誌，避免雜訊 【受眾】自己 + 未來接手的人 + 3 個月後忘記細節的自己
日誌 Prompt · 讓 AI 幫你把第一版寫出來
你是一位熟悉 LLM 應用品質工程的產品工程師。請根據我提供的「這一輪 prompt 改動」與「前後 eval 分數」，幫我產出一條迭代日誌。 輸出規格： 1. 單條日誌 Markdown 格式，標題用 `## v{版本號} · {一句話摘要}` 2. 正文依序有六個欄位：改了什麼 / 為什麼改 / Eval 分數變化 / 失敗案例前 2 筆 / 這次學到什麼 / 下一版預計動什麼 3. 「改了什麼」欄位要寫得像 git diff 摘要，讓人 30 秒內看懂動了哪段 4. 「Eval 分數變化」要同時列 before 與 after，並標出漲跌幅（+X% / -X%） 5. 「這次學到什麼」限 1–2 句，可以被未來的自己 3 個月後看懂 6. 禁用模型版本號具體字串；統一用「你」不用敬稱；全文繁體中文台灣用語 7. 直接輸出 Markdown，不要加前言或解釋 請在下一則訊息中貼上你這輪的改動摘要與 eval 數字，我再產出日誌。

- 下一步：對剩下 30% 推理型任務另開分支保留進階模型

情境 D · 功能上線追蹤 每週 retrospective：上線了什麼、學到什麼、下週改什麼｜約 10 分鐘
日誌模板欄位 · 紀錄頻率 · 受眾情境 D
【日誌模板欄位】 - 週次（W12 / W13...）+ 日期區間 - 本週上線了什麼（3 項以內，每項 1 行） - 本週沒做到什麼（2 項以內） - 數據亮點（2 個最有感的指標） - 使用者意外回饋（沒預期到的 1–2 則） - 本週最大一課（一句話） - 下週要動什麼（3 項以內，分「做」「停」「繼續觀察」） 【紀錄頻率】每週五下班前 20 分鐘寫完，週一同步給團隊 【受眾】自己 + 團隊 + 半年後回顧年度成果的自己
日誌 Prompt · 每週 retrospective 產生器
你是一位熟悉敏捷迭代的資深產品工程師。請根據我提供的「這一週發生的事」，幫我產出一條週報式的迭代日誌。 輸出規格： 1. 單條日誌 Markdown 格式，標題用 `## W{週次} · {一句話本週主題}` 2. 正文依序七欄位：本週上線（最多 3 項，每項 1 行） / 本週未達（最多 2 項） / 數據亮點（2 個指標 before → after） / 意外回饋（1–2 則原話） / 本週最大一課（一句話） / 下週「做」（最多 3 項） / 下週「停 / 繼續觀察」 3. 「本週最大一課」必須是可以直接貼到備忘錄的一句，不是空話（如「測試要在 demo 前跑完」而非「要多測試」） 4. 「下週要動什麼」三區塊清楚分開：做 / 停 / 繼續觀察 5. 統一用「你」不用敬稱；全文繁體中文台灣用語；不出現客戶或模型具體名稱 6. 若本週有事故或回退，必須在「本週最大一課」或「停」裡寫出來 7. 直接輸出 Markdown，不要前言 請在下一則訊息給我本週原始資料：上線項、未達項、兩個指標變化、1–2 則使用者回饋。

(08) · Extension Challenges

延伸挑戰｜把日誌變成你的第二大腦

日誌的價值不是「寫一次」，是「寫到變成肌肉記憶」。下面三個挑戰從補寫歷史、記錄失敗、到月度回顧，層層往外推。每個挑戰做完會累積一份可以直接交接給隊友、或半年後貼進年度簡報的紀錄。

●
先熱身：把你目前手上最活的一個 side project 或工作工具叫出來，問自己「過去三週我動了哪些地方？」——如果你答不出來，就從挑戰 1 開始補；如果能答出來但沒寫下來，就從挑戰 2 開始寫。
🎯 挑戰 1 · 補 3 週迭代日誌約 45 分鐘
幫你目前專案把過去 3 週的改動補齊。打開 git log、翻 Slack 歷史、問隊友 3 件事：這週改了什麼、為什麼改、結果怎樣。用情境 A 的「日誌 Prompt」跑一次，產出 3 條日誌合併成 CHANGELOG.md。重點不是寫得完美，是把「記憶還在的時候」搶救回來。
🎯 挑戰 2 · 寫一份「失敗的嘗試」紀錄約 30 分鐘
挑一個你最近一個月「試了但沒效果、最後回退」的改動——可能是換了某個策略結果品質掉、也可能是加了某個功能但沒人用。用情境 C 的「日誌 Prompt」格式，老實寫下：做了什麼、成本、品質、為什麼 drop、根因觀察。失敗的紀錄比成功的更珍貴，因為半年後你會想再試一次類似的方向，這份紀錄會擋下重複踩雷。
🎯 挑戰 3 · 月度回顧簡報約 60 分鐘
把你這一個月的週報（情境 D）整合成一份月度 retrospective：四週一頁、指標走勢一張圖、最大三課、下月三個押寶方向。讓 AI 讀你四週的日誌，產出月報初稿；你只負責改措辭、加圖表。做完直接貼進月會簡報，或當作個人作品集的素材。這一步會把你從「每週交差」拉升到「每月有累積」。
(09) · Quiz · 5 題

迭代日誌檢核｜看你懂不懂日誌該寫什麼

情境跑完＋三關挑戰之後，用這 5 題自評——每題 1–2 分鐘，全部做完約 8 分鐘。重點不是「答對幾題」，而是看到題目能不能立刻判斷「這份日誌是不是有效的」。做完再展開解析對照。

1一條有效的迭代日誌，最不能缺的三個欄位是？
展開解析
正解 B。日誌的用途是未來的自己能回看「動了哪、為什麼動、有沒有用」，這三欄缺一不可。作者簽核是流程附件，截圖和工時是加分項，都不是核心。
2你試了一個新策略，結果品質沒升甚至掉了 3pp，最後回退沒合併。這條紀錄要不要進日誌？
展開解析
正解 C。半年後你或隊友很可能想再試一次類似方向，這條失敗紀錄就是那時候的護欄。只寫標題等於沒寫，因為根因就藏在細節裡。
3你一天之內改了 5 個地方同時上線，兩天後指標有漲。這個節奏最大的問題是？
展開解析
正解 B。迭代節奏太快、同時動多項的壞處是「歸因失敗」。下次要複製這次的成功時，會發現自己不知道該複製哪一塊，日誌會變成一堆結果但看不出因果。
4指標選錯（例如只看單次轉換率、不看留存）的代價，下列哪一個最關鍵？
展開解析
正解 D。錯的指標會讓你把精力灌在一個跟真正業務結果脫鉤的目標上；更糟的是設計可被 gaming 的指標（例如只看點擊數），最後被使用者或自己的 bot 刷爆卻沒有實際價值。選對指標比跑得快重要。
5你寫的迭代日誌，最核心的讀者是誰？
展開解析
正解 C。日誌的受眾是「三層人」：現在的自己（確認思路）、現在的團隊（同步決策）、未來的自己（回溯根因）。只寫給老闆會變成績效秀，只寫給自己容易寫得跳步，把「未來的自己」當主讀者，內容密度最剛好。
