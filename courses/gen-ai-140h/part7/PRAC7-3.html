PART 7 · 專題衝刺與成果發表
PRAC 7.3 · 3 小時

開發衝刺第二輪

依改進清單執行 v1：UX 修好、錯誤處理補齊、成本控管加上。v1 是你準備部署的版本。

任務目標
從 PRAC7-2 的改進清單中，選出「高優先」項目完成。這一輪重點是三個面向：UX 問題修正、錯誤處理補齊、成本控管。結束前確認 v1 可以準備部署。
1
從改進清單選出 3–5 個高優先項目（15 分鐘）
打開 PRAC7-2 的改進清單，選 3–5 個在這 3 小時內可以做完的項目。不要試圖全部做完，這一輪目標是讓工具「夠好」，不是「完美」。
時間分配建議：UX 修正 90 分鐘 → 錯誤處理 45 分鐘 → 成本控管 30 分鐘 → 最終測試 15 分鐘
2
UX 修正（90 分鐘）
依下方三個方向，修正你的工具中最影響使用體驗的問題。每個改動都要在瀏覽器中確認。
3
錯誤處理補齊（45 分鐘）
把所有可能崩潰的地方加上防護。目標：不管使用者怎麼操作，頁面都不應該顯示技術錯誤訊息或 undefined。
4
成本控管確認（30 分鐘）
確認 max_tokens 設定合理、API 後台有預算上限、輸入驗證阻擋空白和超長輸入。
5
最終測試 + commit（15 分鐘）
跑完 PRAC7-1 設計的 10 個測試案例，全部通過才算 v1 完成。git commit -m "v1 ready for deployment"
UX 修正三個優先方向
  • Loading 狀態：點擊送出後按鈕改為「處理中…」並禁用，收到回應後恢復
  • 複製功能：輸出區旁加「複製全文」按鈕（navigator.clipboard.writeText()）
  • 說明文字：input placeholder 和頁面標題要讓第一次看到的人秒懂這個工具做什麼
錯誤處理必須覆蓋的情況
  • 空白輸入：if (!input.trim()) { showError('請輸入內容'); return; }
  • API 錯誤轉友善訊息：429 → 「請稍後再試」、500 → 「系統暫時忙碌」
  • 輸出安全取值：response?.content?.[0]?.text ?? '無法取得回應'
  • 網路離線偵測：if (!navigator.onLine) { showError('請確認網路連線'); return; }
成本控管部署前必確認
  • 輸入上限：超過 3000 字截斷並提示（避免 token 暴衝）
  • max_tokens 設合理值（創意類 800 / 分析類 500 / 摘要類 300）
  • API 後台設每月預算上限（Anthropic Console → Billing → Usage Limits）
完成進度：0 / 5
第二輪衝刺驗收

延伸案例

第二輪衝刺最關鍵的是「取捨」——哪些要做、哪些要放掉、bug 要補到多嚴。挑一個貼近你處境的情境，把優先級與回歸測試設計完整做一次。每個案例 30–45 分鐘。

案例 01 · 技術債排序
從 8 個 issue 決定做哪 6 個、cut 哪 2 個
難度 ★★★ 約 45 分鐘

你是客服組長，兼差做一個「工單分類 AI 助手」給自家組員用。第一輪衝刺剩 6 個工作天，但同學 demo 後回報的 issue 有 8 個。你要決定 6 個做、2 個砍，並寫理由給指導老師。

原始資料｜8 個 issue 清單
#1 貼長工單（>2000 字）會逾時，前端沒有錯誤提示 #2 AI 分類結果常把「退款申請」誤判成「查詢進度」 #3 手機版排版跑掉，按鈕重疊 #4 沒有歷史記錄，組員每次要重貼 #5 API key 寫死在 client JS，同學說有安全風險 #6 載入時白屏 3 秒沒 loading indicator #7 想加「一鍵轉 Zendesk」按鈕（老闆喜歡） #8 Prompt 沒有 few-shot 範例，新類別辨識率低 你的條件：剩 6 天，每天可工作 3 小時，總共 18 小時 指導老師要求：必須寫「做與不做」的理由，不能只列清單
你的任務
  1. 為每個 issue 估工時（0.5h / 1h / 2h / 4h 四擋），總和應超過 18h
  2. 用「嚴重度 × 使用頻率 × 修補成本」排序，決定 6 做 2 砍
  3. 寫 2 個被砍項的理由各 30 字，能直接貼給指導老師
評估指標
時間吻合
6 個做的項目工時加總 ≤18h，含 20% buffer
安全優先
#5（API key 安全）不能被 cut，就算老師沒特別叮嚀
砍得乾脆
被砍的理由要寫「為什麼後期再補可接受」，不能只寫「時間不夠」
使用者視角
排序有考慮「組員天天遇到的」vs「偶爾才發生的」
參考決策（建議先自己排再看）
點擊展開參考答案
【工時估算】 #1 長工單逾時：2h（前端加字數檢查 + 切分邏輯） #2 退款誤判：4h（加 few-shot + 2 輪評測） #3 手機排版：2h #4 歷史記錄：4h（需 localStorage + UI） #5 API key 安全：2h（移到環境變數 + 後端代理） #6 Loading indicator：0.5h #7 一鍵轉 Zendesk：4h（API 整合） #8 Prompt few-shot：合併到 #2 做（算 0h） 總和：18.5h 【做的 6 項（優先順序）】 #5 → #2+#8 → #1 → #6 → #3 → #4 安全 > 核心準確率 > 穩定性 > UX 小坑 > 手機相容 > 便利 【砍的 2 項 + 理由】 #7 一鍵轉 Zendesk（cut 理由） 4 小時工時大，且老闆的「喜歡」不代表組員日常會用；中期 demo 先用手動複製貼上也能過關，第二期再做。 （#8 沒列獨立工時，已合併入 #2） 實際上被正式 cut 的第 2 項： #3 手機排版（cut 理由） 問過 3 位組員都在桌機操作，手機使用率 <5%。先加一行提示文字「建議桌機使用」，2 週後有真實需求再排工。
變體挑戰
A老闆突然加塞 #7（一鍵 Zendesk）必做——改寫排序，告訴老闆哪一項要順延
B時間從 18h 縮到 12h（要出差 2 天），重排一次
C把決策寫成 Slack 訊息格式（<200 字）發給老師，最多 3 段

案例 02 · 回歸測試
bug 修完後的 10 題回歸測試 checklist
難度 ★★☆ 約 35 分鐘

你是獨立開發者，做了一個「發票條碼 OCR + 自動記帳」網頁工具給小吃店老闆用。上週改了 3 個 bug（上傳大圖 crash、金額抓錯小數點、重複發票沒檢測），今天要交付。你怕這次改又壞了別的地方，想寫一份 10 題的回歸測試 checklist，15 分鐘跑完。

原始資料｜已知修過的 bug 與主要功能
已修 3 個 bug： B1 上傳 >5MB 圖片直接 crash（已加壓縮） B2 金額「NT$1,234.50」被抓成 123450（小數點錯位） B3 同一張發票貼兩次沒提示（已加 hash 檢查） 主要功能（務必不能壞）： F1 拍照 → 自動辨識金額 + 日期 + 店名 F2 手動修正辨識錯的欄位 F3 存入瀏覽器 localStorage F4 匯出 CSV 給會計 F5 統計本月支出 測試環境： - Chrome 桌機（老闆用的） - iOS Safari（老闆老婆用的） - 測試發票：15 張（含 3 張破損、2 張手寫）
你的任務
  1. 寫 10 題測試 checklist：3 題針對修過的 bug、5 題針對主功能、2 題邊界情況
  2. 每題寫「怎麼做」+「預期結果」，可以讓非工程師（你老婆、朋友）照著跑
  3. 標示 3 題必過紅線（失敗就不能交付）
評估指標
可執行
每題有明確步驟與預期結果，不是「檢查 X 是否正常」
覆蓋修過的
3 個 bug 都有對應的驗證題
覆蓋沒改的
5 個主功能都有快速驗證，避免「改 A 壞 B」
紅線清楚
3 題紅線標示明顯，失敗立即 block 交付
參考 Checklist（建議先自己寫再看）
點擊展開參考答案
【修過的 Bug 驗證（B1–B3）】 1. [紅線] 上傳一張 8MB 的 iPhone 照片 → 預期：不 crash，頁面顯示「壓縮中」再辨識成功 2. [紅線] 辨識一張含「NT$1,234.50」的發票 → 預期：金額欄顯示 1234.5 或 1234.50，不是 123450 3. 同一張發票連續貼兩次 → 預期：第二次跳提示「此發票已存過」 【主功能 F1–F5】 4. [紅線] 拍一張標準超商小白單 → 預期：金額/日期/店名三欄至少 2 欄正確 5. 辨識後手動改金額 → 預期：改完存下去，統計數字同步更新 6. 連續存 5 張發票 → 預期：重開瀏覽器後 5 張都還在（localStorage） 7. 按「匯出 CSV」→ 預期：下載 csv 檔，Excel 打開中文不亂碼 8. 看「本月統計」→ 預期：顯示筆數與總金額，與手動加總吻合 【邊界情況】 9. 上傳一張完全模糊的圖 → 預期：不 crash，提示「辨識失敗請重拍」 10. iOS Safari 跑測試 4–8 → 預期：5 題全部通過 【交付規則】 紅線 3 題（#1 #2 #4）任一題失敗 = 不交付、當天修完再跑。 其他 7 題允許 1 題失敗但需在交付 email 註明「已知問題 + 修補時程」。
變體挑戰
A把 10 題壓縮到 5 題「5 分鐘冒煙測試」版，用在每次改 code 後快速跑
B某題失敗時，寫「診斷流程」3 步（看什麼 log、問老闆什麼、該回滾哪段）
C改成讓老闆自己跑的版本，每題用白話改寫（不用技術詞）
