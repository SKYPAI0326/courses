最終發表

Demo Day 工具包：分段計時器 + 12 題 QA 演練 + 評審回饋紀錄 + 3 個月 Roadmap + 一鍵匯出結業包，把 140 小時整理成可帶走的證據。

任務目標
用下方 Demo Day 工具包，把腳本寫好、計時器跑一輪、QA 演練 5 題、Demo 完記錄評審回饋、寫 Roadmap，最後一鍵匯出結業包 Markdown，140 小時收成可帶走的證據。
1
撰寫 Demo 腳本（10 分鐘）
用 M1 工具填 7 欄專題資訊，產出 5 段式 Demo 腳本（問題 / 方案 / Demo / 限制 / 收尾）。資料自動存到 localStorage。
2
計時器演練 + QA 練習（15 分鐘）
用 M2 分段計時器跑一次 5 分鐘流程（含警示），再用 M3 隨機抽 5 題 QA 演練。標記「答得順 / 卡住 / 最怕」，找出要補的洞。
最該練的不是「我會講什麼」，而是「我答不出來什麼」。卡住的題目就是你下一個專題要補的洞。
3
5 分鐘正式 Demo + 評審回饋（15 分鐘）
上場 5 分鐘按照腳本講完。Demo 結束後用 M4 記下每位評審的亮點 / 疑慮 / 建議 / 可引用金句。
4
寫 3 個月 Roadmap（10 分鐘）
用 M5 產 Roadmap Prompt 給外部 AI 跑，結果貼回 M6 三段時間軸（0-2 週 / 1 個月 / 3 個月）。
5
匯出結業包（5 分鐘）
M7 一鍵把全部內容匯出成 Markdown，下載或複製到 Notion / Google Doc。140 小時的證據就在這份檔裡——慶祝一下，這不是小事。

最終發表是把 140 小時累積變成「對方願意買單」的臨門一腳。挑一個貼近你轉職處境的情境完整做一次。每個案例 30–45 分鐘。

案例 01 · QA 題庫
10 題 QA 題庫應對技術評審的刁鑽提問
難度 ★★★ 約 45 分鐘

你是轉職中的上班族，把「合約條款風險檢測 AI」做為發表專題。下週 demo day 有 3 位評審：一位資深工程師、一位法律顧問、一位投資人。你很怕被問到答不出來。要一份 10 題 QA 題庫：答案簡短、誠實、能當場念。

原始資料｜專題背景
專題：ContractScan — 貼合約條款 → AI 標出風險段落 + 建議 技術：前端 + Claude Sonnet API + RAG（嵌入 50 份公開合約範本） 現況： - 準確率自測 78%（20 份測試合約） - 每份分析成本約 3 元台幣 - 尚未接真實客戶，只有 5 位律師朋友試用回饋 - 程式碼放 GitHub Public 你的真實處境： - 不是法律背景、不是工程師科班 - 這是你第一個完整專題 - 目標：拿這個專題去應徵 legal tech 新創的產品職
你的任務
  1. 預想 10 個評審可能問的刁鑽問題，涵蓋技術（4）、商業（3）、法律（2）、個人（1）
  2. 每題寫答案（≤60 字），不懂的直接寫「我不確定，但⋯」的版本
  3. 標出 3 題「最怕被問」——那些答案裡最可能翻車的題
評估指標
答得出口
每題都能念出來 ≤60 字，不是背了很長但 demo 當場忘
誠實優先
不懂的題明確寫「不確定」，不唬爛
涵蓋翻車點
準確率、責任歸屬、法律風險、為什麼不是你這行專家這 4 題有準備
引導下一步
至少 2 題答完主動帶到「下一步我想做 X」
參考題庫（建議先自己寫再看）
點擊展開參考答案
【技術類】 Q1（工程師）：78% 準確率，剩下 22% 是什麼錯法？ A：主要漏掉罕見條款例如競業禁止、境外管轄。錯誤多是 false negative（沒抓到），很少誤報。下一步想加這兩類的訓練樣本。 Q2（工程師）：為什麼用 Sonnet 不用 Haiku？成本差 10 倍。 A：Haiku 跑過一輪準確率掉到 61%。目前捨不得降，但進 100 份文件後會跑 AB 測再決定。 Q3（工程師）：RAG 50 份範本夠嗎？某些產業（醫療、國防）怎麼辦？ A：不夠。現在只涵蓋一般商務合約。醫療國防要另外準備 domain-specific 語料，我還沒做。 Q4（工程師）：Prompt injection 怎麼防？ A：[最怕被問] 老實說這塊我還沒深入做。目前只有基本的 system prompt 隔離，進入生產前要補 input sanitization。 【商業類】 Q5（投資人）：誰會付錢？付多少？ A：假設付費者是中小企業主，每月 500–1500 元訂閱。目前沒有真實客戶，5 位律師朋友願意月付 800。 Q6（投資人）：既有競品例如 LawGeex、Ironclad，你憑什麼？ A：它們是給大型法務部門，起訂價數萬美元/年。我瞄準的是沒法務的中小企業，市場是錯開的。 Q7（投資人）：GitHub public，別人抄怎麼辦？ A：程式碼不是壁壘，真正的壁壘是台灣繁中合約語料庫，這我沒公開。 【法律類】 Q8（法律顧問）：AI 給的「風險建議」若誤導客戶，責任歸誰？ A：[最怕被問] 目前產品會顯示「本工具僅供參考，不構成法律建議」。正式商用前一定要加免責同意頁與律師合作審稿機制。 Q9（法律顧問）：合約內容屬敏感資料，送 Anthropic API 合規嗎？ A：Anthropic enterprise 有 zero-retention 選項，個資部分目前會先做 PII masking。不過這塊我還需要跟律師一起確認各產業合規需求。 【個人類】 Q10（全體）：你非法律、非工程科班，憑什麼做這個？ A：[最怕被問] 我的切入點不是專業深度，是「非專業者看合約時最困惑什麼」。我採訪 12 位被合約坑過的朋友，那份訪談 insight 在 GitHub README 可看。

案例 02 · 作品集卡片
把 3 個 AI 專案寫成 portfolio 網站的 3 張卡片
難度 ★★☆ 約 40 分鐘

你是 pivot 中的創業者，要把 140 小時課程做的 3 個專題整理成一頁式 portfolio（自家網域）。三張卡片各 80 字左右，目標是讓來訪的潛在客戶 10 秒內決定要不要點進去看 demo。

原始資料｜3 個專題
P1：ReplyDraft（Part 4 做的） - 客服回覆草稿生成 - 靜態前端 + Claude Haiku - 朋友公司內部用 3 週，回覆時間 8→2 分鐘 P2：WeeklyMetrics（Part 5 做的） - 廣告週報 AI 生成 - 貼 CSV → 生成中文週報 - 自己用 2 個月，省下每週 4 小時 P3：ContractScan（Part 7 最終專題） - 合約風險掃描 - RAG + Claude Sonnet - 5 位律師朋友測試，自評準確率 78% 訪客是誰： - 主要：找 AI 接案夥伴的中小企業主 - 次要：徵 legal tech 產品職的面試官 - 背景：大多非技術，看程式碼會離開
你的任務
  1. 寫 3 張卡片，每張結構：標題（<12 字）+ 一句話說明 + 3 個數字亮點 + 一句學到什麼
  2. 整頁最上面 30 字的自我介紹：你是誰 + 接什麼案 + 為什麼找你
  3. 決定卡片排序並寫理由（不是時間順序，是「給訪客看的順序」）
評估指標
10 秒決定
每張卡片訪客瞄一眼能抓到「解決什麼問題 + 成效」
有數字
每張都有至少 2 個具體數字，不是「大幅提升」這種
不堆技術
Claude、RAG、API 等詞可以出現但不是主角
排序有理由
解釋為什麼第 1 張放這個而不是另一個
參考內容（建議先自己寫再看）
點擊展開參考答案
【自我介紹 · 30 字】 我做讓中小企業「能立刻用」的 AI 工具。不賣課、不賣夢，只接兩週內能上線的專案。 【卡片 1 · ContractScan】 標題：合約風險掃描 · ContractScan 一句話：把 30 頁合約變成 5 個該注意的點，給沒法務的中小企業主。 數字亮點： - 50 份範本 RAG 訓練，準確率 78% - 每份分析成本 3 元 - 5 位律師試用回饋已整理公開 我學到：AI 的價值不是比專家懂法律，是讓非專家先篩出「要不要找專家」。 【卡片 2 · ReplyDraft】 標題：客服草稿生成 · ReplyDraft 一句話：客服每張工單回覆從 8 分鐘降到 2 分鐘，公司不用換系統。 數字亮點： - 朋友公司實測 3 週、200+ 工單 - 平均回覆時間降 75% - 月 API 成本 1,200 元 我學到：老闆願意買單的不是「AI 超強」，是「下週一就能讓人變輕鬆」。 【卡片 3 · WeeklyMetrics】 標題：廣告週報產生器 · WeeklyMetrics 一句話：貼 CSV 進去，生出 800 字中文週報，我自己每週省下 4 小時。 數字亮點： - 自用 8 週、從沒出過數字錯誤 - 每份報告 token 成本 0.5 元 - 改寫語氣 3 種模式（正式/內部/老闆愛看） 我學到：AI 寫作產品能不能用，不看模型多大，看 prompt 對「不准編數字」寫得多兇。 【排序理由】 ContractScan 放第 1 張：是最新專題、技術最複雜、最能代表我現在能做到哪。 ReplyDraft 放第 2 張：有「真實別人公司」的數字，對潛在客戶可信度最高。 WeeklyMetrics 放第 3 張：自用專案，用來證明我會用自己做的東西，不只寫給別人。

Part 7 · 通關檢測
專題衝刺與成果發表 — 課程終關

走到這裡代表 4 章＋5 實作都跑完了。這份檢測把 Pitch、MVP、架構、測試、Demo、作品集六塊串起來驗收。先自己作答，再展開解析。最後的「140 小時綜合任務」是全課程最硬的一題，45 分鐘一次把四個交付做完，真正把學過的變成能端出去的成品。估時 60–75 分鐘。

觀念題｜10 題
190 秒 pitch 的前 15 秒最該做什麼？
展開解析
正解 B。痛點先行——前 15 秒沒勾到聽者的 KPI／處境，剩下 75 秒都是自言自語。自我介紹與感謝都應壓縮或捨掉。
2MVP 規格書的「先不做」清單最大的用途是？
展開解析
正解 B。MVP 邊界不是寫謙虛話，是事先把「這次刻意不做的 X/Y/Z」釘死，讓你開發時遇到「順便加一下」的誘惑時拒絕得掉。
3「靜態前端直呼 API」和「後端代理」最主要的差別是？
展開解析
正解 B。靜態前端直呼把 key 寫在瀏覽器能被抓，適合個人私用；後端代理讓 key 留在伺服器、可做 rate limit 與成本控管，適合對外商用。
4六層架構圖（使用者／介面／邏輯／AI／資料／部署）最容易被新手漏掉的是？
展開解析
正解 C。新手愛畫「使用者→前端→AI→輸出」就結束，常漏資料持久化與部署成本。評審/客戶真正關心的反而是這兩層。
5測試三案例（正常／邊界／空白輸入）裡，「邊界輸入」指的是？
展開解析
正解 B。邊界測試是合法但極端——最長、最短、全符號、全英文等。最容易在 demo 當場翻車的就是這類「我以為不會有人這樣用」的輸入。
65 分鐘 Demo 的標準五段結構是？
展開解析
正解 B。問題／做法／展示／限制／後續是能撐 5 分鐘又不亂的骨架，「限制」段敢講真正的弱點反而加分。
7Demo 當場 AI 出現幻覺或回應錯誤時，最好的應對是？
展開解析
正解 B。Demo 翻車 SOP：承認 → 歸類（已知／未知）→ 改用備案繼續。停在現場 debug 是大忌。
8作品集卡片寫「大幅提升客服效率 90%」這種描述的問題是？
展開解析
正解 B。可驗證的數字 > 空泛的形容詞。寫「3 位同事試用 2 週、8→2 分鐘」勝過「大幅提升 90%」。
9「QA 題庫應對技術評審」最關鍵的原則是？
展開解析
正解 B。誠實優先＋答得出口。背長不如背精簡，講「不確定」+ 下一步反而比硬掰加分。
10發表後的「後續方向」這段，寫給轉職者聽的最佳版本是？
展開解析
正解 B。「後續方向」是下一個機會的誘餌——具體＋帶需求，讓台下有理由散場後來找你。空話或躺平都浪費了 5 分鐘舞台。
綜合任務｜1 題（45 分鐘）
任務｜140 小時課程總整合
選一個你在 140 小時課程中覺得最有潛力做成產品的點子（可以是本 Part 的最終專題、或前面任一實作延伸），45 分鐘內完成 4 個交付：

(1) 一頁 Pitch：90 秒可念版本，痛點／現況證據／提案／要求四段齊全。
(2) 六層架構圖：使用者／介面／邏輯／AI／資料／部署，每層寫 1 行具體決策（可用文字表示，不一定要畫圖）。
(3) 10 題 QA 題庫：技術 4 ／商業 3 ／法律或合規 2 ／個人 1，每題答案 ≤60 字。
(4) 作品集卡片：標題（<12 字）+ 一句話 + 3 個具體數字 + 一句「我學到什麼」。
驗收指標
Pitch 可念：實際計時 85–95 秒念得完，痛點在前 15 秒出現
架構完整：六層都有、資料與部署層不省略、有具體工具名
QA 耐問：至少 2 題敢寫「不確定」+ 下一步，沒有硬掰
卡片可驗：3 個數字都有母數與期間，不是「大幅提升」
本卷得分 — 按右方按鈕算分，10 題計 100 分；綜合任務 4 項交付各自評 0/5/10 分（總分 60 以上算通關）

140 小時結課只是中途站，真正的差距在「結訓後第 1 天到第 90 天」如何繼續累積。下面列出 3 條已被市場驗證的發展路徑，對應不同的學員畫像、目標與工具棧。挑一條最貼近你現在處境的，把 Prompt 複製到 AI 工具裡，生成你自己的 3 個月週表，再按下方驗收里程碑逐週自我追蹤。

路徑 A AI 助理擴充者｜往 MCP、agent framework、auto tools 發展
路徑 A｜AI 助理擴充者
● 學員畫像：已經天天用 ChatGPT / Claude / Gemini 做日常工作，想把「手動貼輸入」升級成「AI 自己去拿資料、自己跑工具」 ● 3 個月目標：能獨立開發 2–3 個 MCP server（連自家 Notion / Gmail / 客戶 CRM）並部署給團隊共用；能讀懂 agent loop 程式碼並自己寫一個多步驟 agent ● 建議工具棧：Claude Code / Cursor + MCP SDK（Python 或 TypeScript）+ anthropic-sdk / openai-sdk + LangGraph 或 Anthropic Agent SDK + Vercel（部署）+ GitHub（版本管理 + Actions） ● 市場定位：中小企業 AI 助理客製化接案、團隊內部 AI 工具負責人（月薪 5–8 萬或專案 5–20 萬）
3 個月自主學習計畫 Prompt
## 角色 你是 AI 工具開發者的 3 個月自學規劃教練，擅長把模糊的「我想學 AI」拆成可驗收的週級任務。 ## 任務 為想往「AI 助理擴充者」方向發展的學員，產出 12 週自主學習計畫（每週 8–12 小時）。 ## 學員資料 - 目前能力：[填你的起點，例：「會用 ChatGPT 但沒寫過 API 串接」] - 每週可投入：[填 N 小時，建議 8–12] - 手邊真實場景：[填 1 個真實工作痛點，例：「每週要從 Notion 抓會議紀錄整理成週報」] ## 輸出要求 產出 12 週表格，每週包含： 1. 週主題（一句話） 2. 學習輸入（讀什麼 / 看什麼，2–3 項） 3. 動手任務（寫什麼 / 做什麼，1 項可交付） 4. 驗收里程碑（這週結束前，你能對別人示範什麼） 里程碑分段： - 第 1–4 週：建 API 串接基礎（能寫一個讀取 Notion API 的 Python / TS 腳本） - 第 5–8 週：MCP server 第一版（能寫一個自家工具的 MCP，部署到 Claude Desktop 能用） - 第 9–12 週：多步驟 agent（能寫一個「自己規劃、自己呼叫工具、自己回報」的 agent，解決週報自動生成） ## 限制 - 每週動手任務必須是可交付的程式碼 / 部署 URL / GitHub commit，禁止「讀完某本書」這種無法驗證的任務 - 驗收里程碑要能錄成 30 秒示範影片，禁止「理解了某個概念」這種主觀描述 - 12 週結束要有 1 個公開作品（MCP server 或 agent）放到 GitHub，README 寫給會用但不會寫的人看
週表 + 可驗收里程碑
W1–4 API 串接基礎（Notion/Gmail/CRM 至少 1 個可跑）→ W5–8 第一個 MCP server 部署成功（Claude Desktop 能呼叫）→ W9–12 多步驟 agent 解決真實場景（週報自動化或客戶查詢自動化）。每週 1 個 commit，第 12 週交付 1 個公開 GitHub repo + 30 秒示範影片。
路徑 B AI 產品經理｜往 evals、metrics、用戶研究、產品發現發展
路徑 B｜AI 產品經理
● 學員畫像：比起自己寫 code，更有興趣「決定做什麼、做多好、給誰用」；喜歡跟人聊、把模糊需求轉成規格 ● 3 個月目標：能獨立跑完一個 AI 功能從「用戶訪談 → Prompt 雛形 → Eval 集建立 → 上線 A/B 測試 → 指標讀寫」的完整循環；能寫一份 AI 產品規格書（PRD）給工程師照著做 ● 建議工具棧：Anthropic Console（Prompt 開發 + Evals）+ LangSmith / Braintrust（生產監控）+ Notion（用戶訪談紀錄）+ Dovetail 或 Airtable（訪談編碼）+ Looker / Metabase（指標儀表板）+ GitHub（與工程協作） ● 市場定位：AI 產品公司 PM / 新創 AI feature owner（月薪 7–12 萬），或 B2B SaaS 的 AI 模組負責人
3 個月自主學習計畫 Prompt
## 角色 你是 AI 產品經理的 3 個月自學規劃教練，擅長讓轉職者不寫 code 也能被 AI 團隊認可「懂產品、懂模型、懂指標」。 ## 任務 為想轉型成「AI 產品經理」的學員，產出 12 週自主學習計畫（每週 8–12 小時）。 ## 學員資料 - 目前能力：[填你的起點，例：「做過非 AI 的網頁 PM，沒跑過 eval」] - 每週可投入：[填 N 小時] - 手邊真實場景：[填 1 個你想做的 AI 產品點子，例：「給 HR 用的面試逐字稿摘要工具」] ## 輸出要求 產出 12 週表格，每週包含： 1. 週主題 2. 學習輸入（論文 / 部落格 / 案例研究，2–3 項） 3. 動手任務（寫 PRD / 做訪談 / 跑 eval，1 項可交付） 4. 驗收里程碑 里程碑分段： - 第 1–4 週：用戶發現（至少訪談 8 位目標用戶，產出需求分級表） - 第 5–8 週：Prompt + Eval 集（建立 30 筆 eval 案例，用 Anthropic Console 或 LangSmith 跑三輪迭代） - 第 9–12 週：產品上線與指標（用 Replit / Bolt 做 MVP 找 5 位早期使用者，定義 3 個核心指標並追蹤 4 週） ## 限制 - 每週動手任務必須可交付（訪談錄音 + 紀錄 / eval 集 CSV / PRD 文件 / 指標截圖），禁止「讀完某本書」 - 訪談紀錄要有「對方原話」段落，禁止只寫你的摘要 - 第 12 週要交付 1 份完整 PRD（問題、用戶、解法、eval 結果、上線指標），能直接拿去投 AI PM 職缺
週表 + 可驗收里程碑
W1–4 至少訪談 8 位用戶 + 需求分級表 → W5–8 30 筆 eval 集 + Prompt 三輪迭代（命中率數字可驗證）→ W9–12 MVP 上線 + 5 位真實用戶 + 3 個指標 4 週追蹤。第 12 週交付 1 份可投職的 PRD（含 eval 報告 + 指標截圖）。
路徑 C AI 應用工程師｜往後端中介層、deployment、observability 發展
路徑 C｜AI 應用工程師
● 學員畫像：在課程中發現自己對「串 API、寫後端、把東西部署上去跑得穩」特別有興趣；想進軟體團隊當「會用 AI 的工程師」 ● 3 個月目標：能獨立把一個 AI 功能從本機 Prompt 變成穩定運行的正式服務（含 rate limit、error handling、日誌、監控、成本追蹤）；能在履歷上放 1–2 個真實部署 URL 撐場 ● 建議工具棧：Node.js / Python 後端 + Hono 或 FastAPI + Vercel / Railway / Fly.io（部署）+ Upstash Redis（rate limit / cache）+ Sentry（錯誤監控）+ Langfuse 或 Helicone（LLM observability）+ Stripe（如果要收費）+ PostgreSQL / Supabase ● 市場定位：軟體公司後端工程師（AI 加分項）、或 AI 新創 full-stack（月薪 6–10 萬起）
3 個月自主學習計畫 Prompt
## 角色 你是 AI 應用工程師的 3 個月自學規劃教練，擅長讓轉職者寫得出「能上線、能被監控、能撐流量」的 AI 後端服務。 ## 任務 為想當「AI 應用工程師」的學員，產出 12 週自主學習計畫（每週 10–14 小時）。 ## 學員資料 - 目前能力：[填你的起點，例：「會寫基本 Python / JS，沒部署過正式服務」] - 每週可投入：[填 N 小時] - 手邊真實場景：[填 1 個你想做的 AI 服務，例：「給自由工作者用的發票 OCR + 分類 API」] ## 輸出要求 產出 12 週表格，每週包含： 1. 週主題 2. 學習輸入（官方文件 / 教程 / 原始碼，2–3 項） 3. 動手任務（寫 code / 部署 / 加監控，1 項可交付） 4. 驗收里程碑 里程碑分段： - 第 1–4 週：可用的 API 後端（含 auth、rate limit、basic error handling，部署到 Vercel / Railway 拿到公開網址） - 第 5–8 週：Observability + 成本監控（接上 Langfuse 或 Helicone，能看到每個 request 的 token 用量 / 延遲 / 錯誤，每日成本報表） - 第 9–12 週：壓力測試 + SLA（跑一次 100 concurrent 壓測，產出 p50/p95/p99 延遲數字，寫一份 incident runbook） ## 限制 - 每週任務必須對應 git commit 或部署紀錄，禁止「讀完某文件」 - 部署 URL 必須能被外部訪問（不是 localhost），禁止只在本機跑 demo - 第 12 週要交付 1 個部署中的服務 + README（含架構圖、API 文件、成本報表截圖、壓測數據），履歷可直接貼連結
週表 + 可驗收里程碑
W1–4 後端 API 上線公開網址（含 auth + rate limit + error handling）→ W5–8 Observability 接通（每日成本報表 + token/延遲/錯誤可視化）→ W9–12 壓測 + SLA（p50/p95/p99 數字 + incident runbook）。第 12 週交付 1 個穩定運行的服務 URL + 可貼履歷的 README。

（進階選配）找 2 位前同事 / Mentor / 社群朋友 / LinkedIn 群組做月度 review結課後 7 天內，聯絡 2 位前同事、Mentor、社群朋友或 LinkedIn 上的工作領域朋友（選和你路徑不同的人更好），約定每月用 30 分鐘互相 review 進度，為轉職後的學習網絡鋪路。對方看你交付物、你看對方的，彼此給 1 個最狠的回饋。非必要，但能加分——核心驗收靠自己的 LLM 三人格壓測 + 24h 自審就夠（PRAC7-2 框架），找不到對象不會卡住結業。驗收：前 3 個月至少完成 2 輪外部 review，有書面紀錄（Notion / Google Doc 都行）。
每月自我 eval每月最後一天，花 30 分鐘回答 3 題自我 eval：(1) 這個月交付了什麼？（列清單 + 連結）(2) 原本計畫的哪些沒做？為什麼？（3) 下個月要調整哪一週的內容？連寫 3 個月（結訓後第 1、2、3 個月），就能清楚看到自己是「真的在走」還是「原地打轉」。驗收：3 份自我 eval 文件可以直接拿去投職缺當履歷附件。
1結訓後第一件最該做的事是？
看答案
B。結訓當下「作品 + 自述」的鮮度最高、記憶最清楚，是最容易變現成履歷、接案頁、自我介紹的資產。A 會讓你永遠在追新，卻沒有可展示的過去；C 銷毀證據；D 讓熱度與肌肉記憶雙雙流失。先把已有的 140 小時變現，再決定下一步。
2自主學習最容易放棄的關卡通常在哪？
看答案
B。第 1 週靠新鮮感撐；第 10 週靠慣性；真正殺手是第 3 週上下的 plateau：新鮮感消失、難度上升、第一批成品還沒成形。預先知道這個關卡存在，就能在第 3 週刻意降低任務複雜度、找夥伴打氣、回頭交付第 1–2 週累積的半成品，而不是硬撐進入死亡螺旋。
3自主學習地圖該設計到多具體？
看答案
C。太粗（A/D）會讓你第 3 週卡住時不知道自己偏了多少；太細（B）一天沒跟上就崩盤。週級 + 可量測里程碑是被驗證過最穩的粒度：每週結束能明確回答「這週交付了什麼」，又不會因為某一天狀態差就整個計畫廢掉。
4自學過程中抱怨「學不到東西」時，該先怎麼辦？
看答案
B。「學不到東西」的感受有兩個源頭：(1) 只在讀不在做（方法問題：缺動手交付物）(2) 真的內容偏離目標（內容問題：教材錯配）。前者佔 80%，解法是把輸入時間砍半、動手時間加倍、每週強制 1 個交付物；後者才需要換教材。不先做這個判斷就換（A/D），會反覆換課但始終沒動手。
5什麼時候可以考慮從 A/B/C 中換一條路徑？
看答案
C。換路徑成本很高（之前累積部分作廢）但有時必要。判斷標準要同時滿足兩條：(1) 連續 3 週無可驗收進展——不是感覺不好，是真的交不出東西；(2) 已排除方法問題（改動手比例後仍無進展）。A/B 是情緒驅動的換，通常會再換一次、再換一次；D 則忽略真的走錯的情況。連續 3 週 + 方法已調整 = 可以且應該換。
