架構設計 Checklist

開始寫程式之前，先用六個面向把架構想清楚。每少問一個問題，後面就多一個意外。

學完你會
用六層架構 Checklist 評估你的專題設計
判斷你的專題是否需要後端，以及後端要做什麼
知道開始開發前必須確認的安全與成本決策
選定部署平台並知道為什麼
Section 01

六層架構 Checklist

在動手前把這六個面向問清楚。每個面向都會影響開發決策，跳過任何一個都可能導致中途返工。

🖥
前端
頁面要幾個？一頁工具還是多頁應用？
輸入區放什麼元件（textarea / select / file upload）？
輸出如何呈現（直接顯示文字 / 格式化 / 複製按鈕）？
Loading 狀態如何顯示？Streaming 還是 spinner？
⚙️
後端
需不需要後端？（沒有 API Key 保護需求 → 考慮 Vercel 靜態 + 環境變數）
如果需要，後端負責什麼？（代理 API / 資料處理 / RAG 管線）
用 Express / Vercel Functions / Render？
🤖
AI
用哪個模型？Haiku（快/便宜）/ Sonnet（平衡）/ Opus（最強）？
System Prompt 定義什麼角色與限制？
temperature 設多少（創意 → 0.7–1.0 / 精確 → 0–0.3）？
輸出格式要求是什麼（純文字 / JSON / Markdown）？
🗄
資料
需不需要知識庫？（RAG）
需不需要工具呼叫？（Function Calling / n8n）
需不需要儲存使用者輸入或歷史記錄？（大多數 MVP 不需要）
🚀
部署
靜態只有前端 → Vercel / Netlify（免費，秒部署）
含後端 → Render / Railway（免費方案有睡眠機制）
自訂網域 / HTTPS / 環境變數在哪裡設定？
🔒
安全
API Key 放後端環境變數，絕不放前端 JS 程式碼
輸入驗證：空值 / 超長輸入 / 惡意注入怎麼擋？
設每日使用量上限，防止費用爆炸
Section 02

需不需要後端？決策樹

這是最常讓初學者卡關的決策。不是所有 AI 應用都需要後端，但有些情況不加後端會有安全風險。

需要保護 API Key 不暴露在前端？
→
是
需要後端（Express 代理或 Vercel Functions）把 API Key 放環境變數
|
否（內部工具/個人用）
可用前端直呼，但務必不要把含 Key 的程式碼推上 GitHub
需要處理資料庫、RAG 管線或複雜運算？
→
是
後端是必要的，前端不適合做重型資料處理
|
否
靜態部署 + Vercel Functions 可能就足夠
需要使用者上傳檔案並在伺服器端處理？
→
是
需要後端接收 multipart/form-data，處理後送 AI
|
否（純文字輸入）
無後端方案完全可行
💡
MVP 建議：從靜態開始，有需要再加後端。很多人一開始就假設需要後端，結果花了大量時間在基礎設施上，核心功能反而做不完。先讓工具跑起來，再考慮安全與擴展。
Section 03

安全 Checklist：開始開發前必須確認的四件事

🔑
API Key 絕對不放前端
不管多急，API Key 都不能出現在前端 JS 程式碼。一旦推上 GitHub 或被 DevTools 看到，就算馬上刪除也可能已被爬蟲抓走。用 .env 放後端，用 .gitignore 擋掉。
✅
輸入驗證在送 API 之前
空值、超長輸入（建議 5000 字以下）、特殊字元都要在送 API 前處理。不只防惡意輸入，也避免 token 超出上限導致的 API 錯誤。
💰
設 API 使用量上限
在 Anthropic / OpenAI 後台設定每月預算上限。同時在程式碼中設合理的 max_tokens（不是越大越好），避免單次呼叫消耗過多 token。
🛡
System Prompt 邊界設定
在 System Prompt 中明確告訴模型「只做什麼」，而非只說「不做什麼」。正向定義邊界比負向禁止更有效，也讓模型更容易遵守。
Section 04

AI 設定決策：模型、溫度、輸出格式

這三個設定決定你的 AI 表現。不用猜，先設定一個合理的起點，實際測試後再調整。

🏃
模型選擇起點
先用 claude-haiku-4-5-20251001 開發和測試：最快、最便宜，適合迭代
輸出品質不夠時才換 Sonnet；換前先確認是 Prompt 問題還是模型限制
同樣任務 Haiku 已夠好 → 不要為了「感覺更好」換貴的模型
🌡
Temperature 起點
創意類（文案、故事、腦力激盪）：0.7–1.0
分析類（摘要、分類、提取）：0.1–0.3
結構化輸出（JSON、清單）：0 或 0.1
不確定 → 先設 0.5，測試後再調
📝
輸出格式定義
純文字：最簡單，適合摘要、建議
Markdown：標題、清單、粗體，前端用 marked.js 轉 HTML
JSON：需要後製處理時使用，在 System Prompt 明確要求 JSON Schema
多個版本：讓模型輸出編號清單，前端切割顯示
🔑
架構設計完成的標誌：你能在 5 分鐘內向另一個人說清楚「這個工具的前端是什麼、後端做什麼、AI 的角色是什麼、部署在哪、API Key 怎麼保護」。說不清楚的地方就是架構還沒想清楚的地方。
