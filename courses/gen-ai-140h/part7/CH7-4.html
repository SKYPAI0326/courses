PART 7 · 專題衝刺與成果發表
CH 7.4 · 2 小時

發表敘事與
作品集頁建置

作品好不代表發表好。一個清楚的敘事結構讓你的工具被看見、被理解、被記住。

學完你會
用五段結構（問題 → 做法 → 展示 → 限制 → 後續）講 5 分鐘 Demo
誠實說出工具限制，並讓這件事成為你的加分項
向非技術觀眾說明技術選型
把這個專題加入作品集展示首頁
Section 01

五段 Demo 敘事結構

不是每個觀眾都理解技術，但每個人都理解問題與解決。從問題出發，技術只是過程，有用的結果是終點。

1
問題（30 秒）
描述你或你的使用者面對的具體問題。用真實場景，不用技術術語。
「每次寫職缺文案，我都要花一個小時從頭想標題、描述、call to action，而且三個版本還要確保風格一致。」
2
做法（30 秒）
說你做了什麼，技術層面點到為止。重點是「為什麼這樣做」，不是「怎麼實作」。
「我用 Claude API 建了一個工具，輸入品牌關鍵字和訴求，它會自動產出三種不同語氣的文案版本。」
3
展示（2–3 分鐘）
Live Demo 或錄影。先輸入，等輸出，指出最有價值的地方。不要跳過等待過程，那是真實體驗的一部分。
「我現在輸入一個假設的情境，大家可以看到它大概 3 秒後就產出三個版本，這是正式語氣的……這是輕鬆語氣的……」
4
限制（30 秒）
誠實說出這個工具做不到什麼，以及你知道這是個問題。這比隱瞞更有說服力。
「目前輸入超過 500 字時品質會下降，這是我下一步要解決的問題。另外它不支援圖片輸入，這是刻意先不做的邊界。」
5
後續（30 秒）
說一件最想繼續做的事，顯示你已經在思考這個產品的未來。
「如果有更多時間，我想加入品牌語氣的記憶功能，讓它記住你的品牌個性，每次生成都更一致。」
💡
5 分鐘是鐵律。準備 4 分 30 秒的內容，留 30 秒緩衝。超時不只是禮貌問題，也代表你沒有清楚理解什麼最重要。多餘的部分剪掉，不是找地方塞。
Section 02

限制怎麼說：誠實比誇大有說服力

初學者最常犯的錯：要麼完全不提限制，要麼說了限制卻沒有解釋為什麼。觀眾聽不到限制，會自己找——而且他們找到的時候比你說出來更尷尬。

情況低分說法高分說法
輸入長度限制 「目前有一些限制」（模糊） 「超過 800 字輸入品質會下降，這是 token 限制，下一步我會加截斷提示」
沒有做的功能 「這個功能還在開發中」（聽起來像藉口） 「登入功能我刻意先不做，MVP 只驗證核心價值，帳號系統會讓開發時間增加三倍」
AI 輸出不穩定 沉默（最糟糕） 「有時候 AI 會產出格式不一致的結果，我已有測試案例記錄，下一步加 output validation」
只能中文 「英文版還沒做」（暗示是缺陷） 「我的使用者都是中文使用者，所以這個版本只支援中文，這是刻意的選擇」
Section 03

把專題加入作品集：你在 PRAC5-3 建的那個頁面

你在 Part 5 建立了個人作品展示首頁。現在把 Part 7 專題加進去，作品集就完整了。

1
準備四個元素
工具名稱、一句話說明（解決什麼問題）、主要技術標籤（3 個以內）、部署網址。
2
加入一張截圖或 GIF
比文字更快讓人理解工具長什麼樣子。Loom 可以輕鬆錄製操作 GIF。
3
加入「開發中學到的一件事」
這個欄位比技術清單更能顯示你的成長。例如：「學到了 API 後端代理的重要性，第一版把 Key 放前端，上線前一小時才發現」。
4
連結到 GitHub Repo
即使程式碼不完美，公開 repo 顯示你願意被看見。記得確認 .gitignore 有擋住 .env。
🔑
作品集的目的是讓對的人找到你。技術面試官看 repo 和測試記錄；潛在合作者看展示和說明；招募者看敘事和解決的問題。同一個作品，不同人在乎不同面向——確保三者都看得到。
