PART 7 · 專題衝刺與成果發表
CH 7.4 · 2 小時

發表敘事與作品集頁建置

作品好不代表發表好。一個清楚的敘事結構讓你的工具被看見、被理解、被記住。

學完你會
用五段結構（問題 → 做法 → 展示 → 限制 → 後續）講 5 分鐘 Demo
誠實說出工具限制，並讓這件事成為你的加分項
向非技術觀眾說明技術選型
把這個專題加入作品集展示首頁
(01)

五段 Demo 敘事結構

不是每個觀眾都理解技術，但每個人都理解問題與解決。從問題出發，技術只是過程，有用的結果是終點。

1
問題（30 秒）
描述你或你的使用者面對的具體問題。用真實場景，不用技術術語。
「每次寫職缺文案，我都要花一個小時從頭想標題、描述、call to action，而且三個版本還要確保風格一致。」
2
做法（30 秒）
說你做了什麼，技術層面點到為止。重點是「為什麼這樣做」，不是「怎麼實作」。
「我用 Claude API 建了一個工具，輸入品牌關鍵字和訴求，它會自動產出三種不同語氣的文案版本。」
3
展示（2–3 分鐘）
Live Demo 或錄影。先輸入，等輸出，指出最有價值的地方。不要跳過等待過程，那是真實體驗的一部分。
「我現在輸入一個假設的情境，大家可以看到它大概 3 秒後就產出三個版本，這是正式語氣的……這是輕鬆語氣的……」
4
限制（30 秒）
誠實說出這個工具做不到什麼，以及你知道這是個問題。這比隱瞞更有說服力。
「目前輸入超過 500 字時品質會下降，這是我下一步要解決的問題。另外它不支援圖片輸入，這是刻意先不做的邊界。」
5
後續（30 秒）
說一件最想繼續做的事，顯示你已經在思考這個產品的未來。
「如果有更多時間，我想加入品牌語氣的記憶功能，讓它記住你的品牌個性，每次生成都更一致。」
💡
5 分鐘是鐵律。準備 4 分 30 秒的內容，留 30 秒緩衝。超時不只是禮貌問題，也代表你沒有清楚理解什麼最重要。多餘的部分剪掉，不是找地方塞。
迷你練習 1 約 20 分鐘
寫出 5 分鐘 Demo 逐字稿並計時朗讀

不要即興、不要大綱——先用五段結構寫逐字稿，計時朗讀一次，多出來的全部砍掉。4 分 30 秒是合格線。

1照五段結構（問題 30s / 做法 30s / 展示 2–3min / 限制 30s / 後續 30s）分段寫逐字稿。
2展示段落把實際操作步驟寫進去：「我輸入 X → 等待 → 指向 Y 說『這裡最有價值』」。
3計時朗讀一次，記錄總秒數；超過 4:30 就砍，不是加速朗讀。
4砍完再朗讀一次，確認 4:00–4:30 之間，句句都有作用。
範本素材 把專題摘要貼進去，讓 AI 依五段結構寫出 4:30 逐字稿草稿，你再依實際節奏修
原始素材 · 專題成果摘要（約 240 字）貼進 Prompt 最下方
專題名稱：履歷優化小工具 使用者：轉職工程師（我自己 + 5 位同期朋友） 痛點：投 20 個職缺 2 次面試，手動改履歷一份要 40 分鐘 做法：Claude Haiku API + Vercel Functions 代理 + 單頁前端，兩個 textarea 貼履歷/職缺 → 三段改寫輸出 MVP 範圍：只做核心改寫，不做登入、歷史記錄、PDF 匯出 Demo 展示內容：我會現場貼一份真實履歷 + 職缺，展示 3 秒內出三段改寫 已知限制：① 輸入超過 3000 字品質下降（token 限制）② 沒有版本比對 ③ 只支援繁體中文 實際數字：自己 + 5 位朋友試用，平均改履歷時間從 40 分鐘降到 8 分鐘 下一步：想加入「品牌語氣記憶」——讓工具記住個人寫作風格，每次改寫更一致
範例 Prompt · 5 分鐘 Demo 逐字稿產生器可直接貼 ChatGPT / Claude
你是 Demo 教練，專門把專題摘要改寫成 4:30 的 5 段逐字稿。 任務：讀下方專題成果摘要，產出逐字稿。 規則： 1. 五段結構：問題 30s / 做法 30s / 展示 2:30 / 限制 30s / 後續 30s 2. 第一句話直接講「使用者的具體問題」，不做自我介紹 3. 做法段落技術名詞最多 3 個（如 Claude API、Vercel Functions） 4. 限制段落用「高分說法」——誠實 + 解釋 + 下一步 5. 總字數 600-700 字（中文口說每分鐘約 150 字，4:30 = 675 字） 輸出格式： # Demo 逐字稿（目標 4:30） ## ① 問題（目標 30 秒 · 約 75 字） [逐字稿內容] ## ② 做法（目標 30 秒 · 約 75 字） [逐字稿內容] ## ③ 展示（目標 2:30 · 約 375 字，含實際操作旁白） [逐字稿內容] ## ④ 限制（目標 30 秒 · 約 75 字） [逐字稿內容] ## ⑤ 後續（目標 30 秒 · 約 75 字） [逐字稿內容] ---專題成果摘要--- [把上面的原始素材整段貼在這裡]
預期輸出片段（供對照）
① 問題：「我和五位同期轉職的朋友，投了 20 個職缺只拿到 2 次面試。每次改履歷要手動抓職缺關鍵字，一份花 40 分鐘，而且每次都在重做一樣的事。」
② 做法：「我用 Claude Haiku API 搭 Vercel Functions 做了一個單頁小工具，貼上履歷和職缺說明，3 秒內產出三段針對該職缺改寫過的履歷。」
③ 展示（示範操作旁白）：「現在我貼一份真實履歷……接著貼這則資深前端職缺……點產生……這是第一段強調……」
④ 限制：「目前輸入超過 3000 字時品質會下降，這是 token 限制，下一版會加截斷提示。帳號系統我刻意不做，因為會讓開發時間多三倍。」
⑤ 後續：「最想加的是『品牌語氣記憶』，讓工具記住每個人的寫作風格，每次改寫更一致。」
成果產出
一份 4:00–4:30 的 Demo 逐字稿，含五段標題與預估秒數；PRAC7-5 發表當天直接用。
自我檢核
(02)

限制怎麼說：誠實比誇大有說服力

初學者最常犯的錯：要麼完全不提限制，要麼說了限制卻沒有解釋為什麼。觀眾聽不到限制，會自己找——而且他們找到的時候比你說出來更尷尬。

情況低分說法高分說法
輸入長度限制 「目前有一些限制」（模糊） 「超過 800 字輸入品質會下降，這是 token 限制，下一步我會加截斷提示」
沒有做的功能 「這個功能還在開發中」（聽起來像藉口） 「登入功能我刻意先不做，MVP 只驗證核心價值，帳號系統會讓開發時間增加三倍」
AI 輸出不穩定 沉默（最糟糕） 「有時候 AI 會產出格式不一致的結果，我已有測試案例記錄，下一步加 output validation」
只能中文 「英文版還沒做」（暗示是缺陷） 「我的使用者都是中文使用者，所以這個版本只支援中文，這是刻意的選擇」
(03)

把專題加入作品集：你在 PRAC5-3 建的那個頁面

你在 Part 5 建立了個人作品展示首頁。現在把 Part 7 專題加進去，作品集就完整了。

1
準備四個元素
工具名稱、一句話說明（解決什麼問題）、主要技術標籤（3 個以內）、部署網址。
2
加入一張截圖或 GIF
比文字更快讓人理解工具長什麼樣子。Loom 可以輕鬆錄製操作 GIF。
3
加入「開發中學到的一件事」
這個欄位比技術清單更能顯示你的成長。例如：「學到了 API 後端代理的重要性，第一版把 Key 放前端，上線前一小時才發現」。
4
連結到 GitHub Repo
即使程式碼不完美，公開 repo 顯示你願意被看見。記得確認 .gitignore 有擋住 .env。
🔑
作品集的目的是讓對的人找到你。技術面試官看 repo 和測試記錄；潛在合作者看展示和說明；招募者看敘事和解決的問題。同一個作品，不同人在乎不同面向——確保三者都看得到。
迷你練習 2 約 15 分鐘
把專題加進 PRAC5-3 作品集頁面

作品集不是「以後再弄」，現在把這個專題加進去，它才算真正收尾。四個元素 + 一個學習心得，15 分鐘內完成。

1打開 PRAC5-3 建的作品展示頁，在列表裡加一張新卡片。
2填四個元素：工具名稱、一句話說明、技術標籤（≤3 個）、部署網址。
3用 Loom 或手機螢幕錄一段 10 秒 GIF，放進卡片。
4加「開發中學到的一件事」欄位，寫一句真實的學習（愈具體愈好，避開空話）。
範本素材 把專題粗略資訊貼進去，讓 AI 一次產出作品集卡片的四個元素 + 學習心得兩版本
原始素材 · 專題粗略資訊（約 200 字）貼進 Prompt 最下方
專題名稱：履歷優化小工具（暫定，不確定要不要改） 目標使用者：轉職工程師 解決問題：手動改履歷一份 40 分鐘、每次都重做一樣的事 技術：前端 HTML/CSS/JS 單頁、後端 Vercel Functions、AI Claude Haiku API 部署網址：https://resume-tuner.vercel.app GitHub：https://github.com/myname/resume-tuner 開發期間踩的坑： - 第一版把 API Key 放前端 JS，推上 GitHub 後一小時才發現，Key 已經洩漏 - 只有我自己測試時都 OK，給朋友用才發現沒寫職缺說明就按產生會當掉 - temperature 一開始設 0.9，結果 AI 會編使用者沒寫過的工作經歷，改 0.3 才穩
範例 Prompt · 作品集卡片產生器可直接貼 ChatGPT / Claude
你是作品集編輯，專門把專題資訊改寫成吸引人的作品集卡片。 任務：讀下方專題粗略資訊，產出一張作品集卡片需要的全部文案。 規則： 1. 「一句話說明」要讓非技術的 HR 秒懂「解決什麼問題」，不超過 30 字 2. 技術標籤最多 3 個，優先放最具代表性的（AI 模型 / 核心框架 / 部署平台） 3. 「學到的一件事」寫兩個版本：① 給 HR 看（重點是判斷力與學習）② 給工程師看（重點是技術細節與教訓） 4. 學習心得要是「具體經驗」，禁止出現「學到很多」「收穫滿滿」這種空話 5. 如果專題名稱可以更好，額外建議一個替代名稱 輸出格式（Markdown）： ## 作品集卡片 - 工具名稱：… - 一句話說明（≤30 字）：… - 技術標籤（≤3）：#… #… #… - 部署網址：… - GitHub：… ## 學到的一件事 **給 HR 看（2-3 句）：** … **給工程師看（2-3 句）：** … ## 名稱建議 - 現名：… - 替代：… · 理由：… ---專題粗略資訊--- [把上面的原始素材整段貼在這裡]
預期輸出片段（供對照）
一句話說明：「貼上履歷與職缺，3 秒產出三段針對該職缺的改寫」
技術標籤：#Claude-Haiku #Vercel-Functions #原生前端
學到（給 HR 看）：「第一版把 API Key 放前端，上線前一小時發現才補救。這件事讓我養成『部署前先跑一次安全 checklist』的習慣。」
學到（給工程師看）：「API Key 洩漏後我把它轉到 Vercel Functions 環境變數，前端改成 fetch 自家 /api/generate。順便發現 temperature 從 0.9 降到 0.3 才能讓改寫忠於原履歷，這是 prompt engineering 不能光看直覺的證據。」
名稱建議：現名 OK，若想更聚焦可用「JobFit Resume」或「履歷對齊器」
成果產出
作品集頁面多一張完整卡片，含四元素 + GIF + 學習心得 + GitHub 連結；這就是 140h 課程的最終展示物。
