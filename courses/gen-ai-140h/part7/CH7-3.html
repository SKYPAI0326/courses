CH 7.3 · 2 小時

測試與 UX 優化

工具能跑不代表好用。測試讓你找到問題，UX 優化讓非技術使用者也能順暢使用。

學完你會
設計 10 個涵蓋正常、邊界、錯誤情境的測試案例
辨識並修正四個常見 UX 問題
知道如何讓非技術朋友試用並給出有效反饋
設計有意義的錯誤訊息，不讓使用者看到 undefined
(01)

10 個測試案例：正常、邊界、錯誤三類

好的測試不是隨機輸入，而是刻意設計的場景。10 個案例的建議分配：4 個正常情境 + 4 個邊界情境 + 2 個錯誤情境。

正常情境（4 個）
核心使用路徑
最典型的「教科書式輸入」：清楚、完整、格式正確
輸入比較短的案例（100 字以內），確認輸出不過於簡短
輸入比較長的案例（超過 1000 字），確認輸出品質穩定
中文輸入（如果你的工具設計給中文使用者）
邊界情境（4 個）
極端但合法的輸入
剛好到輸入字數上限的內容
包含特殊字元（標點、Emoji、換行符）的輸入
只有一個關鍵字的極短輸入
輸入跟你設計的使用情境完全不相關（但合法）的內容
錯誤情境（2 個）
應該要被擋住的輸入
空白輸入（或只有空格）：應顯示提示，不應送 API
超長輸入（故意超出你設的上限）：應截斷或提示，不應讓 API 超出 token 限制
💡
把測試結果記下來。PRAC7-4 的部署驗收需要提交測試記錄。不需要正式的測試框架，一個表格（輸入 → 預期輸出 → 實際輸出 → 通過/失敗）就夠了。
迷你練習 1 約 20 分鐘
為你的專題寫出 10 個測試案例

依照 4 正常 + 4 邊界 + 2 錯誤的配比，為你的工具設計 10 個具體測試輸入，並預先寫下期待的輸出形狀。

1開一張表格，四欄：案例類型 / 輸入內容 / 期待輸出 / 實際輸出（預留）。
2依 4 / 4 / 2 配比填滿 10 列：正常—短—長—中文、邊界—上限—特殊字元—極短—離題、錯誤—空白—超長。
3對每個案例「期待輸出」寫一句話（不是完整答案，是樣貌，如「三段文字，每段含動詞開頭」）。
4跑過一輪，把實際輸出填進去，標通過 / 失敗。
範本素材 把專題規格貼進去，讓 AI 一次生成 10 列測試案例表（4 正常 + 4 邊界 + 2 錯誤）
原始素材 · 專題規格 + 輸入限制（約 180 字）貼進 Prompt 最下方
專題：AI 履歷優化小工具 輸入欄位： - 履歷段落（textarea，必填，100-3000 字） - 職缺說明（textarea，必填，50-2000 字） 預期輸出：三段改寫履歷（Markdown，每段 60-120 字） 核心使用路徑：使用者同時貼兩個欄位 → 點產生按鈕 → 3 秒內出結果 已知風險：欄位只填一個會編資訊、超過 3000 字 token 爆掉、特殊字元（Emoji / Markdown 符號）干擾 使用情境：轉職工程師自用，繁體中文台灣用語
範例 Prompt · 測試案例產生器可直接貼 ChatGPT / Claude
你是 QA 工程師，專門為 AI 應用設計測試案例表。 任務：讀下方專題規格，產出 10 個具體測試案例，涵蓋三類：4 個正常 + 4 個邊界 + 2 個錯誤。 規則： 1. 每個「輸入內容」必須具體到可以直接複製貼進工具（不要寫「一段長文」這種模糊描述） 2. 每個案例寫明「期待輸出」的樣貌，不是完整答案，是 3-4 個特徵 3. 錯誤情境必須是「應該被程式擋住」的，不是 AI 能幫忙兜的 4. 涵蓋專題規格中「已知風險」的每一項至少一個案例 輸出格式（Markdown 表格）： | # | 類型 | 輸入內容（可直接貼）| 期待輸出樣貌 | |---|---|---|---| | 1 | 正常 | … | … | … 規格如下 ↓ --- [把上面的原始素材整段貼在這裡]
預期輸出片段（供對照）
1 正常：3 年前端履歷 + 資深前端職缺 → 三段改寫、動詞開頭、字數 60-120
2 正常：行銷轉軟體履歷 + Junior 軟體職缺 → 強調跨領域、不編造技術經歷
5 邊界：履歷含 5 個 Emoji → 輸出不應保留 Emoji
6 邊界：履歷剛好 3000 字 → 應正常處理不報錯
7 邊界：職缺說明只有 60 字 → 輸出應聚焦此 60 字提到的需求
8 邊界：兩個欄位皆填但完全不相關（履歷是廚師、職缺是工程師）→ 應提示落差大
9 錯誤：兩個欄位空白 → 應顯示「請貼上履歷與職缺」，不應送 API
10 錯誤：履歷貼 8000 字 → 應截斷或提示超出字數上限

(02)

四個常見 UX 問題及修正方式

問題一
沒有 Loading 狀態
使用者點了按鈕沒有任何反應，不知道 AI 是否在處理，可能重複點擊或以為壞掉。
修正：點擊後按鈕變灰色 + 文字改為「處理中…」，回應後恢復。若用 Streaming 則顯示逐字輸出即可。
問題二
空值時的 undefined 或崩潰
使用者沒有輸入就送出，或 API 回傳非預期格式，頁面顯示 undefined 或直接白畫面。
修正：送出前先驗證輸入（if (!input.trim())），API 回應用 optional chaining（msg?.content?.[0]?.text ?? ''）安全取值。
問題三
輸出過長沒有截斷提示
輸出 2000 字全部顯示在一個 div 裡，使用者需要捲動很長才能看完，也不知道如何使用結果。
修正：加「複製全文」按鈕；如果有多個輸出版本用分頁或卡片呈現；考慮只顯示前 300 字 + 展開按鈕。
問題四
錯誤訊息不友善
API 呼叫失敗時顯示技術錯誤訊息（如 Error: 429 Too Many Requests），使用者不知道該怎麼辦。
修正：把錯誤碼轉為人話——429 → 「請稍等一下再試」，500 → 「系統忙碌，請重新整理後再試」，401 → 「服務暫時無法使用」。
(03)

讓非技術朋友試用：如何收到有效反饋

你自己測試找不到問題，因為你太熟悉這個工具了。讓完全不了解你開發過程的人試用，是找出最真實 UX 問題的唯一方法。

1
只給 URL，不給說明
把網址傳給朋友，說「這是一個工具，你覺得它是做什麼的？」——如果他們猜不出來，你的頁面說明需要改。
2
請他們邊用邊說出想法（放聲思考）
「你正在想什麼？」「這個按鈕你預期它做什麼？」「你現在感到困惑嗎？」——不要解釋，只聽。
3
最後問三個問題
① 這個工具最有用的地方是什麼？② 你最困惑的一個地方是哪裡？③ 如果要改一件事，你希望改什麼？
4
把問題列表排優先順序
不是每個問題都要修，先修「影響核心使用路徑的問題」，次要美觀和進階功能。PRAC7-3 的第二輪衝刺就從這份清單開始。
⚠️
不要在試用時幫使用者解釋。如果他們做不到某件事，你的第一反應是解釋「你應該先點這個」——這個解釋本身就是 UX 問題。把解釋寫進介面，不要靠口頭說明。
迷你練習 2 約 20 分鐘
找 1 位非技術朋友做一次放聲試用

自己測試找不到的問題，讓不熟的人試一次就會冒出來。這不是客套，是 PRAC7-3 第二輪衝刺前最值得的 20 分鐘。

1選一位完全沒看過你工具的非技術朋友，只給 URL，不做任何說明。
2請他邊用邊說出想法（「你現在想什麼？」「你預期這個按鈕做什麼？」）——只聽、不解釋。
3試用結束問三題：最有用的地方 / 最困惑的地方 / 只能改一件事會改哪個。
4把觀察到的問題列出來，標「影響核心路徑」或「美觀細節」，前者優先修。
範本素材 試用結束後，把原始觀察筆記貼進去，讓 AI 整理成有優先順序的修正清單
原始素材 · 試用觀察原始筆記（約 220 字）貼進 Prompt 最下方
試用者：某位朋友，30 歲行銷背景，不懂程式 工具：AI 履歷優化小工具（只給 URL，沒給說明） 觀察紀錄： - 打開頁面 3 秒後問「這是做什麼的？」（頁面只有標題「履歷優化」沒有副標） - 滑了一下找不到「範例」，自己貼了自己的履歷，但沒貼職缺說明就按了產生 - 按產生後等了 5 秒沒看到 loading 狀態，以為壞掉按了第二次 - 輸出出來後說「這三段我要怎麼用？」——不知道是取代原履歷還是加進去 - 試用後三個回答： 1 最有用：輸出段落的「動詞開頭」很專業 2 最困惑：不知道職缺說明要貼多長才夠 3 只改一件事：希望按鈕按了之後有「處理中」字樣
範例 Prompt · UX 反饋整理師可直接貼 ChatGPT / Claude
你是產品經理，專門把 UX 試用觀察整理成可執行的修正清單。 任務：讀下方試用觀察筆記，產出結構化修正清單。 規則： 1. 每一條修正都要對應一個具體觀察（引用原文一句） 2. 分類兩級：「影響核心路徑」（使用者無法完成主要任務）vs「美觀細節」（不影響完成，但體驗不好） 3. 核心路徑問題優先，每條給一個具體修法（UI 文案 / 驗證邏輯 / 新增元件） 4. 不要發明觀察筆記沒提到的問題 輸出格式： ## 核心路徑問題（先修） 1. [觀察原文片段] → 問題：… → 修法：… 2. … ## 美觀細節（後修） 1. … ## 不用修（是使用者個人偏好，非通用問題） - … 最後列「下一輪衝刺要改的 3 件事」。 ---試用觀察筆記--- [把上面的原始素材整段貼在這裡]
預期輸出片段（供對照）
核心 1：「打開 3 秒後問這是做什麼的」→ 加一行副標「貼上履歷+職缺，一鍵改寫三段」
核心 2：「沒貼職缺就按產生」→ 加前端驗證、按鈕在兩欄都有內容才啟用
核心 3：「等 5 秒沒看到 loading」→ 按鈕點擊後改文字「產生中…」並 disable
核心 4：「輸出三段不知怎麼用」→ 每段上方加小標「段落一（對應 XX 需求）」
美觀 1：希望有「處理中」字樣 → 已在核心 3 涵蓋
下一輪衝刺三件事：① 副標 + 範例 ② 按鈕驗證 + loading ③ 輸出段落標題
成果產出
一份試用觀察筆記：朋友猜不到的功能、操作卡住的點、三個回答 + 排好優先順序的修正清單。
