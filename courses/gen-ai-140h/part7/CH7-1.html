PART 7 · 專題衝刺與成果發表
CH 7.1 · 2 小時

選題與 MVP 拆解

好的專題不是功能最多的，而是邊界最清晰的。這一章幫你決定「做什麼」，以及更重要的——「不做什麼」。

學完你會
用使用者 / 問題 / 場景三個問題定義你的專題
區分「MVP 核心功能」與「先不做」的邊界
設計輸入 / AI 處理 / 輸出的基本架構
選定一個方向並寫出一句話專題說明
Section 01

三個問題定義你的專題：使用者、問題、場景

很多人一開始就想「功能」，但功能是解決方案，不是問題本身。先想清楚這三件事，功能自然會浮現。

使用者是誰
這個工具給誰用？
越具體越好。「給我自己用」是合法答案。「給所有人用」通常是功能失焦的前兆。
解決什麼問題
他們現在怎麼解決這個問題？
如果現在的方式已經夠好，這個工具就沒必要。說清楚「痛點在哪、現有方案的缺口是什麼」。
什麼場景使用
他們在哪個情境下會打開這個工具？
場景決定設計：是手機上快速查詢，還是桌面上慢慢編輯？是自用工具，還是要給別人看的展示？
💡
一句話專題說明模板：「這是一個給 [使用者] 的工具，讓他們不需要 [現在的痛苦做法]，就能 [達成目標]。」寫出來給不懂技術的朋友看，如果他們秒懂，你的定義就夠清楚了。
Section 02

MVP 拆解：先做什麼、明確不做什麼

MVP（最小可行產品）不是「做一個爛版本」，而是「只做核心路徑上最少的功能，讓核心價值跑通」。

功能核心路徑？MVP 決策理由
接收使用者輸入 是 做 沒有輸入就沒有工具
AI 處理並輸出結果 是 做 核心價值所在
基本錯誤提示 是 做（簡單版） 空白輸入要擋住，不需要完美措辭
複製結果按鈕 否（但方便） 先不做 使用者可以手動複製，不影響核心
歷史記錄 否 先不做 需要資料庫，複雜度跳升，MVP 不需要
登入 / 帳號系統 否 先不做 MVP 不需要帳號，開放存取即可
多語言支援 否 先不做 先鎖定一種語言確認核心值
⚠️
「先不做」不是永遠不做。寫下來的目的是讓你自己清楚邊界，不要在衝刺過程中被「順便做一下」的念頭拉走。每個「順便」都是時間成本。
Section 03

六個可選專題方向

每個方向都符合「有輸入 → AI 處理 → 有用輸出」的基本結構，也都在 Part 1~6 的技術範圍內可以完成。

✍️
AI 文案助手
輸入：品牌名稱、訴求、目標客群 → 輸出：3–5 個不同風格的文案版本。核心技術：multi-shot prompt + temperature 控制。
📄
AI 摘要整理器
輸入：長文或貼上的逐字稿 → 輸出：重點摘要 + 行動項目。核心技術：長 context 處理 + 結構化輸出。
📋
AI 履歷優化工具
輸入：職缺說明 + 個人經歷 → 輸出：針對該職缺優化的履歷段落。核心技術：角色扮演 + 格式化輸出。
🎓
AI 課程企劃助手
輸入：課程主題 + 目標學員 + 時數 → 輸出：結構化課程大綱含學習目標。核心技術：System Prompt 定義角色 + 階層式輸出。
✈️
AI 旅遊行程規劃器
輸入：目的地、天數、偏好（文化/美食/自然）→ 輸出：每日行程 + 注意事項。核心技術：條件式輸出 + Streaming 顯示。
🔧
自選方向
只要符合「使用者有明確問題 + 輸入清楚 + 輸出有用」就可以。這個方向最能反映你的工作情境與需求。
Section 04

設計輸入 / AI / 輸出架構

確定方向後，把你的專題拆解成這三個元件。這個結構圖是後續所有開發決策的基礎。

輸入
使用者輸入什麼？
文字 / 上傳 / 選擇題？
必填 vs 選填欄位？
輸入驗證規則？
→
AI 處理
用哪個模型？
System Prompt 設定什麼角色？
temperature 設多少？
需要 RAG 或 Function Calling？
→
輸出
輸出什麼格式？
純文字 / Markdown / JSON？
輸出幾個版本？
需要後製（格式化）嗎？
🔑
黃金問題：你的工具在什麼情況下會輸出錯誤或無用的結果？現在就想好，比等上線後遇到更划算。這個問題的答案會直接指引 CH7-3 的測試案例設計。
