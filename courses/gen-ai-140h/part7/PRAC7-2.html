PRAC 7.2 · 3 小時

中期自審與 Demo 壓測

v0 demo 完成後的中期驗收 — 自我錄影 + 24h 隔離 + LLM 三人格壓測，把 demo 前能自己抓到的洞抓乾淨。批評不靠別人施捨，自己壓給自己。

任務目標
把 v0 簡述給 LLM 跑一輪快審：請 LLM 扮 1 種人格（建議「挑剔技術主管」）給「一讚 + 一疑 + 一建議」，看回饋是否點到核心問題，整理出 v1 要先動的 1–2 個改動，決定 PRAC7-3 從哪裡下手。
1
簡述 demo 給 LLM（5 分鐘）
用文字寫下 v0 demo 的五段結構：問題 → 做法 → 操作步驟 → 已知限制 → 下一步。簡短即可（200–400 字），重點是誠實寫出限制，不要假裝沒問題。
2
LLM 扮 1 種人格給回饋（10 分鐘）
把上一步的描述貼給 LLM，請它扮 1 種人格（建議「挑剔技術主管」）給「一讚 + 一疑 + 一建議」，看回饋是否點到核心問題。整理出 v1 要先動的 1–2 個改動，PRAC7-3 從這幾項下手。
想多跑幾種人格、做 24h 隔離、做 16 格矩陣？看下方「進階選配 · 不計入課程時數」。
進階選配 · 不計入課程時數（4 步驟錄影 + 上傳 + LLM 三人格 + 24h 自評）
1
Demo 準備（15 分鐘）
確保你的 v0 可以 Live Demo：網路通暢、瀏覽器開好、有一個準備好的測試輸入。不要花時間在這個環節美化介面——有功能比好看更重要。同時架好手機相機或 Loom 螢幕錄影，準備一鏡到底。
2
限時錄影 5 分鐘 Demo（10 分鐘）
用 CH7-4 學到的五段結構：問題（30 秒）→ 做法（30 秒）→ 展示（2 分鐘）→ 限制（1 分鐘）→ 後續（1 分鐘）。一鏡到底，不暫停、不剪輯。v0 階段最重要的是誠實說出限制，不要假裝沒問題。
如果 demo 中途壞掉：說「這是已知問題，原因是 [X]，我計畫 [Y] 解決」並繼續錄。臨場慌張的真實樣子留在影片裡，比剪掉的版本更有說服力。
3
上傳 + LLM 三人格壓測（30 分鐘）
把錄影上傳到 YouTube unlisted（或 Loom），把連結貼到 LLM 對話開頭，請 LLM 分別扮 3 種人格看完給「一讚 + 一疑 + 一建議」：樂觀使用者（看實用價值）、挑剔技術主管（看可行性與成本）、不懂技術的素人（看聽不聽得懂）。每組對話完整截圖。
影片可能太長 LLM 看不完，那就附 demo 逐字稿（用 YouTube 自動字幕複製即可）+ 截圖 3-5 張關鍵畫面。
4
24h 後自評 + 整合 4 套回饋（30 分鐘）
等 24 小時讓現場記憶冷卻，重看影片寫下你自己的「一讚 + 一疑 + 一建議」。把 3 套 LLM 回饋 + 1 套 24h 後自評共 4 套輸入下方工具，生成 v1 改進優先清單。PRAC7-3 只做「高優先」的項目。
回饋整合器 — 4 套回饋 → v1 改進清單

延伸案例

中期自審是把「能跑」推進到「能看」的關鍵。挑一個貼近你處境的情境，把 demo 腳本與自評矩陣完整做一次。每個案例 30–45 分鐘。

案例 01 · 接案者 Live Demo
5 分鐘 live demo 腳本 + 網路斷線 fallback 劇本
難度 ★★☆ 約 40 分鐘

你是接案設計師，下週要在客戶辦公室現場 demo 一個「品牌 logo 草稿生成器」。客戶會議室 WiFi 不穩、主管只給 5 分鐘。你要一份能照念的 demo 腳本，外加當網路斷線時怎麼救場。

原始資料｜Demo 環境現況
Demo 對象：客戶行銷總監 + 設計主管 + 老闆（3 人） 場地：客戶公司會議室，WiFi 密碼要現場問 時間：下午 3:00，主管 3:05 有另一場會議，只給 5 分鐘 工具：你的筆電 + 投影 HDMI（未測試相容性） Demo 內容：生成「環保餐具品牌」3 組 logo 草稿 預備事項：產品網址、你的 API key、本機備份截圖 客戶最在意：能不能 1 小時內交 50 組草稿給他們挑
你的任務
  1. 寫一份 5 分鐘 demo 腳本：30 秒開場 + 3 分鐘實操 + 1 分 30 秒 QA 引導
  2. 準備 3 個 fallback：WiFi 斷、API 逾時、HDMI 訊號不相容
  3. 最後 60 秒的 closing：明確說你要對方做什麼（付訂金 / 簽試作單 / 約下次）
評估指標
時間可控
計時念過 3 次，平均落在 4:30–5:00 之間
Fallback 真能救
每個 fallback 有可立即切換的備案，不是「再試一次」
客戶視角
不秀技術細節，只講對方會在意的事（速度、風格量、修改成本）
Closing 明確
離場前對方知道下一步要決定什麼、要多少錢、到何時
參考腳本（建議先自己寫再看）
點擊展開參考答案
【0:00–0:30 開場】 謝謝給我 5 分鐘。今天要示範一個我為環保品牌做的小工具：輸入品牌名稱 + 風格關鍵字，1 分鐘內生成 3 組 logo 草稿。我會先跑一次，然後講怎麼套用到你們的「綠沐」品牌。 【0:30–3:30 實操】 （在筆電已開網頁）輸入「綠沐 GreenBath」，風格選「自然簡約」，按生成。 → 等待 API 回應時：「現在後台是在跑 Anthropic 的模型，平均 40 秒出 3 組。」 → 結果出來：「這是 v1 草稿，不是完稿。我通常在這個階段跟客戶對方向，而不是挑顏色。」 → 點其中一張：「這張的綠色色相如果客戶覺得太偏藍，我這邊 30 秒重跑一組。」 【3:30–4:30 QA 引導】 「你們最擔心的應該是：1 小時真的能產 50 組嗎？是的，連續跑 API 約 50 分鐘，我會分批給你們。第二個擔心可能是版權——所有 logo 都是原創生成、有記錄檔可追溯。」 【4:30–5:00 Closing】 「如果今天你們同意，我可以在 3 天內交 50 組初稿，試作費 8,000 元。要不要我下週一把試作合約寄給劉總？」 【Fallback A · WiFi 斷】 「我手機開熱點，20 秒接上。期間先看我準備的上一個客戶案例截圖（打開本機 PDF）。」 【Fallback B · API 逾時】 「偶爾會逾時，通常 30 秒內恢復。我先放一組上次的成品給你們看，等這次跑完再比對。」 【Fallback C · HDMI 不相容】 「直接用我的筆電螢幕看，三個人圍著看沒問題。我把筆電推到你們那側。」
變體挑戰
A改成 2 分鐘濃縮版，客戶臨時說「我只有 120 秒」
B客戶當場說「請幫我們跑 10 組」，你要決定當場跑還是回去跑，寫出你的決策邏輯
C改成給 5 人團隊線上 Google Meet 的版本，fallback 要改寫

案例 02 · 自我審查矩陣
設計 4 維度 × 4 視角的 16 格自審矩陣，逼自己看見盲區
難度 ★★☆ 約 35 分鐘

你是企業內訓講師，做了一個「新進員工 FAQ 自動回覆工具」的 v0.8。手邊找不到 4 位同事願意花時間填審查表，但 demo 在即——你要靠自己 + LLM 把所有可能的盲區挖出來。設計一張自我審查矩陣：4 維度（功能完整度 / UX 直覺度 / 邊界處理 / Demo 敘事力）× 4 視角（自己 / LLM 樂觀使用者 / LLM 挑剔技術主管 / LLM 素人）= 16 格，每格寫出一個具體待修點。

原始資料｜工具現況與卡關點
工具：貼 FAQ 問題 → AI 回覆草稿（使用者是人資同事） 你的 3 個已知困擾： 1. AI 回覆太長（平均 220 字，人資希望 100 字內） 2. 語氣不夠像「自家公司」 3. 不知道該不該加「如仍有疑問請聯絡 HR」這類 boilerplate 時間限制：自審 + LLM 三人格全部跑完限 90 分鐘 最怕的盲區：自己念過 5 遍都覺得 OK，demo 給主管看才被問倒
你的任務
  1. 設計 4 × 4 自審矩陣表頭：4 維度（功能完整度 / UX 直覺度 / 邊界處理 / Demo 敘事力）× 4 視角（自己 / LLM 樂觀使用者 / LLM 挑剔技術主管 / LLM 素人）
  2. 16 格各寫一條「具體待修點」（不是「整體可以更好」這種空話）
  3. 寫一份給 LLM 的 prompt：附 demo 逐字稿 + 現有功能列表，請它分別扮 3 種人格逐維度給意見
  4. 進階選配：把 demo URL（YouTube unlisted）丟到自己的轉職社群 / LinkedIn 群組求 review，當作加分項而非必要
評估指標
16 格全填
每格都有一條具體待修點，沒有「N/A」或「沒問題」這類空格
視角差異化
同一維度跨 4 視角的內容明顯不同，不是把同一句話複製 4 次
LLM prompt 可重用
Prompt 寫成模板，下次 demo 換個工具也能套
至少 3 條進 P0
16 條裡有 3 條被標記為「demo 前必修」，且寫得出怎麼改

(A) · Feedback Triage Drills

情境演練：四種散亂回饋整理成可行動清單

同儕回饋只是最小規模的收資料。真實工作裡，你會遇到訪談逐字稿、問卷自由文字、UT 錄影、社群評論雪崩——資料量從十倍到百倍，但整理的核心動作都一樣：分類、去重、找主題、排優先。下面四個情境涵蓋四種典型資料型態，建議一個花 20–25 分鐘：讀資料 → 複製整理 Prompt 丟 AI → 對照預期產出 → 思考怎麼改 Prompt 套到自己專案。四題加總約 90 分鐘。

情境 A · 訪談逐字稿 5 位使用者各 30 分鐘，把共同痛點與獨特意見一次分層。
資料樣本｜5 份 30 分鐘訪談的摘要片段
受訪者 A（資深行銷，45 歲）：「報表匯出是痛點，每週要花 2 小時整理 Excel。我其實不在意 AI 多聰明，我要它輸出直接能貼回我的模板。」 受訪者 B（新創營運，28 歲）：「我用工具第一週超有感，第二週開始覺得 AI 寫的東西都長一樣。希望可以調語氣。」 受訪者 C（接案設計，32 歲）：「匯出 PDF 每次欄位位置都會跑掉，我客戶會以為我不專業。這個一定要修。」 受訪者 D（企業 HR，38 歲）：「我想要一鍵匯出給主管的版本，現在我要手動重排三次。」 受訪者 E（個人接案，26 歲）：「我覺得 UI 很漂亮，但我有時候找不到『複製』按鈕在哪。」 整理目標：把 5 人意見分成「共同痛點」「獨特意見」「情緒強度」三欄，最後輸出優先級清單。
整理 Prompt（複製後貼到 AI）訪談 · 主題萃取
你是使用者研究員，熟悉訪談主題分析（Thematic Analysis）。我會給你 5 份訪談摘要，請照以下步驟整理： 1. 逐句標註「痛點」「需求」「讚美」「疑問」四種標籤 2. 把相同主題的句子聚成一類，每類寫一個主題名稱（5–10 字） 3. 每個主題標記：幾位提到（N=？）、情緒強度（高/中/低，看用詞判斷）、獨特意見請特別標註 4. 最後輸出一份「主題 × 情緒」矩陣：橫軸主題、縱軸情緒強度，格子裡放人數與代表引述 5. 列出前三名待改事項，每項寫上「證據引述」與「為何優先」 輸出格式：Markdown 表格 + 清單，不需要開場白。
預期產出
主題 1：匯出格式問題（N=3，情緒高，A/C/D 都提）、主題 2：AI 內容同質化（N=1，情緒中，B 獨特）、主題 3：UI 可發現性（N=1，情緒低，E 獨特）。矩陣輸出後會看得很清楚——匯出問題是 P0（3 人高情緒），B 的語氣調整是 P1（1 人但痛在續用率）、E 的 UI 細節是 P2。
為何這樣整理：訪談最大的陷阱是「最後一個受訪者講的印象最深」。用主題聚類 + 人數 + 情緒強度三層過濾，可以避免被單一聲音牽走。
情境 B · 問卷開放欄位 50 份自由文字回答丟給 AI 分類，人腦只負責最後 quality check。
資料樣本｜問卷第 8 題「你最希望我們改善什麼？」（節錄 8 筆代表性回答）
#007：希望可以匯出 CSV 不只是 PDF，我要丟進我的分析工具。 #012：速度還可以再快一點，特別是第一次載入。 #018：我用手機看字太小，希望有夜間模式。 #023：匯出功能有 bug，檔名都是亂碼。 #029：我的同事不會英文，希望有全中文界面，現在還有英文字。 #034：沒有什麼要改，繼續維持就好。 #041：價格太貴，希望有學生方案。 #048：匯出後 Excel 打開格式會跑掉，請修。 （完整資料共 50 筆，建議實作時整份貼進 AI） 整理目標：把 50 筆自由文字分成 5–8 個主題，每主題標記件數、情緒極性、可執行性。
整理 Prompt（複製後貼到 AI）問卷 · 主題分類
你是產品研究員，我會給你 50 份問卷的開放欄位回答。請照以下步驟整理： 1. 先讀完全部，歸納出 5–8 個主題類別（類別要互斥，不要重疊） 2. 把每一筆分進類別，若一筆橫跨 2 類可複製分列 3. 每個類別統計：件數、情緒極性（正面 / 中性 / 負面）、可執行性（立即可做 / 需規劃 / 暫不處理） 4. 標出「異常值」：和其他人意見相反、或只有一人但描述極具體的回饋 5. 最後輸出一份主題 × 情緒矩陣，並給出「建議優先處理」排序（按件數 × 情緒負面權重計算） 輸出格式：先給主題總覽表，再給優先級清單。不要道歉、不要開場白。
預期產出
主題類別像是：匯出 / 效能 / 行動裝置體驗 / 語系 / 定價 / 無需改善。匯出類一定會是最多筆（推估 10–15 件）且負面情緒集中，直接進 P0。定價類即使件數多，可執行性通常是「需規劃」不是 P0。異常值欄可能會抓出 #034 這種「沒什麼要改」——要特別放進報告讓團隊知道不是全部人都不滿。
為何這樣整理：50 筆文字人工看要 1 小時、分類會隨心情變。AI 做分類 + 統計只要 2 分鐘，人類只做最後「這個分類對嗎」的 quality check，效率高 30 倍。
情境 C · Usability Testing 錄影 3 支 UT 錄影先轉逐字，再標卡關時間點，不只看使用者說什麼。
資料樣本｜3 支 UT 錄影的逐字 + 操作紀錄（節選）
受測者 P1（任務：建立第一份報告） [00:42] 「欸這個按鈕在哪？」（滑鼠在畫面上游走 15 秒） [01:30] 「喔原來要先選模板。」 [02:10] 按下「產生」後：「怎麼沒反應？」（其實正在 loading，沒有明顯指示） [03:05] 嘆氣，往後靠。 受測者 P2（任務：匯出 PDF） [00:55] 「匯出在哪裡？我找不到。」 [01:20] 終於找到後：「這個位置太邊邊了。」 [02:30] 匯出成功但：「欸字體怎麼變了？」 受測者 P3（任務：分享給同事） [00:30] 「複製連結要在哪個選單？」（試了 3 個選單） [01:50] 成功後：「其實很簡單，但第一次找不到。」 整理目標：把 3 支錄影的卡關時間點 + 情緒反應 + 推論的根因整理成一張 UT 發現表。
整理 Prompt（複製後貼到 AI）UT · 卡關分析
你是 UX 研究員，我會給你 3 位受測者的逐字稿與操作紀錄。請照以下步驟整理： 1. 每位受測者抓出所有「卡關時間點」：任務中斷、滑鼠游走、聲音嘆氣、按錯按鈕 2. 每個卡關點標註：時間、任務、情緒訊號（用詞 / 嘆氣 / 停頓）、推論的根因假設 3. 彙整跨 3 人共通的卡關點，按「影響使用者完成任務的嚴重度」分 P0/P1/P2 4. 最後輸出：共通卡關點表 + 各受測者獨有卡關 + 建議下一步測什麼 5. 特別標記「使用者沒說但行為顯示」的問題（例如滑鼠游走 = 找不到 UI） 輸出格式：先給共通卡關表，再給個別備註。簡短直接。
預期產出
共通卡關：按鈕可發現性差（P0，3 人都卡）、loading 無回饋（P1，2 人嘆氣）、匯出字體跑掉（P1，1 人但嚴重影響交付）。優先級排完直接對應下一個 sprint 要修的東西。
為何這樣整理：UT 最容易犯的錯是「只聽受測者講話」。嘆氣、滑鼠游走、沉默時間，這些非語言訊號比口頭回饋更真實——使用者會客氣，但身體不會說謊。
情境 D · 社群評論雪崩 新功能上線後當天 100 則評論，AI 先跑情緒＋主題過濾，人類只看 top 20 代表性。
資料樣本｜新功能上線當天 100 則社群評論（節錄 10 則）
#1：「新功能超讚！終於可以一鍵生成了。」 #17：「這個功能根本沒用，和舊版一樣爛。」 #23：「想問新功能可以在 iOS 用嗎？」 #34：「UI 變醜了，我想要回舊版。」 #42：「試了 5 次都失敗，一直顯示 error code 503。」 #56：「很期待但速度好慢，等了 40 秒才出來。」 #68：「和競品 XXX 比起來還是差一截。」 #71：「這個價格不值得，我要取消訂閱。」 #83：「客服回覆我的問題很快，感謝！」 #97：「希望可以加批次處理功能。」 （完整資料共 100 則） 整理目標：當天內要產出一份可貼給老闆的「社群聲量速報」，包含情緒分布、問題主題、行動建議。
整理 Prompt（複製後貼到 AI）社群 · 情緒＋主題過濾
你是社群危機處理員。我會給你 100 則新功能上線當天的評論。請照以下步驟整理： 1. 情緒分類：正面 / 中性 / 負面 / 詢問類，統計每類件數與佔比 2. 主題抽取：把評論分成 6–8 個主題（功能讚 / UI 批評 / bug 回報 / 速度 / 定價 / 功能需求 / 其他） 3. 標紅警報：任何可能發酵成危機的評論（bug、退訂、媒體關注），列出原文 + 建議回覆方向 4. 選出每個主題的「代表性 top 3」評論（最能代表該類聲音的，不是按讚最多） 5. 輸出當日聲量速報：一句話總結 + 情緒圓餅 + 主題排行 + 紅色警報清單 + 建議 24h 內動作 輸出格式：先給老闆一頁摘要，再給附錄細節。語氣簡短，不要情緒化。
預期產出
情緒分布可能：正面 35 / 中性 20 / 負面 30 / 詢問 15。紅色警報會抓到 #42（bug）、#71（退訂）兩種。主題排行第一可能是「UI 批評」，這時要馬上決定：要不要出一份官方說明、要不要提供「回到舊版」開關。老闆版一頁摘要要能在 30 秒看懂當天狀況。
為何這樣整理：100 則評論人工看要 2 小時，還會被最激動的那幾則帶偏。AI 先情緒分類 + 代表性抽取後，人類只看主題 top 3，就能做出「今天要不要開危機會議」的判斷。這是把時間從 2 小時壓到 15 分鐘的關鍵動作。
(B) · Extend & Quiz

延伸挑戰與五題自我檢核

四題情境跑完後，要把能力帶回自己真正在做的專案，不然學會的只是「套 Prompt」。延伸挑戰預留 20–30 分鐘，挑 1–2 項做；Quiz 5 題每題 1–2 分鐘。本 section 做完再回頭跑一次情境演練裡改造過的 Prompt 套到自己資料上，整頁 96 分鐘的練習量就會自然完成。

●
延伸挑戰（20–30 分鐘，挑 1–2 項做）
  • 真實回饋跑一次：把你目前在做的專案中最近收到的 5 筆使用者回饋（可以是訊息、客訴、訪談摘要任何形式），挑情境 A–D 裡最接近的 Prompt 改造後丟給 AI 跑一次，看產出跟你自己腦中整理的版本差在哪。
  • 反推使用者畫像：從你整理出的主題 × 情緒矩陣，反推寫出 2–3 個「最常發聲者」的使用者畫像（職業 / 使用情境 / 最在意什麼 / 最痛的點），看能不能跟你原本設定的 target user 對上。對不上的時候要警覺：你可能在替「理想用戶」設計、但聲量來自「另一群人」。
  • 拆成三層清單：把整理結果拆成「立刻改 / 下版再說 / 不做」三欄，每項都要寫「不做」的理由（資源不足 / 違背定位 / 只有 1 人提 / 違背商業模式），這份「不做清單」比「要做清單」更能保護專案不被聲量牽走。
1面對 50 份問卷開放欄位的自由文字回饋，最該先做的動作是？
50 筆自由文字裡一定有大量重複與表達差異（「太慢」「lag」「要等很久」其實是同一件事）。分類 + 去重是必要前置——沒做這一步就改會變成「誰最大聲改誰的」，容易遺漏沉默多數。
2同一個痛點 3 位使用者都提到（語氣平淡），vs 1 位使用者強烈抱怨（語氣激動），該怎麼判斷優先級？
優先級不只看「人數」也要看「嚴重性」。常規情況 N 較高先做，但若單一回饋描述的是致命問題（資料消失、付款失敗、法規違規）則要獨立判斷，不能被聲量掩蓋。
3使用者說「我希望有 X 功能」，你該怎麼處理？
「想要 X」是表面解法，不是問題本身。使用者講的是他自己想到的解法，真正要挖的是底下那個 job-to-be-done（他想完成的任務是什麼）。常見結果是：對方要 X 但其實你家已經有 Y 可以解，或是 X 很貴但換成 Z 只要 10% 成本就能解掉 80% 痛點。
4看 Usability Testing 錄影時，最該關注的是什麼？
使用者會客氣，會幫你找台階下，口頭回饋容易正向偏誤。但身體不會說謊——嘆氣、滑鼠在畫面上游走找按鈕、按錯後停頓 3 秒、往後靠，這些訊號直接反映介面可用性的真實狀態。UT 最大的價值是看行為，不是聽意見。
5整理完一輪回饋後，該產出什麼樣的文件才算有用？
有用的回饋整理文件要能讓讀者「三分鐘決策」：這個要做嗎？做完長什麼樣？為何優先？before/after 描述讓工程師能照做、不再需要你口頭解釋；不做清單保護團隊不被聲量牽走。只列「要做」不寫「不做」，下次同樣的回饋進來還會被拉回討論。
