Part 1 · 解鎖 AI 工作模式
CH 1.2 · 3 小時

黃金提示詞公式：角色、任務、限制、輸出

一個好的提示詞不是靠運氣寫出來的——它有結構。這節課給你一個可以複用的四要素公式，讓你從「問 AI 試試看」進化到「確定能拿到我要的格式和品質」。

學習成果 · OUTCOMES
能用四要素結構（角色、任務、限制、輸出）改寫任何模糊需求
能指定 5 種常用輸出格式：表格、JSON、SOP、Email、條列
理解 few-shot 範例法與 Chain-of-Thought 的使用時機
(01) · The Problem

模糊指令 vs 精確指令：差距有多大？

「幫我寫一封 Email」和「你是一位專業 HR，請用正式但親切的語氣，寫一封邀請應徵者來二面的通知信，200 字以內，包含時間、地點、聯絡方式欄位」——同樣是請 AI 寫信，結果天差地遠。

AI 不會主動補充你沒說的事。你省略的每一個細節，AI 就會用「平均值」填補——給你最通用、最無特色的結果。要拿到你真正想要的輸出，你必須把需求說清楚。

模糊指令的結果
格式你沒說，AI 自己決定
語氣你沒說，AI 選最通用的
長度你沒說，AI 寫到它覺得夠
角色你沒設，AI 像個中性助理
每次結果都不同，難以複用
精確指令的結果
格式你指定，輸出可直接貼用
語氣你指定，符合使用情境
長度你指定，不多不少
角色你設定，專業度提升
同一個 Prompt 可穩定複用
(02) · The Formula

四要素公式：每個好 Prompt 的骨架

記住這個結構。你不需要每次都寫全四個要素，但每次寫不好的 Prompt，幾乎都是少了其中一個。

01
WHO · 角色
你是誰（AI 扮演的角色）
指定角色讓 AI 切換思維框架——律師、設計師、HR、行銷企劃的寫法完全不同。角色設定是控制語氣和專業度最簡單的方式。
範例
你是一位有 10 年經驗的行銷企劃，擅長 B2B 提案寫作。
02
WHAT · 任務
要做什麼（具體的動作）
用動詞開頭，說清楚是「摘要」「改寫」「分類」「生成」還是「比較」。任務越具體，輸出越可預期。
範例
請將以下會議記錄改寫成可以寄給客戶的正式摘要。
03
CONSTRAINT · 限制
不能做什麼 / 條件是什麼
限制是品質的護欄——長度上限、不能提到競品、必須用正體中文、不加任何評論。沒有限制的 AI 很容易過度發揮。
範例
不超過 150 字，不使用過度客套的開場白，避免模糊詞如「很多」「大家」。
04
FORMAT · 輸出格式
輸出長什麼樣
這是最常被忽略但影響最大的要素。表格？條列？JSON？Email 格式？沒有指定格式，AI 就自己選，而它選的格式往往不是你想要的。
範例
輸出格式為 Markdown 表格，欄位：主題｜決議｜負責人｜期限。
把四要素想成點餐：角色是廚師類型，任務是菜名，限制是過敏原，格式是裝盤方式——說得越清楚，端出來的菜越接近你要的。
Mini Practice · 迷你練習 01
約 10 分鐘
把你平常的模糊指令，用四要素改寫一次

選一個你本週實際用過、但結果不夠好的 Prompt。套用角色 → 任務 → 限制 → 輸出四要素重寫一次，比較兩版差異。

1
回想本週你給 AI 的一個指令（如「幫我寫活動文案」），把原句寫下來當作 v1。
2
依序補上角色（你是誰）、任務（具體動詞）、限制（字數 / 禁忌語 / 語氣）、輸出（格式），組成 v2。
3
在 ChatGPT 或 Claude 分別跑 v1 與 v2，把兩次輸出貼在一起比較。
4
標出 v2 比 v1 好在哪（格式？長度？語氣？專業度？）。

不知道從哪下手？把這份素材、v1、v2 各貼一次到 ChatGPT / Claude，就能立刻看到差別，再換成你的任務。
原始情境｜餐廳外送客訴
某公司 A 經理收到一則 Google 評論負評：「今天中午訂了三人套餐外送，等了 80 分鐘才到，雞腿已經涼掉、湯灑出來一半。店員打電話來只說會再出餐，沒解釋為什麼延遲。結果還是不完整的一餐。整個中午都毀了，不會再訂第二次。」A 經理想在 24 小時內公開回覆這則評論。
v1 · 模糊指令（壞示範）四要素 0/4
幫我回一下這個負評：{貼上方負評內容}
v2 · 套四要素改寫四要素 4/4
你是一位連鎖餐廳的客服主管（角色）。 請針對以下 Google 評論負評撰寫一則公開回覆（任務）。 限制： - 總長不超過 120 字 - 語氣誠懇但不卑躬，不使用「很抱歉造成您的不便」這類套話 - 須說明具體補救行動與後續改善做法 - 禁止承諾退款以外的任何金額賠償 輸出格式：直接可貼在 Google 評論區的純文字，結尾附店家署名「A 店 管理團隊」。 --- 【負評內容】 {貼上方負評內容}
預期輸出片段（對照用）
顧客您好，您提到的外送延誤與餐點溫度問題，我們已向門市確認為 4/12 中午備料排班誤判造成。本週起我們調整外送尖峰時段的出餐流程，並在每筆外送加上 45 分鐘抵達檢核。關於該次訂單，請您私訊訂單編號與聯絡方式，由店長直接處理退款與補寄。A 店 管理團隊
應產出 · OUTPUT
一份 v1 / v2 對照紀錄，加一句話總結：「這次最關鍵的要素是 ____」

(03) · Output Formats

六種你今天就用得上的輸出格式

指定輸出格式是讓 AI 輸出可以直接使用的關鍵。以下是最常用的格式，每種都有對應的指令寫法。

Markdown 表格
適合：比較、整理多筆資料、填寫式報告
輸出為 Markdown 表格，欄位：項目｜說明｜狀態
JSON
適合：資料結構化、後端串接、批量處理
以 JSON 格式輸出，key 為 name, date, amount
SOP 條列
適合：操作流程、工作指引、教學步驟
以編號步驟列出，每步驟不超過 2 句說明
Email 格式
適合：對外溝通、通知信、邀請函
包含：主旨行、稱謂、正文、署名，正式語氣
條列重點
適合：摘要、會議記錄、快速整理
最多 5 點，每點開頭用動詞，不超過 20 字
簡報頁綱
適合：提案、課程設計、報告架構
每頁：標題 + 3 個要點，共 8 頁，第一頁為封面
✅
進階技巧：如果你想要 JSON 但 AI 多說了廢話，在結尾加上「只輸出 JSON，不要其他說明文字」即可。這叫「輸出限制」，是格式控制的進階版。
Mini Practice · 迷你練習 02
約 10 分鐘
同一個任務，三種格式的效果差多少？

選一個你常做的整理任務（會議紀錄、競品比較、行程規劃），分別用 Markdown 表格、JSON、條列三種格式請 AI 輸出，觀察哪種最適合貼進你實際用的工具。

1
準備一段原始資料（會議逐字稿片段、產品規格列表都可以，約 200 字以內）。
2
跑三次，每次只改輸出格式指令：表格 / JSON / 條列，其他要素保持一致。
3
把三份輸出分別貼到你實際會用的工具（Notion、Excel、Slack、Email）看視覺效果。
4
選出「未來這類任務的預設格式」並說明原因。
範本素材 · Sample
下方是同一份會議逐字稿與三種格式的 Prompt。依序複製貼進 ChatGPT / Claude 跑三次，把三種結果並排看一次，差異就浮出來了。
原始資料｜某內訓團隊週會逐字稿（約 180 字）
A 主管：這週體驗設計課進度落後，預計 4/25 交付的學員手冊延到 4/30，B 負責。 B 助理：手冊延後我這邊能吸收，但週四前要拿到案例素材，不然不來不及。 A 主管：素材由 C 規劃師下週一整理完給 B。另外下個月開的 AI 班，報名表截止要改成 5/15，我請行政 D 更新官網和 Facebook。 C 規劃師：下個月班的行銷文案還沒定案，要不要今天先決定主視覺走向？ A 主管：主視覺下週二會議再討論，今天先確認：手冊延到 4/30、素材週一交、報名截止改 5/15、行銷主視覺下週二決。
Prompt 1｜輸出為 Markdown 表格適合貼 Notion / Google Sheets
你是會議紀錄助理。請把以下逐字稿整理成 Markdown 表格，欄位：類型（決議 / 待辦 / 待確認）｜內容｜負責人｜期限。只輸出表格，不要其他說明。 --- {貼上方逐字稿}
Prompt 2｜輸出為 JSON適合接 n8n / Make / API
你是會議紀錄助理。請把以下逐字稿輸出為 JSON，結構如下： { "decisions": [ { "content": "", "owner": "", "due": "" } ], "todos": [ { "content": "", "owner": "", "due": "" } ], "pending": [ { "content": "", "owner": "", "due": "" } ] } 日期請用 YYYY-MM-DD，未提到則留空字串。只輸出 JSON，不要其他說明文字。 --- {貼上方逐字稿}
Prompt 3｜輸出為三級條列適合貼 Slack / Email / Obsidian
你是會議紀錄助理。請把以下逐字稿整理成三級條列： - 第一級：決議事項 / 待辦事項 / 待確認事項 - 第二級：每個項目的內容（不超過 30 字） - 第三級：負責人與期限，格式為「（某某／日期）」 只輸出條列，不要前言或結論。 --- {貼上方逐字稿}
選用建議（格式 × 場景）
同一份素材跑完三版後，你會發現：表格最適合貼 Notion / Google Sheets，方便排序和篩選；JSON看起來最不「給人看」，但它是唯一能直接丟給 n8n 或 Make 做自動化的格式；條列最適合貼 Slack 或 Email，因為是純文字、閱讀時不需切換上下文。
應產出 · OUTPUT
一張「格式適用性比較表」，欄位：格式｜貼進哪個工具｜可讀性｜預設採用與否

(04) · Few-shot

Few-shot 範例法：給例子比給說明更有效

你有沒有試過把要求說得很清楚，但 AI 還是搞錯風格或格式？這時候，與其繼續用文字解釋，不如直接給一個你想要的例子——這就是 few-shot 提示法。

📋
Zero-shot（不給例子）：
「請用簡短、有行動感的語氣改寫以下句子。」

Few-shot（給 2 個例子）：
「請用以下風格改寫句子。
原句：我們計畫推出新產品。→ 新品即將上市，敬請期待。
原句：本次活動歡迎各界人士報名。→ 名額有限，立即搶先報名！
現在請改寫：本季業績表現優異。」

Few-shot 特別適合風格、語氣、格式有嚴格要求的任務。用文字描述「俏皮但不失專業」很難，但給三個例子，AI 馬上就懂了。

💡
最佳實務：2–5 個例子最有效。太少沒有規律，太多 AI 容易死板地複製例子。選你最想要的那種結果當例子。
Mini Practice · 迷你練習 03
約 15 分鐘
Zero-shot vs Few-shot：給例子有沒有差？

選一個有風格要求的任務（改寫標題、翻譯語氣、客服回應），先不給例子跑一次，再給 2 個例子跑一次，對照 AI 輸出的差異。

1
選一個有明確風格要求的任務，例如「把生硬標題改寫成俏皮但專業的版本」。
2
第一輪只用文字描述風格（zero-shot），跑 3 次取最好的。
3
第二輪附上 2 個你覺得理想的前後對照例子（few-shot），跑 3 次取最好的。
4
把兩輪的最佳輸出放一起，主觀評分哪個更貼近你心中的風格。
範本素材 · Sample
同一個任務：幫電子報寫五個主旨行。分別跑 zero-shot（只說風格）與 few-shot（直接給三個範例），立刻感受到「給例子」的差別。
任務背景｜某團隊每週發一封電子報，希望主旨行像「私信」不像「廣告」
本週電子報主題：週末加班文化 內容摘要：文章談某公司取消強制加班後，業績反而提升的現象，並整理三位管理者的對話紀錄。 希望主旨行風格：第一人稱、像朋友分享、避免感嘆號、避免「震驚」「秒懂」這類 clickbait 用語。
Prompt A · Zero-shot（只用文字描述風格）先跑這版
你是電子報主編。請為下列主題撰寫 5 個主旨行，風格要像朋友私信、第一人稱、避免感嘆號、避免「震驚」「秒懂」這類用語。 主題：週末加班文化——某公司取消強制加班後業績反而提升。 只輸出 5 個主旨行，每行獨立一行，不加編號。
Prompt B · Few-shot（加三個範例）再跑這版對照
你是電子報主編。請為下列主題撰寫 5 個主旨行，風格參考以下範例（都是該電子報過去表現最好的三封主旨）： 範例 1：這週讀過最有感的一句話 範例 2：我弄丟了 300 封 email 範例 3：老闆不在的那天 主題：週末加班文化——某公司取消強制加班後業績反而提升。 請模仿上述範例的語氣、長度與人稱，只輸出 5 個主旨行，每行獨立一行，不加編號。
預期差異（跑完兩版對照用）
Zero-shot 版本：即使明文禁用感嘆號，AI 仍容易寫出「原來加班不是必要的？」「一家公司的驚人轉變」這類廣告感句子，因為它沒「看過」你真正想要的風格。Few-shot 版本：AI 接收到「弄丟 300 封 email」「老闆不在的那天」這類具體場景句，會自動生成如「那個週六我沒進公司」「少加班的那一季我們多賺了」之類具體事件 + 第一人稱的結構——這就是範例帶來的風格模仿，文字說不清的部分例子能補起來。
應產出 · OUTPUT
一份實驗紀錄：原始指令 + 兩種方式的最佳輸出 + 主觀評分（1–10 分）+ 一句話心得
自我檢核 · CHECKLIST
(05) · Chain-of-Thought

讓 AI 先想再答：Chain-of-Thought 基礎

對於需要推理或判斷的任務，直接問 AI 答案往往不準。讓 AI 先說明它的思考過程，再給出結論，準確度會顯著提升——這就是 Chain-of-Thought（思維鏈）技巧。

1
加上「請逐步說明你的推理過程」——讓 AI 把判斷邏輯攤開，你也能驗證它的推理是否合理。
2
加上「先分析，再給結論」——迫使 AI 不要跳結論，對複雜問題特別有效。
3
加上「請先整理你的假設，再回答」——對有歧義的問題，先讓 AI 說清楚它的理解，再給答案。
⚠️
注意：Chain-of-Thought 會讓輸出變長、變慢，適合用在需要推理、準確度要求高的任務。只是要產出一段文字的場合，不需要用——那樣只是浪費 token。
✅
本節帶走產出：用四要素公式改寫 10 個工作中常用的指令，並至少做出 2 種輸出格式（建議：表格 + Email）。這是 PRAC 1.1 的前置作業。
